## Feature Specification: Smart Connections Similar — Semantic Similarity for a Single Note

**Feature Branch**: `026-smart-connections-similar`
**Created**: 2026-05-15
**Status**: Draft
**Input**: User description: "smart connections similar — a typed tool that returns the semantically nearest notes to a single named note, using the Smart Connections plugin's similarity index. Callers identify the source note either by an explicit locator (vault display name plus a vault-relative path or vault-root basename) or implicitly by which note the user currently has focused in the editor, and receive a structured list of nearest-neighbour matches with similarity scores. Plan-phase deliverables: update '.architecture/Obsidian CLI MCP - Architecture with Smart Connections.md' AND create ADR-013."

## Clarifications

### Session 2026-05-15

- Q: How should the wrapper handle Smart Connections plugin versions whose API surface diverges from the plan-stage-probed shape (older plugin versions missing methods the JS template depends on, or newer versions with renamed paths)? → A: Option C — **docs-only soft-pin**. The wrapper documents a minimum plugin version in `docs/tools/smart_connections_similar.md` (covered by FR-022 docs surface). No runtime version check at all. When the user's plugin version diverges enough that the API path the JS template depends on no longer resolves, the existing `SMART_CONNECTIONS_NOT_READY` code fires (the in-eval `try`/probe simply doesn't reach a callable function). No new envelope codes; no version-pinning coupling between wrapper releases and plugin point releases; no new FR for version assertion. Locks an addition to FR-022 (the docs MUST name the minimum probed plugin version), an addition to the assumptions block (minimum-version assumption), and an addition to the documented inherited limitations list (plugin-version-drift surfaces as `SMART_CONNECTIONS_NOT_READY`, not as a dedicated discriminator).
- Q: How should the wrapper handle non-finite or absent `score` values returned by the Smart Connections plugin (`NaN`, `Infinity`, `-Infinity`, `null`, `undefined`, or a missing `score` field on a match record)? → A: Option B — **drop bad entries silently**. The in-eval post-fetch pipeline applies a `Number.isFinite(score)` filter alongside the existing self-exclusion filter (FR-010); any match whose `score` is not a finite number is removed from the result list before the wrapper computes `count` and emits the envelope. The outer `count` reflects the post-filter length (consistent with FR-006a's cross-mode invariant — same filter applies in both default and count-only modes). Rationale: a NaN/null score means the plugin failed to compute similarity for that pair, not that the match is meaningful at score=0 or score=worst — silently dropping is the conservative thin-adapter behaviour, consistent with BI-025's defer-to-upstream stance and avoids inventing a value the plugin didn't produce. No new envelope code; the wrapper does NOT surface `INVALID_SCORE` or similar. Locks an addition to FR-009 (the finite-score filter), an Edge Cases addition under "CONTENT — score representation" (non-finite scores dropped silently), and a regression-test addition (handler test case where the eval envelope contains at least one non-finite-score match entry — wrapper drops it).
- Q: When a multi-vault caller passes `vault=<X>` but Obsidian is currently focused on a different vault `<Y>`, the eval reaches `<Y>`'s Smart Connections plugin instance (not `<X>`'s). Should the wrapper detect this mismatch, and if so, how should it be surfaced? → A: Option C — **in-eval detection, reuse `VAULT_NOT_FOUND` with `details.reason: "not-focused"` discriminator**. The JS template compares `app.vault.getName()` against the caller's `vault` argument inside the eval; mismatch emits an envelope error that the handler maps to `CLI_REPORTED_ERROR(details.code = "VAULT_NOT_FOUND", details.reason = "not-focused")`. The same top-level code `VAULT_NOT_FOUND` covers both "the vault name does not exist in the host's registry" (existing 011-R5 inspection clause, `details.reason: "unknown"` or absent) and "the vault name exists but is not the focused vault" (new in-eval detection, `details.reason: "not-focused"`). Rationale: detection is cheap (one comparison inside eval) and gives multi-vault callers actionable signal; using `details.reason` as a sub-discriminator avoids adding a 9th top-level `details.code` to the roster and preserves the eleven-tool zero-new-top-level-codes streak; the existing test surface for unknown-vault routing extends with one new sub-case. Locks the elevation of "multi-vault" from a documented inherited limitation to a STRUCTURED ERROR, an addition to FR-017 (the sub-reason discriminator), an addition to the Edge Cases "LOCATOR — multi-vault" section (now structured-error not inherited-limitation), a reduction in the inherited-limitations list from five to four entries (FR-022 + SC-020), and a new acceptance scenario in US1 (vault-mismatch yields the structured error).
- Q: When multiple error conditions fail simultaneously for a single call (e.g. wrong vault focused AND file is `.canvas` AND plugin not installed), which discriminator surfaces first? → A: Option A — **outer-to-inner / cheapest-first precedence**. Specific mode: `VAULT_NOT_FOUND(unknown)` (cli-adapter 011-R5 inspection BEFORE eval) → `VAULT_NOT_FOUND(not-focused)` (in-eval `app.vault.getName()` comparison) → `SMART_CONNECTIONS_NOT_INSTALLED` (presence of `app.plugins.plugins["smart-connections"]`) → `FILE_NOT_FOUND` (target resolution) → `NOT_MARKDOWN` (file extension check) → `SMART_CONNECTIONS_NOT_READY` (similarity-API callable check) → `SOURCE_NOT_INDEXED` (source entry lookup) → success. Active mode skips the vault step entirely (active mode forbids `vault` arg per ADR-003): `SMART_CONNECTIONS_NOT_INSTALLED` → `NO_ACTIVE_FILE` (focused-file resolution) → `NOT_MARKDOWN` → `SMART_CONNECTIONS_NOT_READY` → `SOURCE_NOT_INDEXED` → success. Rationale: matches the user's mental layering (which vault → which file → which plugin → which source); each check that fires is independently meaningful (an agent that gets `VAULT_NOT_FOUND` doesn't waste effort on file paths in the wrong vault); cheap string comparisons run before expensive cache lookups; `VAULT_NOT_FOUND(unknown)` is naturally first because the cli-adapter's 011-R5 inspection intercepts the eval before it runs. Locks a new FR (FR-017b) specifying the precedence chain, a regression-test addition (one fixture per pair of compound-failure conditions verifying the earlier-priority discriminator fires), and a docs surface addition (FR-022 docs MUST document the precedence order so agent debugging is deterministic).
- Q: What is the long-term relationship between `.architecture/Obsidian CLI MCP - Architecture.md` (the base file, canonical pre-026) and `.architecture/Obsidian CLI MCP - Architecture with Smart Connections.md` (the new file FR-030 targets)? → A: Option C — **coexist as a snapshot**. The base `.architecture/Obsidian CLI MCP - Architecture.md` remains the canonical, forward-going architecture document for non-plugin-backed wrapper architecture and is rolled forward by future plugin-cohort BIs alongside their own BI-specific snapshot files. The new `.architecture/Obsidian CLI MCP - Architecture with Smart Connections.md` is FROZEN as a BI-026-time snapshot — a historical artefact capturing the architecture as of the moment Smart Connections support landed. Future plugin BIs (e.g. `smart_connections_blocks` from 027-NNN, `smart_connections_chat` from 028-NNN, or unrelated plugin wraps from 029-NNN+) do NOT update this file; they update the base file AND create their own BI-time snapshot file if their feature input requests one. The new file's content at end of plan covers exactly the FR-030 scope (the new typed-tool surface, the plugin-namespace tool-naming convention from ADR-013, the plugin-as-runtime-dependency pattern, the eight-entry failure-mode roster, the four documented inherited limitations, the architectural delta of "plugin-API-routing-via-eval"). Rationale: the new file's name explicitly references "with Smart Connections" — semantically a slice, not the canonical doc; preserves the single-source-of-truth pattern for the canonical architecture document; gives future BIs a frozen reference point ("this is what the architecture looked like when SC landed") without imposing the maintenance cost of a parallel forward-going doc; the base file remains the one place to roll plugin-cohort architecture forward. Locks a clarification of FR-030 (snapshot-not-supersede semantics), a new FR-030a (future plugin BIs update the base file, not this snapshot file), and a corresponding SC update (SC-028's commitment that the snapshot file is complete at end of plan AND the base file is rolled forward in the same plan commit).
- Q: **(live-probe-driven amendment)** Live probe on 2026-05-15 contradicted Q3's premise — `obsidian vault=<name> eval code="…"` DOES route the eval to the named vault's `app` instance (`app.vault.getName()` returns the requested vault name, not the focused-window vault name; confirmed across three open vaults: TestVault-Obsidian-CLI-MCP, The Setup, Ways of Working). The "wrong-vault" detection FR-017a was designed for does NOT need to fire when the requested vault is currently open. How should FR-017a be revised? → A: Option B — **repurpose FR-017a for vault-not-currently-open detection**. The discriminator pattern from Q3 (reusing `details.code = "VAULT_NOT_FOUND"` with a `details.reason` sub-discriminator instead of adding a 9th top-level code) is preserved, but the failure mode it detects is replaced. The new failure: vault is REGISTERED in `obsidian vaults` output BUT the vault window is NOT currently open in Obsidian, so eval has no `app` instance to route to. The wrapper surfaces `CLI_REPORTED_ERROR(details.code = "VAULT_NOT_FOUND", details.reason = "not-open")`. The exact CLI behaviour when calling eval against a closed registered vault was NOT directly probable (all three test-rig vaults were open at probe time, and asking the user to close one mid-conversation interrupts workflow); the precise detection mechanism (does eval hang, error out, transparently open the vault, or emit a specific string?) is locked at plan-stage probe per FR-024. The contract shape is fixed by this clarification; the mechanism is the plan's call. Locks an FR-017a rewrite (from focused-vault-mismatch to vault-not-open), an FR-017b precedence-chain update (the rewritten FR-017a still sits between cli-adapter-011-R5 `VAULT_NOT_FOUND(unknown)` and `SMART_CONNECTIONS_NOT_INSTALLED`), Edge Cases LOCATOR rewrite (replace "multi-vault and which-vault-the-plugin-sees" → "vault-registered-but-not-open"), US1 scenario 6b rewrite (closed-vault scenario instead of focused-mismatch scenario), SC-011a + SC-011b update, AND **adds multi-vault routing back to the documented inherited limitations list as #5** (the routing itself works correctly per live probe, but the limitation that "the wrapper cannot disambiguate basename across vault windows" remains — the basename lookup happens inside the requested vault's `app.metadataCache`, not across vaults).
- Q: **(live-probe-driven amendment to grilling Q3)** Live probe on 2026-05-15 contradicted the grilling-session Q3 lock (source-only matches for v1). `find_connections()` on a source item returns BLOCK-LEVEL matches by default — keys carry `#Heading#Subsection` fragments (e.g. `"400-Architecture/Architecture Overview.md#---frontmatter---"`, `"400-Architecture/Components/MCP Layer.md#MCP Layer#Current implementation"`). Even with `limit: 50`, zero source-level matches appeared in the result set; the plugin's `exclude_blocks: true` filter was probed and the result was still 0 source-level matches across the test vaults. The grilling Q3 assumption that source-mode was the natural plugin output is FALSE. How should v1's match granularity be re-cut? → A: Option B — **switch v1 to block-level matches**, the plugin's natural output. The per-match shape becomes `{path, headingPath, score}` where `path` is the source file's vault-relative path (everything before the first `#`, e.g. `"400-Architecture/Components/MCP Layer.md"`) and `headingPath` is an ordered array of heading segments after the first `#` (split on `#`, e.g. `["MCP Layer", "Current implementation"]`; empty array `[]` for source-level matches if any appear; literal-preserved `["---frontmatter---"]` for frontmatter-block matches — the wrapper does NOT normalise the plugin's sentinel). Sort order is `score` descending, with secondary tiebreak `path` byte-compare ascending, tertiary tiebreak `headingPath.join("#")` byte-compare ascending. Self-exclusion (FR-010) is now keyed on the SOURCE path component (`path.split('#')[0]`), not the full block key — blocks WITHIN the source note are still excluded along with the source itself. Tool name stays `smart_connections_similar` (the granularity is the natural shape, not a separate tool). Rationale: matches the plugin's actual return shape (eliminates the lossy block-to-source dedupe transform that would collapse per-block scores into a single source-level score); aligns the wrapper's contract with the Smart Connections sidebar UI users see in Obsidian (which shows block matches); the agent's "what notes are semantically related" query is still answered — agents collapse to source-level client-side via `path` field grouping if they only want source-level. The grilling-session Q6 lock (plugin-namespace tool name) AND ADR-013 (plugin-namespace convention) are UNAFFECTED — the tool name and convention apply regardless of granularity. Locks: rewrite of FR-007 (per-match shape gains `headingPath`); FR-008 (sort order gains tertiary tiebreak on `headingPath`); FR-010 (self-exclusion keyed on source path); all four user stories' acceptance scenarios updated for the new per-match shape; Edge Cases CONTENT — path string format rewritten; Key Entities "Similarity match" definition gains `headingPath`; the "block-level matches deferred to future smart_connections_blocks tool" item REMOVED from the inherited-limitations list (now four entries, was five); out-of-scope block-level-matching item REMOVED from Assumptions OOS block; new SCs SC-007a (headingPath shape) and SC-010a (block-level fixture coverage in FR-025); FR-024 plan-stage probe spec updated (probe MUST confirm whether `find_connections` ever returns source-level entries — if so, how the wrapper emits them with `headingPath: []`).

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Semantic similarity listing for a named note (Priority: P1)

An agent (or a human caller through an MCP-aware client) needs the semantically nearest notes to one specific note — for example "what notes in the vault discuss similar topics to this project brief so I can compile a literature review?", "find related working notes I might have forgotten about", or "before I rewrite this index page, show me the semantic neighbours so I can confirm the page still covers the right cluster." The caller identifies the source note in specific mode by either a vault-relative path or a vault-root basename, optionally scoping the call to a named vault, and optionally setting a top-N cutoff. The tool returns the ordered list of nearest-neighbour notes, each entry carrying the neighbour's vault-relative path and its similarity score, sorted from most-similar to least-similar.

**Why this priority**: This is the dominant use case and the entry point that justifies the feature's existence. Today an agent that wants to find semantically related notes has no efficient discovery path through the typed-tool surface — it must either read the full note body (potentially tens of kilobytes) and run a client-side embedding pipeline (requires the agent to host its own embedding model, duplicates work the Smart Connections plugin has already done, and produces results inconsistent with what the user sees in the Smart Connections sidebar in Obsidian), or fall back to keyword search via `obsidian_exec` (which misses semantic relationships text search can't see — e.g. notes about the same topic written with different vocabulary). A single typed call turns "what notes are related to this one?" into a deterministic answer that mirrors what the user sees in the Smart Connections sidebar. Without this story the feature offers no value; every other story refines this primary read path.

**Independent Test**: With the Smart Connections plugin installed and initial indexing completed against a test vault containing a known set of topically-clustered notes (e.g. five notes about "machine learning", three notes about "cooking", two notes about "travel"), invoke the tool against one of the machine-learning notes in specific mode with `limit: 10`. Assert the response carries `count` matching the number of returned matches, each match's `path` referring to a vault-existing note (and NOT the source note itself), the highest-scoring matches in the result list are the other machine-learning notes, scores are monotonically non-increasing across the list, and the path-string format matches the vault-relative-with-`.md`-extension convention shared with the existing `files` typed tool. Independently testable in isolation; nothing in P2/P3 is required.

**Acceptance Scenarios**:

1. **Given** a vault with the Smart Connections plugin installed and initial indexing completed, AND a source note at vault-relative path `Topics/ML.md` whose semantic neighbours (per the plugin's similarity index) are blocks within `Topics/Neural-Networks.md`, `Topics/Gradient-Descent.md`, `Projects/ML-Project.md` (in decreasing similarity order; live probe on 2026-05-15 confirmed `find_connections()` returns BLOCK-level matches by default), **When** the agent invokes the tool with `{ target_mode: "specific", vault: "Demo", path: "Topics/ML.md" }`, **Then** the response carries the top-K matches (K being the default limit of 20, or fewer if the plugin returns fewer above its internal threshold) with each entry shaped as `{path, headingPath, score}`: `matches[0].path === "Topics/Neural-Networks.md"` with `matches[0].headingPath` an ordered array of the heading segments locating the matched block within that file (e.g. `["Neural Networks", "Backpropagation"]` for a block under H1 "Neural Networks" → H2 "Backpropagation"; `[]` if the plugin returns a source-level match for that file; `["---frontmatter---"]` for a frontmatter-block match); subsequent `matches[i].path` may be `"Topics/Neural-Networks.md"` again (a different block within the same file) OR a different file; each `match.score` a numeric value reflecting the plugin's similarity calculation; and `matches[0].score >= matches[1].score >= matches[2].score` (descending order; secondary tiebreaker `path` byte-compare ascending; tertiary tiebreaker `headingPath.join("#")` byte-compare ascending per FR-008).
2. **Given** the same vault and source note AND a `limit: 5` parameter, **When** the agent invokes the tool, **Then** the response carries at most five matches in the `matches` array (the top five by plugin similarity score).
3. **Given** a source note that exists at the named path AND the basename `ML` resolves unambiguously inside the vault, **When** the agent invokes the tool with `{ target_mode: "specific", vault: "Demo", file: "ML" }` (basename without the `.md` extension), **Then** the response is structurally equivalent to the path-based call against the same source — same `count`, same per-match `path` and `score` values, same order. The two locator forms are interchangeable when the basename resolves unambiguously.
4. **Given** a source note that exists in the vault BUT the Smart Connections plugin has no embedding for it (e.g. the note was created after the last indexing pass, OR the note is in a folder the plugin's settings exclude), **When** the agent invokes the tool, **Then** the response is a structured error with `details.code = "SOURCE_NOT_INDEXED"` rather than an empty success — distinct from a file-not-found error because the file DOES exist in the vault; the failure is specifically that Smart Connections has not indexed it.
5. **Given** a locator that does not resolve to any note in the vault (an unknown `path` or an unknown `file` basename), **When** the agent invokes the tool, **Then** the response is a structured error with `details.code = "FILE_NOT_FOUND"` identifying the missing file; the response is NOT an empty success and NOT a `SOURCE_NOT_INDEXED` error (which would imply the file exists).
6. **Given** a vault display name the host does not recognise (e.g. `vault: "Unknown"`), **When** the agent invokes the tool, **Then** the response is a structured error with `details.code = "VAULT_NOT_FOUND"` (and `details.reason` absent or `"unknown"`) via the cli-adapter's 011-R5 inspection clause — the call MUST NOT silently return an empty success and MUST NOT silently route to the focused vault. Matches BI-014 / BI-015 / BI-025 eval-cohort behaviour.
6b. **Given** a vault display name the host DOES recognise in `obsidian vaults` output (`vault: "Other"`) BUT the corresponding vault window is NOT currently open in Obsidian, **When** the agent invokes the tool, **Then** the response is a structured error with `details.code = "VAULT_NOT_FOUND"` AND `details.reason = "not-open"`. The wrapper detects the closed-vault case via the empty-stdout + exit-0 + `vault=` + known-vault signature locked by live probe on 2026-05-15 (the CLI emits empty stdout with exit 0 for the first eval call against a closed registered vault AND transparently opens the vault as a side effect). The error response signals to the agent that the vault is now opening; a retry of the same call after a brief delay will likely succeed. The wrapper MUST NOT auto-retry on the caller's behalf, MUST NOT hang, MUST NOT silently route to a different vault's Smart Connections index, AND MUST NOT classify the empty-stdout case as a generic JSON parse failure (which would lose the actionable `not-open` signal). Live probe on 2026-05-15 confirmed `vault=<name> eval` routes correctly when the vault IS open — `app.vault.getName()` returns the requested vault name across the three open test vaults.
7. **Given** the source note is a `.canvas` file, a PDF, or any non-Markdown file the Smart Connections plugin does not embed (the free plugin's default scope is `.md` only), **When** the agent invokes the tool with that locator, **Then** the response is a structured error with `details.code = "NOT_MARKDOWN"` rather than a `SOURCE_NOT_INDEXED` error.

### User Story 2 — Semantic similarity listing for the focused note (Priority: P1)

An agent acting on the user's working context needs the semantic neighbours of whichever note the user currently has open in the editor, without restating the path. The caller invokes the tool in active mode. The tool resolves the focused note implicitly and returns its semantic neighbours. When no note is currently focused (a fresh Obsidian window, the user switched to a non-note view, etc.) the tool surfaces a structured no-active-file error rather than silently returning an empty list.

**Why this priority**: Active-mode is the canonical pattern shared by every per-file typed tool in the project (`read`, `outline`, `read_heading`, `read_property`, `delete`, `rename`, `write_note`, `set_property`, `find_by_property`, `links`). Omitting it would create an inconsistency in the typed surface AND would force agents acting on the user's working context to first invoke a separate "what file is focused?" lookup. Pairs equally with US1.

**Independent Test**: With Obsidian open, the Smart Connections plugin installed, indexing complete, and a known note focused, call the tool with `{ target_mode: "active" }`. Assert the response covers the focused note's semantic neighbours. Switch focus away (open a non-note view or close all notes) and call again with `{ target_mode: "active" }`. Assert the second response is a structured `NO_ACTIVE_FILE` error rather than an empty success.

**Acceptance Scenarios**:

1. **Given** the user has a note focused in Obsidian AND the Smart Connections plugin is installed and ready AND that note's semantic neighbours include a known set of related notes, **When** the agent invokes the tool with `{ target_mode: "active" }`, **Then** the response covers exactly that focused note's semantic neighbours — same `count`, same per-match shape, same descending-score order as if the caller had supplied the focused note's vault-relative path in specific mode.
2. **Given** no note is currently focused in Obsidian (the workspace shows an empty pane, a non-note view, or the help panel), **When** the agent invokes the tool with `{ target_mode: "active" }`, **Then** the response is a structured error with `details.code = "NO_ACTIVE_FILE"` — NOT an empty success, NOT a generic CLI failure.
3. **Given** active mode AND any per-file locator field accidentally supplied alongside it (e.g. `{ target_mode: "active", file: "Other" }`), **When** the agent invokes the tool, **Then** the call fails validation BEFORE any underlying CLI invocation — active mode forbids the per-file locator fields per the project's existing target_mode discriminator contract (ADR-003).
4. **Given** active mode AND the focused note is a `.canvas` file (Obsidian Canvas) or any non-Markdown file the Smart Connections plugin does not embed, **When** the agent invokes the tool, **Then** the response is a structured error with `details.code = "NOT_MARKDOWN"` — symmetric with the specific-mode rejection in US1 scenario 7.

### User Story 3 — Validation rejects malformed inputs at the boundary (Priority: P1)

An agent (or a misbehaving caller) submits an input shape that violates the tool's contract. The tool MUST reject the call at the validation boundary, before any underlying CLI invocation occurs, and MUST surface a structured validation error that names the offending field.

**Why this priority**: Validation is the safety contract for every typed tool in this project, and zod-as-source-of-truth is a constitutional requirement. Without it, malformed callers reach the CLI and produce undefined or harmful behaviour. Independently testable because every validation case can be exercised with a mock/spy on the CLI dispatcher to assert the dispatcher was never called.

**Independent Test**: For each invalid input shape, call the tool with a CLI dispatcher spy. Assert the call rejects with a structured `VALIDATION_ERROR` AND that the dispatcher was never invoked. No real CLI or vault required.

**Acceptance Scenarios**:

1. **Given** specific mode WITHOUT a vault parameter (e.g. `{ target_mode: "specific", path: "Topics/ML.md" }`), **When** the agent invokes the tool, **Then** the call fails validation (vault is mandatory in specific mode per ADR-003); no CLI call is made.
2. **Given** specific mode WITHOUT any per-file locator (neither `file` nor `path` supplied), **When** the agent invokes the tool, **Then** the call fails validation (the XOR locator rule from ADR-003); no CLI call is made.
3. **Given** specific mode with BOTH `file` AND `path` supplied, **When** the agent invokes the tool, **Then** the call fails validation (the XOR locator rule rejects supplying both); no CLI call is made.
4. **Given** any input with an unknown top-level key (e.g. `{ target_mode: "specific", vault: "Demo", path: "Topics/ML.md", threshold: 0.7 }` — `threshold` is not part of this tool's surface), **When** the call is forwarded by an MCP client that does NOT strip unknown keys, **Then** the server-side validation fails; no CLI call is made.
5. **Given** `total` set to a non-boolean value (e.g. the string `"true"`), **When** the agent invokes the tool, **Then** the call fails validation; no CLI call is made.
6. **Given** `limit` set to a value outside the `1..100` range (e.g. `0`, `-5`, `101`, `1000`) OR to a non-integer (e.g. `5.5`, `"20"`), **When** the agent invokes the tool, **Then** the call fails validation; no CLI call is made.
7. **Given** `path` containing path-traversal characters (e.g. `"../../etc/passwd"`) or shell metacharacters, **When** the agent invokes the tool, **Then** the call is rejected — either at the input-validation boundary or by the underlying vault-access layer; in both cases a structured error reaches the caller and no escape from the vault occurs.
8. **Given** an input shape where `target_mode` is missing, set to an unknown value (e.g. `"focused"`, `"current"`), or set to a non-string value, **When** the agent invokes the tool, **Then** the call fails validation; no CLI call is made.

### User Story 4 — Count-only mode skips the per-match payload (Priority: P2)

An agent only needs to know how many neighbours the Smart Connections plugin returns for a note — for a structural audit ("how well-connected is this note semantically?"), a heuristic ("does this index note still cluster with its expected neighbourhood?"), or a pre-flight check before deciding whether a full listing is worthwhile. The caller invokes the tool with `total: true`. The response carries the `count` only — the per-match list is empty. The tool MUST NOT pay the per-match payload cost when the caller asks only for a count.

**Why this priority**: Count-only mode matches the established `total: true` precedent shared by every other enumerating typed tool in the project (`files`, `outline`, `properties`, `links`). It is an optimisation, not a requirement of the core read path; agents that always set `total: false` lose no functionality. Independently testable from US1 with a separate single-line input variation.

**Independent Test**: Author a fixture vault where one note has a known number of semantic neighbours above the plugin's score threshold and another note has zero (or near-zero — an outlier semantic-island note). Call the tool with `total: true` against each. Assert the first response carries `count: N` and an empty `matches` list; assert the second response carries `count: 0` and an empty `matches` list. Both succeed.

**Acceptance Scenarios**:

1. **Given** a source note whose Smart Connections similarity query returns exactly seven neighbours under the default `limit`, **When** the agent invokes the tool with `{ target_mode: "specific", vault: "Demo", path: "Topics/ML.md", limit: 20, total: true }`, **Then** the response carries `count: 7` and `matches: []` (empty list).
2. **Given** the same source note AND a `limit: 3` parameter AND `total: true`, **When** the agent invokes the tool, **Then** the response carries `count: 3` and `matches: []` — the `limit` parameter caps the count even in count-only mode (the count reflects what the agent would see in default mode at the same `limit`, NOT the unbounded similarity-query result count).
3. **Given** count-only mode AND active mode AND no focused note, **When** the agent invokes the tool with `{ target_mode: "active", total: true }`, **Then** the response is the same structured `NO_ACTIVE_FILE` error as in default mode (US2 scenario 2) — count-only mode does NOT suppress error paths; it only suppresses the per-match payload on success.
4. **Given** count-only mode AND an unresolved locator in specific mode, **When** the agent invokes the tool, **Then** the response is the same structured `FILE_NOT_FOUND` error as in default mode (US1 scenario 5) — count-only mode never silently masks an unresolved locator with `count: 0`. Symmetric: count-only mode does NOT mask `SOURCE_NOT_INDEXED`, `NOT_MARKDOWN`, `SMART_CONNECTIONS_NOT_INSTALLED`, or `SMART_CONNECTIONS_NOT_READY` either.

### User Story 5 — Documentation surface for the typed tool (Priority: P2)

An operator or agent inspects the project's progressive-disclosure help facility to understand how the semantic-similarity tool works, what the plugin prerequisites are, and how to interpret the `score` field. ### Edge Cases

The implementation MUST handle, document, or explicitly defer each of the following observable shapes.

**CONCURRENCY**

- The Smart Connections plugin's embedding index may be updated (notes added, removed, or re-embedded) between the moment the request is received and the moment the similarity query executes. The response reflects the plugin's index state at the moment of query; there are no stale-read or partial-update guarantees beyond that. Documented as a known limitation.

**PLUGIN LIFECYCLE — plugin not installed**

- If the Smart Connections plugin is not installed (or not enabled) in the vault that processes the request, the tool MUST surface a structured error with `details.code = "SMART_CONNECTIONS_NOT_INSTALLED"`. Remediation: install/enable the plugin via Obsidian's community-plugins settings. This is distinct from `VAULT_NOT_FOUND` (vault doesn't exist) and `FILE_NOT_FOUND` (file doesn't exist).

**PLUGIN LIFECYCLE — plugin not ready**

- If the Smart Connections plugin is installed but its embedding index is not yet built (first-run users sit in this state for minutes-to-hours depending on vault size and the configured embedding model), the tool MUST surface a structured error with `details.code = "SMART_CONNECTIONS_NOT_READY"`. Remediation: wait for indexing to complete (visible in the plugin's status UI). This is distinct from `SOURCE_NOT_INDEXED` (plugin is ready but this specific note has no embedding).

**PLUGIN LIFECYCLE — source note not in plugin's index**

- A source note that exists in the vault as a `.md` file but for which the Smart Connections plugin has no embedding entry (e.g. note created after the last indexing pass, or in a folder the plugin excludes via its `exclude_folders` setting, or never embedded because the indexing run hasn't reached it) MUST yield a structured error with `details.code = "SOURCE_NOT_INDEXED"`. The wrapper does NOT trigger embedding from this call (read-only contract). Remediation: trigger a re-index from the plugin's UI or wait for the next indexing pass.

**CONTENT — non-`.md` source**

- The Smart Connections plugin (free version, default settings) only embeds `.md` files. A source locator pointing at a `.canvas` file, a PDF, an image, or any other non-Markdown extension MUST yield a structured error with `details.code = "NOT_MARKDOWN"` rather than a `SOURCE_NOT_INDEXED` error or an empty success. The wrapper enforces this in-eval via an `f.extension === 'md'` guard before reaching the plugin; parity with BI-025 `links`'s NOT_MARKDOWN contract.

**CONTENT — empty result list**

- A source note that the Smart Connections plugin has indexed but for which the plugin's similarity query returns zero neighbours (an extreme semantic-outlier note, OR a vault with only one note) MUST yield `{ count: 0, matches: [] }` — no error. Both default and count-only modes share this contract.

**CONTENT — self-exclusion**

- The source note itself AND any block inside the source note MUST NOT appear in the `matches` list. Smart Connections' own `find_connections()` implementation excludes the source by default at the block-key level; the wrapper enforces this as defence-in-depth via an in-eval `.filter(m => sourceKey(m) !== sourcePath)` post-fetch — where `sourceKey(m)` is the SOURCE path component of the match key (everything before the first `#`). This excludes both source-level self-matches (`headingPath: []`) AND block-level matches inside the source note (`path === sourcePath` with non-empty `headingPath`). If a future plugin version changes the self-exclusion default, the wrapper's filter still holds.

**CONTENT — score representation**

- The `score` field for each match is the raw numeric value the Smart Connections plugin returns from its similarity calculation. The wrapper does NOT clamp the value to `[0, 1]`, does NOT normalise across queries, and does NOT round to a fixed precision. The score's range and semantics are embedding-model-dependent (transformers.js default model produces approximately cosine-similarity values in `[~0, 1]`; OpenAI ada-002 produces `[0, 1]`; other models may produce different ranges including occasional slightly-negative values). Callers MUST NOT assume a model-independent score scale; the documented inherited limitation surfaces this.

**CONTENT — non-finite scores dropped silently**

- Any match record whose `score` field is not a finite JavaScript number (`NaN`, `Infinity`, `-Infinity`, `null`, `undefined`, or a missing field — `Number.isFinite(score)` returns false) MUST be silently removed from the result list by the wrapper's in-eval post-fetch filter (per the 2026-05-15 clarifications session Q2). The wrapper does NOT coerce non-finite scores to a sentinel value, does NOT surface a dedicated envelope code for the filter event, and does NOT emit any signal in the response about how many entries were dropped. Rationale: a non-finite score means the plugin failed to compute similarity for that pair; dropping is the conservative thin-adapter behaviour and avoids inventing a value the plugin didn't produce. The outer `count` reflects the post-filter length consistently in both default and count-only modes.

**CONTENT — sort order**

- The `matches` list MUST be in stable, deterministic order across repeated calls against an unchanged index state. The primary sort is `score` descending (highest similarity first — `matches[0].score >= matches[1].score >= ... >= matches[N].score`). The SECONDARY tiebreaker for equal scores is `path` byte-compare ascending (the simple `<`/`>` operator order on the JavaScript string, NOT locale-aware `localeCompare`). The TERTIARY tiebreaker for equal `path` (two blocks from the same source note tied on score) is `headingPath.join("#")` byte-compare ascending. The wrapper does NOT expose alternative sort-order parameters in this feature; callers needing a different order re-sort client-side.

**CONTENT — path string format**

- Each match's `path` field MUST be the SOURCE FILE's vault-relative path with the `.md` extension preserved (e.g. `"Folder/Note.md"`) — extracted by taking everything before the FIRST `#` in the plugin's match key. Parity with the `files` typed tool's output. The path is directly pasteable into any other typed tool's `path=` field for follow-on reads. The wrapper does NOT strip the extension, prepend a slash, or otherwise transform the path.
- Each match's `headingPath` field MUST be an ordered array of heading-path segments locating the matched block within the source file — split on `#` AFTER the first `#` in the plugin's match key. Examples: plugin key `"Folder/Note.md#H1#H2"` → `headingPath: ["H1", "H2"]`; plugin key `"Folder/Note.md#---frontmatter---"` → `headingPath: ["---frontmatter---"]` (frontmatter sentinel preserved verbatim); plugin key `"Folder/Note.md"` with no `#` → `headingPath: []` (source-level match, no block fragment). The wrapper does NOT normalise the frontmatter sentinel, does NOT validate that heading segments correspond to real headings in the source file's current state, and does NOT strip whitespace from segments. Multiple matches MAY share the same `path` with different `headingPath` values (different blocks within the same source note).

**CONTENT — very large match lists and the underlying execution layer's output cap**

- The underlying execution layer enforces a 10 MiB output cap on a single CLI invocation (inherited from feature 003). A query whose serialised match list would exceed this cap (essentially impossible at the locked `limit: 1..100` ceiling — 100 matches × ~80 bytes each ≈ 8 KiB, four orders of magnitude below the cap) MUST nonetheless produce a structured `CLI_NON_ZERO_EXIT` (output-cap kill) if it ever fires, never a silent truncation.

**LOCATOR — unresolved locator**

- A `file` or `path` value that does not resolve to any note in the vault MUST yield a structured error with `details.code = "FILE_NOT_FOUND"` rather than an empty success (`count: 0, matches: []`) AND rather than a `SOURCE_NOT_INDEXED` error (which would imply the file exists). Count-only mode (`total: true`) does NOT suppress this error.

**LOCATOR — basename collision**

- A `file` value that is a basename matching more than one note in the vault MUST resolve through the host's existing wikilink-resolution semantics; the wrapper does NOT impose a separate disambiguation contract. Whether the host returns one specific file's neighbours or a structured error is the host's choice; the wrapper inherits whatever surface the host produces.

**LOCATOR — unknown vault**

- A `vault` display name that is not registered MUST surface as a structured error with `details.code = "VAULT_NOT_FOUND"` via the cli-adapter's 011-R5 unknown-vault response-inspection clause. Cohort: matches BI-014 / BI-015 / BI-025 (eval-driven structured-error cohort). Multi-vault callers MUST supply a registered display name. The wrapper does NOT silently route to the focused vault when an unrecognised name is supplied.

**LOCATOR — vault registered but not currently open**

- The Obsidian CLI's `vault=<name>` argument on `eval` ROUTES THE CALL CORRECTLY to the named vault's `app` instance when that vault is currently open in Obsidian (verified by live probe on 2026-05-15 across three open vaults — `app.vault.getName()` inside the eval returns the requested vault name, not the focused-window vault name). The prior premise that "vault= is silently honoured-as-noop by eval" (carried forward in the spec drafts from BI-014 / BI-015 / BI-025) was incorrect for this BI's probed CLI version. **When the requested vault is REGISTERED in `obsidian vaults` output BUT the corresponding vault window is NOT currently open in Obsidian**, the live probe on 2026-05-15 (after the user closed two registered vaults) confirmed the CLI behaviour: the FIRST eval call against the closed vault returns **EMPTY stdout with exit 0** (no `=> ` prefix, no result, no error string, no hang) AND transparently OPENS the vault as a side effect; the SECOND eval call against the same vault (now opened by the first call) reaches the now-open `app` instance and works normally. The cli-adapter's existing 011-R5 inspection clause does NOT fire (no `Vault not found.` string in the empty output) — the closed-vault case requires the new detection mechanism specified in FR-017a (empty stdout + exit 0 + `vault=` supplied + vault present in `obsidian vaults`). The wrapper surfaces `CLI_REPORTED_ERROR(details.code = "VAULT_NOT_FOUND", details.reason = "not-open")` — distinct from `details.reason: "unknown"` (vault not registered, classic 011-R5 path). Agents seeing `not-open` MAY retry after a brief delay since the CLI has already started transparently opening the vault.
- **Multi-vault basename ambiguity (documented inherited limitation)**: when a basename matches notes in multiple open vaults, the wrapper does NOT disambiguate across vaults — the basename lookup happens inside the requested vault's `app.metadataCache.getFirstLinkpathDest`, scoped to that vault only. Callers using the `file=<basename>` locator across multiple vaults MUST supply the `vault=<name>` argument to scope the lookup. Documented in `docs/tools/smart_connections_similar.md`.

**INDEXING — folder exclusions silently filter results**

- The Smart Connections plugin's settings expose `exclude_folders` and similar filters. Notes inside excluded folders are never embedded and therefore never appear in any match list. The wrapper inherits this silently — there is no way to detect "this note was excluded by config" from outside the plugin. Documented inherited limitation.

**INDEXING — staleness vs freshness**

- The plugin's embedding index reflects the state at the last embedding pass, not vault HEAD. A note edited after its last embedding will still match by its old content until the next indexing pass runs. The wrapper does NOT trigger re-indexing and does NOT surface a staleness signal. Documented inherited limitation.

**ACTIVE-MODE — no focused file**

- Active mode with no focused note MUST yield a structured error with `details.code = "NO_ACTIVE_FILE"` rather than an empty success. This applies in both default mode and count-only mode. Parity with every other typed tool that exposes active mode.

**CLIENT-CLASS — unknown-key validation**

- The server-side validation behaviour for "unknown top-level keys" (US3 scenario 4) is directly observable only from MCP clients that forward unknown keys to the server. Strict-naive clients strip unknown keys client-side per the published JSON Schema's `additionalProperties: false`, in which case the server never sees the offending key and validation does not trigger. Both pathways MUST be documented; the test case MUST exercise the server-side path explicitly so the validation contract holds for the client class that does forward unknown keys.

**SECURITY — structural data-passing**

- The `vault`, `file`, and `path` fields are caller-supplied. The implementation MUST pass them to the underlying CLI as **data** — specifically, JSON-stringified + base64-encoded inside the frozen JS template's payload variable (parity with BI-014 / BI-015 / BI-025's anti-injection pattern). Caller-supplied strings MUST NEVER reach the JS source as text. Path-traversal handling for `path` (US3 scenario 7) is independently a separate concern handled either at the schema layer or by the underlying vault-access layer.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST expose a typed MCP tool that returns the semantically nearest notes to a single named source note, sourced from the Smart Connections plugin's similarity index. The tool's registered name is `smart_connections_similar`, using the **plugin-namespace prefix** convention `<plugin_name>_<operation>` — a NEW naming convention this feature establishes for plugin-backed typed tools. The convention is codified in **ADR-013** (created during the plan phase per the user input). The convention is distinct from BI-022's single-word-verbatim-from-upstream rule, which applies only to wrappers of native Obsidian CLI subcommands (not plugin-backed surfaces).
- **FR-002**: The tool's input MUST consume the project's existing target_mode discriminator contract (`specific` vs `active`) per ADR-003. In specific mode, `vault` is mandatory and exactly one of `file` (vault-root basename) or `path` (vault-relative path) is supplied (XOR). In active mode, none of `vault`, `file`, `path` MUST be supplied.
- **FR-003**: The tool MUST accept an optional `limit` integer field with range `1..100` and default `20`. The `limit` parameter caps the number of matches returned (and the value of `count` in count-only mode). Values outside the range or non-integer values MUST be rejected by schema validation.
- **FR-004**: The tool MUST accept an optional `total` boolean field, defaulting to `false`. When `total: true`, the response carries the count only and the per-match list MUST be empty. When `total: false` (or omitted), the response carries the full match inventory up to `limit`.
- **FR-005**: The tool's input schema MUST forbid unknown top-level keys (`additionalProperties: false`).
- **FR-006**: The tool MUST return an output object with two fields: `count` (a non-negative integer — the number of matches returned, capped at `limit`) and `matches` (an ordered list of per-match entries). The list is empty when `total: true`; it is populated up to `limit` entries when `total: false`. The outer `count` ALWAYS represents the number of matches returned at the locked `limit`, NOT the unbounded similarity-query result count.
- **FR-006a**: The outer `count` field MUST have the same value across both default mode (`total: false`) and count-only mode (`total: true`) for the same source note, same `limit`, and same plugin index state at the same instant. Specifically, when `total: false`, `count === matches.length` MUST hold; when `total: true`, `count` MUST equal the value it would have had under `total: false` against the same inputs. This cross-mode invariant is the testable form of the FR-006 commitment.
- **FR-007**: Each per-match entry MUST carry EXACTLY the following fields: `path` (string, the vault-relative path of the source FILE the matched block belongs to, with the `.md` extension preserved — extracted by taking everything before the FIRST `#` in the plugin's match key; parity with the `files` tool's output for the file-path portion), `headingPath` (an ordered array of strings — the heading-path segments AFTER the first `#` in the plugin's match key, split on `#`; empty array `[]` when the plugin returns a source-level match without a block fragment; literal-preserved single-element array `["---frontmatter---"]` for matches inside the source note's frontmatter block — the wrapper does NOT normalise or rewrite the plugin's frontmatter sentinel), and `score` (a JavaScript `number`, the raw similarity score the Smart Connections plugin returns for that match — see FR-009 on score semantics). The per-match shape is EXHAUSTIVE — NO additional fields are surfaced; in particular NO `displayName`, NO `lines` / `lineStart` / `lineEnd`, NO `excerpt` / `content`, NO `kind` discriminator (block vs source is encoded by `headingPath.length === 0`), NO `model` discriminator, NO `embeddingVersion` field, NO `original` raw-key field. Plan-stage implementation MUST NOT silently widen the per-match shape; any addition is a public-contract change requiring a fresh clarifications session.
- **FR-008**: The `matches` list MUST be in stable, deterministic order across repeated calls against an unchanged index state. Primary sort is `score` descending; SECONDARY tiebreak for equal scores is `path` byte-compare ascending (JavaScript `<`/`>` operator order, NOT locale-aware `localeCompare`); TERTIARY tiebreak for equal `path` (e.g. two blocks from the same source note tied on score) is `headingPath.join("#")` byte-compare ascending. The wrapper does NOT expose alternative sort-order parameters in this feature.
- **FR-009**: The `score` field MUST be the raw numeric value returned by the Smart Connections plugin's similarity API. The wrapper MUST NOT clamp, normalise, or round the score. The score's range and semantics are embedding-model-dependent; the documentation MUST surface this as a documented inherited limitation.
- **FR-009a**: The wrapper MUST apply an in-eval post-fetch finite-score filter (`Number.isFinite(score)` — drops `NaN`, `Infinity`, `-Infinity`, `null`, `undefined`, and any non-numeric `score` value) alongside the FR-010 self-exclusion filter. Any match record whose `score` field is not a finite JavaScript number MUST be silently removed from the result list before the wrapper computes the outer `count` and emits the envelope (per the 2026-05-15 clarifications session Q2). The wrapper MUST NOT coerce non-finite scores to a sentinel value AND MUST NOT surface a dedicated envelope code for the filter event — bad entries simply don't appear. The cross-mode invariant (FR-006a) holds against the post-filter `count` value in both default and count-only modes.
- **FR-010**: The source note itself — AND any block inside the source note — MUST NOT appear in the `matches` list. The wrapper MUST enforce this as defence-in-depth via an in-eval post-fetch filter keyed on the SOURCE path component (`.filter(m => m.path !== sourcePath)`) where `m.path` is the per-match `path` field (everything before the first `#` in the plugin's match key) and `sourcePath` is the source note's vault-relative path. This excludes both source-level self-matches (when `headingPath` is empty) AND block-level self-matches inside the source note (when `headingPath` is non-empty but `path` equals the source path), regardless of whether the plugin's `find_connections()` already excludes them.
- **FR-011**: A source note with zero returned neighbours MUST yield `{ count: 0, matches: [] }` — no error. Count-only mode and default mode share this contract.
- **FR-012**: A locator (`file` or `path`) that does not resolve to any note in the vault MUST yield a structured error with `details.code = "FILE_NOT_FOUND"`. Distinct from `SOURCE_NOT_INDEXED` (which fires only when the file exists in the vault but has no Smart Connections embedding entry). Count-only mode does NOT suppress this error.
- **FR-013**: A locator pointing at a source file the host does not classify as Markdown (file extension `!== 'md'`) MUST yield a structured error with `details.code = "NOT_MARKDOWN"`. The wrapper MUST enforce this in-eval via an `f.extension === 'md'` guard (parity with BI-025).
- **FR-014**: A source file that exists in the vault AND is Markdown AND the Smart Connections plugin is loaded and ready, BUT for which the plugin has no embedding entry, MUST yield a structured error with `details.code = "SOURCE_NOT_INDEXED"`.
- **FR-015**: If the Smart Connections plugin is not installed or not enabled in the target vault (`app.plugins.plugins["smart-connections"]` is `undefined`), the tool MUST yield a structured error with `details.code = "SMART_CONNECTIONS_NOT_INSTALLED"`.
- **FR-016**: If the Smart Connections plugin is installed but its similarity-query surface is unavailable (the plugin's API path the wrapper depends on — locked at plan stage per FR-024 — is missing or non-functional), the tool MUST yield a structured error with `details.code = "SMART_CONNECTIONS_NOT_READY"`. This single code covers both "indexing in progress" and "indexing never started" — distinguishing those sub-states requires plugin-internal probing the wrapper does NOT perform.
- **FR-017**: A `vault` display name that is not registered MUST surface as a structured error with `details.code = "VAULT_NOT_FOUND"` via the cli-adapter's 011-R5 unknown-vault response-inspection clause (cohort: matches BI-014 / BI-015 / BI-025; the wrapper inherits the existing cli-adapter clause unchanged for this path).
- **FR-017a**: When a `vault` display name IS registered in `obsidian vaults` output BUT the corresponding vault window is NOT currently open in Obsidian (no running `app` instance for that vault), the wrapper MUST surface a structured error with `details.code = "VAULT_NOT_FOUND"` and `details.reason = "not-open"`. The same top-level `details.code = "VAULT_NOT_FOUND"` covers BOTH "vault unknown / not registered" (011-R5 inspection clause, `details.reason` absent or `"unknown"`) and "vault registered but not currently open" (new path, `details.reason: "not-open"`) — callers needing to distinguish read `details.reason`. The wrapper MUST NOT add a 9th top-level `details.code` value for this case; the eleven-tool zero-new-top-level-error-codes streak since BI-011 is preserved. **Detection mechanism (locked by live probe on 2026-05-15)**: when the wrapper's eval invocation returns EMPTY stdout (zero bytes OR whitespace-only) with exit 0 AND a `vault=<name>` argument WAS supplied AND the supplied vault name IS present in `obsidian vaults` output, the wrapper classifies the response as `VAULT_NOT_FOUND(reason="not-open")`. Live probe findings (the closed-vault behaviour the prior amendment deferred to plan-stage): the CLI emits empty stdout + exit 0 for the FIRST eval call against a closed-but-registered vault AND transparently OPENS the vault as a side effect; subsequent eval calls reach the now-open vault and work normally. The wrapper's structured error therefore signals "the vault was closed; the CLI has now opened it transparently; a retry of the same call after a brief wait will likely succeed" — agents seeing `not-open` MAY retry after a short delay (the open-and-load time is plugin-config-dependent, often seconds to minutes). The wrapper does NOT auto-retry on the caller's behalf; that's an explicit agent-level decision. The cli-adapter's existing 011-R5 inspection clause (which keys on the literal string `"Vault not found."`) does NOT fire for the closed-vault case — the dispatch-layer classifier must be EXTENDED with this new empty-stdout-plus-known-vault check, OR the typed-tool handler can implement the detection at the JSON.parse stage by treating empty stdout as a specific signal rather than a generic parse failure. The locus of the new detection logic is locked at plan stage per FR-024.
- **FR-017b**: When multiple error conditions fail simultaneously for a single call, the wrapper MUST surface the FIRST discriminator according to a fixed precedence order (per the 2026-05-15 clarifications session Q4 — outer-to-inner / cheapest-first, with the live-probe-driven amendment that replaces "not-focused" with "not-open"). **Specific mode precedence**: `VAULT_NOT_FOUND` with `details.reason` absent or `"unknown"` (cli-adapter 011-R5 inspection BEFORE eval runs) → `VAULT_NOT_FOUND` with `details.reason: "not-open"` (closed-vault detection, mechanism locked at plan stage per FR-017a + FR-024) → `SMART_CONNECTIONS_NOT_INSTALLED` (presence of `app.plugins.plugins["smart-connections"]`) → `FILE_NOT_FOUND` (target file resolution) → `NOT_MARKDOWN` (file extension check via `f.extension === 'md'`) → `SMART_CONNECTIONS_NOT_READY` (similarity-API path resolves to a callable function) → `SOURCE_NOT_INDEXED` (source entry lookup in the plugin's index) → success. **Active mode precedence** (active mode forbids the `vault` arg per ADR-003, so the vault steps do not apply): `SMART_CONNECTIONS_NOT_INSTALLED` → `NO_ACTIVE_FILE` (focused-file resolution via `app.workspace.getActiveFile()`) → `NOT_MARKDOWN` → `SMART_CONNECTIONS_NOT_READY` → `SOURCE_NOT_INDEXED` → success. The wrapper MUST NOT short-circuit later checks when an earlier check fires (no `Promise.all` parallel evaluation); each discriminator is the FIRST condition in the chain that fails. Regression tests MUST include at least one compound-failure fixture per adjacent pair in the chain verifying the earlier-priority discriminator surfaces.
- **FR-018**: Active mode with no focused note MUST yield a structured error with `details.code = "NO_ACTIVE_FILE"`. This applies in both default mode and count-only mode.
- **FR-019**: All validation failures MUST occur strictly before any underlying CLI invocation. Tests MUST be able to assert a CLI dispatcher spy was never called for invalid inputs.
- **FR-020**: A `path` value containing path-traversal characters or shell metacharacters MUST be rejected. The locus of rejection MAY be the input-validation boundary, the underlying vault-access layer, or both; in all cases a structured error MUST reach the caller and no escape from the vault MUST occur.
- **FR-021**: Errors MUST flow through the project's existing structured error codes — **NO new top-level error codes** MUST be introduced by this feature. The eight plugin-specific failure modes (`VALIDATION_ERROR`, `VAULT_NOT_FOUND`, `FILE_NOT_FOUND`, `NO_ACTIVE_FILE`, `NOT_MARKDOWN`, `SMART_CONNECTIONS_NOT_INSTALLED`, `SMART_CONNECTIONS_NOT_READY`, `SOURCE_NOT_INDEXED`) are all surfaced as `CLI_REPORTED_ERROR` with a `details.code` discriminator — parity with the BI-014 / BI-015 / BI-025 envelope-error pattern. The eleven-tool zero-new-top-level-error-codes streak since BI-011 is preserved.
- **FR-022**: The tool MUST be registered through the project's existing typed-tool registration factory. The progressive-disclosure help facility's documentation file for the tool MUST be authored with the per-field input contract, the output shape (for both default and count-only modes), the full eight-entry failure-mode roster (including all three plugin-lifecycle codes AND the `details.reason: "not-open"` sub-discriminator under `VAULT_NOT_FOUND` per the 2026-05-15 clarifications session live-probe-driven amendment), the **error-precedence chain** specified by FR-017b (so agents debugging a failure understand which discriminator fires first per the 2026-05-15 clarifications session Q4), the five documented inherited limitations (embedding-model-dependent score bands, indexing freshness, folder exclusions, plugin-version drift surfaces as `SMART_CONNECTIONS_NOT_READY`, multi-vault basename ambiguity), the **minimum Smart Connections plugin version** the wrapper was probed against (captured by the plan-stage probe per FR-024 and surfaced in the docs as a soft-pin per the 2026-05-15 clarifications session Q1), and at least four worked examples covering at least four distinct usage modes from {specific-mode-by-path, specific-mode-by-file, active-mode, count-only-mode, plugin-not-installed error, plugin-not-ready error, source-not-indexed error, no-active-file error, vault-not-open error, validation rejection}.

### Key Entities *(include if feature involves data)*

- **Similarity match**: A record describing one nearest-neighbour result for a single source note — at BLOCK granularity (the plugin's natural return shape per the 2026-05-15 live-probe-driven amendment). Carries: `path` (the source FILE's vault-relative path with `.md` extension preserved — the file containing the matched block); `headingPath` (an ordered array of heading-path segments locating the block within `path`; `[]` for whole-source matches; `["---frontmatter---"]` literal-preserved for frontmatter-block matches); `score` (the raw similarity score from the Smart Connections plugin's similarity calculation — embedding-model-dependent semantics). Multiple matches MAY share the same `path` (different blocks within the same source note); agents wanting source-level dedupe collapse client-side by grouping on `path` and keeping the highest-scoring entry per source.
- **Similarity inventory**: An ordered list of similarity matches plus a `count`. Sort order is `score` descending with `path` byte-compare ascending tiebreaker. List length is capped at `limit` (default 20, max 100). The list is fully populated in default mode and empty in count-only mode (`total: true`); the `count` is identical in both modes per FR-006a.
- **Target locator**: An ADR-003 target_mode discriminator. In `specific` mode, `vault` is mandatory and exactly one of `file` (vault-root basename) or `path` (vault-relative path) is supplied. In `active` mode, none of `vault`, `file`, `path` is supplied — the focused note is resolved implicitly.
- **Plugin lifecycle state**: A three-state enumeration the wrapper distinguishes from external observation: `NOT_INSTALLED` (plugin not present in vault), `NOT_READY` (plugin loaded but similarity-query surface unavailable), `READY` (plugin loaded and similarity-query surface functional). The `READY` state does NOT imply every individual note has an embedding entry — that is captured by the per-call `SOURCE_NOT_INDEXED` discriminator.

## Assumptions

- **Tool name**: The registered tool name is `smart_connections_similar`, using the **plugin-namespace prefix** convention (FR-001). The convention is `<plugin_name>_<operation>` — plugin name as prefix (hyphens replaced by underscores: `smart-connections` → `smart_connections`), operation as suffix (`similar`). The convention is codified in ADR-013 (created during plan phase per the user input). Source dir `src/tools/smart_connections_similar/`; factory `createSmartConnectionsSimilarTool`.
- **Plugin-namespace vs BI-022 single-word convention**: BI-022's single-word-verbatim-from-upstream rule applies to typed tools that wrap a **native Obsidian CLI subcommand** (e.g. `read` wraps `obsidian read`, `links` wraps `obsidian links`). The new plugin-namespace convention applies to typed tools that wrap a **plugin-exposed API via eval** (e.g. `smart_connections_similar` wraps `app.plugins.plugins["smart-connections"].env.smart_sources.<X>.find_connections()`). The two conventions are non-overlapping and chosen by the wrapping mechanism. ADR-013 governs.
- **Per-match shape**: The per-match entry carries exactly `{path, headingPath, score}` (FR-007 — block-level granularity per the 2026-05-15 live-probe-driven amendment to grilling Q3, supersedes the earlier `{path, score}` source-only shape). Rationale: matches the Smart Connections plugin's natural return shape (`find_connections` returns block-level entries by default; even with `exclude_blocks: true` the result set was empty for source-level in live probes); aligns the wrapper's contract with the Smart Connections sidebar UI; agents wanting source-level collapse to `path`-grouped highest-score-per-source client-side. Richer metadata (model identifier, embedding version, distance vs similarity discriminator, line ranges, excerpts) is deferred to future plugin-backed tools (`smart_connections_blocks` with richer per-block metadata, `smart_connections_chat`) under the same plugin-namespace convention; a future BI MAY add a `lines: [start, end]` field if the plugin surfaces them stably, but the v1 contract is exactly `{path, headingPath, score}`.
- **Score representation**: The `score` field is the raw plugin-returned number, pass-through (FR-009). Rationale: the wrapper is a thin adapter; the plugin's similarity calculation is the source of truth; clamping or normalising would lie about the plugin's classification. The score's range is embedding-model-dependent (transformers.js default ≈ `[0, 1]`; OpenAI ada-002 = `[0, 1]`; other models vary including occasional slightly-negative values for cosine-similarity-derived scores).
- **Sort order**: Default ordering is `score` descending with `path` byte-compare ascending secondary tiebreaker and `headingPath.join("#")` byte-compare ascending tertiary tiebreaker (FR-008 — three-level sort per the 2026-05-15 live-probe-driven amendment, was two-level pre-amendment). Rationale: descending matches the user-visible "most similar first" framing of the Smart Connections sidebar; the path-byte-compare and headingPath-byte-compare tiebreakers are deterministic (unlike `localeCompare` which is locale-dependent) and produce stable test snapshots; the tertiary level on `headingPath` is necessary because the per-match shape now permits multiple matches per source file (different blocks within one note) per the block-granularity amendment; parity with BI-025's `_col`-style internal-sort precedent (inherit plugin order, then apply stable secondary / tertiary sort).
- **Self-exclusion**: The source note AND any block inside the source note are filtered out of the match list (FR-010) via an in-eval post-fetch filter keyed on the source path component (`.filter(m => m.path !== sourcePath)`). Rationale: Smart Connections' own `find_connections()` already excludes self in current plugin versions, but the wrapper enforces this as defence-in-depth so a future plugin version that changes the default does not silently leak the source into results. Block-level self-exclusion is necessary under the 2026-05-15 live-probe-driven amendment to grilling Q3 because the per-match shape now permits blocks within a source — blocks inside the source note are still "self" in the semantic-similarity sense and would otherwise dominate the result list for short notes.
- **Limit default and ceiling**: The `limit` field defaults to `20` and is constrained to `1..100` (FR-003). Rationale: 20 matches the typical "top N" agent-consumption window (10 too tight, 50 too verbose); 100 prevents pathological payloads while leaving headroom above the plugin's typical sidebar default of ~50; 1 is the natural minimum.
- **Count-only field name**: The optional boolean `total` (FR-004) follows the established precedent shared by `files` (BI-019), `outline` (BI-023), `properties` (BI-024), and `links` (BI-025). The default is `false` so existing callers unaware of count-only mode receive the full inventory.
- **Output shape consistency between modes**: The same envelope `{ count, matches }` is returned in both default and count-only modes (FR-006); count-only differs only by `matches` being empty. This eliminates a discriminated-union output type and keeps client code uniform.
- **Path format**: Each match's `path` is the vault-relative path with `.md` extension preserved (FR-007 + Edge Cases / path string format). Rationale: parity with BI-019 `files` output; pasteable directly into every other typed tool's `path=` field for follow-on reads; Smart Connections' own source `key` is exactly this format (zero-transform pass-through).
- **Eval-driven cohort**: The wrapper uses the `eval` subcommand (NOT a native CLI subcommand — no native Obsidian CLI subcommand for similarity queries exists) and reaches `app.plugins.plugins["smart-connections"]` from inside Obsidian. Single-call architecture (one `invokeCli({subcommand: "eval"})` per request). Parity with BI-014 / BI-015 / BI-025 eval-driven flows.
- **Anti-injection**: Caller-supplied `vault` / `file` / `path` flow through `JSON.stringify` → `Buffer.from(...).toString("base64")` → `atob` + `JSON.parse` at JS runtime; never reach the JS source as text (FR-028). Parity with BI-014 / BI-015 / BI-025.
- **Plugin-API verification deferred to plan stage**: The exact API path (`env.smart_sources.get(key)` vs `.items[key]` vs `.fs.get(key)` vs `env.collections.smart_sources.items[key]` vs top-level `find_connections({key})` vs another candidate) AND the `find_connections()` return shape (`[{path,score}]` direct vs `[{item, score}]` requiring extraction vs other) are unknown at spec time. The plan-stage live probe (FR-024) is BLOCKING — `/speckit-plan` cannot complete without the findings persisted in research.md.
- **Plan-stage prerequisite**: Before `/speckit-plan` runs, the test vault MUST have the Smart Connections plugin installed AND initial embedding indexing completed. Without this, the live probe (FR-024) cannot resolve the plugin's API surface and the data-model would ship speculative architecture. The probe captures the host plugin version + working path + return shape + any other plugin-internal facts the JS template depends on.

Cohort: matches BI-014 / BI-015 / BI-025 (eval-driven structured-error cohort). The FR-017 structured-error commitment holds without amendment.

- **Read-only wrapper**: The tool MUST NOT trigger embedding/indexing, MUST NOT mutate the plugin's index, MUST NOT change plugin settings. Parity with every other typed read tool in the project.
- **Zero new top-level error codes; additive surface**: The release impact is purely additive at the public-tool-surface level — no existing tool's public surface changes; **zero new top-level error codes** are added (every plugin-specific failure surfaces through `CLI_REPORTED_ERROR` with a `details.code` discriminator); one new ADR (ADR-013) is created during plan phase per the user input; the architecture document is updated during plan phase per the user input. The earlier OOS entry "block-level / heading-section-level matching (defer to `smart_connections_blocks`)" is REMOVED — block-level matches are IN scope for v1 per the 2026-05-15 live-probe-driven amendment.
