# Feature Specification: Add Get Backlinks

**Feature Branch**: `036-get-backlinks`
**Created**: 2026-05-17
**Status**: Draft
**Input**: User description: "Add Get Backlinks — a retrieval surface that returns every note in the vault which references a given note (its backlinks), so callers can audit incoming references without manually scanning the vault."

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Backlinks listing for a named target note (Priority: P1)

An agent (or a human caller through an MCP-aware client) needs the complete list of source notes that reference one specific target note — for example "before I rename or delete this note, show me every note that points at it so I can update references in lockstep", or "audit which working-set notes still cite this index note". The caller identifies the target in specific mode by either a vault-relative path or a vault-root basename, optionally scoping the call to a named vault. The tool returns the ordered list of every source note that contains at least one reference to the target, each entry carrying the source note's vault-relative path. The source list is per-FILE (one entry per source note), not per-occurrence — see User Story 4 for opt-in per-source multiplicity.

**Why this priority**: This is the dominant use case and the entry point that justifies the feature's existence. Today a caller that wants to know which notes reference a given target has no efficient discovery path — they must either scan the vault file-by-file with a body-text grep (fragile against link-syntax edge cases, picks up link-like tokens inside fenced blocks if not careful, misses display-text aliasing where the wikilink target differs from the visible label), parse plain-text CLI output line by line, or guess from convention. A single typed call turns "who references this note?" into a deterministic answer and pairs with the existing outgoing-links primitive (`links`, BI-025) to give complete 1-hop graph reads from any note. Without this story the feature offers no value; every other story refines this primary read path.

**Independent Test**: Construct a vault carrying a known target note `Notes/Target.md` plus a known set of source notes referencing it: `Notes/A.md` (one bare `[[Target]]` link), `Notes/B.md` (one aliased `[[Target|See Target]]` link), `Notes/C.md` (no reference to Target). Call the tool with `{ target_mode: "specific", vault: "Demo", path: "Notes/Target.md" }`. Assert the response carries `count: 2` and exactly two entries with `source: "Notes/A.md"` and `source: "Notes/B.md"` (in deterministic order); `Notes/C.md` MUST NOT appear. Independently testable in isolation; nothing in P2/P3 is required.

**Acceptance Scenarios**:

1. **Given** a vault containing a target note at vault-relative path `Notes/Target.md` AND three source notes — `Notes/A.md` carrying `[[Target]]`, `Notes/B.md` carrying `[[Target|See Target]]`, `Notes/C.md` carrying no reference to Target — **When** the agent invokes the tool with `{ target_mode: "specific", vault: "Demo", path: "Notes/Target.md" }`, **Then** the response carries `count: 2` and exactly two entries with `source: "Notes/A.md"` and `source: "Notes/B.md"` (no `Notes/C.md`); the entries are in deterministic order (per FR-008); no per-source `count` field is present (per FR-005 — counts are opt-in via `with_counts: true`).
2. **Given** the same target note exists at the same vault-relative path AND the basename `Target` resolves unambiguously inside the vault, **When** the agent invokes the tool with `{ target_mode: "specific", vault: "Demo", file: "Target" }` (basename without the `.md` extension) instead of with `path`, **Then** the response is structurally equivalent to the path-based call — same `count`, same per-entry `source` values, same order. The two locator forms are interchangeable when the basename resolves unambiguously.
3. **Given** a target note that no other note in the vault references, **When** the agent invokes the tool against that target, **Then** the response succeeds with `count: 0` and `backlinks: []` — no error.
4. **Given** a locator that does not resolve to any note in the vault (an unknown path or an unknown basename), **When** the agent invokes the tool, **Then** the response is a structured error identifying the missing target; the response is NOT an empty success (`count: 0`).
5. **Given** a vault display name the host does not recognise (e.g. `vault: "Unknown"`), **When** the agent invokes the tool, **Then** the response is a structured error identifying the unknown vault — the call MUST NOT silently return an empty success and MUST NOT silently route to the focused vault.

### User Story 2 — Backlinks listing for the focused note (Priority: P1)

An agent acting on the user's working context needs the backlinks of whichever note the user currently has open in the editor, without restating the path. The caller invokes the tool in active mode. The tool resolves the focused note implicitly and returns its backlinks. When no note is currently focused (a fresh Obsidian window, the user switched to a non-note view, etc.) the tool surfaces a structured error indicating there is no active file rather than silently returning an empty list.

**Why this priority**: Active-mode is the canonical pattern shared by every per-file typed tool in the project (`read`, `outline`, `read_heading`, `read_property`, `delete`, `rename`, `write_note`, `set_property`, `find_by_property`, `links`). Omitting it would create an inconsistency in the typed surface AND would force agents acting on the user's working context to first invoke a separate "what file is focused?" lookup. Pairs equally with US1.

**Independent Test**: With Obsidian open and a known note focused, call the tool with `{ target_mode: "active" }`. Assert the response covers the focused note's backlinks. Switch focus away (open a non-note view or close all notes) and call again with `{ target_mode: "active" }`. Assert the second response is a structured no-active-file error rather than an empty success.

**Acceptance Scenarios**:

1. **Given** the user has a note focused in Obsidian and that note is referenced by a known set of source notes, **When** the agent invokes the tool with `{ target_mode: "active" }`, **Then** the response covers exactly that focused note's backlinks — same `count`, same per-entry shape, same deterministic order as if the caller had supplied the focused note's vault-relative path in specific mode.
2. **Given** no note is currently focused in Obsidian (the workspace shows an empty pane, a non-note view, or the help panel), **When** the agent invokes the tool with `{ target_mode: "active" }`, **Then** the response is a structured error indicating there is no active file — NOT an empty success, NOT a generic CLI failure.
3. **Given** active mode AND any per-file locator field accidentally supplied alongside it (e.g. `{ target_mode: "active", file: "Other" }`), **When** the agent invokes the tool, **Then** the call fails validation BEFORE any underlying CLI invocation — active mode forbids the per-file locator fields per the project's existing target_mode discriminator contract (ADR-003).

### User Story 3 — Validation rejects malformed inputs at the boundary (Priority: P1)

An agent (or a misbehaving caller) submits an input shape that violates the tool's contract. The tool MUST reject the call at the validation boundary, before any underlying CLI invocation occurs, and MUST surface a structured validation error that names the offending field. The locator-validity stories from the user spec — "neither name nor focus" and "both name and focus" — both fall out of the project's existing target_mode discriminator contract.

**Why this priority**: Validation is the safety contract for every typed tool in this project, and zod-as-source-of-truth is a constitutional requirement (Principle III). Without it, malformed callers reach the CLI and produce undefined or harmful behaviour. Independently testable because every validation case can be exercised with a mock/spy on the CLI dispatcher to assert the dispatcher was never called.

**Independent Test**: For each invalid input shape, call the tool with a CLI dispatcher spy. Assert the call rejects with a structured validation error AND that the dispatcher was never invoked. No real CLI or vault required.

**Acceptance Scenarios**:

1. **Given** an input with NEITHER a target name supplied NOR active mode requested (e.g. `{ target_mode: "specific", vault: "Demo" }` — vault but no `file`/`path`, OR `{}` — nothing at all), **When** the agent invokes the tool, **Then** the call fails validation (the user spec's "rejected before any underlying lookup occurs" contract; mapped to the project's target_mode discriminator rule that specific mode requires exactly one of `file`/`path`); no CLI call is made.
2. **Given** an input with BOTH a target name AND active mode requested (e.g. `{ target_mode: "active", file: "Other" }`), **When** the agent invokes the tool, **Then** the call fails validation (the user spec's "rejected before any underlying lookup occurs" contract; mapped to the project's target_mode discriminator rule that active mode forbids `vault`/`file`/`path`); no CLI call is made.
3. **Given** specific mode WITHOUT a vault parameter (e.g. `{ target_mode: "specific", path: "Notes/Target.md" }`), **When** the agent invokes the tool, **Then** the call fails validation (vault is mandatory in specific mode per ADR-003); no CLI call is made.
4. **Given** specific mode with BOTH `file` AND `path` supplied, **When** the agent invokes the tool, **Then** the call fails validation (the XOR locator rule rejects supplying both); no CLI call is made.
5. **Given** any input with an unknown top-level key (e.g. `{ target_mode: "specific", vault: "Demo", path: "Notes/Target.md", filter: "wikilink" }` — `filter` is not part of this tool's surface), **When** the call is forwarded by an MCP client that does NOT strip unknown keys, **Then** the server-side validation fails; no CLI call is made.
6. **Given** `with_counts` set to a non-boolean value (e.g. the string `"true"`), **When** the agent invokes the tool, **Then** the call fails validation; no CLI call is made. Equivalent test for `total` and `limit`.
7. **Given** `limit` set to a value outside the inclusive range `1..10000` (zero, negative, or greater than 10000), **When** the agent invokes the tool, **Then** the call fails validation; no CLI call is made.
8. **Given** `path` containing path-traversal characters (e.g. `"../../etc/passwd"`) or shell metacharacters, **When** the agent invokes the tool, **Then** the call is rejected — either at the input-validation boundary or by the underlying vault-access layer; in both cases a structured error reaches the caller and no escape from the vault occurs.
9. **Given** an input shape where `target_mode` is missing, set to an unknown value (e.g. `"focused"`, `"current"`), or set to a non-string value, **When** the agent invokes the tool, **Then** the call fails validation; no CLI call is made.

### User Story 4 — Per-source multiplicity opt-in (Priority: P2)

An agent prioritising refactoring work needs to know not only WHICH notes reference the target but HOW MANY TIMES each referring note links to the target — high-coupling referrers (notes that reference the target many times) are the highest-leverage cleanup candidates. The caller invokes the tool with `with_counts: true`. The response carries one entry per referring source note (unchanged from default mode) but each entry additionally carries an integer `count` field reflecting how many times that source note references the target. When `with_counts: false` (or omitted) the per-entry shape is minimal — `source` only, no `count` field.

**Why this priority**: Per-source counts are an opt-in optimisation, not a precondition for the read path. Default mode (counts omitted) delivers the full source-note list that satisfies US1. Counts opt-in matters for the refactoring use case but is layered on top of the core retrieval. Independently testable from US1 with a separate single-line input variation.

**Independent Test**: Construct a vault with `Notes/Target.md` (target) plus `Notes/Many.md` (carries `[[Target]]` three times across three different lines) and `Notes/Once.md` (carries `[[Target]]` once). Call the tool with `{ ..., with_counts: true }` against the target. Assert the response carries two entries: one with `source: "Notes/Many.md"` and `count: 3`, one with `source: "Notes/Once.md"` and `count: 1`. Call again with `with_counts: false` (or omitted). Assert the response carries the same two source entries but NEITHER entry carries a `count` field.

**Acceptance Scenarios**:

1. **Given** a vault where source note `Notes/Many.md` references target `Notes/Target.md` three times across three different lines AND source note `Notes/Once.md` references the same target once, **When** the agent invokes the tool with `{ target_mode: "specific", vault: "Demo", path: "Notes/Target.md", with_counts: true }`, **Then** the response carries two entries — `{ source: "Notes/Many.md", count: 3 }` and `{ source: "Notes/Once.md", count: 1 }` — in deterministic order. The source-note list is per-FILE; the multiplicity is collapsed INTO the per-source `count` field.
2. **Given** the same vault state, **When** the agent invokes the tool with `with_counts: false` (or omits the field — defaults to `false`), **Then** the response carries the same two source entries — `{ source: "Notes/Many.md" }` and `{ source: "Notes/Once.md" }` — but NEITHER entry carries a `count` field (per FR-005, the absence of `count` is observable: an MCP client doing `entry.count === undefined` MUST see `true` in default mode).
3. **Given** a source note that references the target multiple times on the SAME line (rare; e.g. `Compare [[Target]] vs [[Target]] for symmetry.`), **When** the agent invokes the tool with `with_counts: true`, **Then** the source appears ONCE in the response with the multiplicity reflected in `count` (≥ 2). The per-source entry is keyed by `{source}` not by `{source, line}`; same-line repetition contributes to the per-source `count` like cross-line repetition does.
4. **Given** a source note that references the target via an aliased wikilink (e.g. `Notes/A.md` contains `[[Target|See Target]]` once and `[[Target]]` once), **When** the agent invokes the tool with `with_counts: true`, **Then** the entry for `Notes/A.md` reports `count: 2` — the aliased reference contributes to the `count` for the RESOLVED target (`Target`), NOT to a count keyed by the alias text (`"See Target"`). The wrapper attributes multiplicity by resolved target, not by display text.

### User Story 5 — Truncation signal for outsized result sets (Priority: P2)

An agent reading the backlinks of a heavily-cross-referenced "MOC" (map-of-content) note could receive a result set spanning thousands of source notes. To keep the response payload bounded and the latency predictable, the tool applies an implicit cap on the number of source-note entries returned AND surfaces a `truncated: true` flag when the underlying source set exceeded that cap. The caller may also supply an explicit `limit` parameter in the range `1..10000` to override the implicit cap in either direction.

**Why this priority**: Cap-and-truncation is the routine response shape for outsized vaults; without it the tool is brittle on heavily-cross-referenced notes. The default cap delivers a usable contract out of the box; the `limit` knob lets explicit callers tune for their workload. Independently testable from US1 with a fixture vault that exceeds the implicit cap.

**Independent Test**: Construct a vault with a target note `Notes/Hub.md` and 1500 source notes each referencing the hub. (a) Call with default cap — assert `count` equals the implicit cap (1000 per FR-010) and `truncated: true`. (b) Call with `limit: 50` — assert `count: 50` and `truncated: true`. (c) Call with `limit: 10000` against the same vault — assert `count: 1500` (under the explicit cap) AND `truncated` is absent (or `false`).

**Acceptance Scenarios**:

1. **Given** a target note referenced by 1500 source notes AND the call omits `limit`, **When** the agent invokes the tool, **Then** the response carries exactly 1000 entries (the implicit cap per FR-010) AND a boolean `truncated: true` indicating the underlying source set was clipped.
2. **Given** the same vault state AND the call supplies `limit: N` where N is in the range `1..10000`, **When** the underlying source set exceeds N, **Then** the response carries exactly N entries AND `truncated: true`. When N is greater than or equal to the underlying source set size, the response carries all source entries AND `truncated` is absent or `false` (per FR-011, callers MUST treat absent `truncated` as equivalent to `false`).
3. **Given** the underlying source set fits the applied cap (implicit 1000 OR explicit `limit`), **When** the agent invokes the tool, **Then** the response carries the full source list AND the `truncated` field is either absent or set to `false`. The wrapper does NOT emit `truncated: true` when no clipping occurred.

### User Story 6 — Documentation surface for the typed tool (Priority: P3)

An operator or agent inspects the project's progressive-disclosure help facility to understand how the backlinks tool works. The published documentation MUST cover the per-field input contract, the output shape (with and without per-source counts; with and without count-only mode), the failure-mode roster, and at least four worked examples covering at least: a specific-mode happy path with `path`, a specific-mode happy path with `file` (basename), an active-mode happy path, a per-source-counts call, and at least one failure path (unresolved locator OR no-active-file OR validation-rejection).

**Why this priority**: The help facility is the primary discovery surface for tool consumers (mirrored from every typed tool). The tool is callable without docs but undiscoverable without them. Should-pass for ship; not required for the read code path itself to function. Independently testable by loading the help facility output and asserting structural completeness.

**Independent Test**: Invoke the help facility for the backlinks tool. Assert the doc carries: per-field input contract, output shape for default / `with_counts: true` / `total: true` modes, failure-mode roster, and at least four worked examples covering at least four distinct usage modes. The registry-consistency test from `005-help-tool` already auto-asserts the file's existence once the tool is registered; this story expands the assertion to content completeness.

**Acceptance Scenarios**:

1. **Given** the help facility, **When** an operator queries the backlinks tool, **Then** the response carries the full per-field input contract (`target_mode`, `vault`, `file`, `path`, `with_counts`, `total`, `limit`; including the active-mode XOR locator rule), the output shape for default / `with_counts: true` / `total: true` modes, the failure-mode roster (unresolved locator, unknown vault, no active file, validation rejection, output-cap kill), and at least four worked examples covering at least four distinct usage modes from {specific-mode-by-path, specific-mode-by-file, active-mode, with-counts, count-only, unresolved-locator error, no-active-file error, validation rejection}.
2. **Given** the help facility, **When** an operator queries the backlinks tool, **Then** the doc explicitly names the practical ceiling for notes with unusually many incoming references (the FR-010 implicit cap AND the FR-019 underlying execution layer's output cap) so callers can choose to set an explicit `limit` or defer to a different discovery strategy when a target's backlinks list is unusually long.
3. **Given** the help facility, **When** an operator queries the backlinks tool, **Then** the doc carries a one-sentence cross-pointer to the outgoing-links sibling (`links`, BI-025) indicating that the two tools together give complete 1-hop graph reads from any note.

### Edge Cases

The implementation MUST handle, document, or explicitly defer each of the following observable shapes.

**CONCURRENCY**

- The vault may be edited (source notes added, removed, links inserted/deleted) between the moment the request is received and the moment the source-set is scanned. The response reflects the vault state at the moment of scan; there are no stale-read or partial-update guarantees beyond that. Documented as a known limitation.

**CONTENT — empty backlinks list**

- A target note that no other note references MUST return `{ count: 0, backlinks: [] }` — no error. Both default and count-only modes share this contract.

**CONTENT — self-reference**

- A note that links to itself (e.g. `Notes/Self.md` contains `[[Self]]`) is INCLUDED in its own backlinks list — the source-note list carries an entry with `source: "Notes/Self.md"` and (under `with_counts: true`) a `count` reflecting the number of self-references (per Assumptions — self-reference inclusion is locked to match the host's own backlinks-pane semantic and the bidirectional symmetry with the outgoing-links sibling `links` BI-025; the host's combined cache surfaces self-references uniformly). Callers wanting "external" backlinks only do a one-line client-side filter `entry.source !== target_path`.

**CONTENT — body-content opacity (fenced code blocks)**

- A reference appearing only inside a fenced code block or an indented code block (e.g. `[[Target]]` written verbatim inside a triple-backtick block as documentation of wikilink syntax) MUST NOT contribute to the backlinks list. The source note containing only such code-block-only references MUST NOT appear in the response at all. The implementation MAY satisfy this requirement by deferring to the upstream CLI / Obsidian metadata cache, which already separates real links from body-content tokens; no wrapper-side Markdown parser is required.

**CONTENT — display-text aliasing**

- A reference using piped wikilink alias syntax (`[[Target|Display]]`) contributes one reference to the source-note entry for the RESOLVED target (`Target`), NOT to any entry keyed by the alias text (`Display`). The wrapper attributes multiplicity by the resolved target, never by the display label. Mixed-form notes (a source note carrying both `[[Target|Display]]` and `[[Target]]`) report a single per-source entry with `count` reflecting BOTH occurrences.

**CONTENT — markdown link syntax**

- A reference using markdown link syntax (`[Display](Target)` or `[Display](Target.md)`) contributes to the backlinks list when the host's link resolver classifies the URL portion as resolving to the target note. The wrapper does NOT add separate detection for markdown links — it inherits whatever the upstream cache surfaces (which mirrors the host's "Backlinks" pane semantic).

**CONTENT — embed reference**

- An embed reference (`![[Target]]` or `![alt](Target.md)`) contributes to the backlinks list. The wrapper does NOT distinguish embeds from links in the backlinks listing (the per-entry shape carries `source` only — no `kind` discriminator, unlike the outgoing-links sibling); the multiplicity counts embed and non-embed references uniformly. Callers wanting embed-vs-link distinction use the outgoing-links sibling `links` (BI-025) on each source note.

**CONTENT — frontmatter-declared references**

- A reference declared inside YAML frontmatter (e.g. `related: "[[Target]]"`, a property typed as a list-of-wikilinks containing the target) contributes to the backlinks list AND to the per-source `count` (when `with_counts: true`). Frontmatter references and body references are uniformly counted — no separate `source: "frontmatter" | "body"` discriminator. This matches Obsidian's own "Backlinks" pane semantic and preserves bidirectional symmetry with the outgoing-links sibling `links` (BI-025 FR-006b).

**CONTENT — reference to heading or block fragment**

- A reference of the form `[[Target#Heading]]` or `[[Target#^block-id]]` contributes to the backlinks list of `Target.md`. The wrapper does NOT distinguish fragment-anchored references from whole-file references in the listing — they uniformly contribute one to the source-note entry's `count` (under `with_counts: true`). Callers wanting per-heading or per-block backlinks use a different discovery strategy (e.g. body-text search via `search`/`context_search`).

**CONTENT — multiple references on the same line**

- A source note referencing the target multiple times on the same line (e.g. `Compare [[Target]] vs [[Target]] for symmetry.`) contributes ONCE to the source list and (under `with_counts: true`) the per-source `count` reflects the number of occurrences on that line (≥ 2). The per-source entry is keyed by `{source}` — same-line repetition collapses into the per-source `count` the same way cross-line repetition does.

**CONTENT — ordering**

- The per-entry list MUST be in stable, deterministic order across repeated calls on an unchanged vault state. Order is `source` ascending (UTF-16 code-unit order). The wrapper does NOT expose alternative sort-order parameters in this feature; callers needing a different order (e.g. by `count` descending for refactoring prioritisation) re-sort the list client-side.

**CONTENT — implicit cap and truncation**

- When the underlying source-note set exceeds the applied cap (implicit 1000 per FR-010 OR an explicit `limit`), the response carries exactly the cap's worth of entries AND `truncated: true` (per FR-011). The cap unit is `backlinks.length` (one entry per source note, post-aggregation). The wrapper does NOT include the pre-cap source count as a separate field; callers needing the full count set `limit: 10000` (the maximum) and accept the higher payload cost.

**CONTENT — output-cap kill on extraordinarily large responses**

- The underlying execution layer enforces a 10 MiB output cap on a single CLI invocation (inherited from feature 003). A response that, AFTER the applicable cap (`limit` or implicit 1000) has been applied, would STILL exceed this output cap MUST produce a structured `CLI_NON_ZERO_EXIT` (output-cap kill) rather than a silent truncation. This is rare — only fires if individual source-note paths are extraordinarily long. The practical ceiling MUST be documented in the published help facility.

**LOCATOR — unresolved target locator**

- A `file` or `path` value that does not resolve to any note in the vault MUST yield a structured error identifying the missing file rather than an empty success (`count: 0, backlinks: []`). Count-only mode (`total: true`) does NOT suppress this error — the user spec's "name that resolves to no existing note → structured error stating that the target was not found" contract.

**LOCATOR — basename collision**

- A `file` value that is a basename matching more than one note in the vault (e.g. both `Notes/Foo.md` and `Archive/Foo.md` exist) MUST resolve through the host's existing wikilink-resolution semantics; the wrapper does NOT impose a separate disambiguation contract. Whether the host returns the first-matched file's backlinks or a structured error is the host's choice; the wrapper inherits whatever surface the host produces.

**LOCATOR — unknown vault**

- A `vault` display name that is not registered MUST surface as a structured error naming the unknown vault. Cohort placement (eval-driven structured error vs native-subcommand inherited-limitation) is locked at plan stage based on the chosen execution path; in either case, the wrapper MUST NOT silently route to the focused vault.

**LOCATOR — target is a binary attachment**

- A target locator pointing to a binary attachment (image, PDF, or any non-`.md` file the host does not treat as a link target) MUST yield a structured error rather than silently returning a zero-backlinks list — attachments are not part of the note-link graph.

**ACTIVE-MODE — no focused file**

- Active mode with no focused note MUST yield a structured no-active-file error rather than an empty success. This applies in both default mode, `with_counts: true` mode, AND `total: true` mode (none of the opt-in modes mask the error). Parity with every other typed tool that exposes active mode. The user spec's "given no note is focused, when the caller asks for the focused note's backlinks, then the response is a structured error directing the caller either to focus a note or to name a target explicitly" contract.

**CLIENT-CLASS — unknown-key validation**

- The server-side validation behaviour for "unknown top-level keys" (US3 scenario 5) is directly observable only from MCP clients that forward unknown keys to the server. Strict-naive clients strip unknown keys client-side per the published JSON Schema's `additionalProperties: false`, in which case the server never sees the offending key and validation does not trigger. Both pathways MUST be documented; the test case MUST exercise the server-side path explicitly so the validation contract holds for the client class that does forward unknown keys.

**SECURITY — structural data-passing**

- The `vault` / `file` / `path` fields are caller-supplied. The implementation MUST pass them to the underlying CLI as **data** (process arguments or structured parameters), never as text concatenated into a shell command, an `eval` payload, or any other text-based execution surface. Path-traversal handling for `path` (US3 scenario 8) is independently a separate concern handled either at the schema layer or by the underlying vault-access layer.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST expose a typed MCP tool that returns the catalogue of source notes referencing a single named target note. The tool's registered name is `backlinks`, per ADR-010's single-word-verbatim-from-upstream convention (upstream Obsidian CLI subcommand is `backlinks`; parity with `links` (BI-025), `outline` (BI-023), `files` (BI-019), `properties` (BI-024)).
- **FR-002**: The tool's input MUST consume the project's existing target_mode discriminator contract (`specific` vs `active`) per ADR-003. In specific mode, `vault` is mandatory and exactly one of `file` (vault-root basename) or `path` (vault-relative path) is supplied (XOR). In active mode, none of `vault`, `file`, `path` MUST be supplied. The user spec's "neither name nor focus" rejection is the schema-level XOR rule firing on specific mode with no locator; the "both name and focus" rejection is the schema-level rule firing on active mode with a locator field present.
- **FR-003**: The tool MUST accept an optional `with_counts` boolean field, defaulting to `false`. When `with_counts: true`, each per-source entry in the response carries an additional integer `count` field (≥ 1) reflecting the number of times that source note references the target (across body, frontmatter, and embed contexts; aliased and bare references contribute uniformly; same-line repetition contributes the same way cross-line repetition does — see Edge Cases). When `with_counts: false` (or omitted), per-source entries are minimal — `source` only, NO `count` field.
- **FR-004**: The tool MUST accept an optional `total` boolean field, defaulting to `false`. When `total: true`, the response carries the count of source notes only — the per-entry list MUST be empty. When `total: false` (or omitted), the response carries the per-source list as governed by FR-003. The two flags are independent: `total: true` always returns an empty `backlinks` list (the `with_counts` flag has no effect under `total: true` because there are no entries to attach counts to).
- **FR-005**: The tool MUST return an output object with two REQUIRED fields — `count` (a non-negative integer — the total number of distinct source notes referencing the target, post-cap if FR-010 fires) and `backlinks` (an ordered list of per-source entries; empty under `total: true` per FR-004; fully populated otherwise) — AND one OPTIONAL field `truncated` (a boolean, present and set to `true` ONLY when FR-010 fires; absent or `false` otherwise per FR-011). Per-source entries are shaped `{ source: string }` under default mode and `{ source: string, count: integer }` under `with_counts: true`. The outer `count` represents the per-source-note count (FR-005a defines the cross-mode invariant); it does NOT represent total-occurrences-across-all-sources.
- **FR-005a**: The outer `count` field MUST equal `backlinks.length` when `total: false` AND the applied cap was not exceeded (FR-010 did not fire). When `total: true`, the outer `count` MUST equal the value it would have had under `total: false` against the same vault state, AGAIN post-cap if FR-010 fires. When FR-010 fires (truncation), the outer `count` reflects the post-cap source-note count — NOT the pre-cap underlying source-set size; the pre-cap count is NOT separately surfaced. This cross-mode invariant is the testable form of the FR-005 commitment.
- **FR-006**: The input schema MUST forbid unknown top-level keys (`additionalProperties: false`).
- **FR-007**: Repeated references to the target from the same source note (across body lines, frontmatter values, and embed syntaxes) MUST collapse into ONE per-source entry — the source-note list is per-FILE, not per-occurrence. Under `with_counts: true`, the per-source `count` integer reflects the total occurrences from that source. Under `with_counts: false`, the multiplicity is invisible (one entry per source, no count field).
- **FR-008**: The per-entry list MUST be in stable, deterministic order across repeated calls on an unchanged vault state. Sort is by `source` ascending (UTF-16 code-unit order). The wrapper does NOT expose alternative sort-order parameters in this feature; callers re-sort client-side.
- **FR-009**: A target with zero source notes referencing it MUST return `{ count: 0, backlinks: [] }` — no error. Count-only mode (`total: true`) and `with_counts: true` mode share this contract.
- **FR-010**: When `limit` is omitted, the tool MUST apply an implicit hard cap of 1000 source-note entries on the response `backlinks` array. When `limit` IS supplied, that value MUST take precedence over the implicit cap in both directions (`limit: 10` returns up to 10 sources; `limit: 5000` returns up to 5000 sources). The cap unit is `backlinks.length` (one entry per source note, post-aggregation). The `limit` parameter accepts positive integers in the inclusive range `1..10000`.
- **FR-011**: The response MUST include the boolean field `truncated` whenever the underlying pre-cap source-note set exceeded the applied cap (implicit 1000 OR a user-supplied `limit`); the value MUST be `true` in that case. When the underlying set fit within the cap, the wrapper MAY OMIT the field or set it to `false` — callers MUST treat `truncated` absent as equivalent to `false`. The response MUST NOT include the pre-cap source-set count as a separate field.
- **FR-012**: The tool MUST reject a `limit` outside the inclusive range `1..10000` (zero, negative, or greater than 10000) with a structured `VALIDATION_ERROR` BEFORE any vault scan.
- **FR-013**: Self-reference inclusion is LOCKED to **include** — a note that links to itself appears in its own backlinks list, and under `with_counts: true` the per-source `count` reflects the number of self-references (per Assumptions, this matches the host's "Backlinks" pane semantic and preserves bidirectional symmetry with the outgoing-links sibling `links` BI-025). The user spec's "the contract picks either include or exclude and locks it" contract is satisfied by this commitment. Callers wanting "external" backlinks only do a one-line client-side filter on `entry.source !== target_path`.
- **FR-014**: References appearing only inside fenced code blocks or indented code blocks MUST NOT contribute to the backlinks list. The implementation MAY satisfy this requirement by deferring to the upstream CLI / Obsidian metadata cache, which already separates real links from body-content tokens. No wrapper-side Markdown parser is required.
- **FR-015**: References using piped wikilink alias syntax (`[[Target|Display]]`) MUST be attributed to the resolved target (`Target`), NOT to the alias text (`Display`). The per-source `count` (under `with_counts: true`) sums aliased and bare references to the same target uniformly. The wrapper does NOT surface alias text in the response at all (the per-entry shape carries `source` and optionally `count` — no `alias` or `displayText` field).
- **FR-016**: References declared inside YAML frontmatter (e.g. `related: "[[Target]]"`, properties typed as a list of wikilinks containing the target) MUST contribute to the backlinks list AND to the per-source `count` (under `with_counts: true`), uniformly with body references. No separate `source: "frontmatter" | "body"` discriminator is surfaced. Matches Obsidian's "Backlinks" pane semantic and preserves bidirectional symmetry with the outgoing-links sibling `links` (BI-025 FR-006b).
- **FR-017**: A `file` or `path` value that does not resolve to any note in the vault MUST yield a structured error identifying the missing target rather than an empty success. Count-only mode does NOT suppress this error. The user spec's "name that resolves to no existing note → structured error" contract.
- **FR-018**: A `vault` display name that is not registered MUST surface as a structured error naming the unknown vault. The exact envelope (eval-driven structured error via the cli-adapter's 011-R5 unknown-vault response-inspection clause, OR native-subcommand inherited-limitation surface) depends on the execution path chosen at plan stage. In either case, the wrapper MUST NOT silently route to the focused vault.
- **FR-019**: Active mode with no focused note MUST yield a structured no-active-file error rather than an empty success. This applies in default mode, `with_counts: true` mode, AND `total: true` mode (none of the opt-in modes mask the error). The user spec's "no note focused → structured error directing caller either to focus a note or to name a target explicitly" contract.
- **FR-020**: A target locator pointing to a binary attachment (image, PDF, or any file the host does not treat as a link target) MUST yield a structured error rather than silently returning a zero-backlinks list.
- **FR-021**: All validation failures MUST occur strictly before any underlying CLI invocation. Tests MUST be able to assert a CLI dispatcher spy was never called for invalid inputs.
- **FR-022**: A `path` value containing path-traversal characters or shell metacharacters MUST be rejected. The locus of rejection MAY be the input-validation boundary, the underlying vault-access layer, or both; in all cases a structured error MUST reach the caller and no escape from the vault MUST occur.
- **FR-023**: Errors MUST flow through the project's existing structured error codes — NO new top-level error codes MUST be introduced by this feature (Constitution Principle IV; preserves the project's zero-new-top-level-codes streak across the typed-tool cohort). Validation failures surface as `VALIDATION_ERROR`; CLI failures surface through the existing CLI-failure codes; no-active-file surfaces through the project's existing no-active-file code; unresolved-locator surfaces through the project's existing unresolved-locator code; unknown-vault surfaces through `CLI_REPORTED_ERROR(code: 'VAULT_NOT_FOUND')` per ADR-015's existing sub-discriminator usage when the eval-driven path is chosen.
- **FR-024**: The output-too-large structured error MUST fire only when the response — AFTER the applicable cap (user-supplied `limit` or implicit 1000) has been applied — STILL exceeds the cli-adapter's output budget. The error surfaces as the inherited `CLI_NON_ZERO_EXIT` envelope from the cli-adapter's output-cap-kill; the wrapper does NOT construct a custom message. The error MUST NOT fire on a normally-capped response; the `truncated` flag (FR-010 / FR-011) is the routine signal that a larger underlying source set was clipped.
- **FR-025**: The response MUST NOT echo the locator inputs (`vault`, `file`, `path`, `with_counts`, `total`, `limit`, `target_mode`) — read-tool response convention (the project-wide convention that read tools return data only; only write tools echo for write-verification).
- **FR-026**: The tool MUST be registered through the project's existing typed-tool registration factory (per ADR-006). The progressive-disclosure help facility's documentation file for the tool MUST be authored with the per-field input contract, the output shape (for default / `with_counts: true` / `total: true` modes), the failure-mode roster (including the unresolved-locator, unknown-vault, no-active-file, validation-rejection, and output-cap-kill modes), the practical ceiling (FR-010 implicit cap + FR-024 output-cap kill), at least four worked examples covering at least four distinct usage modes from {specific-mode-by-path, specific-mode-by-file, active-mode, with-counts, count-only, unresolved-locator error, no-active-file error, validation rejection}, AND a one-sentence cross-pointer to the outgoing-links sibling `links` (BI-025).
- **FR-027**: Each acceptance criterion across US1–US6 MUST be locked by at least one regression test that survives subsequent re-runs unchanged. The test count MUST be sufficient to cover schema validation, handler behaviour, and registration consistency.
- **FR-028**: The feature MUST run a live-CLI characterisation pass before ship that documents observable CLI behaviour for: specific-mode by-path happy path against a target carrying a mix of source references (bare wikilink, aliased wikilink, embed, markdown link, frontmatter wikilink); specific-mode by-file happy path against the same fixture (verifies basename↔path equivalence); specific-mode against a target with zero backlinks (verifies the empty-list contract); specific-mode against a target with one source note that references it three times across three lines (verifies per-source aggregation FR-007); specific-mode against a target with one source note that references it twice on the same line (verifies same-line aggregation under `with_counts: true`); specific-mode against a target referenced via aliased wikilink only (verifies FR-015 alias-attribution); specific-mode against a target referenced from frontmatter only (verifies FR-016 frontmatter inclusion); specific-mode against a target referenced from inside a fenced code block only (verifies FR-014 code-block exclusion — source MUST NOT appear); specific-mode against a target that references itself (verifies FR-013 self-reference inclusion); active-mode happy path against a focused note; active-mode with no focused file (verifies the no-active-file error path); specific-mode with an unresolved `path` value (verifies the unresolved-locator error path); specific-mode with an unresolved `file` basename (verifies the unresolved-locator error path); specific-mode with `path` pointing at a binary attachment (verifies the FR-020 non-`.md` rejection contract); specific-mode with `vault: "Unknown"` (locks the FR-018 outcome — structured error OR documented inherited limitation depending on execution path); `with_counts: true` against a populated target (verifies the per-source `count` integer is present and accurate); `total: true` against a populated target (verifies the FR-005a cross-mode invariant — outer `count` matches the value returned in default mode for the same vault state); cap-and-truncate behaviour against a target referenced by more than 1000 source notes (verifies FR-010 implicit cap fires and `truncated: true` is present); cap-and-truncate behaviour with `limit: 50` against the same fixture (verifies explicit cap precedence); upstream wire-format characterisation — verifies what fields upstream emits per backlink, locking the wrapper's parsing strategy per FR-005 (in particular: whether upstream emits per-source `count` natively, whether upstream supports a `format=json` flag undocumented in `obsidian help backlinks`, whether the wrapper uses `eval`-driven access to the Obsidian metadata cache or the native `backlinks` subcommand). Findings MUST be persisted in the feature's research artefact.
- **FR-029**: The feature MUST NOT change the public surface of any existing typed tool (`read`, `delete`, `files`, `write_note`, `read_property`, `find_by_property`, `set_property`, `read_heading`, `rename`, `outline`, `properties`, `links`, `move`, `search`, `context_search`, `tag`, `paths`, `smart_connections_similar`, `smart_connections_query`, `obsidian_exec`, the help tool). The only permitted edit to existing source is the addition of the new tool to the registration list AND the corresponding rolled-forward FR-018-style baseline (per feature 022's durable registry-stability machinery).
- **FR-030**: All new source files introduced by this feature MUST carry the project's "Original — no upstream." attribution header per the project Constitution's originality principle (Principle V).

### Key Entities *(include if feature involves data)*

- **Backlink source**: A record describing one source note that references a single target note. Carries `source` (the source note's vault-relative path with forward-slash separators) and, ONLY under `with_counts: true`, `count` (a positive integer counting how many times that source note references the target, aggregated across body / frontmatter / embed contexts, with aliased and bare references uniformly counted, and same-line and cross-line repetition collapsed into a single per-source integer). The per-source entry is keyed by `{source}` (one entry per source note); same-target multiplicity collapses into the optional `count` field, NEVER into separate entries.
- **Backlinks inventory**: An ordered list of backlink-source entries in stable deterministic order (`source` ascending, UTF-16 code-unit order) plus a `count` (the list's length, post-cap if FR-010 fires) and an optional `truncated` flag (`true` only when FR-010 fires). The list is fully populated in default and `with_counts: true` modes; the list is empty under `total: true`. The outer `count` is identical across modes for the same vault state per FR-005a.
- **Target locator**: An ADR-003 target_mode discriminator. In `specific` mode, `vault` is mandatory and exactly one of `file` (vault-root basename) or `path` (vault-relative path) is supplied (XOR). In `active` mode, none of `vault`, `file`, `path` is supplied — the focused note is resolved implicitly.

## Assumptions

- **Tool name**: The registered tool name is `backlinks`, per ADR-010's single-word-verbatim-from-upstream convention (FR-001). Upstream Obsidian CLI subcommand is `backlinks` (single-word, verified by analogy with the `links` (BI-025) sibling whose spec references `backlinks` as a sibling tool surface in its out-of-scope clause). Parity with `links` (BI-025), `outline` (BI-023), `files` (BI-019 post-rename), `properties` (BI-024). Source dir `src/tools/backlinks/`; factory `createBacklinksTool`. The upstream subcommand's exact wire format (whether per-source counts are natively emitted, whether a JSON output flag is supported, whether to route via native subcommand or `eval`) is locked at plan stage based on live-CLI characterisation per FR-028.
- **Per-source aggregation, NOT per-occurrence**: The backlinks listing is per-FILE (one entry per source note), not per-occurrence. This contrasts with the outgoing-links sibling `links` (BI-025) which is per-occurrence (one entry per link in the note's source). Rationale: the user spec frames the response as "list of source notes that reference it" (per-FILE framing); the multiplicity concern is satisfied via the opt-in `with_counts` flag (FR-003) which adds a per-source `count` integer aggregating all references from that source. Per-occurrence backlinks (with line numbers / kind / displayText per occurrence) would be a different tool surface — out of scope at v1; if demand emerges, a future sibling tool `backlinks_per_occurrence` could be added without changing this tool's contract.
- **Per-source entry shape is minimal**: The per-source entry carries `source` only (default mode) or `source` + `count` (under `with_counts: true`). NO `kind` discriminator (unlike the outgoing-links sibling — backlinks aggregates across kinds), NO `line` / `column` field (per-occurrence position information is collapsed into the count), NO `displayText` / `alias` field (alias text is never surfaced — FR-015), NO `source: "frontmatter" | "body"` field (frontmatter and body references are uniformly counted — FR-016), NO `resolved: boolean` flag (broken-backlink reporting is out of scope — the listing only reflects references the host's resolver classifies as targeting the named note). Plan-stage implementation MUST NOT silently widen the per-entry shape; any addition is a public-contract change requiring a fresh clarifications session.
- **Self-reference inclusion is LOCKED to include** (FR-013). Rationale: matches Obsidian's own "Backlinks" pane semantic (self-references appear in the pane), preserves bidirectional symmetry with the outgoing-links sibling `links` (BI-025) which lists a note's own self-link if any, and matches the host's combined cache surface which does not distinguish self-references from external references. The user spec's "the contract picks either include or exclude and locks it" clause is satisfied by this commitment; callers wanting "external" backlinks only do a one-line client-side filter on `entry.source !== target_path`.
- **`with_counts` flag name**: The opt-in counts flag is named `with_counts` (FR-003), defaulting to `false`. Rationale: matches the project's convention for boolean-opt-in fields (`total: true` for count-only mode is already established across `files`, `outline`, `properties`, `links`); the `with_counts` name is unambiguous about what flipping it adds to the response shape; doesn't collide with the `total` flag (which controls a different concern — total source-note count with empty list, NOT per-source occurrence count). If the live-CLI characterisation (FR-028) reveals the upstream `backlinks` subcommand uses a different flag name natively, the wrapper's flag name MAY be renamed to mirror upstream at plan stage — the user-facing semantic is unchanged.
- **`total` count-only mode included for project convention**: The optional boolean `total` (FR-004) follows the established precedent shared by `files` (BI-019), `outline` (BI-023), `properties` (BI-024), `links` (BI-025), and the upstream `backlinks` subcommand's likely own `total` flag (verified at plan stage). Rationale: every other enumerating typed tool in the project exposes this mode; consistency reduces the typed-surface's cognitive load. The user spec did NOT explicitly request `total: true` mode; this is a project-convention addition layered on top of the user spec's core retrieval. If the live-CLI characterisation reveals `total` is not natively supported upstream, the wrapper can compute it client-side (return-length-only) without an additional CLI invocation when paired with the same call's source list.
- **Implicit 1000-source cap**: The implicit cap on `backlinks.length` is locked at 1000 entries (FR-010) — parity with the context_search sibling (BI-035 FR-010) and with the cli-adapter's general output-budget posture. Rationale: a target with > 1000 backlinks is rare in practice; for such "MOC" notes, callers should either set `limit: 10000` explicitly (the maximum) and accept the higher payload cost or paginate via a different discovery strategy. The 1000-entry cap is a wrapper invariant — NOT a plumbed-through schema knob.
- **Output ordering**: Sort is `source` ascending in UTF-16 code-unit order (FR-008). Rationale: matches the deterministic-ordering convention shared by `search` (FR-019 line-mode) and `context_search` (FR-018); the alternative orderings — by `count` descending for refactoring prioritisation, by recency of last modification, etc. — are caller concerns satisfiable by a one-line client-side re-sort and do NOT warrant a request-side sort parameter at v1.
- **Vault targeting uses the project-wide vault-targeting convention** for per-file typed tools: target_mode discriminator (specific vs active) per ADR-003 (FR-002). Active-mode TOCTOU is handled uniformly by the project-wide approach noted in the assistant's auto-memory; this tool does not retrofit per-tool active-mode protection.
- **Read-tool response convention applies**: the tool returns data only; locator inputs (`vault`, `file`, `path`, `with_counts`, `total`, `limit`, `target_mode`) are NOT echoed in the response (FR-025).
- **Execution-path choice deferred to plan stage**: Whether the wrapper routes via the native `backlinks` subcommand or via `eval`-driven access to the Obsidian metadata cache (`app.metadataCache.getBacklinksForFile()` or equivalent) is a plan-stage decision based on live-CLI characterisation (FR-028). The wire-format and per-source-count availability differ between the two paths; the user-facing contract (FRs 001–030, SCs 001–029) is unchanged by the choice. Cohort placement for the unknown-vault behaviour (FR-018) likewise depends on the execution-path choice: `eval`-driven flows match the BI-014 / BI-015 / BI-025 cohort with structured error via the cli-adapter's 011-R5 clause; native-subcommand flows match the BI-019 / BI-023 / BI-024 cohort with an inherited-limitation surface. The plan-stage research artefact MUST lock the choice and document the cohort placement.
- **No new error codes; additive surface**: The release impact is purely additive — no existing tool's public surface changes; no error codes are added; no ADRs are amended. Version bump locked at plan stage — PATCH-level bump per BI-023 / BI-024 / BI-025 precedent (additive surface).
- **The post-008 module-layout convention** (`{schema, handler, index}.ts` plus co-located `*.test.ts`) and the post-022 registration-baseline machinery (`npm run baseline:write` rolls the registry-stability baseline forward in the same commit) are the conventions this feature consumes. No precedent feature's spec or plan is amended.

## Out of scope

- **Per-occurrence backlinks**: Returning one entry per link occurrence (with `line` / `kind` / `displayText` per occurrence) — covered by the outgoing-links sibling `links` (BI-025) when iterating per source note. The per-occurrence direction for backlinks would be a different tool surface; if demand emerges, a future sibling tool can be added without changing this tool's contract.
- **Outgoing links FROM a note** — covered by the existing `links` tool (BI-025); this tool is the inbound sibling.
- **Inventory of unresolved links** (references pointing at non-existent targets) — a separate future feature; the backlinks tool only reflects references the host's resolver classifies as targeting the named target.
- **Shortest-path traversal between two notes** — a separate future feature; this tool returns the direct 1-hop inbound neighbourhood only.
- **Cluster or community detection across the link graph** — a separate future feature; this tool returns a flat per-source list with no graph-structural metadata.
- **Cross-vault backlinks** — each call targets a single vault. Multi-vault callers iterate per vault.
- **Per-heading or per-block backlinks** (e.g. "which notes reference `Target#Heading` specifically") — the tool aggregates fragment-anchored and whole-file references uniformly into the per-source `count`. Callers needing per-heading granularity use a different discovery strategy (body-text search via `search` / `context_search`).
- **Backlinks classification by kind** (wikilink vs embed vs markdown vs frontmatter-property) — the per-source entry collapses across kinds. Callers needing kind-aware breakdown iterate the source notes via the outgoing-links sibling `links` (BI-025) and filter by `target`.
- **Alternative sort orders at the request layer** — sort is locked to `source` ascending (FR-008). Callers re-sort client-side.
- **Pagination (`offset` parameter)** — `limit` only at v1. Callers needing pagination iterate via narrower `path`/`file` locators or a different discovery strategy.
- **Backlinks for the alias text rather than the resolved target** — references via aliased syntax are uniformly attributed to the resolved target (FR-015). Callers needing to find "all references using a specific display label" use a body-text search (`search` / `context_search`) on the alias text.
- **Vault-wide backlinks inventory across every note at once** — this feature operates on one target at a time; callers iterate by composing with `files`.
- **Bare URL references** — bare URLs in body prose (e.g. `https://example.org`) are not part of the note-link graph and do NOT contribute to backlinks (consistent with the outgoing-links sibling's closed `{wikilink, embed, markdown}` enum per BI-025 Q3). Callers needing bare-URL discovery run a body-text grep.
- **Replace-in-vault** (write-side counterpart that updates all source notes' references in lockstep with a target rename) — separate BI when demand emerges; backlinks is read-only retrieval.
