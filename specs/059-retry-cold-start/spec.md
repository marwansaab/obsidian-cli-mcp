# Feature Specification: Retry Cold Start

**Feature Branch**: `059-retry-cold-start`
**Created**: 2026-05-30
**Status**: Draft
**Input**: User description: "Retry Cold Start — when a command targets a vault that is registered but not currently running, the tool transparently absorbs the vault's one-time cold-launch failure and tries the command once more, so the caller receives the real result instead of a spurious 'command not found'. Implements ADR-029."

## Clarifications

### Session 2026-05-30

- Q: When attempt 1 shows the cold-start signature and the retry returns neither success nor the cold-start signature — the vault has launched, the command ran, and it returns a different real error (e.g. FILE_NOT_FOUND or a non-zero exit) — which result does the caller see? → A: The second attempt's outcome is authoritative — success → success; any failure → attempt 2's classified error. Attempt 1's cold-start signal is known-spurious and is discarded once the retry fires; it is never surfaced over a genuine post-launch error.
- Q: Should the intermittent transport-level `Stream closed` first-attempt failure also trigger the single retry, or only the exact `exitCode: 0` + "command not found" stdout signature? → A: Both forms are recognized as cold-start triggers, but they are NOT equally safe to retry unconditionally — refined this session by the plan-phase adversarial review (see the follow-up Q below). Form (a) command-not-found is retried unconditionally; form (b) `Stream closed` is probe-gated. (Originally answered "yes, unconditional for both"; narrowed because a dropped pipe can fire after a mutation applied.)
- Q: Should the retry fire immediately, or after a short fixed delay to let the cold-launch window close? → A: Immediate retry, no delay, is the firm default (matches ADR-029's "up by the second call"; keeps success-path latency at zero extra). The plan-phase T0 probe (OQ-003) measures the launch window and is authorized to introduce a single small fixed bounded delay before the retry ONLY if an immediate retry demonstrably races the launch; no unbounded or growing wait is ever permitted.
- Q: How should the canonical cold-start stdout signature be matched — exact full-string equality, or a substring/normalized match on the invariant phrase? → A: Match `exitCode: 0` AND stdout containing the invariant, command-name-independent phrase (e.g. `not found. It may require a plugin to be enabled.`). This is command-agnostic (FR-002 — no per-command signature table) and mirrors the existing R5 vault-not-found stdout-signature precedent in the ADR-007 four-priority classification. Exact full-string equality is rejected (would require interpolating the per-call command name, breaking command-agnosticism); a broad `Error:`-prefix match is rejected (would retry unrelated exitCode-0 errors). The exact invariant literal is pinned in the plan against the T0 probe and covered by a regression test (ADR-029 brittleness note).
- Q: (plan-phase adversarial review) Is it safe to retry the `Stream closed` (form b) variant unconditionally for every command, including non-idempotent mutating ones (`rename`/`move`/`delete`, and the opaque `obsidian_exec` passthrough)? → A: No. Form (a) command-not-found is provably side-effect-safe for ALL commands (registry-not-ready means the command never executed), so it is retried unconditionally. Form (b) `Stream closed` is a dropped pipe that carries no evidence of where in the command lifecycle it occurred — it can fire after a mutation already applied — so it is probe-gated, all-or-nothing: form (b) triggers the retry only if the extended T0 probe (OQ-005) proves `Stream closed` always fires pre-execution, in which case it is enabled blanket for all commands; otherwise form (b) is dropped entirely. It is never gated per-command (no idempotency flag per tool), to avoid adding a per-tool adaptation surface a future mutating tool could forget.
- Q: (plan-phase adversarial review) The "no tool can forget the retry" property holds only because every command flows through `dispatchCli`; what prevents a future tool from importing `node:child_process` `spawn` directly (e.g. for a streaming command) and silently bypassing the retry? → A: A structural guardrail test (FR-012) that source-scans `src/**` and fails the build if a `spawn`/`spawnSync`/`exec`/`execFile` value-import appears outside `src/cli-adapter/_dispatch.ts`, or if `dispatchCli` gains a production caller beyond the two facades. This converts the bypass from a silent regression into a failing build — the load-bearing enforcement of the no-per-tool-adaptation guarantee (FR-009).

## User Scenarios & Testing *(mandatory)*

### User Story 1 - A valid command against a closed vault succeeds after launch (Priority: P1)

An automation agent issues a valid command whose target vault is registered with Obsidian but is not currently running. The very act of issuing the command launches the vault as a side effect, and the command — invisibly to the agent — is retried once after the launch window closes. The agent receives the command's normal successful result on that single call, with no trace of the transient cold-launch failure.

**Why this priority**: This is the whole point of the feature and the minimum viable slice. Today the first command against a not-yet-open vault returns a misleading "command not found" even though the command is valid and the vault launches anyway (ADR-029). An automation agent reading that message misdiagnoses a valid command — or the entire tool — as broken. Absorbing the one-time cold-start failure so the caller sees the real result is the core value; everything else refines or guards it.

**Independent Test**: Against a registered-but-closed vault, issue a known-valid command (for example a read) as the first command. Verify the caller receives the command's normal successful result on a single call, and that the cold-start "command not found" failure is never surfaced to the caller.

**Acceptance Scenarios**:

1. **Given** a registered vault that is not currently open, **When** a caller issues a valid command targeting it, **Then** the command returns its normal successful result and the transient cold-start failure is not surfaced to the caller.
2. **Given** that same scenario, **When** the cold-start window is encountered on the first attempt, **Then** the caller observes a single final outcome — the success — not an intermediate error followed by a result.

---

### User Story 2 - Genuine failures still surface unchanged (Priority: P2)

An automation agent issues a command that is genuinely invalid — a command name that does not exist, a typo, a TUI-only command, or one that requires a capability/plugin that is unavailable — or a command that fails for a reason unrelated to cold-start. The retry behaviour never hides such failures: the original, real error reaches the caller intact.

**Why this priority**: The retry is only safe if it cannot mask a real error. Because the cold-start failure and a genuinely-unknown-command failure are observably identical (ADR-029: the stdout string is the same for both), the retry must absorb only the transient window and then pass through whatever the second attempt produces. A no-masking guarantee that is verifiable in isolation is what makes the P1 recovery safe to ship; without it the feature would trade a misleading error for a hidden one. It is P2 rather than P1 only because the recovery path (US1) is demonstrable on its own — but in practice the two ship together, since the retry's behaviour on the second attempt is exactly what preserves genuine failures.

**Independent Test**: Issue a genuinely unknown command (a typo, or a TUI-only command such as `vault:open`) against a closed vault. Verify it still fails after exactly one retry, with the original error preserved and programmatically unchanged. Separately, issue a command that fails for a non-cold-start reason (a missing target file, or a call that times out) and verify it surfaces on the first attempt without being retried.

**Acceptance Scenarios**:

1. **Given** a command name that does not exist or requires an unavailable capability, **When** a caller issues it, **Then** the caller still receives the failure unchanged after the retry — the behaviour does not mask it.
2. **Given** a command that fails for a reason unrelated to cold-start (for example the target file does not exist, or the call times out), **When** a caller issues it, **Then** that error surfaces without being retried as if it were a cold-start.
3. **Given** the retry's second attempt also returns the cold-start signature, **When** the outcome is classified, **Then** the second attempt's structured error (identical to the first attempt's) is propagated unchanged rather than swallowed or defaulted.

---

### User Story 3 - One consistent behaviour for every command (Priority: P3)

A tool author wants the cold-start retry handled once, in the shared command-dispatch layer, so that every command — whether issued through a purpose-built tool or through the general command passthrough — inherits the same behaviour with no per-command opt-in, and so that no tool re-implements retry logic and drifts.

**Why this priority**: P1 and P2 define the behaviour for one command; P3 makes that behaviour uniform and bounded across the whole tool surface. Centralising the retry in the dispatch layer is the structural decision that keeps the behaviour consistent and prevents the per-tool retry duplication ADR-029 was written to avoid. It also fixes the latency and termination bounds — at most one extra attempt, and a guaranteed terminate-and-surface when the vault never becomes available. It is P3 because it is a generalisation/hardening of the single-command behaviour already established by US1/US2, not a separate user-facing capability.

**Independent Test**: Confirm the retry fires identically for a command issued through a purpose-built tool and for the same command issued through the general command passthrough, with no per-command configuration. Confirm at most one extra attempt is ever made (worst-case latency is one additional round-trip). Confirm that when the vault never becomes available (Obsidian not running at all), the operation terminates after the single retry and surfaces the real error rather than hanging or retrying indefinitely.

**Acceptance Scenarios**:

1. **Given** any command in the tool surface, **When** it meets the cold-start condition, **Then** the same retry behaviour applies with no per-command opt-in.
2. **Given** the cold-start condition, **When** the second attempt succeeds, **Then** no more than one additional attempt is made, so added latency is bounded.
3. **Given** the vault never becomes available (for example the app is not running at all), **When** a caller issues a command, **Then** the operation terminates and surfaces the real error rather than hanging or retrying indefinitely.
4. **Given** a command issued through the general command passthrough as opposed to a purpose-built tool, **When** it meets the cold-start condition, **Then** it inherits the identical retry behaviour.

### Edge Cases

- **Cold-launch window outlasts a single round-trip**: if the vault is still launching when the retry fires, the retry also returns the cold-start signature. Because the retry is bounded to exactly one, that still-cold signature (the second attempt's) is then classified and propagated as the final error (no further retries). The retry fires immediately by default; whether a single small fixed bounded pre-retry delay is warranted to close the window reliably is measured by the plan-phase T0 probe (OQ-003).
- **First attempt surfaces the transport-level variant instead of the not-found signature**: ADR-029 notes the first attempt occasionally surfaces as a transport-level `Stream closed` rather than the `exitCode: 0` + "command not found" stdout. This variant (form b) is **probe-gated** (FR-001, Clarification 2026-05-30 follow-up): unlike form (a), a dropped pipe carries no evidence of where in the command lifecycle it occurred, so it is enabled only if the extended T0 probe (OQ-005) proves it always fires pre-execution (then blanket for all commands), else dropped. Its empirical frequency (reliable vs intermittent) and lifecycle position are characterised by the plan-phase T0 probe (OQ-002 / OQ-005).
- **Transport-level `Stream closed` arrives as an exit-0 success (the dangerous resolve path)**: if `Stream closed` ever manifests as `exitCode: 0` with stdout that does not begin `Error:`, the dispatch classifier resolves it as a *success* rather than rejecting — so a catch-only retry wrapper would never see it and the caller would receive a phantom success. The retry decision MUST therefore also inspect the resolved output for the pinned `Stream closed` literal, not only the rejection path. Whether this manifestation is real is pinned by the T0 probe (OQ-005); the resolve-path inspection is added only if the probe shows it occurs.
- **Eval-composed cold-start manifests as a well-formed envelope (out of scope)**: an eval-composed tool (e.g. `open_file`, `query_base`) may, during the launch window, run its eval against the wrong/previous focused vault and return a clean `exitCode: 0` eval envelope (e.g. `open_file`'s `VAULT_NOT_FOCUSED`, or `query_base`'s empty-stdout → `detectIfClosed` → `VAULT_NOT_FOUND/reason:"not-open"`). The dispatch layer sees a *success*, so the cold-start retry does not (and is not meant to) fire. This focused-vault-mismatch manifestation is a distinct failure mode already handled by those tools' own guards (per BI-057 / ADR-015) and is **out of ADR-029's dispatch-layer scope** (FR-013). The dispatch retry fixes cold-start only when it surfaces as form (a) or form (b).
- **Genuinely unknown command (typo, TUI-only `vault:open`, disabled-plugin command)**: fails identically on the retry; that same error is propagated after exactly one retry. The retry cannot — and does not attempt to — distinguish cold-start from genuine-unknown by signature, because the two are observably identical.
- **Retry returns a different real error than attempt 1**: attempt 1 shows the cold-start signature, the retry fires, and the now-launched vault runs the command and returns a *different* real error (e.g. `FILE_NOT_FOUND`, or a non-zero exit). The second attempt is authoritative (FR-005, FR-007, Clarification 2026-05-30): the post-launch error surfaces, and attempt 1's spurious cold-start "command not found" is discarded rather than masking it.
- **Obsidian not running at all**: the first attempt may not produce a usable vault even after the launch side effect; the operation terminates after the single retry and surfaces the real error, never hanging or looping.
- **Mutating command in the cold-start window**: for **form (a)** (command-not-found), the signature means the command registry was not ready and the command never executed against the vault, so the only side effect of the first attempt is the vault launch and the retry cannot double-apply — safe for `rename`/`move`/`delete` and the `obsidian_exec` passthrough alike. For **form (b)** (`Stream closed`), this guarantee does NOT automatically transfer: a dropped pipe can fire after a mutation already applied but before stdout flushed, so an unconditional form-(b) retry of a non-idempotent mutating command could double-apply (append twice, duplicate) or mask a real success (e.g. `rename A→B` landed, retry sees A gone, reports failure). This is exactly why form (b) is probe-gated (FR-001): the extended T0 probe (OQ-005) must force a `Stream closed` against a mutating command on a closed vault and inspect the vault state for partial application before form (b) is enabled.
- **Non-cold-start failure (file-not-found, timeout, validation, path-safety reject)**: never retried; surfaces on the first attempt with current single-shot latency.

## Requirements *(mandatory)*

### Functional Requirements

#### Trigger detection

- **FR-001**: System MUST detect the cold-start failure signature on the first attempt of a one-shot command — the observable signal produced when a valid command targets a registered-but-not-yet-running vault during the cold-launch window. The trigger has two recognized first-attempt forms (Clarification 2026-05-30): (a) the canonical CLI-reported signature (ADR-029) — `exitCode: 0` with stdout `Error: Command "<cmd>" not found. It may require a plugin to be enabled.`; and (b) the intermittent transport-level `Stream closed` failure. Form (a) triggers the single retry **unconditionally for all commands** — registry-not-ready provably means the command never executed, so retrying is side-effect-safe even for mutating commands. Form (b) is **probe-gated, all-or-nothing** (Clarification 2026-05-30 follow-up): because a dropped pipe carries no evidence of where in the command lifecycle it occurred, form (b) triggers the retry only if the extended T0 probe (OQ-005) proves `Stream closed` always fires pre-execution, in which case it is enabled blanket for all commands; otherwise form (b) is dropped — never gated per-command. Form (a) is matched as `exitCode: 0` AND stdout containing the invariant, command-name-independent phrase (e.g. `not found. It may require a plugin to be enabled.`) — a substring/normalized match, not exact full-string equality, so it stays command-agnostic (FR-002) and mirrors the existing R5 vault-not-found stdout-signature precedent in the ADR-007 four-priority classification (Clarification 2026-05-30). The exact invariant literal is pinned in the plan against the T0 probe and covered by a regression test; the spec fixes that the trigger is this transient cold-launch window — in both forms — and nothing else.
- **FR-002**: System MUST treat trigger detection as command-agnostic — the same signature governs the retry regardless of which command (read, list, search, write/mutating, or tab/open) produced it. The system MUST NOT maintain a per-command signature table. Any command whose cold-start signature is found to differ from the canonical one is recorded as a divergence (see Open Questions) rather than special-cased silently.

#### Retry behaviour

- **FR-003**: On detecting the cold-start signature on the first attempt, System MUST issue exactly one additional attempt of the identical command before classifying or propagating any error. The retry relies on the vault-launch side effect of the first attempt: by the second attempt, the cold-launched vault is up and its command registry is ready.
- **FR-004**: System MUST retry at most once. There MUST be no retry loop and no escalating backoff. After the single retry, the outcome — success or failure — is final.
- **FR-005**: On a successful retry, System MUST return the second attempt's successful result as the caller's sole, single final outcome. The transient first-attempt failure MUST NOT be surfaced to the caller, and the caller MUST NOT observe an intermediate error followed by a result. More generally, once the retry fires the second attempt is authoritative for both success and failure (Clarification 2026-05-30; see FR-007).
- **FR-006**: System MUST keep total added latency bounded. In the success path (no cold-start signature on the first attempt), zero extra attempts and zero added delay are incurred. In the cold-start / genuine-unknown path, at most one extra attempt is made. The retry fires immediately by default (Clarification 2026-05-30). A single small fixed bounded pre-retry delay MAY be introduced only if the plan-phase T0 probe (OQ-003) shows an immediate retry races the launch window; such a delay MUST be a small fixed bound, never an unbounded or growing wait, and applies only on the retry path (never the success path).

#### No-masking guarantee

- **FR-007**: When the retry returns any failure, System MUST classify and propagate the **second attempt's** error — attempt 1's cold-start signal is discarded once the retry fires (Clarification 2026-05-30). Because a genuinely unknown command, a command behind a disabled plugin, a TUI-only command, or a typo returns the same failure on the retry, it surfaces intact after exactly one retry. When the retry instead returns a *different* real error (the vault having launched in the interim), that post-launch error is what surfaces — attempt 1's stale cold-start "command not found" is never surfaced over it, which would itself be a masking bug. No failure is swallowed, defaulted, or rewritten by the retry (Constitution Principle IV).
- **FR-008**: System MUST NOT retry any failure that is not the cold-start signature. File-not-found, timeouts, validation errors, path-safety rejects, and every other classified failure MUST keep their current single-shot behaviour and surface on the first attempt without an added retry.

#### Scope of application

- **FR-009**: System MUST apply the retry uniformly across the command-dispatch layer so that every command — those issued through a purpose-built tool and those issued through the general command passthrough — inherits the identical behaviour. There MUST be no per-command opt-in and no per-tool re-implementation of retry logic.
- **FR-010**: System MUST bound termination. When the vault never becomes available (for example Obsidian is not running at all), the operation MUST terminate after the single retry and surface the real error, rather than hanging or retrying indefinitely.

#### Side-effect safety

- **FR-011**: For **form (a)** (command-not-found), System MUST rely on the signature meaning the command never executed against the vault (the registry was not yet ready), so that retrying a mutating command does not double-apply — the only side effect of a form-(a) first attempt is the vault launch. For **form (b)** (`Stream closed`), System MUST NOT assume this property: a dropped pipe may fire after a mutation applied, so form (b) is enabled (FR-001) only after the extended T0 probe (OQ-005) empirically proves `Stream closed` always fires pre-execution; otherwise form (b) is dropped. Any evidence of partial application on a first attempt of the form that is being enabled invalidates that form's retry and is surfaced rather than ignored.

#### No-bypass enforcement and scope

- **FR-012**: System MUST enforce the single-dispatch-chokepoint invariant with a build-failing structural guardrail test, not by convention alone. The test MUST source-scan `src/**` and fail if a `node:child_process` value-import of `spawn` / `spawnSync` / `exec` / `execFile` appears in any production file other than the sole spawn primitive (`src/cli-adapter/_dispatch.ts`), or if the spawn primitive `dispatchCli` acquires a production caller beyond the two dispatch facades. This converts "a future tool reaches the CLI without inheriting the retry" from a silent regression into a failing build — it is the load-bearing enforcement of FR-009's no-per-tool-adaptation guarantee.
- **FR-013**: System's retry MUST be understood to fix cold-start ONLY for first-attempt failures that surface to the dispatch layer as form (a) or form (b). A cold-start that an eval-composed tool converts into a well-formed `exitCode: 0` eval envelope (e.g. a focused-vault mismatch returning `VAULT_NOT_FOUND/reason:"not-open"`) is a distinct failure mode handled by that tool's own guards and is explicitly out of scope for this dispatch-layer retry. The spec MUST NOT claim the retry fixes that manifestation; any future need to retry the eval-envelope manifestation is a separate decision outside ADR-029.

### Key Entities

- **Cold-start failure signature**: The observable signal that identifies the transient cold-launch window, in two recognized first-attempt forms (Clarification 2026-05-30): (a) the canonical `exitCode: 0` together with the stdout `Error: Command "<cmd>" not found. It may require a plugin to be enabled.` (ADR-029); and (b) the intermittent transport-level `Stream closed` failure. Form (a) triggers the retry unconditionally for all commands; form (b) is probe-gated, all-or-nothing (FR-001). Form (a) is observably indistinguishable from a genuinely-unknown-command failure, which is why the retry must be self-limiting rather than signature-discriminating.
- **One-shot command invocation**: The unit of work the retry wraps — a single command dispatched against a vault, whether on behalf of a purpose-built tool or the general command passthrough.
- **Retry attempt**: The single additional invocation of the identical command, issued only when the first attempt returns the cold-start signature, and never more than once.
- **Cold-launch window**: The short period (empirically a few hundred milliseconds — to be quantified, see Open Questions) during which the vault is launching as a side effect of the first command and its command registry is not yet ready.
- **Command-dispatch layer**: The single shared layer through which both the purpose-built tools and the general command passthrough issue commands, and where the retry is centralised so the behaviour is consistent and not re-implemented per tool.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: A valid command issued as the first command against a registered-but-closed vault returns its normal successful result on a single call — across a representative set of commands (a read, a list, a search, a write/mutating command, and a tab/open command) whose cold-start surfaces as the command-not-found signature, the spurious first-call "command not found" failure is surfaced to the caller in 0% of cases. (A tab/open eval whose cold-start instead manifests as a `VAULT_NOT_FOCUSED` envelope is out of scope per FR-013; OQ-001 records which manifestation each kind produces.)
- **SC-002**: The caller observes exactly one final outcome per command — never an intermediate error followed by a separate result.
- **SC-003**: A genuinely unknown or unsupported command still fails, with its original error preserved unchanged, after no more than one extra attempt — 0% of genuine failures are masked by the retry.
- **SC-004**: No more than one additional attempt is ever made for any command. Worst-case added latency is one extra command round-trip (plus at most a small fixed pre-retry delay if one proves necessary), incurred only on the cold-start / unknown-command path; the success path adds zero extra attempts.
- **SC-005**: The retry behaviour is identical for a command issued through a purpose-built tool and the same command issued through the general command passthrough, with no per-command configuration in either path.
- **SC-006**: When the target vault never becomes available, the operation terminates and surfaces the real error in bounded time, in 100% of such cases — it never hangs or retries indefinitely.
- **SC-007**: Errors unrelated to cold-start (file-not-found, timeout, validation, path-safety) are retried as cold-start in 0% of cases — they surface on the first attempt with the current single-shot latency.
- **SC-008**: The no-per-tool-adaptation guarantee is build-enforced: a structural guardrail test fails the build if any production file outside the sole spawn primitive value-imports a `node:child_process` spawn function, or if the spawn primitive gains a caller beyond the two dispatch facades (FR-012). A future tool that tries to reach the CLI without inheriting the retry cannot merge.
- **SC-009**: No non-idempotent mutating command is ever double-applied or has a real success masked by the retry. Form (a) is safe by construction; form (b) (`Stream closed`) ships only after the OQ-005 probe proves pre-execution-only firing, and is dropped otherwise (FR-001, FR-011) — so in 100% of shipped configurations a mutating command is retried only when provably non-executing on the first attempt.

## Assumptions

- **Implements ADR-029 (proposed); this spec gates its ratification**: The decision to retry once on the cold-start signature in the dispatch layer is recorded in ADR-029 (status: Proposed). This feature turns that decision into behaviour. The empirical probes in Open Questions resolve ADR-029's open parameters (exact trigger-signature match, whether the probe-gated transport-level `Stream closed` variant ships or is dropped, any small fixed pre-retry delay) and gate its flip to Decided. The architecture description's error-mapping / dispatch pipeline is amended to record the retry step on ship, not before (per the connector documentation discipline cited in ADR-029).
- **Centralised in the dispatch layer, not per tool**: The retry is placed in the shared command-dispatch / classification path (the locus ADR-029 names) so both dispatch facades — the typed tools and the general command passthrough — inherit it. This is the deliberate alternative to per-tool retry, which ADR-029 rejected as duplication-prone drift.
- **No new top-level error code (Constitution Principle IV)**: The retry adds no new failure vocabulary. On persistent failure the second attempt's structured error is propagated unchanged; the retry only suppresses the transient first-attempt cold-start failure when the second attempt succeeds. The zero-new-top-level-codes streak is preserved.
- **Relies on the upstream lazy-launch contract**: The retry depends on the documented upstream behaviour that the first command against a closed/registered vault launches it as a side effect. If upstream ever stops launching-on-first-call, the retry would no longer self-resolve — it would simply pass the error through after one retry (safe, but the benefit would disappear). Pre-launching the vault before the first command is explicitly out of scope.
- **The trigger is a string/exit-code match on upstream output**: The cold-start signature is matched against upstream stdout wording plus exit code. This is brittle if Obsidian changes the message; the matched literal must be kept in sync and covered by a test. The spec fixes the behaviour and the match strategy (a substring/normalized match on the invariant, command-name-independent phrase plus `exitCode: 0`, FR-001); the exact invariant literal is pinned in the plan against the T0 probe.
- **Side-effect safety is form-specific**: For form (a) (command-not-found), the signature is a false negative from a not-yet-ready command registry, so the first attempt provably did not apply any mutation — the unconditional retry is safe for all commands. For form (b) (`Stream closed`), no such guarantee exists a priori (a dropped pipe can fire post-mutation), which is why form (b) is probe-gated: it ships only if OQ-005 empirically proves pre-execution-only firing, else it is dropped. The plan-phase adversarial review (2026-05-30) surfaced this distinction; the original spec conflated the two forms.
- **Test scope is unit-level plus empirical T0 probes**: Behavioural coverage is the project's vitest unit tests over the dispatch layer (cold-start signature → one retry → success; persistent signature → propagate unchanged; non-cold-start failure → no retry; both facades). The empirical probes in Open Questions are run against the authorised test vault per `.memory/test-execution-instructions.md` at the implement-phase T0 step (tasks T003–T010), not encoded as in-repo integration tests.

## Out of Scope

- Proactively pre-launching or opening a vault before the first command is issued.
- Retrying any error class other than the cold-start condition — timeouts, genuine not-found, validation errors, and the like keep their current single-shot behaviour.
- Multi-attempt retry loops or exponential backoff — at most one extra attempt.
- Changing which vault is focused, or any vault-switching behaviour.

## Open Questions & Validation Probes

These are empirical probes to run against a real closed vault at the implement-phase T0 step (tasks T003–T010), per `.memory/test-execution-instructions.md`. Their answers resolve ADR-029's open parameters (gating its flip to Decided) and are fed back into the architecture description on ship. They are recorded here rather than as `[NEEDS CLARIFICATION]` markers because ADR-029 explicitly defers them to the plan/implement-T0 flow and a reasonable default exists for each.

- **OQ-001 — Signature uniformity across command kinds** (match strategy decided 2026-05-30: substring match on the invariant phrase + `exitCode: 0` — FR-001): confirm the cold-start failure signature's invariant phrase is identical across a representative set (a read, a list, a search, a write/mutating command, and a tab/open command) issued against a closed vault, and pin the exact invariant literal for the regression test. Note any command whose signature differs (it would force a re-think of the command-agnostic FR-002 assumption).
- **OQ-002 — Transport-level variant** (inclusion **re-opened and probe-gated** by the plan-phase adversarial review — FR-001): characterise (1) how reliably vs intermittently the `Stream closed` first-attempt failure occurs during cold-start, AND (2) its exact surface form through `dispatchCli` — does it arrive as a raw rejected `Error`, a non-zero exit (`CLI_NON_ZERO_EXIT`), or the dangerous `exitCode: 0` stdout that resolves as a phantom success? The matcher and the resolve-path inspection both depend on this. Feeds the OQ-005 gate; form (b) is enabled only if OQ-005 also clears it.
- **OQ-003 — Cold-launch window timing** (default decided 2026-05-30: immediate retry, no delay — FR-006): measure the cold-launch window to confirm an immediate second attempt is sufficient. A single small fixed bounded delay is introduced only if the measurement shows the immediate retry races the launch; quantify the typical launch time to set that bound if so.
- **OQ-004 — No-masking proof**: Verify empirically that a genuinely unknown command (a TUI-only command, or a typo) still fails after exactly one retry, with the original error preserved (FR-007).
- **OQ-005 — Mutating-command side-effect safety (and the form-(b) gate)**: For **form (a)**, confirm the failed first attempt of a mutating command did not partially apply (the failure means the command never executed) — expected safe by registry-not-ready semantics. For **form (b)**, this is the gate that decides whether `Stream closed` is enabled at all: force a `Stream closed` against a mutating command (`rename`/`move`/`delete`) targeting a closed vault and inspect the vault state to determine whether the mutation applied on attempt 1 (did the rename/move/delete land? did an append double?). Form (b) is enabled blanket only if `Stream closed` is proven to always fire pre-execution; if it can fire post-mutation, form (b) is dropped entirely (FR-001, FR-011). Pin both the surface form (per OQ-002) and the lifecycle position here before freezing the predicate.
- **OQ-006 — Both dispatch facades exhibit cold-start today**: Confirm the behaviour applies whether a command is issued through a purpose-built tool or the general command passthrough, and that both exhibit the cold-start failure today (FR-009).
- **OQ-007 — Re-confirm ADR-029's empirical claims**: Re-confirm the empirical claims ADR-029 rests on (command-agnostic cold-start; a single retry succeeds once the vault is up) hold across the tested set; surface any divergence so the ADR is corrected before it is ratified.

## Dependencies

- **ADR-029 — Retry Once on Cold-Start Vault-Launch Failure** (Proposed): the decision this spec implements; ratified to Decided when the trigger signature locks (Open Questions resolved) and the retry ships.
- **The shared command-dispatch / classification layer** (ADR-007 — Centralized CLI Bounds with Selective Override; ADR-004 — Centralized Obsidian CLI Adapter): the locus where the retry wraps the existing classification so both dispatch facades inherit it.
- **The upstream lazy-launch behaviour** (Obsidian CLI — Upstream Issues and Limitations): the vault-launch-on-first-call side effect the retry depends on.
