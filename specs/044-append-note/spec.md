# Feature Specification: Append Note

**Feature Branch**: `044-append-note`
**Created**: 2026-05-25
**Status**: Draft
**Input**: User description: "Add Append Note — a typed MCP tool that appends content to the end of an existing vault note in a single call. By default the appended content starts on its own new line; an explicit opt-in joins it directly to the trailing line. Eliminates the read-then-rewrite cycle that today forces callers to ship the whole note through the full-replace write surface just to add a line, and removes the unstructured escape-hatch (universal exec) path that has no per-call validation, no structured errors, and no progressive-disclosure documentation. Targets either an explicit note (vault + vault-relative path, or wikilink-form name) or the note currently focused in the user's editor. Fails loud on missing target, invalid input, and oversized content. Out of scope: auto-creating the target when it does not exist (the full-replace write surface owns creation); appending after a specific heading or after the frontmatter block (the heading-patch surface owns sub-section writes); prepending; mid-file insertion; search-and-replace; partial rewrites; atomicity guarantees beyond what the underlying execution layer already provides; the internal mechanism used to write the bytes (argv pipe vs filesystem-direct vs any future channel — the contract is observable behaviour and a documented size ceiling, not the choice of pipeline)."

## Clarifications

### Session 2026-05-25

- Q: Default-separator behaviour when the target file already ends with a line break — single line break (use the existing trailing newline as the separator), always-insert (producing a blank line of separation), or defer to upstream? → A: **Single line break.** Per FR-006a, when the target file's existing content ends with a line break (`\n` or `\r\n`), the wrapper does NOT insert an additional separator — the existing trailing line break IS the separator. Result for `prior\n` appended with `new_content`: `prior\nnew_content`. When the file ends with a non-newline character, the wrapper inserts the line-ending-convention-matching separator. Rationale: matches the user-facing contract ("starts on its own new line" describes a single new line, not a blank line of separation); byte-stable against the dominant Obsidian shape (notes that already end in `\n`); avoids inflating files by one byte per append in steady-state journaling / log-line / progressive-list workflows. Always-insert was rejected because it would produce a visually-distracting blank line of separation between every appended chunk and the prior content in the common case. Defer-to-upstream was rejected because the cohort discipline is to publish a settled byte-level rule at the wrapper boundary, not to delegate the contract to the substrate where caller behaviour would shift if the upstream's behaviour changed; the plan phase verifies that the upstream `obsidian append` subcommand matches this rule and patches the wrapper if it does not.
- Q: Wikilink-form locator bracket handling — strict bare-name (reject `[[…]]` brackets at the input-validation boundary), trim-and-accept (silently strip brackets before resolution), or accept-both-no-stripping? → A: **Strict bare-name.** Per FR-001a, the wikilink-form name locator MUST be the bare note name only; locators containing `[[` or `]]` characters are rejected at the input-validation boundary with a typed validation error naming the offending characters and the expected shape. Cohort parity with every existing `file`-parameter tool (`read`, `delete`, `rename`, `files`, `set_property`, and the rest) — none of them strip brackets today, and adopting silent stripping on this one tool would surprise callers who switch between read-side and write-side surfaces. Rejection produces a clear typed error at the schema boundary, before any filesystem access or subprocess invocation. Trim-and-accept was rejected because it breaks cohort parity and silently normalises caller input — callers who include brackets are reaching for the wrong locator shape and deserve a diagnostic, not a silent fix-up. Accept-both-no-stripping was rejected because passing the literal `[[Note]]` to the upstream resolver produces a worse caller experience (post-validation `NOTE_NOT_FOUND` after the wrapper has already initiated subprocess work) than a clear pre-validation error.
- Q: Active-mode explicit-opt-in requirement (cohort parity with `write_note`'s `overwrite: true`) — require an explicit opt-in flag in active mode, no opt-in required (cohort exception), or conditional opt-in (only when the inline opt-in is also enabled)? → A: **No opt-in required (cohort exception).** Per FR-004a, an active-mode append requires only `target_mode: "active"` plus `content` (subject to the locator-mutually-exclusivity rule of FR-014). The `write_note` cohort precedent (mandatory `overwrite: true` in active mode, rationale "active mode is destructive by definition") does NOT apply: `write_note` replaces the focused note's entire prior content with the supplied content (wrong-target = total destruction of prior content), whereas `append` only adds bytes at end-of-file (wrong-target = additive noise at the end of an unintended note, recoverable by removing the appended tail). The asymmetric safety profile justifies the asymmetric opt-in posture. Requiring an opt-in flag would impose ergonomic friction on the dominant authoring case (rapid in-editor journaling, list growth, table-row addition against the focused note) without a matching safety benefit. Strict-cohort-parity (Option A) was rejected because it conflates destructive overwrite with additive append. Conditional-opt-in tied to the inline flag (Option C) was rejected because the inline opt-in already requires explicit caller action — layering a second opt-in on top would be redundant ceremony.
- Q: Content payload's own trailing newline handling — preserve the caller's `content` bytes verbatim (no trim, no normalisation), auto-trim trailing `\n` / `\r\n`, or auto-append `\n` if missing? → A: **Verbatim.** Per FR-010a, the wrapper writes the caller's `content` bytes byte-for-byte after the FR-006 / FR-006a separator decision; it does NOT strip, trim, normalise, or auto-append trailing newlines in the content. The caller chooses whether to include a trailing `\n` and lives with the consequence — a `content` ending in `\n` leaves the resulting file ending in `\n`, which under the next default-mode append triggers FR-006a (existing trailing newline IS the separator) and produces no blank line. Cohort parity with `write_note`'s `content` handling (whole-file write, byte-for-byte). Auto-trim (Option B) was rejected because it silently mutates caller input and breaks symmetry with `write_note`; callers using the line-builder pattern (no trailing `\n`) and callers using the paragraph-builder pattern (trailing `\n` included) both deserve byte-stable behaviour. Auto-append-`\n` (Option C) was rejected for the same reason — and because the canonical "ends with `\n`" Obsidian shape is preserved naturally under verbatim semantics when the caller cooperates, without the wrapper imposing it.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Append to a named note with default separator semantics (Priority: P1)

A vault author identifies a note by either its vault-relative path or its wikilink-form name (the bare note name, the same shape Obsidian resolves inside `[[…]]` links) and supplies content to append. The wrapper writes the content at the end of the named note. By default the appended content begins on its own new line so that bullet lists, journal entries, tables, and successive appended chunks remain visually and structurally readable. The wrapper preserves every byte of the prior content. The response identifies the file by its vault-relative path so the caller can attribute the change or roll it back.

**Why this priority**: The whole feature exists to give authors a single-call append surface anchored to an existing note. Without P1 there is no MVP. The default-separator behaviour is what makes the surface usable for the dominant authoring intent — progress logs, list growth, journal entries, table rows — where every appended chunk wants to be its own line. P1 stands alone as a viable shipped slice: an author who can name a note and append a line at the end already has every authoring shape this feature aims to deliver, plus the response payload needed to track or undo the change. P1 covers BOTH locator shapes (vault-relative path and wikilink-form name) because the cohort's CLI-wrapper write tools accept both natively and an asymmetry here would surprise callers.

**Independent Test**: Stand up a vault with a note containing prior content. Issue the operation against that note using its vault-relative path, and then again against a second note using its wikilink-form name. For each call, verify that (a) the appended content is present at the end of the file, (b) a line break sits between the prior trailing line and the new content, (c) every byte of the prior content is byte-stable from the pre-call state, and (d) the response payload names the vault-relative path of the file that was written.

**Acceptance Scenarios**:

1. **Given** an existing note that contains prior content and ends on a non-empty line (no trailing line break), **When** the caller invokes the tool with the note's vault-relative path and replacement content, **Then** the new content appears at the end of the file with a single line-ending-convention-matching separator inserted between the prior trailing content and the new content, the prior bytes are unchanged, and the response identifies the file by its vault-relative path.
2. **Given** an existing note whose content ends with a line break (`\n` or `\r\n` — the dominant Obsidian shape), **When** the caller invokes the tool with the note's vault-relative path and replacement content under the default-separator behaviour, **Then** the new content appears immediately after the existing trailing line break with NO additional separator inserted (per FR-006a — the existing trailing line break IS the separator), and no blank line of separation appears between the prior content and the new content.
3. **Given** an existing note identified by its wikilink-form name (the bare note name, no `[[…]]` brackets — the same shape the cohort's read-side tools accept under the `file` locator parameter), **When** the caller invokes the tool with that name and content, **Then** the wrapper resolves the name to the matching vault note, the append lands on that note, and the response identifies the file by its vault-relative path (the resolved canonical path, not the wikilink-form input the caller supplied).
4. **Given** the target is a 0-byte file, **When** the caller appends content with default settings, **Then** no leading line break is inserted and the file contains only the appended content.
5. **Given** a successful append, **When** the call completes, **Then** the response identifies the patched note's vault-relative location so the caller can attribute the change or roll it back.

### User Story 2 - Distinguish unresolvable-target and input failures through typed errors (Priority: P2)

A vault author who supplies a note locator that does not correspond to an existing note, an empty content payload, both a path and a wikilink-form name simultaneously, no locator at all when one is required, a locator while intending to target the focused note, an unknown extra input field, or content that exceeds the supported size ceiling, receives a typed, structured error identifying which failure mode occurred. No file is created. No file is modified. The caller can switch on the failure mode directly — no prose parsing.

**Why this priority**: P1 gives the happy path. Without P2, an author who mistypes a path, ships an empty content payload, or attempts to send a chunk above the size ceiling has no programmatic way to learn which problem they hit other than reading the inputs and the vault state side-by-side. Worse, an undetected missing-target could be misread as a silent success — a future caller could re-read the would-have-been-appended location and find stale content. P2 is the diagnostic contract that makes failure self-describing — "note not found", "validation error" (with sub-discriminators for the specific input violation), and "content too large" are programmatically distinguishable states. With P1+P2 the caller has both the operation and the feedback loop needed to use it reliably. The size-ceiling failure mode is non-negotiable because the underlying execution layer has a hard limit (whether argv-pipe length, in-memory buffer size, or any successor channel) and a clear typed error preventing silent corruption, hangs, or opaque failures is a contract obligation.

**Independent Test**: With P1 in place, issue the operation naming a note path or wikilink-form name that does not correspond to any existing note in the vault and verify the response is a typed error programmatically distinguishable as "note not found" and that no note is created. Issue the operation with each of the following inputs in turn — empty content payload, both `path` and the wikilink-form locator supplied at the same time, neither locator supplied while explicit-target mode is requested, a locator supplied while focused-note mode is requested, an unknown extra input field — and verify each is rejected at the input-validation boundary with a typed error naming the violation, before any filesystem access or subprocess invocation. Issue the operation with content that exceeds the supported size ceiling and verify the response is a typed error naming the size limit and that the file on disk is byte-identical to its pre-call state.

**Acceptance Scenarios**:

1. **Given** a note locator that does not correspond to any existing note in the vault, **When** the caller invokes the tool, **Then** the operation fails with a typed error programmatically identifying the failure as "note not found", the error payload names the missing locator, and no note is created.
2. **Given** an empty content payload, **When** the caller invokes the tool, **Then** the request is rejected at the input-validation boundary before any filesystem access or subprocess invocation, the typed error names the offending field and the reason, and no file is modified.
3. **Given** both an explicit path locator and a wikilink-form name locator supplied in the same call, **When** the caller invokes the tool, **Then** the request is rejected at the input-validation boundary, the typed error names the mutually-exclusive-locators violation, and no file is modified.
4. **Given** explicit-target mode is requested but no locator (neither vault-relative path nor wikilink-form name) is supplied, **When** the caller invokes the tool, **Then** the request is rejected at the input-validation boundary, the typed error names the missing-locator violation, and no file is modified.
5. **Given** focused-note mode is requested AND a locator is supplied, **When** the caller invokes the tool, **Then** the request is rejected at the input-validation boundary, the typed error names the locator-not-allowed-in-active-mode violation, and no file is modified.
6. **Given** an unknown extra input field is present in the call, **When** the caller invokes the tool, **Then** the request is rejected at the input-validation boundary, the typed error names the unknown field, and no file is modified. (Strict-mode schema, cohort parity with `write_note`.)
6a. **Given** a wikilink-form name locator supplied with the literal `[[…]]` brackets (e.g. `[[My Note]]`), **When** the caller invokes the tool, **Then** the request is rejected at the input-validation boundary, the typed error names the offending bracket characters and the expected bare-name shape, and no filesystem access or subprocess invocation occurs. (Cohort parity with every existing `file`-parameter tool — none strip brackets.)
7. **Given** a content payload that exceeds the supported size ceiling, **When** the caller invokes the tool, **Then** the operation fails with a typed error naming the size limit and the file on disk is byte-identical to its pre-call state.
8. **Given** a content payload at or below the supported size ceiling, **When** the caller invokes the tool, **Then** the operation succeeds end-to-end.
9. **Given** a caller consults the tool's documentation, **When** the caller reads it, **Then** the supported size ceiling is stated explicitly so the caller can predict the limit ahead of time without trial-and-error.

### User Story 3 - Inline (no-separator) opt-in for finishing a partial trailing line (Priority: P2)

A vault author who has a note whose trailing line ends with a partial token (an in-progress sentence, a half-built list-item bullet, a table-row prefix awaiting its remaining cells) supplies content with an explicit "inline" opt-in. The wrapper fuses the new content directly onto the trailing line with no intervening line break. Every byte of the prior content remains unchanged.

**Why this priority**: P1's default-separator behaviour serves the dominant authoring intent (new line per append). The inline-fuse intent is the legitimate minority case — finishing a partial line, building up a single composite line across multiple calls, completing a half-typed table row. Without P3, that authoring intent has no surface — the only workaround is to read the trailing line, splice the new content client-side, and ship the full result through the full-replace write surface, which is exactly the wastage the feature exists to eliminate. P3 is also a contract symmetry obligation: the upstream CLI exposes the equivalent flag, and the wrapper's job is to publish that surface in a typed, validated, structured-error form rather than gate-keep it. P3 is at the same priority tier as P2 because both layer onto P1 directly and neither depends on the other; the caller can use P1 alone, P1+P2 (diagnostics), P1+P3 (inline opt-in), or P1+P2+P3 (the full slice).

**Independent Test**: Stand up a vault with a note whose trailing line ends with the partial token `Partial` and no trailing newline. Issue the operation against that note with content `Tail` and the inline opt-in. Verify that (a) the last line of the resulting file reads `PartialTail` with no intervening line break, (b) no byte of the prior content other than the trailing line's continuation is changed, and (c) the response identifies the file by its vault-relative path.

**Acceptance Scenarios**:

1. **Given** an existing note whose trailing line ends with the partial token `Partial`, **When** the caller invokes the tool with content `Tail` and the inline opt-in enabled, **Then** the resulting trailing line reads `PartialTail` with no intervening line break and every other byte of the file is unchanged from its pre-call state.
2. **Given** a successful inline append, **When** the call completes, **Then** the response identifies the patched note's vault-relative location, identical in shape to the default-separator success response.

### User Story 4 - Operate on the focused note (Priority: P3)

A vault author working with a note focused in their editor appends content to that note without restating its location. When no note is focused, the operation fails with a typed error directing the author to focus a note or to supply an explicit locator. No filesystem access occurs and no write happens in the no-focused-note failure case.

**Why this priority**: P1 already accepts an explicit note locator (vault-relative path or wikilink-form name), so the feature is shippable without P4. P4 layers a convenience surface on top: an author who is already working in a note does not need to restate its path or name on every append. This is cohort parity with the active-mode surface across the rest of the typed-tool family — every typed tool that accepts a note locator also accepts a focused-note variant via the cohort's `target_mode: "active"` discriminator (ADR-003, ADR-004). P4 stands independently — it can be added after P1+P2+P3 without disturbing them, and P1 callers who always pass an explicit locator are unaffected.

**Independent Test**: With P1+P2+P3 in place, focus a known note in the user's editor, issue the operation using the focused-note locator (no explicit vault/path/wikilink-form name supplied), and verify the append lands on the focused note. Clear the focused-note state and issue the same request, and verify the operation fails with a typed error directing the author to focus a note or to supply an explicit locator — and that no filesystem access occurred.

**Acceptance Scenarios**:

1. **Given** a note is focused in the author's editor, **When** the caller submits a focused-note append request with `target_mode: "active"` plus content and no further opt-in flag (per FR-004a — append is a deliberate cohort exception to `write_note`'s mandatory `overwrite: true` in active mode), **Then** the append lands on the focused note in the same way as when the note is named explicitly, and the response identifies the file by its vault-relative path (resolved at request time from the focused-note state).
2. **Given** no note is focused, **When** the caller submits a focused-note append request, **Then** the operation fails with the existing active-mode cohort error directing the caller to focus a note or to supply an explicit locator; no filesystem access is performed and no write occurs.

### Edge Cases

- **Target file does not exist (any locator shape — path, wikilink-form name, or focused-note resolution)**: per FR-016, the operation fails with `code: "CLI_REPORTED_ERROR"` + `details.code: "NOTE_NOT_FOUND"` and the file is not auto-created. Auto-create is the full-replace write surface's responsibility (cohort parity — `write_note` owns creation; `append` assumes existence). This is the deliberate scope split per the user's Out of scope statement.
- **Target is a 0-byte file (default separator)**: per FR-009, no leading line break is inserted; the file becomes exactly the appended content. This is the only case in which the default-separator behaviour does not insert a leading line break.
- **Target's trailing line ends with a line break already (default separator)**: per FR-006a (settled 2026-05-25), the wrapper does NOT insert an additional separator — the existing trailing line break (`\n` or `\r\n`) IS the separator. Result for a file `prior\n` appended with `new_content` under the default-separator behaviour: `prior\nnew_content`. No blank line of separation is introduced.
- **Note's line-ending convention is CRLF**: the wrapper preserves the note's existing line-ending convention. When the default separator inserts a new line, the bytes of that separator match the note's existing convention (LF or CRLF). The wrapper does not invent a new convention or convert between conventions.
- **Trailing newline convention**: the wrapper preserves the note's existing trailing-newline convention except where the append itself necessarily alters it. A note that ended with `\n` before the append still ends in a way consistent with the underlying append semantics; the wrapper does not strip or invent a trailing newline beyond what the append operation produces.
- **Inline opt-in against a 0-byte file**: accepted — the file becomes exactly the appended content (same observable result as the default-separator behaviour against a 0-byte file). Inline-vs-default only differs when there is prior content to fuse against.
- **Inline opt-in against a file whose trailing line ends with a line break**: the appended content fuses onto whatever the underlying execution layer considers the "trailing line" — which, for a file ending in `\n`, may be either the last non-empty line or a fresh empty trailing line. The wrapper publishes the observable contract (no leading line break inserted by the wrapper) and inherits the underlying behaviour. See Assumptions.
- **Content payload contains line breaks**: accepted as-is. Whether the resulting markdown is structurally valid (e.g. a list-item-shaped first chunk followed by a paragraph-shaped second chunk) is the caller's responsibility. The wrapper does not invent shape-specific structural validation.
- **Content payload itself ends with a line break (`\n` or `\r\n`)**: per FR-010a (settled 2026-05-25), preserved verbatim. The wrapper does not strip, trim, or normalise the trailing newline. A caller using a paragraph-builder pattern (`content` ends with `\n` to "finish" the paragraph) leaves the file ending in `\n`; the next default-mode append against that file triggers FR-006a (existing trailing line break IS the separator) and produces no blank line. A caller using a line-builder pattern (`content` without trailing `\n`) leaves the file ending on a non-newline character; the next default-mode append inserts the line-ending-convention-matching separator per FR-006. Both patterns produce byte-stable concatenation under repeated default-mode appends; the caller chooses which pattern by choosing whether to include the trailing newline.
- **Wikilink-form name resolves to multiple notes in the vault** (ambiguous shortname when no folder path is supplied): the wrapper inherits the underlying execution layer's resolution rule (cohort parity with `read`, `delete`, `rename`, and the rest of the read-side and write-side tools that accept the `file` locator parameter). The wrapper does not introduce a new ambiguous-name typed error; if the underlying layer raises one, it surfaces unchanged.
- **Wikilink-form name resolves to no note**: per FR-016, surfaces as `NOTE_NOT_FOUND`. Same shape as the vault-relative-path-not-found case — one typed error, not two.
- **Locator passed with the literal `[[…]]` brackets (the full wikilink syntactic shape)**: per FR-001a (settled 2026-05-25), rejected at the input-validation boundary with a typed validation error naming the offending bracket characters and the expected bare-name shape. The wrapper does NOT silently strip brackets — cohort parity with every existing `file`-parameter tool (`read`, `delete`, `rename`, `files`, `set_property`, and the rest). No filesystem access or subprocess invocation occurs.
- **Active mode with `file`/`path` supplied (i.e. `target_mode: "active"` plus a locator)**: per FR-014, rejected at the input-validation boundary with a typed validation error naming the conflict. Cohort parity with `write_note`'s active-mode discipline.
- **Unknown extra input fields**: per FR-015, the input schema is strict; unknown fields are rejected at the input-validation boundary with a typed error naming the offending field. Cohort parity with `write_note`, `patch_heading`, and the strict-mode-by-default discipline.
- **External editor has the target open with unsaved changes**: when the underlying execution layer signals the condition to the wrapper, the operation fails loud per the cohort `EXTERNAL_EDITOR_CONFLICT` discriminator (`details.reason` ∈ `unsaved-changes` / `file-locked`). When the layer does not signal the condition, the append lands and the editor sees a refreshed file on next focus. This detection-capability caveat is inherited from the cohort (BI-040 `patch_heading` FR-021, BI-043 `patch_block` FR-021); the spec does not invent new detection mechanics.

## Requirements *(mandatory)*

### Functional Requirements

#### Locator and addressing

- **FR-001**: System MUST accept exactly one of three target shapes per call: an explicit vault-relative path locator, an explicit wikilink-form name locator (the bare note name — same shape the cohort's read-side tools accept under their `file` parameter), or a focused-note locator referencing the note currently open in the author's editor. The vault-relative path locator and the wikilink-form name locator are mutually exclusive — the caller supplies one or the other, not both. The focused-note locator is mutually exclusive with both explicit locator shapes.
- **FR-001a**: System MUST reject wikilink-form name locators that contain the `[[` or `]]` bracket characters at the input-validation boundary, before any filesystem access or subprocess invocation, with a typed validation error naming the offending characters and the expected shape (bare note name without brackets). The wrapper does NOT silently strip brackets or otherwise normalise the locator — cohort parity with every existing `file`-parameter tool. Settled by Clarifications Session 2026-05-25; rationale and rejected alternatives are recorded there.
- **FR-002**: System MUST resolve the wikilink-form name locator using the cohort's existing name-resolution rule (the same rule the read-side `file` parameter follows). The wrapper does not introduce a new resolution algorithm.
- **FR-003**: System MUST canonicalise the response's reported file location to the vault-relative path of the file that was actually written, regardless of which locator shape the caller supplied. A caller who supplies a wikilink-form name receives the resolved vault-relative path in the response, not the wikilink-form name they supplied.
- **FR-004**: When the focused-note locator is requested but no note is focused, System MUST fail with the existing active-mode cohort error (whichever top-level `code` the active-mode ADR specifies for "no focused note"; this BI introduces no new sub-discriminator for this condition). No filesystem access MUST occur and no write MUST be performed.
- **FR-004a**: System MUST NOT require an explicit opt-in flag (e.g. `confirmActive: true` or equivalent) for active-mode append requests. A valid active-mode call needs only `target_mode: "active"` plus a non-empty `content` payload (subject to the locator-mutually-exclusivity rule of FR-014). This is a deliberate cohort exception to `write_note`'s mandatory `overwrite: true` in active mode — append is additive (wrong-target = recoverable additive noise) rather than destructive (wrong-target = total prior-content destruction), and the asymmetric safety profile justifies the asymmetric opt-in posture. Settled by Clarifications Session 2026-05-25; rationale and rejected alternatives are recorded there.

#### Append placement and separator semantics

- **FR-005**: System MUST support exactly one append operation — write the supplied content at the end of the target file. The operation has no mid-file placement, no prepend, no after-heading, no after-frontmatter, and no search-and-replace modes; callers needing those shapes use the full-replace write surface or the heading-patch surface per the published scope split.
- **FR-006**: System MUST default the separator behaviour to "the appended content begins on its own new line" — that is, when the target file has prior content, a line break sits between the prior trailing content and the new content in the resulting file.
- **FR-006a**: When the target file's existing content ends with a line break (`\n` or `\r\n`), System MUST NOT insert an additional separator under the default-separator behaviour — the existing trailing line break IS the separator. The wrapper inserts a separator under the default-separator behaviour ONLY when the file's existing content does not already end with a line break. Settled by Clarifications Session 2026-05-25; rationale and rejected alternatives are recorded there.
- **FR-007**: System MUST accept an explicit "inline" opt-in input field. When the opt-in is enabled, the appended content fuses directly onto the trailing content with no line break inserted by the wrapper. The opt-in is opt-in only — its default value is the default-separator behaviour of FR-006.
- **FR-008**: System MUST preserve the note's existing line-ending convention (LF, CRLF, or other). Any line break the wrapper inserts under the default-separator behaviour MUST match the note's existing convention. The wrapper MUST NOT convert between line-ending conventions.
- **FR-009**: When the target file is 0 bytes, System MUST NOT insert a leading line break under the default-separator behaviour. The resulting file contains exactly the appended content.
- **FR-010**: System MUST preserve every byte of the prior content. The append operation MUST NOT modify any byte of the file other than (a) the bytes the wrapper inserts as part of the separator (under the default-separator behaviour) and (b) the bytes of the appended content itself.
- **FR-010a**: System MUST write the caller's `content` payload byte-for-byte verbatim — no trimming, no normalisation, no auto-appended trailing newline, no character-set conversion beyond what the transport layer already requires. A `content` payload supplied with a trailing `\n` results in a file whose new tail ends with `\n`; a `content` payload supplied without a trailing `\n` results in a file whose new tail does NOT end with `\n`. Subsequent default-mode appends against the resulting file follow FR-006a (existing trailing line break IS the separator) when applicable. Settled by Clarifications Session 2026-05-25; rationale and rejected alternatives are recorded there. Cohort parity with `write_note`'s `content` handling.
- **FR-011**: System MUST NOT modify YAML frontmatter. The append lands at end-of-file, and end-of-file is below the frontmatter block by definition; frontmatter writes are handled by the frontmatter-write capability.
- **FR-012**: System MUST NOT auto-create the target file when it does not exist. File creation is handled by the full-replace write surface per the published scope split.

#### Input validation

- **FR-013**: System MUST reject an empty content payload at the input-validation boundary, before any filesystem access or subprocess invocation, with a typed validation error naming the offending field and the reason.
- **FR-014**: System MUST reject any input that supplies both an explicit locator (vault-relative path or wikilink-form name) AND a focused-note request in the same call. The locator shapes per FR-001 are mutually exclusive; mixing them surfaces as a typed validation error at the input-validation boundary.
- **FR-015**: System MUST reject unknown extra input fields at the input-validation boundary with a typed validation error naming the offending field. The input schema is strict; cohort parity with `write_note`, `patch_heading`, and the strict-mode-by-default discipline.
- **FR-016**: When the resolved target does not correspond to an existing note in the vault (whether the locator was a vault-relative path, a wikilink-form name, or a focused-note resolution that pointed to a path the vault no longer carries), System MUST fail with `code: "CLI_REPORTED_ERROR"` + `details.code: "NOTE_NOT_FOUND"` (cohort parity with the existing read-side `NOTE_NOT_FOUND` discriminator). No note MUST be created.

#### Size ceiling and oversized content

- **FR-017**: System MUST publish a maximum supported content-payload size — the "size ceiling". The ceiling is a single concrete number stated in the tool's documentation; callers can predict the limit ahead of time without trial-and-error. The ceiling's exact value (and the underlying constraint that motivates it — argv-pipe length, in-memory buffer cap, or any successor channel's analogue) is settled at the plan phase and codified in the tool's published help text.
- **FR-018**: When the content payload exceeds the size ceiling, System MUST fail with a typed validation error naming the size ceiling, before any filesystem access or subprocess invocation. The file on disk MUST be byte-identical to its pre-call state.
- **FR-019**: When the content payload is at or below the size ceiling, System MUST complete the append operation end-to-end (subject to the other failure modes specified in FR-004, FR-013-FR-016, FR-021).

#### Response shape

- **FR-020**: On a successful append, System MUST return a response identifying the vault-relative location of the file that was written. The response shape supports caller attribution and rollback. Cohort parity: write-side tools echo the locator for write-verification (per the project's read-vs-write echo convention).

#### Atomicity and external-editor conflict

- **FR-021**: A single invocation MUST NOT leave the note on disk in a half-written or otherwise inconsistent state at any observable instant within the wrapper's control. The wrapper inherits whatever single-invocation atomicity the underlying execution layer provides; the spec does not invent new atomicity guarantees. Interrupted invocations (power loss, kill -9) leave on-disk state to the operating system.
- **FR-022**: When the target note is open in an external editor with unsaved changes AND the underlying execution layer signals the condition to the wrapper, System MUST fail loud with `code: "CLI_REPORTED_ERROR"` + `details.code: "EXTERNAL_EDITOR_CONFLICT"` + `details.reason` set to `"unsaved-changes"` or `"file-locked"` matching the OS-level signal; the file on disk MUST NOT be modified. When the underlying layer does not signal the condition, the edit lands and the editor sees a refreshed file on next focus — this detection-capability caveat is inherited from the cohort (BI-040 FR-021, BI-043 FR-021) and is documented in the tool's help text.
- **FR-023**: For every failure mode in FR-004, FR-013-FR-016, FR-018, and FR-022, System MUST NOT modify the file on disk. "Failed loud" means the typed error is the only side effect visible to the caller and the filesystem.

#### Out of scope at the contract boundary

- **FR-024**: System MUST NOT accept any placement mode other than the end-of-file append defined in FR-005. After-heading, after-frontmatter, prepend, mid-file insertion, search-and-replace, and partial rewrites are out of scope; callers needing those shapes use the heading-patch surface, the prepend surface (a separate future feature), or the full-replace write surface per the published scope split.
- **FR-025**: System MUST NOT auto-create the target file. File creation is handled by the full-replace write surface per the published scope split (FR-012).
- **FR-026**: System MUST NOT publish any transactional contract across concurrent invocations beyond the single-invocation atomicity of FR-021. The published cross-invocation contract is last-write-wins; callers needing stronger guarantees coordinate externally.
- **FR-027**: System MUST NOT publish any guarantee about the internal mechanism used to write the bytes (argv-pipe length, filesystem-direct read-modify-write, or any future channel). The published contract is the observable behaviour (FR-001 through FR-023) plus the documented size ceiling (FR-017). The plan phase chooses the pipeline; the spec describes what the caller sees.

### Key Entities

- **Note**: A markdown file inside the vault, addressed by vault + vault-relative path, by wikilink-form name (the bare note name, resolved by the same rule as the cohort's read-side `file` parameter), or by the focused-note locator. Contains an optional YAML frontmatter block followed by a body composed of paragraphs, lists, tables, fenced code, callouts, and other markdown constructs. Has a line-ending convention (LF / CRLF) and a trailing-newline convention that the append operation preserves.
- **Append content**: The string of bytes the caller supplies for the wrapper to write at the end of the target file. Must be non-empty (FR-013). Must not exceed the size ceiling (FR-017, FR-018). May contain line breaks (caller responsibility for resulting markdown shape).
- **Default separator**: A line break inserted by the wrapper between the prior trailing content and the new content, present by default (FR-006) and matching the note's existing line-ending convention (FR-008). Absent when the file is 0 bytes (FR-009) or when the inline opt-in (FR-007) is enabled.
- **Inline opt-in**: A boolean input field whose default value preserves the default-separator behaviour and whose enabled value suppresses the wrapper-inserted separator so that the appended content fuses directly onto the prior trailing content.
- **Size ceiling**: A single concrete number stating the maximum supported content-payload size (FR-017). Published in the tool's help text. Exceeded payloads surface as a typed validation error (FR-018) without modifying the file on disk.
- **Focused note**: The note the user currently has open in their editor. Resolved at request time; failure to resolve (no note focused) surfaces a typed error per FR-004.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: A vault author can append content to any existing note in the vault — addressed by vault-relative path, by wikilink-form name, or by the focused-note locator — with a single call, without first reading the note, computing a splice, or shipping any byte of the prior content back to the wrapper.
- **SC-002**: For every append operation that succeeds, every byte of the prior content (every other line, every other character, the frontmatter, the line-ending convention) is unchanged from the pre-call state. Only the bytes the wrapper inserts as separator (under the default-separator behaviour) plus the bytes of the appended content itself are added.
- **SC-003**: Every failure mode (note not found, empty content, mutually-exclusive locators, missing locator in explicit mode, locator supplied in active mode, unknown extra field, content exceeds size ceiling, no focused note in active mode, external-editor conflict) surfaces as a typed, programmatically distinguishable error. No failure mode produces a silently-empty success, a silently-wrong write, or a half-written file.
- **SC-004**: Repeat invocations of the same request against an unchanged vault produce identical outcomes (same write, same response) — the operation is deterministic given a fixed input and a fixed vault state.
- **SC-005**: An author's payload size for a single append is bounded by the size of the appended content plus the locator metadata — *not* by the size of the surrounding note. A 50-byte append against a 50,000-byte note moves 50 bytes plus the locator, not 50,050 bytes round-trip. This is the wastage the feature exists to eliminate.
- **SC-006**: The response payload identifies the appended file's vault-relative location, enabling caller attribution and rollback without re-reading the vault.
- **SC-007**: The size ceiling stated in the tool's published documentation matches the value the wrapper actually enforces — callers who size their payloads against the documented number do not encounter unexpected oversize failures, and callers who exceed the documented number receive the typed oversize error before any filesystem access.

## Assumptions

- **Native CLI subcommand exists**: The upstream Obsidian Integrated CLI exposes a native `append` subcommand with `file=<name>` / `path=<path>` / `content=<text>` / `inline` parameters. This is the wrapping target and the source of the default-separator and inline-opt-in semantics. The wrapper publishes a typed, validated, structured-error surface over the same observable contract. Confirmed by inspection of the local CLI's help output at spec time; the plan phase confirms the version pin and the exact behaviour against the authorised test vault per `.memory/test-execution-instructions.md`.
- **Tool name follows ADR-010**: Because a native upstream `append` subcommand exists, the wrapper's tool name mirrors it exactly — `append` — per ADR-010 (Typed Tool Names Mirror Upstream CLI Subcommand). Cohort precedent: `read`, `delete`, `rename`, `files`, `set_property`. The "Append Note" feature title is the human-facing name; the tool's registry name is the CLI subcommand name.
- **Locator schema cohort parity**: The two explicit locator shapes (vault-relative path under `path`, wikilink-form name under `file`) and the focused-note locator (via `target_mode: "active"`) follow the locator-shape conventions established by the cohort under ADR-003 / ADR-004 / ADR-013 — same shape as `write_note`, `patch_heading`, `patch_block`, and the read-side tools. The plan phase confirms the exact schema shape; the spec describes the contract.
- **Error code cohort parity**: Typed errors surface under the existing top-level error codes per Constitution Principle IV — zero new top-level codes; Principle IV streak preserved. `NOTE_NOT_FOUND` (FR-016) parallels the read-side `NOTE_NOT_FOUND` discriminator. `EXTERNAL_EDITOR_CONFLICT` (FR-022) reuses BI-040's discriminator unchanged with the same `details.reason` enum. The "no focused note" condition (FR-004) reuses the existing active-mode cohort discriminator. Validation-error sub-discriminators (FR-013 empty-content, FR-014 mutually-exclusive-locators, FR-015 unknown-field, FR-018 content-too-large) surface under `VALIDATION_ERROR` with `details.code` sub-discrimination per ADR-015; the exact `details.code` and `details.reason` values are settled at the plan phase against the cohort's existing validation-error vocabulary.
- **Size ceiling exact value deferred to plan**: The user's Out of scope statement explicitly defers "the internal mechanism used to write the bytes" to the plan phase. The size ceiling is the observable consequence of whichever pipeline the plan picks (argv-pipe length, in-memory buffer cap, or any successor channel); the spec publishes the contract (a ceiling exists, it is documented, the error names it) and the plan phase chooses the value. Cohort: if the plan picks the CLI-wrapper path (passing `content=<text>` via argv to `obsidian append`), the ceiling is the platform argv limit minus a safety margin — practical examples include the Windows command-line maximum (~32,767 chars) and Linux's `ARG_MAX` (~128 KiB); the plan picks a portable number that is safe on every supported platform.
- **Default-separator interaction with existing trailing newline**: Settled by Clarifications Session 2026-05-25 — per FR-006a, when the target file's existing content ends with a line break (`\n` or `\r\n`), the wrapper does NOT insert an additional separator; the existing trailing line break IS the separator. Result for `prior\n` appended with `new_content`: `prior\nnew_content`. Always-insert was rejected because it would produce a visually-distracting blank line of separation between every appended chunk and the prior content in the common case (notes ending in `\n`). Defer-to-upstream was rejected because the cohort discipline is to publish a settled byte-level rule at the wrapper boundary, not to delegate the contract to the substrate where caller behaviour would shift if the upstream's behaviour changed. The plan phase verifies that the upstream `obsidian append` subcommand matches this rule and patches the wrapper if not.
- **Wikilink-form bracket handling**: Settled by Clarifications Session 2026-05-25 — per FR-001a, wikilink-form locators containing `[[` or `]]` brackets are rejected at the input-validation boundary with a typed validation error. The wrapper does NOT silently strip brackets. Cohort parity with every existing `file`-parameter tool (none of them strip brackets today). Trim-and-accept and accept-both-no-stripping alternatives were rejected; rationale recorded in Clarifications.
- **Empty content rejection**: Settled at FR-013 — empty content payloads are validation errors. Rationale: appending zero bytes is a no-op masquerading as an operation; surfacing it as a typed validation error gives the caller a clear signal that no work was requested and prevents the wrapper from invoking the underlying execution layer for nothing.
- **Markdown shape responsibility**: The wrapper does not invent shape-specific structural validation on the content payload — callers may append line breaks, partial list items, partial table rows, or any other shape, and whether the resulting markdown is structurally valid is the caller's responsibility. Cohort parity with the rest of the write-side tools' "shape is the caller's responsibility" discipline.
- **Cross-invocation concurrency**: Per the user's explicit out-of-scope statement, the wrapper publishes no transactional contract across concurrent invocations beyond the single-invocation atomicity of FR-021. The cross-invocation behaviour is last-write-wins; callers needing stronger guarantees coordinate externally. This is the deliberate scope contract, not a defect to be patched later.
- **Pipeline-agnostic contract**: Per FR-027 and the user's explicit out-of-scope statement, the spec does not commit the wrapper to a specific write mechanism (argv pipe, filesystem-direct, any future channel). The plan phase chooses the pipeline based on the size-ceiling trade-off, the cohort substrate, and the constitution's in-tree-bias rule for dependencies.
