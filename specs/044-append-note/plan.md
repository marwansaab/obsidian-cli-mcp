# Implementation Plan: Append Note

**Branch**: `044-append-note` | **Date**: 2026-05-25 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/044-append-note/spec.md`

## Summary

Add a typed MCP tool `append_note` that writes the caller's `content` payload at the end of an existing vault note in a single call, with a default-separator behaviour that places the new content on its own new line (FR-006 / FR-006a — settled by Clarifications) and an opt-in `inline` mode that fuses the new content directly onto the trailing line (FR-007). Single placement mode — *append-at-end-of-file*; no mid-file insertion, no after-heading, no after-frontmatter, no prepend (callers needing those shapes use `write_note`, `patch_heading`, `patch_block`, or a future `prepend` surface). The wrapper performs fs read-modify-write through the existing reliable-writer substrate (ADR-009): vault-registry resolution, two-layer path safety, atomic write-temp-then-rename, post-write `metadataCache` invalidation eval. The default-separator rule is implemented in TS (the wrapper inspects the file's last-byte to decide whether to insert a line-ending-convention-matching separator per FR-006a). `content` is preserved byte-for-byte verbatim per FR-010a (no trim, no normalise, no auto-appended trailing newline). Active mode requires only `target_mode: "active"` plus `content` — no opt-in flag (FR-004a: deliberate cohort exception to `write_note`'s mandatory `overwrite: true`, justified by the additive-vs-destructive safety asymmetry settled at Clarifications). Errors surface through `UpstreamError` with the existing `CLI_REPORTED_ERROR` / `VALIDATION_ERROR` / `PATH_ESCAPES_VAULT` / `FS_WRITE_FAILED` / `VAULT_NOT_FOUND` / `ERR_NO_ACTIVE_FILE` top-level codes; one new `details.code` value (`CONTENT_EMPTY` under `VALIDATION_ERROR` for FR-013) surfaces via `details.code` sub-discrimination per ADR-015. `NOTE_NOT_FOUND` is reused from the read-side cohort unchanged. Wikilink-form bracket rejection (FR-001a) surfaces under the existing `STRUCTURALLY_UNSAFE_PATH_MESSAGE` channel via a `file`-field refinement that augments the cohort's structural path-safety check. Constitution Principle IV streak preserved — count becomes nineteen tools after BI-044 ships.

## Technical Context

**Language/Version**: TypeScript 5.x (strict mode, `tsc --noEmit` clean)
**Primary Dependencies**: `@modelcontextprotocol/sdk` (MCP transport), `zod` (boundary validation), `zod-to-json-schema` (MCP `inputSchema` publication via the centralised `registerTool` factory). No new runtime dependencies — append-with-separator semantics are in-tree (a ~30-40 LOC pure function in `src/tools/append_note/append-edit.ts`, well under the 150-LOC in-tree bias).
**Storage**: Vault filesystem — direct `fs` read-modify-write through `src/path-safety/`, `src/vault-registry/`, and the cohort's `node:fs/promises` substrate. The native upstream `obsidian append` subcommand IS NOT WRAPPED — this is an fs-direct tool in the `write_note` / `patch_heading` / `patch_block` lineage (ADR-009 substrate), cohort-aligned with the project's "user content NEVER crosses the CLI argv pipe at any size" discipline (write_note handler header comment, FR-005, SC-007 cohort lineage). The deliberate divergence from a CLI-wrap approach is documented in `research.md` R1 with the BI-0038 Windows argv-defect rationale.
**Testing**: `vitest` with `@vitest/coverage-v8`; tests co-located as `*.test.ts` alongside source modules per Principle II.
**Target Platform**: Node.js >= 22.11 LTS; MCP server transport over stdio; cross-platform (Linux / macOS / Windows). The `EXTERNAL_EDITOR_CONFLICT` detection-capability caveat (FR-022) is inherited from `write_note` / `patch_heading` / `patch_block` byte-stably: Windows surfaces sharing-violation errors on `fs.rename` / `fs.open` for files held by external editors; Linux / macOS surface the condition only when the editor takes an exclusive flock.
**Project Type**: MCP server typed-tool wrapper using the project's direct-fs substrate (ADR-009 lineage; cohort precedents: `write_note`, `patch_heading`, `patch_block`).
**Performance Goals**: Read + separator-decide + write within the cohort's per-call budget (`write_note`'s empirical p95 is ~10 ms for typical notes, dominated by the `metadataCache`-invalidation eval at ~150 ms). No streaming or chunked rewrites — whole-file read into memory, modified in memory, written back atomically. Append's per-call dominant cost is the substrate's atomic-rename overhead, identical to `write_note`'s rewrite cost.
**Constraints**: 1000 UTF-16 code-unit input cap on `file` / `path` locators (cohort parity with BI-033 / BI-038 / BI-039 / BI-040 / BI-043); no input cap on `content` itself beyond what the substrate accepts (the entire point of going fs-direct is to escape the argv-pipe ceiling that would otherwise apply); zero new top-level error codes (Constitution Principle IV); no modifications to YAML frontmatter (FR-011); preserve note's existing line-ending and trailing-newline conventions (FR-008, FR-009, FR-010); no auto-create of missing target (FR-012, FR-025); `content` payload preserved byte-for-byte verbatim (FR-010a — no trim, no normalise); wikilink-form locators MUST NOT contain `[[` or `]]` brackets (FR-001a — schema-layer rejection).
**Scale/Scope**: One new typed tool (`append_note`), one new source-tree module (`src/tools/append_note/`) with four production files (`index.ts` / `schema.ts` / `handler.ts` / `append-edit.ts`) and four co-located test files. One new single-state `details.code` value (`CONTENT_EMPTY` under `VALIDATION_ERROR` for FR-013). Two existing `details.code` / top-level-code reuses unchanged (`NOTE_NOT_FOUND` under `CLI_REPORTED_ERROR`; `EXTERNAL_EDITOR_CONFLICT` under `CLI_REPORTED_ERROR` with its `unsaved-changes` / `file-locked` `details.reason` enum inherited from BI-040; `ERR_NO_ACTIVE_FILE` for active-mode no-focused-file). No new top-level codes. The `file`-field bracket-rejection (FR-001a) extends the cohort's existing `STRUCTURALLY_UNSAFE_PATH_MESSAGE` channel — one shared `safeWikilinkNameField` Zod refinement may be lifted to a small shared helper, deferred to T0 as the only decision narrowed from "lift now" to "lift later when a second consumer appears" if a second cohort tool needs the same constraint within the next two BIs.

**Structure Decision**: Single project, per-surface module at `src/tools/append_note/` per Principle I. The module follows the four-file shape (`{index, schema, handler}.ts` + one helper) — simpler than `patch_heading/`'s and `patch_block/`'s five-file shape because append has only ONE pure helper (the separator-decide-and-concatenate function is small and self-contained; there is no block-scan / heading-walk equivalent because end-of-file append needs no in-document scan). Precedent: `delete/`, `rename/`, `move/` ship a three-file shape (`{index, schema, handler}.ts` with no helper); `find_and_replace/` ships a six-file shape with three helpers. The new tool wires into the boot spine at `src/server.ts` via the standard one-line import + one-line factory call (cohort precedent — 28 prior tools follow this pattern). The `_register-baseline.json` fixture gets a one-entry append. All existing infrastructure (`invokeCli` for the small active-mode pre-write eval and the metadataCache-invalidation post-write eval, `vault-registry/`, `path-safety/`, `target-mode/`, `UpstreamError`) is consumed unchanged.
