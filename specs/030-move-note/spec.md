# Feature Specification: Move Note Typed MCP Tool

**Feature Branch**: `030-move-note`
**Created**: 2026-05-15
**Status**: Draft
**Input**: User description: "Add Move Note — A typed MCP tool that moves (or moves-and-renames) a vault note to a new location. Honours the vault's 'Automatically update internal links' setting; the auto-update behaviour is vault-config-dependent, not wrapper-controlled."

## Background *(non-mandatory context)*

This feature adds the **sixteenth** typed-tool wrap on top of the foundation completed by features 003–010 and extended by 011 / 012 / 013 / 014 / 015 / 016 / 017 / 018 / 019 / 020 / 021 / 022 / 023 / 024 / 025 / 026 / 027 / 028 / 029. Where the prior fifteen typed surfaces (`read`, `write_note`, `delete`, `read_property`, `find_by_property`, `read_heading`, `set_property`, `files`, `rename`, `outline`, `properties`, `links`, `smart_connections_similar`, `smart_connections_query`, `tag`, `tree`) wrap CLI subcommands for reading, creating/overwriting, deleting, frontmatter access, heading-body reads, folder listing, in-place rename, outline / properties / link enumeration, plugin similarity / query, tag-index retrieval, and recursive subtree enumeration, `move` wraps the **move (or move-and-rename)** subcommand of the Obsidian CLI. It composes on the same load-bearing primitives:

- **[004-target-mode-schema](../004-target-mode-schema/spec.md) / [010-flatten-target-mode](../010-flatten-target-mode/spec.md)** — the shared `target_mode` schema primitive at [src/target-mode/target-mode.ts](../../src/target-mode/target-mode.ts). Post-010 the canonical reuse idiom is `applyTargetModeRefinement(targetModeBaseSchema.extend({ <tool-specific fields> }))` (the Pattern (a) flat-extension form). Per Constitution Principle III, this primitive owns the `target_mode` discriminator, the `vault`/`file`/`path` keys, the specific-branch XOR locator rule, the active-branch forbidden-key rule, and the strict-mode posture (`additionalProperties: false`); `move` MUST compose on top of it rather than re-state any of those rules inline.
- **[003-cli-adapter](../003-cli-adapter/spec.md)** — the centralised CLI adapter at [src/cli-adapter/cli-adapter.ts](../../src/cli-adapter/cli-adapter.ts). The handler routes every CLI invocation through `invokeCli(...)` (or its bounded counterpart through the shared queue). The adapter owns argv assembly, vault hoisting, active-mode locator stripping, and the four-priority error classification (`CLI_BINARY_NOT_FOUND`, `CLI_NON_ZERO_EXIT`, `ERR_NO_ACTIVE_FILE`, `CLI_REPORTED_ERROR`). The 011-write-note unknown-vault response-inspection clause and the 017-cross-platform-support binary-resolver are inherited unchanged.
- **[005-help-tool](../005-help-tool/spec.md)** — the `help` tool and the schema-stripping utility at [src/help/strip-schema.ts](../../src/help/strip-schema.ts). The registry-consistency test at [src/server.test.ts](../../src/server.test.ts) gates the existence of `docs/tools/move.md`; this BI authors that file as a non-stub Markdown body (the file does not currently exist).
- **[021-rename-note](../021-rename-note/spec.md) (registered as `rename` after the 022 rename sweep) / [012-delete-note](../012-delete-note/spec.md) (registered as `delete`)** — the closest siblings on the typed-tool track. `rename` handles in-place name changes; `move` handles path changes (with optional rename when `to` is a full path including a filename). The two were always intended to split cleanly along this axis: `rename` is filename-on-disk; `move` is path-on-disk. The folder-separator-rejection rule on `rename`'s `name` field (locked at 021's /speckit-clarify Q2) is the structural enforcement point that kept `rename` scoped to in-place rename and deferred path changes to `move`. This BI consumes that scope deferral. `move` follows the same module layout (`schema.ts` / `handler.ts` / `index.ts` + co-located tests under `src/tools/move/`), the same `RegisterDeps` shape (`logger`, `queue`, optional `deps` pass-through), and the same handler-thinness ceiling (a tight argv-shape + result-mapping body; no per-call logger events at the tool layer per the 011-R1 reconciliation). The factory name is `createMoveTool` per the 022-aligned single-word convention.
- **[022-rename-typed-tools](../022-rename-typed-tools/spec.md)** — the rename sweep that aligned typed-tool surface names to upstream Obsidian CLI subcommand names. `move` is a single-word verbatim-from-upstream name (matches the CLI's `move` subcommand referenced as a future-BI handoff point in 021's narrative). The post-022 registry-stability baseline at `src/tools/_register-baseline.json` (per 022 FR-018) MUST be rolled forward in the same commit that registers `move`, via `npm run baseline:write`.

The user-facing contract this feature delivers: an MCP client calls `move({ target_mode: "specific", vault: "MyVault", path: "Inbox/Old.md", to: "Archive/" })` and receives `{ moved: true, fromPath: "Inbox/Old.md", toPath: "Archive/Old.md" }` — the file at `Inbox/Old.md` is now at `Archive/Old.md` on disk, and any wikilinks pointing at the old path have been rewritten (subject to the vault's "Automatically update internal links" setting). For move-and-rename in one operation, the caller passes `to` as a full vault-relative path: `move({ target_mode: "specific", vault: "MyVault", path: "Inbox/Old.md", to: "Archive/Renamed.md" })` → `{ moved: true, fromPath: "Inbox/Old.md", toPath: "Archive/Renamed.md" }`. Locator XOR, mode-forbidden-key, empty-`to`, and strict-mode (`additionalProperties: false`) rejections all happen at the zod parse boundary; CLI failures (binary missing, non-zero exit, in-band `Error:` reply, no active file, source-not-found, destination-exists) flow through `UpstreamError`. No new error codes are introduced by this BI.

The feature scope is **single-file move (optionally with rename) only**. Bulk moves are a future concern (the caller iterates today). Forcing link-update regardless of the vault's auto-update-links setting is out of scope — that would require config mutation, which the typed surface does not perform. Pre-move backup is the caller's responsibility. Conflict-resolution policy when the destination exists is surfaced as-is from the CLI (no tool-layer `overwrite` / `auto_suffix` / `on_conflict` knob).

Link-rewriting is **vault-config-dependent**: when the vault's "Automatically update internal links" setting is enabled (Settings → Files & Links → "Automatically update internal links"), the underlying Obsidian move surface rewrites every wikilink pointing at the old path to the new path; when the setting is disabled, the move happens but stale links remain. The typed surface does NOT enforce the setting and does NOT implement a wrapper-side link-rewriting fallback (which would duplicate the link-graph machinery Obsidian already owns). The surface documents the dependency prominently so callers do not expect unconditional link-rewriting.

**Active-mode no-focused-note classifier behaviour** is a load-bearing inherited quirk this BI must surface explicitly: per the user input (and empirically confirmed at TC-049 for `delete` and TC-171 for `rename`), the native CLI's `move` subcommand emits `Error: No active file.` (capital-N) on no-focused-note, while the bridge's dispatch-layer classifier (per [003-cli-adapter](../003-cli-adapter/spec.md)) targets lowercase `Error: no active file`. The capital-N reply therefore does NOT classify as `ERR_NO_ACTIVE_FILE`; it falls through to `CLI_REPORTED_ERROR` with `details.message` carrying the verbatim CLI line. This is consistent across all native-subcommand typed tools (`delete`, `rename`, and now `move`) and is tracked under [[BI-0027 - Audit Tool Descriptions]] dimension C.2. The spec's error roster, the published `docs/tools/move.md` body, and the User Story 5 acceptance scenarios all reflect this — `move`'s active-mode no-focused-note surfaces as `CLI_REPORTED_ERROR`, not `ERR_NO_ACTIVE_FILE`. No new error code is introduced; the spec records the actual observable behaviour of the bridge.

## Clarifications

### Session 2026-05-15

- Q: `.md` append rule on full-path-target `to`'s filename portion — apply unconditionally (parity with 021-rename Q1), apply only when source is `.md`, or never apply? → A: **Apply only when source is `.md`**. The wrapper appends `.md` to `to`'s filename portion IFF (a) the source `path` ends in `.md` (literal, case-sensitive byte equality) AND (b) `to`'s filename portion does not end in `.md`. Non-`.md` source files (`.canvas`, `.pdf`, attachments) bypass the append rule entirely; the wrapper forwards `to` verbatim to the CLI. This preserves the 021-rename Q1 ergonomic default for the common `.md → .md` move case while preventing silent cross-type conversion (e.g., `Board.canvas + to: "Archive/Renamed"` would otherwise have become `Archive/Renamed.md`, an unintended `.canvas → .md` rename). Rationale: the user-input framing of `to`'s full-path shape as "a full vault-relative path with filename" already presumes the caller provides the complete filename; the source-`.md` guard catches the common-case ergonomic footgun (forgetting `.md` on a `.md`-source move-and-rename) without inventing a wrapper-owned type-coercion rule for non-`.md` sources. The check is literal byte equality on both predicates, mirroring the 020-fix-write-gaps R2 lock.
- Q: Trailing-`/` discriminator strictness for ambiguous `to: "Archive"` (no trailing `/`, no extension) — keep strict trailing-`/`, switch to a heuristic, validation-reject ambiguous shapes, or probe source location and disambiguate? → A: **Keep strict trailing-`/` discriminator**. `to` ending in `/` → folder-target (preserve source basename); otherwise → full-path-target (apply source-`.md`-guarded append rule per Q1). `to: "Archive"` is full-path-target by the strict rule: against `.md` source produces `Archive.md` at vault root; against non-`.md` source produces `Archive` extensionless at vault root. Callers who want folder-target shape MUST include the trailing `/`. Rationale: predictable, deterministic, no heuristics, no extra CLI roundtrips, no schema-layer validation overhead. Mirrors the project's anti-magic posture (020-fix-write-gaps R2 byte-equality lock; 021-rename Q2 schema-layer folder-separator reject). The one-byte ergonomic burden on the caller buys zero ambiguity. `docs/tools/move.md` prominently documents the rule with the "ALWAYS include trailing `/` for folder-target" guidance and explicit worked examples of the surprise case (`to: "Archive"` and what it produces in both source-extension cases).

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Specific-mode move with `to` as a folder path (Priority: P1)

An MCP client (typically an LLM agent) needs to relocate a note from one folder to another while preserving the filename — the archival / regrouping flow. The agent calls `move({ target_mode: "specific", vault: "MyVault", path: "Inbox/Tax-2026.md", to: "Archive/2026/" })` and receives `{ moved: true, fromPath: "Inbox/Tax-2026.md", toPath: "Archive/2026/Tax-2026.md" }`. The file at `Inbox/Tax-2026.md` is no longer there; `Archive/2026/Tax-2026.md` exists with the same body content as before. With the vault's auto-update-links setting enabled, every wikilink that previously read `[[Inbox/Tax-2026]]` or `[[Tax-2026]]` now reads `[[Archive/2026/Tax-2026]]` or `[[Tax-2026]]` (the bare-form resolves to the new location via Obsidian's link-resolution machinery).

**Why this priority**: This is the headline value of the feature. The archive / regroup flow is the most common move driver, and the trailing-`/` folder-target shape is the most ergonomic surface for callers who don't want to think about the filename — they just want the file relocated to a different folder and they're done. Without this story, the typed-tool track has no move primitive, and every move either goes through the freeform escape hatch (which inherits none of the per-mode validation, structured errors, or link-rewriting guarantees) or — worse — through a raw filesystem move which breaks every wikilink across the vault.

**Independent Test**: With a stub `spawnFn` injected via `deps`, calling `move({ target_mode: "specific", vault: "MyVault", path: "Inbox/Tax-2026.md", to: "Archive/2026/" })` against a stub child that exits `0` and emits a CLI-shaped success response (canonical from/to paths on stdout, or substantively equivalent — exact wording captured during plan-stage CLI characterisation per FR-019) results in (a) the spawn being invoked with binary `"obsidian"` and an argv whose first non-binary token is the `move` subcommand and which contains `vault=MyVault`, `path=Inbox/Tax-2026.md`, and a destination token reflecting the resolved `to` value (exact argv-key names captured during plan-stage research per FR-019), and (b) the handler returning `{ moved: true, fromPath: "Inbox/Tax-2026.md", toPath: "Archive/2026/Tax-2026.md" }`.

**Acceptance Scenarios**:

1. **Given** the `move` tool is registered and a stub adapter that exits `0` reporting a successful folder-target move, **When** an MCP client calls `move({ target_mode: "specific", vault: "MyVault", path: "Inbox/Tax-2026.md", to: "Archive/2026/" })`, **Then** the tool returns a successful MCP response whose JSON-serialised payload is `{ moved: true, fromPath: "Inbox/Tax-2026.md", toPath: "Archive/2026/Tax-2026.md" }` AND the adapter was invoked with argv that includes `vault=MyVault`, the source locator (`path=Inbox/Tax-2026.md` or the CLI argv-key rename per FR-019), and a destination token whose value preserves the source's filename (`Tax-2026.md`) at the new folder (`Archive/2026/`) AND the argv does NOT contain a `file=` token (locator XOR enforced at the schema layer means only `path` was provided).
2. **Given** the same tool, **When** an MCP client calls `move` with `to: "Archive/"` (single-segment folder, trailing slash), **Then** the resulting `toPath` is `Archive/<source-filename>` and the call succeeds — the trailing-`/` discriminator (per FR-003) reliably distinguishes folder-target from full-path-target.
3. **Given** the same tool, **When** an MCP client calls `move` against a source whose source filename contains internal periods or unusual extensions (`source path = "Drafts/Doc.v1.draft.md"`, `to = "Archive/"`), **Then** the resulting `toPath` preserves the entire source filename verbatim (`Archive/Doc.v1.draft.md`) — the folder-target branch does NOT apply the `.md` append rule and does NOT re-derive the filename from any other input.

### User Story 2 — Specific-mode move-and-rename via full-path `to` (Priority: P1)

When the agent wants to relocate a note AND rename it in one call, it passes `to` as a full vault-relative path including the destination filename. The agent calls `move({ target_mode: "specific", vault: "MyVault", path: "Inbox/Tax-2026.md", to: "Archive/2026-Tax-Return.md" })` and the CLI moves the file from `Inbox/Tax-2026.md` to `Archive/2026-Tax-Return.md` in one operation. The handler reports `{ moved: true, fromPath: "Inbox/Tax-2026.md", toPath: "Archive/2026-Tax-Return.md" }`. The combined operation is a single CLI call; the typed surface does NOT compose `rename` + `move` client-side (which would require two CLI calls, double the link-rewriting roundtrips, and risk a half-applied state).

**Why this priority**: P1 alongside Story 1. The two `to` shapes — folder-target (Story 1) and full-path-target (Story 2) — together cover the entire move surface. Downgrading either would leave callers without a clean path for one of the two natural use cases. Story 2 is the load-bearing case for renaming-as-part-of-archival (e.g., "move this to the archive AND give it a normalised name in one step").

**Independent Test**: With a stub adapter that exits `0` and reports `Inbox/Tax-2026.md` as the canonical source and `Archive/2026-Tax-Return.md` as the canonical destination, calling `move({ target_mode: "specific", vault: "MyVault", path: "Inbox/Tax-2026.md", to: "Archive/2026-Tax-Return.md" })` results in the spawn being invoked with argv that includes `vault=MyVault`, the source locator, and the destination token whose value is `Archive/2026-Tax-Return.md`, and the handler returning `{ moved: true, fromPath: "Inbox/Tax-2026.md", toPath: "Archive/2026-Tax-Return.md" }`.

**Acceptance Scenarios**:

1. **Given** the `move` tool and a stub adapter reporting `Inbox/Tax-2026.md` and `Archive/2026-Tax-Return.md` as the canonical from/to paths, **When** an MCP client calls `move({ target_mode: "specific", vault: "MyVault", path: "Inbox/Tax-2026.md", to: "Archive/2026-Tax-Return.md" })`, **Then** the tool returns `{ moved: true, fromPath: "Inbox/Tax-2026.md", toPath: "Archive/2026-Tax-Return.md" }` AND the adapter was invoked with argv that includes `vault=MyVault`, the resolved source locator, and the destination token whose value is `Archive/2026-Tax-Return.md` (exact argv key captured during plan-stage research per FR-019).
2. **Given** the same tool with a `.md` source, **When** an MCP client calls `move` with a full-path `to` whose filename portion does NOT end in `.md` (e.g., source `path: "Inbox/Tax-2026.md"`, `to: "Archive/2026-Tax-Return"`), **Then** the wrapper's source-`.md`-guarded `.md` append rule (per FR-003, locked at /speckit-clarify Q1 session 2026-05-15) applies: the effective forwarded destination is `Archive/2026-Tax-Return.md` and the resulting `toPath` is `Archive/2026-Tax-Return.md`. The check is literal byte equality (case-sensitive on both predicates: source-`.md` and `to`-filename-`.md`), mirroring the 020-fix-write-gaps R2 lock and the 021-rename Q1 lock for `name`.
3. **Given** the same tool with a non-`.md` source (e.g., `path: "Boards/Plan.canvas"`), **When** an MCP client calls `move` with a full-path `to` whose filename portion does NOT end in `.md` (e.g., `to: "Archive/Renamed"`), **Then** the source-`.md` guard suppresses the append rule and the effective forwarded destination is `Archive/Renamed` verbatim (no `.md` appended; the CLI handles the destination as-given). No silent `.canvas → .md` cross-type conversion. Per the /speckit-clarify Q1 resolution.
4. **Given** the same tool, **When** an MCP client calls `move` via the wikilink-form source locator (`file: "Tax-2026"` instead of `path: "Inbox/Tax-2026.md"`), **Then** the CLI resolves the wikilink to a concrete on-disk source and the handler reports `{ moved: true, fromPath: "<CLI-resolved canonical source path>", toPath: "<resolved destination path>" }`. The `file=` vs `path=` choice is at the caller's discretion; both reach the same CLI subcommand via the same handler path.

### User Story 3 — Source-not-found and destination-already-exists fail with structured errors (Priority: P1)

When the source path does not exist on disk, or when a file already exists at the requested destination, the CLI surfaces a structured failure that the handler propagates as an `UpstreamError`. The agent calls `move({ target_mode: "specific", vault: "MyVault", path: "Inbox/Missing.md", to: "Archive/" })` against an empty vault and receives an MCP error response with `code: "CLI_REPORTED_ERROR"` (or `CLI_NON_ZERO_EXIT` depending on the CLI's exit-code convention — captured during plan-stage research per FR-019) whose `details.message` carries the verbatim CLI wording. Similarly, calling `move({ target_mode: "specific", vault: "MyVault", path: "Inbox/A.md", to: "Archive/A.md" })` when `Archive/A.md` already exists fails with a structured error whose message identifies the destination collision.

**Why this priority**: P1. These are the two most common move failure modes and they MUST surface as actionable structured errors — not as silent no-ops, not as opaque exit codes, and definitely not as silent overwrites (which would destroy caller data without recovery). The agent's recovery path depends on distinguishing "the source is wrong, retry with the right locator" from "the destination is taken, pick a different name or auto-suffix" from "something else went wrong, surface the error to the user." Per the user input's [P1] AC #3 and AC #4.

**Independent Test**: Two handler tests, each driving the adapter through an injected stub. (a) stub child exits non-zero with stderr `"Error: file not found at Inbox/Missing.md\n"` OR exits `0` with stdout `"Error: file not found at Inbox/Missing.md\n"` (whichever shape the live CLI produces per FR-019) → handler returns the appropriate `CLI_NON_ZERO_EXIT` or `CLI_REPORTED_ERROR` code AND `details.message` carries the verbatim CLI line. (b) stub child surfaces the destination-exists error → handler propagates with the verbatim message and the typed surface does NOT silently overwrite, does NOT auto-suffix, does NOT swallow the error.

**Acceptance Scenarios**:

1. **Given** the `move` tool and a stub adapter reporting a source-not-found failure for `Inbox/Missing.md`, **When** an MCP client calls `move({ target_mode: "specific", vault: "MyVault", path: "Inbox/Missing.md", to: "Archive/" })`, **Then** the tool returns an MCP error response whose `code` is the one the adapter classified (either `CLI_REPORTED_ERROR` or `CLI_NON_ZERO_EXIT` per the CLI's actual signal) AND `details.message` carries the verbatim wording from the CLI.
2. **Given** the `move` tool and a stub adapter reporting a destination-exists failure (e.g., the file `Archive/A.md` already exists and the CLI refuses to overwrite), **When** an MCP client calls `move({ target_mode: "specific", vault: "MyVault", path: "Inbox/A.md", to: "Archive/A.md" })` OR `move({ target_mode: "specific", vault: "MyVault", path: "Inbox/A.md", to: "Archive/" })` (folder-target whose preserved filename collides), **Then** the tool returns an MCP error response whose `details.message` identifies the destination collision — the typed surface does NOT silently overwrite, does NOT auto-suffix the destination, and does NOT swallow the error.

### User Story 4 — Schema layer rejects malformed inputs before any CLI call (Priority: P1)

Per Constitution Principle III, the zod schema is the gatekeeper. `move`'s schema composes the post-010 flat target-mode primitive (which enforces `vault` required in specific mode, exactly-one-of `file`/`path` in specific mode, `vault`/`file`/`path` forbidden in active mode, `additionalProperties: false` strict-mode against unknown top-level keys) and adds `to: z.string().min(1)` as the single `move`-specific required field. Seven classes of malformed input MUST fail at the parse step: (a) `target_mode === "specific"` with neither `file` nor `path`; (b) `target_mode === "specific"` with both `file` AND `path`; (c) `target_mode === "specific"` without `vault`; (d) `target_mode === "active"` with any of `vault`/`file`/`path`; (e) any input with an unknown top-level key (e.g., `pancakes: "yes"`); (f) `target_mode` absent or with a value other than `"specific"` / `"active"`; (g) `to` absent OR `to === ""` OR `to` present but a non-string (e.g., `to: 42`). None of these reach the adapter; the failure is a `VALIDATION_ERROR` with the offending field path. **No file on disk MUST be touched by any input that fails validation.** The CLI is never invoked.

**Why this priority**: P1. Boundary validation is the constitutional baseline — ADR-003's promise that the schema enforces the intent contract is empty if `move` doesn't actually parse against the primitive. For a move operation, skipping this story is doubly bad: every malformed input either becomes an opaque CLI failure (uninformative to the agent) or — worse — undefined behaviour where the CLI accepts both `file` AND `path` and silently moves the wrong source, or accepts an empty `to` and silently produces a no-op or a vault-root collision. The empty-`to` carve-out is the additional safety net — `to: ""` MUST fail at the schema layer, not bubble up as an opaque CLI failure or — worst case — silently produce a move to the vault root or to an empty-string path that the OS may or may not refuse depending on platform. Per the user input's [P1] ACs #5, #6, #7, #8, #9.

**Independent Test**: Seven parse-only tests, each driving the handler with a stub adapter that MUST NOT be called. Each asserts the resulting error is `VALIDATION_ERROR` with the expected `details.issues[].path` and message keywords; the stub spawn function is asserted never to have been called.

**Acceptance Scenarios**:

1. **Given** the `move` tool and a stub adapter that records every invocation, **When** an MCP client calls `move({ target_mode: "specific", vault: "MyVault", to: "Archive/" })` (neither `file` nor `path`), **Then** the tool returns `code: "VALIDATION_ERROR"` whose `details.issues` includes an entry whose `message` mentions `"exactly one of"` AND the stub adapter was never invoked.
2. **Given** the `move` tool, **When** an MCP client calls `move({ target_mode: "specific", vault: "MyVault", file: "F", path: "F.md", to: "Archive/" })` (both locators), **Then** the tool returns `code: "VALIDATION_ERROR"` whose issues mention `"exactly one of"` AND the stub adapter was never invoked.
3. **Given** the `move` tool, **When** an MCP client calls `move({ target_mode: "specific", path: "P.md", to: "Archive/" })` (vault missing), **Then** the tool returns `code: "VALIDATION_ERROR"` whose `details.issues[].path` includes `"vault"` AND the stub adapter was never invoked.
4. **Given** the `move` tool, **When** an MCP client calls `move({ target_mode: "active", vault: "V", to: "Archive/" })` (forbidden key in active mode), **Then** the tool returns `code: "VALIDATION_ERROR"` whose issues mention `"vault"` AND `"active mode"` AND the stub adapter was never invoked. The same shape applies for forbidden `file` and forbidden `path`.
5. **Given** the `move` tool, **When** an MCP client calls `move({ target_mode: "specific", vault: "V", path: "P.md", to: "Archive/", pancakes: "yes" })` (unknown top-level key), **Then** the tool returns `code: "VALIDATION_ERROR"` whose `details.issues[]` includes a `code: "unrecognized_keys"` entry naming `pancakes` AND the stub adapter was never invoked. (Per [010-flatten-target-mode](../010-flatten-target-mode/spec.md) FR-002, this is the post-010 strict-mode behaviour; this BI inherits it.)
6. **Given** the `move` tool, **When** an MCP client calls `move({ target_mode: "specific", vault: "V", path: "P.md", to: "" })` (empty `to`), **Then** the tool returns `code: "VALIDATION_ERROR"` whose `details.issues[].path` includes `"to"` AND whose error code is the standard zod `too_small` / `min(1)` shape AND the stub adapter was never invoked.
7. **Given** the `move` tool, **When** an MCP client calls `move({ target_mode: "specific", vault: "V", path: "P.md" })` (`to` absent) OR `move({ target_mode: "specific", vault: "V", path: "P.md", to: 42 })` (`to` non-string), **Then** the tool returns `code: "VALIDATION_ERROR"` whose `details.issues[].path` includes `"to"` AND the stub adapter was never invoked.

### User Story 5 — Active-mode moves the focused note; no-focused-note surfaces as `CLI_REPORTED_ERROR` (Priority: P1)

When the agent declares `target_mode: "active"`, the call's intent is "move whatever note is currently open in Obsidian." The schema layer (per the target-mode primitive) forbids `vault`, `file`, and `path` in this mode. `to` remains required. The handler routes `{ target_mode: "active", to }` through the adapter, which emits argv beginning with the `move` subcommand and the `to`-derived destination token but no source-locator key=value tokens. The CLI moves the focused note's file to the requested destination; the handler returns `{ moved: true, fromPath: "<focused note's vault-relative path>", toPath: "<resolved destination>" }`.

**Active-mode no-focused-note classifier behaviour** is the load-bearing inherited quirk this story locks: the native CLI's `move` subcommand emits `Error: No active file.` (capital-N) on no-focused-note, while the bridge's dispatch-layer classifier targets lowercase `Error: no active file`. The capital-N reply does NOT classify as `ERR_NO_ACTIVE_FILE`; it falls through to `CLI_REPORTED_ERROR` with `details.message` carrying the verbatim CLI line. This is consistent across `delete` (empirically confirmed at TC-049), `rename` (TC-171), and `move` (this BI). The spec's error roster, the published `docs/tools/move.md` body, and this story's AC #3 reflect the actual `CLI_REPORTED_ERROR` code — not the hypothetical `ERR_NO_ACTIVE_FILE` that the rename-spec template (021) would have written for a tool wrapping an eval-driven subcommand. Per the user input's [P1] AC #10 and AC #11.

**Independent Test**: With a stub adapter that exits `0` and reports `Daily/2026-05-15.md` as the source and `Archive/2026-05-15.md` as the destination, calling `move({ target_mode: "active", to: "Archive/" })` results in the adapter being invoked with argv that contains no `vault=…`, `file=…`, or `path=…` tokens, includes the destination token derived from `to`, and the handler returns `{ moved: true, fromPath: "Daily/2026-05-15.md", toPath: "Archive/2026-05-15.md" }`. ### User Story 6 — CLI failures (binary missing, non-zero exit, in-band Error) flow through `UpstreamError` (Priority: P1)

When the underlying CLI fails — the binary is missing, the process exits non-zero, the CLI replies with `Error: …` on stdout, the focused note is missing in active mode — the failure surfaces to the MCP client as an `UpstreamError` carrying the adapter's classified `code`. Per the cli-adapter's four-priority classification, `move` does not classify; it propagates whatever code the adapter raised. This is the same propagation guarantee as every other typed surface.

**Why this priority**: P1. Constitution Principle IV requires every boundary failure to surface as a structured `UpstreamError` with a stable code. The user input names source-not-found, destination-collision, external-editor-locks-the-file, permission-denied, and unknown-vault-display-name as adversarial cases that MUST be handled rather than silently succeed; all flow through this propagation path (per FR-019).

**Acceptance Scenarios**:

### User Story 8 — Same-folder move = rename equivalence (Priority: P3)

When the agent calls `move` with a `to` whose folder portion resolves to the source's folder (`dirname(toPath) === dirname(fromPath)`), the operation is effectively a rename — same folder, different filename. The wrapper does NOT special-case this; it forwards to the CLI's `move` subcommand uniformly. The CLI handles the same-folder move identically to a cross-folder move (the underlying surface treats relocation and rename as the same operation), and the typed surface returns `{ moved: true, fromPath: <X>, toPath: <Y> }` with `dirname(fromPath) === dirname(toPath)`. Callers can detect the rename-equivalence case structurally via the path comparison. `docs/tools/move.md` documents the equivalence so callers who want a pure rename can choose between `rename` (filename-only field, dedicated tool) and `move` (path-shaped `to` with same parent) based on ergonomic preference.

**Why this priority**: P3. The equivalence is a documented observation, not a load-bearing contract. Most callers wanting a rename use `rename` (the dedicated tool with a `name` field); most callers wanting a move use `move` with a different folder. The same-folder-move case exists at the boundary between the two and the story exists primarily to (a) record that the typed surface does not branch on it and (b) document the equivalence for callers who care. Per the user input's [P3] AC #15.

**Independent Test**: With a stub adapter that exits `0` and reports `Inbox/Old.md` as source and `Inbox/New.md` as destination, calling `move({ target_mode: "specific", vault: "V", path: "Inbox/Old.md", to: "Inbox/New.md" })` results in the handler returning `{ moved: true, fromPath: "Inbox/Old.md", toPath: "Inbox/New.md" }` AND the structural assertion `dirname(fromPath) === dirname(toPath)` holds. A second test asserts `move` with `to: "Inbox/"` (folder-target whose folder matches the source) returns `{ moved: true, fromPath: "Inbox/Old.md", toPath: "Inbox/Old.md" }` — the same-name same-folder case (effectively a no-op from the caller's perspective).

**Acceptance Scenarios**:

1. **Given** the `move` tool and a stub adapter reporting a same-folder move (`Inbox/Old.md` → `Inbox/New.md`), **When** an MCP client calls `move({ target_mode: "specific", vault: "V", path: "Inbox/Old.md", to: "Inbox/New.md" })`, **Then** the tool returns `{ moved: true, fromPath: "Inbox/Old.md", toPath: "Inbox/New.md" }` AND `dirname(fromPath) === dirname(toPath)` AND the documentation at `docs/tools/move.md` notes the equivalence to `rename`.

### Edge Cases

- **`move({ target_mode: "specific", vault: "", path: "P.md", to: "Archive/" })`** (empty-string vault): MUST fail at the zod parse boundary as `VALIDATION_ERROR` whose `details.issues[].path` includes `"vault"`. The target-mode primitive enforces `vault: z.string().min(1)`. Reasonable default — error message mirrors zod's standard `min(1)` message.
- **`move({ target_mode: "specific", vault: "V", path: "P.md", to: "../../etc/notes/" })`** (path-traversal-shaped destination): MUST be either rejected at the validation boundary or verified to be rejected by the underlying CLI. Per the user input's FILESYSTEM adversarial bullet, **silent vault-escape is a critical security defect** — moving files outside the vault root is unacceptable. Plan-stage characterisation per FR-019 verifies the CLI's behaviour against `../`-shaped `to` values (and against `../`-shaped `path` values for the source); if the CLI does NOT reject such inputs, this BI is amended pre-ship to add a tool-layer reject (a precondition, not an open question — gated by SC-012). The dominant security concern for `move` is path-traversal on `to` (a new attack surface vs `rename`'s name-only field) because shell-injection is structurally prevented by the cli-adapter's argv-array passing.
- **`to` targets a folder that does not exist** (e.g., `to: "NonExistentFolder/"`): observable behaviour captured during plan-stage characterisation per FR-019 — the CLI may auto-create the folder, may fail with a structured error, or may produce undefined behaviour. The typed surface propagates whatever the CLI does. `docs/tools/move.md` documents the observed behaviour per FR-014. Per the user input's FILESYSTEM adversarial bullet.
- **`move({ target_mode: "specific", vault: "DoesNotExist", path: "P.md", to: "Archive/" })`** (unknown vault display name): inherits the response-inspection clause that 011-write-note's R5 added to the cli-adapter. The unknown-vault response is re-classified by the adapter to `CLI_REPORTED_ERROR` with the verbatim CLI message preserved in `details.message`. `move` propagates this without modification. Plan-stage characterisation per FR-019 confirms the CLI's response signature for the `move` subcommand matches the create / delete / rename subcommand signatures; if it differs, the adapter's response-inspection logic is extended (additive change to the adapter, not to `move`). Per the user input's UNDERLYING adversarial bullet ("An unknown vault display name may produce a response that the bridge's classifier does not treat as an error" — capture and verify).
- **Source deleted out-of-band before the call** (e.g., another agent or the user removed it via Obsidian's UI): surfaces as a structured `CLI_REPORTED_ERROR` (or `CLI_NON_ZERO_EXIT` depending on the CLI's exit code). Plan-stage research per FR-019 captures the verbatim wording and the handler propagates it. No silent no-op — the second caller MUST see a structured error per the user input's CONCURRENCY adversarial bullet.
- **Two concurrent `move` calls against the same source**: the shared in-flight CLI queue (per FR-008, inherited from [006-read-note](../006-read-note/spec.md) FR-016 and [011-write-note](../011-write-note/spec.md) FR-008) serializes both calls behind one channel. The first call moves the file; the second call sees the on-disk state after the first call completed and surfaces a `CLI_REPORTED_ERROR` (source-not-found, because the source has been moved away). Deterministic ordering is guaranteed by the queue; silent no-ops are precluded by the source-not-found path. Per the user input's CONCURRENCY adversarial bullet.
- **Concurrent move + rename + delete against the same source**: same queue serialization. The order of operations is the order of acquisition of the queue; whichever call lands first wins; subsequent calls operate on the post-first-call state. Per the user input's CONCURRENCY adversarial bullet.
- **External editor has the file open** (Obsidian, VS Code, anything else) when `move` is invoked: observable behaviour is captured during plan-stage characterisation per FR-019 and documented in `docs/tools/move.md`. `move` does NOT lock the file or coordinate with external editors. If the OS rejects the move with EBUSY (Windows), the CLI surfaces non-zero exit; the handler propagates as `CLI_NON_ZERO_EXIT`. If the OS allows the move and Obsidian closes/reopens the buffer (or shows a stale view), that is the editor's behaviour, not `move`'s. Per the user input's CONCURRENCY adversarial bullet.
- **Active-mode TOCTOU**: the focused note may change between the schema's parse step and the CLI's actual invocation. The operation runs against whichever note is focused at execution time. Documented as a known limitation; `move` does NOT attempt to lock or pin the focused note. Agents that need certainty about which file is being moved MUST use specific mode with an explicit locator. The project-wide active-mode TOCTOU concern is governed at the target-mode primitive layer, not retrofitted per-tool (per `feedback_no_locator_echo_in_read_responses.md`). Per the user input's CONCURRENCY adversarial bullet.
- **Case-only moves on Windows / macOS-default filesystems** (`Inbox/Note.md` → `inbox/note.md`): platform-dependent. On case-insensitive filesystems (Windows NTFS-default, macOS APFS-default), the OS may treat the operation as a no-op (the underlying inode is the same), or as a rename (the on-disk filename's case changes but the inode stays). On case-sensitive filesystems (Linux ext4-default, macOS APFS case-sensitive option, Windows WSL ext4), the operation is a regular move. Plan-stage characterisation per FR-019 captures the observed CLI behaviour on each platform; the typed surface propagates the CLI's response verbatim. Per the user input's FILESYSTEM adversarial bullet.
- **Reserved Windows characters in `to`** (`:?*<>|"`): platform-dependent. The CLI is the trust boundary; `move` forwards verbatim (except where `/` and `\` are folder-separator-treated per FR-003). Plan-stage CLI characterisation per FR-019 records the observed behaviour for the remaining reserved characters (likely platform-specific — Windows kernel rejects with `EINVAL`, POSIX OSes accept everything except `/` and `\0`); the documented behaviour is captured in `docs/tools/move.md` per FR-014. Per the user input's FILESYSTEM adversarial bullet.
- **Reserved Windows device names in `to`** (`CON`, `PRN`, `AUX`, `NUL`, `COM1`–`COM9`, `LPT1`–`LPT9`): platform-dependent. The CLI is the trust boundary; same posture as reserved-characters above. Documented in `docs/tools/move.md`.
- **Unicode normalisation of `to`** (e.g., NFC vs NFD on macOS HFS+ vs APFS): the input `to` may not round-trip exactly through the on-disk path. macOS HFS+ stores filenames as NFD-normalised; APFS is case-insensitive-but-normalisation-preserving. Windows preserves the input bytes. Linux ext4 preserves the input bytes. `move` does NOT normalise the `to` input; it forwards verbatim to the CLI. The CLI / OS may store a normalised form. The handler's response `toPath` reflects whatever the CLI reports — which on macOS HFS+ may differ byte-for-byte from the input `to`. Documented as a platform limitation. Per the user input's FILESYSTEM adversarial bullet.
- **Aliased wikilinks** (`[[Real Path|Display Text]]`): when the vault's auto-update-links setting is enabled, the auto-update rewrites the path side (`Real Path`) leaving the display text unchanged. `move` does NOT control this — Obsidian does. Documented in `docs/tools/move.md` so callers understand display text persists. Per the user input's LINK-UPDATE BEHAVIOUR adversarial bullet.
- **Outgoing-link references inside the moved file**: relative paths inside the moved file's body (e.g., embedded images via `![[path/to/image.png]]` or relative URL references) may now be broken if the destination is at a different folder depth. Whether the auto-update setting rewrites internal references is captured during plan-stage characterisation per FR-019 and documented in `docs/tools/move.md`. Plausible Obsidian behaviour: outgoing links to vault-resolved targets are rewritten when the setting is enabled (Obsidian re-resolves and emits a new link), but raw relative URL references (rare in markdown notes) are NOT. The typed surface does NOT mediate this. Per the user input's LINK-UPDATE BEHAVIOUR adversarial bullet.
- **Auto-update-links setting disabled in the vault**: links pointing at the old path remain stale after the move. The file move succeeds; the link-rewriting does not happen. `move` does NOT detect or warn about this — the typed surface has no observability into the vault's settings. Documented prominently in `docs/tools/move.md` per FR-014 and per the user input's [P1] AC #12.
- **Caller-supplied `path` / `file` / `to` containing argv-shell metacharacters** (e.g., `to: "Archive/; rm -rf /; #"`): SAFE. The cli-adapter assembles argv as a child_process.spawn array (no shell), so no character in `path` / `file` / `to` can break out into a shell. This is a structural property of the adapter, not a runtime check. Path-traversal (the dominant remaining security concern for `to`) is covered by the bullet above. Per the user input's SECURITY bullet.
- **Strict-rich vs strict-naive client-class observability of unknown-key rejection** (Story 4 AC #5): strict-rich clients (Claude Desktop, MCP Inspector) read the `additionalProperties: false` from the published `inputSchema` and either forward unknown keys (in which case the bridge-side `unrecognized_keys` rejection is observable to the test) or strip them client-side (in which case the bridge-side rejection is non-observable but the schema-side invariant — no untyped passthrough at runtime — still holds). Strict-naive clients (Cowork) do not consult `additionalProperties`; they forward unknown keys to the bridge, which always observes the rejection. Tests assume the strict-rich-forwarding path (the worst case) and assert the bridge-side rejection occurs. Documented per the user input's CLIENT-CLASS adversarial bullet.

## Requirements *(mandatory)*

### Functional Requirements

#### Module layout

- **FR-001**: A new typed tool MUST be created at `src/tools/move/` following the existing per-surface module convention (per Constitution Principle I and the `src/tools/{delete, files, find_by_property, help, links, obsidian_exec, outline, properties, read, read_heading, read_property, rename, set_property, smart_connections_query, smart_connections_similar, tag, tree, write_note}/` precedents). The module MUST contain six files at minimum (three source + three co-located tests): `schema.ts` (the zod input schema, the inferred TypeScript types, and the JSON Schema export via `zod-to-json-schema`), `handler.ts` (the function that calls the adapter and maps the result to the output shape), `index.ts` (the `createMoveTool` factory returning a `RegisteredTool` per the aggregator pattern — `index.ts` per the post-011 file-name convention, NOT `tool.ts`), and one co-located test file per source file (`schema.test.ts`, `handler.test.ts`, `index.test.ts`). Per Constitution Principle V, every new source file MUST carry an `// Original — no upstream. <one-line description>.` header.

#### Schema

- **FR-002**: The input schema MUST compose the post-010 flat `target_mode` primitive (per [010-flatten-target-mode](../010-flatten-target-mode/spec.md)) using the canonical Pattern (a) flat-extension idiom: `applyTargetModeRefinement(targetModeBaseSchema.extend({ to }))`. Per the contract at [specs/010-flatten-target-mode/contracts/flat-target-mode.contract.md](../010-flatten-target-mode/contracts/flat-target-mode.contract.md), `.extend()` (NOT `.merge()`) is the binding API call — `.merge()` resets `unknownKeys` to `"strip"` and silently drops the strict-mode safety net for unknown top-level keys. The composed schema inherits the primitive's full per-mode contract end-to-end: `vault` required in specific mode, exactly-one-of `file`/`path` in specific mode, `vault`/`file`/`path` forbidden in active mode, top-level `additionalProperties: false` against unknown keys.

  **Active-mode `superRefine` clauses for `move`**: NONE. The target-mode primitive's existing rules govern locator XOR and forbidden-key checks unchanged.

- **FR-003**: The schema MUST add one `move`-specific required field:
    - `to: z.string().min(1)` — required in BOTH modes. The `.min(1)` rejects empty-string `to` at the parse step per Story 4 AC#6 / the user input's [P1] AC #7. The field is NOT optional — every move must supply a destination regardless of mode.

  **`to` shape discriminator and `.md` append rule** (wrapper-owned, file-local helper):
    - **Folder-target shape**: `to.endsWith("/")` (literal byte equality, case-sensitive). The forwarded destination preserves the source's basename: `effectiveTo = to + basename(fromPath)`. No `.md` append rule applied (the source's existing extension carries over verbatim). Worked examples: `path="Inbox/Tax.md", to="Archive/"` → forwarded `Archive/Tax.md`; `path="Drafts/Doc.v1.draft.md", to="Archive/"` → forwarded `Archive/Doc.v1.draft.md`.
    - **Full-path-target shape**: `to` does NOT end in `/`. The forwarded destination applies the **source-`.md`-guarded** `.md` append rule on the filename portion (the segment after the last `/`, or the entire string if no `/`), per /speckit-clarify Q1 (session 2026-05-15): `if (sourcePath.endsWith(".md") && !filenamePortion.endsWith(".md")) forward (to + ".md"); else forward verbatim`. Both predicates are literal byte equality (case-sensitive), mirroring the 020-fix-write-gaps R2 lock. The source-`.md` guard prevents silent cross-type conversion on non-`.md` sources (`Board.canvas + to: "Archive/Renamed"` forwards `Archive/Renamed` verbatim, NOT `Archive/Renamed.md`) while preserving the 021-rename Q1 ergonomic default for `.md → .md` moves. Worked examples (source = `Inbox/Note.md`): `to="Archive/Renamed.md"` → forwarded `Archive/Renamed.md` (verbatim; both predicates fail the `&&`); `to="Archive/Renamed"` → forwarded `Archive/Renamed.md` (append; source-`.md` AND filename-non-`.md`); `to="Archive/Doc.v1.draft"` → forwarded `Archive/Doc.v1.draft.md` (append; internal periods preserved, `.draft` not in allowlist); `to="Archive/Renamed.MD"` → forwarded `Archive/Renamed.MD.md` (case-sensitive non-match on filename; uppercase `.MD` does not satisfy the rule); `to="Archive/Plan.canvas"` (cross-type rename intent) → forwarded `Archive/Plan.canvas.md` (append; `.canvas` not in allowlist; caller wanting a literal `.canvas` destination routes through `obsidian_exec` per the 021-rename Q1 fallback). Worked examples (source = `Boards/Plan.canvas` — non-`.md`): `to="Archive/Renamed"` → forwarded `Archive/Renamed` (verbatim; source-`.md` guard fires, append rule suppressed); `to="Archive/Renamed.md"` → forwarded `Archive/Renamed.md` (verbatim; the `.md` is the caller's explicit choice); `to="Archive/Plan"` → forwarded `Archive/Plan` (verbatim; cross-type conversion to `.md` is NOT default behaviour on non-`.md` sources).
    - **The discriminator is the trailing `/`** — not a heuristic on whether the trailing segment looks like a filename, not a check on extension presence, not a comparison against the source's basename. Trailing-`/` ⇒ folder; absent trailing-`/` ⇒ full-path. Callers who want unambiguous folder-target shape MUST include the trailing `/`. This is documented prominently in `docs/tools/move.md`.
    - **Backslash (`\`) in `to`**: forwarded verbatim to the CLI. The typed surface does NOT normalise `\` to `/` and does NOT reject `\` at the validation boundary. Plan-stage characterisation per FR-019 case (xi) records the CLI's behaviour against backslash-shaped inputs (likely platform-dependent: Windows-native CLI may accept; POSIX CLI may reject or treat as literal character). Documented in `docs/tools/move.md`. If plan-stage research surfaces that `\` causes silent vault-escape or silent platform-dependent corruption, this BI is amended pre-ship to add a validation-boundary reject (the same SC-012 precondition pattern as path-traversal).

- **FR-004**: The schema MUST NOT add `.describe()` annotations to any field. Parameter documentation lives in `docs/tools/move.md` (per ADR-005's progressive-disclosure design).

- **FR-005**: The schema MUST export the inferred TypeScript type via `z.infer<typeof moveInputSchema>` (or equivalent name). No hand-written `interface MoveInput` or `type MoveInput = { … }` is permitted (Constitution Principle III). The output type MUST also be a zod schema (or its `z.infer<…>`): `moveOutputSchema = z.object({ moved: z.literal(true), fromPath: z.string(), toPath: z.string() }).strict()`. The `moved` field is a `z.literal(true)` because every successful return path of `move` produces `moved: true` — the failure path goes through `UpstreamError`, not through a `moved: false` shape (parity with `delete`'s `deleted: true` literal and `rename`'s `renamed: true` literal). Hand-rolled output types are forbidden by the same principle.

- **FR-006**: The schema MUST export a JSON Schema document via `zod-to-json-schema(...)` (mirroring `obsidian_exec`, `read`, `write_note`, `delete`, `rename`). The exported JSON Schema MUST round-trip through the schema-stripping utility at registration time per FR-011. The emitted JSON Schema MUST be the post-010 flat shape (one top-level `type: "object"`, `properties` with all five top-level fields typed inline, `additionalProperties: false`, no `oneOf` envelope) — verifiable via the consolidated drift detector at [src/tools/_register.test.ts](../../src/tools/_register.test.ts) which automatically covers every registered tool.

#### Handler

- **FR-007**: The handler MUST route the validated input through the cli-adapter's `invokeCli` entry point (or its bounded counterpart through the shared queue per FR-008). The adapter call shape is `invokeCli({ command: "move", parameters: <derived>, flags: <derived>, target_mode: <validated>, vault: <hoisted-or-undefined> }, deps?)`. The subcommand name `move` is the user-input-supplied name and parity with the 022 rename sweep convention; plan-stage research per FR-019 confirms the live CLI exposes the subcommand under this name (or captures the divergence and the handler retargets). The `parameters` and `flags` records are derived from the validated input as follows:
    - **Specific mode**: `vault: parsed.vault` is passed as a TOP-LEVEL field (per [011-write-note](../011-write-note/spec.md) PSR-3) NOT inside `parameters`. `parameters` includes exactly one of `file: parsed.file` or `path: parsed.path` (with the user-facing field name potentially renamed to a CLI argv key per the 011-write-note PSR-5 precedent — confirmed during plan-stage research per FR-019), plus the destination token derived from `parsed.to` after the FR-003 transform (or its CLI argv-key rename — `to=` / `name=` / `dest=` per the 011-write-note PSR-5 precedent).
    - **Active mode**: `parameters` contains the destination token only (no `vault`, no `file`, no `path` — the schema layer has already rejected those and the adapter's defensive locator-stripping is a redundant safety net). The top-level `vault` field passed to `invokeCli` is `undefined` in active mode.
    - The choice between key=value form (`to=Archive/Renamed.md`) and flag form for the destination token is a plan-stage research decision per FR-019, parity with the 011-write-note PSR-4 resolution for `overwrite`/`open`. The structural contract (the destination token is emitted for every call, never omitted) binds; the placement (parameters vs flags) does not.

  **`to`-transform rule** (per FR-003, locked 2026-05-15 /speckit-clarify Q1): the handler computes the forwarded destination via a file-local helper `resolveTo(to: string, fromPath: string): string` of ~12 LOC implementing the two-branch logic: trailing-`/` → append source basename; otherwise → apply the source-`.md`-guarded `.md` append rule on filename portion (`if (fromPath.endsWith(".md") && !filenamePortion.endsWith(".md")) append`; both predicates literal byte equality, case-sensitive, mirroring the 020-fix-write-gaps R2 lock). The handler's input to `resolveTo` is the validated `to` and the source's resolved `fromPath`; for active mode where `fromPath` is not pre-known at handler entry, the handler may either (a) forward `to` verbatim to the CLI and let upstream resolve (accepting that the source-`.md` guard cannot be applied wrapper-side without a pre-resolve roundtrip and the CLI's response `toPath` is canonical), or (b) defer the active-mode `to`-transform to the CLI entirely and accept that the source-`.md` guard's protection only fully binds in specific mode. Plan-stage research per FR-019 case (viii) characterises the CLI's native handling of the active-mode `to` extension contract; if active-mode and specific-mode would diverge observably, the spec is amended pre-ship. The structural contract (the source-`.md`-guarded `.md` append rule applies to full-path-target `to` in specific mode; active-mode behaviour matches the CLI's native handling per FR-019) binds; the implementation timing (wrapper-side pre-compute vs CLI-side compute) does not.

  Plan-stage research per FR-019 confirms the live CLI's actual move-subcommand name, argv shape, and response wording for both successful moves and failure modes (source-not-found, destination-exists, no-active-file, unknown-vault, path-traversal, missing-destination-folder). The handler's argv-mapping and response-parsing logic lock against the live CLI's behaviour.

- **FR-008**: The handler MUST execute through the same single-in-flight CLI queue that gates `obsidian_exec`, `read`, `write_note`, `delete`, `read_property`, `find_by_property`, `read_heading`, `set_property`, `files`, `rename`, `outline`, `properties`, `links`, `smart_connections_similar`, `smart_connections_query`, `tag`, `tree` (per [006-read-note](../006-read-note/spec.md) FR-016 and successor specs). `createMoveTool`'s `RegisterDeps` MUST accept `queue: Queue`, and `src/server.ts` MUST pass the *same* `queue` instance to `createMoveTool` and to every other CLI-invoking tool so concurrent CLI invocations across all surfaces serialize through one channel. This forecloses the "two concurrent moves against the same source" race in Edge Cases.

- **FR-009**: The handler MUST be a thin `invokeCli` wrapper with no per-call logger events at the tool layer, parity with the actual `read` / `write_note` / `delete` / `rename` handler implementations (per [011-write-note](../011-write-note/spec.md) PSR-1 / R1 — the spec-level requirement that earlier specs phrased as "logger.callStart / callEndSuccess / callEndFailure events" superseded by the actual sibling implementations). Observability flows through the cli-adapter's existing `dispatchTimeout` / `dispatchCap` / `dispatchKill` events end-to-end. `RegisterDeps` MUST still accept `logger: Logger` for forwarding to the adapter / queue layer; `src/server.ts` MUST pass the same logger instance to all tool registrations.

- **FR-010**: The handler MUST return a successful tool-call result whose JSON-serialised payload conforms to the output schema: `{ moved: true, fromPath: "<canonical source path>", toPath: "<canonical destination path>" }`. Both `fromPath` and `toPath` are derived from the CLI's response (or, if the CLI does not echo canonical paths, from the input locator and the post-FR-003 forwarded destination — captured during plan-stage research per FR-019). The handler MUST NOT trim, normalise, or post-process the path values; it propagates the CLI's reported values verbatim. Per the user input's response shape definition. The audit-trail invariant for same-folder moves (Story 8 AC#1) is `dirname(fromPath) === dirname(toPath)` — verifiable by the caller via path comparison.

### Out of Scope

The following are explicitly excluded from this feature and remain the responsibility of separate BIs or future work (mirroring the user input's "Out of scope" enumeration plus consequential exclusions surfaced during spec drafting):

- **Bulk multi-file moves**: out of scope per the user input. Callers iterate today, or a future batch-move BI ships if usage justifies. The single-file typed surface does not aggregate calls.
- **Forcing link-update regardless of vault setting**: out of scope per the user input. The vault's "Automatically update internal links" setting governs link-rewriting; the typed surface does NOT mutate that setting (which would require config writes the CLI does not necessarily expose) and does NOT force link-rewriting independently (which would require a parallel link-rewriting implementation, defeating the point of wrapping the CLI's native surface). If empirical use surfaces a strong need for forced link-rewriting, a future BI ships either (a) a `force_update_links: true` flag with explicit caveats, or (b) a separate `rewrite_links` primitive that operates orthogonally.
- **Conflict-resolution policy when the destination exists**: out of scope per the user input. The typed surface surfaces the underlying CLI's behaviour as-is — if the CLI refuses to overwrite, the surface returns `CLI_REPORTED_ERROR` with the verbatim message; if the CLI auto-suffixes, the surface returns success with the suffixed `toPath`; if the CLI overwrites silently (unlikely for Obsidian's move surface but verified during plan-stage characterisation per FR-019 case (vi)), the documented behaviour is captured in `docs/tools/move.md`. The typed surface does NOT add a tool-layer `overwrite` flag, `auto_suffix` flag, or `on_conflict` policy enum.
- **Pre-move backup**: out of scope per the user input. Callers that need rollback on move failure are responsible for capturing the source state before invoking `move`.
- **Path-traversal validation at the tool layer**: out of scope per Edge Cases — UNLESS plan-stage characterisation per FR-019 case (x) surfaces that the CLI does not reject `../`-shaped paths in `to` or `path`, in which case the spec is amended pre-ship to add a tool-layer reject. This is a precondition, not an open question (gated by SC-012). Silent vault-escape on a move is a critical security defect.
- **Backslash-in-`to` validation at the tool layer**: out of scope unless plan-stage characterisation per FR-019 case (xii) surfaces that the CLI silently produces vault-escape or platform-dependent corruption from `\`-containing `to` values, in which case the spec is amended pre-ship (same SC-012 pattern).
- **Case-only-move normalisation across platforms**: out of scope. The typed surface propagates the CLI's behaviour verbatim. Plan-stage characterisation per FR-019 captures the observed behaviour on each platform; `docs/tools/move.md` documents it.
- **Unicode normalisation of `to`**: out of scope. The typed surface forwards verbatim to the CLI; the OS / filesystem normalises (or doesn't) per platform. macOS HFS+ NFD-normalisation is documented as a platform limitation, not mitigated.
- **Automatic destination-folder creation at the tool layer**: out of scope. Plan-stage characterisation per FR-019 case (xi) records whether the CLI auto-creates missing destination folders; the typed surface propagates the CLI's behaviour. If the CLI does NOT auto-create and callers find this disruptive, a future BI may ship a `create_destination: true` flag (analogous to `mkdir -p`); current scope is "surface the CLI's behaviour as-is."
- **A `list_vaults` primitive for vault-display-name pre-validation**: out of scope. The unknown-vault edge case is handled by the cli-adapter's response-inspection clause (per [011-write-note](../011-write-note/spec.md) R5). If response-inspection proves brittle, `list_vaults` ships as a follow-up BI and pre-validation moves there.
- **New error codes**: out of scope per the user input. `move`'s entire failure surface is covered by `VALIDATION_ERROR` + the adapter's four codes (`CLI_BINARY_NOT_FOUND`, `CLI_NON_ZERO_EXIT`, `ERR_NO_ACTIVE_FILE`, `CLI_REPORTED_ERROR`). If a `move`-specific failure mode emerges (e.g., "destination exists and the CLI's response wording doesn't classify cleanly"), a future amendment introduces a code with the same rigour as [005-help-tool](../005-help-tool/spec.md) introduced `HELP_TOOL_NOT_FOUND` / `HELP_DOCS_MISSING`.
- **Bridge-classifier change to match capital-N `Error: No active file.`**: out of scope per the user input's UNDERLYING bullet (and per `feedback_no_locator_echo_in_read_responses.md`'s "project-wide concerns get ADRs applied uniformly, not per-tool retrofits"). The classifier mismatch affects every native-subcommand typed tool (`delete`, `rename`, `move`, and any future native-subcommand additions) and belongs in a dedicated cross-cutting BI tracked under [[BI-0027 - Audit Tool Descriptions]] dimension C.2. This BI's job is to record the actual observable behaviour and ensure the help-doc / error-roster claims match — not to fix the classifier.
- **Refactoring the existing typed tools**: out of scope. The fifteen existing typed tools co-exist with the new tool unchanged. `move` adds rather than perturbs (per SC-009).
- **Modifying ADR-003, ADR-005, ADR-006, ADR-010, ADR-013, ADR-014, or ADR-015**: this BI implements all seven; it does not change them. Conflicts surface as new ADRs per the project's amendment procedure.
- **Active-mode TOCTOU mitigations specific to `move`**: out of scope. The project-wide TOCTOU concern is governed at the target-mode primitive layer. `move` inherits whatever mitigation that layer offers (currently none beyond documentation); future TOCTOU work lands as an ADR amendment, not a `move` patch.
- **Cross-extension moves with type conversion**: in scope (per /speckit-clarify Q1, session 2026-05-15: extension-agnostic with source-`.md`-guarded `.md` append). Full-path-target `to` may end in any extension (`.md`, `.canvas`, `.pdf`, no extension at all). The source-`.md`-guarded `.md` append rule applies ONLY when (a) source ends in `.md` AND (b) `to`'s filename portion does not end in `.md` — meaning `.md → .canvas` requires a caller-supplied explicit `.canvas` extension (e.g., `to: "Archive/Doc.canvas"` forwards verbatim, because `.canvas` is not `.md` and the case-sensitive byte equality check on filename-`.md` fails, so the append rule short-circuits before the source-`.md` check matters — same wire-level result as without the guard, but the rationale is cleaner). For `.canvas → .md`, the caller MUST supply an explicit `.md` extension; the wrapper does NOT append `.md` to non-`.md` sources. The CLI's behaviour on cross-extension moves (does it permit `.md → .canvas` at all? Does it permit `.canvas → .md`?) is captured during plan-stage research per FR-019. Callers should be aware that `move` does NOT validate the destination extension matches the source — the typed surface trusts the caller's explicit intent on cross-extension destinations but refuses to silently CREATE that intent via wrapper-side append on non-`.md` sources.
