# Feature Specification: Find Replace

**Feature Branch**: `038-find-replace`
**Created**: 2026-05-18
**Status**: Draft
**Input**: User description: "Add Find Replace — across a chosen scope of vault notes, the user can replace every occurrence of a chosen text pattern (a literal string or a regular expression) with a chosen replacement. Preview by default, commit only on explicit opt-in. Code blocks and HTML comments skipped by default, opt-in to include. Optional subfolder scope. Refuses to run when matches exceed a configured safe upper bound."

## Clarifications

### Session 2026-05-18

- Q: Response envelope shape — is the preview response and the commit response a single discriminated-union schema or two distinct top-level schemas? → A: **Single Zod `z.discriminatedUnion("mode", […])` keyed on a top-level `mode: "preview" | "commit"` literal.** The preview branch carries `{ mode: "preview", affected_notes, total_occurrences }`. The commit branch carries `{ mode: "commit", changed_notes, total_occurrences_replaced, partial }`. The `mode` literal lets agents type-narrow via Zod's discriminated-union semantics; agents pattern-match on the literal rather than checking field presence. The `partial: boolean` flag in the commit branch signals whether the batch completed in full (`partial: false`) or was halted by an `FS_WRITE_FAILED` (FR-021) mid-batch (`partial: true`, with the response carrying counts up to the failing note plus the failing-note locator). Constitution Principle III favours single-schema discrimination over fragmented top-level types — both branches share a single `*EvalResponseSchema` (or equivalent) rather than splitting into `PreviewResult` / `CommitResult`. Find_and_replace is the project's first preview-then-commit tool — no cohort parity to inherit; the discriminated-union pattern is established here for any future tool with the same two-mode shape.
- Q: Per-note atomic write strategy — how is per-note write atomicity (FR-015) implemented? → A: **Inherit ADR-009 §3 write-temp-then-rename pattern verbatim, with a UUID-suffixed tmp path for concurrent-commit safety.** Each per-note commit write MUST: (1) `fs.writeFile(<target>.<uuid>.tmp, newContent)` writing the rewritten note bytes to a tmp path adjacent to the target with a UUID suffix (parity with write_note's `${absPath}.${randomUUID()}.tmp` pattern at handler.ts:203); (2) `fs.rename(<target>.<uuid>.tmp, <target>)` atomically swapping the tmp file over the target. Same-volume rename atomicity (POSIX inode swap; Windows `MoveFileEx`). Mid-process-crash MUST leave either the old content or the new content intact at the target path — never a torn write. The UUID-suffixed tmp path prevents collision between two concurrent commits both writing temps for the same target (Q2's per-note Queue acquisition guarantees the rename serialization, but the tmp paths must not collide). Parent directories already exist at commit time (commit only writes notes that existed at preview time), so no `mkdir({recursive: true})` is needed. The implementation MAY reuse an extracted utility OR inline the pattern in the handler — planning's call; the spec contract is the semantic guarantee, not the code-organisation.
- Q: Pattern minimum length — does the operation accept or reject an empty `pattern: ""`? → A: **Reject at the input-validation boundary.** Empty pattern surfaces `code: "VALIDATION_ERROR"`, `details.code: "INVALID_PATTERN"`, `details.reason: "empty"` (third sub-state under the existing `INVALID_PATTERN` `details.code`, alongside `"too-long"` and `"regex-syntax"`). Rejection fires BEFORE any regex compilation or vault traversal; no note is modified. The "I forgot to set the pattern field" caller bug surfaces as a hard error rather than a silent zero-match success (which would otherwise look identical to a no-occurrence success per FR-017's zero-width-skip rule). Parity with BI-033 query-non-empty validation. ADR-015 sub-discriminator pattern — same `details.code` namespace as the other pattern-rejection sub-states, distinguished by `details.reason`.
- Q: Concurrent-commit serialization — how does the handler coordinate writes when two `find_and_replace` commits fire simultaneously against the same vault with overlapping note sets? → A: **Per-note Queue acquisition via the project's injected `Queue` dep (from the `createQueue()` god-node), parity with write_note / rename_note / move_note.** Commit-time writes for a single invocation MUST be serialized through the injected `Queue` per-note — each note write acquires the queue independently, mirroring the per-note atomicity of FR-015. Two concurrent commits against the same vault interleave at note-granularity but MUST NOT produce torn writes on any single note. Preview operations (read-only) MUST NOT acquire the queue. The handler MUST declare `Queue` as an injected dep in its `ExecuteDeps` shape per the project's DI discipline (Constitution Principle I — no handler reaches back into the composition root at runtime). Source reference: write_note's existing pattern at `src/tools/write_note/handler.ts:12`.
- Q: Canonical tool name — what string does the operation register under for MCP discovery and source-tree layout? → A: **`find_and_replace`.** Three-word snake_case with explicit conjunction. The two operations (find + replace) are surfaced explicitly in the name rather than collapsed; agents reading the tool list see "find AND replace" as one atomic transaction rather than guessing at the verb_verb form. Diverges from the project's two-word default (`write_note`, `read_note`, `pattern_search`, `move_note`) but the conjunction is semantically load-bearing — it signals the find-then-replace coupling that distinguishes this tool from the read-only `pattern_search` sibling. Source layout: `src/tools/find_and_replace/{index,schema,tool,handler,*.test}.ts`. The feature directory (`specs/038-find-replace/`) and git branch (`038-find-replace`) retain their `find-replace` short-name from spec scaffolding — directory naming and tool naming are independent per the speckit convention. ADR-010 (typed tool names mirror upstream CLI subcommand) does NOT bind here — find_and_replace is a direct-FS tool per ADR-009, not a CLI-bridge tool.
- Q: Discriminator alignment for `INVALID_PATTERN` — should invalid-regex-syntax (FR-010) and over-cap-pattern (FR-022) share a `details.code` and split via `details.reason`, or use separate `details.code` values? → A: **Single `details.code: "INVALID_PATTERN"` with `details.reason` sub-discriminator.** Over-cap pattern surfaces `code: "VALIDATION_ERROR"` + `details.code: "INVALID_PATTERN"` + `details.reason: "too-long"`. Syntactically invalid regex (regex mode only) surfaces `code: "VALIDATION_ERROR"` + `details.code: "INVALID_PATTERN"` + `details.reason: "regex-syntax"`. ADR-015 sub-discriminator pattern, parity with the existing `VAULT_NOT_FOUND` + `unknown` / `not-open` pair already used in the spec. Keeps the discriminator namespace tight — agents pattern-match on `details.code` for the category ("the pattern is bad") and on `details.reason` for the remediation path. SC-006 enumerates eleven distinct `(top-level code, details.code, details.reason)` triples; the `details.code` namespace itself stays compact. The same alignment applies to the over-cap-replacement case as a single-state `details.code: "INVALID_REPLACEMENT"` (no sub-state needed today; `details.reason` MAY be absent or set to `"unknown"` per ADR-015's default-sub-state allowance).
- Q: Input length caps on `pattern` and `replacement_substring` — are the find pattern and the replacement text bounded at the input-validation boundary, and at what length? → A: **1000-character cap on `pattern` and 1000-character cap on `replacement`, both enforced at the input-validation boundary BEFORE any vault scan or regex compilation, parity with BI-033 FR-004 / clarifications-2026-05-16 (1000-char `query` cap).** Over-cap `pattern` surfaces `code: "VALIDATION_ERROR"` with `details.code: "INVALID_PATTERN"`. Over-cap `replacement` surfaces `code: "VALIDATION_ERROR"` with `details.code: "INVALID_REPLACEMENT"`. Both reject before any vault traversal begins; no note is modified. Typical caller usage is far below 1000 chars (most patterns/replacements fit in 50 chars); over-cap input is virtually always a paste-a-whole-file accident the caller wants surfaced quickly rather than executed. No new top-level codes — REUSE of `VALIDATION_ERROR` (streak preserved per Constitution Principle IV).
- Q: Indented code blocks — does the skip-by-default for "fenced code blocks" extend to CommonMark indented code blocks (lines indented by 4+ spaces or a tab at line start with no surrounding fence)? → A: **Indented code blocks are treated as prose — only paired fences (` ``` ` / `~~~`) are skip-regions.** Detecting CommonMark indented code blocks requires distinguishing them from list-item continuation lines (a `1.` list whose body is indented 4 spaces is not a code block), which forces a real CommonMark parser. Parity with the current "text-level, not full-CommonMark-parser-level" framing in the spec's fence-detection Assumption. Indented code in modern Obsidian notes is rare enough that the safe-default benefit is small while the implementation cost (full CommonMark parsing for list disambiguation) is high. Callers wanting to protect indented code regions wrap them in fences (the conventional Obsidian usage anyway).
- Q: Generic FS write-error envelope on commit — how does the operation surface non-traversal, non-bound write failures (ENOSPC / EACCES / EROFS / EIO etc.) during commit? → A: **Inherit ADR-009 §7 verbatim. Top-level `code: "FS_WRITE_FAILED"` (existing ADR-009 code — REUSE, no new top-level code) with `details.errno` carrying the underlying Node errno string (e.g., `"ENOSPC"`, `"EACCES"`, `"EROFS"`).** Commit halts at the failing note; per FR-015 there is no cross-note rollback — notes written successfully before the failure stay written on disk. The response surfaces the partial counts (notes-changed-so-far, occurrences-replaced-so-far, locator of the failing note) ALONGSIDE the `FS_WRITE_FAILED` envelope, so the caller knows where to retry from. The wrapper does NOT auto-retry the failing note — caller decides whether to re-issue (parity with the "No retry semantics" Assumption).
- Q: Eligible-file filter and hidden-folder skip — which files in the vault are scanned and rewritten, and are Obsidian's hidden / system folders excluded? → A: **`.md` extension only (case-insensitive on the extension) AND skip any directory whose name begins with `.` at every traversal level.** Files whose path includes a `.`-prefixed segment anywhere — `.obsidian/`, `.trash/`, `.git/`, `.obsidian/plugins/foo/data.md` — MUST NOT be scanned or rewritten, even when the extension matches. Non-`.md` files (`.canvas`, `.base`, `.txt`, attachments, plugin data, JSON configs, images) MUST NOT be scanned or rewritten. The filter MUST be applied defensively as a post-filter at the wrapper, even though the upstream subcommand natively restricts to `.md` (parity with BI-033 FR-021's defensive future-proofing against an upstream version that broadens indexing). Hidden-folder skip and extension filter both apply uniformly under whole-vault scope AND under the subfolder scope from FR-008.
- Q: Vault target mode — does the operation expose `target_mode` / `@active` sentinel for focused-vault selection, or follow some other vault-selection contract? → A: **Vault-wide cohort contract — single optional `vault?: string` field, focused-vault default, no `target_mode` axis, no `@active` sentinel.** Find-and-replace is inherently a vault-wide surface (operates across many notes per invocation), not file-targeted. Per the project architecture's verbatim statement, "ADR-003's scope explicitly does not reach inherently-vault-wide surfaces" — ADR-003's target-mode discipline applies to single-file-target tools (BI-011 write_note, BI-018 write_property, BI-021 rename_note, BI-030 move_note), NOT to vault-wide cohort tools. The correct cohort parity is `find_by_property`, `properties`, `search`, `context_search`, and `pattern_search` (BI-037) — all of which use an optional `vault?: string` field with NO `target_mode` axis. **Input contract**: a single optional `vault?: string`. When absent, the operation defaults to the focused vault (cohort default). When present, the operation resolves the supplied display name via the lazy vault registry established by ADR-009 — REUSE of the existing write-surface infrastructure, no new resolver. **Error contract**: an unknown vault display name surfaces `code: "CLI_REPORTED_ERROR"` with `details.code: "VAULT_NOT_FOUND"` and `details.reason: "unknown"`; a registered-but-closed vault surfaces the same `details.code` with `details.reason: "not-open"` (ADR-015 sub-discriminator pattern; closed-but-registered detection branch per BI-026 v0.5.4 — `pattern_search` handler.ts:74 and handler.test.ts:293 already implement this exact pair). **Path-safety** from Q4 applies POST-resolution: once the vault is resolved (focused or named), the resolved vault root is `fs.realpath`'d before the subfolder check and the per-note canonical-path check.
- Q: Subfolder argument — path-traversal handling — how does the operation reject subfolder arguments that attempt to escape the vault root (via `../`, leading `/` or `\`, drive-letter prefix, control characters, or in-vault symlinks pointing outside)? → A: **Two-layer path safety per ADR-009, inheriting the existing posture from the write surface.** Find-and-replace is a write-side tool and adopts the full write-side path-safety stance, not the lighter read-side check. **Layer 1 (schema-level pre-flight)**: the input schema rejects path-traversal-shaped inputs (`../`, leading `/` or `\`, drive-letter prefix, control characters) at validation boundary and surfaces top-level `code: "VALIDATION_ERROR"` + `details.code: "INVALID_SUBFOLDER"` + `details.reason: "path-traversal"` (ADR-015 sub-discriminator). **Layer 2 (runtime canonical-path check)**: after resolving the absolute filesystem path, the handler invokes `fs.realpath` on the resolved subfolder and verifies `startsWith(realVaultRoot + sep)` on the canonical result. On escape (e.g., an in-vault symlink that resolves outside the vault) the handler surfaces top-level `code: "PATH_ESCAPES_VAULT"` (REUSE of the existing ADR-009 code — no new top-level code) and emits a `pathEscapeAttempt` security event via the project logger for the operator audit trail. Adopting Layer 1 alone would leave symlink-escape attacks blind; reusing `PATH_ESCAPES_VAULT` preserves the eleven-tool zero-new-top-level-codes streak (Constitution Principle IV) and unifies the security posture across the write surface. The same posture is applied to the per-note write path (each affected note's resolved location must also lie under the vault root); no commit MUST proceed if any affected note's canonical path escapes the vault root.
- Q: Whole-word matching for literal mode — does the operation expose an opt-in whole-word flag so that `foo` in `literal` mode does not match inside `food` / `foobar`? → A: **No whole-word flag in this BI.** Callers that need word-boundary anchoring drop to `regex` mode and write `\bfoo\b` explicitly. Parity with BI-037 pattern_search, which similarly does not expose a whole-word flag. The input surface stays narrow (no third axis beyond pattern mode + case sensitivity); the ECMAScript `\b` escape hatch is one-line and already supported by the regex-mode path. A future feature can add whole-word as a literal-mode opt-in if real use cases force it.
- Q: Stable ordering of affected notes and per-occurrence entries — what is the deterministic order in which affected notes appear in the preview/commit response, and the order of occurrences within each note? → A: **Two-level deterministic order — path-ascending across notes, then line-number-ascending within each note.** Affected notes MUST appear in ascending lexicographic order of their vault-relative path (parity with BI-019 list_files, BI-029 list_files_recursive, BI-033 search_vault_content). Within each note's per-occurrence list, occurrences MUST appear in ascending line-number order, and within a single line in ascending byte-offset order. The ordering is stable across runs given the same vault content — deterministic for tests and chunkable for agents that iterate the response top-to-bottom. Commit responses MUST surface the locators of changed notes in the same path-ascending order.
- Q: Line-ending preservation on commit write — when a note is rewritten on disk, are the source's line endings (CRLF / LF / mixed) preserved? → A: **Byte-for-byte preservation of unmatched content; per-line endings untouched.** Commit-time writes MUST preserve every byte of the source note that was not part of a matched occurrence — including line endings (CRLF, LF, or any mix), trailing-newline-or-not at EOF, BOM, and other bytes outside the replaced spans. Parity with BI-016 FR-004 / SC-002 (write_note byte-for-byte persistence including mixed CRLF/LF). Only the matched_substring bytes are replaced with the replacement_substring bytes; the surrounding line-ending context is left untouched. This forbids the implementation from "normalising" line endings as a side effect of round-tripping the note through a string split/join pipeline. Cross-platform-support invariant (BI-017) intact — Windows-authored CRLF notes do not churn to LF on rewrite, and macOS-authored LF notes do not gain `\r`. If the caller's replacement_substring itself contains an embedded newline character, that character is written verbatim — the operation does not normalise replacement_substring to the per-note dominant style.
- Q: Case-sensitivity control — does the operation support case-insensitive matching, and what is the default? → A: **Parity with BI-037 FR-007.** Default is case-sensitive matching. An explicit `case_insensitive: true` opt-in flips matching to ignore letter case. The flag applies uniformly to both `literal` and `regex` modes — in `regex` mode it is equivalent to setting the `i` flag on the compiled `RegExp`. Zero divergence from the sibling read-side tool; agents that learned the surface from `pattern_search` carry the same mental model into `find_replace`.
- Q: Multi-line pattern scope — can the find-pattern span line boundaries within a note? → A: **Line-scoped only, parity with BI-037 FR-012.** The find-pattern MUST NOT span line boundaries; matching runs line-by-line per note. In `regex` mode, `\n` metacharacters and dot-matches-newline behaviour do NOT enable cross-line matching — a pattern that uses `\n` simply does not match (no line in the note contains a literal `\n` byte). In `literal` mode, a pattern string that contains an embedded `\n` does not match (no single line contains a literal newline). Multi-line refactors are expressed as N single-line replacements; a separate future tool would handle multi-line spans if real use cases force it. Keeps per-occurrence B-shape (FR-004) unchanged — no `end_line` field needed — and preserves zero divergence from the established read-side regex semantics.
- Q: Per-note proposed-change shape in preview — how is the proposed rewrite represented for each affected note? → A: **Per-occurrence list, B-shape verbatim.** Each occurrence carries `{ line_number, full_line, matched_substring, replacement_substring }`. The `full_line` field is clipped to 500 UTF-16 code units (with trailing `…` U+2026 when clipped) per BI-033 FR-024 / BI-037 FR-005; trailing `\r` stripping follows BI-035 context_search FR-012 precedent. `matched_substring` and `replacement_substring` are never clipped — the predicate's hit and its replacement are always returned in full, even when the surrounding `full_line` is clipped. Parity with BI-037 `pattern_search` per-match envelope plus the new `replacement_substring` field. Structured per-occurrence shape Zod-schematises cleanly per Constitution Principle III; unified-diff text (rejected option A) is opaque to Zod, and full-note before/after content (rejected options C and D) violates the project's payload-bound discipline (parity with BI-033 / BI-035's 1000-entry truncation cap and the in-flight BI-0085 bound). This shape holds for single-line patterns; multi-line patterns are a separate scope question deferred to a subsequent clarification (see Outstanding below).
- Q: Safe-upper-bound source and default value — where does the configured ceiling on total occurrences live, and what is the default? → A: **Operator-tunable environment variable `OBSIDIAN_FIND_REPLACE_MAX_OCCURRENCES`, default 500.** The bound is read from the process environment **lazily on the first handler invocation** and cached for the process lifetime (parity with `resolveVaultPath`'s lazy-cache pattern and the existing `OBSIDIAN_BIN` env-var convention); env-var changes after the first invocation do not propagate — operators restart the server to change the bound. Invalid values (unset, non-numeric, ≤ 0) fall back to default 500 with a WARN log naming the offending value. Default 500 is small enough that an agent issuing a typical ADR-rename refactor surfaces the bound as a guardrail in normal use (the refusal forces the agent to narrow the scope or raise the bound deliberately) rather than as a rarely-tripped backstop, while still permitting routine cross-folder refactors. Operators with larger vaults set the env var to a higher value on install; the bound is enforced uniformly across preview and commit per FR-011 + FR-012.
- Q: Vault drift between preview and commit — what happens if a note's content changes between a successful preview and the subsequent commit, so that committing would produce a different rewrite than the preview promised? → A: **Count-only drift detection under the existing `VALIDATION_ERROR` top-level code.** At commit time, the operation re-runs occurrence-counting across the chosen scope on the current vault content. When the commit-time total occurrence count differs from the preview-time total occurrence count, the commit refuses with a typed error: top-level `code: "VALIDATION_ERROR"`, `details.code: "OCCURRENCE_COUNT_DRIFT"`, and no note is modified. No new top-level error code — the eleven-tool zero-new-top-level-codes streak since BI-011 / ADR-009 / Constitution Principle IV is preserved by surfacing the state-precondition failure through `details.code` discrimination per the ADR-015 sub-discriminator pattern. Count-only drift detection accepts the known false-negative of an equal-count add+remove (e.g., one occurrence appears elsewhere while one is deleted, leaving the total unchanged); the project's BI-Safety-Net clean-git-tree precondition shoulders that residual TOCTOU window. Per-note content-hash fingerprinting was rejected as forcing the caller to round-trip per-file fingerprints for marginal coverage gain over the count-based check.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Preview the replacement before any note changes (Priority: P1)

An agent issues a find-and-replace request against a vault — supplying the pattern, the replacement, and (optionally) a subfolder scope — without opting in to commit. The response lists every affected note together with the proposed change for that note and the per-note occurrence count, while no note on disk is modified. The agent can read the preview, confirm the scope is what it expected, and then re-issue the same call with the explicit commit opt-in to apply the rewrite. A preview that affects nothing returns a successful empty result rather than an error.

**Why this priority**: Bulk refactors against a vault are irreversible once committed; without a preview the agent cannot confirm the blast radius before mutation. Preview-then-commit is the safety contract that makes the whole feature usable — every other behaviour (skip-defaults, scope, upper-bound guard) builds on top of it. Without P1 there is no MVP; with only P1 the agent already gets a vault-wide refactor capability that is strictly safer than the manual read-rewrite loop it replaces.

**Independent Test**: Stand up a vault containing notes with a known occurrence count of the pattern across multiple notes. Issue the call without the commit opt-in. Verify the response lists every affected note with its proposed change, verify no file on disk is modified (compare mtimes and contents before/after), then re-issue the same call with the commit opt-in and verify the same set of notes is updated on disk with the same change shown in the preview. Separately, issue a pattern that matches nothing and verify the response is a successful empty result, not an error.

**Acceptance Scenarios**:

1. **Given** a vault with notes containing the pattern in prose, **When** the agent issues the operation without the commit opt-in, **Then** the response lists every affected note with the proposed change and the per-note occurrence count, and no note on disk is modified.
2. **Given** the agent has just received a preview, **When** the agent re-issues the same request with the explicit commit opt-in, **Then** the same set of notes is updated on disk and the response reports how many notes changed and how many occurrences were replaced overall.
3. **Given** the pattern matches no occurrence anywhere in scope, **When** the agent issues either a preview or a commit, **Then** the response is a successful empty result with zero affected notes and zero replacements, not an error.
4. **Given** the agent issues a request without specifying the commit opt-in, **When** the operation runs, **Then** it behaves identically to an explicit preview — no note on disk is modified — so that an accidental call cannot mutate the vault.

---

### User Story 2 - Skip code blocks and HTML comments by default; opt back in when needed (Priority: P2)

By default, occurrences inside fenced code blocks and inside HTML comments are left alone — code samples embedded in notes are not corrupted, and audit-trail comments embedded in notes are not silently rewritten. When the refactor genuinely targets those regions (e.g., a deliberate rename of a symbol that appears inside code samples), the agent can opt back in to either region — independently — and have occurrences inside that region replaced alongside the prose occurrences.

**Why this priority**: Without skip-by-default, every find-and-replace risks corrupting embedded code, command lines, examples, and the comment-based audit trail. The default is the safety contract that lets agents reach for this tool without hand-auditing every match. The opt-back-in is the escape hatch that prevents the safe default from blocking the legitimate "rename this symbol everywhere, including in code samples" case. P2 because P1 alone can already operate against a no-code-blocks vault, but most real vaults mix prose with fenced code and HTML comments — without P2, the tool would be unsafe for the common case.

**Independent Test**: With P1 in place, prepare a note where the pattern appears once in prose, once inside a fenced code block, and once inside an HTML comment. Run a preview with the default skip behaviour and verify only the prose occurrence is in the proposed change. Run a preview with the code-block opt-in and verify the prose and code-block occurrences both appear. Run a preview with the HTML-comment opt-in and verify the prose and comment occurrences both appear. Run with both opt-ins and verify all three appear.

**Acceptance Scenarios**:

1. **Given** a note where the pattern appears in prose and again inside a fenced code block, **When** the operation runs with the default skip behaviour, **Then** the prose occurrence is replaced and the code-block occurrence is left unchanged.
2. **Given** a note where the pattern appears inside an HTML comment, **When** the operation runs with the default skip behaviour, **Then** the comment is left unchanged.
3. **Given** the agent opts to include fenced code blocks, **When** a matching occurrence appears inside a fenced code block, **Then** that occurrence is included in the replacement.
4. **Given** the agent opts to include HTML comments, **When** a matching occurrence appears inside an HTML comment, **Then** that occurrence is included in the replacement.
5. **Given** the agent opts in to one region but not the other, **When** the operation runs, **Then** the opted-in region is included and the other region is still skipped — the two opt-ins are independent.

---

### User Story 3 - Scope the operation to a chosen subfolder (Priority: P3)

An agent that already knows the relevant area of the vault can pass a vault-relative subfolder so only notes under that subtree are examined. Folders that do not exist in the vault are rejected with a typed error, so the agent can distinguish "no matches under that folder" from "the folder name was wrong".

**Why this priority**: Subfolder scoping is a blast-radius control, not a correctness lever — P1 already runs against the whole vault correctly. Targeting a known area matters most for large vaults where the same pattern appears in unrelated regions; smaller vaults can ship value with whole-vault scope alone.

**Independent Test**: With P1 in place, prepare a vault where the pattern appears in two sibling subtrees. Issue the operation against one subtree and verify the response only references notes under that subtree (whether previewing or committing). Issue the operation against a folder that does not exist and verify a typed error names the unknown folder rather than returning an empty success.

**Acceptance Scenarios**:

1. **Given** the agent names a vault-relative subfolder as the scope, **When** the operation runs, **Then** only notes under that subtree are examined and the response reflects the narrowed scope.
2. **Given** the named subfolder does not exist in the vault, **When** the operation runs, **Then** the operation returns a typed error identifying the missing subfolder and no note is modified.

---

### User Story 4 - Refuse when the safe upper bound on occurrences is exceeded (Priority: P3)

When the pattern matches more occurrences than the configured safe upper bound across the chosen scope, the operation refuses to run — for both preview and commit — and returns a typed error indicating the bound was exceeded. The guard fires before any note is modified.

**Why this priority**: A too-broad pattern (an unescaped wildcard, a regex shorter than intended) can match thousands of occurrences across the vault. Without an upper-bound guard, a single careless call could silently rewrite the entire vault. The guard is a blast-radius cap that complements the preview default — preview tells you what will change, the guard prevents the over-broad request from running at all. P3 because the preview default (P1) already protects against accidental mutation; the guard adds protection against accidental enumeration of a vault-sized change set.

**Independent Test**: Configure the safe upper bound to a value below the known occurrence count of a pattern in the test vault. Issue a preview and verify a typed error names the upper bound and the offending occurrence count, with no note modified. Issue a commit and verify the same typed error is returned, with no note modified. Lower the occurrence count below the bound (or raise the bound) and verify the operation now succeeds.

**Acceptance Scenarios**:

1. **Given** the pattern matches more occurrences than the configured upper bound across the chosen scope, **When** the operation is invoked as a preview, **Then** it returns a typed error indicating the bound was exceeded and no note is modified.
2. **Given** the same too-broad pattern, **When** the operation is invoked as a commit, **Then** it returns the same typed error and no note is modified.
3. **Given** the occurrence count is exactly at or below the bound, **When** the operation runs, **Then** it succeeds normally.

### Edge Cases

- The pattern appears multiple times on the same line in the same note: each occurrence is counted and replaced independently; the per-note occurrence count in the preview reflects all of them.
- A note contains the pattern only inside a fenced code block (or only inside an HTML comment) and the corresponding region opt-in is not set: the note is not listed as affected; preview shows zero proposed change for it.
- A fenced code block is left open (no closing fence) at end-of-file: text from the unclosed fence to EOF is treated as code-block content for skip-by-default purposes; this is the safer interpretation when the markdown is malformed.
- The pattern matches text in the frontmatter block at the top of a note: the frontmatter content is treated as prose for matching — frontmatter is NOT separately skipped (out-of-scope clause in the user input). Dedicated frontmatter-key migration is a separate feature.
- The replacement is the empty string (deletion): occurrences are deleted in place. Preview shows the proposed change as a removal; commit removes the text from the note.
- The replacement contains text that itself matches the pattern: the operation makes one pass — replaced text is not re-scanned, so no infinite-loop risk.
- The pattern matches zero-width (e.g., `a*`, `^`, `$`, `\b`, lookarounds in regex mode): zero-width matches are skipped, parity with the sibling read-only `pattern_search` tool (BI-037 FR-016).
- The same note matches in both prose and an opted-out region (e.g., HTML comments off, but the pattern also appears in HTML comments): only the prose occurrences are counted, previewed, and replaced; the opted-out region occurrences do not appear in the per-note occurrence count.
- The chosen subfolder is valid but contains no notes that match: the response is a successful empty result with the narrowed-scope signal — same shape as a whole-vault call that matched nothing.
- The caller passes a subfolder argument containing `../`, a leading `/` or `\`, a drive-letter prefix (e.g., `C:\`), or a control character: the operation rejects at the input-validation boundary with `code: "VALIDATION_ERROR"`, `details.code: "INVALID_SUBFOLDER"`, `details.reason: "path-traversal"` before any vault scan begins. No note is modified, no preview is built.
- Two `find_and_replace` commits fire concurrently against the same vault with an overlapping note set: writes serialize per-note through the injected `Queue` (FR-024). Both commits succeed; the later-acquiring write for any given note observes the earlier commit's content. Per-occurrence count drift between preview and commit (FR-012) catches any case where the earlier commit changed the eligible-occurrence count of a note the later commit targets.
- The caller passes an empty `pattern: ""`: the operation surfaces `code: "VALIDATION_ERROR"`, `details.code: "INVALID_PATTERN"`, `details.reason: "empty"` BEFORE any regex compilation or vault traversal. No note is modified. Catches the "I forgot to set the pattern field" caller bug as a hard error rather than a silent zero-match success.
- The caller passes a `pattern` longer than 1000 UTF-16 code units (e.g., accidentally pasting the contents of a whole note): the operation surfaces `code: "VALIDATION_ERROR"`, `details.code: "INVALID_PATTERN"`, `details.reason: "too-long"` BEFORE any regex compilation or vault traversal. No note is modified.
- The caller passes a syntactically invalid regex pattern in `regex` mode (e.g., `pattern: "[unclosed"`): the operation surfaces `code: "VALIDATION_ERROR"`, `details.code: "INVALID_PATTERN"`, `details.reason: "regex-syntax"` BEFORE any vault traversal. Same `details.code` as the over-cap case but distinguished by `details.reason`. No note is modified.
- The caller passes a `replacement` longer than 1000 UTF-16 code units: the operation surfaces `code: "VALIDATION_ERROR"`, `details.code: "INVALID_REPLACEMENT"` BEFORE any vault traversal. No note is modified.
- A note contains an indented code block (4-space or tab indent at line start, no surrounding fence) with the pattern inside it: the occurrence IS counted and IS replaced (default behaviour), because indented code blocks are treated as prose per FR-006. Callers who want to protect those regions must wrap them in fences.
- A note contains the pattern inside an inline code span (single-backtick span, e.g. `` `OldName` ``): the occurrence IS counted and IS replaced — inline code spans are NOT skip-regions per FR-006.
- The commit halts midway because a single note's write fails with a non-path-safety, non-bound FS error (e.g., disk full ENOSPC, permission denied EACCES, read-only filesystem EROFS, hardware IO error EIO): the operation surfaces `code: "FS_WRITE_FAILED"` with `details.errno` carrying the underlying Node errno string. Notes written successfully before the failure stay written on disk (per FR-015 — no cross-note rollback). The response carries the partial counts (notes-changed-so-far, occurrences-replaced-so-far) and the locator of the failing note so the caller can decide whether to resume. The operation does NOT auto-retry.
- A `.md` file lives under a `.`-prefixed directory (e.g., `.obsidian/plugins/some-plugin/notes.md`): the file is NOT scanned and is NOT eligible for rewrite, even though its extension matches. The `.`-prefixed-directory rule overrides the extension-match rule per FR-020.
- A non-`.md` file in the vault contains the pattern (e.g., `.canvas`, `.base`, `.txt`, `.json` plugin config): the file is NOT scanned and is NOT included in the response. The defensive `.md`-only post-filter applies regardless of what the underlying scan path would otherwise return.
- The caller passes a subfolder argument that points at a `.`-prefixed directory (e.g., `subfolder: ".obsidian"`): the operation surfaces `code: "VALIDATION_ERROR"` with `details.code: "INVALID_SUBFOLDER"` and the response is a successful empty result OR the unknown-subfolder error depending on whether the underlying directory exists. The traversal-time `.`-prefixed-skip in FR-020 means that even if the directory exists, no eligible files are returned from it.
- The caller omits the `vault` argument: the operation runs against the currently-focused vault per the vault-wide cohort default (parity with `find_by_property`, `properties`, `search`, `context_search`, `pattern_search`).
- The caller passes a `vault` display name that does not appear in the lazy vault registry: the operation surfaces `code: "CLI_REPORTED_ERROR"` with `details.code: "VAULT_NOT_FOUND"` and `details.reason: "unknown"`. No note is modified.
- The caller passes a `vault` display name that is registered but the corresponding vault window is not currently open in Obsidian: the operation surfaces `code: "CLI_REPORTED_ERROR"` with `details.code: "VAULT_NOT_FOUND"` and `details.reason: "not-open"`. No note is modified. Caller's recovery path is to open the vault in Obsidian and re-issue.
- The caller passes a syntactically-valid subfolder, but the canonical path (resolved via `fs.realpath`) escapes the vault root — e.g., an in-vault symlink that points outside the vault: the operation surfaces `code: "PATH_ESCAPES_VAULT"` and emits a `pathEscapeAttempt` security event via the project logger. No note is modified.
- A note is unreadable during the scan (e.g., transient `fs.readFile` failure with errno EACCES / EIO / ENOENT-mid-walk): the operation surfaces `code: "FS_WRITE_FAILED"`, `details.reason: "read"`, `details.errno` carrying the underlying Node errno string, and the locator of the failing note. The operation aborts BEFORE any write — partial commits are not initiated; the response carries the error envelope with NO `partial` flag (because no write was attempted). FR-021 covers both per-note read failures during scan AND per-note write failures during commit under the same top-level code, distinguished by `details.reason`.
- The safe upper bound is breached BETWEEN the first scan and the second scan of a single commit invocation: the second-scan count is re-checked against the bound per FR-012(a). When another writer adds occurrences in the window between the two scans pushing the total over the bound, the commit refuses with the bound-exceeded error, even though the first-scan count was under the bound. The two scans run within the same commit invocation — there is no separate "preview retry" call required.
- The vault content drifts BETWEEN the first scan and the second scan of a single commit invocation without breaching the upper bound — e.g., another writer edits a targeted note between the two scans so that the second-scan count is non-zero but differs from the first-scan count: the commit refuses with `code: "VALIDATION_ERROR"`, `details.code: "OCCURRENCE_COUNT_DRIFT"`, `details.preview_count` (first-scan count), `details.commit_count` (second-scan count); no note is modified. The caller's recovery path is to re-issue the commit (which will rerun two fresh scans against the current vault state).
- A literal-mode pattern `foo` is invoked against a note containing both `foo` and `food`: both substrings match — `food` matches because `foo` is a substring of it. Callers who want word-boundary anchoring must use `regex` mode with `\bfoo\b`; there is no separate whole-word flag in this BI.
- A note's source contains mixed CRLF and LF line endings (e.g., a Windows-authored note that has had LF-only lines pasted in): the rewrite preserves each line's original ending byte-for-byte. The operation does not unify the mix as a side effect of rewriting (FR-015).
- A note ends without a trailing newline (no `\n` after the last line): the rewrite preserves the absence of the trailing newline. The operation does not add a trailing newline as a side effect (FR-015).
- The replacement_substring itself contains an embedded `\n` (LF) — e.g., a regex-mode replacement that inserts a paragraph break: that newline is written verbatim into the note, regardless of the per-note dominant line-ending style. The caller is responsible for choosing CRLF / LF in the replacement_substring if it matters.
- Equal-count add+remove drift between the first scan and the second scan of a single commit invocation — e.g., one occurrence is deleted from one note while a new occurrence appears in another note between the two scans, leaving the total occurrence count unchanged: this drift is NOT detected by FR-012's count-only check; the commit proceeds against the second-scan map and writes a rewrite that may differ per-note from what a separate preview call would have surfaced. The clean-git-tree precondition (caller-supplied backup before commit) is the recovery path for this residual TOCTOU window.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST accept a find-pattern argument and an explicit pattern-mode argument with values `literal` or `regex`. Default mode MUST be `literal`. Literal mode MUST treat the pattern as exact text (no metacharacter interpretation). Regex mode MUST treat the pattern as an ECMAScript regular expression, parity with the sibling `pattern_search` tool (BI-037 FR-001).
- **FR-002**: System MUST accept a replacement-text argument. In `literal` mode the replacement MUST be inserted verbatim. In `regex` mode the replacement MUST support ECMAScript replacement-string semantics — `$1`–`$9` for capture groups, `$&` for the whole match, `$$` for a literal dollar sign — parity with the platform's standard string-replace behaviour.
- **FR-003**: System MUST accept an explicit commit-opt-in argument. When the commit opt-in is absent or false, the operation MUST run as a preview and MUST NOT modify any note on disk. When the commit opt-in is true, the operation MUST apply the replacement to every affected note on disk.
- **FR-004**: Preview responses MUST list every affected note with the per-note occurrence count and a per-occurrence list of proposed changes for that note. Each occurrence entry MUST carry, at minimum, the fields `line_number`, `full_line`, `matched_substring`, and `replacement_substring`. The `full_line` field MUST be capped at 500 UTF-16 code units; when the original line is longer, the field MUST carry the first 500 code units followed by `…` (U+2026), parity with BI-033 FR-024 / BI-037 FR-005. Trailing `\r` characters MUST be stripped from `full_line` before clipping, parity with BI-035 context_search FR-012. The `matched_substring` and `replacement_substring` fields MUST NOT be capped — the predicate's hit and its replacement are always returned in full, even when the surrounding `full_line` is clipped. The per-occurrence list MUST be sufficient for the caller to verify what will be written without re-reading the note from disk. Affected notes MUST appear in ascending lexicographic order of their vault-relative path (parity with BI-019 / BI-029 / BI-033). Within each note's per-occurrence list, occurrences MUST appear in ascending `line_number` order; within a single line, in ascending byte-offset order. The ordering MUST be deterministic across runs against the same vault content.
- **FR-005**: Commit responses MUST report the count of notes changed and the count of occurrences replaced overall, plus the `partial` flag and the optional `failing_note_locator` field per FR-025's commit-branch envelope. A commit that matched zero occurrences MUST return zero counts as a successful result, not as an error. The locators of changed notes MUST be surfaced in ascending lexicographic order of their vault-relative path, matching FR-004's preview ordering.
- **FR-006**: System MUST skip occurrences inside fenced markdown code blocks (paired triple-backtick ` ``` ` or paired triple-tilde `~~~`) by default. The caller MUST be able to opt back in to including fenced-code-block occurrences via an explicit include-code-blocks argument. Indented code blocks (CommonMark blocks formed by 4+ spaces or a tab at line start without a surrounding fence) MUST NOT be skipped — they are treated as prose. Inline code spans (single-backtick spans within a line) MUST NOT be skipped either. The fence-skip rule operates at a text level (matching paired fences) and MUST NOT require a full CommonMark parse to disambiguate indented-code-block-vs-list-continuation.
- **FR-007**: System MUST skip occurrences inside HTML comments (`<!-- … -->`) by default. The caller MUST be able to opt back in to including HTML-comment occurrences via an explicit include-html-comments argument. The include-code-blocks and include-html-comments opt-ins MUST be independent — opting into one MUST NOT change the default of the other.
- **FR-008**: System MUST accept an optional vault-relative subfolder argument. When supplied, the operation MUST examine only notes whose path is under that subfolder. When absent, the operation MUST examine every note in the vault.
- **FR-009**: System MUST reject an unknown subfolder with a typed error that names the missing subfolder and MUST NOT silently return an empty result. No note MUST be modified when this error is raised. Path-traversal-shaped subfolder arguments (`../`, leading `/` or `\`, drive-letter prefix, control characters) MUST be rejected at the input-validation boundary with `code: "VALIDATION_ERROR"`, `details.code: "INVALID_SUBFOLDER"`, `details.reason: "path-traversal"` (ADR-015 sub-discriminator) before any vault scan begins. After path normalisation, the handler MUST perform a runtime canonical-path check via `fs.realpath` on the resolved subfolder and MUST verify the canonical result lies under the vault root (`startsWith(realVaultRoot + sep)`); on canonical-level escape (e.g., an in-vault symlink that resolves outside the vault) the handler MUST surface `code: "PATH_ESCAPES_VAULT"` (the existing ADR-009 top-level code — REUSE, not new) and MUST emit a `pathEscapeAttempt` security event via the project logger. The same two-layer check MUST be applied to each affected note's resolved location at commit time; no commit MUST write to any note whose canonical path escapes the vault root. Parity with ADR-009 / BI-016 / BI-021 / BI-030 — full write-side path-safety posture, not the lighter read-side check.
- **FR-010**: System MUST reject a syntactically invalid regular expression (in `regex` mode) with `code: "VALIDATION_ERROR"`, `details.code: "INVALID_PATTERN"`, `details.reason: "regex-syntax"` (ADR-015 sub-discriminator). The error MUST explain the pattern is syntactically invalid. No note MUST be modified when this error is raised. Literal-mode patterns MUST NOT be validated for regex syntax (literal mode has no syntax to invalidate).
- **FR-011**: System MUST enforce a configured safe upper bound on the total number of occurrences across the chosen scope. The bound MUST be read from the process environment variable `OBSIDIAN_FIND_REPLACE_MAX_OCCURRENCES` lazily on the first handler invocation and cached for the process lifetime (parity with `resolveVaultPath`'s lazy-cache pattern and the existing `OBSIDIAN_BIN` env-var convention). When the variable is unset or set to a non-finite-positive-integer value, the bound MUST default to 500 and a WARN MUST be logged via the project logger naming the invalid value. Env-var changes after the first invocation MUST NOT propagate — operators restart the server to change the bound. When the total occurrence count exceeds the bound, the operation MUST return a typed error that names the bound and the offending count, for both preview and commit invocations, and no note MUST be modified.
- **FR-012**: When the commit opt-in is true, the operation MUST run TWO occurrence-counting scans within the same commit invocation: a first scan ("first-scan count" — the same scan that would produce a preview response) followed by a second scan ("second-scan count") taken immediately before any write is performed. Drift detection compares the two counts of the SAME commit invocation — there is no client-side state, no per-call echo, no requirement that the caller round-trip a preview-time count. The handler MUST enforce two state-preconditions before any note is written: (a) the second-scan total occurrence count MUST NOT exceed the safe upper bound — re-checking FR-011 against current content — so that vault drift cannot push a previously-under-bound request over the bound silently; and (b) the second-scan total occurrence count MUST equal the first-scan total occurrence count — when the two counts differ the commit MUST refuse with a typed error envelope `code: "VALIDATION_ERROR"`, `details.code: "OCCURRENCE_COUNT_DRIFT"`, `details.preview_count` (the first-scan count), `details.commit_count` (the second-scan count), and no note MUST be modified. The drift check is count-only — an equal-count add+remove that leaves the total unchanged is a known false negative and is NOT detected by this guard (the project's BI-Safety-Net clean-git-tree precondition covers that residual window).
- **FR-013**: System MUST operate on exactly one vault per invocation; callers that need rewrites across multiple vaults invoke the operation once per vault. The vault is identified by a single optional `vault?: string` input field. When the field is **present**, the operation MUST resolve the supplied display name via the lazy vault registry established by ADR-009 (REUSE of the existing write-surface infrastructure — no new resolver). When the field is **absent**, the operation MUST resolve the focused vault's filesystem root via a small bug-safe `obsidian eval` query against `app.vault.adapter.basePath` (parity with write_note's `FOCUSED_FILE_TEMPLATE` at [src/tools/write_note/handler.ts:48](src/tools/write_note/handler.ts#L48) — constant ~80-byte argv well under the IPC limit). The direct-FS cohort (write_note, find_and_replace) MUST resolve focused-vault path explicitly via this eval; the CLI-bridge cohort siblings (`find_by_property`, `properties`, `search`, `context_search`, `pattern_search`) delegate focused-vault resolution to the upstream CLI by passing `vault: undefined` to `invokeCli`. The operation MUST NOT expose a `target_mode` axis and MUST NOT accept any `@active` sentinel value — ADR-003's target-mode discipline applies to file-targeted tools (BI-011, BI-018, BI-021, BI-030) and explicitly does not reach inherently-vault-wide surfaces per the project architecture. An unknown vault display name MUST surface `code: "CLI_REPORTED_ERROR"`, `details.code: "VAULT_NOT_FOUND"`, `details.reason: "unknown"`. A registered-but-closed vault MUST surface the same `details.code` with `details.reason: "not-open"` (ADR-015 sub-discriminator; closed-but-registered detection per BI-026 v0.5.4 — parity with `pattern_search` handler.ts:74).
- **FR-014**: Preview responses MUST NOT mutate any note on disk under any circumstance, including when the response is later discarded due to a downstream error.
- **FR-015**: Commit responses MUST update notes on disk atomically per note — a note that is updated is updated in full; a note that fails to update mid-write MUST NOT leave a half-written file at its path. Atomicity MUST be implemented via the ADR-009 §3 write-temp-then-rename pattern: each per-note write MUST first `fs.writeFile` to a UUID-suffixed temp path adjacent to the target (e.g. `<target>.<uuid>.tmp`, parity with write_note's `${absPath}.${randomUUID()}.tmp` pattern), then `fs.rename(<temp>, <target>)` to atomically swap the temp over the target. Same-volume rename atomicity (POSIX inode swap; Windows `MoveFileEx`). Mid-process-crash MUST leave either the old content or the new content intact at the target — never a torn write. The UUID-suffixed temp path prevents collision between two concurrent commits both writing temps for the same target (the per-note `Queue` from FR-024 serializes the rename step). Cross-note transactional rollback is out of scope (the caller is responsible for taking a backup before committing). Every byte of a rewritten note that was NOT part of a matched occurrence MUST be preserved byte-for-byte from the source — including line endings (CRLF, LF, or any mix), the presence or absence of a trailing newline at EOF, byte-order marks (BOM), and any other bytes outside the replaced spans (parity with BI-016 FR-004 / SC-002). The operation MUST NOT normalise line endings as a side effect of rewriting. Replacement_substring bytes are written verbatim — including any embedded newline characters the caller supplies — and are NOT normalised to a per-note dominant line-ending style.
- **FR-016**: System MUST scope each occurrence to the position within a single line of a single note where it was found. A single pattern match MUST NOT span line boundaries within a note (parity with BI-037 FR-012) and MUST NOT span notes. In `regex` mode, `\n` metacharacters and any dot-matches-newline behaviour MUST NOT cause cross-line matching — patterns that require newline-spanning will simply not match. In `literal` mode, a pattern string containing an embedded newline MUST NOT match (no line carries an embedded newline). Multi-line refactors are expressed as N single-line replacements.
- **FR-017**: System MUST skip zero-width regex matches (positions where the pattern matches the empty string) — parity with the sibling `pattern_search` tool (BI-037 FR-016). Zero-width matches MUST NOT contribute to the occurrence count and MUST NOT trigger the safe-upper-bound guard.
- **FR-018**: Frontmatter content at the top of a note MUST be treated as prose for matching purposes — frontmatter is NOT a separately-skipped region. Dedicated frontmatter-key migration is a separate feature outside this scope.
- **FR-019**: System MUST accept an optional case-sensitivity control. Default behaviour MUST be case-sensitive matching. An explicit `case_insensitive: true` argument MUST cause the operation to ignore letter case during matching. In `regex` mode this MUST be equivalent to compiling the pattern with the `i` flag. In `literal` mode this MUST apply **ASCII-lower-fold only** — both the pattern and the line content are folded via `String.prototype.toLowerCase()` for the comparison; folding is effective on the range `A-Z` ↔ `a-z`; non-ASCII characters (`É`, `ß`, `İ`, CJK, emoji, etc.) MUST be compared verbatim — `É` does NOT match `é` under case-insensitive literal mode (parity with BI-033's case-handling Q5 / R-Q5 precedent; Unicode case-folding is out of scope for this BI). Parity with BI-037 FR-007 for the case-sensitive default + the opt-in shape.
- **FR-020**: System MUST scan and rewrite only files whose extension is `.md` (case-insensitive on the extension). Non-`.md` files (`.canvas`, `.base`, `.txt`, attachments, plugin data, JSON configs, images, and every other extension) MUST NOT be scanned or rewritten under any circumstance. The filter MUST be applied as a defensive post-filter at the wrapper, parity with BI-033 FR-021 (future-proof against an upstream that broadens indexing). System MUST also skip any path that traverses a directory whose name begins with `.` at any level — including (but not limited to) `.obsidian/`, `.trash/`, `.git/`. A file at `.obsidian/plugins/foo/data.md` MUST NOT be scanned even though its extension matches; the `.`-prefixed-directory rule overrides the extension-match rule. Both filters MUST apply uniformly under whole-vault scope AND under the subfolder scope from FR-008.
- **FR-021**: Generic filesystem failures (ENOSPC, EACCES, EROFS, EIO, ENOENT-during-read, and other Node errno strings outside the path-safety and bound categories already covered by FR-009 / FR-011 / FR-012) MUST surface as `code: "FS_WRITE_FAILED"` (the existing ADR-009 §7 top-level code — REUSE, not new) with `details.errno` carrying the underlying Node errno string AND a `details.reason: "read" | "write"` sub-discriminator (ADR-015 sub-discriminator pattern) distinguishing per-note read failures during the scan from per-note write failures during commit. Per-note **read** failures during the preview-time OR commit-time scan MUST surface `details.reason: "read"` and abort the operation BEFORE any write — no partial commit is initiated; the response carries the error envelope and the failing-note locator with NO `partial` flag (because no write was attempted). Per-note **write** failures during commit MUST surface `details.reason: "write"` and halt the batch at the failing note; per FR-015 there is no cross-note rollback — notes written successfully before the failure stay written, and the commit response MUST carry `partial: true` + the partial counts + the failing-note locator per FR-025. The operation MUST NOT auto-retry the failing note in either case.
- **FR-022**: System MUST require the `pattern` input to be between 1 and 1000 UTF-16 code units (inclusive). The replacement input MUST be capped at 1000 UTF-16 code units; the replacement MAY be the empty string (deletion semantics per the existing edge case). Both length checks MUST be enforced at the input-validation boundary BEFORE any regex compilation, vault traversal, or scan begins. An empty `pattern` MUST surface `code: "VALIDATION_ERROR"`, `details.code: "INVALID_PATTERN"`, `details.reason: "empty"` (ADR-015 sub-discriminator — third sub-state alongside `"too-long"` and `"regex-syntax"` under the shared `INVALID_PATTERN` `details.code`). An over-cap `pattern` MUST surface `code: "VALIDATION_ERROR"`, `details.code: "INVALID_PATTERN"`, `details.reason: "too-long"`. An over-cap `replacement` MUST surface `code: "VALIDATION_ERROR"` with `details.code: "INVALID_REPLACEMENT"` (`details.reason` MAY be absent or set to `"unknown"` — single-state per ADR-015's default-sub-state allowance). No note MUST be modified when any of these errors is raised. Parity with BI-033's 1000-character `query` cap (clarifications-2026-05-16); no new top-level error codes (Constitution Principle IV preserved).
- **FR-023**: System MUST register under the MCP tool name `find_and_replace` (snake_case, with explicit conjunction). The source layout MUST follow the project's per-tool module convention at `src/tools/find_and_replace/{index,schema,tool,handler,*.test}.ts`. The feature directory `specs/038-find-replace/` and the git branch `038-find-replace` retain their `find-replace` short-name from spec scaffolding — directory naming and tool naming are independent per the speckit convention. ADR-010 (typed tool names mirror upstream CLI subcommand) does NOT bind — find_and_replace is a direct-FS tool per ADR-009, not a CLI-bridge tool.
- **FR-024**: The handler MUST take an injected `Queue` dep (from the project's `createQueue()` god-node) as part of its `ExecuteDeps` shape. Commit-time writes within a single invocation MUST be serialized through this `Queue` per-note — each note write acquires the queue independently, mirroring the per-note atomicity of FR-015. Two concurrent `find_and_replace` commits against the same vault MUST interleave at note-granularity but MUST NOT produce torn writes on any single note. Preview operations (read-only) MUST NOT acquire the queue. The handler MUST NOT reach back into the composition root at runtime — `Queue` arrives via DI per Constitution Principle I. Parity with `write_note` / `rename_note` / `move_note` handler shape.
- **FR-025**: The operation's response envelope MUST be a single Zod discriminated union keyed on a top-level `mode: "preview" | "commit"` literal. The `mode: "preview"` branch MUST carry `{ mode: "preview", affected_notes, total_occurrences }` where `affected_notes` is the per-occurrence list specified by FR-004. The `mode: "commit"` branch MUST carry `{ mode: "commit", changed_notes, total_occurrences_replaced, partial }` where `changed_notes` is the locator list per FR-005, `total_occurrences_replaced` is the overall replacement count, and `partial: boolean` indicates whether the batch completed in full (`partial: false`) or was halted mid-batch by an `FS_WRITE_FAILED` write-side failure (`partial: true`). When `partial: true`, the commit response MUST also carry the locator of the failing note alongside the `FS_WRITE_FAILED` error envelope per FR-021. The two branches MUST share a single response schema (Zod discriminated union per Constitution Principle III) rather than splitting into two distinct top-level schemas. Agents MUST be able to type-narrow on the `mode` literal.

### Key Entities

- **Find request**: The optional `vault?: string` field (absent = focused-vault default; present = resolved via the ADR-009 vault registry), the pattern, the pattern mode (`literal` or `regex`), the replacement text, the optional case-sensitivity control, the optional subfolder scope, the include-code-blocks and include-html-comments opt-ins, and the commit opt-in. Defines what the agent asked for.
- **Affected note**: A single note whose content contains at least one non-skipped occurrence of the pattern under the current scope and opt-in settings. Carries the note locator, the per-note occurrence count, and the proposed change.
- **Proposed change**: The shape of the rewrite for a single affected note — sufficient for the caller to verify what would be written if the commit opt-in were set.
- **Response envelope**: A single Zod discriminated union keyed on a top-level `mode: "preview" | "commit"` literal (FR-025). Both branches share the schema; agents type-narrow on the literal.
- **Preview result** *(the `mode: "preview"` branch of the response envelope)*: The list of affected notes with their proposed changes, plus the total-occurrences count across all affected notes. Carries no on-disk side effect signal beyond the `mode` literal itself.
- **Commit result** *(the `mode: "commit"` branch of the response envelope)*: The locators of changed notes, the total-replacements count, and a `partial: boolean` flag indicating whether the batch completed in full (`false`) or was halted mid-batch by an `FS_WRITE_FAILED` (`true`).
- **Safe upper bound**: The configured ceiling on the total occurrence count across the chosen scope above which the operation refuses to run. A property of the tool's configured environment rather than of any single call.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: An agent can preview and then commit a vault-wide find-and-replace in exactly two tool calls (one preview, one commit) without re-authoring the request or copying the preview shape into a different argument schema. 100% of preview-then-commit pairs against an unchanged vault produce the same set of affected notes and the same per-note proposed changes.
- **SC-002**: 100% of preview invocations leave every note on disk unchanged — verifiable by comparing per-note content and mtime before and after the preview call.
- **SC-003**: When code-block and HTML-comment skip-by-default is in effect, 0% of returned proposed changes include an occurrence that lies inside a fenced code block or inside an HTML comment.
- **SC-004**: When the include-code-blocks or include-html-comments opt-ins are set, 100% of occurrences inside the corresponding region are included in the proposed changes.
- **SC-005**: When a subfolder scope is supplied, 0% of affected notes in the response come from outside that subfolder subtree.
- **SC-006**: Unknown subfolder, invalid regex (in `regex` mode), safe-upper-bound-exceeded, commit-time occurrence-count drift, schema-level path-traversal rejection, canonical-level vault-escape rejection, unknown-vault, registered-but-closed-vault, per-note FS read failure during scan, per-note FS write failure during commit, over-cap pattern input, empty-pattern input, and over-cap replacement input are surfaced as thirteen distinct typed-error discriminators so an agent can branch on the error sub-code without parsing human-readable error prose. Drift, schema-level path-traversal, and (existing) unknown-subfolder surface under the existing top-level `VALIDATION_ERROR` with distinct `details.code` (path-traversal carries an additional `details.reason: "path-traversal"` per the ADR-015 sub-discriminator pattern). Canonical-level vault escape surfaces under the existing `PATH_ESCAPES_VAULT` top-level code. Unknown-vault and registered-but-closed-vault surface under the existing `CLI_REPORTED_ERROR` top-level code with `details.code: "VAULT_NOT_FOUND"` and `details.reason: "unknown"` vs `"not-open"` respectively (ADR-015 sub-discriminator; BI-026 v0.5.4 / BI-037 parity). FS read and FS write failures SHARE the existing `FS_WRITE_FAILED` top-level code (ADR-009 §7) with `details.errno` carrying the Node errno string, distinguished by `details.reason: "read"` vs `"write"` (ADR-015 sub-discriminator). Over-cap pattern, empty pattern, and syntactically invalid regex SHARE `details.code: "INVALID_PATTERN"` under the existing `VALIDATION_ERROR` top-level code, distinguished by `details.reason: "too-long"` / `"empty"` / `"regex-syntax"` respectively (ADR-015 sub-discriminator, parity with the `VAULT_NOT_FOUND` + `unknown` / `not-open` shape). Over-cap replacement surfaces under `VALIDATION_ERROR` with `details.code: "INVALID_REPLACEMENT"` (single-state; `details.reason` MAY be absent). All discriminators REUSE existing top-level codes — the eleven-tool zero-new-top-level-codes streak (Constitution Principle IV) is preserved across all thirteen `(top-level code, details.code, details.reason)` triples.
- **SC-007**: A pattern that matches nothing returns a successful empty result with zero affected notes and zero replacements 100% of the time, never as an error.
- **SC-008**: When the safe upper bound is exceeded, 100% of invocations — preview or commit — refuse to run, with no note modified.

## Assumptions

- **Regex dialect**: The `regex` mode uses ECMAScript (Node's built-in `RegExp`) and ECMAScript replacement-string semantics, parity with the sibling `pattern_search` tool (BI-037). The user input excluded regex-dialect negotiation from scope, so the dialect is fixed.
- **Pattern mode default**: When the caller does not declare a pattern mode, the operation runs in `literal` mode. Literal-as-default is the safer choice — a caller who pastes `foo.bar` and forgets to declare regex mode does not accidentally match `fooXbar` and corrupt unrelated notes.
- **Safe upper bound source** *(locked by 2026-05-18 clarifications session)*: The upper bound is read from the process environment variable `OBSIDIAN_FIND_REPLACE_MAX_OCCURRENCES` **lazily on the first handler invocation** and cached for the process lifetime (parity with `resolveVaultPath`'s lazy-cache pattern and the existing `OBSIDIAN_BIN` env-var convention). Default 500 when the variable is unset or invalid; invalid values log a WARN naming the offending value. Env-var changes after the first invocation do NOT propagate — operators restart the server to change the bound. The bound is server-side configured rather than a per-call argument. The bound is enforced uniformly across preview and commit per FR-011 + FR-012.
- **Literal-mode case-folding rule** *(locked by 2026-05-18 analyze-remediation)*: When `case_insensitive: true` is supplied in `literal` mode (FR-019), folding is **ASCII-lower-fold only** — both the pattern and the line content are folded via `String.prototype.toLowerCase()` for the comparison; folding is effective on `A-Z` ↔ `a-z`. Non-ASCII characters (`É`, `ß`, `İ`, CJK, emoji, etc.) are compared verbatim — `É` does NOT match `é` in literal mode under case-insensitive matching. Parity with BI-033 R-Q5 (which also chose ASCII-lower-fold for the same reason: deterministic across Node/ICU versions; avoids Intl/ICU dependency overhead). Unicode case-folding is out of scope for this BI and remains a candidate for a sibling BI if real-user demand surfaces. Regex mode (which uses `RegExp`'s `i` flag) is unaffected by this clarification — `i`-flag matching has its own Unicode semantics governed by ECMAScript.
- **Focused-vault resolution** *(locked by 2026-05-18 analyze-remediation)*: When the `vault` argument is absent (FR-013), the operation queries the focused vault's filesystem root via a small bug-safe `obsidian eval` against `app.vault.adapter.basePath` — parity with write_note's `FOCUSED_FILE_TEMPLATE` at [src/tools/write_note/handler.ts:48](src/tools/write_note/handler.ts#L48), constant ~80-byte argv well under the IPC limit. Find_and_replace is direct-FS for the scan + write paths (read AND write bypass the CLI), but the vault-root discovery itself uses one CLI eval round-trip when the caller does not name the vault explicitly. The CLI-bridge cohort siblings (`find_by_property`, `properties`, `search`, `context_search`, `pattern_search`) delegate focused-vault resolution to the upstream CLI implicitly by passing `vault: undefined` to `invokeCli`; the direct-FS cohort cannot delegate and resolves explicitly. The eval response carries the absolute base path; the handler then applies `fs.realpath` + the Layer-2 canonical check before any scan begins.
- **Per-note atomicity, not cross-note transactionality**: Commit writes are atomic per note (a note is fully written or unchanged), but the operation does not transactionally roll back already-written notes if a later note fails to write. Cross-note rollback is out of scope per the user input; the caller is responsible for taking a backup before committing.
- **Eligible files** *(locked by 2026-05-18 clarifications session)*: Only files with extension `.md` (case-insensitive) are scanned and rewritten. Hidden / system directories — every path that traverses a `.`-prefixed directory name (`.obsidian/`, `.trash/`, `.git/`, etc.) — are skipped at every level; a `.md` file inside `.obsidian/plugins/...` is NOT scanned. Defensive post-filter at the wrapper, parity with BI-033 FR-021 (future-proofs against upstream broadening). Non-`.md` files (`.canvas`, `.base`, `.txt`, attachments, plugin data, images) are not scanned or modified.
- **Single-vault per invocation** *(locked by 2026-05-18 clarifications session)*: One vault per invocation. Vault is identified via a single optional `vault?: string` field; absent ⇒ focused-vault default (vault-wide cohort parity with `find_by_property`, `properties`, `search`, `context_search`, `pattern_search`); present ⇒ resolved via the ADR-009 lazy vault registry. No `target_mode` axis, no `@active` sentinel — ADR-003's target-mode discipline applies to file-targeted tools and explicitly does not reach inherently-vault-wide surfaces per the project architecture. Cross-vault aggregation is out of scope per the user input.
- **Fenced code block detection**: A "fenced code block" is identified by paired triple-backtick (` ``` `) or paired triple-tilde (`~~~`) fences in standard CommonMark style. Inline code spans (single backticks) are not skipped — only fenced blocks are. Detection is text-level, not full-CommonMark-parser-level — sufficient for the safe default in well-formed notes.
- **HTML comment detection**: An "HTML comment" is identified by `<!--` … `-->` pairs in the note's text, including multi-line comments.
- **Subfolder semantics**: "Subfolder of the vault" means a path interpreted relative to the vault root, consistent with sibling tools in this surface that take a folder argument.
- **Ordering of affected notes** *(locked by 2026-05-18 clarifications session)*: Affected notes appear in ascending lexicographic order of vault-relative path; within each note, occurrences appear in ascending `line_number` then ascending byte-offset within a line. Deterministic across runs against the same vault content. Parity with sibling list/search tools (BI-019, BI-029, BI-033). See FR-004 + FR-005.
- **No retry semantics**: A failed commit (e.g., due to a transient file-system error mid-batch) is not automatically retried by the operation. The caller decides whether to re-issue.
- **No whole-word flag** *(locked by 2026-05-18 clarifications session)*: There is no whole-word opt-in in this BI. `literal` mode matches anywhere the pattern appears as a substring — `foo` matches inside `foobar`. Callers wanting word-boundary anchoring use `regex` mode with `\b` (parity with BI-037 pattern_search). A future feature can add whole-word if use cases force it.
