# Feature Specification: Find Replace

**Feature Branch**: `038-find-replace`
**Created**: 2026-05-18
**Status**: Draft
**Input**: User description: "Add Find Replace — across a chosen scope of vault notes, the user can replace every occurrence of a chosen text pattern (a literal string or a regular expression) with a chosen replacement. Preview by default, commit only on explicit opt-in. Code blocks and HTML comments skipped by default, opt-in to include. Optional subfolder scope. Refuses to run when matches exceed a configured safe upper bound."

## Clarifications

### Session 2026-05-18

- Q: Vault target mode — does the operation expose `target_mode` / `@active` sentinel for focused-vault selection, or follow some other vault-selection contract? → A: **Vault-wide cohort contract — single optional `vault?: string` field, focused-vault default, no `target_mode` axis, no `@active` sentinel.** Find-and-replace is inherently a vault-wide surface (operates across many notes per invocation), not file-targeted. Per the project architecture's verbatim statement, "ADR-003's scope explicitly does not reach inherently-vault-wide surfaces" — ADR-003's target-mode discipline applies to single-file-target tools (BI-011 write_note, BI-018 write_property, BI-021 rename_note, BI-030 move_note), NOT to vault-wide cohort tools. The correct cohort parity is `find_by_property`, `properties`, `search`, `context_search`, and `pattern_search` (BI-037) — all of which use an optional `vault?: string` field with NO `target_mode` axis. **Input contract**: a single optional `vault?: string`. When absent, the operation defaults to the focused vault (cohort default). When present, the operation resolves the supplied display name via the lazy vault registry established by ADR-009 — REUSE of the existing write-surface infrastructure, no new resolver. **Error contract**: an unknown vault display name surfaces `code: "CLI_REPORTED_ERROR"` with `details.code: "VAULT_NOT_FOUND"` and `details.reason: "unknown"`; a registered-but-closed vault surfaces the same `details.code` with `details.reason: "not-open"` (ADR-015 sub-discriminator pattern; closed-but-registered detection branch per BI-026 v0.5.4 — `pattern_search` handler.ts:74 and handler.test.ts:293 already implement this exact pair). **Path-safety** from Q4 applies POST-resolution: once the vault is resolved (focused or named), the resolved vault root is `fs.realpath`'d before the subfolder check and the per-note canonical-path check.
- Q: Subfolder argument — path-traversal handling — how does the operation reject subfolder arguments that attempt to escape the vault root (via `../`, leading `/` or `\`, drive-letter prefix, control characters, or in-vault symlinks pointing outside)? → A: **Two-layer path safety per ADR-009, inheriting the existing posture from the write surface.** Find-and-replace is a write-side tool and adopts the full write-side path-safety stance, not the lighter read-side check. **Layer 1 (schema-level pre-flight)**: the input schema rejects path-traversal-shaped inputs (`../`, leading `/` or `\`, drive-letter prefix, control characters) at validation boundary and surfaces top-level `code: "VALIDATION_ERROR"` + `details.code: "INVALID_SUBFOLDER"` + `details.reason: "path-traversal"` (ADR-015 sub-discriminator). **Layer 2 (runtime canonical-path check)**: after resolving the absolute filesystem path, the handler invokes `fs.realpath` on the resolved subfolder and verifies `startsWith(realVaultRoot + sep)` on the canonical result. On escape (e.g., an in-vault symlink that resolves outside the vault) the handler surfaces top-level `code: "PATH_ESCAPES_VAULT"` (REUSE of the existing ADR-009 code — no new top-level code) and emits a `pathEscapeAttempt` security event via the project logger for the operator audit trail. Adopting Layer 1 alone would leave symlink-escape attacks blind; reusing `PATH_ESCAPES_VAULT` preserves the eleven-tool zero-new-top-level-codes streak (Constitution Principle IV) and unifies the security posture across the write surface. The same posture is applied to the per-note write path (each affected note's resolved location must also lie under the vault root); no commit MUST proceed if any affected note's canonical path escapes the vault root.
- Q: Whole-word matching for literal mode — does the operation expose an opt-in whole-word flag so that `foo` in `literal` mode does not match inside `food` / `foobar`? → A: **No whole-word flag in this BI.** Callers that need word-boundary anchoring drop to `regex` mode and write `\bfoo\b` explicitly. Parity with BI-037 pattern_search, which similarly does not expose a whole-word flag. The input surface stays narrow (no third axis beyond pattern mode + case sensitivity); the ECMAScript `\b` escape hatch is one-line and already supported by the regex-mode path. A future feature can add whole-word as a literal-mode opt-in if real use cases force it.
- Q: Stable ordering of affected notes and per-occurrence entries — what is the deterministic order in which affected notes appear in the preview/commit response, and the order of occurrences within each note? → A: **Two-level deterministic order — path-ascending across notes, then line-number-ascending within each note.** Affected notes MUST appear in ascending lexicographic order of their vault-relative path (parity with BI-019 list_files, BI-029 list_files_recursive, BI-033 search_vault_content). Within each note's per-occurrence list, occurrences MUST appear in ascending line-number order, and within a single line in ascending byte-offset order. The ordering is stable across runs given the same vault content — deterministic for tests and chunkable for agents that iterate the response top-to-bottom. Commit responses MUST surface the locators of changed notes in the same path-ascending order.
- Q: Line-ending preservation on commit write — when a note is rewritten on disk, are the source's line endings (CRLF / LF / mixed) preserved? → A: **Byte-for-byte preservation of unmatched content; per-line endings untouched.** Commit-time writes MUST preserve every byte of the source note that was not part of a matched occurrence — including line endings (CRLF, LF, or any mix), trailing-newline-or-not at EOF, BOM, and other bytes outside the replaced spans. Parity with BI-016 FR-004 / SC-002 (write_note byte-for-byte persistence including mixed CRLF/LF). Only the matched_substring bytes are replaced with the replacement_substring bytes; the surrounding line-ending context is left untouched. This forbids the implementation from "normalising" line endings as a side effect of round-tripping the note through a string split/join pipeline. Cross-platform-support invariant (BI-017) intact — Windows-authored CRLF notes do not churn to LF on rewrite, and macOS-authored LF notes do not gain `\r`. If the caller's replacement_substring itself contains an embedded newline character, that character is written verbatim — the operation does not normalise replacement_substring to the per-note dominant style.
- Q: Case-sensitivity control — does the operation support case-insensitive matching, and what is the default? → A: **Parity with BI-037 FR-007.** Default is case-sensitive matching. An explicit `case_insensitive: true` opt-in flips matching to ignore letter case. The flag applies uniformly to both `literal` and `regex` modes — in `regex` mode it is equivalent to setting the `i` flag on the compiled `RegExp`. Zero divergence from the sibling read-side tool; agents that learned the surface from `pattern_search` carry the same mental model into `find_replace`.
- Q: Multi-line pattern scope — can the find-pattern span line boundaries within a note? → A: **Line-scoped only, parity with BI-037 FR-012.** The find-pattern MUST NOT span line boundaries; matching runs line-by-line per note. In `regex` mode, `\n` metacharacters and dot-matches-newline behaviour do NOT enable cross-line matching — a pattern that uses `\n` simply does not match (no line in the note contains a literal `\n` byte). In `literal` mode, a pattern string that contains an embedded `\n` does not match (no single line contains a literal newline). Multi-line refactors are expressed as N single-line replacements; a separate future tool would handle multi-line spans if real use cases force it. Keeps per-occurrence B-shape (FR-004) unchanged — no `end_line` field needed — and preserves zero divergence from the established read-side regex semantics.
- Q: Per-note proposed-change shape in preview — how is the proposed rewrite represented for each affected note? → A: **Per-occurrence list, B-shape verbatim.** Each occurrence carries `{ line_number, full_line, matched_substring, replacement_substring }`. The `full_line` field is clipped to 500 UTF-16 code units (with trailing `…` U+2026 when clipped) per BI-033 FR-024 / BI-037 FR-005; trailing `\r` stripping follows BI-035 context_search FR-012 precedent. `matched_substring` and `replacement_substring` are never clipped — the predicate's hit and its replacement are always returned in full, even when the surrounding `full_line` is clipped. Parity with BI-037 `pattern_search` per-match envelope plus the new `replacement_substring` field. Structured per-occurrence shape Zod-schematises cleanly per Constitution Principle III; unified-diff text (rejected option A) is opaque to Zod, and full-note before/after content (rejected options C and D) violates the project's payload-bound discipline (parity with BI-033 / BI-035's 1000-entry truncation cap and the in-flight BI-0085 bound). This shape holds for single-line patterns; multi-line patterns are a separate scope question deferred to a subsequent clarification (see Outstanding below).
- Q: Safe-upper-bound source and default value — where does the configured ceiling on total occurrences live, and what is the default? → A: **Operator-tunable environment variable `OBSIDIAN_FIND_REPLACE_MAX_OCCURRENCES`, default 500.** The bound is read from the process environment at server start (parity with the existing `OBSIDIAN_BIN` env-var convention) rather than from a per-call argument or a hard-coded constant. Default 500 is small enough that an agent issuing a typical ADR-rename refactor surfaces the bound as a guardrail in normal use (the refusal forces the agent to narrow the scope or raise the bound deliberately) rather than as a rarely-tripped backstop, while still permitting routine cross-folder refactors. Operators with larger vaults set the env var to a higher value on install; the bound is enforced uniformly across preview and commit per FR-011 + FR-012.
- Q: Vault drift between preview and commit — what happens if a note's content changes between a successful preview and the subsequent commit, so that committing would produce a different rewrite than the preview promised? → A: **Count-only drift detection under the existing `VALIDATION_ERROR` top-level code.** At commit time, the operation re-runs occurrence-counting across the chosen scope on the current vault content. When the commit-time total occurrence count differs from the preview-time total occurrence count, the commit refuses with a typed error: top-level `code: "VALIDATION_ERROR"`, `details.code: "OCCURRENCE_COUNT_DRIFT"`, and no note is modified. No new top-level error code — the eleven-tool zero-new-top-level-codes streak since BI-011 / ADR-009 / Constitution Principle IV is preserved by surfacing the state-precondition failure through `details.code` discrimination per the ADR-015 sub-discriminator pattern. Count-only drift detection accepts the known false-negative of an equal-count add+remove (e.g., one occurrence appears elsewhere while one is deleted, leaving the total unchanged); the project's BI-Safety-Net clean-git-tree precondition shoulders that residual TOCTOU window. Per-note content-hash fingerprinting was rejected as forcing the caller to round-trip per-file fingerprints for marginal coverage gain over the count-based check.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Preview the replacement before any note changes (Priority: P1)

An agent issues a find-and-replace request against a vault — supplying the pattern, the replacement, and (optionally) a subfolder scope — without opting in to commit. The response lists every affected note together with the proposed change for that note and the per-note occurrence count, while no note on disk is modified. The agent can read the preview, confirm the scope is what it expected, and then re-issue the same call with the explicit commit opt-in to apply the rewrite. A preview that affects nothing returns a successful empty result rather than an error.

**Why this priority**: Bulk refactors against a vault are irreversible once committed; without a preview the agent cannot confirm the blast radius before mutation. Preview-then-commit is the safety contract that makes the whole feature usable — every other behaviour (skip-defaults, scope, upper-bound guard) builds on top of it. Without P1 there is no MVP; with only P1 the agent already gets a vault-wide refactor capability that is strictly safer than the manual read-rewrite loop it replaces.

**Independent Test**: Stand up a vault containing notes with a known occurrence count of the pattern across multiple notes. Issue the call without the commit opt-in. Verify the response lists every affected note with its proposed change, verify no file on disk is modified (compare mtimes and contents before/after), then re-issue the same call with the commit opt-in and verify the same set of notes is updated on disk with the same change shown in the preview. Separately, issue a pattern that matches nothing and verify the response is a successful empty result, not an error.

**Acceptance Scenarios**:

1. **Given** a vault with notes containing the pattern in prose, **When** the agent issues the operation without the commit opt-in, **Then** the response lists every affected note with the proposed change and the per-note occurrence count, and no note on disk is modified.
2. **Given** the agent has just received a preview, **When** the agent re-issues the same request with the explicit commit opt-in, **Then** the same set of notes is updated on disk and the response reports how many notes changed and how many occurrences were replaced overall.
3. **Given** the pattern matches no occurrence anywhere in scope, **When** the agent issues either a preview or a commit, **Then** the response is a successful empty result with zero affected notes and zero replacements, not an error.
4. **Given** the agent issues a request without specifying the commit opt-in, **When** the operation runs, **Then** it behaves identically to an explicit preview — no note on disk is modified — so that an accidental call cannot mutate the vault.

---

### User Story 2 - Skip code blocks and HTML comments by default; opt back in when needed (Priority: P2)

By default, occurrences inside fenced code blocks and inside HTML comments are left alone — code samples embedded in notes are not corrupted, and audit-trail comments embedded in notes are not silently rewritten. When the refactor genuinely targets those regions (e.g., a deliberate rename of a symbol that appears inside code samples), the agent can opt back in to either region — independently — and have occurrences inside that region replaced alongside the prose occurrences.

**Why this priority**: Without skip-by-default, every find-and-replace risks corrupting embedded code, command lines, examples, and the comment-based audit trail. The default is the safety contract that lets agents reach for this tool without hand-auditing every match. The opt-back-in is the escape hatch that prevents the safe default from blocking the legitimate "rename this symbol everywhere, including in code samples" case. P2 because P1 alone can already operate against a no-code-blocks vault, but most real vaults mix prose with fenced code and HTML comments — without P2, the tool would be unsafe for the common case.

**Independent Test**: With P1 in place, prepare a note where the pattern appears once in prose, once inside a fenced code block, and once inside an HTML comment. Run a preview with the default skip behaviour and verify only the prose occurrence is in the proposed change. Run a preview with the code-block opt-in and verify the prose and code-block occurrences both appear. Run a preview with the HTML-comment opt-in and verify the prose and comment occurrences both appear. Run with both opt-ins and verify all three appear.

**Acceptance Scenarios**:

1. **Given** a note where the pattern appears in prose and again inside a fenced code block, **When** the operation runs with the default skip behaviour, **Then** the prose occurrence is replaced and the code-block occurrence is left unchanged.
2. **Given** a note where the pattern appears inside an HTML comment, **When** the operation runs with the default skip behaviour, **Then** the comment is left unchanged.
3. **Given** the agent opts to include fenced code blocks, **When** a matching occurrence appears inside a fenced code block, **Then** that occurrence is included in the replacement.
4. **Given** the agent opts to include HTML comments, **When** a matching occurrence appears inside an HTML comment, **Then** that occurrence is included in the replacement.
5. **Given** the agent opts in to one region but not the other, **When** the operation runs, **Then** the opted-in region is included and the other region is still skipped — the two opt-ins are independent.

---

### User Story 3 - Scope the operation to a chosen subfolder (Priority: P3)

An agent that already knows the relevant area of the vault can pass a vault-relative subfolder so only notes under that subtree are examined. Folders that do not exist in the vault are rejected with a typed error, so the agent can distinguish "no matches under that folder" from "the folder name was wrong".

**Why this priority**: Subfolder scoping is a blast-radius control, not a correctness lever — P1 already runs against the whole vault correctly. Targeting a known area matters most for large vaults where the same pattern appears in unrelated regions; smaller vaults can ship value with whole-vault scope alone.

**Independent Test**: With P1 in place, prepare a vault where the pattern appears in two sibling subtrees. Issue the operation against one subtree and verify the response only references notes under that subtree (whether previewing or committing). Issue the operation against a folder that does not exist and verify a typed error names the unknown folder rather than returning an empty success.

**Acceptance Scenarios**:

1. **Given** the agent names a vault-relative subfolder as the scope, **When** the operation runs, **Then** only notes under that subtree are examined and the response reflects the narrowed scope.
2. **Given** the named subfolder does not exist in the vault, **When** the operation runs, **Then** the operation returns a typed error identifying the missing subfolder and no note is modified.

---

### User Story 4 - Refuse when the safe upper bound on occurrences is exceeded (Priority: P3)

When the pattern matches more occurrences than the configured safe upper bound across the chosen scope, the operation refuses to run — for both preview and commit — and returns a typed error indicating the bound was exceeded. The guard fires before any note is modified.

**Why this priority**: A too-broad pattern (an unescaped wildcard, a regex shorter than intended) can match thousands of occurrences across the vault. Without an upper-bound guard, a single careless call could silently rewrite the entire vault. The guard is a blast-radius cap that complements the preview default — preview tells you what will change, the guard prevents the over-broad request from running at all. P3 because the preview default (P1) already protects against accidental mutation; the guard adds protection against accidental enumeration of a vault-sized change set.

**Independent Test**: Configure the safe upper bound to a value below the known occurrence count of a pattern in the test vault. Issue a preview and verify a typed error names the upper bound and the offending occurrence count, with no note modified. Issue a commit and verify the same typed error is returned, with no note modified. Lower the occurrence count below the bound (or raise the bound) and verify the operation now succeeds.

**Acceptance Scenarios**:

1. **Given** the pattern matches more occurrences than the configured upper bound across the chosen scope, **When** the operation is invoked as a preview, **Then** it returns a typed error indicating the bound was exceeded and no note is modified.
2. **Given** the same too-broad pattern, **When** the operation is invoked as a commit, **Then** it returns the same typed error and no note is modified.
3. **Given** the occurrence count is exactly at or below the bound, **When** the operation runs, **Then** it succeeds normally.

---

### Edge Cases

- The pattern appears multiple times on the same line in the same note: each occurrence is counted and replaced independently; the per-note occurrence count in the preview reflects all of them.
- A note contains the pattern only inside a fenced code block (or only inside an HTML comment) and the corresponding region opt-in is not set: the note is not listed as affected; preview shows zero proposed change for it.
- A fenced code block is left open (no closing fence) at end-of-file: text from the unclosed fence to EOF is treated as code-block content for skip-by-default purposes; this is the safer interpretation when the markdown is malformed.
- The pattern matches text in the frontmatter block at the top of a note: the frontmatter content is treated as prose for matching — frontmatter is NOT separately skipped (out-of-scope clause in the user input). Dedicated frontmatter-key migration is a separate feature.
- The replacement is the empty string (deletion): occurrences are deleted in place. Preview shows the proposed change as a removal; commit removes the text from the note.
- The replacement contains text that itself matches the pattern: the operation makes one pass — replaced text is not re-scanned, so no infinite-loop risk.
- The pattern matches zero-width (e.g., `a*`, `^`, `$`, `\b`, lookarounds in regex mode): zero-width matches are skipped, parity with the sibling read-only `pattern_search` tool (BI-037 FR-016).
- The same note matches in both prose and an opted-out region (e.g., HTML comments off, but the pattern also appears in HTML comments): only the prose occurrences are counted, previewed, and replaced; the opted-out region occurrences do not appear in the per-note occurrence count.
- The chosen subfolder is valid but contains no notes that match: the response is a successful empty result with the narrowed-scope signal — same shape as a whole-vault call that matched nothing.
- The caller passes a subfolder argument containing `../`, a leading `/` or `\`, a drive-letter prefix (e.g., `C:\`), or a control character: the operation rejects at the input-validation boundary with `code: "VALIDATION_ERROR"`, `details.code: "INVALID_SUBFOLDER"`, `details.reason: "path-traversal"` before any vault scan begins. No note is modified, no preview is built.
- The caller omits the `vault` argument: the operation runs against the currently-focused vault per the vault-wide cohort default (parity with `find_by_property`, `properties`, `search`, `context_search`, `pattern_search`).
- The caller passes a `vault` display name that does not appear in the lazy vault registry: the operation surfaces `code: "CLI_REPORTED_ERROR"` with `details.code: "VAULT_NOT_FOUND"` and `details.reason: "unknown"`. No note is modified.
- The caller passes a `vault` display name that is registered but the corresponding vault window is not currently open in Obsidian: the operation surfaces `code: "CLI_REPORTED_ERROR"` with `details.code: "VAULT_NOT_FOUND"` and `details.reason: "not-open"`. No note is modified. Caller's recovery path is to open the vault in Obsidian and re-issue.
- The caller passes a syntactically-valid subfolder, but the canonical path (resolved via `fs.realpath`) escapes the vault root — e.g., an in-vault symlink that points outside the vault: the operation surfaces `code: "PATH_ESCAPES_VAULT"` and emits a `pathEscapeAttempt` security event via the project logger. No note is modified.
- A note is unreadable (e.g., transient file-system error) during the scan: the operation surfaces a typed error rather than silently skipping the note; partial commits are not initiated unless every targeted note can be read.
- The safe upper bound is exceeded during a commit retry after a successful preview: the bound is re-checked at commit time. If the vault changed between preview and commit and the new occurrence count exceeds the bound, the commit refuses with the bound-exceeded error, even though the preview succeeded.
- The vault content drifts between preview and commit without breaching the upper bound — e.g., another agent edits a targeted note in the interim so that the commit-time occurrence count is non-zero but differs from the preview-time count: the commit refuses with `code: "VALIDATION_ERROR"`, `details.code: "OCCURRENCE_COUNT_DRIFT"`, and no note is modified. The caller's recovery path is to re-preview against the current vault state and re-commit.
- A literal-mode pattern `foo` is invoked against a note containing both `foo` and `food`: both substrings match — `food` matches because `foo` is a substring of it. Callers who want word-boundary anchoring must use `regex` mode with `\bfoo\b`; there is no separate whole-word flag in this BI.
- A note's source contains mixed CRLF and LF line endings (e.g., a Windows-authored note that has had LF-only lines pasted in): the rewrite preserves each line's original ending byte-for-byte. The operation does not unify the mix as a side effect of rewriting (FR-015).
- A note ends without a trailing newline (no `\n` after the last line): the rewrite preserves the absence of the trailing newline. The operation does not add a trailing newline as a side effect (FR-015).
- The replacement_substring itself contains an embedded `\n` (LF) — e.g., a regex-mode replacement that inserts a paragraph break: that newline is written verbatim into the note, regardless of the per-note dominant line-ending style. The caller is responsible for choosing CRLF / LF in the replacement_substring if it matters.
- Equal-count add+remove drift between preview and commit — e.g., one occurrence is deleted from one note while a new occurrence appears in another note, leaving the total occurrence count unchanged: this drift is NOT detected by FR-012's count-only check; the commit proceeds against the current vault content and writes a rewrite that may differ per-note from the preview. The clean-git-tree precondition (caller-supplied backup before commit) is the recovery path for this residual TOCTOU window.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST accept a find-pattern argument and an explicit pattern-mode argument with values `literal` or `regex`. Default mode MUST be `literal`. Literal mode MUST treat the pattern as exact text (no metacharacter interpretation). Regex mode MUST treat the pattern as an ECMAScript regular expression, parity with the sibling `pattern_search` tool (BI-037 FR-001).
- **FR-002**: System MUST accept a replacement-text argument. In `literal` mode the replacement MUST be inserted verbatim. In `regex` mode the replacement MUST support ECMAScript replacement-string semantics — `$1`–`$9` for capture groups, `$&` for the whole match, `$$` for a literal dollar sign — parity with the platform's standard string-replace behaviour.
- **FR-003**: System MUST accept an explicit commit-opt-in argument. When the commit opt-in is absent or false, the operation MUST run as a preview and MUST NOT modify any note on disk. When the commit opt-in is true, the operation MUST apply the replacement to every affected note on disk.
- **FR-004**: Preview responses MUST list every affected note with the per-note occurrence count and a per-occurrence list of proposed changes for that note. Each occurrence entry MUST carry, at minimum, the fields `line_number`, `full_line`, `matched_substring`, and `replacement_substring`. The `full_line` field MUST be capped at 500 UTF-16 code units; when the original line is longer, the field MUST carry the first 500 code units followed by `…` (U+2026), parity with BI-033 FR-024 / BI-037 FR-005. Trailing `\r` characters MUST be stripped from `full_line` before clipping, parity with BI-035 context_search FR-012. The `matched_substring` and `replacement_substring` fields MUST NOT be capped — the predicate's hit and its replacement are always returned in full, even when the surrounding `full_line` is clipped. The per-occurrence list MUST be sufficient for the caller to verify what will be written without re-reading the note from disk. Affected notes MUST appear in ascending lexicographic order of their vault-relative path (parity with BI-019 / BI-029 / BI-033). Within each note's per-occurrence list, occurrences MUST appear in ascending `line_number` order; within a single line, in ascending byte-offset order. The ordering MUST be deterministic across runs against the same vault content.
- **FR-005**: Commit responses MUST report the count of notes changed and the count of occurrences replaced overall. A commit that matched zero occurrences MUST return zero counts as a successful result, not as an error. The locators of changed notes MUST be surfaced in ascending lexicographic order of their vault-relative path, matching FR-004's preview ordering.
- **FR-006**: System MUST skip occurrences inside fenced markdown code blocks by default. The caller MUST be able to opt back in to including fenced-code-block occurrences via an explicit include-code-blocks argument.
- **FR-007**: System MUST skip occurrences inside HTML comments (`<!-- … -->`) by default. The caller MUST be able to opt back in to including HTML-comment occurrences via an explicit include-html-comments argument. The include-code-blocks and include-html-comments opt-ins MUST be independent — opting into one MUST NOT change the default of the other.
- **FR-008**: System MUST accept an optional vault-relative subfolder argument. When supplied, the operation MUST examine only notes whose path is under that subfolder. When absent, the operation MUST examine every note in the vault.
- **FR-009**: System MUST reject an unknown subfolder with a typed error that names the missing subfolder and MUST NOT silently return an empty result. No note MUST be modified when this error is raised. Path-traversal-shaped subfolder arguments (`../`, leading `/` or `\`, drive-letter prefix, control characters) MUST be rejected at the input-validation boundary with `code: "VALIDATION_ERROR"`, `details.code: "INVALID_SUBFOLDER"`, `details.reason: "path-traversal"` (ADR-015 sub-discriminator) before any vault scan begins. After path normalisation, the handler MUST perform a runtime canonical-path check via `fs.realpath` on the resolved subfolder and MUST verify the canonical result lies under the vault root (`startsWith(realVaultRoot + sep)`); on canonical-level escape (e.g., an in-vault symlink that resolves outside the vault) the handler MUST surface `code: "PATH_ESCAPES_VAULT"` (the existing ADR-009 top-level code — REUSE, not new) and MUST emit a `pathEscapeAttempt` security event via the project logger. The same two-layer check MUST be applied to each affected note's resolved location at commit time; no commit MUST write to any note whose canonical path escapes the vault root. Parity with ADR-009 / BI-016 / BI-021 / BI-030 — full write-side path-safety posture, not the lighter read-side check.
- **FR-010**: System MUST reject a syntactically invalid regular expression (in `regex` mode) with a typed error that explains the pattern is invalid. No note MUST be modified when this error is raised. Literal-mode patterns MUST NOT be validated for regex syntax.
- **FR-011**: System MUST enforce a configured safe upper bound on the total number of occurrences across the chosen scope. The bound MUST be read from the process environment variable `OBSIDIAN_FIND_REPLACE_MAX_OCCURRENCES` at server start; when the variable is unset or invalid, the bound MUST default to 500. When the total occurrence count exceeds the bound, the operation MUST return a typed error that names the bound and the offending count, for both preview and commit invocations, and no note MUST be modified.
- **FR-012**: At commit time (when the commit opt-in is true), the operation MUST re-run occurrence-counting across the chosen scope on the current vault content and MUST enforce two state-preconditions before any note is written: (a) the commit-time total occurrence count MUST NOT exceed the safe upper bound — re-checking FR-011 against current content — so that vault drift cannot push a previously-under-bound request over the bound silently; and (b) the commit-time total occurrence count MUST equal the preview-time total occurrence count — when the two counts differ the commit MUST refuse with a typed error envelope `code: "VALIDATION_ERROR"`, `details.code: "OCCURRENCE_COUNT_DRIFT"`, and no note MUST be modified. The drift check is count-only — an equal-count add+remove that leaves the total unchanged is a known false negative and is NOT detected by this guard (the project's BI-Safety-Net clean-git-tree precondition covers that residual window).
- **FR-013**: System MUST operate on exactly one vault per invocation; callers that need rewrites across multiple vaults invoke the operation once per vault. The vault is identified by a single optional `vault?: string` input field. When absent, the operation MUST default to the focused vault (the vault-wide-cohort default — parity with `find_by_property`, `properties`, `search`, `context_search`, `pattern_search`). When present, the operation MUST resolve the supplied display name via the lazy vault registry established by ADR-009 (REUSE of the existing write-surface infrastructure — no new resolver). The operation MUST NOT expose a `target_mode` axis and MUST NOT accept any `@active` sentinel value — ADR-003's target-mode discipline applies to file-targeted tools (BI-011, BI-018, BI-021, BI-030) and explicitly does not reach inherently-vault-wide surfaces per the project architecture. An unknown vault display name MUST surface `code: "CLI_REPORTED_ERROR"`, `details.code: "VAULT_NOT_FOUND"`, `details.reason: "unknown"`. A registered-but-closed vault MUST surface the same `details.code` with `details.reason: "not-open"` (ADR-015 sub-discriminator; closed-but-registered detection per BI-026 v0.5.4 — parity with `pattern_search` handler.ts:74).
- **FR-014**: Preview responses MUST NOT mutate any note on disk under any circumstance, including when the response is later discarded due to a downstream error.
- **FR-015**: Commit responses MUST update notes on disk atomically per note — a note that is updated is updated in full; a note that fails to update mid-write does not leave a half-written file at its path. Cross-note transactional rollback is out of scope (the caller is responsible for taking a backup before committing). Every byte of a rewritten note that was NOT part of a matched occurrence MUST be preserved byte-for-byte from the source — including line endings (CRLF, LF, or any mix), the presence or absence of a trailing newline at EOF, byte-order marks (BOM), and any other bytes outside the replaced spans (parity with BI-016 FR-004 / SC-002). The operation MUST NOT normalise line endings as a side effect of rewriting. Replacement_substring bytes are written verbatim — including any embedded newline characters the caller supplies — and are NOT normalised to a per-note dominant line-ending style.
- **FR-016**: System MUST scope each occurrence to the position within a single line of a single note where it was found. A single pattern match MUST NOT span line boundaries within a note (parity with BI-037 FR-012) and MUST NOT span notes. In `regex` mode, `\n` metacharacters and any dot-matches-newline behaviour MUST NOT cause cross-line matching — patterns that require newline-spanning will simply not match. In `literal` mode, a pattern string containing an embedded newline MUST NOT match (no line carries an embedded newline). Multi-line refactors are expressed as N single-line replacements.
- **FR-017**: System MUST skip zero-width regex matches (positions where the pattern matches the empty string) — parity with the sibling `pattern_search` tool (BI-037 FR-016). Zero-width matches MUST NOT contribute to the occurrence count and MUST NOT trigger the safe-upper-bound guard.
- **FR-018**: Frontmatter content at the top of a note MUST be treated as prose for matching purposes — frontmatter is NOT a separately-skipped region. Dedicated frontmatter-key migration is a separate feature outside this scope.
- **FR-019**: System MUST accept an optional case-sensitivity control. Default behaviour MUST be case-sensitive matching. An explicit `case_insensitive: true` argument MUST cause the operation to ignore letter case during matching; in `regex` mode this MUST be equivalent to compiling the pattern with the `i` flag. The same flag MUST apply uniformly to both `literal` and `regex` modes. Parity with BI-037 FR-007.

### Key Entities

- **Find request**: The optional `vault?: string` field (absent = focused-vault default; present = resolved via the ADR-009 vault registry), the pattern, the pattern mode (`literal` or `regex`), the replacement text, the optional case-sensitivity control, the optional subfolder scope, the include-code-blocks and include-html-comments opt-ins, and the commit opt-in. Defines what the agent asked for.
- **Affected note**: A single note whose content contains at least one non-skipped occurrence of the pattern under the current scope and opt-in settings. Carries the note locator, the per-note occurrence count, and the proposed change.
- **Proposed change**: The shape of the rewrite for a single affected note — sufficient for the caller to verify what would be written if the commit opt-in were set.
- **Preview result**: The list of affected notes with their proposed changes, plus the total-occurrences count across all affected notes and a signal indicating no note on disk was modified.
- **Commit result**: The notes-changed count and the total-replacements count for an executed commit, plus the locators of the notes that were updated.
- **Safe upper bound**: The configured ceiling on the total occurrence count across the chosen scope above which the operation refuses to run. A property of the tool's configured environment rather than of any single call.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: An agent can preview and then commit a vault-wide find-and-replace in exactly two tool calls (one preview, one commit) without re-authoring the request or copying the preview shape into a different argument schema. 100% of preview-then-commit pairs against an unchanged vault produce the same set of affected notes and the same per-note proposed changes.
- **SC-002**: 100% of preview invocations leave every note on disk unchanged — verifiable by comparing per-note content and mtime before and after the preview call.
- **SC-003**: When code-block and HTML-comment skip-by-default is in effect, 0% of returned proposed changes include an occurrence that lies inside a fenced code block or inside an HTML comment.
- **SC-004**: When the include-code-blocks or include-html-comments opt-ins are set, 100% of occurrences inside the corresponding region are included in the proposed changes.
- **SC-005**: When a subfolder scope is supplied, 0% of affected notes in the response come from outside that subfolder subtree.
- **SC-006**: Unknown subfolder, invalid regex (in `regex` mode), safe-upper-bound-exceeded, commit-time occurrence-count drift, schema-level path-traversal rejection, canonical-level vault-escape rejection, unknown-vault, and registered-but-closed-vault are surfaced as eight distinct typed-error discriminators so an agent can branch on the error sub-code without parsing human-readable error prose. Drift, schema-level path-traversal, and (existing) unknown-subfolder surface under the existing top-level `VALIDATION_ERROR` with distinct `details.code` (path-traversal carries an additional `details.reason: "path-traversal"` per the ADR-015 sub-discriminator pattern). Canonical-level vault escape surfaces under the existing `PATH_ESCAPES_VAULT` top-level code. Unknown-vault and registered-but-closed-vault surface under the existing `CLI_REPORTED_ERROR` top-level code with `details.code: "VAULT_NOT_FOUND"` and `details.reason: "unknown"` vs `"not-open"` respectively (ADR-015 sub-discriminator; BI-026 v0.5.4 / BI-037 parity). All discriminators REUSE existing top-level codes — the eleven-tool zero-new-top-level-codes streak (Constitution Principle IV) is preserved across all eight discriminators.
- **SC-007**: A pattern that matches nothing returns a successful empty result with zero affected notes and zero replacements 100% of the time, never as an error.
- **SC-008**: When the safe upper bound is exceeded, 100% of invocations — preview or commit — refuse to run, with no note modified.

## Assumptions

- **Regex dialect**: The `regex` mode uses ECMAScript (Node's built-in `RegExp`) and ECMAScript replacement-string semantics, parity with the sibling `pattern_search` tool (BI-037). The user input excluded regex-dialect negotiation from scope, so the dialect is fixed.
- **Pattern mode default**: When the caller does not declare a pattern mode, the operation runs in `literal` mode. Literal-as-default is the safer choice — a caller who pastes `foo.bar` and forgets to declare regex mode does not accidentally match `fooXbar` and corrupt unrelated notes.
- **Safe upper bound source** *(locked by 2026-05-18 clarifications session)*: The upper bound is read from the process environment variable `OBSIDIAN_FIND_REPLACE_MAX_OCCURRENCES` at server start; default 500 when the variable is unset or invalid. The bound is server-side configured rather than a per-call argument. The bound is enforced uniformly across preview and commit per FR-011 + FR-012.
- **Per-note atomicity, not cross-note transactionality**: Commit writes are atomic per note (a note is fully written or unchanged), but the operation does not transactionally roll back already-written notes if a later note fails to write. Cross-note rollback is out of scope per the user input; the caller is responsible for taking a backup before committing.
- **Eligible files**: Only markdown notes (the same set scanned by sibling read/list tools in this surface) are eligible. Other files in the vault (images, attachments, plugin data) are not scanned or modified.
- **Single-vault per invocation** *(locked by 2026-05-18 clarifications session)*: One vault per invocation. Vault is identified via a single optional `vault?: string` field; absent ⇒ focused-vault default (vault-wide cohort parity with `find_by_property`, `properties`, `search`, `context_search`, `pattern_search`); present ⇒ resolved via the ADR-009 lazy vault registry. No `target_mode` axis, no `@active` sentinel — ADR-003's target-mode discipline applies to file-targeted tools and explicitly does not reach inherently-vault-wide surfaces per the project architecture. Cross-vault aggregation is out of scope per the user input.
- **Fenced code block detection**: A "fenced code block" is identified by paired triple-backtick (` ``` `) or paired triple-tilde (`~~~`) fences in standard CommonMark style. Inline code spans (single backticks) are not skipped — only fenced blocks are. Detection is text-level, not full-CommonMark-parser-level — sufficient for the safe default in well-formed notes.
- **HTML comment detection**: An "HTML comment" is identified by `<!--` … `-->` pairs in the note's text, including multi-line comments.
- **Subfolder semantics**: "Subfolder of the vault" means a path interpreted relative to the vault root, consistent with sibling tools in this surface that take a folder argument.
- **Ordering of affected notes** *(locked by 2026-05-18 clarifications session)*: Affected notes appear in ascending lexicographic order of vault-relative path; within each note, occurrences appear in ascending `line_number` then ascending byte-offset within a line. Deterministic across runs against the same vault content. Parity with sibling list/search tools (BI-019, BI-029, BI-033). See FR-004 + FR-005.
- **No retry semantics**: A failed commit (e.g., due to a transient file-system error mid-batch) is not automatically retried by the operation. The caller decides whether to re-issue.
- **No whole-word flag** *(locked by 2026-05-18 clarifications session)*: There is no whole-word opt-in in this BI. `literal` mode matches anywhere the pattern appears as a substring — `foo` matches inside `foobar`. Callers wanting word-boundary anchoring use `regex` mode with `\b` (parity with BI-037 pattern_search). A future feature can add whole-word if use cases force it.
