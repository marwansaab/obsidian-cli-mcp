# Implementation Plan: Open Vault File

**Branch**: `057-open-vault-file` | **Date**: 2026-05-29 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/057-open-vault-file/spec.md`

## Summary

Add a typed MCP tool `open_file` that surfaces an existing vault file — of any type Obsidian recognises (markdown note, canvas, PDF, image, or other attachment) — as the focused, visible file in the running Obsidian workspace, decoupling the hand-off affordance from `write_note`'s existing best-effort `open` parameter (ADR-009). The tool is **eval-composed**: a single bug-safe `obsidian eval` round-trip (the open argv carries only a vault-relative path + a boolean, far below the ~4 KB IPC ceiling of ADR-009) joins the eval-composed cohort alongside `backlinks`, `links`, `find_and_replace`, and `query_base` rather than the fs-direct write cohort. There is no native upstream `obsidian open` subcommand, so the tool name is descriptive (`open_file`) and ADR-010 is N/A.

The defining contract is the **active focused-vault guard** settled at Clarifications 2026-05-29: because upstream limitation B1 routes every `eval` to Obsidian's focused vault regardless of `vault=`, the handler pre-resolves the requested vault's base path via the vault registry (`resolveVaultPath`, ADR-009) and the single open-eval compares it against `app.vault.adapter.basePath` *before* resolving or opening the file. A mismatch fails loud with `CLI_REPORTED_ERROR` + `details.code: "VAULT_NOT_FOUND"` + `details.reason: "not-open"` (the `"not-open"` reason reused with a broadened semantic: not-the-focused-vault, closed OR background) and opens nothing — this is the cohort-precedented "no silent routing to focused vault" guard that `links`/`backlinks` already implement via their vault-resolution inspection clause. Folding the guard and the open into ONE eval (rather than a separate `FOCUSED_VAULT_TEMPLATE` eval + open eval) eliminates the focus-change TOCTOU window between guard and open.

Errors reuse existing top-level codes — **zero new top-level codes** (Constitution Principle IV streak preserved). `VAULT_NOT_FOUND` (`details.reason: "unknown" | "not-open"`) and `FILE_NOT_FOUND` are both reused unchanged from the eval-composed cohort (`backlinks`/`links` already carry both as `details.code` sub-discriminators under `CLI_REPORTED_ERROR`). One new single-state `details.code` is introduced: `UNSUPPORTED_FILE_TYPE` under `CLI_REPORTED_ERROR` (FR-009), surfaced via ADR-015 sub-discrimination with no `details.reason` enumeration. Input failures surface via `VALIDATION_ERROR` (the cohort's Zod-issue channel). The classifier stage order per ADR-014 is `VAULT_NOT_FOUND(unknown)` → `VAULT_NOT_FOUND(not-open)` → `FILE_NOT_FOUND` → `UNSUPPORTED_FILE_TYPE`, with the vault guard firing before file resolution (FR-012a). The success envelope echoes `{ opened, vault, new_tab }` for write-verification per the project's read-vs-write echo convention (opening mutates observable workspace state).

## Technical Context

**Language/Version**: TypeScript 5.x (strict mode, `tsc --noEmit` clean).
**Primary Dependencies**: `@modelcontextprotocol/sdk` (MCP transport), `zod` (boundary validation), `zod-to-json-schema` (MCP `inputSchema` publication via the centralised `registerTool` factory, ADR-006). No new runtime dependencies — the eval pipeline reuses the existing `invokeCli` / cli-adapter substrate (ADR-004) and the `vault-registry` (`resolveVaultPath`, ADR-009) unchanged.
**Storage**: Vault workspace state, indirectly, via one `obsidian eval` round-trip against Obsidian's `app.workspace` / `app.vault` API. The wrapper performs **no filesystem syscalls** — it does not `fs.readFile`/`fs.writeFile`/`fs.realpath`/`fs.stat`; Obsidian owns the file resolution and the view open. This is the eval-composed cohort posture (parity with `backlinks`/`links`/`find_and_replace`/`query_base`), distinct from the fs-direct write cohort (`write_note`/`append_note`/`patch_*`) that does touch `fs`.
**Testing**: `vitest` with `@vitest/coverage-v8`; tests co-located as `*.test.ts` alongside source modules per Principle II. Behaviour tests drive a mocked `invokeCli` returning canned `=> <json>` eval stdout for each stage (`ok` / `vault-not-focused` / `file-not-found` / `unsupported-type`); a T0 live-CLI probe against the authorised test vault (`.memory/test-execution-instructions.md`) confirms the real eval-result shapes at implement time.
**Target Platform**: Node.js >= 22.11 LTS; MCP server transport over stdio; cross-platform (Linux / macOS / Windows). The basePath comparison in the guard must be separator/case-normalised on Windows (research R5).
**Project Type**: MCP server typed-tool wrapper, eval-composed via the `invokeCli` substrate. Cohort precedents: `backlinks`, `links` (eval-composed link-graph reads with a vault-resolution guard), `find_and_replace`, `query_base` (eval-composed with `FOCUSED_VAULT_TEMPLATE`), and `write_note`'s existing `openLinkText` open eval (ADR-009).
**Performance Goals**: One `eval` spawn per call (the combined guard+resolve+open eval). Amortised: the first `open_file` (or any prior fs-direct/registry) call hydrates the vault registry via one `obsidian vaults verbose` probe (~150 ms, cached for the process lifetime per ADR-009); steady-state is a single ~50–150 ms Electron-startup-dominated eval spawn. No second spawn — the guard is folded into the open eval (research R2), improving on the "one extra round-trip" cost accepted at Clarifications 2026-05-29.
**Constraints**: 1000 UTF-16 code-unit cap on `path` / `file` locators (cohort parity); zero new top-level error codes (Principle IV); wikilink-form `file` locators MUST NOT contain `[[`/`]]` brackets (FR-004, schema-layer rejection, cohort parity with `append_note`'s `safeFileField`); structural-path-safety on `path`/`file` at the schema layer (no leading `/`, `\`, drive letter, `..` segments, control chars → `VALIDATION_ERROR`); no filesystem write, no vault switch (FR-010), no content edit (FR-020), no intra-file navigation (FR-021).
**Scale/Scope**: One new typed tool (`open_file`), one new source-tree module (`src/tools/open_file/`) with three production files (`index.ts` / `schema.ts` / `handler.ts`) plus an internal eval-template builder (a small pure helper akin to `backlinks`' `composeEvalCode`) and four co-located test files. One new single-state `details.code` value (`UNSUPPORTED_FILE_TYPE` under `CLI_REPORTED_ERROR`, FR-009). Reused unchanged: `VAULT_NOT_FOUND` (`unknown`/`not-open`), `FILE_NOT_FOUND`, `VALIDATION_ERROR`, `CLI_REPORTED_ERROR`, `CLI_BINARY_NOT_FOUND`/`CLI_NON_ZERO_EXIT`. No new top-level codes. One `_register-baseline.json` one-entry append.

The ADR-003 row is the one judgement call: it is **Y on intent** (explicit-vault-required, no implicit execution) while deliberately omitting the `target_mode` discriminator because the tool has a single mode. This is recorded in research R4 with the rejected single-value-literal alternative. It is not an `N` — ADR-003 governs *how intent is forced*, and the focused-vault guard forces it strictly.

## Graphify structural check

*Per the CLAUDE.md `/speckit-plan` rule: affected communities + kernel-node touch surface.*

- **Affected community**: `open_file`'s handler joins **community 2** — the runtime/error-spine + eval-composed-handler cluster that holds `invokeCli`, `UpstreamError`, and the sibling eval-composed handlers (`executeBacklinks` verified at degree 6 in community 2: `<-- invokeCli [calls]`, `<-- composeEvalCode [calls]`, `--> mapEnvelopeError [calls]`). `executeOpenFile` will present the same shape: `<-- invokeCli [calls]`, `<-- composeOpenEval [calls]`, `--> <error-classifier> [calls]`, `--> UpstreamError`. New-symbol placement in community 2 is the *expected* community, not a surprise — the post-implement structural-verification step (CLAUDE.md) should confirm this and flag if `executeOpenFile` lands elsewhere.
- **Kernel-node touch surface** (the validated kernel nodes: `createLogger`, `createQueue`, `UpstreamError`, `createServer`):
  - `UpstreamError` — **touched in the standard way**: imported and classified through, exactly as every handler does (the error-spine star). No new top-level error code is added to `src/errors.ts`, so no new error-class node appears outside the `errors.ts` community (post-implement check 1 will hold).
  - `createLogger` / `createQueue` — **NOT touched directly**. `executeOpenFile` receives `logger` + `queue` as injected `ExecuteDeps`; it does not construct either factory. `server.ts` remains the sole constructor (post-implement check 2 will hold — no production handler imports the boot-time DI factories).
  - `createServer` — **touched only via the standard boot-spine attach**: one `createOpenFileTool(deps)` line in `server.ts`'s registration block. No change to `createServer`'s own logic.
- **Explicit no-touch claim** (verified at post-implement): this BI adds **no new top-level error code**, constructs **neither** `createLogger` nor `createQueue` in any production file, and modifies **none** of the kernel nodes' internals — it attaches one new handler to the existing three spines (boot via `_register`/`server.ts`, runtime via `handler → invokeCli → eval`, error via `UpstreamError`). High-blast-radius surface: none beyond the one-line `server.ts` registration and the `_register-baseline.json` append.
