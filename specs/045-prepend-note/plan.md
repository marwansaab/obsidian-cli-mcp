# Implementation Plan: Prepend Note

**Branch**: `045-prepend-note` | **Date**: 2026-05-26 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/045-prepend-note/spec.md`

## Summary

Add a typed MCP tool `prepend` that writes the caller's `content` payload at the logical top of an existing vault note in a single call. The wrapper CLI-wraps the native upstream `obsidian prepend` subcommand (whose published parameter shape ‚Äî `file=<name>` / `path=<path>` / `content=<text>` / `inline` ‚Äî matches the spec's contract surface byte-for-byte) rather than re-implementing frontmatter detection in TS. This is a deliberate cohort divergence from BI-044's fs-direct pipeline pick: BI-045's defining contract (FR-005a frontmatter-aware insertion + FR-005b "defer detection to the underlying execution layer; no wrapper-side frontmatter parser") makes upstream's well-tested YAML-frontmatter detector the contract source, and re-implementing it in-tree would be both a literal FR-005b violation and an exposure-surface increase relative to the upstream pin. The cohort precedent for frontmatter-aware operations (`set_property`, `read_property`, `properties`, `find_by_property` ‚Äî all CLI-wrap) authorises the divergence; the content-write family (`write_note`, `append_note`, `patch_heading`, `patch_block` ‚Äî all fs-direct under ADR-009) was driven by the argv-pipe Windows-defect cap (BI-0038), which BI-045 mitigates by activating FR-018 with a concrete documented size ceiling instead. Active mode reuses the cohort's existing focused-file eval pattern (cohort parity with `set_property`, `append_note`); wikilink-form `file` locators resolve via a pre-flight `obsidian file file=<name>` TSV call (cohort parity with `set_property`, `append_note`) so the response can canonicalise the vault-relative path per FR-003. Errors surface through `UpstreamError` with the existing top-level error codes (`CLI_REPORTED_ERROR`, `VALIDATION_ERROR`, `ERR_NO_ACTIVE_FILE`); one new `details.code` value (`CONTENT_TOO_LARGE` under `VALIDATION_ERROR` for FR-018) surfaces via `details.code` sub-discrimination per ADR-015. `CONTENT_EMPTY` is reused unchanged from BI-044's `(VALIDATION_ERROR, CONTENT_EMPTY)` pair. `NOTE_NOT_FOUND` reuses the read-side cohort's `(CLI_REPORTED_ERROR, NOTE_NOT_FOUND)` discriminator. Constitution Principle IV streak preserved ‚Äî zero new top-level codes; the streak count becomes twenty after BI-045 ships.

## Technical Context

**Language/Version**: TypeScript 5.x (strict mode, `tsc --noEmit` clean).
**Primary Dependencies**: `@modelcontextprotocol/sdk` (MCP transport), `zod` (boundary validation), `zod-to-json-schema` (MCP `inputSchema` publication via the centralised `registerTool` factory). No new runtime dependencies ‚Äî the CLI-wrap pipeline reuses the existing `invokeCli` / `dispatchCli` infrastructure unchanged. No in-tree frontmatter parser (FR-005b ‚Äî deliberately not implemented).
**Storage**: Vault filesystem indirectly, via the upstream Obsidian Integrated CLI's `prepend` subcommand. The wrapper does NOT call `fs.readFile` / `fs.writeFile` / `fs.rename` / `fs.unlink` ‚Äî the upstream process owns the open/read/modify/write/rename pipeline. The wrapper DOES use two read-only filesystem syscalls: `fs.realpath` for the Layer 2 canonical-path safety check (cohort parity with the rest of the typed-tool family ‚Äî the same check applies regardless of write pipeline) and `fs.stat` to bracket the upstream prepend call for the `bytes_written` delta computation (tasks.md T009). Both calls are read-only file-info syscalls, not opens/reads/writes; the wrapper never touches file contents. This is a deliberate cohort divergence from BI-044 (and from the write-side family generally) on the WRITE pipeline ‚Äî rationale in `research.md` R1; the read-only fs touchpoints (realpath, stat) remain cohort-uniform with every other typed tool.
**Testing**: `vitest` with `@vitest/coverage-v8`; tests co-located as `*.test.ts` alongside source modules per Principle II.
**Target Platform**: Node.js >= 22.11 LTS; MCP server transport over stdio; cross-platform (Linux / macOS / Windows). The Windows command-line maximum (~32,767 chars per Microsoft's CreateProcess documentation) is the dominant constraint that makes FR-018 active in this BI ‚Äî see Constraints below.
**Project Type**: MCP server typed-tool wrapper using the project's `invokeCli` substrate. Cohort precedents: `set_property` (CLI-wrap to `property:set`), `read_property` (CLI-wrap to `property:read`), `properties` (CLI-wrap to `properties`), and the read-side family generally.
**Performance Goals**: One spawn per call for specific+path mode; two spawns per call for specific+file mode (pre-flight `obsidian file` resolver + `obsidian prepend`) and active mode (focused-file eval + `obsidian prepend`). The cohort's per-spawn cost is empirically ~50-150 ms dominated by Electron startup; total per-call budget ~100-300 ms for the dominant case. No additional substrate cost beyond `set_property`'s 2-spawn pattern.
**Constraints**: 1000 UTF-16 code-unit input cap on `file` / `path` locators (cohort parity with BI-033 / BI-038 / BI-039 / BI-040 / BI-043 / BI-044); **24 576 UTF-16 code-unit cap on `content` (FR-018 active ‚Äî see R3)** to keep the total argv envelope below the Windows ~32 767-char command-line maximum with safety headroom; zero new top-level error codes (Constitution Principle IV); no modifications to YAML frontmatter (FR-011 ‚Äî inherited from upstream's contract); preserve note's existing line-ending convention (FR-008 ‚Äî inherited from upstream's contract); no auto-create of missing target (FR-012, FR-025); `content` payload preserved byte-for-byte verbatim (FR-010a ‚Äî inherited from upstream's contract); wikilink-form locators MUST NOT contain `[[` or `]]` brackets (FR-001a ‚Äî schema-layer rejection, cohort parity with BI-044).
**Scale/Scope**: One new typed tool (`prepend`), one new source-tree module (`src/tools/prepend/`) with three production files (`index.ts` / `schema.ts` / `handler.ts`) and three co-located test files (no `prepend-edit.ts` helper ‚Äî the bytes are written by upstream, not by an in-tree pure helper). One new `details.code` value (`CONTENT_TOO_LARGE` under `VALIDATION_ERROR` for FR-018) ‚Äî single-state, ADR-015 sub-discriminator without `details.reason` enumeration. Three existing `details.code` / top-level-code reuses unchanged (`CONTENT_EMPTY` from BI-044's `(VALIDATION_ERROR, CONTENT_EMPTY)` pair; `NOTE_NOT_FOUND` from the read-side cohort; `ERR_NO_ACTIVE_FILE` for active mode with no focused file). No new top-level codes. The `file`-field bracket rejection (FR-001a) reuses the cohort's existing `safeFileField` / `safeWikilinkNameField` refinement pattern (BI-044 ships an inline `safeFileField` in `src/tools/append_note/schema.ts`; BI-045 may lift this helper to a shared location if a second consumer cohort emerges ‚Äî deferred to T0 evaluation, narrowed from "lift now" to "lift later when a second consumer appears" per BI-044's research.md R5).

**Structure Decision**: Single project, per-surface module at `src/tools/prepend/` per Principle I. The module follows the three-file shape (`{index, schema, handler}.ts` with NO helpers) ‚Äî simpler than BI-044's `append_note/` (which carries a private `append-edit.ts` byte-concatenation helper) because BI-045's bytes are written by the upstream process, not by an in-tree pure helper. Precedent: `set_property/`, `read_property/`, `delete/`, `rename/`, `move/`, `properties/`, `files/`, `read/`, and the rest of the CLI-wrap cohort ship the three-file shape. The new tool wires into the boot spine at `src/server.ts` via the standard one-line import + one-line factory call (cohort precedent ‚Äî 29 prior tools follow this pattern). The `_register-baseline.json` fixture gets a one-entry append. All existing infrastructure (`invokeCli` for the upstream subcommand calls, `vault-registry/`, `path-safety/`, `target-mode/`, `UpstreamError`) is consumed unchanged.
