# Feature Specification: Prepend Note

**Feature Branch**: `045-prepend-note`
**Created**: 2026-05-26
**Status**: Draft
**Input**: User description: "Add Prepend Note — a typed MCP tool that prepends content to the top of an existing vault note in a single call. The insertion is frontmatter-aware: when the note has a YAML frontmatter block, the new content lands immediately after the closing `---` rather than before it, leaving the frontmatter intact. By default a line break separates the prepended content from the existing body; an explicit opt-in joins it directly to the leading line. Targets either an explicit note (vault-relative path or wikilink-form name) or the note currently focused in the user's editor. Fails loud on missing target, invalid input, and oversized content. Out of scope: prepending BEFORE the frontmatter block (full-replace write surface owns that shape); auto-creating the target when it does not exist (the full-replace write surface owns creation); prepending at a specific heading or inside a specific section (the heading-patch surface owns sub-section writes); mid-file insertion, search-and-replace, partial rewrites; heuristic behaviour for malformed frontmatter blocks (the contract follows whatever the underlying execution layer detects); atomicity guarantees beyond what the underlying execution layer already provides; the internal mechanism used to write the bytes (argv pipe vs filesystem-direct vs any future channel — the contract is observable behaviour and a documented size ceiling, not the choice of pipeline)."

## Clarifications

### Session 2026-05-26

No new clarifications were required for BI-045. Every design choice that was clarified in the sibling BI-044 (`append_note`) settles here by cohort parity, and the user's input statement explicitly settles the choices unique to BI-045 (frontmatter-aware insertion point as the defining contract, malformed-frontmatter handling deferred to upstream, default separator placed between the prepended content and the existing body). The cohort-parity inheritances and the specifically-settled choices are recorded inline in the FR rationale and the Assumptions section.

The four BI-044 clarifications inherited verbatim into this spec:

- **Default-separator interaction with an adjacent existing newline** → "the existing newline IS the separator" (FR-006a). For append, the relevant adjacency was the existing file's trailing newline; for prepend, the symmetric adjacency is the supplied content's trailing newline (the byte that sits next to the existing body's leading line). When the supplied content ends with `\n` or `\r\n`, the wrapper does NOT insert an additional separator — the content's trailing newline IS the separator. Mirrors BI-044 Session 2026-05-25 Q1.
- **Wikilink-form bracket handling** → strict bare-name (FR-001a). Locators containing `[[` or `]]` are rejected at the input-validation boundary; the wrapper does NOT strip brackets. Cohort parity with every existing `file`-parameter tool. Mirrors BI-044 Session 2026-05-25 Q2.
- **Active-mode explicit-opt-in requirement** → no opt-in required (FR-004a). Active-mode prepend needs only `target_mode: "active"` plus `content` (subject to FR-014). Prepend is additive — wrong-target produces recoverable additive noise at the top of an unintended note, not total prior-content destruction; the asymmetric safety profile justifies the asymmetric opt-in posture. Mirrors BI-044 Session 2026-05-25 Q3 (same additive-safety argument applies symmetrically to prepend; the additive noise lands at the top instead of the bottom but is equally recoverable).
- **Content payload's own trailing newline handling** → verbatim, byte-for-byte (FR-010a). The wrapper does NOT strip, trim, normalise, or auto-append trailing newlines. Mirrors BI-044 Session 2026-05-25 Q4. The interaction with FR-006a (above) is symmetric to the BI-044 case: a content payload ending in `\n` triggers FR-006a in the prepend direction (content's trailing newline IS the separator); a content payload without a trailing `\n` triggers the wrapper-inserted separator per FR-006.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Prepend to a named note with frontmatter-aware placement and default separator semantics (Priority: P1)

A vault author identifies a note by either its vault-relative path or its wikilink-form name (the bare note name, the same shape Obsidian resolves inside `[[…]]` links) and supplies content to prepend. The wrapper writes the content at the LOGICAL top of the named note — which means: when the note has a YAML frontmatter block, the new content lands immediately after the closing `---` (the frontmatter is preserved byte-for-byte); when the note has no frontmatter block, the new content lands at byte zero. By default a line break separates the prepended content from the existing body so that the new content reads as its own block above the body and does not run on into the existing leading line. The wrapper preserves every byte of the prior content other than the bytes it inserts as separator. The response identifies the file by its vault-relative path so the caller can attribute the change or roll it back.

**Why this priority**: The whole feature exists to give authors a single-call prepend surface anchored to an existing note. Without P1 there is no MVP. Frontmatter-aware placement is not an optional refinement — it is the defining contract of this surface, because a naive byte-zero prepend would destroy the YAML metadata that is a near-universal convention in this vault. Bundling "vault-relative path locator", "wikilink-form name locator", "frontmatter-aware insertion point", and "default-separator behaviour" into a single P1 slice is deliberate: any subset of them would either ship a broken surface (insertion before frontmatter destroys metadata; no default separator collides the new content with the existing leading line; one locator shape but not the other is asymmetric with the cohort) or ship no usable surface at all. P1 stands alone as a viable shipped slice: an author who can name a note (by either locator shape) and prepend a TL;DR, a status line, or a header block at the logical top — with frontmatter preserved and a readable separator before the existing body — has every authoring shape this feature aims to deliver at MVP, plus the response payload needed to track or undo the change.

**Independent Test**: Stand up a vault with two notes — one with a YAML frontmatter block (opening `---`, body, closing `---`) followed by a non-empty body, and one with no frontmatter block (the body starts at byte zero). Issue the operation against the frontmatter note using its vault-relative path and against the second note using its wikilink-form name. For each call, verify that (a) the prepended content is present at the logical top of the file (after the closing `---` for the frontmatter note; at byte zero for the no-frontmatter note), (b) a line break sits between the prepended content and the existing leading body line, (c) every byte of the prior content above and below the insertion point is byte-stable from the pre-call state (frontmatter bytes byte-for-byte preserved for the frontmatter note), and (d) the response payload names the vault-relative path of the file that was written (the resolved canonical path, not the wikilink-form input the caller supplied for the second note).

**Acceptance Scenarios**:

1. **Given** an existing note whose first lines form a valid YAML frontmatter block (opening `---`, body, closing `---`) followed by a non-empty body, **When** the caller invokes the tool with the note's vault-relative path and content under the default-separator behaviour, **Then** the new content appears immediately after the closing `---` (no separator inserted between the closing `---` and the new content — the prepended content sits flush against the existing line break that terminates the closing `---`), a single line-ending-convention-matching separator sits between the new content and the existing first body line, the YAML block above the insertion point is preserved byte-for-byte, the existing body content below the insertion point is preserved byte-for-byte, and the response identifies the file by its vault-relative path.
2. **Given** an existing note with no YAML frontmatter block (the body starts at byte zero) and a non-empty leading line, **When** the caller invokes the tool with the note's vault-relative path and content under the default-separator behaviour, **Then** the new content appears at byte zero of the file, a single line-ending-convention-matching separator sits between the new content and the existing leading line, the existing content below the insertion point is preserved byte-for-byte, and the response identifies the file by its vault-relative path.
3. **Given** the supplied content ends with a line break (`\n` or `\r\n`), **When** the caller invokes the tool under the default-separator behaviour against a note with a non-empty existing body, **Then** the content's trailing line break IS the separator — the wrapper does NOT insert an additional separator, no blank line of separation appears between the prepended content and the existing leading body line (per FR-006a — symmetric to BI-044's FR-006a "existing trailing newline IS the separator" rule).
4. **Given** an existing note identified by its wikilink-form name (the bare note name, no `[[…]]` brackets — the same shape the cohort's read-side tools accept under the `file` locator parameter), **When** the caller invokes the tool with that name and content, **Then** the wrapper resolves the name to the matching vault note, the prepend lands on that note under the same frontmatter-aware rule (FR-005a), and the response identifies the file by its vault-relative path (the resolved canonical path, not the wikilink-form input the caller supplied).
5. **Given** the target is a 0-byte file, **When** the caller prepends content with default settings, **Then** no trailing separator is inserted and the file contains exactly the prepended content (per FR-009 — symmetric to the BI-044 0-byte-file rule; no existing body to separate against).
6. **Given** a successful prepend, **When** the call completes, **Then** the response identifies the patched note's vault-relative location so the caller can attribute the change or roll it back.

### User Story 2 - Distinguish unresolvable-target and input failures through typed errors (Priority: P2)

A vault author who supplies a note locator that does not correspond to an existing note, an empty content payload, both a path and a wikilink-form name simultaneously, no locator at all when one is required, a locator while intending to target the focused note, an unknown extra input field, or content that exceeds the supported size ceiling, receives a typed, structured error identifying which failure mode occurred. No file is created. No file is modified. The caller can switch on the failure mode directly — no prose parsing.

**Why this priority**: P1 gives the happy path. Without P2, an author who mistypes a path, ships an empty content payload, or attempts to send a chunk above the size ceiling has no programmatic way to learn which problem they hit other than reading the inputs and the vault state side-by-side. Worse, an undetected missing-target could be misread as a silent success — a future caller could re-read the would-have-been-prepended location and find stale content. P2 is the diagnostic contract that makes failure self-describing — "note not found", "validation error" (with sub-discriminators for the specific input violation), and "content too large" are programmatically distinguishable states. With P1+P2 the caller has both the operation and the feedback loop needed to use it reliably. The size-ceiling failure mode is non-negotiable for the same reason it is non-negotiable in the sibling append surface: the underlying execution layer has a hard limit (whether argv-pipe length, in-memory buffer size, or any successor channel) and a clear typed error preventing silent corruption, hangs, or opaque failures is a contract obligation.

**Independent Test**: With P1 in place, issue the operation naming a note path or wikilink-form name that does not correspond to any existing note in the vault and verify the response is a typed error programmatically distinguishable as "note not found" and that no note is created. Issue the operation with each of the following inputs in turn — empty content payload, both `path` and the wikilink-form locator supplied at the same time, neither locator supplied while explicit-target mode is requested, a locator supplied while focused-note mode is requested, an unknown extra input field, a wikilink-form name supplied with literal `[[…]]` brackets — and verify each is rejected at the input-validation boundary with a typed error naming the violation, before any filesystem access or subprocess invocation. Issue the operation with content that exceeds the supported size ceiling (when an explicit ceiling is enforced — see FR-018 deferral) and verify the response is a typed error naming the size limit and that the file on disk is byte-identical to its pre-call state.

**Acceptance Scenarios**:

1. **Given** a note locator that does not correspond to any existing note in the vault, **When** the caller invokes the tool, **Then** the operation fails with a typed error programmatically identifying the failure as "note not found", the error payload names the missing locator, and no note is created.
2. **Given** an empty content payload, **When** the caller invokes the tool, **Then** the request is rejected at the input-validation boundary before any filesystem access or subprocess invocation, the typed error names the offending field and the reason, and no file is modified.
3. **Given** both an explicit path locator and a wikilink-form name locator supplied in the same call, **When** the caller invokes the tool, **Then** the request is rejected at the input-validation boundary, the typed error names the mutually-exclusive-locators violation, and no file is modified.
4. **Given** explicit-target mode is requested but no locator (neither vault-relative path nor wikilink-form name) is supplied, **When** the caller invokes the tool, **Then** the request is rejected at the input-validation boundary, the typed error names the missing-locator violation, and no file is modified.
5. **Given** focused-note mode is requested AND a locator is supplied, **When** the caller invokes the tool, **Then** the request is rejected at the input-validation boundary, the typed error names the locator-not-allowed-in-active-mode violation, and no file is modified.
6. **Given** an unknown extra input field is present in the call, **When** the caller invokes the tool, **Then** the request is rejected at the input-validation boundary, the typed error names the unknown field, and no file is modified. (Strict-mode schema, cohort parity with `write_note` and `append_note`.)
7. **Given** a wikilink-form name locator supplied with the literal `[[…]]` brackets (e.g. `[[My Note]]`), **When** the caller invokes the tool, **Then** the request is rejected at the input-validation boundary, the typed error names the offending bracket characters and the expected bare-name shape, and no filesystem access or subprocess invocation occurs. (Cohort parity with every existing `file`-parameter tool — none strip brackets; inherited from BI-044 FR-001a.)
8. **Given** an explicit content-size ceiling exists on the wrapper AND a caller exceeds it, **When** the caller invokes the tool, **Then** the operation fails with a typed validation error naming the size limit, before any filesystem access or subprocess invocation, and the file on disk is byte-identical to its pre-call state. (See FR-018; the trigger condition is unsatisfiable in BI-045 because the wrapper inherits BI-044's no-explicit-cap discipline — the acceptance check is conditionally satisfied when and if a future BI introduces a concrete cap.)
9. **Given** a content payload of any size accepted by the substrate, **When** the caller invokes the tool, **Then** the operation succeeds end-to-end (subject to other failure modes). With no explicit wrapper-imposed cap, every in-substrate payload is accepted.
10. **Given** a caller consults the tool's documentation, **When** the caller reads it, **Then** the supported size ceiling is stated explicitly so the caller can predict the limit ahead of time without trial-and-error. In BI-045 the docs state "substrate-bounded; no wrapper-imposed cap" — cohort parity with BI-044, satisfying SC-007's contract-and-implementation match without committing to a concrete number prematurely.

### User Story 3 - Inline (no-separator) opt-in for fusing onto the existing leading line (Priority: P2)

A vault author who has a note whose existing leading body line begins with content that the new prepend should flow directly into (a continuation token, a sentence the prepend completes a prefix for, or a list bullet whose marker should fuse with new prefix text) supplies content with an explicit "inline" opt-in. The wrapper fuses the new content directly onto the leading line with no intervening line break. Every byte of the prior content other than the insertion point's immediate neighbourhood remains unchanged. Frontmatter handling is unchanged by the inline opt-in: when frontmatter is present, the prepended content still lands after the closing `---`; the inline opt-in only suppresses the wrapper's default-inserted separator between the prepended content and the existing leading body line.

**Why this priority**: P1's default-separator behaviour serves the dominant authoring intent (new line / block per prepend — TL;DR above body, status block above body, header above body). The inline-fuse intent is the legitimate minority case — extending an existing leading sentence, prefixing a leading list-item marker, completing a partial prefix that the existing body's leading line continues. Without P3, that authoring intent has no surface — the only workaround is to read the leading line, splice the new content client-side, and ship the full result through the full-replace write surface, which is exactly the wastage the feature exists to eliminate. P3 is also a contract symmetry obligation: the upstream CLI exposes the equivalent flag, and the wrapper's job is to publish that surface in a typed, validated, structured-error form rather than gate-keep it. P3 is at the same priority tier as P2 because both layer onto P1 directly and neither depends on the other; the caller can use P1 alone, P1+P2 (diagnostics), P1+P3 (inline opt-in), or P1+P2+P3 (the full slice). Cohort parity with BI-044's inline opt-in (US3).

**Independent Test**: Stand up a vault with a note whose existing leading body line starts with the text `Existing`. Issue the operation against that note with content `Lead` and the inline opt-in. Verify that (a) the resulting leading body line reads `LeadExisting` with no intervening line break, (b) no byte of the prior content other than the leading line's prefix is changed, (c) when the note has a YAML frontmatter block, the YAML block is preserved byte-for-byte and the inline-fused content lands after the closing `---` (not before — the inline opt-in does not suppress the frontmatter-aware insertion-point rule of FR-005a), and (d) the response identifies the file by its vault-relative path.

**Acceptance Scenarios**:

1. **Given** an existing note whose existing leading body line begins with the text `Existing`, **When** the caller invokes the tool with content `Lead` and the inline opt-in enabled, **Then** the resulting leading body line reads `LeadExisting` with no intervening line break and every other byte of the file is unchanged from its pre-call state.
2. **Given** an existing note whose first lines form a valid YAML frontmatter block followed by a non-empty body whose leading line begins with the text `Existing`, **When** the caller invokes the tool with content `Lead` and the inline opt-in enabled, **Then** the YAML block above the closing `---` is preserved byte-for-byte, the resulting first body line reads `LeadExisting` with no intervening line break, and every other byte of the file is unchanged from its pre-call state.
3. **Given** a successful inline prepend, **When** the call completes, **Then** the response identifies the patched note's vault-relative location, identical in shape to the default-separator success response.

### User Story 4 - Operate on the focused note (Priority: P3)

A vault author working with a note focused in their editor prepends content to that note without restating its location. When no note is focused, the operation fails with a typed error directing the author to focus a note or to supply an explicit locator. No filesystem access occurs and no write happens in the no-focused-note failure case. Frontmatter handling, default-separator behaviour, and inline opt-in semantics are identical to the explicit-locator cases — the focused-note locator is a convenience surface, not a separate insertion-rule contract.

**Why this priority**: P1 already accepts an explicit note locator (vault-relative path or wikilink-form name), so the feature is shippable without P4. P4 layers a convenience surface on top: an author who is already working in a note does not need to restate its path or name on every prepend. This is cohort parity with the active-mode surface across the rest of the typed-tool family — every typed tool that accepts a note locator also accepts a focused-note variant via the cohort's `target_mode: "active"` discriminator (ADR-003, ADR-004). P4 stands independently — it can be added after P1+P2+P3 without disturbing them, and P1 callers who always pass an explicit locator are unaffected.

**Independent Test**: With P1+P2+P3 in place, focus a known note in the user's editor, issue the operation using the focused-note locator (no explicit vault/path/wikilink-form name supplied), and verify the prepend lands on the focused note under the same frontmatter-aware rule as the explicit-locator case. Clear the focused-note state and issue the same request, and verify the operation fails with a typed error directing the author to focus a note or to supply an explicit locator — and that no filesystem access occurred.

**Acceptance Scenarios**:

1. **Given** a note is focused in the author's editor, **When** the caller submits a focused-note prepend request with `target_mode: "active"` plus content and no further opt-in flag (per FR-004a — prepend is a deliberate cohort exception to `write_note`'s mandatory `overwrite: true` in active mode, inherited from BI-044), **Then** the prepend lands on the focused note under the same frontmatter-aware rule as when the note is named explicitly, and the response identifies the file by its vault-relative path (resolved at request time from the focused-note state).
2. **Given** no note is focused, **When** the caller submits a focused-note prepend request, **Then** the operation fails with the existing active-mode cohort error directing the caller to focus a note or to supply an explicit locator; no filesystem access is performed and no write occurs.

### Edge Cases

- **Target file does not exist (any locator shape — path, wikilink-form name, or focused-note resolution)**: per FR-016, the operation fails with `code: "CLI_REPORTED_ERROR"` + `details.code: "NOTE_NOT_FOUND"` and the file is not auto-created. Auto-create is the full-replace write surface's responsibility (cohort parity — `write_note` owns creation; `prepend_note` assumes existence, mirrors `append_note` FR-012). This is the deliberate scope split per the user's Out of scope statement.
- **Target is a 0-byte file (default separator)**: per FR-009, no trailing separator is inserted; the file becomes exactly the prepended content. This is the only case in which the default-separator behaviour does not insert a separator between the prepended content and the (non-existent) existing body. Cohort parity with BI-044's 0-byte handling.
- **Target has a YAML frontmatter block (opening `---`, body, closing `---`) followed by a non-empty body**: per FR-005a, the prepended content lands immediately after the closing `---`. The frontmatter block above the insertion point is preserved byte-for-byte. The existing body below the insertion point is preserved byte-for-byte (apart from any wrapper-inserted separator under FR-006).
- **Target has a YAML frontmatter block followed by an EMPTY body (frontmatter only, no body content after the closing `---`)**: per FR-005a, the prepended content lands immediately after the closing `---` and becomes the entire body. Under the default-separator behaviour, no trailing separator is inserted (FR-009 applies symmetrically — there is no existing body to separate against). The frontmatter is preserved byte-for-byte.
- **Target has no YAML frontmatter block at all**: per FR-005a, the prepended content lands at byte zero. The existing content below the insertion point is preserved byte-for-byte (apart from any wrapper-inserted separator under FR-006).
- **Malformed frontmatter (missing closing `---`, or leading `---` used as a horizontal rule rather than a frontmatter delimiter)**: per the user's explicit out-of-scope statement and FR-005b, the wrapper inherits whatever the underlying execution layer detects. The wrapper does NOT introduce its own frontmatter parser, does NOT publish a separate typed error for malformed-frontmatter, and does NOT promise byte-exact placement in this case. Callers needing byte-exact placement against malformed-frontmatter notes use the full-replace write surface.
- **Target's existing body's leading line ends with a line break already vs. starts with content immediately**: irrelevant to the prepend semantics. The default separator (FR-006) is inserted between the prepended content and the existing leading line — what the existing leading line itself contains does not change the wrapper's behaviour. (The symmetric question for prepend is "does the SUPPLIED CONTENT end with a newline" — see FR-006a, the prepend-direction analogue of BI-044's FR-006a.)
- **Note's line-ending convention is CRLF**: the wrapper preserves the note's existing line-ending convention. When the default separator inserts a new line break, the bytes of that separator match the note's existing convention (LF or CRLF). The wrapper does not invent a new convention or convert between conventions. (Frontmatter delimiters are preserved with whatever line-ending convention the existing block uses.)
- **Inline opt-in against a 0-byte file**: accepted — the file becomes exactly the prepended content (same observable result as the default-separator behaviour against a 0-byte file). Inline-vs-default only differs when there is existing body content to fuse against.
- **Inline opt-in against a file with a YAML frontmatter block followed by an EMPTY body**: same observable result as default-separator behaviour in this case (no existing body line to fuse with; the prepended content lands after the closing `---` and becomes the entire body). The inline opt-in only suppresses the wrapper's default-inserted separator between the prepended content and the existing leading body line; when no existing leading body line exists, there is no separator for the opt-in to suppress.
- **Content payload contains line breaks**: accepted as-is. Whether the resulting markdown is structurally valid (e.g. a paragraph-shaped prepend followed by an existing list-item leading line) is the caller's responsibility. The wrapper does not invent shape-specific structural validation. Cohort parity with BI-044.
- **Content payload itself ends with a line break (`\n` or `\r\n`)**: per FR-010a, preserved verbatim — and per FR-006a, the content's trailing newline IS the default separator (no additional separator inserted). Result for a file with existing body `Existing` prepended with content `Lead\n` under the default-separator behaviour: `Lead\nExisting`. Result for a file with existing body `Existing` prepended with content `Lead` (no trailing newline) under the default-separator behaviour: `Lead\nExisting` (wrapper inserts the separator per FR-006). Both produce the same observable result; the difference is whether the separator came from the caller's bytes (FR-006a) or the wrapper's insertion (FR-006).
- **Content payload starts with a line break (`\n` or `\r\n`)**: preserved verbatim per FR-010a. The result has a leading blank line above the prepended content's first non-newline byte — and, when frontmatter is present, that blank line sits between the closing `---` and the prepended content's first non-newline byte. The wrapper does NOT trim leading newlines from the content payload. Callers who do not want a leading blank line do not include one in their content.
- **Wikilink-form name resolves to multiple notes in the vault** (ambiguous shortname when no folder path is supplied): the wrapper inherits the underlying execution layer's resolution rule (cohort parity with `read`, `delete`, `rename`, `append_note`, and the rest of the read-side and write-side tools that accept the `file` locator parameter). The wrapper does not introduce a new ambiguous-name typed error; if the underlying layer raises one, it surfaces unchanged.
- **Wikilink-form name resolves to no note**: per FR-016, surfaces as `NOTE_NOT_FOUND`. Same shape as the vault-relative-path-not-found case — one typed error, not two. Cohort parity with BI-044.
- **Locator passed with the literal `[[…]]` brackets (the full wikilink syntactic shape)**: per FR-001a (inherited verbatim from BI-044 Clarifications Session 2026-05-25), rejected at the input-validation boundary with a typed validation error naming the offending bracket characters and the expected bare-name shape. The wrapper does NOT silently strip brackets. No filesystem access or subprocess invocation occurs.
- **Active mode with `file`/`path` supplied (i.e. `target_mode: "active"` plus a locator)**: per FR-014, rejected at the input-validation boundary with a typed validation error naming the conflict. Cohort parity with `write_note`, `append_note`, and the rest of the active-mode-supporting cohort.
- **Unknown extra input fields**: per FR-015, the input schema is strict; unknown fields are rejected at the input-validation boundary with a typed error naming the offending field. Cohort parity with `write_note`, `append_note`, `patch_heading`, `patch_block`, and the strict-mode-by-default discipline.
- **External editor has the target open with unsaved changes**: when the underlying execution layer signals the condition to the wrapper, the operation fails loud per the cohort `EXTERNAL_EDITOR_CONFLICT` discriminator (`details.reason` ∈ `unsaved-changes` / `file-locked`). When the layer does not signal the condition, the prepend lands and the editor sees a refreshed file on next focus. This detection-capability caveat is inherited from the cohort (BI-040 `patch_heading` FR-021, BI-043 `patch_block` FR-021, BI-044 `append_note` FR-022); the spec does not invent new detection mechanics.

## Requirements *(mandatory)*

### Functional Requirements

#### Locator and addressing

- **FR-001**: System MUST accept exactly one of three target shapes per call: an explicit vault-relative path locator, an explicit wikilink-form name locator (the bare note name — same shape the cohort's read-side tools accept under their `file` parameter), or a focused-note locator referencing the note currently open in the author's editor. The vault-relative path locator and the wikilink-form name locator are mutually exclusive — the caller supplies one or the other, not both. The focused-note locator is mutually exclusive with both explicit locator shapes.
- **FR-001a**: System MUST reject wikilink-form name locators that contain the `[[` or `]]` bracket characters at the input-validation boundary, before any filesystem access or subprocess invocation, with a typed validation error naming the offending characters and the expected shape (bare note name without brackets). The wrapper does NOT silently strip brackets or otherwise normalise the locator — cohort parity with every existing `file`-parameter tool. Inherited verbatim from BI-044 Clarifications Session 2026-05-25; rationale and rejected alternatives are recorded in BI-044 Clarifications.
- **FR-002**: System MUST resolve the wikilink-form name locator using the cohort's existing name-resolution rule (the same rule the read-side `file` parameter follows). The wrapper does not introduce a new resolution algorithm.
- **FR-003**: System MUST canonicalise the response's reported file location to the vault-relative path of the file that was actually written, regardless of which locator shape the caller supplied. A caller who supplies a wikilink-form name receives the resolved vault-relative path in the response, not the wikilink-form name they supplied. Cohort parity with BI-044 FR-003.
- **FR-004**: When the focused-note locator is requested but no note is focused, System MUST fail with the existing active-mode cohort error (whichever top-level `code` the active-mode ADR specifies for "no focused note"; this BI introduces no new sub-discriminator for this condition). No filesystem access MUST occur and no write MUST be performed. Cohort parity with BI-044 FR-004.
- **FR-004a**: System MUST NOT require an explicit opt-in flag (e.g. `confirmActive: true` or equivalent) for active-mode prepend requests. A valid active-mode call needs only `target_mode: "active"` plus a non-empty `content` payload (subject to the locator-mutually-exclusivity rule of FR-014). This is a deliberate cohort exception to `write_note`'s mandatory `overwrite: true` in active mode and is inherited from BI-044's FR-004a — prepend is additive (wrong-target = recoverable additive noise at the TOP of an unintended note) rather than destructive (wrong-target = total prior-content destruction), and the asymmetric safety profile justifies the asymmetric opt-in posture. The symmetric argument applies regardless of insertion direction (top vs bottom): additive noise is recoverable whether it lands at the top or the bottom of the unintended note.

#### Prepend placement and separator semantics

- **FR-005**: System MUST support exactly one prepend operation — write the supplied content at the LOGICAL top of the target file. The operation has no after-heading, no after-frontmatter-of-arbitrary-depth, no mid-file placement, no append, and no search-and-replace modes; callers needing those shapes use the full-replace write surface, the heading-patch surface, the append surface (BI-044), or another existing typed surface per the published scope split.
- **FR-005a** *(defining contract — frontmatter-aware insertion point)*: System MUST place the prepended content's first byte immediately after the closing `---` of the target file's YAML frontmatter block when one is present, and at byte zero of the target file when no frontmatter block is present. "Frontmatter block is present" is whatever the underlying execution layer detects as a YAML frontmatter block; the wrapper does not introduce its own parser. The frontmatter block above the insertion point — the opening `---`, the YAML body, the closing `---`, and the line break terminating the closing `---` — MUST be preserved byte-for-byte. The existing body content below the insertion point (when any exists) MUST be preserved byte-for-byte apart from any wrapper-inserted separator under FR-006.
- **FR-005b**: System MUST defer the detection of YAML frontmatter to the underlying execution layer and MUST NOT introduce a wrapper-side frontmatter parser. When the frontmatter is malformed (missing closing `---`, leading `---` interpreted as a horizontal rule rather than a delimiter, or any other detection edge case), the wrapper inherits whatever placement the underlying layer produces; no wrapper-side malformed-frontmatter typed error is published. Callers needing byte-exact placement against malformed-frontmatter notes use the full-replace write surface. Settled by the user's explicit out-of-scope statement.
- **FR-006**: System MUST default the separator behaviour to "a line break separates the prepended content from the existing body" — that is, when the target file has existing body content below the insertion point determined by FR-005a, a line break sits between the prepended content and the existing leading body line in the resulting file.
- **FR-006a**: When the supplied content's last bytes already form a line break (`\n` or `\r\n`), System MUST NOT insert an additional separator under the default-separator behaviour — the content's trailing line break IS the separator. The wrapper inserts a separator under the default-separator behaviour ONLY when the supplied content does not already end with a line break. This is the prepend-direction symmetric of BI-044 FR-006a (which addressed the existing-file's-trailing-newline adjacency); the prepend symmetric addresses the supplied-content's-trailing-newline adjacency. Same rationale: byte-stable against the dominant Obsidian shape, avoids inflating files by one byte per prepend in steady-state authoring, avoids visually-distracting blank lines of separation between every prepended chunk and the existing body.
- **FR-007**: System MUST accept an explicit "inline" opt-in input field. When the opt-in is enabled, the wrapper does NOT insert a separator between the prepended content and the existing leading body line — the prepended content fuses directly onto the existing leading body line. The opt-in is opt-in only — its default value is the default-separator behaviour of FR-006. The opt-in does NOT change the frontmatter-aware insertion-point rule of FR-005a (the prepended content still lands after the closing `---` when frontmatter is present); it only suppresses the wrapper's default-inserted separator.
- **FR-008**: System MUST preserve the note's existing line-ending convention (LF, CRLF, or other). Any line break the wrapper inserts under the default-separator behaviour MUST match the note's existing convention. The wrapper MUST NOT convert between line-ending conventions. The frontmatter block's existing line-ending convention is preserved as part of FR-005a's byte-for-byte requirement.
- **FR-009**: When the target file is 0 bytes (and therefore has no frontmatter and no existing body), System MUST NOT insert a trailing separator under the default-separator behaviour. The resulting file contains exactly the prepended content. The same rule applies when the target file consists of a frontmatter block followed by an empty body — no separator is inserted because there is no existing leading body line to separate against; the prepended content lands after the closing `---` and becomes the entire body.
- **FR-010**: System MUST preserve every byte of the prior content other than the bytes the wrapper inserts as separator under FR-006. The prepend operation MUST NOT modify any byte of the file other than (a) the bytes the wrapper inserts as separator (under the default-separator behaviour) and (b) the bytes of the prepended content itself.
- **FR-010a**: System MUST write the caller's `content` payload byte-for-byte verbatim — no trimming, no normalisation, no auto-appended trailing newline, no leading-newline stripping, no character-set conversion beyond what the transport layer already requires. A `content` payload supplied with a trailing `\n` results in a file whose new leading body section ends with `\n` (and FR-006a applies — that `\n` IS the default separator); a `content` payload supplied without a trailing `\n` results in a file whose new leading body section does NOT end with `\n` (and FR-006 inserts the default separator). A `content` payload supplied with a leading `\n` results in a file whose new leading body section begins with `\n` — under a frontmatter-present case, the resulting bytes are `---\n\n[rest of content]`, i.e. a blank line between the closing `---` and the content's first non-newline byte. Cohort parity with BI-044 FR-010a and `write_note`'s `content` handling.
- **FR-011** *(frontmatter preservation — restatement of the defining contract)*: System MUST NOT modify any byte of the YAML frontmatter block. The frontmatter block above the insertion point determined by FR-005a — including the opening `---`, the YAML body, the closing `---`, and the line break terminating the closing `---` — is preserved byte-for-byte across every prepend operation, default-separator or inline-opt-in alike. Frontmatter writes are handled by the frontmatter-write capability (e.g. `set_property`); prepend's job is to leave the frontmatter alone and place the new content below it.
- **FR-012**: System MUST NOT auto-create the target file when it does not exist. File creation is handled by the full-replace write surface per the published scope split. Cohort parity with BI-044 FR-012.

#### Input validation

- **FR-013**: System MUST reject an empty content payload at the input-validation boundary, before any filesystem access or subprocess invocation, with a typed validation error naming the offending field and the reason. Cohort parity with BI-044 FR-013.
- **FR-014**: System MUST reject any input that supplies both an explicit locator (vault-relative path or wikilink-form name) AND a focused-note request in the same call, OR supplies neither locator while explicit-target mode is requested, OR supplies both locator shapes (path AND wikilink-form name) at the same time. The locator shapes per FR-001 are mutually exclusive; mixing them surfaces as a typed validation error at the input-validation boundary. Cohort parity with BI-044 FR-014.
- **FR-015**: System MUST reject unknown extra input fields at the input-validation boundary with a typed validation error naming the offending field. The input schema is strict; cohort parity with `write_note`, `append_note`, `patch_heading`, `patch_block`, and the strict-mode-by-default discipline.
- **FR-016**: When the resolved target does not correspond to an existing note in the vault (whether the locator was a vault-relative path, a wikilink-form name, or a focused-note resolution that pointed to a path the vault no longer carries), System MUST fail with `code: "CLI_REPORTED_ERROR"` + `details.code: "NOTE_NOT_FOUND"` (cohort parity with the existing read-side `NOTE_NOT_FOUND` discriminator and with BI-044 FR-016). No note MUST be created.

#### Size ceiling and oversized content

- **FR-017**: System MUST publish a maximum supported content-payload size — the "size ceiling" — in the tool's documentation. The ceiling MAY be expressed as a concrete number OR as a substrate-bound description (e.g. "bounded by available memory and the filesystem's max-file-size limit"); either form is acceptable as long as callers can predict the limit ahead of time. The plan-phase pipeline decision (cohort parity with BI-044 research.md R1 — fs-direct via the ADR-009 substrate) removes the argv-pipe motivation for an aggressive cap, and the wrapper publishes no explicit ceiling in BI-045 (cohort parity with BI-044's no-cap discipline); the docs state the substrate-bound rule explicitly.
- **FR-018** *(deferred to a future BI — same trigger condition as BI-044 FR-018)*: When the wrapper imposes an explicit ceiling on the content payload AND a caller exceeds it, System MUST fail with a typed validation error naming the size ceiling, before any filesystem access or subprocess invocation, and the file on disk MUST be byte-identical to its pre-call state. The trigger condition is unsatisfiable in BI-045 because the wrapper imposes no explicit cap (cohort parity with BI-044); the implementation path (`z.string().max(N)` schema refinement + `details.code: "CONTENT_TOO_LARGE"` under `VALIDATION_ERROR`) is named for the future BI that surfaces a concrete need. Cohort discipline: this BI does NOT prematurely introduce a cap whose value would be speculative.
- **FR-019** *(trivially satisfied while FR-018 is deferred)*: When the content payload is at or below the (substrate-bound) ceiling, System MUST complete the prepend operation end-to-end (subject to the other failure modes specified in FR-004, FR-013-FR-016, FR-022). With no explicit wrapper-imposed cap, every in-substrate content payload satisfies this requirement; the cohort's existing fs-direct happy-path tests provide the verifiable evidence template.

#### Response shape

- **FR-020**: On a successful prepend, System MUST return a response identifying the vault-relative location of the file that was written. The response shape supports caller attribution and rollback. Cohort parity: write-side tools echo the locator for write-verification (per the project's read-vs-write echo convention; same shape as BI-044 FR-020).

#### Atomicity and external-editor conflict

- **FR-021**: A single invocation MUST NOT leave the note on disk in a half-written or otherwise inconsistent state at any observable instant within the wrapper's control. The wrapper inherits whatever single-invocation atomicity the underlying execution layer provides; the spec does not invent new atomicity guarantees. Interrupted invocations (power loss, kill -9) leave on-disk state to the operating system. Cohort parity with BI-044 FR-021.
- **FR-022**: When the target note is open in an external editor with unsaved changes AND the underlying execution layer signals the condition to the wrapper, System MUST fail loud with `code: "CLI_REPORTED_ERROR"` + `details.code: "EXTERNAL_EDITOR_CONFLICT"` + `details.reason` set to `"unsaved-changes"` or `"file-locked"` matching the OS-level signal; the file on disk MUST NOT be modified. When the underlying layer does not signal the condition, the edit lands and the editor sees a refreshed file on next focus — this detection-capability caveat is inherited from the cohort (BI-040 FR-021, BI-043 FR-021, BI-044 FR-022) and is documented in the tool's help text.
- **FR-023**: For every failure mode in FR-004, FR-013-FR-016, FR-018, and FR-022, System MUST NOT modify the file on disk. "Failed loud" means the typed error is the only side effect visible to the caller and the filesystem.

#### Out of scope at the contract boundary

- **FR-024**: System MUST NOT accept any placement mode other than the logical-top prepend defined in FR-005 / FR-005a. After-heading, after-frontmatter-with-an-arbitrary-skip, BEFORE-frontmatter, mid-file insertion, append, search-and-replace, and partial rewrites are out of scope; callers needing those shapes use the heading-patch surface, the append surface (BI-044), the full-replace write surface, or another existing typed surface per the published scope split.
- **FR-025**: System MUST NOT auto-create the target file. File creation is handled by the full-replace write surface per the published scope split (FR-012).
- **FR-026**: System MUST NOT publish any transactional contract across concurrent invocations beyond the single-invocation atomicity of FR-021. The published cross-invocation contract is last-write-wins; callers needing stronger guarantees coordinate externally.
- **FR-027**: System MUST NOT publish any guarantee about the internal mechanism used to write the bytes (argv-pipe length, filesystem-direct read-modify-write, or any future channel). The published contract is the observable behaviour (FR-001 through FR-023) plus the documented size ceiling (FR-017). The plan phase chooses the pipeline; the spec describes what the caller sees. Cohort parity with BI-044 FR-027.

### Key Entities

- **Note**: A markdown file inside the vault, addressed by vault + vault-relative path, by wikilink-form name (the bare note name, resolved by the same rule as the cohort's read-side `file` parameter), or by the focused-note locator. May optionally begin with a YAML frontmatter block (opening `---`, body, closing `---`) followed by a body composed of paragraphs, lists, tables, fenced code, callouts, and other markdown constructs. Has a line-ending convention (LF / CRLF) and a trailing-newline convention. The prepend operation preserves the frontmatter (when present), the line-ending convention, and every byte of the existing body other than the bytes the wrapper inserts as separator.
- **YAML frontmatter block**: An optional block at the top of the note delimited by an opening `---` line and a closing `---` line, used by Obsidian to store note metadata. When present, defines the boundary between "above the prepend's insertion point" (preserved byte-for-byte) and "the rest of the body" (the prepend lands at the start of this region). Detection is delegated to the underlying execution layer (FR-005b).
- **Prepend content**: The string of bytes the caller supplies for the wrapper to write at the logical top of the target file (after the closing `---` when frontmatter is present, at byte zero otherwise). Must be non-empty (FR-013). May contain line breaks (caller responsibility for resulting markdown shape). May begin or end with line breaks; the wrapper preserves both verbatim (FR-010a); the trailing-newline byte interacts with the default-separator rule per FR-006a.
- **Default separator**: A line break inserted by the wrapper between the prepended content and the existing leading body line, present by default (FR-006) and matching the note's existing line-ending convention (FR-008). Absent when the file is 0 bytes (FR-009), when the existing body is empty (frontmatter-only note — FR-009 symmetric), when the supplied content already ends with a line break (FR-006a — the content's trailing newline IS the separator), or when the inline opt-in (FR-007) is enabled.
- **Inline opt-in**: A boolean input field whose default value preserves the default-separator behaviour and whose enabled value suppresses the wrapper-inserted separator between the prepended content and the existing leading body line. Does NOT change the frontmatter-aware insertion-point rule of FR-005a.
- **Size ceiling**: A documented maximum supported content-payload size, expressed as a concrete number OR as a substrate-bound description (FR-017). Published in the tool's help text. In BI-045 the wrapper imposes no explicit cap (cohort parity with BI-044); a future BI may introduce one via the deferred FR-018 path.
- **Focused note**: The note the user currently has open in their editor. Resolved at request time; failure to resolve (no note focused) surfaces a typed error per FR-004.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: A vault author can prepend content to any existing note in the vault — addressed by vault-relative path, by wikilink-form name, or by the focused-note locator — with a single call, without first reading the note, computing a splice, or shipping any byte of the prior content back to the wrapper.
- **SC-002**: For every prepend operation that succeeds against a note with a YAML frontmatter block, every byte of the frontmatter block (opening `---`, YAML body, closing `---`, terminating line break) is unchanged from the pre-call state. The wrapper-side guarantee is byte-for-byte; this is the defining contract of the prepend surface (FR-005a, FR-011).
- **SC-003**: For every prepend operation that succeeds, every byte of the existing body content below the insertion point (every other line, every other character, the line-ending convention) is unchanged from the pre-call state. Only the bytes the wrapper inserts as separator (under the default-separator behaviour) plus the bytes of the prepended content itself are added.
- **SC-004**: Every failure mode (note not found, empty content, mutually-exclusive locators, missing locator in explicit mode, locator supplied in active mode, unknown extra field, content exceeds size ceiling, no focused note in active mode, external-editor conflict) surfaces as a typed, programmatically distinguishable error. No failure mode produces a silently-empty success, a silently-wrong write, or a half-written file.
- **SC-005**: Repeat invocations of the same request against an unchanged vault produce identical outcomes (same write, same response) — the operation is deterministic given a fixed input and a fixed vault state.
- **SC-006**: An author's payload size for a single prepend is bounded by the size of the prepended content plus the locator metadata — *not* by the size of the surrounding note. A 50-byte prepend against a 50,000-byte note moves 50 bytes plus the locator, not 50,050 bytes round-trip. This is the wastage the feature exists to eliminate (cohort parity with BI-044 SC-005).
- **SC-007**: The response payload identifies the prepended file's vault-relative location, enabling caller attribution and rollback without re-reading the vault.
- **SC-008**: The size ceiling stated in the tool's published documentation matches the value the wrapper actually enforces — callers who size their payloads against the documented number do not encounter unexpected oversize failures, and callers who exceed the documented number receive the typed oversize error before any filesystem access. (Cohort parity with BI-044 SC-007; in BI-045 the documented contract is "substrate-bounded; no wrapper-imposed cap" and the implementation matches.)

- **Error code cohort parity**: Typed errors surface under the existing top-level error codes per Constitution Principle IV — zero new top-level codes; Principle IV streak preserved. `NOTE_NOT_FOUND` (FR-016) parallels the read-side and `append_note` `NOTE_NOT_FOUND` discriminator. `EXTERNAL_EDITOR_CONFLICT` (FR-022) reuses BI-040's discriminator unchanged with the same `details.reason` enum (cohort parity with BI-043 and BI-044). - **Default-separator interaction with the content's trailing newline**: Settled by FR-006a as the prepend-direction symmetric of BI-044's FR-006a — when the supplied content ends with a line break, the wrapper does NOT insert an additional separator; the content's trailing line break IS the separator. Always-insert was rejected (by inheritance) because it would produce a visually-distracting blank line of separation between every prepended chunk and the existing body in the common case. Defer-to-upstream was rejected (by inheritance) because the cohort discipline is to publish a settled byte-level rule at the wrapper boundary. The wrapper does NOT silently strip brackets. Cohort parity with every existing `file`-parameter tool.
- **Empty content rejection**: Settled at FR-013 — empty content payloads are validation errors. Rationale: prepending zero bytes is a no-op masquerading as an operation; surfacing it as a typed validation error gives the caller a clear signal that no work was requested and prevents the wrapper from invoking the underlying execution layer for nothing. Cohort parity with BI-044.
- **Markdown shape responsibility**: The wrapper does not invent shape-specific structural validation on the content payload — callers may prepend line breaks, partial blocks, headers, paragraphs, or any other shape, and whether the resulting markdown is structurally valid is the caller's responsibility. Cohort parity with the rest of the write-side tools' "shape is the caller's responsibility" discipline.
- **Cross-invocation concurrency**: Per the user's explicit out-of-scope statement, the wrapper publishes no transactional contract across concurrent invocations beyond the single-invocation atomicity of FR-021. The cross-invocation behaviour is last-write-wins; callers needing stronger guarantees coordinate externally. This is the deliberate scope contract, not a defect to be patched later. Cohort parity with BI-044.
- **Pipeline-agnostic contract**: Per FR-027 and the user's explicit out-of-scope statement, the spec does not commit the wrapper to a specific write mechanism (argv pipe, filesystem-direct, any future channel). The plan phase chooses the pipeline based on the size-ceiling trade-off, the cohort substrate, and the constitution's in-tree-bias rule for dependencies. Cohort parity with BI-044.
- **Insertion point relative to the closing `---`'s terminating line break**: FR-005a places the prepended content's first byte immediately after the closing `---`'s terminating line break — i.e., on its own fresh line below the closing `---`, not appended to the same line as the closing `---`. This is the natural reading of "lands after the closing `---`" and matches Obsidian-side expectations. The plan phase verifies the upstream subcommand's behaviour matches this contract; any divergence is patched at the wrapper.
