**Organisation**: Tasks are grouped by user story. Setup + Foundational (Phase 1 + 2) build the shared Zod schema and the module scaffold. **Cohort divergences from sibling BI-044 (append_note)** worth noting upfront:

- **Pipeline pick is CLI-wrap, NOT fs-direct** (research.md R1). This is the FIRST BI in the project's history to choose CLI-wrap for a content-carrying write tool. The wrapper does NOT touch the filesystem directly; the upstream `obsidian prepend` subcommand owns the entire write. Driven by FR-005b's hard MUST NOT on a wrapper-side frontmatter parser + the property-cohort precedent for frontmatter-aware ops (set_property / read_property / properties / find_by_property).
- **Tool name `prepend`, not `prepend_note`** — ADR-010 APPLIES (not N/A), because the wrapper wraps the upstream `obsidian prepend` subcommand. Cohort naming asymmetry with BI-044's `append_note` is deliberate and cohort-discipline-consistent.
- **Three-file module shape (no helper file)** — `index.ts` + `schema.ts` + `handler.ts` + co-located tests. NO `prepend-edit.ts` because the upstream owns the byte-level work (frontmatter detection, insertion-point selection, separator decision, line-ending preservation, atomic write). Cohort precedent: `set_property/`, `read_property/`, `delete/`, `rename/`, `move/`, `properties/`, `files/`, `read/`.
- **One new `details.code` (`CONTENT_TOO_LARGE` under `VALIDATION_ERROR`, single state, no `details.reason`)** — sibling of BI-044's `CONTENT_EMPTY` which is reused unchanged. Bracket rejection (FR-001a) surfaces via the cohort's standard Zod-issue-path channel — no new sub-code introduced.
- **FR-018 IS ACTIVE in BI-045** (not deferred like BI-044). The schema layer enforces a 24576 UTF-16 code-unit cap on `content`; oversized payloads surface as the new `CONTENT_TOO_LARGE` sub-discriminator before any spawn occurs. Rationale: research.md R3.
- **No fs.* error classification** — the wrapper does not call fs.readFile / fs.writeFile / fs.rename. Filesystem-level upstream failures (disk full, read-only filesystem, permission denied, ENOENT, EBUSY) surface from upstream's stderr inspection instead of from Node.js errno. EXTERNAL_EDITOR_CONFLICT classification is driven by upstream's stderr pattern matching, not fs.rename errno.
- **No `FS_WRITE_FAILED` code in the error roster** — replaced by generic `CLI_REPORTED_ERROR` with `details.stdout` + `details.stderr` for unrecognised upstream failures (cohort parity with `set_property`'s unrecognised-error path).
- **US ordering reflects the spec's four-story shape**: US1 (P1 named-locator + frontmatter-aware placement + default separator + response shape), US2 (P2 typed failures), US3 (P2 inline opt-in), US4 (P3 active mode). US2 and US3 share the P2 tier — independent layers on top of US1.
- **Active mode requires NO explicit opt-in flag** (FR-004a, inherited verbatim from BI-044 — deliberate cohort exception to `write_note`'s mandatory `overwrite: true`).
- **EXTERNAL_EDITOR_CONFLICT classification (FR-022) folded into US1's handler task** rather than split into a separate phase — cohort precedent from BI-043 T014 + BI-044 T011.
