# Feature Specification: Rename Note Typed MCP Tool

**Feature Branch**: `021-rename-note`
**Created**: 2026-05-12
**Status**: Draft
**Input**: User description: "Add Rename Note — A typed MCP tool that renames a vault note in place. Honours the vault's 'Automatically update internal links' setting; the auto-update behaviour is vault-config-dependent."

## Background *(non-mandatory context)*

This feature adds the **ninth** typed-tool wrap on top of the foundation completed by features 003–010 and extended by 011 / 012 / 013 / 014 / 015 / 016 / 017 / 018 / 019 / 020. Where the prior eight typed surfaces (`read_note`, `write_note`, `delete_note`, `read_property`, `find_by_property`, `read_heading`, `write_property`, `list_files`) wrap CLI subcommands for reading, creating/overwriting, deleting, frontmatter-property access, heading-body reads, and folder listing, `rename_note` wraps the **in-place rename** subcommand of the Obsidian CLI. It composes on the same load-bearing primitives:

- **[004-target-mode-schema](../004-target-mode-schema/spec.md) / [010-flatten-target-mode](../010-flatten-target-mode/spec.md)** — the shared `target_mode` schema primitive at [src/target-mode/target-mode.ts](../../src/target-mode/target-mode.ts). Post-010 the canonical reuse idiom is `applyTargetModeRefinement(targetModeBaseSchema.extend({ <tool-specific fields> }))` (the Pattern (a) flat-extension form). Per Constitution Principle III, this primitive owns the `target_mode` discriminator, the `vault`/`file`/`path` keys, the specific-branch XOR locator rule, the active-branch forbidden-key rule, and the strict-mode posture (`additionalProperties: false`); `rename_note` MUST compose on top of it rather than re-state any of those rules inline.
- **[003-cli-adapter](../003-cli-adapter/spec.md)** — the centralised CLI adapter at [src/cli-adapter/cli-adapter.ts](../../src/cli-adapter/cli-adapter.ts). The handler routes every CLI invocation through `invokeCli(...)` (or its bounded counterpart through the shared queue). The adapter owns argv assembly, vault hoisting, active-mode locator stripping, and the four-priority error classification (`CLI_BINARY_NOT_FOUND`, `CLI_NON_ZERO_EXIT`, `ERR_NO_ACTIVE_FILE`, `CLI_REPORTED_ERROR`). The 011-write-note unknown-vault response-inspection clause and the 017-cross-platform-support binary-resolver are inherited unchanged.
- **[005-help-tool](../005-help-tool/spec.md)** — the `help` tool and the schema-stripping utility at [src/help/strip-schema.ts](../../src/help/strip-schema.ts). The registry-consistency test at [src/server.test.ts](../../src/server.test.ts) gates the existence of `docs/tools/rename_note.md`; this BI authors that file as a non-stub Markdown body.
- **[012-delete-note](../012-delete-note/spec.md) / [018-write-property](../018-write-property/spec.md) / [019-list-files](../019-list-files/spec.md)** — the empirical validators of the typed-tool pattern for write-side surfaces. `rename_note` follows the same module layout (`schema.ts` / `handler.ts` / `index.ts` + co-located tests under `src/tools/rename_note/`), the same `RegisterDeps` shape (`logger`, `queue`, optional `deps` pass-through), and the same handler-thinness ceiling (a tight argv-shape + result-mapping body; no per-call logger events at the tool layer per the 011-R1 reconciliation). The factory name is `createRenameNoteTool` per the existing precedent.

The user-facing contract this feature delivers: an MCP client calls `rename_note({ target_mode: "specific", vault: "MyVault", path: "Inbox/Old.md", name: "New" })` and receives `{ renamed: true, fromPath: "Inbox/Old.md", toPath: "Inbox/New.md" }` — the file at `Inbox/Old.md` is now at `Inbox/New.md` on disk, and any wikilinks pointing at the old name have been rewritten (subject to the vault's "Automatically update internal links" setting). Locator XOR, mode-forbidden-key, empty-name, and strict-mode (`additionalProperties: false`) rejections all happen at the zod parse boundary; CLI failures (binary missing, non-zero exit, in-band `Error:` reply, no active file, source-not-found, destination-exists) flow through `UpstreamError`. No new error codes are introduced by this BI.

The feature scope is **in-place rename only** — the file's directory does not change. Renaming-with-move (changing both filename and folder) is the responsibility of a future `move_note` tool referenced in this spec's out-of-scope section. The two concerns split cleanly: rename is a name-on-disk operation with link-rewriting; move is a path-on-disk operation with link-rewriting and folder-creation semantics. Conflating them would force every caller to think about both even when they only need one.

Link-rewriting is **vault-config-dependent**: when the vault's "Automatically update internal links" setting is enabled (Settings → Files & Links → "Automatically update internal links"), the underlying Obsidian rename surface rewrites every wikilink pointing at the old name to the new name; when the setting is disabled, the rename happens but stale links remain. The typed surface does NOT enforce the setting — that would require config mutation, which is out of scope. The surface documents the dependency prominently so callers don't expect unconditional link-rewriting.

## Clarifications

### Session 2026-05-12

- Q: Extension-handling rule — when `name` lacks an extension, what does the typed surface do? And what counts as "having an extension"? → A: **`.md`-only allowlist, case-sensitive byte match** (mirroring the 020-fix-write-gaps R2 lock). Rule: if `name.endsWith(".md")` (literal, case-sensitive), forward verbatim. Otherwise, append `.md`. Internal periods are preserved (e.g., `Doc.v1.draft` → `Doc.v1.draft.md`, NOT `.draft`-treated-as-extension). The allowlist is exactly `{".md"}` — NOT widened to `.canvas` / `.pdf` / image types. **`rename_note` is scoped to `.md` notes** per the tool's name; **non-`.md` filename targets (renaming `.canvas` files, attachments, cross-extension type conversion like `.md → .canvas`) are out of scope** — callers route those through `obsidian_exec` with the CLI's rename subcommand. Rationale: a broader allowlist would invent a vault-filetype taxonomy absent from any ADR; source-introspection would force per-call CLI lookups inconsistent with the wrapper-side determinism that 020 locked.
- Q: Folder-separator-rejection rule — may `name` contain `/` or `\`? → A: **Validation-layer reject**. The schema validates `name` against `z.string().min(1).regex(/^[^/\\]+$/)` (or equivalent). Inputs containing `/` or `\` fail at the zod parse boundary as `VALIDATION_ERROR` whose `details.issues[].path` includes `"name"` and whose message names the rule with the `move_note` recovery hint (e.g., "name must not contain folder separators; use move_note to relocate the file to a different folder"). Preserves the in-place-rename scope cleanly vs the future `move_note` surface; produces an actionable error message at the boundary rather than relying on platform-dependent CLI / OS behaviour. Rejected at the schema layer means no CLI call is made for inputs that would partially-rename-and-move on Windows or error on POSIX.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Specific-mode in-place rename via path locator with extension preservation (Priority: P1)

An MCP client (typically an LLM agent) needs to rename a note at a known vault-relative path to a new name, expecting the `.md` extension to carry over without being repeated. The agent calls `rename_note({ target_mode: "specific", vault: "MyVault", path: "Inbox/Typo.md", name: "Fixed" })` and receives `{ renamed: true, fromPath: "Inbox/Typo.md", toPath: "Inbox/Fixed.md" }`. The file at `Inbox/Typo.md` is no longer there; `Inbox/Fixed.md` exists with the same body content as before. With the vault's auto-update-links setting enabled, every wikilink that previously read `[[Inbox/Typo]]` or `[[Typo]]` now reads `[[Inbox/Fixed]]` or `[[Fixed]]`.

**Why this priority**: This is the headline value of the feature. The typo-fix / naming-normalisation flow is the most common rename driver, and extension preservation is the default ergonomic — callers almost always want the rename to stay within the same file type. Without this story, the typed-tool track has no rename primitive and every rename either goes through the freeform escape hatch (which inherits none of the per-mode validation, structured errors, or link-rewriting guarantees) or — worse — through a raw filesystem rename which breaks every wikilink across the vault.

**Independent Test**: With a stub `spawnFn` injected via `deps`, calling `rename_note({ target_mode: "specific", vault: "MyVault", path: "Inbox/Typo.md", name: "Fixed" })` against a stub child that exits `0` and emits a CLI-shaped success response (canonical from/to paths on stdout, or substantively equivalent — exact wording captured during plan-stage CLI characterisation per FR-019) results in (a) the spawn being invoked with binary `"obsidian"` and an argv whose first non-binary token is the rename subcommand and which contains `vault=MyVault`, `path=Inbox/Typo.md`, and `name=Fixed` (exact argv-key names captured during plan-stage research per FR-019), and (b) the handler returning `{ renamed: true, fromPath: "Inbox/Typo.md", toPath: "Inbox/Fixed.md" }`.

**Acceptance Scenarios**:

1. **Given** the `rename_note` tool is registered and a stub adapter that exits `0` reporting a successful in-place rename, **When** an MCP client calls `rename_note({ target_mode: "specific", vault: "MyVault", path: "Inbox/Typo.md", name: "Fixed" })`, **Then** the tool returns a successful MCP response whose JSON-serialised payload is `{ renamed: true, fromPath: "Inbox/Typo.md", toPath: "Inbox/Fixed.md" }` AND the adapter was invoked with argv that includes `vault=MyVault`, the source locator (`path=Inbox/Typo.md` or the CLI argv-key rename per FR-019), and the destination name token (`name=Fixed` or the CLI argv-key rename per FR-019) AND the argv does NOT contain a `file=` token (locator XOR enforced at the schema layer means only `path` was provided).
2. **Given** the same tool, **When** an MCP client calls `rename_note` with `name: "Renamed"` against a source ending in `.md`, **Then** the resulting `toPath` ends in `.md` exactly once (no double-extension `Renamed.md.md`). Per the /speckit-clarify Q1 resolution: `name.endsWith(".md")` (literal, case-sensitive) forwards verbatim; otherwise the typed surface appends `.md`. So `name: "Renamed"` → argv token `Renamed.md`; `name: "Renamed.md"` → argv token `Renamed.md` (verbatim, no double-append).

### User Story 2 — Specific-mode rename via wikilink locator (`file=`) (Priority: P1)

When the agent has a wikilink-form name for the source — typically because that's how Obsidian linked to it, or because the user referred to it by name — it uses `file` instead of `path`. The agent calls `rename_note({ target_mode: "specific", vault: "MyVault", file: "QuickNote", name: "Quick Note" })` and the CLI resolves the wikilink to a concrete on-disk location. The handler reports `{ renamed: true, fromPath: "<CLI-resolved canonical source path>", toPath: "<resolved destination path in the same folder>" }`. The destination's folder is identical to the source's folder (in-place rename); only the filename changes.

**Why this priority**: P1 alongside Story 1. The two locators target the same CLI subcommand but resolve different ambiguity classes — `file=` is wikilink-resolved by Obsidian, `path=` is exact. Both are part of the typed surface that retires `obsidian_exec` for renames; downgrading either would imply the surface is incomplete.

**Independent Test**: With a stub adapter that exits `0` and reports `Inbox/QuickNote.md` as the canonical source and `Inbox/Quick Note.md` as the canonical destination, calling `rename_note({ target_mode: "specific", vault: "MyVault", file: "QuickNote", name: "Quick Note" })` results in the spawn being invoked with argv that includes `vault=MyVault`, the wikilink-form source locator (in whichever argv-token shape plan-stage research per FR-019 confirms), and the destination-name token, and the handler returning `{ renamed: true, fromPath: "Inbox/QuickNote.md", toPath: "Inbox/Quick Note.md" }`.

**Acceptance Scenarios**:

1. **Given** the `rename_note` tool and a stub adapter reporting `Inbox/QuickNote.md` and `Inbox/Quick Note.md` as the canonical from/to paths, **When** an MCP client calls `rename_note({ target_mode: "specific", vault: "MyVault", file: "QuickNote", name: "Quick Note" })`, **Then** the tool returns `{ renamed: true, fromPath: "Inbox/QuickNote.md", toPath: "Inbox/Quick Note.md" }` AND the adapter was invoked with argv that includes `vault=MyVault` and the resolved wikilink-form source locator (exact CLI argv key captured during plan-stage research per FR-019; user-facing field stays `file` for cross-tool uniformity).
2. **Given** the `rename_note` tool, **When** an MCP client calls it with `file` containing UTF-8 multi-byte characters (e.g., `file: "笔记", name: "日记"`), **Then** the adapter is invoked with that exact byte sequence in both the source-locator and destination-name tokens AND the call succeeds. `rename_note` does not transcode, normalise, or escape the locator or the name.

### User Story 3 — Internal periods in `name` preserved; non-`.md` trailing segments are NOT treated as extensions (Priority: P2)

When the agent supplies a `name` containing internal periods that look like extension suffixes but are NOT `.md` (e.g., `Doc.v1.draft`, `version.1.2.3`, or a deliberate `.txt` rename request), the typed surface treats the trailing segment as part of the filename stem and appends `.md`. The rule is literal `name.endsWith(".md")` — case-sensitive, byte-equality — mirroring the 020-fix-write-gaps R2 lock. The agent calls `rename_note({ target_mode: "specific", vault: "MyVault", path: "Drafts/Sketch.md", name: "Doc.v1.draft" })` and receives `{ renamed: true, fromPath: "Drafts/Sketch.md", toPath: "Drafts/Doc.v1.draft.md" }`. The destination is a `.md` file whose stem happens to contain dots; internal periods are preserved verbatim, not interpreted as extension delimiters.

**Why this priority**: P2 (was P1 in the pre-clarify draft; demoted by /speckit-clarify Q1's narrowing). The story still matters — agents that work with versioned filenames or supply `name` values containing dots need to know exactly what happens — but it's no longer the load-bearing "explicit-extension verbatim use" story it was before. The pre-clarify Q1 candidate (broad allowlist) would have made this a P1 contract about cross-type conversion; the actual /speckit-clarify Q1 lock narrows the wrapper to `.md`-only and routes cross-type renames to `obsidian_exec`. The remaining value of Story 3 is the **"no surprises from internal periods"** corollary: callers can pass any UTF-8 stem with dots and the wrapper does exactly one thing — append `.md` unless the stem already ends with `.md`.

**Independent Test**: With a stub adapter that exits `0` and reports a CLI-shaped success response for `Drafts/Sketch.md` → `Drafts/Doc.v1.draft.md`, calling `rename_note({ target_mode: "specific", vault: "MyVault", path: "Drafts/Sketch.md", name: "Doc.v1.draft" })` results in the spawn being invoked with argv that includes a destination-name token whose value is exactly `Doc.v1.draft.md` (`.md` appended; `.draft` is NOT in the allowlist) AND the handler returns `{ renamed: true, fromPath: "Drafts/Sketch.md", toPath: "Drafts/Doc.v1.draft.md" }`.

**Acceptance Scenarios**:

1. **Given** the `rename_note` tool and a stub adapter reporting a successful rename of `Drafts/Sketch.md` to `Drafts/Doc.v1.draft.md`, **When** an MCP client calls `rename_note({ target_mode: "specific", vault: "MyVault", path: "Drafts/Sketch.md", name: "Doc.v1.draft" })`, **Then** the tool returns `{ renamed: true, fromPath: "Drafts/Sketch.md", toPath: "Drafts/Doc.v1.draft.md" }` AND the adapter's argv name-token's value is exactly `Doc.v1.draft.md` (`.md` appended because `Doc.v1.draft.endsWith(".md") === false`).
2. **Given** the same tool, **When** an MCP client calls `rename_note` with `name: "Renamed.MD"` (uppercase `.MD`) against a source ending in `.md`, **Then** the typed surface appends `.md` because the allowlist match is **case-sensitive byte equality** per the /speckit-clarify Q1 resolution — the argv name-token's value is `Renamed.MD.md`. Callers that want literal case-sensitive `.md` extension recognition pass `name: "Renamed.md"` (all lowercase) verbatim. (Note: on case-insensitive filesystems like Windows NTFS-default or macOS APFS-default, the on-disk result may still collide with an existing `Renamed.md` per the case-only-rename Edge Case; that's a separate concern from the wrapper's allowlist behaviour.)
3. **Given** the `rename_note` tool, **When** an MCP client calls `rename_note({ target_mode: "specific", vault: "MyVault", path: "Drafts/Sketch.md", name: "Sketch.canvas" })` (the cross-extension type-conversion case that the pre-clarify draft treated as a P1 contract), **Then** the typed surface produces `toPath: "Drafts/Sketch.canvas.md"` — `.canvas` is NOT in the allowlist, so `.md` is appended. Callers who genuinely want a `.md → .canvas` type conversion MUST route through `obsidian_exec` with the CLI's rename subcommand (per the /speckit-clarify Q1 scope narrowing). `docs/tools/rename_note.md` documents this routing explicitly.

### User Story 4 — Source-not-found and destination-name-collision fail with structured errors (Priority: P1)

When the source path does not exist on disk, or when a file with the requested destination name already exists in the same folder, the CLI surfaces a structured failure that the handler propagates as an `UpstreamError`. The agent calls `rename_note({ target_mode: "specific", vault: "MyVault", path: "Inbox/Missing.md", name: "Renamed" })` against an empty vault and receives an MCP error response with `code: "CLI_REPORTED_ERROR"` (or `CLI_NON_ZERO_EXIT` depending on the CLI's exit-code convention — captured during plan-stage research per FR-019) whose `details.message` carries the verbatim CLI wording (likely `Error: file not found at Inbox/Missing.md` or similar). Similarly, calling `rename_note({ target_mode: "specific", vault: "MyVault", path: "Inbox/A.md", name: "B" })` when `Inbox/B.md` already exists fails with a structured error whose message identifies the destination collision.

**Why this priority**: P1. These are the two most common rename failure modes and they MUST surface as actionable structured errors — not as silent no-ops, not as opaque exit codes. The agent's recovery path depends on distinguishing "the source is wrong, retry with the right locator" from "the destination is taken, pick a different name" from "something else went wrong, surface the error to the user." Per the user input's [P1] AC #4 and AC #5.

**Independent Test**: Two handler tests, each driving the adapter through an injected stub. (a) stub child exits non-zero with stderr `"Error: file not found at Inbox/Missing.md\n"` OR exits `0` with stdout `"Error: file not found at Inbox/Missing.md\n"` (whichever shape the live CLI produces per FR-019) → handler returns the appropriate `CLI_NON_ZERO_EXIT` or `CLI_REPORTED_ERROR` code AND `details.message` carries the verbatim CLI line. (b) stub child surfaces the destination-exists error → handler propagates with the verbatim message.

**Acceptance Scenarios**:

1. **Given** the `rename_note` tool and a stub adapter reporting a source-not-found failure for `Inbox/Missing.md`, **When** an MCP client calls `rename_note({ target_mode: "specific", vault: "MyVault", path: "Inbox/Missing.md", name: "Renamed" })`, **Then** the tool returns an MCP error response whose `code` is the one the adapter classified (either `CLI_REPORTED_ERROR` or `CLI_NON_ZERO_EXIT` per the CLI's actual signal) AND `details.message` carries the verbatim wording from the CLI.
2. **Given** the `rename_note` tool and a stub adapter reporting a destination-exists failure (e.g., the file `Inbox/B.md` already exists and the CLI refuses to overwrite), **When** an MCP client calls `rename_note({ target_mode: "specific", vault: "MyVault", path: "Inbox/A.md", name: "B" })`, **Then** the tool returns an MCP error response whose `details.message` identifies the destination collision — the typed surface does NOT silently overwrite, does NOT auto-suffix the destination, and does NOT swallow the error.

### User Story 5 — Active-mode renames the focused note (Priority: P1)

When the agent declares `target_mode: "active"`, the call's intent is "rename whatever note is currently open in Obsidian." The schema layer (per the target-mode primitive) forbids `vault`, `file`, and `path` in this mode. `name` remains required (renames must supply a destination name regardless of how the source is resolved). The handler routes `{ target_mode: "active", name }` through the adapter, which emits argv beginning with the rename subcommand and the `name` token but no source-locator key=value tokens. The CLI renames the focused note's file in place; the handler returns `{ renamed: true, fromPath: "<focused note's vault-relative path>", toPath: "<focused note's folder + new name>" }`.

**Why this priority**: P1. The active-mode contract is the entire reason `target_mode` is a discriminated union — collapsing it to "specific only" defeats ADR-003. The agent uses active mode when it operates on the user's current focus; making that path work end-to-end (schema → adapter → CLI) for the in-place rename is part of this BI's value. Per the user input's [P1] AC #10 and AC #11.

**Independent Test**: With a stub adapter that exits `0` and reports `Daily/2026-05-12.md` as the source and `Daily/Today.md` as the destination, calling `rename_note({ target_mode: "active", name: "Today" })` results in the adapter being invoked with argv that contains no `vault=…`, `file=…`, or `path=…` tokens, includes the destination-name token (`name=Today`), and the handler returns `{ renamed: true, fromPath: "Daily/2026-05-12.md", toPath: "Daily/Today.md" }`. A second test asserts that calling `rename_note({ target_mode: "active", name: "X" })` against a stub adapter that surfaces `ERR_NO_ACTIVE_FILE` flows that error through to the MCP error response unchanged.

**Acceptance Scenarios**:

1. **Given** the `rename_note` tool and a stub adapter reporting a successful focused-note rename, **When** an MCP client calls `rename_note({ target_mode: "active", name: "Today" })`, **Then** the tool returns `{ renamed: true, fromPath: "<focused source path>", toPath: "<focused folder>/Today.md" }` AND the adapter was invoked with argv that contains the rename subcommand, a `name`-shaped destination token, and no locator tokens.
2. **Given** the `rename_note` tool, **When** an MCP client calls `rename_note({ target_mode: "active", vault: "V", name: "X" })` (forbidden key in active mode), **Then** the call fails at the zod parse boundary as `VALIDATION_ERROR` whose `details.issues[].path` includes `"vault"` AND whose message names `"vault"` AND `"active mode"` per the target-mode primitive's contract. The same applies for forbidden `file` and `path`.
3. **Given** the `rename_note` tool and a stub adapter that surfaces `UpstreamError({ code: "ERR_NO_ACTIVE_FILE", … })`, **When** an MCP client calls `rename_note({ target_mode: "active", name: "X" })`, **Then** the tool returns an MCP error response (`isError: true`) whose JSON-serialised payload carries `code: "ERR_NO_ACTIVE_FILE"` and a recovery-hint message naming "focus a note or switch to specific mode" — propagated via the adapter's verbatim `.message` field, not synthesised at the `rename_note` layer.

### User Story 6 — Schema layer rejects malformed inputs before any CLI call (Priority: P1)

Per Constitution Principle III, the zod schema is the gatekeeper. `rename_note`'s schema composes the post-010 flat target-mode primitive (which enforces `vault` required in specific mode, exactly-one-of `file`/`path` in specific mode, `vault`/`file`/`path` forbidden in active mode, `additionalProperties: false` strict-mode against unknown top-level keys) and adds `name: z.string().min(1).regex(/^[^/\\]+$/)` as the single `rename_note`-specific required field. Eight classes of malformed input MUST fail at the parse step: (a) `target_mode === "specific"` with neither `file` nor `path`; (b) `target_mode === "specific"` with both `file` AND `path`; (c) `target_mode === "specific"` without `vault`; (d) `target_mode === "active"` with any of `vault`/`file`/`path`; (e) any input with an unknown top-level key (e.g., `pancakes: "yes"`); (f) `target_mode` absent or with a value other than `"specific"` / `"active"`; (g) `name` absent OR `name === ""` OR `name` present but a non-string (e.g., `name: 42`); (h) `name` contains a folder separator (`/` or `\`) per the /speckit-clarify Q2 resolution. None of these reach the adapter; the failure is a `VALIDATION_ERROR` with the offending field path. **No file on disk MUST be touched by any input that fails validation.** The CLI is never invoked.

**Why this priority**: P1. Boundary validation is the constitutional baseline — ADR-003's promise that the schema enforces the intent contract is empty if `rename_note` doesn't actually parse against the primitive. For a rename operation, skipping this story is doubly bad: every malformed input either becomes an opaque CLI failure (uninformative to the agent) or — worse — undefined behaviour where the CLI accepts both `file` AND `path` and silently renames the wrong source. The empty-name carve-out is the additional safety net — `name: ""` MUST fail at the schema layer, not bubble up as an opaque CLI failure or — worst case — silently produce a rename to the empty-string filename that the OS may or may not refuse depending on platform. The folder-separator-rejection carve-out (per /speckit-clarify Q2) is the structural enforcement of the in-place-rename scope boundary — `name: "Sub/X"` MUST NOT be silently forwarded to the CLI where it might be interpreted as a rename-with-move operation on Windows (conflating with a future `move_note` surface) or surface as an opaque CLI error on POSIX. Per the user input's [P1] ACs #6, #7, #8, #9, #10.

**Independent Test**: Eight parse-only tests, each driving the handler with a stub adapter that MUST NOT be called. Each asserts the resulting error is `VALIDATION_ERROR` with the expected `details.issues[].path` and message keywords; the stub spawn function is asserted never to have been called.

**Acceptance Scenarios**:

1. **Given** the `rename_note` tool and a stub adapter that records every invocation, **When** an MCP client calls `rename_note({ target_mode: "specific", vault: "MyVault", name: "X" })` (neither `file` nor `path`), **Then** the tool returns `code: "VALIDATION_ERROR"` whose `details.issues` includes an entry whose `message` mentions `"exactly one of"` AND the stub adapter was never invoked.
2. **Given** the `rename_note` tool, **When** an MCP client calls `rename_note({ target_mode: "specific", vault: "MyVault", file: "F", path: "F.md", name: "X" })` (both locators), **Then** the tool returns `code: "VALIDATION_ERROR"` whose issues mention `"exactly one of"` AND the stub adapter was never invoked.
3. **Given** the `rename_note` tool, **When** an MCP client calls `rename_note({ target_mode: "specific", path: "P.md", name: "X" })` (vault missing), **Then** the tool returns `code: "VALIDATION_ERROR"` whose `details.issues[].path` includes `"vault"` AND the stub adapter was never invoked.
4. **Given** the `rename_note` tool, **When** an MCP client calls `rename_note({ target_mode: "active", vault: "V", name: "X" })` (forbidden key in active mode), **Then** the tool returns `code: "VALIDATION_ERROR"` whose issues mention `"vault"` AND `"active mode"` AND the stub adapter was never invoked. The same shape applies for forbidden `file` and forbidden `path`.
5. **Given** the `rename_note` tool, **When** an MCP client calls `rename_note({ target_mode: "specific", vault: "V", path: "P.md", name: "X", pancakes: "yes" })` (unknown top-level key), **Then** the tool returns `code: "VALIDATION_ERROR"` whose `details.issues[]` includes a `code: "unrecognized_keys"` entry naming `pancakes` AND the stub adapter was never invoked. ### User Story 7 — CLI failures (binary missing, non-zero exit, in-band Error, no active file) flow through `UpstreamError` (Priority: P1)

When the underlying CLI fails — the binary is missing, the process exits non-zero, the CLI replies with `Error: …` on stdout, the focused note is missing in active mode — the failure surfaces to the MCP client as an `UpstreamError` carrying the adapter's classified `code`. Per the cli-adapter's four-priority classification, `rename_note` does not classify; it propagates whatever code the adapter raised.

**Why this priority**: P1. Constitution Principle IV requires every boundary failure to surface as a structured `UpstreamError` with a stable code. The user input names source-not-found, destination-collision, external-editor-locks-the-file, permission-denied, and unknown-vault-display-name as adversarial cases that MUST be handled rather than silently succeed; all flow through this propagation path (per FR-019). The actual code on the wire depends on what the CLI does — the spec's job is to require propagation, not to claim a single code for a failure mode the CLI controls.

**Independent Test**: Five handler tests, each driving the adapter through an injected stub: (a) stub child fails to spawn with `ENOENT` → handler returns `code: "CLI_BINARY_NOT_FOUND"`; (b) stub child exits `1` with stderr `"permission denied"` → handler returns `code: "CLI_NON_ZERO_EXIT"`; (c) stub child exits `0` with stdout `"Error: file not found at Inbox/Missing.md\n"` → handler returns `code: "CLI_REPORTED_ERROR"` AND `details.message` carries the verbatim CLI line; (d) stub child surfaces `ERR_NO_ACTIVE_FILE` → handler returns `code: "ERR_NO_ACTIVE_FILE"`; (e) any non-`UpstreamError` exception escapes the adapter — handler re-throws verbatim WITHOUT `asToolError` wrapping (parity with `read_note` / `write_note` / `delete_note`).

**Acceptance Scenarios**:

1. **Given** the `rename_note` tool and a stub adapter whose `spawnFn` raises `ENOENT`, **When** an MCP client calls `rename_note({ target_mode: "specific", vault: "V", path: "P.md", name: "X" })`, **Then** the tool returns an MCP error response with `code: "CLI_BINARY_NOT_FOUND"` AND the error's `details` carries the binary-resolver's structured failure shape per [017-cross-platform-support](../017-cross-platform-support/spec.md).
2. **Given** the `rename_note` tool and a stub adapter that exits `1` with stderr `"permission denied"`, **When** an MCP client calls `rename_note({ target_mode: "specific", vault: "V", path: "P.md", name: "X" })`, **Then** the tool returns an MCP error response with `code: "CLI_NON_ZERO_EXIT"` AND `details.exitCode: 1` AND `details.stderr` carries the verbatim CLI stderr.
3. **Given** the `rename_note` tool and a stub adapter that exits `0` with stdout `"Error: file not found at Inbox/Missing.md\n"`, **When** an MCP client calls `rename_note({ target_mode: "specific", vault: "V", path: "Inbox/Missing.md", name: "X" })`, **Then** the tool returns an MCP error response with `code: "CLI_REPORTED_ERROR"` AND `details.message` carries the verbatim wording from the CLI — the adapter's classification is propagated unchanged.
4. **Given** the `rename_note` tool, **When** any non-`UpstreamError` exception escapes the adapter (an unexpected runtime error), **Then** the handler re-throws it (does NOT wrap it as a tool error), matching the `obsidian_exec` / `read_note` / `write_note` / `delete_note` precedent.

### User Story 9 — Same-name rename is a successful no-op (Priority: P3)

When the agent calls `rename_note` with a `name` that resolves to the same canonical destination path as the source — i.e., the rename produces `fromPath === toPath` — the call returns success with `renamed: true` and `fromPath === toPath`. The underlying CLI's behaviour is captured during plan-stage research per FR-019: either (a) the CLI accepts the no-op rename and returns success (the typed surface's preferred behaviour); or (b) the CLI rejects the no-op rename with an error; or (c) the CLI silently does nothing and returns success. The typed surface propagates whichever shape the CLI exposes — but documents the observed behaviour explicitly so callers know whether to filter same-name calls client-side. The audit-trail signal (`renamed: true` + `fromPath === toPath`) is the structural marker callers can use to detect no-op renames without parsing the response further.

**Why this priority**: P3. Same-name renames are an edge case — most callers either filter them client-side (a comparison against the existing name is cheap) or accept the CLI's behaviour as-is. The story exists to (a) lock the structural invariant `fromPath === toPath` for no-op cases and (b) document the CLI's observed behaviour so callers can choose to filter or not. Per the user input's [P3] AC #16.

**Independent Test**: With a stub adapter that exits `0` and reports an unchanged from/to path pair (e.g., source `Inbox/Note.md`, destination `Inbox/Note.md`), calling `rename_note({ target_mode: "specific", vault: "V", path: "Inbox/Note.md", name: "Note" })` results in the handler returning `{ renamed: true, fromPath: "Inbox/Note.md", toPath: "Inbox/Note.md" }` (the audit-trail invariant holds AND the call did not error).

**Acceptance Scenarios**:

1. **Given** the `rename_note` tool and a stub adapter reporting an unchanged path pair, **When** an MCP client calls `rename_note` with a `name` that resolves to the same canonical destination as the source, **Then** the tool returns `{ renamed: true, fromPath: <path>, toPath: <path> }` (with `fromPath === toPath` by string equality) AND the call does NOT surface as an error.

### Edge Cases

- **`rename_note({ target_mode: "specific", vault: "", path: "P.md", name: "X" })`** (empty-string vault): MUST fail at the zod parse boundary as `VALIDATION_ERROR` whose `details.issues[].path` includes `"vault"`. The target-mode primitive enforces `vault: z.string().min(1)`. Reasonable default — error message mirrors zod's standard `min(1)` message.
- **`rename_note({ target_mode: "specific", vault: "V", path: "../../../etc/passwd.md", name: "X" })`** (path-traversal-shaped source): MUST be either rejected at the validation boundary or verified to be rejected by the underlying CLI. Per the user input's FILESYSTEM adversarial bullet, **silent vault-escape is a critical security defect** — renaming files outside the vault root is unacceptable. Plan-stage characterisation per FR-019 verifies the CLI's behaviour against `../`-shaped paths; if the CLI does NOT reject such inputs, this BI is amended pre-ship to add a tool-layer reject (a precondition, not an open question — gated by SC-012). The dominant security concern for `rename_note` is path-traversal because shell-injection is structurally prevented by the cli-adapter's argv-array passing.
- **`rename_note({ target_mode: "specific", vault: "DoesNotExist", path: "P.md", name: "X" })`** (unknown vault display name): inherits the response-inspection clause that 011-write-note's R5 added to the cli-adapter. The unknown-vault response is re-classified by the adapter to `CLI_REPORTED_ERROR` with the verbatim CLI message preserved in `details.message`. `rename_note` propagates this without modification. Plan-stage characterisation per FR-019 confirms the CLI's response signature for the rename subcommand matches the create / delete subcommand signature; if it differs, the adapter's response-inspection logic is extended (additive change to the adapter, not to `rename_note`).
- **Source deleted out-of-band before the call** (e.g., another agent or the user removed it via Obsidian's UI): surfaces as a structured `CLI_REPORTED_ERROR` (or `CLI_NON_ZERO_EXIT` depending on the CLI's exit code). Plan-stage research per FR-019 captures the verbatim wording and the handler propagates it. No silent no-op — the second caller MUST see a structured error per the user input's CONCURRENCY adversarial bullet.
- **Two concurrent `rename_note` calls against the same source**: the shared in-flight CLI queue (per FR-008, inherited from [006-read-note](../006-read-note/spec.md) FR-016 and [011-write-note](../011-write-note/spec.md) FR-008) serializes both calls behind one channel. The first call renames the file; the second call sees the on-disk state after the first call completed and surfaces a `CLI_REPORTED_ERROR` (source-not-found, because the source has been renamed away). Deterministic ordering is guaranteed by the queue; silent no-ops are precluded by the source-not-found path. Per the user input's CONCURRENCY adversarial bullet.
- **External editor has the file open** (Obsidian, VS Code, anything else) when `rename_note` is invoked: observable behaviour is captured during plan-stage characterisation per FR-019 and documented in `docs/tools/rename_note.md`. `rename_note` does NOT lock the file or coordinate with external editors. If the OS rejects the rename with EBUSY (Windows), the CLI surfaces non-zero exit; the handler propagates as `CLI_NON_ZERO_EXIT`. If the OS allows the rename and Obsidian closes/reopens the buffer (or shows a stale view), that is the editor's behaviour, not `rename_note`'s.
- **Active-mode TOCTOU**: the focused note may change between the schema's parse step and the CLI's actual invocation. The operation runs against whichever note is focused at execution time. Documented as a known limitation; `rename_note` does NOT attempt to lock or pin the focused note. Agents that need certainty about which file is being renamed MUST use specific mode with an explicit locator. The project-wide active-mode TOCTOU concern is governed at the target-mode primitive layer, not retrofitted per-tool. Per the user input's CONCURRENCY adversarial bullet.
- **Case-only renames on Windows / macOS-default filesystems** (`Note.md` → `note.md`): platform-dependent. On case-insensitive filesystems (Windows NTFS-default, macOS APFS-default), the OS may treat the operation as a no-op (the underlying inode is the same), or as a rename (the on-disk filename's case changes but the inode stays). On case-sensitive filesystems (Linux ext4-default, macOS APFS case-sensitive option, Windows WSL ext4), the operation is a regular rename. Plan-stage characterisation per FR-019 captures the observed CLI behaviour on each platform; the typed surface propagates the CLI's response verbatim. Per the user input's FILESYSTEM adversarial bullet.
- **Unicode normalisation of `name`** (e.g., NFC vs NFD on macOS HFS+ vs APFS): the input `name` may not round-trip exactly through the on-disk path. macOS HFS+ stores filenames as NFD-normalised; APFS is case-insensitive-but-normalisation-preserving. Windows preserves the input bytes. Linux ext4 preserves the input bytes. `rename_note` does NOT normalise the `name` input; it forwards verbatim to the CLI. The CLI / OS may store a normalised form. The handler's response `toPath` reflects whatever the CLI reports — which on macOS HFS+ may differ byte-for-byte from the input `name`. Documented as a platform limitation. Per the user input's FILESYSTEM adversarial bullet.
- **Reserved Windows characters in `name`** (`:?*<>|"` / `\` / `/`): platform-dependent. The CLI is the trust boundary; `rename_note` forwards verbatim (except where the folder-separator-rejection rule per /speckit-clarify Q2 catches `/` and `\` at the schema layer). Plan-stage CLI characterisation per FR-019 records the observed behaviour for the remaining reserved characters (likely platform-specific — Windows kernel rejects with `EINVAL`, POSIX OSes accept everything except `/` and `\0`); the documented behaviour is captured in `docs/tools/rename_note.md` per FR-014. Per the user input's FILESYSTEM adversarial bullet.
- **Reserved Windows device names in `name`** (`CON`, `PRN`, `AUX`, `NUL`, `COM1`–`COM9`, `LPT1`–`LPT9`): platform-dependent. The CLI is the trust boundary; same posture as reserved-characters above. Documented in `docs/tools/rename_note.md`.
- **`name` contains a folder separator** (e.g., `name: "Subfolder/New"`): rejected at the validation boundary as `VALIDATION_ERROR` per the /speckit-clarify Q2 resolution. The schema enforces `name: z.string().min(1).regex(/^[^/\\]+$/)`. The error message names the rule and suggests `move_note` as the recovery path for callers that intended a rename-with-move operation. Preserves the in-place-rename scope cleanly vs the future `move_note` surface; produces an actionable error message at the boundary rather than relying on platform-dependent CLI / OS behaviour (POSIX errors with invalid-filename; Windows might partially-move). Per the user input's "Out of scope: Rename + move in one operation" bullet.
- **Aliased wikilinks** (`[[Real Path|Display Text]]`): when the vault's auto-update-links setting is enabled, the auto-update rewrites the path side (`Real Path`) leaving the display text unchanged. `rename_note` does NOT control this — Obsidian does. Documented in `docs/tools/rename_note.md` so callers understand display text persists. Per the user input's LINK-UPDATE BEHAVIOUR adversarial bullet.
- **Auto-update-links setting disabled in the vault**: links pointing at the old name remain stale after the rename. The file rename succeeds; the link-rewriting does not happen. `rename_note` does NOT detect or warn about this — the typed surface has no observability into the vault's settings. Documented prominently in `docs/tools/rename_note.md` per FR-014 and per the user input's [P1] AC #13.
- **Caller-supplied `path` / `file` / `name` containing argv-shell metacharacters** (e.g., `name: "; rm -rf /; #"`): SAFE. The cli-adapter assembles argv as a child_process.spawn array (no shell), so no character in `path` / `file` / `name` can break out into a shell. This is a structural property of the adapter, not a runtime check. Path-traversal (the dominant remaining security concern for `path`) is covered by the bullet above; for `name`, the reserved-character behaviour is the OS / CLI's responsibility per the user input's SECURITY bullet.
- **Strict-rich vs strict-naive client-class observability of unknown-key rejection** (Story 6 AC #5): strict-rich clients (Claude Desktop, MCP Inspector) read the `additionalProperties: false` from the published `inputSchema` and either forward unknown keys (in which case the bridge-side `unrecognized_keys` rejection is observable to the test) or strip them client-side (in which case the bridge-side rejection is non-observable but the schema-side invariant — no untyped passthrough at runtime — still holds). Strict-naive clients (Cowork) do not consult `additionalProperties`; they forward unknown keys to the bridge, which always observes the rejection. Tests assume the strict-rich-forwarding path (the worst case) and assert the bridge-side rejection occurs. Documented per the user input's CLIENT-CLASS adversarial bullet.

## Requirements *(mandatory)*

### Functional Requirements

#### Module layout

- **FR-001**: A new typed tool MUST be created at `src/tools/rename_note/` following the existing per-surface module convention (per Constitution Principle I and the `src/tools/{delete_note, find_by_property, help, list_files, obsidian_exec, read_heading, read_note, read_property, write_note, write_property}/` precedents). The module MUST contain six files at minimum (three source + three co-located tests): `schema.ts` (the zod input schema, the inferred TypeScript types, and the JSON Schema export via `zod-to-json-schema`), `handler.ts` (the function that calls the adapter and maps the result to the output shape), `index.ts` (the `createRenameNoteTool` factory returning a `RegisteredTool` per the aggregator pattern — `index.ts` per the post-011 file-name convention, NOT `tool.ts`), and one co-located test file per source file (`schema.test.ts`, `handler.test.ts`, `index.test.ts`). Per Constitution Principle V, every new source file MUST carry an `// Original — no upstream. <one-line description>.` header.

#### Schema

- **FR-002**: The input schema MUST compose the post-010 flat `target_mode` primitive (per [010-flatten-target-mode](../010-flatten-target-mode/spec.md)) using the canonical Pattern (a) flat-extension idiom: `applyTargetModeRefinement(targetModeBaseSchema.extend({ name }))`. Per the contract at [specs/010-flatten-target-mode/contracts/flat-target-mode.contract.md](../010-flatten-target-mode/contracts/flat-target-mode.contract.md), `.extend()` (NOT `.merge()`) is the binding API call — `.merge()` resets `unknownKeys` to `"strip"` and silently drops the strict-mode safety net for unknown top-level keys. The composed schema inherits the primitive's full per-mode contract end-to-end: `vault` required in specific mode, exactly-one-of `file`/`path` in specific mode, `vault`/`file`/`path` forbidden in active mode, top-level `additionalProperties: false` against unknown keys.

  **Active-mode `superRefine` clauses for `rename_note`**: NONE beyond the folder-separator-rejection rule on `name` (which applies uniformly in BOTH modes via the regex on `name` — see FR-003). The target-mode primitive's existing rules govern locator XOR and forbidden-key checks unchanged.

- **FR-003**: The schema MUST add one `rename_note`-specific required field:
    - `name: z.string().min(1).regex(/^[^/\\]+$/)` — required in BOTH modes. The `.min(1)` rejects empty-string `name` at the parse step per Story 6 AC#6 / the user input's [P1] AC #8. The `.regex(/^[^/\\]+$/)` rejects folder-separator characters (`/` and `\`) per the /speckit-clarify Q2 resolution; the regex error message names the rule and suggests `move_note` as the recovery path. The field is NOT optional — every rename must supply a destination name regardless of mode. (Implementation detail: the regex clause may be expressed as `.refine()` or `.superRefine()` if a custom error message is required; the structural contract — reject `/` or `\` in `name` — binds, the API call shape does not.)

- **FR-004**: The schema MUST NOT add `.describe()` annotations to any field. Parameter documentation lives in `docs/tools/rename_note.md` (per ADR-005's progressive-disclosure design).

- **FR-005**: The schema MUST export the inferred TypeScript type via `z.infer<typeof renameNoteInputSchema>` (or equivalent name). No hand-written `interface RenameNoteInput` or `type RenameNoteInput = { … }` is permitted (Constitution Principle III). The output type MUST also be a zod schema (or its `z.infer<…>`): `renameNoteOutputSchema = z.object({ renamed: z.literal(true), fromPath: z.string(), toPath: z.string() }).strict()`. The `renamed` field is a `z.literal(true)` because every successful return path of `rename_note` produces `renamed: true` — the failure path goes through `UpstreamError`, not through a `renamed: false` shape (parity with `delete_note`'s `deleted: true` literal). Hand-rolled output types are forbidden by the same principle.

- **FR-006**: The schema MUST export a JSON Schema document via `zod-to-json-schema(...)` (mirroring `obsidian_exec`, `read_note`, `write_note`, `delete_note`). The exported JSON Schema MUST round-trip through the schema-stripping utility at registration time per FR-011. The emitted JSON Schema MUST be the post-010 flat shape (one top-level `type: "object"`, `properties` with all five top-level fields typed inline, `additionalProperties: false`, no `oneOf` envelope) — verifiable via the consolidated drift detector at [src/tools/_register.test.ts](../../src/tools/_register.test.ts) which automatically covers every registered tool.

#### Handler

- **FR-007**: The handler MUST route the validated input through the cli-adapter's `invokeCli` entry point (or its bounded counterpart through the shared queue per FR-008). The adapter call shape is `invokeCli({ command: "<rename subcommand name>", parameters: <derived>, flags: <derived>, target_mode: <validated>, vault: <hoisted-or-undefined> }, deps?)`, where the rename subcommand name (`rename`, `mv`, or whatever the live CLI exposes) is captured during plan-stage research per FR-019. The `parameters` and `flags` records are derived from the validated input as follows:
    - **Specific mode**: `vault: parsed.vault` is passed as a TOP-LEVEL field (per [011-write-note](../011-write-note/spec.md) PSR-3) NOT inside `parameters`. `parameters` includes exactly one of `file: parsed.file` or `path: parsed.path` (with the user-facing field name potentially renamed to a CLI argv key per the 011-write-note PSR-5 precedent — confirmed during plan-stage research per FR-019), plus `name: parsed.name` (or its CLI argv-key rename).
    - **Active mode**: `parameters` contains the `name` token only (no `vault`, no `file`, no `path` — the schema layer has already rejected those and the adapter's defensive locator-stripping is a redundant safety net). The top-level `vault` field passed to `invokeCli` is `undefined` in active mode.
    - The choice between key=value form (`name=NewName`) and flag form for the destination-name token is a plan-stage research decision per FR-019, parity with the 011-write-note PSR-4 resolution for `overwrite`/`open`. The structural contract (the name token is emitted for every call, never omitted) binds; the placement (parameters vs flags) does not.

  **Extension-handling rule** (per /speckit-clarify Q1, locked 2026-05-12): the handler computes the forwarded `name` argv-token value via a file-local helper `appendMdIfMissing(name)` of ~3 LOC: `return name.endsWith(".md") ? name : name + ".md"`. The check is literal byte equality (case-sensitive), mirroring the 020-fix-write-gaps R2 lock. Worked examples:
    - `name: "Fixed"` → argv token `Fixed.md` (no `.md` suffix → append)
    - `name: "Fixed.md"` → argv token `Fixed.md` (verbatim)
    - `name: "Doc.v1.draft"` → argv token `Doc.v1.draft.md` (internal periods preserved; `.draft` not `.md` → append)
    - `name: "Renamed.MD"` → argv token `Renamed.MD.md` (case-sensitive match → append; uppercase `.MD` does not satisfy the rule)
    - `name: "Sketch.canvas"` → argv token `Sketch.canvas.md` (cross-extension type conversion not in scope per /speckit-clarify Q1; callers route through `obsidian_exec` if they want a literal `.canvas` destination)

  The forwarded value is what reaches the CLI argv's `name` token; the response's `toPath` reflects the CLI's canonical destination path. The extension-handling rule is wrapper-owned and does NOT depend on the CLI's behaviour. If the CLI itself ALSO auto-appends `.md` when omitted, the wrapper's pre-append is idempotent (the forwarded value already has `.md`, so the CLI's auto-append is a no-op); if the CLI does NOT auto-append, the wrapper's pre-append is the load-bearing step. Plan-stage research per FR-019 verifies this is true; if the CLI auto-appends a DIFFERENT extension (unlikely for an Obsidian-internal rename surface), the wrapper's rule still binds the agent-visible behaviour.

  Plan-stage research per FR-019 confirms the live CLI's actual rename-subcommand name, argv shape, and response wording for both successful renames and failure modes (source-not-found, destination-exists, no-active-file, unknown-vault). The handler's argv-mapping and response-parsing logic lock against the live CLI's behaviour.

- **FR-008**: The handler MUST execute through the same single-in-flight CLI queue that gates `obsidian_exec`, `read_note`, `write_note`, `delete_note`, `read_property`, `find_by_property`, `read_heading`, `write_property`, and `list_files` (per [006-read-note](../006-read-note/spec.md) FR-016 and successor specs). `createRenameNoteTool`'s `RegisterDeps` MUST accept `queue: Queue`, and `src/server.ts` MUST pass the *same* `queue` instance to `createRenameNoteTool` and to every other CLI-invoking tool so concurrent CLI invocations across all surfaces serialize through one channel. ### Out of Scope

The following are explicitly excluded from this feature and remain the responsibility of separate BIs or future work (mirroring the user input's "Out of scope" enumeration plus consequential exclusions surfaced during spec drafting):

- **Rename + move in one operation**: out of scope per the user input. A combined operation that changes both filename AND folder is a separate concern that splits cleanly from in-place rename. A future `move_note` tool (referenced in this spec but NOT a precondition of `rename_note`'s ship) handles the path-on-disk case; `rename_note` handles the filename-on-disk case. The `name` field's folder-separator-rejection rule (locked at /speckit-clarify Q2, session 2026-05-12) is the enforcement point that keeps the two concerns separate.
- **Bulk multi-file renames**: callers iterate today, or a future batch-rename BI ships if usage justifies. The single-file typed surface does not aggregate calls.
- **Forcing link-update regardless of vault setting**: out of scope. The vault's "Automatically update internal links" setting governs link-rewriting; the typed surface does NOT mutate that setting (which would require config writes the CLI does not necessarily expose) and does NOT force link-rewriting independently (which would require a parallel link-rewriting implementation, defeating the point of wrapping the CLI's native surface). If empirical use surfaces a strong need for forced link-rewriting, a future BI ships either (a) a `force_update_links: true` flag with explicit caveats, or (b) a separate `rewrite_links` primitive that operates orthogonally.
- **Conflict-resolution policy when the new name exists**: out of scope per the user input. The typed surface surfaces the underlying CLI's behaviour as-is — if the CLI refuses to overwrite, the surface returns `CLI_REPORTED_ERROR` with the verbatim message; if the CLI auto-suffixes, the surface returns success with the suffixed `toPath`; if the CLI overwrites silently (unlikely for Obsidian's rename surface), plan-stage characterisation per FR-019 case (vi) surfaces this as a known platform behaviour. The typed surface does NOT add a tool-layer `overwrite` flag, `auto_suffix` flag, or `on_conflict` policy enum.
- **Path-traversal validation at the tool layer**: out of scope per Edge Cases — UNLESS plan-stage characterisation per FR-019 case (ix) surfaces that the CLI does not reject `../`-shaped paths, in which case the spec is amended pre-ship to add a tool-layer reject. This is a precondition, not an open question (gated by SC-012). Silent vault-escape on a rename is a critical security defect.
- **Case-only-rename normalisation across platforms**: out of scope. The typed surface propagates the CLI's behaviour verbatim. Plan-stage characterisation per FR-019 case (x) captures the observed behaviour on each platform; `docs/tools/rename_note.md` documents it.
- **Unicode normalisation of `name`**: out of scope. The typed surface forwards verbatim to the CLI; the OS / filesystem normalises (or doesn't) per platform. macOS HFS+ NFD-normalisation is documented as a platform limitation, not mitigated.
- **A `list_vaults` primitive for vault-display-name pre-validation**: out of scope. The unknown-vault edge case is handled by the cli-adapter's response-inspection clause (per [011-write-note](../011-write-note/spec.md) R5). If response-inspection proves brittle, `list_vaults` ships as a follow-up BI and pre-validation moves there.
- **New error codes**: out of scope per the user input. `rename_note`'s entire failure surface is covered by `VALIDATION_ERROR` + the adapter's four codes. If a `rename_note`-specific failure mode emerges (e.g., "destination exists and the CLI's response wording doesn't classify cleanly"), a future amendment introduces a code with the same rigour as [005-help-tool](../005-help-tool/spec.md) introduced `HELP_TOOL_NOT_FOUND` / `HELP_DOCS_MISSING`.
- **Refactoring the existing typed tools**: out of scope. The eight existing typed tools co-exist with the new tool unchanged. `rename_note` adds rather than perturbs (per SC-009).
- **Modifying ADR-003, ADR-005, or ADR-006**: this BI implements all three; it does not change them. Conflicts surface as new ADRs per the project's amendment procedure.
- **Active-mode TOCTOU mitigations specific to `rename_note`**: out of scope. The project-wide TOCTOU concern is governed at the target-mode primitive layer. `rename_note` inherits whatever mitigation that layer offers (currently none beyond documentation); future TOCTOU work lands as an ADR amendment, not a `rename_note` patch.
- **Cross-extension renames (`.md → .canvas`, `.md → .pdf`, etc.)**: out of scope per the /speckit-clarify Q1 scope narrowing (locked 2026-05-12). The typed surface's extension-handling rule treats only `.md` as a literal extension; any other trailing segment in `name` is part of the filename stem and `.md` is appended. Callers that want a genuine cross-extension type conversion route through `obsidian_exec` with the CLI's rename subcommand. `docs/tools/rename_note.md` documents this routing explicitly per FR-014.
- **Renaming non-`.md` files via `rename_note`** (`.canvas` boards, `.pdf` attachments, image files in the vault): out of scope per the same /speckit-clarify Q1 scope narrowing. The tool's name (`rename_note`) and the wrapper's `.md`-only allowlist bind it to Markdown notes. The CLI's rename subcommand exposed via `obsidian_exec` remains the fallback for non-`.md` filename targets. If empirical use surfaces a strong need for typed renames of other Obsidian-native file types, a future BI ships a dedicated tool (e.g., `rename_canvas`) with its own scope; widening `rename_note`'s allowlist mid-life would be a contract break.

## Assumptions

- **The user input's `(required)` annotation on `target_mode` and `name`** is binding. The schema MUST NOT make either optional. `vault` is "required in specific mode, forbidden in active mode" by reference to the target-mode primitive's contract; `file` / `path` are "mutually exclusive in specific mode, forbidden in active mode" by the same reference.
- **The CLI exposes a single-file rename subcommand** (likely named `rename` or `mv` — captured during plan-stage research per FR-019) that accepts the argv shape `["<rename-subcommand>", "vault=…", "file=… | path=…", "name=…"]` and emits a structured success or failure response on stdout/stderr that the adapter can classify. This is the precondition the tool is built on; if the CLI uses a different shape, plan-stage characterisation per FR-019 captures the actual shape and the handler's argv-assembly retargets accordingly. Reasonable default: argv assembly is `[command, vault=…, locator=…, name=…, …extras]` parity with the existing subcommands.
- **The CLI's response signal for from/to canonical paths** is captured during plan-stage research per FR-019 (cases (i)–(iv) + (viii)). Plausible signals include: a stdout marker (`"Renamed Inbox/Old.md → Inbox/New.md"`); the two paths on stdout alone (one per line); or a JSON-shaped response. The handler's response-parsing logic locks against whichever signal the live CLI uses. If the CLI does not echo canonical paths (e.g., it returns only `"OK"` regardless of input), this BI is amended pre-ship to either (a) derive `fromPath` and `toPath` from the input locator + extension-preservation rule (with a fallback rule documented in `docs/tools/rename_note.md`), or (b) request a CLI-layer enhancement to disambiguate. The amendment lands before merge per FR-019.
- **The active-mode CLI invocation** for the rename subcommand (`obsidian <rename> name=…` with no vault/file/path) renames the focused note in the focused vault. If no vault is focused, the CLI surfaces `Error: no active file` (or equivalent), classified by the adapter as `ERR_NO_ACTIVE_FILE`. This is consistent with the read / create / delete subcommands' active-mode behaviour observed during prior BIs; if the rename subcommand's active-mode behaviour differs, plan-stage characterisation per FR-019 case (viii) captures the difference and the spec is amended.
- **The post-010 strict-mode `additionalProperties: false`** is the source-of-truth for the unknown-top-level-key rejection (Story 6 AC#5). `rename_note` does NOT add `.strict()` redundantly at the tool layer; it inherits strict mode from `targetModeBaseSchema.extend(...)` per [010-flatten-target-mode](../010-flatten-target-mode/spec.md) FR-002 (which verified that `.extend()` preserves strict mode while `.merge()` does not).
- **The cli-adapter's `deps.spawnFn` test seam** is the canonical test-injection mechanism. `rename_note`'s handler tests inject `deps.spawnFn` via the adapter call, NOT by mocking the adapter module itself. This matches `read_note`'s, `write_note`'s, `delete_note`'s, and `obsidian_exec`'s test patterns.
- **No new error codes** are introduced. Per Constitution Principle IV, error codes are the project's stable API. `rename_note`'s entire failure surface is covered by `VALIDATION_ERROR` (zod) and the adapter's four codes. If empirical use surfaces a `rename_note`-specific failure mode (destination-collision wording that resists classification is the anticipated candidate), a future amendment introduces the code with the same rigour as ADR-006 / [005-help-tool](../005-help-tool/spec.md).
- **Constitution Principle II binds the test-set co-location** at `src/tools/rename_note/{schema,handler,index}.test.ts`. The cases enumerated in FR-016 are the minimum set; reviewers may add more.
- **Constitution Principle V binds the module-header convention**: `// Original — no upstream. <one-line description>.` is required on every new source file in `src/tools/rename_note/`. The Markdown documentation file at `docs/tools/rename_note.md` is exempt per [005-help-tool](../005-help-tool/spec.md) FR-019.
- **Link-rewriting is vault-config-dependent** per the user input's [P1] AC #13. `rename_note` does NOT verify, mutate, or warn about the vault's "Automatically update internal links" setting. The CLI / Obsidian app honours the setting at rename time. The typed surface's documentation calls this out prominently so callers know stale links may remain when the setting is disabled.
- **`move_note` is a future BI** referenced in this spec but is NOT a precondition of `rename_note`'s ship. The user input names it ("Out of scope: Rename + move in one operation — use `move_note` with a destination path that includes the new name."); the spec mirrors the reference. The folder-separator-rejection rule (locked at /speckit-clarify Q2, session 2026-05-12) is the structural enforcement point that keeps `rename_note` scoped to in-place rename; `move_note`'s existence (or non-existence) does not affect `rename_note`'s ship gate.
- **Plan-stage decisions deferred** to `/speckit-plan`: the live CLI's rename-subcommand name (`rename` / `mv`); the exact argv shape (locator argv-key rename — `file=` vs `name=` vs `from=` per the 011-write-note PSR-5 precedent; the destination-name argv-key form); the exact wording of the top-level `description` field; the exact wording of `docs/tools/rename_note.md`; the registration order in `src/server.ts`; the response-parsing shape for the canonical `fromPath` and `toPath` fields; and the case-only-rename behaviour observed on each supported platform. The structural contracts in FR-001 through FR-019 bind; the surface-level wording does not.
- **/speckit-clarify decisions LOCKED at session 2026-05-12** (see ## Clarifications section near the top of this spec): Q1 — extension-handling rule resolved to **`.md`-only allowlist with case-sensitive byte equality** (`name.endsWith(".md")` → verbatim; else append `.md`); cross-extension renames and non-`.md` filename targets are out of scope and route through `obsidian_exec`. Q2 — folder-separator-rejection rule resolved to **validation-layer reject** (`name` MUST NOT contain `/` or `\`, with the `move_note` recovery hint). Both decisions bind the schema, handler, docs, and tests. No further /speckit-clarify session is required before /speckit-plan.
