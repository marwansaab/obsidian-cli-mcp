# Implementation Plan: Progressive Disclosure Help Tool

**Branch**: `005-help-tool` | **Date**: 2026-05-06 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from [specs/005-help-tool/spec.md](./spec.md)

## Summary

Implement [ADR-005 — Token-Optimized Tool Definitions via Progressive Disclosure](../../.decisions/ADR-005%20-%20Token-Optimized%20Tool%20Definitions%20via%20Progressive%20Disclosure.md) (BI-030) by shipping two co-located components plus the documentation directory they depend on:

1. **A pure schema-stripping utility** at [src/help/strip-schema.ts](../../src/help/strip-schema.ts) (with co-located vitest at [src/help/strip-schema.test.ts](../../src/help/strip-schema.test.ts)) — exports a single function `stripSchemaDescriptions(schema)` that takes a `zod-to-json-schema` output object and returns a deep-copy with `description` removed at every nesting level inside `properties`, `anyOf`, `oneOf`, `items`, and `additionalProperties` (root-level `description` preserved per FR-003). Uses `structuredClone` for the deep copy and a recursive walker that visits the canonical JSON Schema constructs. Pure: no `node:fs`, no `node:path`, no logger, no global state (FR-005, SC-004).

2. **A new public MCP tool named `help`** registered via the per-surface module layout at [src/tools/help/](../../src/tools/help/) — `schema.ts` (zod input schema with `tool_name: z.string().min(1).optional()` per Clarification Q1), `handler.ts` (path-resolution from `import.meta.url`, directory-existence check returning `HELP_DOCS_MISSING` per Clarification Q4, per-tool name resolution returning `HELP_TOOL_NOT_FOUND` if the file is missing, path-traversal defense via the canonical `node:path` resolution-then-prefix-check pattern), and `tool.ts` (the SDK registration carrying the stripped schema + the verb-led top-level description that mentions `help("help")` per FR-016). The tool is registered alongside `obsidian_exec` in [src/server.ts](../../src/server.ts) by a new `registerHelpTool` call, with both registrations passed through the strip utility per FR-006.

3. **The bundled `docs/tools/` directory** at the package root containing: `help.md` (full doc for the help tool itself; response to `help({ tool_name: "help" })`), `index.md` (the listing — response to `help({})`), `obsidian_exec.md` (full doc transcribed from [001's contracts](../001-add-cli-bridge/contracts/) per Clarification Q2, NOT a stub), and six stub files (`read_note.md`, `write_note.md`, `append_note.md`, `search_vault.md`, `list_notes.md`, `list_vaults.md` — the FR-012 ∪ architecture-committed roster per Clarification Q3) each carrying the `<!-- TODO(BI-XXX): populate this doc -->` marker. The directory is added to `package.json`'s `files` array as `"docs/tools/**/*.md"` (FR-014) so it ships with the npm release.

In addition to the two component-level vitest test sets (FR-017's strip-utility minimum and help-tool minimum), this BI ships a **server-level registry-consistency test** in [src/server.test.ts](../../src/server.test.ts) per Clarification Q5 — it iterates every tool the server registers and asserts a corresponding `docs/tools/<tool_name>.md` file exists. The same test (or a sibling) also asserts the strip utility was applied: every registered tool's `inputSchema.properties` tree is description-free at every depth (Story 1 AC#5 / SC-002, the bypass-detection assertion per Edge Case "Tool registration via the SDK that bypasses the stripping utility"). Per the P6 plan-stage decision below the two assertions live in the same test for cohesion.

The existing `obsidian_exec` tool's `inputSchema` is now passed through the strip utility at registration time, and its top-level description in [src/tools/obsidian_exec/tool.ts:15-16](../../src/tools/obsidian_exec/tool.ts#L15-L16) is condensed to the verb-led summary that mentions `help("obsidian_exec")` per FR-015. The schema fields, handler logic, and error contract are unchanged. Two new `UpstreamError` codes are introduced — `HELP_TOOL_NOT_FOUND` and `HELP_DOCS_MISSING` (Clarification Q4) — and registered in [specs/001-add-cli-bridge/contracts/errors.contract.md](../001-add-cli-bridge/contracts/errors.contract.md) plus the README error-codes table per the established 002/003 precedent.

Constitution Principles I (one new directory per component, downward imports), II (co-located tests on every new public surface), III (zod is the single source of truth at the help tool's boundary), IV (every help-tool failure surfaces as `UpstreamError` with a stable `code`), and V (original-contribution headers on every new source file) all bind `Y`. The plan resolves seven plan-stage decisions deferred from the spec (module paths and naming, strip-utility implementation tactic, path-traversal defense implementation, top-level description wording, bypass-detection test location, SC-006 measurement mechanism, registry-consistency test location) in the Phase 0 Research section below.

## Technical Context

**Language/Version**: TypeScript 5.6+ in strict mode; runtime Node.js >= 22.11 (LTS floor per constitution). Same as 001/002/003/004.

**Primary Dependencies**: No new runtime dependencies. The strip utility uses only standard-library primitives (`structuredClone` for the deep copy — available since Node 17 and LTS-stable since 18; recursive object walking via plain `for…in` / `Array.isArray`). The help tool uses `node:fs/promises` (`readFile`, `access`, `readdir`), `node:path` (`resolve`, `join`, `dirname`, `relative`), and `node:url` (`fileURLToPath`) — all in the standard library. Existing project deps unchanged: `@modelcontextprotocol/sdk` ^1.0.4 (SDK registration), `zod` ^3.23.8 (the help tool's input schema), `zod-to-json-schema` ^3.23.5 (input to the strip utility). Vitest ^4.1.5 + `@vitest/coverage-v8` ^4.1.5 continue as the test framework.

**Storage**: N/A. Both components are stateless. The strip utility produces a fresh deep copy per call. The help tool reads files on each invocation — no module-level cache. (Cross-session caching is explicitly out-of-scope per the spec.) The `docs/tools/` directory is read-only at runtime; the package treats it as part of its bundled artifact.

**Testing**: Vitest, co-located `*.test.ts` per Principle II.

- **Strip utility**: 4 minimum test cases at [src/help/strip-schema.test.ts](../../src/help/strip-schema.test.ts) per FR-017 minimum (Story 1 AC#1 happy-path flat, AC#2 happy-path nested, AC#3 boundary no-descriptions, AC#4 mutation safety) plus 2 recommended (AC#6 structural-key preservation, plus a `description: { foo: "bar" }` non-string-value Edge Case). Total: ~6 test bodies.
- **Help tool**: 8 minimum cases at [src/tools/help/handler.test.ts](../../src/tools/help/handler.test.ts) per FR-017 (Story 2 AC#1 named-tool, AC#2 index, AC#3 unknown-tool → `HELP_TOOL_NOT_FOUND`, plus the two failure cases pinned by clarifications: empty-string `tool_name` → `VALIDATION_ERROR` (Q1), missing `docs/tools/` directory → `HELP_DOCS_MISSING` (Q4), plus path-traversal probe → `HELP_TOOL_NOT_FOUND`, plus cwd-independence (Story 2 AC#4 / Story 4 AC#2), plus non-string `tool_name` → `VALIDATION_ERROR` (Story 2 AC#6)). Schema-level tests at [src/tools/help/schema.test.ts](../../src/tools/help/schema.test.ts) — ~3 cases (zod parse happy + empty-string rejection + omitted-field acceptance). Registration-level tests at [src/tools/help/tool.test.ts](../../src/tools/help/tool.test.ts) — ~3 cases (top-level description mentions `help`, stripped schema has no nested descriptions, `HELP_TOOL_NOT_FOUND` round-trips through the SDK error response). Total: ~14 test bodies across the help tool's three co-located `*.test.ts` files.
- **Registry-consistency + bypass-detection**: a single new test block in [src/server.test.ts](../../src/server.test.ts) per Clarification Q5 + the P6 plan-stage decision below — iterates every tool the server registers and asserts (a) a corresponding `docs/tools/<tool_name>.md` file exists and (b) the tool's `inputSchema.properties` tree contains zero `description` keys at any depth. ~2 test bodies in one block.

The aggregate statements coverage floor (84.3% per [vitest.config.ts:20](../../vitest.config.ts#L20), pinned by feature 002 and reaffirmed by 003 + 004) MUST not regress — FR-020 mandates this. Pre-implementation projection: the new code is small and exhaustively tested, so coverage moves *up* by ~0.4–0.6 pp.

**Target Platform**: Windows host (per [CLAUDE.md](../../CLAUDE.md) and the 001/002/003/004 baselines). Path-resolution uses `node:path` primitives that are platform-portable. The case-sensitivity of doc-file lookup follows the underlying filesystem (Edge Cases — Windows is case-insensitive; CI runs on Linux which is case-sensitive). Tests assert behavior under the present platform's filesystem; cross-platform case-folding is explicitly out-of-scope per the spec's case-mismatch Edge Case.

**Project Type**: Single library (Node.js MCP server). Same as 001/002/003/004.

**Performance Goals**: Strip utility — O(N) over the schema's total node count, where N is the count of schema-construct objects (typically ≲ 50 per tool). **Constraints**: Strip utility — no filesystem, no network, no logger imports (FR-005, SC-004). Help tool — no `process.cwd()` reference (FR-009, SC-005); path resolution anchored to `import.meta.url`. Both — no new runtime dependencies (FR-001 implicit, package.json deps unchanged).

**Plan-stage decisions resolved during this Phase 0** (full rationale + alternatives in [research.md](./research.md)):

- **P1 (FR-001, strip utility module path + export name)**: Module at `src/help/strip-schema.ts` (with test at `src/help/strip-schema.test.ts`); exported function name is `stripSchemaDescriptions`. Verb-led name matches `executeObsidianExec` / `invokeCli` / `applyTargetModeSpecificRefinement` precedents. The `src/help/` directory groups the two components belonging to the help feature (the strip utility + the help tool's per-surface module folder under `src/tools/help/`). Alternative `src/strip-schema/` rejected because the utility is logically a piece of the help feature; alternative `src/tools/help/strip-schema.ts` rejected because the strip utility is consumed by every tool registration site, not exclusively by the help tool — placing it inside `src/tools/help/` would invert the dependency direction (tools/* → tools/help/strip-schema).
- **P2 (FR-002, recursive walker tactic)**: One pure recursive function over the JSON Schema constructs `properties` / `anyOf` / `oneOf` / `items` / `additionalProperties`. Implementation: `structuredClone` first, then in-place delete on the clone. Avoids the typed-DAG-walker library temptation — JSON Schema's recursive structure has six relevant node kinds at most and a hand-rolled walker is ≲ 30 LOC. The walker explicitly does NOT visit `definitions` / `$defs` / `$ref` targets — `zod-to-json-schema` is invoked with `$refStrategy: "none"` (per [src/tools/obsidian_exec/schema.ts:19](../../src/tools/obsidian_exec/schema.ts#L19) precedent), so refs do not appear in our schemas. If a future schema needs `$ref` (e.g., very large discriminated unions), the walker is extended in a follow-up BI.
- **P3 (FR-007, help tool input-schema annotation choice)**: The `tool_name` field carries no `.describe()` annotation in the zod schema. Rationale: per FR-007 + Clarification Q1, the schema is `tool_name: z.string().min(1).optional()`. A `.describe(...)` annotation would only matter if the strip utility weren't applied — and it IS applied per FR-006, so any annotation is stripped at registration time anyway. Adding it would create the false impression that the description survives to `tools/list`. The top-level (registration-object-level) description is what the agent sees pre-strip, and FR-016 + the P5 plan-stage decision below pin its wording. Constitution Principle III is satisfied — the schema is the single source of truth for runtime parse + structural shape.
- **P4 (FR-010, path-traversal defense implementation)**: The handler resolves the candidate path via `path.resolve(docsDir, `${tool_name}.md`)` and rejects (returns `HELP_TOOL_NOT_FOUND`) if `path.relative(docsDir, candidate)` starts with `..` OR contains a path separator after the `.md` suffix logic OR equals `..`. Equivalently: the canonical "resolve then check that the relative path's first segment is the bare `<tool_name>.md` filename" pattern. The error message MUST NOT echo `tool_name` per FR-010 — the message lists the *available* names from `readdir`, and `details.requestedName` (the structured field) carries the original input for operator-side debugging. Path separators in `tool_name` (e.g., `tool_name: "../../etc/passwd"`) are caught by the `path.relative` check; null bytes in `tool_name` would be caught by Node's filesystem-call rejection (filesystem APIs reject embedded NULs), but a defensive `tool_name.includes("\0")` check is added at the start of the handler for clarity and to surface as `HELP_TOOL_NOT_FOUND` rather than as a surprise `ERR_INVALID_ARG_VALUE` from `readFile`. Alternatives rejected: regex-validation at the zod boundary (would shift the failure to `VALIDATION_ERROR` and complicate Q1's empty-string-only rejection; rejected because traversal is a runtime path concern, not a structural-validity concern).
- **P5 (FR-015 + FR-016, top-level description wording)**: Pinned strings, derived from the Constitution's "concise verb-led" requirement and the user input's "explicit mention of `help("<tool_name>")`":
    - **`help` tool**: `"Look up full Markdown documentation for any registered MCP tool. Call help() with no arguments for an index of available docs, or help({ tool_name: \"<name>\" }) for a specific tool's full parameter docs. Self-describing — call help({ tool_name: \"help\" })."`
    - **`obsidian_exec` tool** (replaces the existing 200-word string at [src/tools/obsidian_exec/tool.ts:15-16](../../src/tools/obsidian_exec/tool.ts#L15-L16)): `"Invoke any Obsidian Integrated CLI subcommand. Returns stdout, stderr, exitCode, and the exact argv invoked. Failures (non-zero exit, in-band Error: prefix, missing binary, timeout, output > 10 MiB) surface as structured errors with stable codes. Call help({ tool_name: \"obsidian_exec\" }) for full parameter docs and the error-code roster."`
    The original 200-word `OBSIDIAN_EXEC_DESCRIPTION` is removed (the surrounding constant declaration becomes `export const OBSIDIAN_EXEC_DESCRIPTION = "...";` with the new short string). Token-cost reduction at the description alone is ~85% (~1100 → ~170 chars), corroborating the SC-006 directional claim.
- **P6 (Clarification Q5 + Edge Case bypass-detection, registry-consistency test location)**: A single new test block at the bottom of [src/server.test.ts](../../src/server.test.ts) (NOT a separate test file) holds both assertions per Q5: (a) every registered tool has a `docs/tools/<name>.md` file, (b) every registered tool's `inputSchema.properties` tree is description-free at every depth (the bypass-detection assertion). Rationale: both assertions iterate the same `server.tools` collection; splitting them into two files would duplicate the iteration scaffold without adding value. The block is named `describe("registry consistency", () => { ... })` for grep-discoverability. The Edge Case "Tool registration via the SDK that bypasses the stripping utility" therefore lands at this test block, not at a sibling.
- **P7 (SC-006 measurement mechanism)**: A one-off measurement noted in the PR description, NOT a benchmark test. Rationale: SC-006's claim is directional (~70%) and environmental — running it as a vitest benchmark would tie the merge gate to a number that drifts every time a tool ships a new `.describe()` annotation, creating a maintenance burden out of proportion to the value. The PR author measures the JSON-serialized `tools/list` response byte count (a) before applying the strip utility (with the existing 200-word `OBSIDIAN_EXEC_DESCRIPTION`-style descriptions on `obsidian_exec`'s schema fields if any) and (b) after, and records the delta in the PR description as a one-liner. SC-006 is satisfied when the PR description carries this measurement and the directional claim (≥ 50% reduction at the typical-tool surface, weighted toward the ~70% claim from ADR-005) holds.

**ADR-005 / Architecture alignment**: [.decisions/ADR-005 - Token-Optimized Tool Definitions via Progressive Disclosure.md](../../.decisions/ADR-005%20-%20Token-Optimized%20Tool%20Definitions%20via%20Progressive%20Disclosure.md) is the design decision this feature implements. The plan respects ADR-005's decisions verbatim: schema stripping at registration time (decision #1), `help` tool serving Markdown (decision #2), embedded `docs/tools/*.md` (decision #3), caveman descriptions on tools that mention help (decision #4). No ADR amendment, no new ADR. The [Architecture document](../../.architecture/Obsidian%20CLI%20MCP%20-%20Architecture.md) names BI-030 as this implementation; this feature IS BI-030.
