# Feature Specification: Progressive Disclosure Help Tool

**Feature Branch**: `005-help-tool`
**Created**: 2026-05-06
**Status**: Draft
**Input**: User description: "Add Help Tool — Implement progressive disclosure for tool definitions via two components: (1) a schema-stripping utility that removes parameter descriptions from the JSON Schema at tool registration time, and (2) a new public MCP tool called `help` that serves full Markdown documentation for any tool on demand. This is the implementation of ADR-005."

## Background *(non-mandatory context)*

[ADR-005 — Token-Optimized Tool Definitions via Progressive Disclosure](../../.decisions/ADR-005%20-%20Token-Optimized%20Tool%20Definitions%20via%20Progressive%20Disclosure.md) commits the project to a progressive-disclosure pattern for the MCP tool surface. The connector exposes 20+ typed tools, and each tool's `inputSchema` (the JSON Schema returned by `tools/list`) is the LLM's only structural guide to constructing valid payloads. Including full parameter-level descriptions on every tool consumes ~25 kB of the context window at session start — a fixed cost paid before the agent has done any work.

ADR-005's resolution is two-part: **strip the descriptions** from the JSON Schema at registration time (preserving structural validation — types, enums, required arrays, anyOf/oneOf — so the agent can still construct valid payloads from the stripped schema), and **expose a `help` tool** that serves the full Markdown documentation for any tool on demand. The agent reads a tool's docs once per session by calling `help({ tool_name: "<name>" })`; the response stays in chat history for the rest of the session. Structural correctness is preserved by zod's runtime validation at the tool boundary (Constitution Principle III) — if the agent constructs a malformed payload despite the stripped schema, the per-tool zod parser rejects it with a structured error, not the JSON Schema validator.

This feature is the implementation of ADR-005. It ships two components in one BI:

- **Component 1 — schema-stripping utility**: a pure function consumed by every tool registration site. It accepts a JSON Schema object (the output of `zod-to-json-schema`) and returns a deep copy with all parameter-level `description` fields removed.
- **Component 2 — `help` MCP tool**: a public MCP tool that reads Markdown documentation files from a bundled `docs/tools/` directory and returns their contents.

The Markdown documentation files for the public typed tools (BI-003 through BI-025: `read_note`, `write_note`, `append_note`, `search_vault`, `list_notes`, …) are **stubs** in this BI — each typed-tool BI populates its own doc file when it ships. This BI's responsibility is the doc-file machinery and the `help.md` + `index.md` files; the per-tool doc bodies are owned by the per-tool BIs.

The feature does not modify the `obsidian_exec` tool's behaviour — but the schema-stripping utility MUST be applied to its registration, and a stub `obsidian_exec.md` file MUST exist so the existing tool is reachable via `help({ tool_name: "obsidian_exec" })` from day one.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Tool registration emits a stripped JSON Schema with structural validation intact (Priority: P1)

Every MCP tool registered with the server passes its `zod-to-json-schema` output through the schema-stripping utility before that schema is handed to the SDK. As a result, an MCP client calling `tools/list` receives, for every tool, a JSON Schema document with **no** `description` field anywhere inside the `properties` tree (or any nested schema construct: `anyOf`, `oneOf`, `items`, `additionalProperties`, discriminator branches), but with all structural keys (`type`, `properties`, `required`, `enum`, `anyOf`, `oneOf`, `items`, `$ref`, `additionalProperties`) preserved exactly. The top-level tool registration object's own `description` field — the concise verb-led summary attached to the tool by the SDK — is unchanged; only the descriptions inside the `inputSchema` are stripped.

**Why this priority**: This is the load-bearing token-economy property. Without it, ADR-005's central claim — ~70% reduction in tool-discovery token cost — does not hold. Every other story in this feature presupposes the stripping utility is correct.

**Independent Test**: With the schema-stripping utility implemented and applied at every tool registration site, an MCP `tools/list` request returns each tool's `inputSchema` with: (a) no `description` key at any nesting level inside the schema's `properties` tree, (b) all `type`, `enum`, `required`, `anyOf`, `oneOf`, `items` structural keys present and identical (modulo the stripped descriptions) to a reference schema produced without stripping. A separate unit test on the utility itself verifies (a) flat-schema stripping, (b) nested-schema stripping (discriminated union, array items, anyOf/oneOf), (c) no-mutation of the input object, and (d) preservation of all structural keys.

**Acceptance Scenarios**:

1. **Given** a flat zod object schema with two fields each carrying `.describe(...)` annotations, **When** the schema is converted via `zod-to-json-schema` and then passed to the stripping utility, **Then** the returned schema's `properties` tree contains no `description` field on either property, but `type`, `required`, and any other structural keys are unchanged.
2. **Given** a nested zod schema (a discriminated union with two branches, each branch containing a `z.array(z.object({...}))` field with `.describe(...)` annotations on inner properties), **When** the converted JSON Schema is passed to the stripping utility, **Then** the returned schema has no `description` field at any depth (top level of branches, array `items`, object `properties` of array elements, discriminator-branch object `properties`) — the strip is recursive and visits every nested schema construct (`anyOf`, `oneOf`, `items`, `additionalProperties`).
3. **Given** a JSON Schema produced from a zod schema that contains zero `.describe(...)` annotations (no descriptions to strip), **When** the schema is passed to the stripping utility, **Then** the returned schema is structurally equivalent to the input (no errors, no mutations, no missing keys).
4. **Given** a JSON Schema object passed to the stripping utility, **When** the utility returns, **Then** the original input object is unchanged — the utility produces a deep copy and never mutates its input. Verifiable by deep-equality check against a pre-call snapshot of the input.
5. **Given** the MCP server's `tools/list` handler responds, **When** an MCP client inspects the returned tool descriptors, **Then** every tool's `inputSchema` has no `description` field at any depth inside the schema's `properties` tree, while every tool's top-level (registration-object-level) `description` field is present and non-empty.
6. **Given** a tool's stripped JSON Schema, **When** it is checked for structural keys, **Then** the schema retains `type`, `properties`, `required` (where the original had it), `enum` arrays, `anyOf`/`oneOf` arrays, `items`, `$ref`, and `additionalProperties` (where present) at every nesting level — the strip removes ONLY the `description` field, nothing else.

### User Story 2 — `help` tool serves a tool's full Markdown documentation on demand (Priority: P1)

A LLM agent that needs the full parameter semantics for `read_note` calls `help({ tool_name: "read_note" })`. The MCP server resolves the request by reading `docs/tools/read_note.md` (relative to the package's installed location, not `process.cwd()`) and returns its contents as the tool's text response. The agent can now construct a correct payload for `read_note` using the full parameter docs, and that documentation stays in the chat history for the rest of the session — meaning subsequent `read_note` calls in the same session do not need to re-invoke `help`.

**Why this priority**: P1 alongside Story 1. Without the `help` tool, the stripped schema (Story 1) is information loss with no recovery path: the agent has no way to learn parameter semantics it does not already know. Story 1 + Story 2 together are the progressive-disclosure pattern; either alone is insufficient.

**Independent Test**: With the `help` tool registered and `docs/tools/help.md`, `docs/tools/index.md`, and at least one stub doc file present, an MCP client calling `help({ tool_name: "help" })` receives the contents of `help.md`, `help({})` receives the contents of `index.md`, and `help({ tool_name: "obsidian_exec" })` receives the contents of `obsidian_exec.md`. Doc-file resolution succeeds when the MCP server is started from a working directory other than the package root (e.g., `cd /tmp && node /path/to/server.js`) — the path resolution is anchored to the module's own location.

**Acceptance Scenarios**:

1. **Given** the `help` tool is registered and `docs/tools/help.md` exists, **When** an MCP client calls `help({ tool_name: "help" })`, **Then** the tool returns the full contents of `docs/tools/help.md` as the response text.
2. **Given** the `help` tool is registered and `docs/tools/index.md` exists, **When** an MCP client calls `help({})` (empty input object) or `help` with no `tool_name` field, **Then** the tool returns the full contents of `docs/tools/index.md` as the response text.
3. **Given** the `help` tool is registered, **When** an MCP client calls `help({ tool_name: "nonexistent_xyz" })`, **Then** the tool fails with an `UpstreamError` whose `code` is `HELP_TOOL_NOT_FOUND` and whose `message` lists the names of the tools whose doc files exist (so the agent can self-correct). The structured error is serialized via the project's existing MCP error-response shape (per Constitution Principle IV / `UpstreamError` contract).
4. **Given** the MCP server is started with the process's current working directory set to a path other than the package root, **When** an MCP client calls `help({ tool_name: "help" })`, **Then** the tool resolves the doc file relative to the module's own location (using e.g. `import.meta.url` or equivalent) and returns the file's contents successfully — `process.cwd()` is NOT used in resolution.
5. **Given** the `help` tool is registered alongside the `obsidian_exec` tool and a stub `docs/tools/obsidian_exec.md` exists, **When** an MCP client calls `help({ tool_name: "obsidian_exec" })`, **Then** the tool returns the contents of the stub file (which carries a TODO marker indicating the per-tool BI will populate it).
6. **Given** the `help` tool's input schema, **When** an MCP client calls `help` with a `tool_name` value that is not a string (e.g., `help({ tool_name: 42 })`), **Then** the tool fails with a structured zod validation error (per the standard tool-boundary validation contract — Constitution Principle III) whose `path` includes `"tool_name"`. This is the standard `VALIDATION_ERROR` code, not `HELP_TOOL_NOT_FOUND`.

### User Story 3 — Top-level tool descriptions are concise verb-led summaries that mention `help()` (Priority: P1)

Every tool registered with the MCP server (the existing `obsidian_exec` tool plus the future typed tools BI-003 through BI-025) carries a top-level `description` field — the string passed to the MCP SDK alongside the schema, which an MCP client sees in `tools/list` as `tools[i].description`. That description MUST be a concise verb-led summary that gives the agent enough information to decide whether to invoke `help()` for full docs, and MUST explicitly mention `help("<tool_name>")` (or substantively equivalent — `help({ tool_name: "<name>" })`) so the agent knows the recovery path is available.

**Why this priority**: P1, not P2. The progressive-disclosure pattern only works if the agent knows it CAN call `help()`. A tool whose top-level description is verbose (defeating the token economy) or is silent on the help recovery path (orphaning the agent when the stripped schema is insufficient) breaks ADR-005's contract — both the token economy and the agent-self-recovery loop depend on this story being satisfied for every registered tool.

**Independent Test**: An MCP client calling `tools/list` on the running server inspects each tool's `description` field and verifies (a) the field is present and non-empty, and (b) the string contains the literal substring `"help"` (case-insensitive match acceptable) AND a reference to invoking it with the tool's own name (e.g., the literal substring `"help("` or `"help({"` followed somewhere by the tool's name). Existing tools (`obsidian_exec`, the `help` tool itself) and any tools landed by future BIs MUST satisfy this contract — verifiable by a registry-level test that iterates over the registered tools.

**Acceptance Scenarios**:

1. **Given** the MCP server is running with the existing tools registered (the `obsidian_exec` tool and the new `help` tool), **When** an MCP client calls `tools/list`, **Then** every returned tool descriptor has a `description` field that is present, non-empty, and mentions `help` invocation as the recovery path for full parameter docs.
2. **Given** the `obsidian_exec` tool's existing top-level description, **When** this BI lands, **Then** the description is updated (or augmented) to satisfy Story 3's contract — verb-led summary plus explicit mention of `help("obsidian_exec")`. The semantic intent of the tool is preserved; only the description string is condensed and the `help` reference is added.
3. **Given** the `help` tool's own top-level description, **When** an MCP client inspects it, **Then** it is a concise verb-led summary that mentions self-invocation via `help("help")` (or `help({ tool_name: "help" })`) for full documentation.

### User Story 4 — Doc directory is bundled with the package and resolved from the module's own location (Priority: P2)

The `docs/tools/` directory and its Markdown files are bundled with the npm release of `obsidian-cli-mcp` so that consumers who install the package via `npm install` (or `npx`) get the docs along with the binary. The `help` tool's path-resolution logic MUST be anchored to the module's own location (e.g., via `import.meta.url` or equivalent) rather than `process.cwd()`, so that the tool works regardless of the working directory the MCP server process was started from.

**Why this priority**: P2 because Stories 1–3 are the contract; this story is the operational property that makes the contract hold once the package is installed. Without bundling, a published consumer's `help` invocation would fail with `HELP_TOOL_NOT_FOUND` for every tool — so the bundling configuration is mandatory for shipping. It is P2 only because it is a packaging concern (a `package.json` `files` field configuration) rather than a runtime-validation concern.

**Independent Test**: Two operational tests verify this story:
- `npm pack` produces a tarball containing `docs/tools/help.md`, `docs/tools/index.md`, and the per-tool stub files. Verifiable by `tar -tzf` on the produced tarball OR by inspecting `npm pack --dry-run` output.
- A test starts the MCP server with `process.chdir()` set to a temporary directory unrelated to the package root, then issues `help({ tool_name: "help" })` against the server, and verifies the response carries the `help.md` contents — proving the resolver is anchored to the module's location, not `cwd`.

**Acceptance Scenarios**:

1. **Given** the project's `package.json` has been updated to include the `docs/tools/` directory in the `files` array (or equivalent npm-publish configuration), **When** `npm pack` is run, **Then** the produced tarball contains `docs/tools/help.md`, `docs/tools/index.md`, `docs/tools/obsidian_exec.md`, and the stub files for any other registered tools.
2. **Given** the MCP server is started from a working directory unrelated to the package root, **When** the `help` tool is invoked for any registered tool whose doc file exists, **Then** the tool returns the file contents successfully — the path resolution does NOT depend on `process.cwd()`.

### Edge Cases

- **`help({ tool_name: "" })`** (empty string): MUST fail with `HELP_TOOL_NOT_FOUND` (no doc file is named `.md`). Reasonable default — empty string is treated as a name that does not resolve, not as the absent case (the absent case is `tool_name === undefined` per Story 2 AC#2). Alternative: reject empty string at the zod boundary as a `VALIDATION_ERROR`. Plan-stage decides which surface raises the failure; the behavioral outcome (the call fails, the agent sees a structured error) binds.
- **`help({ tool_name: "Help" })`** (case mismatch — file is `help.md`): on case-sensitive filesystems (Linux, the typical CI environment, npm publish) MUST fail with `HELP_TOOL_NOT_FOUND`. On case-insensitive filesystems (macOS default, Windows) the lookup may succeed — implementation does NOT add a custom case-folding layer; the behavior follows the underlying filesystem. Documented as a known cross-platform gotcha; the resolution policy is "look up the literal name as a `.md` file in the `docs/tools/` directory."
- **`help({ tool_name: "../../etc/passwd" })`** (path traversal attempt): MUST fail with `HELP_TOOL_NOT_FOUND` and MUST NOT read any file outside `docs/tools/`. Implementation MUST validate that the resolved path is within the `docs/tools/` directory before reading. The error message MUST NOT echo the attacker-provided path (to avoid log-injection). The list of valid tool names included in the error message is constructed by directory-listing `docs/tools/` and filtering to `.md` files (excluding `index.md`), not by reflecting the request.
- **`help({ tool_name: "index" })`**: ambiguous — `index.md` exists but is the index page, not a tool doc. MUST fail with `HELP_TOOL_NOT_FOUND`. The list of valid tool names returned in the error MUST exclude `index` (and `help.md`'s exclusion is a per-tool decision: `help` IS a tool, so it appears in the list). Reasonable default: the valid-names list is the set of `.md` files in `docs/tools/` minus `index.md`.
- **`docs/tools/` directory missing entirely** (e.g., a packaging mistake): the `help` tool's first invocation MUST fail with a structured error. Reasonable default: surface as `HELP_TOOL_NOT_FOUND` with a message indicating the docs directory is missing, OR a distinct code (e.g., `HELP_DOCS_MISSING`). Plan-stage chooses; the behavioral contract (the call fails with an `UpstreamError`, not a silent empty response) binds.
- **A doc file exists but is empty** (zero bytes): MUST succeed and return the empty string. The agent's prompt-engineering layer interprets the empty response. The `help` tool does NOT validate doc-file content beyond reading it.
- **A registered tool has NO corresponding doc file** (e.g., a tool was registered but its doc file was forgotten): `help({ tool_name: "<that_tool>" })` MUST fail with `HELP_TOOL_NOT_FOUND`. Reasonable default — there is no alternative; failing loud is correct. A separate test (a "registry consistency" test) MAY assert that every registered tool has a corresponding doc file at build time, but the runtime behavior on miss is `HELP_TOOL_NOT_FOUND`.
- **A doc file exists but no tool with that name is registered** (e.g., a leftover doc from a removed tool): `help({ tool_name: "<that_name>" })` MUST succeed and return the file contents. The `help` tool does NOT cross-reference the registered-tool list — it operates purely on the filesystem of `docs/tools/`. Rationale: registry-doc consistency is a build-time concern, and the `help` tool's job is to serve docs, not to police drift. (See Assumptions.)
- **Stripping utility called with non-object input** (`null`, `undefined`, `42`, a string): outside scope — the utility's input is the output of `zod-to-json-schema`, which is always a plain object. A defensive guard at the utility boundary is acceptable but not required; the function's typed signature (the parameter is typed as a JSON Schema object) is the contract. Reasonable default: type signature alone, no runtime guard.
- **Stripping utility called with a JSON Schema containing a `description` field that holds a non-string value** (e.g., `description: { foo: "bar" }`): the utility removes the field regardless of value type. Rationale: the strip is keyed on the field name, not on the value's type. A non-string `description` is a malformed JSON Schema, but the strip's job is to remove all `description` fields under `properties` recursively, regardless.
- **Top-level (root) `description` on the JSON Schema input**: MUST be preserved. The "do not strip the top-level schema description" rule from the user input refers to the root of the schema document — strip recursion starts ONE LEVEL down from the root (inside `properties`, `anyOf`, `oneOf`, `items`, `additionalProperties`). The root's `description` field, if present, is left intact. (Note: in the project's tool registration pattern, the agent-facing top-level description is the SDK-level `tools[i].description` field on the registration object, not the JSON Schema's own root `description`. The two are distinct fields; the strip utility leaves both intact — it only touches descriptions inside the schema's `properties` tree and nested constructs.)
- **Tool registration via the SDK that bypasses the stripping utility** (a future regression where a developer forgets to apply the utility): the `tools/list` response would carry that tool's full descriptions, defeating the token economy for that tool but not breaking correctness. This is a process risk, not a runtime failure. Mitigation: a registry-level test (per Story 3 AC#1's iteration) MAY assert the absence of `description` keys in every registered tool's stripped schema, catching the regression at PR review time. Plan-stage decides whether to add the registry-consistency test or rely on per-tool registration tests.
- **`help({ tool_name: "help" })` is invoked but `help.md` itself is missing**: same outcome as any other tool's missing doc — `HELP_TOOL_NOT_FOUND`. The `help` tool is not exempt from its own contract; if its doc file is missing, the call fails the standard way. (A bootstrap test ensures `help.md` exists before the tool is registered, so this case is unreachable in practice.)

## Requirements *(mandatory)*

### Functional Requirements

#### Component 1 — Schema-stripping utility

- **FR-001**: A new module MUST be created with co-located tests per Constitution Principle II that exports a single pure function — the schema-stripping utility — accepting one parameter (a JSON Schema object as produced by `zod-to-json-schema`) and returning a JSON Schema object. The exact module path and exported function name are plan-stage decisions; reasonable default per the per-surface module-layout convention is `src/help/strip-schema.ts` (with tests at `src/help/strip-schema.test.ts`) and an exported function name like `stripDescriptions` or `stripSchemaDescriptions`. The function's typed signature MUST treat the input parameter as a JSON Schema object type (a typed alias for the relevant zod-to-json-schema output) and the return as the same type — no `any` in public signatures (Constitution Technical Standards).
- **FR-002**: The utility MUST recursively remove every `description` field appearing inside the schema's `properties` tree at every nesting level. The recursion MUST visit all of: (a) every property object inside `properties`, (b) the `items` schema of array constructs, (c) every branch object inside `anyOf` and `oneOf` arrays, (d) the `additionalProperties` schema when it is an object, and (e) every nested `properties` tree inside any of the above. Discriminated-union JSON Schema output (typically expressed as `oneOf` with discriminator) MUST have descriptions stripped from every branch. Coverage of every nested construct is verifiable by the nested-schema test scenario (Story 1 AC#2).
- **FR-003**: The utility MUST NOT remove the top-level (root) `description` field on the input schema document. The strip recursion starts one level down — inside `properties`, `anyOf`, `oneOf`, `items`, `additionalProperties`. (See Edge Cases for the distinction between this root-level description and the SDK-level `tools[i].description`.)
- **FR-004**: The utility MUST preserve every other key at every nesting level — `type`, `properties`, `required`, `enum`, `anyOf`, `oneOf`, `items`, `$ref`, `additionalProperties`, `default`, `minimum`, `maximum`, `minLength`, `maxLength`, `pattern`, and any other JSON Schema keys present in the input. The strip is targeted: ONLY `description` is removed, nothing else.
- **FR-005**: The utility MUST be a pure function with no side effects: (a) it MUST NOT mutate its input — the input object after the call MUST deep-equal the input object before the call; (b) it MUST NOT read or write the filesystem; (c) it MUST NOT log; (d) it MUST NOT depend on any global state. The function returns a deep copy of its input with the descriptions removed. The deep-copy guarantee is verifiable by a mutation-safety test (Story 1 AC#4).
- **FR-006**: Every MCP tool registration site in this codebase — the existing `obsidian_exec` registration in [`src/tools/obsidian_exec/tool.ts`](../../src/tools/obsidian_exec/tool.ts), the new `help` tool registration in this BI, and every future typed-tool BI's registration — MUST pass its `zod-to-json-schema` output through this utility before handing the schema to the MCP SDK. A tool that bypasses the utility violates ADR-005's contract; reviewers MUST flag any registration that constructs an `inputSchema` without the strip applied.

#### Component 2 — `help` MCP tool

- **FR-007**: A new MCP tool named `help` MUST be registered with the MCP server. The tool's name (the string passed to the SDK as the tool's identifier) is the literal `"help"`. The tool's input schema is a single zod object schema with one field: `tool_name: z.string().optional()` (optional string). Per Constitution Principle III, the zod schema is the single source of truth for both the runtime parse and the SDK-published `inputSchema` (after the strip utility per FR-006).
- **FR-008**: The `help` tool's behavior on input is:
    - When `tool_name` is provided as a non-empty string AND `docs/tools/<tool_name>.md` exists relative to the module's own location, the tool returns the full contents of that file as the response text.
    - When `tool_name` is omitted (the input object has no `tool_name` field) OR the input object is empty (`{}`), the tool returns the full contents of `docs/tools/index.md` as the response text.
    - When `tool_name` is provided as a non-empty string but `docs/tools/<tool_name>.md` does NOT exist (or is not a regular file, or is outside the `docs/tools/` directory after path resolution), the tool fails with an `UpstreamError` whose `code` is the literal string `HELP_TOOL_NOT_FOUND`, whose `details` includes the requested name and the list of available tool names (the `.md` files in `docs/tools/`, excluding `index.md`), and whose `message` lists the available tool names so the agent can self-correct. The error MUST be serialized via the project's existing MCP error-response shape (per Constitution Principle IV's `UpstreamError` serialization contract).
    - When `tool_name` is provided as an empty string (`""`): the call MUST fail. The exact failure surface (zod `VALIDATION_ERROR` versus filesystem `HELP_TOOL_NOT_FOUND`) is a plan-stage decision per Edge Cases; reasonable default is `HELP_TOOL_NOT_FOUND`.
- **FR-009**: The `help` tool's doc-file resolution MUST be anchored to the module's own location (e.g., via `import.meta.url` resolved through `node:url`'s `fileURLToPath` and `node:path`'s `dirname`/`resolve`) rather than `process.cwd()`. The `help` tool MUST work correctly when the MCP server process is started with a working directory other than the package root — verifiable by Story 4 AC#2.
- **FR-010**: The `help` tool's path resolution MUST defend against path-traversal attempts: a `tool_name` value containing `..`, `/`, `\`, or any other path-segment characters that would resolve outside the `docs/tools/` directory MUST cause the lookup to fail with `HELP_TOOL_NOT_FOUND` (the same failure as a non-existent tool, so a probe cannot distinguish "no such tool" from "you tried to escape"). Implementation MUST verify that the fully-resolved path is within `docs/tools/` before any filesystem read. The error message MUST NOT echo the attacker-provided `tool_name` value (Edge Cases).
- **FR-011**: The `help` tool's failure surfaces — `HELP_TOOL_NOT_FOUND`, schema validation failures, any unexpected I/O errors — MUST flow through the `UpstreamError` class defined in [`src/errors.ts`](../../src/errors.ts), serialized via the project's existing tool-error-response shape. Plain `throw new Error("…")` is a Constitution Principle IV violation. The `code`, `cause`, and `details` fields of the error MUST be populated per the existing `UpstreamErrorArgs` contract.

#### Documentation files

- **FR-012**: A `docs/tools/` directory MUST be created at the package root, containing:
    - `index.md` — the index page that lists all available tool documentation files with a one-line description per tool. The `index.md` is the response to `help({})`.
    - `help.md` — full Markdown documentation for the `help` tool itself: input schema, output behavior on each branch (named tool, omitted name, unknown name), error codes, and a usage example. This file is the response to `help({ tool_name: "help" })`.
    - `obsidian_exec.md` — full Markdown documentation for the `obsidian_exec` tool. Reasonable default: this BI may either ship a non-stub file (since the tool exists today and its behavior is documented in [001's contracts](../001-add-cli-bridge/contracts/)) OR a stub with a TODO marker pointing at the existing 001 contracts. Plan-stage decides.
    - One stub `<tool_name>.md` file per other tool that will be registered in future BIs (BI-003 through BI-025: at least `read_note.md`, `write_note.md`, `append_note.md`, `search_vault.md`, `list_notes.md` — the exact list is a plan-stage decision, but every tool that will be registered before the help tool BI ships its `1.0` MUST have a stub). Each stub MUST carry an explicit TODO marker (e.g., the literal string `<!-- TODO(BI-XXX): populate this doc -->`) so the unfilled docs are mechanically discoverable.
- **FR-013**: The `index.md` file MUST list every tool that has a corresponding `<tool_name>.md` file in `docs/tools/`, with one line per tool naming the tool and giving a one-line summary. The summary may be authored by hand for the tools that exist today (`help`, `obsidian_exec`); for stubbed tools it MAY be a placeholder mirroring the stub TODO marker. A tool whose doc file is a stub MUST still be listed in `index.md` so that `help({})` discloses the surface to the agent — the stub's existence, not its content, is the listing trigger.
- **FR-014**: The `package.json` MUST be updated to ensure the `docs/tools/` directory is included in the npm-publish artifact. Reasonable default: add `"docs/tools/**/*.md"` to the `files` array in `package.json`. Verifiable by Story 4 AC#1 (`npm pack` includes the docs).
- **FR-015**: Existing tools registered before this BI ships — currently just `obsidian_exec` — MUST have their top-level descriptions updated (or augmented) to satisfy Story 3's contract: concise verb-led summary plus explicit mention of `help("<tool_name>")` as the recovery path for full parameter docs. The semantic intent of each tool's description is preserved; only the description string is condensed and the `help` reference is added. The exact verbatim wording is a plan-stage decision; the structural contract (summary + `help` mention) binds.
- **FR-016**: The `help` tool's own top-level description MUST be a concise verb-led summary that mentions self-invocation via `help("help")` (or `help({ tool_name: "help" })`) for the tool's own full documentation, AND mentions that calling with no arguments returns the index of all available docs.

#### Test, header, and gating requirements

- **FR-017**: Tests for both components MUST be co-located per Constitution Principle II. The minimum test set (per the user input's acceptance criterion #10):
    - Schema-stripping utility — happy-path flat (Story 1 AC#1), happy-path nested (Story 1 AC#2), boundary-path no descriptions to strip (Story 1 AC#3), mutation safety (Story 1 AC#4).
    - `help` tool — happy-path named tool (Story 2 AC#1), happy-path index (Story 2 AC#2), failure-path unknown tool (Story 2 AC#3).
    - Optional but recommended additions: cwd-independence (Story 2 AC#4 / Story 4 AC#2) and structural-key preservation (Story 1 AC#6). Reviewers MAY add more.
- **FR-018**: All new modules — the schema-stripping utility, its test file, the `help` tool registration, the `help` tool handler (if separated), the `help` tool schema (if separated), and any helper modules — MUST carry an original-contribution header per Constitution Principle V, of the form `// Original — no upstream. <one-line description>.` (per the user input's acceptance criterion #11).
- **FR-019**: Every Markdown file in `docs/tools/` (`help.md`, `index.md`, the stub files) is documentation, not code, and is exempt from the source-code header convention. However, stub files MUST carry the explicit TODO marker per FR-012 so they are mechanically discoverable.
- **FR-020**: The vitest aggregate statements coverage threshold (the merge gate per Constitution v1.1.0 §Development Workflow #5) MUST remain at or above the floor in effect when this feature merges. The new tests MUST NOT cause coverage to drop below that floor.
- **FR-021**: The PR landing this feature MUST update the Constitution Compliance checklist with one Y/N/N/A per principle (per Constitution v1.1.0 §Development Workflow #8). All five principles are expected to evaluate as: Principle I (`Y` — components live in their own modules with co-located tests, tool → handler → utility direction), Principle II (`Y` — co-located tests covering the per-component happy + failure + boundary set), Principle III (`Y` — `help` tool input is zod-validated, types via `z.infer`), Principle IV (`Y` — `help` tool surfaces failures via `UpstreamError`), Principle V (`Y` — original-contribution headers on every new source file).

### Out of Scope

The following are explicitly excluded from this feature and remain the responsibility of separate BIs or future work:

- **Populating `docs/tools/<tool>.md` with real content for tools that do not yet exist**: each public typed tool BI (BI-003 through BI-025) owns its own doc file and populates the body when it ships. This BI ships stubs for them (per FR-012). The `help` tool itself and `index.md` are populated in this BI; `obsidian_exec.md` is populated in this BI (or stubbed pointing at 001's contracts — plan-stage decision).
- **Versioned doc files** (e.g., `read_note.v1.md`, `read_note.v2.md`): documentation is bundled with the npm release; the agent gets the docs that match the package version. Per-tool versioning, if ever desired, lands as a separate spec.
- **Localisation**: `docs/tools/*.md` files are English-only. If localisation is desired, it lands as a separate spec with its own resolution policy.
- **Cross-session persistence of `help` output**: the LLM's context window handles within-session retention natively. The MCP server does NOT cache or persist `help` responses across sessions; each session that needs the docs invokes `help` itself.
- **Enforcing a maximum word count or token-cost ceiling on top-level tool descriptions** (FR-015 / FR-016): the user input states that the conciseness guidance is advisory, not a schema-level constraint. Reviewers verify "concise" by inspection; the build does not measure the description string's length. A future spec MAY add a mechanical cap if drift becomes a problem.
- **Refactoring `obsidian_exec`'s existing schema or handler**: only the registration's `inputSchema` (now passed through the strip utility) and the top-level `description` (now condensed and mentions `help`) change. The schema fields, handler logic, and error contract are unchanged.
- **A registry-consistency test that asserts every registered tool has a corresponding doc file**: optional per Edge Cases. If added, it lives in `src/server.test.ts` or similar; if not, the runtime behavior on registry-doc drift is the standard `HELP_TOOL_NOT_FOUND` failure. Plan-stage decides.
- **A dynamic `help` invocation that introspects the registered tool list and emits `index.md` content from the registry rather than from a static file**: not in this BI. The `index.md` is a static file; introspection-based help is a future enhancement that would land with its own spec.
- **Modifying ADR-005**: this feature implements ADR-005; it does not change it. If the implementation surfaces a contradiction with the ADR, the resolution is a new ADR per the project's amendment procedure, not a silent override.

### Key Entities *(include if data involved)*

- **Schema-stripping utility (`src/help/strip-schema.ts` — module path is a plan-stage decision)**: a single pure function. Input: a JSON Schema object as produced by `zod-to-json-schema`. Output: a deep-copy JSON Schema object with all `description` fields below the root removed. No filesystem, no network, no logging, no mutation. Used at every MCP tool registration site.
- **`help` MCP tool**: a public MCP tool registered with the SDK alongside `obsidian_exec`. Input schema — a zod object with one field `tool_name: z.string().optional()`. Output — the contents of a Markdown file from the bundled `docs/tools/` directory, OR a structured `UpstreamError` with code `HELP_TOOL_NOT_FOUND` when the requested tool's doc file does not exist. Path resolution is anchored to the module's own location (not `process.cwd()`).
- **`docs/tools/` directory**: a package-bundled directory containing one Markdown file per registered tool, plus `help.md` (the help tool's own documentation), plus `index.md` (the listing of all available docs). Bundled with the npm release via `package.json`'s `files` configuration. Per-tool BIs own the per-tool file's body; this BI ships the directory, `help.md`, `index.md`, `obsidian_exec.md`, and stubs for the future tools.
- **`HELP_TOOL_NOT_FOUND` error code**: a stable string identifier on the `UpstreamError` class used by the `help` tool when a requested tool's doc file does not exist or resolves outside `docs/tools/`. The error's `details` carries the requested name and the list of available tool names; the error's `message` lists the available tool names.
- **Top-level tool descriptions**: the `description` strings attached to each tool's MCP-SDK registration object (the value the SDK puts at `tools[i].description` in the `tools/list` response). Distinct from the JSON Schema's own root `description` (which the strip utility leaves intact). All top-level descriptions, post-this-BI, are concise verb-led summaries that mention `help("<tool_name>")` as the recovery path for full parameter docs.

## Assumptions

- **The token-economy claim from ADR-005 (~70% reduction)** is approximately correct for this codebase's eventual full surface (20+ tools). Empirical validation lands as part of this BI (SC-006); if the realized reduction is materially below the claimed number, plan-stage records the discrepancy but does NOT block on it — the strip utility's correctness is independent of the realized reduction.
- **The `zod-to-json-schema` library** is the JSON Schema generator the project uses for MCP `inputSchema` registration, per the existing `obsidian_exec` registration in [`src/tools/obsidian_exec/schema.ts`](../../src/tools/obsidian_exec/schema.ts) and the Constitution's MCP and Validation rules. If a different generator is adopted in the future, the strip utility's input-shape contract (FR-001) re-targets to that generator's output.
- **JSON Schema's `description` field** is the only field name targeted by the strip — `title` and other documentation-adjacent fields are NOT stripped. The user input is explicit on `description`; broader semantic-stripping (e.g., stripping `title` too, or stripping `examples`) is not in scope.
- **The MCP SDK's `tools/list` handler** publishes the `inputSchema` exactly as registered, without further mutation. The strip utility's effect at registration time is therefore observable end-to-end via `tools/list`. Verifiable by Story 1 AC#5.
- **The `help` tool is itself registered through the same registration pipeline** that applies the strip utility — meaning `help`'s own input schema (a single optional string field) appears in `tools/list` with no descriptions inside its properties tree. The `help` tool is not exempt from the strip; the strip leaves the SDK-level top-level description intact (where the agent sees the `help("<tool_name>")` mention), so the agent can still discover the help tool's purpose from its top-level description alone.
- **`docs/tools/*.md` files are read as UTF-8 text** — Reasonable default. Non-UTF-8 file content would be a packaging mistake; the `help` tool does not transcode.
- **Doc file resolution is filesystem-based, not registry-based**: the `help` tool's source of truth for "what tools have docs" is the contents of `docs/tools/`, NOT the SDK's registered-tool list. A registered tool whose doc is missing fails with `HELP_TOOL_NOT_FOUND`; an orphaned doc with no registered tool succeeds and returns the file. Registry-doc consistency is a build-time / review-time concern, not a runtime concern (per Edge Cases).
- **The error code namespace is the project's existing `UpstreamError` codespace** (per [001's errors contract](../001-add-cli-bridge/contracts/)). `HELP_TOOL_NOT_FOUND` is a new code added by this BI and lives in the same codespace. No code-collision risk with existing codes (the existing codes are CLI-execution-focused; this is a tool-resolution code).
- **The `obsidian_exec` tool is the only tool registered today**, and the typed-tool BIs (BI-003 through BI-025) have not yet shipped. This BI's stub files cover the to-be-shipped tools; each typed-tool BI populates its own stub when it lands. The set of stubbed tools is a plan-stage decision based on the BI roster in [Architecture](../../.architecture/Obsidian%20CLI%20MCP%20-%20Architecture.md).
- **The `help` tool is itself a public MCP tool subject to Constitution Principle II's test contract**: it ships with at least one happy-path test AND one failure-or-boundary test (per the user input's acceptance criterion #10). The minimum test set is named in FR-017.
- **Constitution Principle III binds the help tool's input validation**: the zod schema is the single source of truth, and the SDK-published `inputSchema` is derived from the same zod schema (post-strip) — there is no parallel hand-written interface for `help`'s input.
- **Constitution Principle IV binds the help tool's error contract**: every failure surface flows through `UpstreamError` with a stable `code` string. `HELP_TOOL_NOT_FOUND` is the only new code introduced by this BI; zod validation failures use the existing `VALIDATION_ERROR` code.
- **Constitution Principle V binds every new source file** to carry an original-contribution header. Markdown documentation files in `docs/tools/` are exempt from the source-code header convention (per FR-019) but stub files MUST carry the explicit TODO marker (per FR-012).
