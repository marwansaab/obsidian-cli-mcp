## Summary

Add three independent typed MCP tools тАФ `bases` (vault-wide `.base` file enumeration, BI-0049), `views_base` (view enumeration within the focused `.base` file, BI-0082), and `create_base` (item creation within a base, BI-0083) тАФ completing the Obsidian Bases surface alongside the already-shipped `query_base`. Each tool is independently shippable with its own co-located test set.

**Scale/Scope**: Three new tool modules (9 production files + 9 test files), one `server.ts` boot-spine addition.

## Implementation Strategy

### Tool 1: `bases` (BI-0049)

**Handler pipeline** (3 stages):
1. **Vault resolution** тАФ resolve vault if provided (parity with `query_base`), noting R-001 silent-ignore.
2. **CLI invocation** тАФ `invokeCli({ command: "bases", parameters: {}, flags: [], target_mode: "active" })`. Parse stdout: split by `\n`, filter empty lines.
3. **Post-process** тАФ sort paths lexicographically (FR-005), construct output envelope `{ bases, count }`.

**Error classification**: CLI failures classified under `CLI_REPORTED_ERROR`. Empty stdout тЖТ `{ bases: [], count: 0 }` (not an error).

### Tool 2: `views_base` (BI-0082)

**Handler pipeline** (3 stages):
1. **Vault resolution** тАФ same as `bases`, noting R-003 silent-ignore.
2. **CLI invocation** тАФ `invokeCli({ command: "base:views", parameters: {}, flags: [], target_mode: "active" })`. Parse stdout: split by `\n`, filter empty lines.
3. **Post-process** тАФ construct output envelope `{ views, count }`.

**Error classification**: "Active file is not a base file:" pattern тЖТ `CLI_REPORTED_ERROR` with contextual `details.code`. Empty stdout тЖТ `{ views: [], count: 0 }`.

**Key limitation**: Active-mode-only. No path input. Tool description MUST state this limitation prominently.

### Tool 3: `create_base` (BI-0083)

**Handler pipeline** (5 stages):
1. **Input validation** тАФ `path` (base path validation), `name` (1тАУ1000 chars, non-empty), `content` size pre-check against platform argv ceiling.
2. **Vault resolution** тАФ same as `bases`, noting R-004 silent-ignore.
3. **CLI invocation** тАФ `invokeCli({ command: "base:create", parameters: { path, name, content?, view? }, flags: [], target_mode: "specific" })`.
4. **Parse response** тАФ extract filename from `Created: <filename>.md` pattern.
5. **Construct path** тАФ derive vault-relative path from base directory + returned filename.

**Error classification**: "Base file not found:" тЖТ `CLI_REPORTED_ERROR` / `BASE_NOT_FOUND`. Content over limit тЖТ `VALIDATION_ERROR` / `CONTENT_TOO_LARGE`.

### Cross-cutting: server.ts wiring

Add three imports and three `createXTool(deps)` calls in `server.ts`, following the existing pattern for `createQueryBaseTool`. Update `_register-baseline.json` with three new entries.

### Cross-cutting: documentation

Each tool gets a `docs/tools/<name>.md` help doc with:
- Worked examples (happy path, error cases)
- Error roster (all `details.code` values)
- Known limitations (vault routing, active-mode-only for `views_base`)
- Cohort cross-references (Bases-family: `bases`, `query_base`, `views_base`, `create_base`)
