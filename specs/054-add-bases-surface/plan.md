# Implementation Plan: Add Bases Surface

**Branch**: `054-add-bases-surface` | **Date**: 2026-05-28 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `specs/054-add-bases-surface/spec.md`

## Summary

Add three independent typed MCP tools вЂ” `bases` (vault-wide `.base` file enumeration, BI-0049), `views_base` (view enumeration within the focused `.base` file, BI-0082), and `create_base` (item creation within a base, BI-0083) вЂ” completing the Obsidian Bases surface alongside the already-shipped `query_base`. Each tool is independently shippable with its own co-located test set.

## Technical Context

**Language/Version**: TypeScript, strict mode, Node.js >= 22.11
**Primary Dependencies**: `@modelcontextprotocol/sdk`, `zod`, `zod-to-json-schema`
**Storage**: N/A (CLI passthrough to Obsidian vault)
**Testing**: vitest with `@vitest/coverage-v8`, co-located `*.test.ts` files
**Target Platform**: Cross-platform (Windows, Linux, macOS)
**Project Type**: MCP server (CLI wrapper)
**Performance Goals**: 10s timeout per CLI call (TYPED_TOOL_TIMEOUT_MS), 10 MiB output cap
**Constraints**: Zero new top-level error codes (Constitution Principle IV). Three tools independently shippable.
**Scale/Scope**: Three new tool modules (9 production files + 9 test files), one `server.ts` boot-spine addition.

## Implementation Strategy

### Tool 1: `bases` (BI-0049)

**Handler pipeline** (3 stages):
1. **Vault resolution** вЂ” resolve vault if provided (parity with `query_base`), noting R-001 silent-ignore.
2. **CLI invocation** вЂ” `invokeCli({ command: "bases", parameters: {}, flags: [], target_mode: "active" })`. Parse stdout: split by `\n`, filter empty lines.
3. **Post-process** вЂ” sort paths lexicographically (FR-005), construct output envelope `{ bases, count }`.

**Error classification**: CLI failures classified under `CLI_REPORTED_ERROR`. Empty stdout в†’ `{ bases: [], count: 0 }` (not an error).

### Tool 2: `views_base` (BI-0082)

**Handler pipeline** (3 stages):
1. **Vault resolution** вЂ” same as `bases`, noting R-003 silent-ignore.
2. **CLI invocation** вЂ” `invokeCli({ command: "base:views", parameters: {}, flags: [], target_mode: "active" })`. Parse stdout: split by `\n`, filter empty lines.
3. **Post-process** вЂ” construct output envelope `{ views, count }`.

**Error classification**: "Active file is not a base file:" pattern в†’ `CLI_REPORTED_ERROR` / `BASE_NOT_FOUND` (reuse existing sub-discriminator). Empty stdout в†’ `{ views: [], count: 0 }`.

**Key limitation**: Active-mode-only. No path input. Tool description MUST state this limitation prominently.

### Tool 3: `create_base` (BI-0083)

**Handler pipeline** (5 stages):
1. **Input validation** вЂ” `path` (base path validation), `name` (1вЂ“1000 chars, non-empty), `content` size pre-check against `MAX_CONTENT_LENGTH` (3072 UTF-16 code units, parity with `prepend`).
2. **Vault resolution** вЂ” same as `bases`, noting R-004 silent-ignore.
3. **CLI invocation** вЂ” `invokeCli({ command: "base:create", parameters: { path, name, content?, view? }, flags: [], target_mode: "specific" })`.
4. **Parse response** вЂ” extract filename from `Created: <filename>.md` pattern.
5. **Construct path** вЂ” derive vault-relative path from base directory + returned filename.

**Error classification**: "Base file not found:" в†’ `CLI_REPORTED_ERROR` / `BASE_NOT_FOUND`. Content over 3072 chars в†’ `VALIDATION_ERROR` / `CONTENT_TOO_LARGE`.

### Cross-cutting: server.ts wiring

Add three imports and three `createXTool(deps)` calls in `server.ts`, following the existing pattern for `createQueryBaseTool`. Update `_register-baseline.json` with three new entries.

### Cross-cutting: documentation

Each tool gets a `docs/tools/<name>.md` help doc with:
- Worked examples (happy path, error cases)
- Error roster (all `details.code` values)
- Known limitations (vault routing, active-mode-only for `views_base`)
- Cohort cross-references (Bases-family: `bases`, `query_base`, `views_base`, `create_base`)
