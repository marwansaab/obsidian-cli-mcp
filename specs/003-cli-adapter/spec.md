# Feature Specification: CLI Adapter

**Feature Branch**: `003-cli-adapter`
**Created**: 2026-05-05
**Status**: Draft
**Input**: User description: "Add CLI Adapter — A centralised internal module that all typed MCP tool handlers route through to execute the Obsidian CLI binary. This is NOT a public MCP tool — it is a shared primitive that encapsulates CLI invocation, target-mode argument stripping, and error detection in one place so that individual tool handlers stay thin. Composes on top of the error-detection contract shipped in BI-026."

## Background *(non-mandatory context)*

The bridge tool `obsidian_exec` (feature 001-add-cli-bridge) and the in-band stdout-error detection it gained in feature 002-detect-cli-errors together establish how this project talks to the Obsidian Integrated CLI: argv-as-array spawn (no shell), positional command followed by `key=value` parameter pairs and bare-word flags, exit-code classification, and the post-exit `Error:` prefix check on stdout. None of that mechanism is currently centralised — it lives inline in `src/tools/obsidian_exec/handler.ts`. The next phase of the project (typed MCP tools — `read_note`, `write_note`, `search_vault`, etc.) will add multiple thin handlers, each of which must produce identical argv quirks (vault hoisting), identical error classification, and identical handling of the `Error: no active file` failure mode that arises when an agent calls a focused-note operation against a vault with no open note. ADR-004 commits the project to a single internal CLI adapter module so that this plumbing is implemented once and reused by every typed tool. This spec is the introduction of that adapter.

The adapter is **internal**: it is not registered as an MCP tool, has no zod schema of its own (callers validate at their own boundary per Principle III), and has no slash in the public surface. Typed tool handlers import the adapter, hand it a structured request, and receive either `{ stdout, stderr }` or a structured `UpstreamError`. Tool handlers MUST NOT inspect raw stdout for error signals — that is the adapter's job, single-sourced. The adapter introduces one new stable error code (`ERR_NO_ACTIVE_FILE`) for the focused-note-missing case, alongside reuse of the two CLI error codes shipped earlier (`CLI_NON_ZERO_EXIT` from BI-001 and `CLI_REPORTED_ERROR` from BI-026).

## Clarifications

### Session 2026-05-05

- Q: Adapter call signature & spawn test-seam injection mechanism? → A: Two-arg signature mirroring `executeObsidianExec` — `invokeCli(input: { command, parameters, flags, target_mode }, deps?: { spawnFn?, env? })`. Tests inject via `deps.spawnFn`.
- Q: Name of the new stable error code for the focused-note-missing case? → A: `ERR_NO_ACTIVE_FILE` — matching the deliberate naming in [ADR-004 - Centralized Obsidian CLI Adapter](../../.decisions/ADR-004%20-%20Centralized%20Obsidian%20CLI%20Adapter.md) and [the Architecture document](../../.architecture/Obsidian%20CLI%20MCP%20-%20Architecture.md). The `ERR_*` prefix marks this code as semantically distinct from the `CLI_*` failure family: `ERR_NO_ACTIVE_FILE` is a recoverable user-action signal (the user/agent corrects by opening a note or switching to `target_mode: "specific"`), not a CLI-process failure. An initial answer renamed to `CLI_NO_ACTIVE_FILE` for prefix uniformity but was reversed in /speckit-plan when the conflict with ADR-004 surfaced — staying with the ADR's deliberate semantics.
- Q: When the child terminates by signal (`code === null`, `signal !== null`), what value lands in `details.exitCode` and `cause.exitCode`? → A: Sentinel `-1`, mirroring the existing `executeObsidianExec` precedent at `src/tools/obsidian_exec/handler.ts:238` (`const exitCode = code ?? -1`). `details.signal` carries the signal name.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Specific-mode CLI invocation routes through one centralised primitive (Priority: P1)

A typed tool handler (e.g., the future `read_note`) accepts a validated zod-typed input and needs to invoke the Obsidian CLI with `vault`, `file`, and other key=value parameters. Instead of spawning the binary directly and re-implementing argv assembly, the handler hands a structured request to the CLI adapter — `command`, a `parameters` record, a `flags` array, and `target_mode: "specific"` — and receives `{ stdout, stderr }` on success. The adapter is the single place in the codebase where the CLI's documented argv conventions (positional command first, `vault=…` hoisted to be the first key=value token, remaining `key=value` pairs, then bare-word flags) are encoded.

**Why this priority**: This is the primary value of the feature. Without a centralised primitive, every typed tool handler re-implements argv assembly, the vault-hoisting rule, and stream collection — divergence is inevitable, and a quirk-fix in one handler doesn't propagate to the others. A single adapter is the only way the project can ship more than one typed tool without duplicating the contract.

**Independent Test**: With a stub spawn function injected, calling the adapter with `command: "read"`, `parameters: { vault: "MyVault", file: "Note" }`, `flags: []`, `target_mode: "specific"` against a stub child that exits `0` with stdout `"# Note body\n"` results in (a) the stub spawn being invoked with the binary `"obsidian"` and argv `["read", "vault=MyVault", "file=Note"]` (in exactly that order), and (b) the adapter resolving with `{ stdout: "# Note body\n", stderr: "" }`. No typed tool handler is required for this test — the adapter is exercised in isolation by injecting a synthetic `spawnFn`.

**Acceptance Scenarios**:

1. **Given** a stub spawn function and a stub child that exits `0` with stdout `"OK\n"`, **When** the caller invokes the adapter with `command: "read"`, `parameters: { vault: "MyVault", file: "Note" }`, `flags: []`, `target_mode: "specific"`, **Then** the spawn function is called with argv `["read", "vault=MyVault", "file=Note"]` and the adapter resolves with `{ stdout: "OK\n", stderr: "" }`.
2. **Given** a stub spawn function and a stub child that exits `0`, **When** the caller invokes the adapter with `parameters: { vault: "V" }` and `flags: ["--copy"]`, **Then** argv is `["<command>", "vault=V", "--copy"]` — flags appear as bare-word tokens after all key=value pairs.
3. **Given** a stub spawn function and a stub child that exits `0`, **When** the caller invokes the adapter with `parameters: { vault: "V", file: undefined, query: "q" }`, **Then** argv is `["<command>", "vault=V", "query=q"]` — the `file` entry whose value is `undefined` produces zero tokens in argv.
4. **Given** a stub spawn function, **When** the caller invokes the adapter with `parameters: { file: "F", vault: "V" }` (vault second in the literal object), **Then** argv hoists `vault=V` to the first key=value position so the result is `["<command>", "vault=V", "file=F"]` — vault-first is invariant regardless of insertion order in the input record.

### User Story 2 — Active-mode invocations strip target-locator keys before assembling argv (Priority: P1)

When the caller signals `target_mode: "active"`, the adapter strips the keys `vault`, `file`, and `path` from the `parameters` record before assembling argv. The CLI's "active" semantics — operate on the currently focused note — require that no target-locator parameters be forwarded; doing so would either contradict or silently override the focused-note convention. By centralising the strip in the adapter, every typed tool handler that supports an active-mode call benefits without each one having to remember the rule.

**Why this priority**: P1 alongside Story 1, not P2, because the most common failure when an agent uses active mode is forwarding a stale `vault=…` key from a prior call shape — the CLI's behavior in that case is unspecified and varies between Obsidian versions. The strip is a behavior-correctness invariant, not a convenience.

**Independent Test**: With a stub spawn function injected, calling the adapter with `command: "read"`, `parameters: { vault: "MyVault", file: "Note" }`, `flags: []`, `target_mode: "active"` against a stub child that exits `0` results in the stub spawn being invoked with argv `["read"]` exactly — no `vault=…` token, no `file=…` token. The same call with `parameters: { path: "P", lines: 5 }, target_mode: "active"` results in argv `["read", "lines=5"]` — `path` is stripped, `lines` (a non-target-locator key) is preserved.

**Acceptance Scenarios**:

1. **Given** a stub spawn function and `target_mode: "active"`, **When** the caller invokes the adapter with `parameters: { vault: "MyVault", file: "Note" }`, **Then** argv is `["<command>"]` — both target-locator keys are stripped, no key=value tokens remain.
2. **Given** a stub spawn function and `target_mode: "active"`, **When** the caller invokes the adapter with `parameters: { path: "some/path.md", query: "term" }`, **Then** argv is `["<command>", "query=term"]` — `path` is stripped; `query` (not a target-locator key) is preserved.
3. **Given** a stub spawn function and `target_mode: "active"`, **When** the caller invokes the adapter with `parameters: {}` (empty record) and `flags: []`, **Then** argv is `["<command>"]` — the empty record contributes no tokens, the empty flag list contributes no tokens.
4. **Given** a stub spawn function and `target_mode: "active"`, **When** the caller invokes the adapter with `parameters: { vault: "V", custom: "value" }` and `flags: ["--copy"]`, **Then** argv is `["<command>", "custom=value", "--copy"]` — `vault` is stripped, `custom` is forwarded, the flag is appended bare-word at the end.

### User Story 3 — Layered error classification surfaces the right `UpstreamError` for each failure mode (Priority: P1)

When the spawned CLI process closes, the adapter classifies the outcome in this strict priority order: (a) non-zero exit code → `CLI_NON_ZERO_EXIT`; (b) exit `0` with stdout starting with `Error: no active file` → `ERR_NO_ACTIVE_FILE`; (c) exit `0` with stdout starting with `Error:` (any other suffix) → `CLI_REPORTED_ERROR`; (d) otherwise resolve with `{ stdout, stderr }`. Each thrown `UpstreamError` preserves the input `command`, the captured `stdout` and `stderr`, and the exit-code state — so that the calling tool handler has full diagnostic context without re-running the call.

**Why this priority**: P1 because the typed tools depend on this classification to map adapter errors to MCP error responses. Without correct classification, either (i) `ERR_NO_ACTIVE_FILE` (the most common failure for agents using active mode against an empty Obsidian session) is misclassified as a generic CLI error and the agent receives a useless message instead of a recovery instruction, or (ii) genuine non-zero crashes are misclassified as in-band CLI errors. Both are silent regressions.

**Independent Test**: With three stub spawn outcomes — (a) child exits `1` with `stderr: "boom"`, (b) child exits `0` with `stdout: "Error: no active file\n"`, (c) child exits `0` with `stdout: "Error: File not found\n"` — three identically-shaped adapter calls produce three distinctly-coded `UpstreamError`s: `CLI_NON_ZERO_EXIT`, `ERR_NO_ACTIVE_FILE`, `CLI_REPORTED_ERROR`. A fourth stub outcome where the child exits `0` with `stdout: "OK\n"` resolves successfully. All four tests run with no real CLI binary involved.

**Acceptance Scenarios**:

1. **Given** a stub child that exits with code `1` and `stderr: "boom"`, **When** the adapter is invoked, **Then** it rejects with `UpstreamError` whose `code` is `"CLI_NON_ZERO_EXIT"`, `cause` is `{ exitCode: 1, signal: null }`, and `details` is `{ command, stdout, stderr: "boom", exitCode: 1, signal: null }`.
2. **Given** a stub child that exits with code `0` and `stdout: "Error: no active file\n"`, **When** the adapter is invoked, **Then** it rejects with `UpstreamError` whose `code` is `"ERR_NO_ACTIVE_FILE"`, `cause` is `null`, and `details` is `{ command, stdout: "Error: no active file\n", stderr: "", exitCode: 0, message: "Error: no active file" }`. The `UpstreamError`'s `.message` property MUST instruct the caller to either open a file in Obsidian or switch the call to `target_mode: "specific"`.
3. **Given** a stub child that exits with code `0` and `stdout: "Error: File not found\n"`, **When** the adapter is invoked, **Then** it rejects with `UpstreamError` whose `code` is `"CLI_REPORTED_ERROR"`, `cause` is `null`, and `details` is `{ command, stdout: "Error: File not found\n", stderr: "", exitCode: 0, message: "Error: File not found" }`.
4. **Given** a stub child that exits with code `0` and `stdout: "OK\n"`, **When** the adapter is invoked, **Then** it resolves with `{ stdout: "OK\n", stderr: "" }` — no `UpstreamError` is raised.
5. **Given** a stub child that exits with code `1` and stdout that happens to start with `"Error: no active file"`, **When** the adapter is invoked, **Then** it rejects with `CLI_NON_ZERO_EXIT` (priority a) — the non-zero exit dominates the stdout-prefix detection. Exit-code precedence applies; `ERR_NO_ACTIVE_FILE` and `CLI_REPORTED_ERROR` only fire on exit `0`.
6. **Given** a stub child that exits with code `0` and stdout `"Error: no active file. Open one or use specific mode.\n"`, **When** the adapter is invoked, **Then** it rejects with `ERR_NO_ACTIVE_FILE` (priority b) — the longer suffix does not change the prefix anchor; priority b dominates priority c.

### User Story 4 — Re-exported `UpstreamError` keeps tool-handler imports compact (Priority: P3)

Typed tool handlers consuming the adapter receive `UpstreamError` instances on rejection. They need both the runtime function (to call the CLI) and the class (to `instanceof`-check the error in their own catch blocks). The adapter module re-exports the project's `UpstreamError` from `src/errors.ts` so that consumers have one import line instead of two.

**Why this priority**: P3 because it is a developer-ergonomics convenience, not a correctness requirement — typed tool handlers could equally import from both modules. But Principle I's modular-import-direction rule (`command/tool → service → adapter → external SDK`) is more readable when the adapter exposes the full surface its consumers need.

**Independent Test**: A consumer module compiles and runs with a single import line of the form `import { invokeCli, UpstreamError } from "<adapter-module-path>"` — no separate import from `src/errors.ts` is required for the `instanceof` check.

**Acceptance Scenarios**:

1. **Given** a typed tool handler that imports the adapter module, **When** the handler imports the runtime entry point (e.g., the adapter call function), **Then** it can also obtain the `UpstreamError` class via the same import statement — the adapter module re-exports it.

### Edge Cases

- **Empty `parameters` record**: argv has no key=value tokens (just `[command, ...flags]`). Both `target_mode` values produce identical argv when parameters is `{}`.
- **Empty `flags` array**: argv has no trailing flag tokens. Both `target_mode` values produce identical trailing-tail when flags is `[]`.
- **`parameters` whose values are `undefined`**: those entries produce zero tokens in argv (skipped during assembly). This is the documented escape-hatch for "this parameter is optional and the caller decided not to pass it."
- **`parameters` whose values are non-string primitives** (e.g., `5`, `true`): coerced to strings via `String(value)` and emitted as `"<key>=<stringified>"`. The CLI consumes these as raw text per its documented `key=value` convention.
- **`parameters` whose values are objects/arrays**: caller error — the adapter's input shape forbids non-primitive values; if ever passed they are stringified via `String(value)` (yielding `"[object Object]"` or comma-joined arrays), producing a CLI argument that the binary will likely reject. Typed tool handlers MUST validate at their zod boundary that `parameters` values are primitives — this matches Principle III and is not something the adapter re-validates.
- **`parameters.vault` present in specific mode**: hoisted to be the first key=value token after the command. Other keys retain their input insertion order.
- **`parameters.vault` absent in specific mode**: no `vault=…` token is added; remaining keys appear in insertion order. The adapter does NOT inject a default vault.
- **Active mode with no target-locator keys to strip**: `parameters` is forwarded unchanged. The strip is a no-op.
- **Active mode with a non-target-locator key whose name happens to be a substring of `vault`/`file`/`path`** (e.g., `"vault_id"`, `"file_count"`): NOT stripped — the strip is keyed on exact key match, not substring.
- **stdout starts with leading whitespace then `Error: no active file`**: the prefix check is performed against `stdout.trimStart()`, so leading whitespace is tolerated and the call routes to `ERR_NO_ACTIVE_FILE`. (Same anchoring discipline as FR-005 in spec 002.)
- **stdout is `"Error:"` exactly (just six chars + newline) on exit `0`**: routes to `CLI_REPORTED_ERROR` — the prefix anchor matches; `details.message` is `"Error:"` (the trimmed first line).
- **stdout starts with `"error:"` (lowercase) or `"ERROR:"` (uppercase) on exit `0`**: routes to (d) success — the prefix detection is case-sensitive on the exact `Error:` prefix per the convention pinned in spec 002 FR-006.
- **stdout starts with `"Error: no active file"` on exit `0`, AND non-zero exit code**: impossible by definition — the priority order classifies by exit code first; non-zero always wins regardless of stdout content.
- **stdout starts with `"Error: no active file"` followed by additional same-line text** (e.g., `"Error: no active file. Open one."`): routes to `ERR_NO_ACTIVE_FILE` (priority b dominates priority c). The longer first line is preserved verbatim in `details.stdout` and `details.message`.
- **Exit code `0` with non-empty stderr but empty stdout**: routes to (d) success — the success path is exit `0` AND no `Error:` prefix in stdout; stderr content does not change classification.
- **Exit code `0` with stdout that starts with `"Error: File"` (a CLI_REPORTED_ERROR pattern that happens to share the first six characters with `"Error: no active file"` but diverges at character 7)**: routes to `CLI_REPORTED_ERROR` (priority c) — the prefix `"Error: no active file"` only matches its full literal, not a partial. Priority b only fires on the full literal anchored prefix.
- **Child terminates by signal (`code === null`, `signal !== null`)**: routes to `CLI_NON_ZERO_EXIT` (priority a) with `details.exitCode: -1` (sentinel coercion via `code ?? -1`) and `details.signal: signalName`. Mirrors the existing handler's precedent at `src/tools/obsidian_exec/handler.ts:238` so signal-termination is uniformly classified across the project.
- **Spawn fails with `ENOENT` (binary not found on PATH)**: routes to `CLI_BINARY_NOT_FOUND` — the adapter mirrors the existing `obsidian_exec` handler's behavior (handler.ts:82-91, handler.ts:155-163) so binary-not-found is uniformly classified across the project.
- **Spawn fails with a non-`ENOENT` native error**: re-thrown as the native error (not wrapped in `UpstreamError`). Caller decides how to surface unknown native errors. This matches the existing handler's behavior (handler.ts:92-93, handler.ts:165).
- **CRLF line endings in stdout** (`"Error: no active file\r\n"`): `details.message` is `"Error: no active file"` — the same `stdout.split('\n', 1)[0].trim()` algorithm pinned in spec 002 FR-003 absorbs the trailing `\r`.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: A new internal module MUST be created at `src/cli-adapter/cli-adapter.ts` with its co-located tests at `src/cli-adapter/cli-adapter.test.ts` (per Constitution Principle II's co-location rule). The module MUST export a single async runtime entry point that callers invoke; the exact export name is a plan-stage decision.
- **FR-002**: The adapter's runtime entry point MUST accept two arguments: (1) an `input` object containing exactly four fields — `command: string`, `parameters: Record<string, string | number | boolean | undefined>`, `flags: string[]`, and `target_mode: "specific" | "active"`; (2) an optional `deps` object with optional `spawnFn` and `env` fields used as the test-seam injection mechanism. The TypeScript type for `target_mode` MUST be a string literal union, not a generic `string`. This signature mirrors the precedent established at `src/tools/obsidian_exec/handler.ts:51` (`executeObsidianExec(input, deps)`); tests inject `deps.spawnFn` per the existing project convention, and production callers omit `deps` to use the default `node:child_process` `spawn` and `process.env`.
- **FR-003**: When `target_mode === "active"`, the adapter MUST remove the keys `vault`, `file`, and `path` from a copy of the `parameters` record before assembling argv. The exact key match is case-sensitive and is performed at the top level of the record (no recursion into nested values, since values are constrained to primitives by FR-002).
- **FR-004**: When `target_mode === "specific"`, the adapter MUST forward every entry of the `parameters` record into argv assembly without modification.
- **FR-005**: argv assembly MUST produce a vector of the form `[command, ...vaultPrefix, ...remainingKvParams, ...flags]`, where:
    - `vaultPrefix` is `["vault=<value>"]` if and only if the (post-strip) parameters contain a `vault` key whose value is not `undefined`; otherwise `vaultPrefix` is `[]`.
    - `remainingKvParams` is the entries of (post-strip parameters minus the vault key) in literal-object insertion order, each emitted as `"<key>=<String(value)>"`.
    - Entries whose value is `undefined` MUST be skipped — they produce zero tokens.
    - `flags` is appended verbatim as bare-word tokens (no `=`-pair wrapping, no quoting).
- **FR-006**: The adapter MUST invoke the CLI binary via `child_process.spawn` with `shell: false` and the assembled argv as a literal array. The binary path MUST be resolved from `(deps?.env ?? process.env).OBSIDIAN_BIN ?? "obsidian"`, mirroring the convention established by `src/tools/obsidian_exec/handler.ts:60-61`. The `deps.env` override path is the test-seam injection mechanism introduced by Q1's two-arg signature (FR-002): production callers omit `deps` entirely (the optional chaining handles that case), so resolution falls through to `process.env`; tests inject a synthetic `env` via `deps.env` to exercise the binary-path resolution chain (e.g., to drive `CLI_BINARY_NOT_FOUND` on a real PATH system).
- **FR-007**: The adapter MUST collect stdout and stderr via stream-event concatenation (Buffer chunks accumulated, then `Buffer.concat(...).toString("utf8")` on close). It MUST await the child's `close` (or equivalent terminal) event before classifying the outcome. Streaming is out of scope per the user input — full-buffer capture only.
- **FR-008**: On the close event, the adapter MUST classify the outcome in this strict priority order. No later branch is reachable while an earlier branch's predicate holds:
    - **(a) Non-zero exit code** (`code !== 0` OR a terminating signal that the runtime represents as a non-zero conventional exit): raise `UpstreamError` with `code: "CLI_NON_ZERO_EXIT"`, `cause: { exitCode, signal }`, and `details: { command, stdout, stderr, exitCode, signal }`. The `command` field in `details` is the input string verbatim. The `exitCode` value MUST be coerced via `code ?? -1` so that signal-only termination (`code === null`, `signal !== null`) resolves to the sentinel `-1` while `details.signal` carries the signal name; this mirrors the precedent at `src/tools/obsidian_exec/handler.ts:238`.
    - **(b) Exit code 0 AND `stdout.trimStart()` starts with the literal string `"Error: no active file"`** (case-sensitive, anchored): raise `UpstreamError` with `code: "ERR_NO_ACTIVE_FILE"`, `cause: null`, and `details: { command, stdout, stderr, exitCode: 0, message: parsedFirstLine }`. The `UpstreamError` instance's `.message` property MUST be a recovery instruction of the form `"No active file in Obsidian. Open a note in the editor, or call this tool with target_mode: \"specific\" and an explicit vault/file."` (or substantively equivalent — the exact wording is a plan-stage decision but the semantic content MUST be present).
    - **(c) Exit code 0 AND `stdout.trimStart()` starts with `"Error:"` AND priority (b) did not fire**: raise `UpstreamError` with `code: "CLI_REPORTED_ERROR"`, `cause: null`, and `details: { command, stdout, stderr, exitCode: 0, message: parsedFirstLine }`. This is the same code shipped in feature 002, reused here for non-`no-active-file` in-band CLI failures.
    - **(d) Otherwise**: resolve with `{ stdout, stderr }`.
- **FR-009**: The `parsedFirstLine` algorithm used in branches (b) and (c) of FR-008 MUST be `stdout.split('\n', 1)[0].trim()` — split on the first LF, take the first segment, apply `String.prototype.trim()` to both ends. This matches spec 002 FR-003 verbatim so the two CLI in-band error paths share one parser.
- **FR-010**: Spawn-time `ENOENT` errors (binary not on PATH) MUST be raised as `UpstreamError` with `code: "CLI_BINARY_NOT_FOUND"`, mirroring the existing handler at `src/tools/obsidian_exec/handler.ts:82-91` and `:155-163`. Other native spawn errors MUST be propagated to the caller as-is (i.e., not wrapped in `UpstreamError`) — the caller decides whether to map them.
- **FR-011**: The adapter module MUST re-export the project's `UpstreamError` class (defined at `src/errors.ts:10`) so that consumers can import the runtime entry point and the error class from a single module path.
- **FR-012**: The new stable error code `ERR_NO_ACTIVE_FILE` MUST be added to the canonical contract at `specs/001-add-cli-bridge/contracts/errors.contract.md` alongside the existing codes (`CLI_NON_ZERO_EXIT`, `CLI_BINARY_NOT_FOUND`, `CLI_TIMEOUT`, `CLI_OUTPUT_TOO_LARGE`, `CLI_REPORTED_ERROR`, `VALIDATION_ERROR`, `TOOL_NOT_FOUND`). The new row MUST list `cause: null` and `details: { command: string, stdout: string, stderr: string, exitCode: 0, message: string }` to mirror the implementation. The contract MUST also describe the trigger condition (exit `0` with stdout that, after leading-whitespace trim, starts with `"Error: no active file"`).
- **FR-013**: The README error-codes table MUST gain a `ERR_NO_ACTIVE_FILE` row with the same trigger condition and key `details` fields, matching the formatting precedent set by the `CLI_REPORTED_ERROR` row added in feature 002.
- **FR-014**: The adapter module MUST carry an original-contribution header per Constitution Principle V, of the form `// Original — no upstream. <one-line description of the adapter's intent>.` The test module MUST also carry an original-contribution header.
- **FR-015**: The adapter MUST NOT register itself as an MCP tool. It is consumed by typed tool modules that each have their own tool registration. The MCP server's `Server` registration list (in `src/server.ts`) MUST NOT change as a result of this feature.
- **FR-016**: Tests for the adapter MUST be co-located at `src/cli-adapter/cli-adapter.test.ts` per Principle II. The test set MUST include — at minimum — the following cases, each driving the adapter through a stub `spawnFn`:
    - **(a) happy-path specific mode**: `command: "read"`, `parameters: { vault: "MyVault", file: "Note" }`, `flags: []`, `target_mode: "specific"` against a stub child that exits `0` with stdout `"# Note body\n"` → spawn called with argv `["read", "vault=MyVault", "file=Note"]`, adapter resolves with `{ stdout: "# Note body\n", stderr: "" }`.
    - **(b) happy-path active-mode stripping (vault + file)**: `parameters: { vault: "V", file: "F", lines: 5 }`, `target_mode: "active"` against a stub child that exits `0` → spawn called with argv `["<command>", "lines=5"]`, adapter resolves successfully.
    - **(c) happy-path active-mode stripping (path)**: `parameters: { path: "P", query: "q" }`, `target_mode: "active"` against a stub child that exits `0` → spawn called with argv `["<command>", "query=q"]`, adapter resolves successfully.
    - **(d) failure-path non-zero exit**: stub child exits `1` with `stderr: "boom"` → adapter rejects with `UpstreamError`, `code: "CLI_NON_ZERO_EXIT"`, `details.exitCode: 1`, `details.stderr: "boom"`, `details.command` matches the input command string.
    - **(e) failure-path ERR_NO_ACTIVE_FILE**: stub child exits `0` with `stdout: "Error: no active file\n"` → adapter rejects with `UpstreamError`, `code: "ERR_NO_ACTIVE_FILE"`, `details.exitCode: 0`, `details.message: "Error: no active file"`, `.message` property is the recovery-instruction string.
    - **(f) failure-path CLI_REPORTED_ERROR**: stub child exits `0` with `stdout: "Error: File not found\n"` → adapter rejects with `UpstreamError`, `code: "CLI_REPORTED_ERROR"`, `details.message: "Error: File not found"`.
    - **(g) boundary-path undefined parameter values**: `parameters: { vault: "V", file: undefined, query: "q" }`, `target_mode: "specific"` → spawn called with argv `["<command>", "vault=V", "query=q"]` (no `file=` token).
    - **(h) boundary-path priority discrimination — `ERR_NO_ACTIVE_FILE` beats `CLI_REPORTED_ERROR`**: stub child exits `0` with `stdout: "Error: no active file. Open one or use specific mode.\n"` → adapter rejects with `code: "ERR_NO_ACTIVE_FILE"` (priority b), not `CLI_REPORTED_ERROR` (priority c).
    - **(i) boundary-path priority discrimination — non-zero exit beats stdout-prefix**: stub child exits `1` with `stdout: "Error: no active file\n"` → adapter rejects with `code: "CLI_NON_ZERO_EXIT"` (priority a), not `ERR_NO_ACTIVE_FILE` (priority b).
    - **(j) boundary-path signal-only termination**: stub child closes with `(code: null, signal: "SIGTERM")` → adapter rejects with `UpstreamError`, `code: "CLI_NON_ZERO_EXIT"`, `details.exitCode: -1` (sentinel via `code ?? -1`), `details.signal: "SIGTERM"`, `cause.exitCode: -1`, `cause.signal: "SIGTERM"`.
- **FR-017**: The vitest aggregate statements coverage threshold (the merge gate per Constitution v1.1.0 §Development Workflow #5) MUST remain at or above the floor in effect when this feature merges. The new tests MUST NOT cause coverage to drop below that floor.
- **FR-018**: The PR landing this feature MUST update the Constitution Compliance checklist with one Y/N/N/A per principle (per Constitution v1.1.0 §Development Workflow #8). All five principles are expected to be `Y` for this feature: Principle I (new module with single responsibility), Principle II (co-located tests covering happy + failure + boundary paths), Principle III (no boundary surface — the adapter is internal — but it does forward zod-typed parameters opaquely), Principle IV (every error path returns `UpstreamError`), Principle V (original-contribution header).

### Out of Scope

The following are explicitly excluded from this feature and remain the caller's concern (or land in a separate future BI):

- **Timeout / `AbortController` / `AbortSignal`**: the adapter fires one CLI call and waits for `close`. Timeout policy is the caller's responsibility — `obsidian_exec` already has its own (handler.ts:127), and typed tool handlers either wrap the adapter with their own timeout or accept the CLI's natural turnaround.
- **Retry logic**: the adapter performs exactly one spawn-and-wait cycle. Retry on transient failures (if any) is the caller's decision.
- **Output cap (`CLI_OUTPUT_TOO_LARGE`)**: the existing `obsidian_exec` handler implements this (handler.ts:13, :131-135) for its untyped escape-hatch role, where output size is unbounded by definition. Typed tools have known output shapes and either trust the size or implement a cap themselves; the adapter does not.
- **Streaming stdout / stderr to the caller**: the adapter only resolves on `close` with the full captured strings. Live-stream consumers are not a v1 use case.
- **MCP tool registration**: the adapter is internal and never appears on `src/server.ts`'s tool list. Each typed tool that consumes the adapter is responsible for its own registration in its own BI.
- **Structured logging**: the adapter emits no log lines. The existing project `Logger` (`src/logger.ts`) is invoked by the calling tool handler at the points where it cares (call start, call end, etc.). Centralising log emission inside the adapter would duplicate per-call lines when both the tool and the adapter logged.
- **zod input validation of the adapter's four parameters**: callers (typed tool handlers) validate at their own boundary per Principle III; the adapter is internal and trusts its inputs.
- **Refactoring the existing `src/tools/obsidian_exec/handler.ts` to use the adapter**: `obsidian_exec` retains its current implementation. It is the freeform escape-hatch tool with its own timeout, output cap, queue, and logger semantics from feature 001 — those concerns do not match the adapter's narrow scope, and refactoring would be a regression risk for no functional gain. If a future BI decides the overlap is harmful, it lands as a separate change.
- **Implementing the typed tools (`read_note`, `write_note`, `search_vault`, …)**: each typed tool is its own BI and its own spec; this feature only ships the foundation they will compose on top of.
- **Localised CLI error prefixes**: the prefix detections in branches (b) and (c) of FR-008 are case-sensitive on the English `Error:` and `Error: no active file` literals. Non-English Obsidian builds may emit different prefixes (unverified at the time of this spec); same out-of-scope reasoning as spec 002.

### Key Entities *(include if data involved)*

- **CLI Adapter (the new internal module)**: a single async runtime entry point at `src/cli-adapter/cli-adapter.ts`. Signature: `(input: { command, parameters, flags, target_mode }, deps?: { spawnFn?, env? })`. Output: `Promise<{ stdout, stderr }>` resolved on success, rejected with `UpstreamError` on any of the three failure modes. The `deps` argument is the test-seam injection mechanism (mirroring `executeObsidianExec`); production callers omit it. No public MCP surface; no zod schema of its own.
- **`target_mode` literal type**: the string literal union `"specific" | "active"`. This is a new TypeScript type introduced by this feature, owned by the adapter module. It does not appear in the `obsidian_exec` schema (which has its own argv assembly with no active-mode concept) — it is a typed-tool concept.
- **`ERR_NO_ACTIVE_FILE` (new stable error code)**: a new member of the `UpstreamError.code` enumeration, added to the canonical contract. Triggered when the adapter detects exit `0` with stdout starting with the literal `"Error: no active file"` after `.trimStart()`. `cause: null`, `details: { command, stdout, stderr, exitCode: 0, message }`. Co-equal with the seven existing codes.
- **`UpstreamError` (re-exported, not redefined)**: the project's existing structured boundary error class from `src/errors.ts`. The adapter module re-exports it so consumers have one import path; the class itself is unchanged.

## Assumptions

- **The new stable error code is `ERR_NO_ACTIVE_FILE`**, matching the deliberate naming committed in [ADR-004](../../.decisions/ADR-004%20-%20Centralized%20Obsidian%20CLI%20Adapter.md) and the [Architecture document](../../.architecture/Obsidian%20CLI%20MCP%20-%20Architecture.md). The `ERR_*` prefix is intentionally distinct from the `CLI_*` family used by `CLI_NON_ZERO_EXIT`, `CLI_BINARY_NOT_FOUND`, `CLI_TIMEOUT`, `CLI_OUTPUT_TOO_LARGE`, and `CLI_REPORTED_ERROR`: those five codes signal CLI-process failures the caller cannot directly recover from in-call, while `ERR_NO_ACTIVE_FILE` signals a user-action condition that the calling LLM/human can correct (open a note, or switch to `target_mode: "specific"`). The semantic split is what the prefix conveys. An initial /speckit-clarify answer (Q2, 2026-05-05) renamed to `CLI_NO_ACTIVE_FILE` for cosmetic prefix uniformity, but the rename was reversed during /speckit-plan when the ADR-004 conflict surfaced and the deliberate semantic split was reaffirmed.
- **The "no active file" detection algorithm is an anchored, case-sensitive prefix check on `stdout.trimStart()` against the literal string `"Error: no active file"`** — same anchoring discipline as spec 002 FR-005/FR-006. The user input's "(or equivalent)" parenthetical in AC #6 is interpreted as: the documented Obsidian CLI message format for the focused-note-missing case as observed in BI-026 reproductions. Any divergent variants discovered empirically in future Obsidian versions are tracked as a follow-up PFI.
- **Path-key stripping in active mode operates at the top level of the `parameters` record** — the keys `vault`, `file`, `path` are removed by exact case-sensitive key match. No recursion into nested values (since values are constrained to primitives by FR-002).
- **Vault-hoisting in argv assembly is a hard invariant**: when (post-strip) `parameters.vault` is present and not `undefined`, the resulting argv MUST place `vault=<value>` as the first key=value token after the command, regardless of the literal-object insertion order in the input. This matches the existing `assembleSpawnArgs` convention at `src/tools/obsidian_exec/handler.ts:254` and the CLI's documented scoping convention from feature 001.
- **Spawn-time `ENOENT` is mapped to `CLI_BINARY_NOT_FOUND`**, mirroring the existing `obsidian_exec` handler. Other native spawn errors propagate as-is. The user input's enumerated priority order (a)–(d) describes post-close classification only; the spawn-time error path is a separate concern that follows project precedent rather than introducing a new code.
- **The adapter has no internal logger**: structured logging is the calling tool handler's concern, since it's the typed tool that owns the per-call correlation context (request id, queue depth, etc.). Centralising log emission inside the adapter would either drop that context or duplicate log lines.
- **The adapter has no zod schema**: it is internal and trusts inputs already validated by the calling tool handler at its own boundary per Principle III.
- **The CLI binary is resolved from `process.env.OBSIDIAN_BIN ?? "obsidian"`**, matching the convention at `src/tools/obsidian_exec/handler.ts:61`. No new env var is introduced by this feature.
- **The existing `src/tools/obsidian_exec/handler.ts` is NOT refactored** to use the adapter as part of this BI. It retains its own argv assembly, error detection, timeout, output cap, and logger integration. The two implementations co-exist; future BIs may consolidate if reviewers conclude the duplication is harmful.
- **Constitution Principle IV (every boundary error is an `UpstreamError`) continues to bind**: every adapter rejection is an `UpstreamError`, never a plain `Error`, `null`, or `undefined`. The exception is non-`ENOENT` native spawn errors per FR-010, which are propagated rather than wrapped — this matches the existing handler's behavior and reflects the principle that wrapping unknown native errors masks information that the caller needs.
- **Constitution Principle II binds the test set co-location** at `src/cli-adapter/cli-adapter.test.ts`. The test cases enumerated in FR-016 are the minimum set; reviewers may add more.
- **Constitution Principle V binds the module-header convention**: `// Original — no upstream. <one-line description>.` is required on both the adapter source file and its test file.
- **The first typed-tool BI that lands on top of this adapter (e.g., `read_note`)** is the empirical validator of the adapter's interface. If that BI surfaces a real-world need not anticipated here, it gets added to a future revision of this spec rather than retrofitted silently.
