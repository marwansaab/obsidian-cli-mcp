# Implementation Plan: File Scope

**Branch**: `066-file-scope` | **Date**: 2026-06-30 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `specs/066-file-scope/spec.md`

> **Modify the existing `find_and_replace` typed tool** (it is NOT a new tool). Add a single-note scope alongside the existing vault-wide default and `subfolder` scope: confine a find-and-replace to exactly one note, addressed either by a named locator (`path` vault-relative, or `file` bare name) or by an open-note opt-in (`active_note`). The schema-modelling shape is settled by **[ADR-032](../../.decisions/ADR-032%20-%20Find%20And%20Replace%20File%20Scope%20via%20Sibling%20Locators.md)** (dedicated optional sibling locator fields; NO `target_mode` adoption) ‚Äî this plan implements that decision, it does not re-open it. The clarify session locked the input shape, the error-discriminator roster, and bare-name resolution; the only open items are empirical T0 verification gates, not design forks.

## Summary

`find_and_replace` today scans every eligible `.md` note in a vault (or under a `subfolder`) ‚Äî it has no way to confine a change to one note, and its description warns "there is NO single-file scoping option." That forces callers to craft a globally-unique pattern for a surgical edit; one mis-scoped call can rewrite every matching note, and on a vault without version control that is unrecoverable. This feature adds a structural single-note limit so the scope, not the pattern, bounds the blast radius ‚Äî and closes the gap where every other note-level tool can already act on the open note but find-and-replace cannot.

**Primary requirements** (from spec, post-clarify): a single-note scope confines matches/replacements to exactly one note, leaving every other note untouched (FR-001 / SC-001); the target is addressed by vault-relative `path` or bare `file` name with the cohort's `[[‚Ä¶]]` bracket rejection, a bare name resolved via `resolveFileByTsv` for shortest-unique-name parity (FR-002 / FR-003 / SC-006); an `active_note` opt-in confines to the currently-open note and reports its location, with `ERR_NO_ACTIVE_FILE` when none is open (FR-004 / FR-005 / SC-003); single-note XOR folder, and `active_note` additionally exclusive with `file`/`path`/`subfolder`/`vault`, all rejected before any read (FR-006 / FR-007 / SC-004); a missing named note and an ineligible (non-`.md` / dot-dir) target are typed errors, never silent empties (FR-008 / FR-012); the affected-notes set is bounded to ‚â§ 1 (FR-009 / SC-002); preview/commit, the safe-upper-bound guard, two-scan drift detection, region skip/opt-in, atomic write, and byte-for-byte preservation all carry over unchanged within the scope (FR-010 / FR-011); unscoped calls are byte-stable ‚Äî no existing call changes meaning (FR-014 / SC-005); and no new top-level error code is introduced ‚Äî every new state reuses an existing code (FR-016 / SC-007, Constitution Principle IV).

**Technical approach** (confined to `src/tools/find_and_replace/**` + the already-imported shared `src/tools/_active-file.ts`; **no kernel-node touch**; the published surface DOES move ‚Äî description + input schema change, so the `_register-baseline.json` `find_and_replace` fingerprints regen):

- **The single-note scope is a front-end scope-resolution change.** The existing handler resolves a `scanRoot` (vault root or subfolder) and walks it to a list of eligible rel-paths; Stages 4‚Äì7 (scan ‚Üí bound-check ‚Üí drift-check ‚Üí atomic write) then operate on that list. The single-note scope replaces only the resolve-and-walk front end: it resolves the one target note to a single rel-path and feeds `eligible = [relPath]` into the unchanged downstream. Bound, drift, region-skip, atomic write, and byte-for-byte preservation are inherited verbatim ‚Äî the scope narrows WHICH notes are examined, not HOW matches are found or written (FR-011).
- **Three resolution forks, all reusing `_active-file.ts` (already imported by the handler).** `active_note` ‚Üí `resolveActiveFocusedFile(deps, "find_and_replace")` returns `{ vaultRoot, relPath }` directly (and throws `ERR_NO_ACTIVE_FILE` when nothing is open). `path` ‚Üí resolve the vault root as today (registry when `vault` is named, else the existing focused-vault eval), then take the given vault-relative path. `file` ‚Üí resolve the vault root as today, then resolve the bare name via `resolveFileByTsv` (the cohort's `obsidian file file=<name>` round-trip ‚Üí TSV `path` line) for shortest-unique-name parity. Each fork then runs the existing two-layer path-safety canonical guard on the resolved note and an eligibility/existence check before handing `eligible = [relPath]` downstream.
- **Schema: three optional fields + a mutual-exclusivity `superRefine`.** `file` (bare name; `[[`/`]]` rejected per the cohort's `WIKILINK_BRACKET_REJECTION_MESSAGE`; structural path-safety), `path` (vault-relative; structural path-safety), `active_note` (boolean, default false). A `superRefine` enforces the scope matrix; all three fields are optional so omitting them is byte-stable with today's vault-wide-or-folder default (FR-014).
- **Errors reuse existing codes (Principle IV streak preserved).** Schema-detected conflicts ‚Üí `VALIDATION_ERROR` + `details.code:"SCOPE_CONFLICT"` + `details.reason` (the conflicting pair) via the existing `mapZodIssuesToToolError` hook in `index.ts`; the `[[‚Ä¶]]` rejection surfaces through the cohort's standard validation channel (no sub-code). Handler-detected states ‚Üí `VALIDATION_ERROR` + `details.code:"INVALID_NOTE"` + `details.reason:"not-found"` (missing) / `"not-eligible"` (non-`.md` or dot-dir), parity with the existing `INVALID_SUBFOLDER`/`not-found` shape. No-note-open ‚Üí the existing `ERR_NO_ACTIVE_FILE`. Canonical escape ‚Üí the existing `PATH_ESCAPES_VAULT`. Per ADR-015 sub-discriminators; per ADR-032 the whole feature stays on existing top-level codes.
- **Surface text corrected.** The `FIND_AND_REPLACE_DESCRIPTION` "no single-file mode" warning and `docs/tools/find_and_replace.md` are rewritten to document the single-note scope; both move the `find_and_replace` fingerprints in `_register-baseline.json`, which is regenerated and re-asserted by the FR-018 baseline-stability test in the same change.

**T0 forcing-gate probes (implement-time, drive `Obsidian.com` per `.memory/test-execution-instructions.md`)**: P1 ‚Äî `obsidian file file=<bare-name>` against the focused vault (no `vault=` arg): confirm the TSV `path` line resolves shortest-unique-name, fixing the focused + bare-name resolution branch. P2 ‚Äî `obsidian eval` focused-file (`FOCUSED_FILE_TEMPLATE`) with a `.md` note open vs. a non-`.md` (PDF/canvas) active file vs. nothing open: confirm the `{path, base}` shape and the `path === null` no-active-file signal. See [contracts/t0-probe-plan.md](contracts/t0-probe-plan.md). The probes verify resolution channels the design already assumes; they are verification gates, not design forks.

## Phase 0: Research ‚Üí research.md

Decisions resolving the design (full detail in [research.md](research.md)):

- **D1 ‚Äî Modify, don't add.** `find_and_replace` exists and is registered; this BI changes its schema + handler front end + description + tests. No new tool, no new `server.ts` registration, no new DI argument.
- **D2 ‚Äî Single-note scope is a front-end narrowing of `eligible`.** The scope resolves to `eligible = [relPath]` and feeds the unchanged Stages 4‚Äì7. The downstream scan/bound/drift/region-skip/atomic-write logic is inherited verbatim (FR-011), so the safety guards and byte-for-byte preservation apply within the one note for free.
- **D3 ‚Äî Sibling locator fields, no `target_mode` (ADR-032).** `file` / `path` / `active_note` optional fields alongside `subfolder` / `vault`; a `superRefine` enforces the scope matrix. Backward-compatible: all new fields optional, unscoped byte-stable (FR-014). `target_mode` is NOT adopted (no new `schema ‚Üí target-mode` edge; ADR-003 untouched).
- **D4 ‚Äî Three resolution forks reusing `_active-file.ts`.** `active_note` ‚Üí `resolveActiveFocusedFile` ‚Üí `{ vaultRoot, relPath }` (and `ERR_NO_ACTIVE_FILE` when nothing is open); `path` ‚Üí existing vault-root resolve + the given vault-relative path; `file` ‚Üí existing vault-root resolve + `resolveFileByTsv` (`obsidian file file=<name>`) for shortest-unique-name parity. Each fork then applies the existing canonical guard + the eligibility/existence check.
- **D5 ‚Äî Eligibility + existence checks for the single target.** The resolved note must end in `.md` (case-insensitive) and traverse no `.`-prefixed segment ‚Üí else `VALIDATION_ERROR` + `INVALID_NOTE`/`not-eligible`; it must exist on disk ‚Üí else `VALIDATION_ERROR` + `INVALID_NOTE`/`not-found`. An explicitly-targeted ineligible/missing note is a hard error, never a silent empty result (FR-008 / FR-012; Principle IV).
- **D6 ‚Äî Focused + bare-name resolution (T0-verified).** When `file` is supplied with `vault` absent, the vault root comes from the existing focused-vault eval; the bare name is resolved against the focused vault via the `obsidian file` subcommand. Plan of record: reverse-resolve the focused vault display name via `resolveVaultDisplayName(vaultRegistry, base)` and pass it to `resolveFileByTsv` (no shared-helper signature change). T0 P1 confirms `obsidian file file=<name>` resolves shortest-unique-name for the focused vault; if it accepts focused mode without a `vault=` arg, a simpler no-reverse-lookup variant is available (recorded, not blocking).
- **D7 ‚Äî Scope mutual-exclusivity matrix (schema `superRefine`).** Conflicts emit `custom` issues with `params: { subCode: "SCOPE_CONFLICT", subReason: <pair> }`; `index.ts` `mapZodIssuesToToolError` maps them to `VALIDATION_ERROR` + `details.code:"SCOPE_CONFLICT"` + `details.reason`. Reasons: `note+folder`, `active+note`, `active+folder`, `active+vault`, `file+path`. The `[[‚Ä¶]]` reject on `file` surfaces via the cohort's standard `VALIDATION_ERROR` channel (no sub-code, parity with `append_note`).
- **D8 ‚Äî Commit re-scan reuse for single-note.** The existing commit path re-walks the scan root for the second (drift) scan. Under single-note there is nothing to re-walk ‚Äî the second scan re-reads the same fixed `[relPath]`, so drift detection (content changed between the two reads ‚Üí count differs) still fires, while folder/vault-wide scopes keep re-walking as today.
- **D9 ‚Äî Surface text + baseline regen.** `FIND_AND_REPLACE_DESCRIPTION` and `docs/tools/find_and_replace.md` are rewritten to document the single-note scope (the old "no single-file mode" warning is removed). Both fingerprints (`descriptionFingerprint`, `schemaFingerprint`) for `find_and_replace` in `_register-baseline.json` move; the file is regenerated and the FR-018 stability test re-asserts the new baseline in the same change.

**Output**: research.md with all decisions recorded. No `NEEDS CLARIFICATION` remains; the two open items are T0 verification gates whose expected outcomes the existing cohort behaviour implies.
