# Feature Specification: Delete Note Typed MCP Tool

**Feature Branch**: `012-delete-note`
**Created**: 2026-05-08
**Status**: Draft
**Input**: User description: "Add Delete Note — A typed MCP tool that deletes a vault note. Default sends the file to the system trash (recoverable from the OS); explicit opt-in is required for permanent (no-undo) deletion."

## Background *(non-mandatory context)*

This feature adds the third typed-tool wrap on top of the foundation completed by features 003–010, and the second since [011-write-note](../011-write-note/spec.md) closed the create/overwrite leg of the typed-write surface. Where `read_note` retired `obsidian_exec` for reads and `write_note` retired it for create/overwrite, `delete_note` retires it for the destructive single-file removal subcommand of the Obsidian CLI. It composes on the same load-bearing primitives:

- **[004-target-mode-schema](../004-target-mode-schema/spec.md) / [010-flatten-target-mode](../010-flatten-target-mode/spec.md)** — the shared `target_mode` schema primitive at [src/target-mode/target-mode.ts](../../src/target-mode/target-mode.ts). Post-010 the canonical reuse idiom is `applyTargetModeRefinement(targetModeBaseSchema.extend({ <tool-specific fields> }))` (the Pattern (a) flat-extension form). Per Constitution Principle III, this primitive owns the `target_mode` discriminator, the `vault`/`file`/`path` keys, the specific-branch XOR locator rule, the active-branch forbidden-key rule, and the strict-mode posture (`additionalProperties: false`); `delete_note` MUST compose on top of it rather than re-state any of those rules inline.
- **[003-cli-adapter](../003-cli-adapter/spec.md)** — the centralised CLI adapter at [src/cli-adapter/cli-adapter.ts](../../src/cli-adapter/cli-adapter.ts). The handler routes every CLI invocation through `invokeCli({ command, parameters, flags, target_mode }, deps?)` (or its bounded counterpart through the shared queue). The adapter owns argv assembly, vault hoisting, active-mode locator stripping, and the four-priority error classification (`CLI_BINARY_NOT_FOUND`, `CLI_NON_ZERO_EXIT`, `ERR_NO_ACTIVE_FILE`, `CLI_REPORTED_ERROR`). `delete_note` MUST NOT spawn child processes directly. The 011-write-note plan-stage research ratified an unknown-vault response-inspection clause in the adapter (R5); `delete_note` inherits that benefit without modification.
- **[005-help-tool](../005-help-tool/spec.md)** — the `help` tool and the schema-stripping utility at [src/help/strip-schema.ts](../../src/help/strip-schema.ts). The registry-consistency test at [src/server.test.ts](../../src/server.test.ts) gates the existence of `docs/tools/delete_note.md`; this BI replaces that file's TODO stub (or creates it if no stub exists) with a non-stub Markdown body.
- **[006-read-note](../006-read-note/spec.md) / [011-write-note](../011-write-note/spec.md)** — the empirical validators of the typed-tool pattern. `delete_note` follows the same module layout (`schema.ts` / `handler.ts` / `index.ts` + co-located tests under `src/tools/delete_note/`), the same `RegisterDeps` shape (`logger`, `queue`, optional `deps` pass-through), and the same handler-thinness ceiling (a tight argv-shape + result-mapping body; no per-call logger events at the tool layer per the 011-write-note R1 reconciliation). The factory name is `createDeleteNoteTool` per the `createReadNoteTool` / `createWriteNoteTool` precedent.

The user-facing contract this feature delivers: an MCP client calls `delete_note({ target_mode: "specific", vault: "MyVault", path: "Inbox/Old.md" })` and receives `{ deleted: true, path: "Inbox/Old.md", toTrash: true }` — the file has been moved to the OS trash, recoverable. To delete irreversibly, the caller passes `permanent: true` and receives `{ deleted: true, path: "Inbox/Old.md", toTrash: false }`. Locator XOR, mode-forbidden-key, and strict-mode (`additionalProperties: false`) rejections all happen at the zod parse boundary; CLI failures (binary missing, non-zero exit, in-band `Error:` reply, no active file) flow through `UpstreamError`. No new error codes are introduced by this BI.

The feature replaces the universal escape hatch (`obsidian_exec`) for delete operations. After this BI lands, agents that today reach for `obsidian_exec` to issue `obsidian delete …` (or whatever the CLI's actual subcommand name is — captured during plan-stage research per FR-019) get a typed, schema-validated, structurally-error-classified surface in `delete_note`; `obsidian_exec` remains as the freeform escape hatch for subcommands not yet wrapped.

The destructive nature of the operation drives the safety posture of the typed surface: the default is the recoverable path (trash), and the irreversible variant requires an explicit typed `permanent: true` opt-in. The tool's structured response carries `toTrash: boolean` so operators auditing logs can filter for permanent deletes from the response shape alone — the typed `permanent` flag IS the audit point.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Specific-mode delete to trash via path locator (Priority: P1)

An MCP client (typically an LLM agent) needs to remove a note at a known vault-relative path, with the safety net that the OS trash provides. The agent calls `delete_note({ target_mode: "specific", vault: "MyVault", path: "Inbox/Old.md" })` against a vault where `Inbox/Old.md` exists, and receives `{ deleted: true, path: "Inbox/Old.md", toTrash: true }`. The file is no longer at `Inbox/Old.md` on disk; it has been moved to the OS trash and is recoverable via the platform's standard trash-restore UI.

**Why this priority**: This is the headline value of the feature. The to-trash default is the safe path — "do no harm to user state" — and is the variant any non-deliberate cleanup workflow uses. Without this story, the typed-tool track has no delete primitive and every delete goes through the freeform escape hatch (which inherits none of the per-mode validation, structured errors, or progressive-disclosure documentation that typed tools provide).

**Independent Test**: With a stub `spawnFn` injected via `deps`, calling `delete_note({ target_mode: "specific", vault: "MyVault", path: "Inbox/Old.md" })` against a stub child that exits `0` and emits a CLI-shaped success response (canonical path on stdout, or substantively equivalent — exact wording captured during plan-stage CLI characterisation per FR-019) results in (a) the spawn being invoked with binary `"obsidian"` and an argv whose first non-binary token is the delete subcommand and which contains `vault=MyVault` and `path=Inbox/Old.md` (and does NOT contain a `permanent` flag), and (b) the handler returning `{ deleted: true, path: "Inbox/Old.md", toTrash: true }`.

**Acceptance Scenarios**:

1. **Given** the `delete_note` tool is registered and a stub adapter that exits `0` reporting a successful to-trash delete, **When** an MCP client calls `delete_note({ target_mode: "specific", vault: "MyVault", path: "Inbox/Old.md" })`, **Then** the tool returns a successful MCP response whose JSON-serialised payload is `{ deleted: true, path: "Inbox/Old.md", toTrash: true }` AND the adapter was invoked with argv that includes `vault=MyVault` and `path=Inbox/Old.md` (vault hoisted first per the adapter's argv-assembly contract) AND the argv does NOT include any `permanent`-shaped token.
2. **Given** the `delete_note` tool, **When** an MCP client calls it with `permanent: false` (explicit-false), **Then** the result is identical to the omitted case — the schema's default-false and an explicit `false` are observationally indistinguishable, AND the argv passed to the adapter does NOT include any `permanent`-shaped token (default-false MUST NOT emit in argv, parity with `write_note`'s `overwrite` rule).

### User Story 2 — Specific-mode delete to trash via wikilink locator (`file=`) (Priority: P1)

When the agent has a wikilink-form name (no extension, no folder) — typically because that is how Obsidian linked to it, or because the agent is removing a note the user referred to by name — it uses `file` instead of `path`. The agent calls `delete_note({ target_mode: "specific", vault: "MyVault", file: "QuickNote" })` and the CLI resolves the wikilink to a concrete on-disk location. The handler reports `{ deleted: true, path: "<CLI-resolved canonical path>", toTrash: true }` — the canonical path is derived from the CLI's response (or, where the CLI does not echo a canonical path, the input wikilink expanded with the inferred default-folder + `.md` extension; the precise derivation strategy is captured during plan-stage research per FR-019).

**Why this priority**: P1 alongside Story 1, not P2. The two locators target the same CLI subcommand but resolve different ambiguity classes — `file=` is wikilink-resolved by Obsidian, `path=` is exact. Both are part of the typed surface that retires `obsidian_exec` for deletes; downgrading either would imply the surface is incomplete.

**Independent Test**: With a stub adapter that exits `0` and reports a CLI-shaped delete response naming `Inbox/QuickNote.md` as the canonical path, calling `delete_note({ target_mode: "specific", vault: "MyVault", file: "QuickNote" })` results in the spawn being invoked with argv that includes `vault=MyVault` and the wikilink-form locator (in whichever argv-token shape plan-stage research per FR-019 confirms — `file=QuickNote` is the cross-tool-uniform user-facing name; the CLI argv token may rename to a different key per the 011-write-note PSR-5 precedent) and the handler returning `{ deleted: true, path: "Inbox/QuickNote.md", toTrash: true }`.

**Acceptance Scenarios**:

1. **Given** the `delete_note` tool and a stub adapter reporting `Inbox/QuickNote.md` as the canonical resolved path, **When** an MCP client calls `delete_note({ target_mode: "specific", vault: "MyVault", file: "QuickNote" })`, **Then** the tool returns `{ deleted: true, path: "Inbox/QuickNote.md", toTrash: true }` AND the adapter was invoked with argv that includes `vault=MyVault` and the resolved wikilink-form locator token (exact CLI argv key captured during plan-stage research per FR-019; user-facing field stays `file` for cross-tool uniformity).
2. **Given** the `delete_note` tool, **When** an MCP client calls it with `file` containing UTF-8 multi-byte characters (e.g., `file: "笔记"`), **Then** the adapter is invoked with that exact byte sequence in the locator token AND the call succeeds. `delete_note` does not transcode, normalise, or escape the file value.

### User Story 3 — Specific-mode permanent delete with explicit opt-in (Priority: P1)

The agent that explicitly knows it wants an irreversible delete (e.g., during automated test cleanup of throwaway fixtures, or when removing a confirmed-stale generated artifact) passes `permanent: true`. The CLI bypasses the OS trash; the file's bytes are unrecoverable from the OS. The handler returns `{ deleted: true, path: "<canonical path>", toTrash: false }` — `toTrash: false` is the audit-trail signal that the destructive variant ran. Operators inspecting logs filter for permanent deletes by `toTrash === false`.

**Why this priority**: P1. The permanent branch is the second half of Story 1's safety contract — the to-trash default is liveable only because explicit-opt-in irreversible-delete is available for callers that genuinely need it. Without an explicit permanent path, agents seeking irreversible deletion would fall back to `obsidian_exec`, which is exactly the regression this BI prevents. The typed `permanent` flag IS the audit point per the user input's SECURITY adversarial bullet — every call's structured response identifies which variant ran via `toTrash`.

**Independent Test**: With a stub adapter that exits `0` and reports a CLI-shaped permanent-delete success response, calling `delete_note({ target_mode: "specific", vault: "V", path: "Old.md", permanent: true })` results in the spawn being invoked with argv that includes a `permanent`-shaped token (placement and exact form — key=value vs flag — captured during plan-stage research per FR-019, parity with the 011-write-note PSR-4 resolution for `overwrite`/`open`) AND the handler returns `{ deleted: true, path: "Old.md", toTrash: false }`.

**Acceptance Scenarios**:

1. **Given** the `delete_note` tool and a stub adapter reporting a successful permanent delete of `Old.md`, **When** an MCP client calls `delete_note({ target_mode: "specific", vault: "V", path: "Old.md", permanent: true })`, **Then** the tool returns `{ deleted: true, path: "Old.md", toTrash: false }` AND the adapter was invoked with argv that includes the `permanent`-true token (exact placement per the adapter's argv-assembly contract and plan-stage characterisation per FR-019).
2. **Given** the same tool and a stub adapter that records its argv, **When** an MCP client calls `delete_note` with `permanent` omitted OR `permanent: false`, **Then** the argv passed to the adapter does NOT contain any `permanent`-shaped token — default-false MUST NOT emit in argv (parity with `write_note`'s `overwrite`/`open` rule per [011-write-note](../011-write-note/spec.md) FR-007).
3. **Given** the `delete_note` tool, **When** `permanent: true` is supplied against a vault where the underlying CLI's default would be the OPPOSITE (i.e., where the CLI's default is permanent and to-trash requires a flag), **Then** the typed surface inverts where necessary so the user-facing default remains to-trash. The typed surface's safety posture binds regardless of the CLI's underlying default; plan-stage research per FR-019 captures the CLI's actual default and the handler's argv-mapping locks against it (per the user input's SAFETY DEFAULTS adversarial bullet).

### User Story 4 — Active-mode deletes the focused note (Priority: P1)

When the agent declares `target_mode: "active"`, the call's intent is "delete whatever note is currently open in Obsidian." The schema layer (per the target-mode primitive) forbids `vault`, `file`, and `path` in this mode. `permanent` remains a permitted optional field in active mode (its semantics are well-defined — irreversibility applies regardless of how the locator is resolved); the active-mode safety-default rule is identical to specific-mode (default to-trash, opt in to permanent). The handler routes `{ target_mode: "active" }` through the adapter, which emits argv beginning with the delete subcommand and no locator key=value tokens. The CLI removes the focused note's file (to trash by default); the handler returns `{ deleted: true, path: "<focused note's vault-relative path>", toTrash: true }` (or `toTrash: false` if `permanent: true` was passed).

**Why this priority**: P1. The active-mode contract is the entire reason `target_mode` is a discriminated union — collapsing it to "specific only" defeats ADR-003. The agent uses active mode when it operates on the user's current focus; making that path work end-to-end (schema → adapter → CLI) for the destructive single-file removal is part of this BI's value.

**Independent Test**: With a stub adapter that exits `0` and reports a CLI-shaped success response naming `Daily/2026-05-08.md` as the focused path, calling `delete_note({ target_mode: "active" })` results in the adapter being invoked with argv that contains no `vault=…`, `file=…`, or `path=…` tokens AND the handler returns `{ deleted: true, path: "Daily/2026-05-08.md", toTrash: true }`. A second test asserts that calling `delete_note({ target_mode: "active" })` against a stub adapter that surfaces `ERR_NO_ACTIVE_FILE` flows that error through to the MCP error response unchanged.

**Acceptance Scenarios**:

1. **Given** the `delete_note` tool and a stub adapter reporting a successful focused-note to-trash delete, **When** an MCP client calls `delete_note({ target_mode: "active" })`, **Then** the tool returns `{ deleted: true, path: "<focused path>", toTrash: true }` AND the adapter was invoked with argv that contains the delete subcommand but no locator tokens.
2. **Given** the `delete_note` tool, **When** an MCP client calls `delete_note({ target_mode: "active", permanent: true })`, **Then** the tool returns `{ deleted: true, path: "<focused path>", toTrash: false }` AND the adapter argv includes the `permanent`-true token AND no locator tokens.
3. **Given** the `delete_note` tool, **When** an MCP client calls `delete_note({ target_mode: "active", vault: "V" })` (forbidden key in active mode), **Then** the call fails at the zod parse boundary as `VALIDATION_ERROR` whose `details.issues[].path` includes `"vault"` and whose message names `"vault"` AND `"active mode"` per the target-mode primitive's contract. The same applies for forbidden `file` and `path`.
4. **Given** the `delete_note` tool and a stub adapter that surfaces `UpstreamError({ code: "ERR_NO_ACTIVE_FILE", … })`, **When** an MCP client calls `delete_note({ target_mode: "active" })`, **Then** the tool returns an MCP error response (`isError: true`) whose JSON-serialised payload carries `code: "ERR_NO_ACTIVE_FILE"` and a recovery-hint message naming "focus a note or switch to specific mode" — propagated via the adapter's verbatim `.message` field, not synthesised at the `delete_note` layer.

### User Story 5 — Schema layer rejects malformed inputs before any CLI call (Priority: P1)

Per Constitution Principle III, the zod schema is the gatekeeper. `delete_note`'s schema composes the post-010 flat target-mode primitive (which enforces `vault` required in specific mode, exactly-one-of `file`/`path` in specific mode, `vault`/`file`/`path` forbidden in active mode, `additionalProperties: false` strict-mode against unknown top-level keys) and adds `permanent` (optional, default false) as the single `delete_note`-specific field. Seven classes of malformed input MUST fail at the parse step: (a) `target_mode === "specific"` with neither `file` nor `path`; (b) `target_mode === "specific"` with both `file` AND `path`; (c) `target_mode === "specific"` without `vault`; (d) `target_mode === "active"` with any of `vault`/`file`/`path`; (e) any input with an unknown top-level key (e.g., `pancakes: "yes"`); (f) `target_mode` absent or with a value other than `"specific"` / `"active"`; (g) `permanent` present with a non-boolean value (e.g., `permanent: "true"`). None of these reach the adapter; the failure is a `VALIDATION_ERROR` with the offending field path. **No file on disk MUST be touched by any input that fails validation.** The CLI is never invoked.

**Why this priority**: P1. Boundary validation is the constitutional baseline — ADR-003's promise that the schema enforces the intent contract is empty if `delete_note` doesn't actually parse against the primitive. For a destructive operation, skipping this story is doubly bad: every malformed input either becomes an opaque CLI failure (uninformative to the agent) or — worse — undefined behaviour where the CLI accepts both `file` AND `path` and silently deletes the wrong one. The unknown-top-level-key carve-out (post-010 strict mode) is the additional safety net that prevents typos like `permannent: true` from silently producing a to-trash delete when the caller intended permanent (or, conversely, a permanent delete when the caller wrote `permanenet: false` and meant to opt out). For an irreversible operation, that silent fall-through is the kind of defect schema validation exists to prevent.

**Independent Test**: Seven parse-only tests, each driving the handler with a stub adapter that MUST NOT be called. Each asserts the resulting error is `VALIDATION_ERROR` with the expected `details.issues[].path` and message keywords; the stub spawn function is asserted never to have been called.

**Acceptance Scenarios**:

1. **Given** the `delete_note` tool and a stub adapter that records every invocation, **When** an MCP client calls `delete_note({ target_mode: "specific", vault: "MyVault" })` (neither `file` nor `path`), **Then** the tool returns `code: "VALIDATION_ERROR"` whose `details.issues` includes an entry whose `message` mentions `"exactly one of"` AND the stub adapter was never invoked.
2. **Given** the `delete_note` tool, **When** an MCP client calls `delete_note({ target_mode: "specific", vault: "MyVault", file: "F", path: "F.md" })` (both locators), **Then** the tool returns `code: "VALIDATION_ERROR"` whose issues mention `"exactly one of"` AND the stub adapter was never invoked.
3. **Given** the `delete_note` tool, **When** an MCP client calls `delete_note({ target_mode: "specific", file: "F" })` (vault missing), **Then** the tool returns `code: "VALIDATION_ERROR"` whose `details.issues[].path` includes `"vault"` AND the stub adapter was never invoked.
4. **Given** the `delete_note` tool, **When** an MCP client calls `delete_note({ target_mode: "active", vault: "V" })` (forbidden key in active mode), **Then** the tool returns `code: "VALIDATION_ERROR"` whose issues mention `"vault"` AND `"active mode"` AND the stub adapter was never invoked. The same shape applies for forbidden `file` and forbidden `path`.
5. **Given** the `delete_note` tool, **When** an MCP client calls `delete_note({ target_mode: "specific", vault: "V", path: "P.md", pancakes: "yes" })` (unknown top-level key), **Then** the tool returns `code: "VALIDATION_ERROR"` whose `details.issues[]` includes a `code: "unrecognized_keys"` entry naming `pancakes` AND the stub adapter was never invoked. (Per [010-flatten-target-mode](../010-flatten-target-mode/spec.md) FR-002, this is the post-010 strict-mode behaviour; this BI inherits it.)
6. **Given** the `delete_note` tool, **When** an MCP client calls `delete_note({ target_mode: "unknown", vault: "V", path: "P.md" })` (invalid discriminator), **Then** the tool returns `code: "VALIDATION_ERROR"` whose issues identify `target_mode` as the offending field with an invalid-value error AND the stub adapter was never invoked.
7. **Given** the `delete_note` tool, **When** an MCP client calls `delete_note({ target_mode: "specific", vault: "V", path: "P.md", permanent: "true" })` (string instead of boolean), **Then** the tool returns `code: "VALIDATION_ERROR"` whose `details.issues[].path` includes `"permanent"` and whose error code is `invalid_type` (or zod's equivalent) AND the stub adapter was never invoked.

### User Story 6 — CLI failures (binary missing, non-zero exit, in-band Error, no active file, file-not-found) flow through `UpstreamError` (Priority: P1)

When the underlying CLI fails — the binary is missing, the process exits non-zero, the CLI replies with `Error: …` on stdout, the focused note is missing in active mode, the target file does not exist (e.g., already deleted out-of-band) — the failure surfaces to the MCP client as an `UpstreamError` carrying the adapter's classified `code`. Per the cli-adapter's four-priority classification, `delete_note` does not classify; it propagates whatever code the adapter raised.

**Why this priority**: P1. Constitution Principle IV requires every boundary failure to surface as a structured `UpstreamError` with a stable code. The user input names file-not-found, permission-denied, concurrent-delete-races, and unknown-vault-display-name as adversarial cases that MUST be handled rather than silently succeed; all flow through this propagation path (per FR-019). The actual code on the wire depends on what the CLI does — the spec's job is to require propagation, not to claim a single code for a failure mode the CLI controls.

**Independent Test**: Five handler tests, each driving the adapter through an injected stub: (a) stub child fails to spawn with `ENOENT` → handler returns `code: "CLI_BINARY_NOT_FOUND"`; (b) stub child exits `1` with stderr `"permission denied"` → handler returns `code: "CLI_NON_ZERO_EXIT"`; (c) stub child exits `0` with stdout `"Error: file not found at Inbox/Missing.md\n"` → handler returns `code: "CLI_REPORTED_ERROR"` AND `details.message` carries the verbatim CLI line; (d) stub child surfaces `ERR_NO_ACTIVE_FILE` → handler returns `code: "ERR_NO_ACTIVE_FILE"`; (e) any non-`UpstreamError` exception escapes the adapter — handler re-throws verbatim WITHOUT `asToolError` wrapping (parity with `read_note` / `write_note`). In all cases, the handler does NOT swallow, mask, or rewrite the adapter's error; it routes through `asToolError()` per the existing precedent at [src/tools/_shared.ts](../../src/tools/_shared.ts).

**Acceptance Scenarios**:

1. **Given** the `delete_note` tool and a stub adapter whose `spawnFn` raises `ENOENT`, **When** an MCP client calls `delete_note({ target_mode: "specific", vault: "V", path: "P.md" })`, **Then** the tool returns an MCP error response with `code: "CLI_BINARY_NOT_FOUND"` AND the error's `details` carries the binary-attempted name and `PATH` per the adapter's contract.
2. **Given** the `delete_note` tool and a stub adapter that exits `1` with stderr `"permission denied"` (e.g., a read-only file or vault root), **When** an MCP client calls `delete_note({ target_mode: "specific", vault: "V", path: "P.md" })`, **Then** the tool returns an MCP error response with `code: "CLI_NON_ZERO_EXIT"` AND `details.exitCode: 1` AND `details.stderr` carries the verbatim CLI stderr.
3. **Given** the `delete_note` tool and a stub adapter that exits `0` with stdout `"Error: file not found at Inbox/Missing.md\n"`, **When** an MCP client calls `delete_note({ target_mode: "specific", vault: "V", path: "Inbox/Missing.md" })` (file deleted out-of-band before the call), **Then** the tool returns an MCP error response with `code: "CLI_REPORTED_ERROR"` AND `details.message` carries the verbatim wording from the CLI — the adapter's classification is propagated unchanged.
4. **Given** the `delete_note` tool, **When** any non-`UpstreamError` exception escapes the adapter (an unexpected runtime error), **Then** the handler re-throws it (does NOT wrap it as a tool error), matching the obsidian_exec / read_note / write_note precedent.

### User Story 7 — Tool registration emits a stripped JSON Schema and a `help`-aware top-level description (Priority: P2)

Per ADR-005, every tool registered with the MCP server passes its `zod-to-json-schema` output through the schema-stripping utility before handing the schema to the SDK, AND its top-level `description` is a concise verb-led summary that mentions `help("<tool_name>")` as the recovery path for full parameter docs. `delete_note` MUST satisfy both contracts. The published `inputSchema` MUST be a flat JSON Schema with `additionalProperties: false` (the post-010 shape; see [010-flatten-target-mode](../010-flatten-target-mode/spec.md) FR-007 / contract.md), describing `target_mode` (enum), `vault`/`file`/`path` (string, optional), and `permanent` (boolean, optional) at the top level — no `oneOf` envelope, no nested per-branch unions.

**Why this priority**: P2 because the underlying machinery (the strip utility, the help tool, the post-010 emit shape) already exists and is gated by registry-consistency tests. This BI's job is to slot `delete_note` into that pipeline correctly and replace its stub doc.

**Independent Test**: Three registration-level tests at `src/tools/delete_note/index.test.ts`: (a) `createDeleteNoteTool()` returns a `RegisteredTool` whose `descriptor.name === "delete_note"`; (b) the descriptor's `inputSchema` has zero `description` keys at any depth AND has `additionalProperties: false` at the top level AND has all five top-level properties (`target_mode`, `vault`, `file`, `path`, `permanent`) typed inline (no `oneOf`); (c) the descriptor's `description` field is non-empty and contains the literal substring `"help"` (case-insensitive) plus a reference to `delete_note` by name, AND surfaces the safety-default disclosure (the to-trash default and the irreversibility of `permanent: true`). The post-010 consolidated drift detector at [src/tools/_register.test.ts](../../src/tools/_register.test.ts) automatically covers `delete_note` via its `it.each` registry walk; the docs-existence assertion at [src/server.test.ts](../../src/server.test.ts) automatically asserts `docs/tools/delete_note.md` exists and is non-stub.

**Acceptance Scenarios**:

1. **Given** the MCP server with `delete_note` registered, **When** an MCP client calls `tools/list`, **Then** the returned descriptor for `delete_note` has an `inputSchema` with `type: "object"`, `additionalProperties: false`, top-level `properties` listing all five fields (`target_mode`, `vault`, `file`, `path`, `permanent`), and `required` listing `["target_mode"]` (or equivalent — the exact `required` set is determined by the flat-emit shape post-010; `vault` is conditional on `target_mode` and is enforced by the runtime `superRefine` rather than the JSON Schema's `required`).
2. **Given** the same `tools/list` response, **When** the descriptor for `delete_note`'s `inputSchema` is walked, **Then** there are zero `description` keys at any depth (per the strip utility's contract).
3. **Given** the same response, **When** the top-level `description` field is inspected, **Then** it is a concise verb-led summary, non-empty, contains the literal substring `"help"` (case-insensitive), references invoking it with the tool's own name (e.g., `help("delete_note")` or `help({ tool_name: "delete_note" })`), AND surfaces the safety-default disclosure (the to-trash default and the irreversibility of `permanent: true`).
4. **Given** the `help` tool, **When** an MCP client calls `help({ tool_name: "delete_note" })`, **Then** it returns the full Markdown body of `docs/tools/delete_note.md` — the file does NOT carry a TODO/stub marker because this BI populates it. The body documents the input schema (target_mode + vault + file/path + permanent), the output shape (`{ deleted: true, path: string, toTrash: boolean }`), the propagated error codes (`VALIDATION_ERROR`, `CLI_BINARY_NOT_FOUND`, `CLI_NON_ZERO_EXIT`, `CLI_REPORTED_ERROR`, `ERR_NO_ACTIVE_FILE`), the safety defaults (permanent false → to trash by default, with explicit irreversibility warning for `permanent: true`), and **at least four worked examples** covering: (i) specific-mode `path=` to-trash delete, (ii) specific-mode `file=` to-trash delete, (iii) specific-mode `path=` permanent delete, and (iv) failure recovery from a non-existent file. Per the user input AC #11.

### User Story 8 — Audit-trail `toTrash` field is correctly populated for every successful call (Priority: P3)

The output's `toTrash: boolean` field is the audit-trail signal for whether a given call ran the destructive variant. Every successful call MUST populate `toTrash` correctly: `true` when the file went to trash, `false` when it was permanently deleted. Operators auditing logs filter on `toTrash === false` to surface every irreversible deletion. The field's correctness is therefore a precondition for the security claim that "the typed `permanent` flag IS the audit point" (per the user input's SECURITY adversarial bullet).

**Why this priority**: P3. The structural contracts in Stories 1–4 already require the field to be populated; this story exists to make the audit-trail claim a first-class verifiable invariant rather than a side effect of the other stories. Every handler test covering a successful delete (Stories 1–4) also asserts on `toTrash`; this story reframes those assertions as a single auditability requirement so reviewers can trace the user input's security posture to a concrete check.

**Independent Test**: A parameterised test that drives the handler across all four success-path combinations (specific + permanent omitted; specific + permanent: false; specific + permanent: true; active + permanent: true) and asserts each output's `toTrash` matches the inverse of the call's effective `permanent` value. The test does NOT need a real CLI; the stub adapter reports the appropriate success-shaped response in each case.

**Acceptance Scenarios**:

1. **Given** the `delete_note` tool and a stub adapter reporting a successful to-trash delete, **When** an MCP client calls `delete_note` in any of the to-trash configurations (specific + omitted, specific + false, active + omitted, active + false), **Then** the tool's response carries `toTrash: true` in every case.
2. **Given** the `delete_note` tool and a stub adapter reporting a successful permanent delete, **When** an MCP client calls `delete_note` in any of the permanent configurations (specific + true, active + true), **Then** the tool's response carries `toTrash: false` in every case.

### Edge Cases

- **`delete_note({ target_mode: "specific", vault: "", path: "P.md" })`** (empty-string vault): MUST fail at the zod parse boundary as `VALIDATION_ERROR` whose `details.issues[].path` includes `"vault"`. The target-mode primitive enforces `vault: z.string().min(1)`. Reasonable default — error message mirrors zod's standard `min(1)` message.
- **`delete_note({ target_mode: "specific", vault: "V", path: "../../../etc/passwd.md" })`** (path-traversal-shaped value): MUST be either rejected at the validation boundary or verified to be rejected by the underlying CLI. Per the user input's FILESYSTEM adversarial bullet, **silent vault-escape is a critical security defect** — deleting files outside the vault root is unacceptable. Plan-stage characterisation per FR-019 case (vii) verifies the CLI's behaviour against `../`-shaped paths; if the CLI does NOT reject such inputs, this BI is amended pre-ship to add a tool-layer reject (a precondition, not an open question — gated by SC-012). The dominant security concern for `delete_note` is path-traversal because shell-injection is structurally prevented by the cli-adapter's argv-array passing.
- **`delete_note({ target_mode: "specific", vault: "DoesNotExist", path: "P.md" })`** (unknown vault display name): inherits the response-inspection clause that 011-write-note's R5 added to the cli-adapter. The unknown-vault response is re-classified by the adapter to `CLI_REPORTED_ERROR` with the verbatim CLI message preserved in `details.message`. `delete_note` propagates this without modification. Plan-stage characterisation per FR-019 case (v) confirms the CLI's response signature is identical for the delete subcommand to the create subcommand (where 011 verified `Vault not found.` on stdout); if it differs, the adapter's response-inspection logic is extended (additive change to the adapter, not to `delete_note`).
- **File deleted out-of-band before the call** (e.g., another agent or the user removed it via Obsidian's UI): surfaces as a structured `CLI_REPORTED_ERROR` (or `CLI_NON_ZERO_EXIT` depending on the CLI's exit code for the not-found case). Plan-stage research per FR-019 captures the verbatim wording (likely `Error: file not found …` or similar) and the handler propagates it. No silent no-op — the second caller MUST see a structured error per the user input's CONCURRENCY adversarial bullet.
- **Two concurrent `delete_note` calls against the same path**: the shared in-flight CLI queue (per FR-008, inherited from [006-read-note](../006-read-note/spec.md) FR-016 and [011-write-note](../011-write-note/spec.md) FR-008) serializes both calls behind one channel. The first call deletes the file; the second call sees the on-disk state after the first call completed and surfaces a `CLI_REPORTED_ERROR` (file-not-found). Deterministic ordering is guaranteed by the queue; silent no-ops are precluded by the file-not-found path. Per the user input's CONCURRENCY adversarial bullet.
- **External editor has the file open** (Obsidian, VS Code, anything else) when `delete_note` is invoked: observable behaviour is captured during plan-stage characterisation per FR-019 and documented in `docs/tools/delete_note.md`. `delete_note` does NOT lock the file or coordinate with external editors. If the OS rejects the delete with EBUSY (Windows), the CLI surfaces non-zero exit; the handler propagates as `CLI_NON_ZERO_EXIT`. If the OS allows the delete and Obsidian closes the buffer (or shows a stale view), that is the editor's behaviour, not `delete_note`'s.
- **A read operation against the same path arriving in flight while delete is mid-execution**: the shared CLI queue serializes both calls. If the read is queued first, it returns the file's contents and then the delete proceeds; if the delete is queued first, the read returns a `CLI_REPORTED_ERROR` (file-not-found). No torn reads — the queue's serialization guarantees atomicity at the call boundary, which is the strongest guarantee the wrapper offers (the OS / FS provides whatever atomicity it provides at the syscall layer, which is out of scope for this BI). Per the user input's CONCURRENCY adversarial bullet.
- **Active-mode TOCTOU**: the focused note may change between the schema's parse step and the CLI's actual invocation. The operation runs against whichever note is focused at execution time. Documented as a known limitation; `delete_note` does NOT attempt to lock or pin the focused note. For an irreversible operation, this is the load-bearing caveat — agents that need certainty about which file is being deleted MUST use specific mode with an explicit locator. Per the user input's CONCURRENCY adversarial bullet.
- **Permissions error on the file or vault root** (file marked read-only, vault root marked read-only, OS-level ACL refusing the user): the CLI surfaces non-zero exit with stderr describing the OS error; the adapter classifies as `CLI_NON_ZERO_EXIT`; the handler propagates. Plan-stage characterisation per FR-019 captures the exact code surfaced for this case (likely `CLI_NON_ZERO_EXIT` per the existing taxonomy).
- **Reserved Windows characters / names** (`CON`, `PRN`, `AUX`, `NUL`, files with `:?*<>|"` in the name): platform-dependent. The CLI is the trust boundary; `delete_note` forwards verbatim. Plan-stage CLI characterisation per FR-019 records the observed behaviour (likely platform-specific); the documented behaviour is captured in `docs/tools/delete_note.md` per FR-014. Per the user input's FILESYSTEM adversarial bullet.
- **Trash-volume-full**: on some platforms (Windows in particular), the recycle bin has a per-volume size cap. Sending a large file to trash may produce a structured error (the CLI surfaces non-zero exit) or, on some platforms, may fall back to permanent delete WITHOUT the caller's opt-in. **The fall-back-to-permanent case is a safety violation** of the typed surface's contract. Plan-stage research per FR-019 captures the CLI's behaviour for this case; if the CLI silently falls back, this BI is amended pre-ship to either (a) detect and report the fall-back as a structured error, or (b) document the platform-specific limitation prominently in `docs/tools/delete_note.md` and surface a `toTrash: false` audit-trail signal in the response. The first option (refuse to silently fall back) is preferred. Per the user input's FILESYSTEM adversarial bullet.
- **`delete_note({ target_mode: "active", file: "Foo" })`** (forbidden key in active mode): MUST fail at the zod parse boundary, mirroring `read_note`'s and `write_note`'s behaviour. The same applies for `vault` and `path`. The error message MUST NOT include recovery instructions per the target-mode primitive's clarification.
- **`delete_note({ target_mode: "specific", vault: "V", path: "P.md", permanent: undefined })`** (explicit-undefined): zod's `.optional()` accepts `undefined` as the omitted form; this resolves to the schema's default of `false` (per FR-005 / FR-007). Behaviour is identical to omitting the key entirely.
- **Caller-supplied `path` / `file` containing argv-shell metacharacters** (e.g., `path: "; rm -rf /; #"`): SAFE. The cli-adapter assembles argv as a child_process.spawn array (no shell), so no character in `path` / `file` can break out into a shell. This is a structural property of the adapter, not a runtime check. Path-traversal (the dominant remaining security concern) is covered by the bullet above. Per the user input's SECURITY bullet.
- **Strict-rich vs strict-naive client-class observability of unknown-key rejection** (Story 5 AC #5): strict-rich clients (Claude Desktop, MCP Inspector) read the `additionalProperties: false` from the published `inputSchema` and either forward unknown keys (in which case the bridge-side `unrecognized_keys` rejection is observable to the test) or strip them client-side (in which case the bridge-side rejection is non-observable but the schema-side invariant — no untyped passthrough at runtime — still holds). Strict-naive clients (Cowork) do not consult `additionalProperties`; they forward unknown keys to the bridge, which always observes the rejection. Tests assume the strict-rich-forwarding path (the worst case) and assert the bridge-side rejection occurs. Documented per the user input's CLIENT-CLASS adversarial bullet.

### Functional Requirements

#### Module layout

- **FR-001**: A new typed tool MUST be created at `src/tools/delete_note/` following the existing per-surface module convention (per Constitution Principle I and the `src/tools/help/`, `src/tools/obsidian_exec/`, `src/tools/read_note/`, `src/tools/write_note/` precedents). The module MUST contain six files at minimum (three source + three co-located tests): `schema.ts` (the zod input schema, the inferred TypeScript types, and the JSON Schema export via `zod-to-json-schema`), `handler.ts` (the function that calls the adapter and maps the result to the output shape), `index.ts` (the `createDeleteNoteTool` factory returning a `RegisteredTool` per the aggregator pattern — `index.ts` per the post-011 file-name convention, NOT `tool.ts`), and one co-located test file per source file (`schema.test.ts`, `handler.test.ts`, `index.test.ts`). Per Constitution Principle V, every new source file MUST carry an `// Original — no upstream. <one-line description>.` header.

#### Schema

- **FR-002**: The input schema MUST compose the post-010 flat `target_mode` primitive (per [010-flatten-target-mode](../010-flatten-target-mode/spec.md)) using the canonical Pattern (a) flat-extension idiom: `applyTargetModeRefinement(targetModeBaseSchema.extend({ permanent }))`. Per the contract at [specs/010-flatten-target-mode/contracts/flat-target-mode.contract.md](../010-flatten-target-mode/contracts/flat-target-mode.contract.md), `.extend()` (NOT `.merge()`) is the binding API call — `.merge()` resets `unknownKeys` to `"strip"` and silently drops the strict-mode safety net for unknown top-level keys. The composed schema inherits the primitive's full per-mode contract end-to-end: `vault` required in specific mode, exactly-one-of `file`/`path` in specific mode, `vault`/`file`/`path` forbidden in active mode, top-level `additionalProperties: false` against unknown keys.

  **Active-mode `superRefine` clauses for `delete_note`**: NONE. Unlike `write_note`, `delete_note` does not need a tool-specific active-mode refinement. The `permanent` field has well-defined semantics in both modes (irreversibility applies regardless of how the locator is resolved), so it is permitted in both. The target-mode primitive's existing forbidden-key rules (vault/file/path) are sufficient. The schema therefore reduces to `applyTargetModeRefinement(targetModeBaseSchema.extend({ permanent: z.boolean().optional().default(false) }))` with no further `.superRefine(...)` chain.

- **FR-003**: The schema MUST add one `delete_note`-specific field:
    - `permanent: z.boolean().optional().default(false)` — optional with default `false` in BOTH modes. The default-false coercion guarantees `parsed.permanent` is a boolean (never `undefined`) after parse; the handler's "emit only when true" rule (FR-007) is a serialisation choice on top of guaranteed booleans, parity with `write_note`'s `overwrite`/`open` rules per [011-write-note](../011-write-note/spec.md) FR-005.

- **FR-004**: The schema MUST NOT add `.describe()` annotations to any field. Parameter documentation lives in `docs/tools/delete_note.md` (per ADR-005's progressive-disclosure design).

- **FR-005**: The schema MUST export the inferred TypeScript type via `z.infer<typeof deleteNoteInputSchema>` (or equivalent name). No hand-written `interface DeleteNoteInput` or `type DeleteNoteInput = { … }` is permitted (Constitution Principle III). The output type MUST also be a zod schema (or its `z.infer<…>`): `deleteNoteOutputSchema = z.object({ deleted: z.literal(true), path: z.string(), toTrash: z.boolean() }).strict()`. The `deleted` field is a `z.literal(true)` because every successful return path of `delete_note` produces `deleted: true` — the failure path goes through `UpstreamError`, not through a `deleted: false` shape (parity with `read_note`'s response shape, which does not carry a `read: false` discriminator). Hand-rolled output types are forbidden by the same principle.

- **FR-006**: The schema MUST export a JSON Schema document via `zod-to-json-schema(...)` (mirroring `obsidian_exec`, `read_note`, and `write_note`). The exported JSON Schema MUST round-trip through the schema-stripping utility at registration time per FR-011. The emitted JSON Schema MUST be the post-010 flat shape (one top-level `type: "object"`, `properties` with all five top-level fields typed inline, `additionalProperties: false`, no `oneOf` envelope) — verifiable via the consolidated drift detector at [src/tools/_register.test.ts](../../src/tools/_register.test.ts) which automatically covers every registered tool.

#### Handler

- **FR-007**: The handler MUST route the validated input through the cli-adapter's `invokeCli` entry point (or its bounded counterpart through the shared queue per FR-008). The adapter call shape is `invokeCli({ command: "<delete subcommand name>", parameters: <derived>, flags: <derived>, target_mode: <validated>, vault: <hoisted-or-undefined> }, deps?)`, where the delete subcommand name (`delete`, `trash`, `rm`, or whatever the live CLI exposes) is captured during plan-stage research per FR-019. The `parameters` and `flags` records are derived from the validated input as follows:
    - **Specific mode**: `vault: parsed.vault` is passed as a TOP-LEVEL field (per [011-write-note](../011-write-note/spec.md) PSR-3) NOT inside `parameters`. `parameters` includes exactly one of `file: parsed.file` or `path: parsed.path` (with the user-facing field name potentially renamed to a CLI argv key per the 011-write-note PSR-5 precedent — confirmed during plan-stage research per FR-019). `permanent` is conditionally emitted only if `parsed.permanent === true` — the schema's default false MUST NOT emit a `permanent`-shaped token in argv (parity with `write_note`'s `overwrite`/`open` rules per Story 1 AC#2 + Story 3 AC#2).
    - **Active mode**: `parameters` is empty (or contains only the `permanent` token if `parsed.permanent === true`). The schema layer has already rejected `vault`/`file`/`path` so the handler does not include them; the adapter's defensive locator-stripping is a redundant safety net. The top-level `vault` field passed to `invokeCli` is `undefined` in active mode.
    - The choice between key=value form (`permanent=true`) and flag form (a bare-word `permanent` in `flags: []`) for the `permanent` token is a plan-stage research decision per FR-019, parity with the 011-write-note PSR-4 resolution for `overwrite`/`open`. The structural contract (no argv emission for default-false booleans, structured emission for explicit-true booleans) binds; the placement (parameters vs flags) does not.

  Plan-stage research per FR-019 confirms the live CLI's actual delete-subcommand name and the exact argv shape (whether the safety default is to-trash with a `permanent` opt-in flag, or the inverse — permanent by default with a `--trash` flag — in which case the typed surface inverts the convention so the user-facing default remains to-trash, per Story 3 AC#3 and the user input's SAFETY DEFAULTS adversarial bullet). The handler's argv-mapping locks against the live CLI's behaviour.

- **FR-008**: The handler MUST execute through the same single-in-flight CLI queue that gates `obsidian_exec`, `read_note`, and `write_note` (per [006-read-note](../006-read-note/spec.md) FR-016 and [011-write-note](../011-write-note/spec.md) FR-008). `createDeleteNoteTool`'s `RegisterDeps` MUST accept `queue: Queue`, and `src/server.ts` MUST pass the *same* `queue` instance to `createDeleteNoteTool`, `createWriteNoteTool`, `createReadNoteTool`, and `createObsidianExecTool` so concurrent CLI invocations across all four tools serialize through one channel. This forecloses the "two concurrent deletes to the same path" race in Edge Cases.

- **FR-009**: The handler MUST be a thin `invokeCli` wrapper with no per-call logger events at the tool layer, parity with the actual `read_note` and `write_note` handler implementations (per [011-write-note](../011-write-note/spec.md) PSR-1 / R1 — the spec-level requirement that earlier specs phrased as "logger.callStart / callEndSuccess / callEndFailure events" superseded by the actual sibling implementations). Observability flows through the cli-adapter's existing `dispatchTimeout` / `dispatchCap` / `dispatchKill` events end-to-end. `RegisterDeps` MUST still accept `logger: Logger` for forwarding to the adapter / queue layer; `src/server.ts` MUST pass the same logger instance to all tool registrations.

- **FR-010**: The handler MUST return a successful tool-call result whose JSON-serialised payload conforms to the output schema: `{ deleted: true, path: "<canonical vault-relative path>", toTrash: <bool> }`. The `toTrash` boolean is derived from the call's effective `permanent` value: `toTrash = !parsed.permanent`. This derivation is structural (not parsed from the CLI's response) because the typed surface owns the safety-default contract — the handler knows which variant ran, regardless of how the CLI describes its action. The `path` field is the canonical vault-relative path the CLI reports (which may differ from the input `file`-form locator's wikilink). Plan-stage research per FR-019 captures the CLI's exact response wording for both successful to-trash and successful permanent-delete cases; the handler's response-parsing logic for the path field is locked against that wording. **Important**: the response's `path` reflects the file's vault-relative path AT THE MOMENT OF DELETION (not after — the file is gone), so callers can audit / log what was removed. Per the user input's [P3] AC #13. The handler MUST NOT trim, normalise, or post-process the path; it propagates the CLI's reported value verbatim.

### Out of Scope

The following are explicitly excluded from this feature and remain the responsibility of separate BIs or future work (mirroring the user input's "Out of scope" enumeration plus consequential exclusions surfaced during spec drafting):

- **Recursive directory delete**: the underlying CLI surface is single-file. Folder deletion is a separate concern; future BI if the use case justifies it.
- **Bulk multi-file delete**: callers iterate today, or a future batch-delete BI ships if usage justifies. The single-file typed surface does not aggregate calls.
- **Soft-delete with a custom restore mechanism**: Obsidian's File Recovery plugin and the OS trash already cover restoration paths; the wrapper does not add a parallel scheme.
- **Pre-delete confirmation prompts**: confirmation is the caller's responsibility; the typed surface returns immediately. An MCP client UI may surface a confirmation step before invoking `delete_note` — that is a client-side concern, not a wrapper concern.
- **Atomicity / undo guarantees beyond what the underlying execution layer provides**: interrupted deletes (power loss, kill -9 mid-spawn) leave the on-disk state to the OS, not the wrapper. If empirical use surfaces a real-world atomicity defect, the resolution lands at the CLI layer or in a primitive amendment.
- **Path-traversal validation at the tool layer**: out of scope per Edge Cases — UNLESS plan-stage characterisation per FR-019 case (vii) surfaces that the CLI does not reject `../`-shaped paths, in which case the spec is amended pre-ship to add a tool-layer reject. This is a precondition, not an open question (gated by SC-012). Silent vault-escape on a destructive operation is a critical security defect.
- **Trash-recovery automation**: the wrapper does not surface "restore from trash" as a tool. Restoration is a per-platform UI workflow (Windows recycle bin, macOS Finder trash, Linux distro-specific). If empirical use surfaces a need for programmatic restoration, a future `restore_from_trash` BI ships separately.
- **A `list_vaults` primitive for vault-display-name pre-validation**: out of scope. The unknown-vault edge case is handled by the cli-adapter's response-inspection clause (per [011-write-note](../011-write-note/spec.md) R5). If response-inspection proves brittle, `list_vaults` ships as a follow-up BI and pre-validation moves there.
- **New error codes**: out of scope per the user input. `delete_note`'s entire failure surface is covered by `VALIDATION_ERROR` + the adapter's four codes. If a `delete_note`-specific failure mode emerges (e.g., "trash volume full and the CLI silently falls back to permanent"), a future amendment introduces a code with the same rigour as [005-help-tool](../005-help-tool/spec.md) introduced `HELP_TOOL_NOT_FOUND` / `HELP_DOCS_MISSING`.
- **Refactoring the existing typed tools**: out of scope. `obsidian_exec`, `help`, `read_note`, and `write_note` co-exist with the new tool unchanged. `delete_note` adds rather than perturbs (per SC-009).
- **Modifying ADR-003, ADR-005, or ADR-006**: this BI implements all three; it does not change them. Conflicts surface as new ADRs per the project's amendment procedure.

### Key Entities *(include if data involved)*

- **`delete_note` MCP tool**: a public MCP tool registered alongside `obsidian_exec`, `help`, `read_note`, and `write_note`. Tool name is the literal `"delete_note"`. Lives at `src/tools/delete_note/` with the project's per-surface module layout (`schema.ts`, `handler.ts`, `index.ts` + co-located tests).
- **`delete_note` input schema** (composed via the post-010 Pattern (a) flat-extension idiom per [010-flatten-target-mode](../010-flatten-target-mode/spec.md)): `applyTargetModeRefinement(targetModeBaseSchema.extend({ permanent: z.boolean().optional().default(false) }))`. The composed schema's runtime contract: `target_mode: "specific" | "active"` (required); `vault` required in specific, forbidden in active; exactly one of `file`/`path` in specific, both forbidden in active; `additionalProperties: false` against any other top-level key. `permanent` is optional in both modes (default false). No tool-specific active-mode `superRefine` clauses (unlike `write_note`).
- **`delete_note` output schema**: `z.object({ deleted: z.literal(true), path: z.string(), toTrash: z.boolean() }).strict()`. The `deleted` field is a literal `true` because every successful return path produces it; failures surface through `UpstreamError`. JSON-serialised into the MCP `content[0].text` envelope per the existing tool-call-result convention. The output type is also derived via `z.infer<typeof deleteNoteOutputSchema>` per Constitution Principle III; no hand-rolled `interface` or `type` declaration.
- **CLI invocation**: `invokeCli({ command: "<delete subcommand>", parameters, flags, target_mode, vault }, deps?)` per [003-cli-adapter](../003-cli-adapter/spec.md), executed through the shared single-in-flight CLI queue (per FR-008). Logger interaction is at the queue / adapter layer only; no per-call events at the tool layer per FR-009. The adapter handles vault hoisting, active-mode locator stripping, and the four-priority error classification including the unknown-vault response-inspection clause inherited from 011-write-note R5. `delete_note` does not interact with `child_process.spawn` directly.
- **`RegisterDeps` shape**: the `createDeleteNoteTool` factory accepts `{ logger: Logger, queue: Queue }` plus the cli-adapter's `deps` pass-through (`spawnFn`, `env`) for tests, mirroring `createWriteNoteTool`'s and `createReadNoteTool`'s shapes exactly. `src/server.ts`'s aggregator wiring stays symmetrical across all five tools.

Operators auditing logs filter by `toTrash === false` to surface every irreversible deletion. The typed `permanent` flag IS the audit point per the user input's SECURITY adversarial bullet.

Per Constitution Principle IV, every failure surface flows through `UpstreamError`; this BI introduces zero new codes.

Documents input schema, output shape, error roster (with explicit irreversibility warning for `permanent: true`), safety defaults, four worked examples, adversarial-edge-case behaviours captured during plan-stage research, audit-trail guidance, and cross-references per FR-014.

## Assumptions

- **The user input's `(required)` annotation on `target_mode`** is binding. The schema MUST NOT make `target_mode` optional. The `(optional)` annotation on `permanent` is also binding (default false); `vault` is "required in specific mode, forbidden in active mode" by reference to the target-mode primitive's contract.
- **The CLI exposes a single-file delete subcommand** (likely named `delete`, `trash`, or `rm` — captured during plan-stage research per FR-019) that accepts the argv shape `["<delete-subcommand>", "vault=…", "file=… | path=…"]` plus an optional `permanent`-shaped token (key=value form `permanent=true`, or flag form bare-word `permanent`, captured during plan-stage research), and emits a structured success or failure response on stdout/stderr that the adapter can classify. This is the precondition the tool is built on; if the CLI uses a different shape, plan-stage characterisation per FR-019 captures the actual shape and the handler's argv-assembly retargets accordingly. Reasonable default: argv assembly is `[command, vault=…, locator=…, …extras]` parity with the create subcommand.
- **The CLI's safety default is to-trash with a `permanent` opt-in flag** (matching the typed surface's exposed contract). If plan-stage research per FR-019 case (ix) finds the CLI's default is the OPPOSITE (permanent-by-default with a `--trash` opt-in flag), the typed surface inverts the convention so the user-facing default remains to-trash per Story 3 AC#3 / the user input's SAFETY DEFAULTS adversarial bullet. The structural contract (typed surface's default is to-trash regardless of the CLI's underlying convention) binds; the argv-assembly placement does not.
- **The CLI's response signal for to-trash vs permanent** is captured during plan-stage research per FR-019 (cases (i) + (iii)). Because `toTrash` is derived structurally (from the call's effective `permanent`, not parsed from the response), the CLI is NOT required to distinguish in its output — but if it does, the handler's path-extraction logic locks against whichever wording the live CLI uses for both cases.
- **The CLI's response signal for the canonical path** is captured during plan-stage research per FR-019 (cases (i)–(iii) + (vi)). Plausible signals include: a stdout marker (`"Trashed Inbox/Old.md"` / `"Deleted Inbox/Old.md"`); the path on stdout alone; or a JSON-shaped response. The handler's response-parsing logic locks against whichever signal the live CLI uses. If the CLI does not echo the canonical path (e.g., it returns only `"OK"` regardless of input), this BI is amended pre-ship to either (a) accept that the response's `path` mirrors the input locator (with the wikilink expanded to a default-folder path, requiring a fallback rule documented in `docs/tools/delete_note.md`), or (b) request a CLI-layer enhancement to disambiguate. The amendment lands before merge per FR-019.
- **The active-mode CLI invocation** for the delete subcommand (`obsidian <delete>` with no vault/file/path) removes the focused note in the focused vault. If no vault is focused, the CLI surfaces `Error: no active file` (or equivalent), classified by the adapter as `ERR_NO_ACTIVE_FILE`. This is consistent with the read and create subcommands' active-mode behaviour observed during prior BIs; if the delete subcommand's active-mode behaviour differs, plan-stage characterisation per FR-019 case (vi) captures the difference and the spec is amended.
- **The `permanent` boolean schema default** (`false`) is part of the schema's parsed output, NOT of the handler's logic. After `schema.parse(args)` returns, `parsed.permanent` is guaranteed to be a boolean (no `undefined`); the handler's "emit only when true" rule (FR-007) and the `toTrash = !parsed.permanent` derivation (FR-010) are serialisation choices on top of guaranteed booleans, not fallback chains. This avoids the "two-place default" anti-pattern.
- **The post-010 strict-mode `additionalProperties: false`** is the source-of-truth for the unknown-top-level-key rejection (Story 5 AC#5). `delete_note` does NOT add `.strict()` redundantly at the tool layer; it inherits strict mode from `targetModeBaseSchema.extend(...)` per [010-flatten-target-mode](../010-flatten-target-mode/spec.md) FR-002 (which verified that `.extend()` preserves strict mode while `.merge()` does not).
- **The cli-adapter's `deps.spawnFn` test seam** is the canonical test-injection mechanism. `delete_note`'s handler tests inject `deps.spawnFn` via the adapter call, NOT by mocking the adapter module itself. This matches `read_note`'s, `write_note`'s, and `obsidian_exec`'s test patterns.
- **No new error codes** are introduced. Per Constitution Principle IV, error codes are the project's stable API. `delete_note`'s entire failure surface is covered by `VALIDATION_ERROR` (zod) and the adapter's four codes. If empirical use surfaces a `delete_note`-specific failure mode (trash-volume-full silent-fall-back is the anticipated candidate), a future amendment introduces the code with the same rigour as ADR-006 / [005-help-tool](../005-help-tool/spec.md).
- **Constitution Principle II binds the test-set co-location** at `src/tools/delete_note/{schema,handler,index}.test.ts`. The cases enumerated in FR-016 are the minimum set; reviewers may add more.
- **Constitution Principle V binds the module-header convention**: `// Original — no upstream. <one-line description>.` is required on every new source file in `src/tools/delete_note/`. The Markdown documentation file at `docs/tools/delete_note.md` is exempt per [005-help-tool](../005-help-tool/spec.md) FR-019.
- **`toTrash` is derived structurally** (not from the CLI's response). The handler computes `toTrash = !parsed.permanent` after a successful adapter call. This is a deliberate design choice: the typed surface owns the safety-default contract, so the user-facing `toTrash` audit signal must be locked to the typed surface's input semantics rather than the CLI's response wording (which might be platform-dependent or otherwise unstable). If the CLI's actual behaviour drifts from this invariant (e.g., it silently falls back to permanent on trash-volume-full), SC-013 catches the drift and the handler is amended to detect and surface the fall-back as a structured error.
