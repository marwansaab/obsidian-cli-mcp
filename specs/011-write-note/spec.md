# Feature Specification: Write Note Typed MCP Tool

**Feature Branch**: `011-write-note`
**Created**: 2026-05-08
**Status**: Draft
**Input**: User description: "Add Write Note — A typed MCP tool that creates a new vault note, or overwrites an existing one when the caller explicitly opts in, with given UTF-8 content. Direct one-to-one wrap of the Obsidian CLI's `create` subcommand."

## Background *(non-mandatory context)*

This feature adds the second typed-tool wrap on top of the foundation completed by features 003–010. It is the symmetric counterpart of [006-read-note](../006-read-note/spec.md): where `read_note` is the typed read primitive that retired `obsidian_exec` for reads, `write_note` is the typed create/overwrite primitive that retires `obsidian_exec` for writes. It composes on three load-bearing primitives:

- **[004-target-mode-schema](../004-target-mode-schema/spec.md) / [010-flatten-target-mode](../010-flatten-target-mode/spec.md)** — the shared `target_mode` schema primitive at [src/target-mode/target-mode.ts](../../src/target-mode/target-mode.ts). Post-010 the canonical reuse idiom is `applyTargetModeRefinement(targetModeBaseSchema.extend({ <tool-specific fields> }))` (the Pattern (a) flat-extension form). Per Constitution Principle III, this primitive owns the `target_mode` discriminator, the `vault`/`file`/`path` keys, the specific-branch XOR locator rule, the active-branch forbidden-key rule, and the strict-mode posture (`additionalProperties: false`); `write_note` MUST compose on top of it rather than re-state any of those rules inline.
- **[003-cli-adapter](../003-cli-adapter/spec.md)** — the centralised CLI adapter at [src/cli-adapter/cli-adapter.ts](../../src/cli-adapter/cli-adapter.ts). The handler routes every CLI invocation through `invokeCli({ command, parameters, flags, target_mode }, deps?)` (or its bounded counterpart through the shared queue). The adapter owns argv assembly, vault hoisting, active-mode locator stripping, and the four-priority error classification (`CLI_BINARY_NOT_FOUND`, `CLI_NON_ZERO_EXIT`, `ERR_NO_ACTIVE_FILE`, `CLI_REPORTED_ERROR`). `write_note` MUST NOT spawn child processes directly.
- **[005-help-tool](../005-help-tool/spec.md)** — the `help` tool and the schema-stripping utility at [src/help/strip-schema.ts](../../src/help/strip-schema.ts). The registry-consistency test at [src/server.test.ts](../../src/server.test.ts) gates the existence of `docs/tools/write_note.md`; this BI replaces that file's TODO stub (or creates it if no stub exists) with a non-stub Markdown body.
- **[006-read-note](../006-read-note/spec.md)** — the empirical validator of the foundation. `write_note` follows its module layout (`schema.ts` / `handler.ts` / `tool.ts` + co-located tests under `src/tools/write_note/`), its `RegisterDeps` shape (`logger`, `queue`, optional `deps` pass-through), and its handler-thinness ceiling (the handler is a tight argv-shape + result-mapping body bracketed by structured log events).

The user-facing contract this feature delivers: an MCP client calls `write_note({ target_mode: "specific", vault: "MyVault", path: "Inbox/Note.md", content: "<bytes>" })` (or with `file` instead of `path`, or with `target_mode: "active"` for the focused note) and receives `{ created: <bool>, path: "<canonical vault-relative path>" }`. The content bytes are written to disk verbatim by the underlying CLI; locator XOR, mode-forbidden-key, and strict-mode (`additionalProperties: false`) rejections all happen at the zod parse boundary; CLI failures (binary missing, non-zero exit, in-band `Error:` reply, no active file) flow through `UpstreamError`. No new error codes are introduced by this BI.

The feature replaces the universal escape hatch (`obsidian_exec`) for create/overwrite operations. After this BI lands, agents that today reach for `obsidian_exec` to issue `obsidian create …` get a typed, schema-validated, structurally-error-classified surface in `write_note`; `obsidian_exec` remains as the freeform escape hatch for subcommands not yet wrapped (the CLI's `newtab` flag, future subcommands, etc.).

## Clarifications

### Session 2026-05-08

- Q: When `write_note` is invoked in active mode WITHOUT setting `overwrite: true`, what should happen? → A: **Schema MUST require `overwrite: true` in active mode.** The safety-default rule ("MUST NOT overwrite an existing file by default ... explicit-opt-in via typed boolean fields") applies uniformly across both modes — active mode is destructive by definition (the focused note exists), so the same explicit-opt-in posture binds. Calls without `overwrite: true` in active mode fail at the schema layer as `VALIDATION_ERROR`. (See FR-002, FR-003, Story 5, Story 6 AC#8.)
- Q: Should the schema enforce a maximum size on the `content` field? → A: **No schema cap.** The OS argv ceiling (≈32 KiB Windows, ≈2 MiB Linux) applies; oversized writes fail at spawn time and are classified by the cli-adapter (typically `CLI_BINARY_NOT_FOUND` or a platform-specific code from the spawn layer). The OS ceiling is documented in `docs/tools/write_note.md` per FR-014. Rationale: keeps the schema platform-independent and aligns with the "CLI is the trust boundary for content/format concerns" pattern that `read_note` already follows. (Confirms current spec stance — no change.)
- Q: Should the schema forbid `template` and `open` in active mode (mirroring how `vault`/`file`/`path` are forbidden)? → A: **Both forbidden in active mode.** Their semantics in active mode are nonsensical — `template` is contradictory (templates apply at creation; active-mode rewrites the content of a note that already exists), `open` is meaningless (focused = already open). The schema's `superRefine` rejects either at parse time as `VALIDATION_ERROR`, mirroring the locator-forbids pattern. Constitutional consistency: schema is the gatekeeper for contractual-shape concerns; the CLI is not the trust boundary for "is this combination semantically valid." (See FR-002, FR-003, Story 6 AC#9, AC#10.)

## Plan-Stage Resolutions

Plan-stage research surfaced a small set of spec-vs-actual-codebase discrepancies that were ratified in [plan.md](plan.md) and [research.md](research.md) rather than via spec edits (per the project convention that research.md is the source of record for plan-stage discoveries; the spec captures intent at scaffold-time). This section indexes the resolutions so a reader of spec.md alone is not surprised at implementation time.

- **PSR-1 (logger surface — supersedes FR-009)**: FR-009 mandates handler-emitted `logger.callStart` / `logger.callEndSuccess` / `logger.callEndFailure` events "in parity with `read_note`". Live verification at plan-stage found those Logger methods do not exist (the [Logger interface](../../src/logger.ts) defines only `shutdown` / `dispatchTimeout` / `dispatchCap` / `dispatchKill`) AND that the actual [read_note handler](../../src/tools/read_note/handler.ts) does NOT emit per-call events. Per "spec follows the code that exists, not the code that was sketched", `write_note`'s handler mirrors the actual sibling shape: thin `invokeCli` wrapper, no per-call events. Observability via the cli-adapter's existing `dispatch*` events is preserved end-to-end. Resolution: [plan.md R1](plan.md), [research.md R1](research.md), [tasks.md T004 (4a)](tasks.md). FR-009's wording is left in place for historical traceability; this PSR is the operative contract.

- **PSR-2 (module file naming — supersedes FR-001 / FR-008 / FR-011 / FR-012 / FR-016 file-name references)**: FR-001 specifies the registration source file as `tool.ts` and the factory as `registerWriteNoteTool`. The actual project convention (verified against the sibling `read_note` and `obsidian_exec` modules) is `index.ts` and `createXxxTool`. `write_note` follows the actual convention: `src/tools/write_note/index.ts` exports `createWriteNoteTool(deps)`. The factory's wrapping behaviour (the `RegisteredTool` aggregator pattern, `stripSchemaDescriptions`, `ZodError → VALIDATION_ERROR`, `UpstreamError → asToolError`, JSON serialisation via `responseFormat: "json"`) is identical regardless of the file/factory name. Resolution: [plan.md Project Structure](plan.md), [data-model.md §Module Layout](data-model.md), [tasks.md T005](tasks.md). FR-016's "Tool-registration tests at `src/tools/write_note/tool.test.ts`" reads as `index.test.ts` in implementation.

- **PSR-3 (`vault` placement in `invokeCli` — supersedes FR-007 vault wording)**: FR-007's specific-mode bullet implies `vault` lives inside the `parameters` record passed to `invokeCli`. The actual `invokeCli` signature ([src/cli-adapter/cli-adapter.ts:13-26](../../src/cli-adapter/cli-adapter.ts#L13-L26)) takes `vault` as a TOP-LEVEL field alongside `command` / `parameters` / `flags` / `target_mode`. The handler passes `vault: parsed.vault` at the top level; `parameters` carries only the `name` / `path` / `content` / `template` tokens. Resolution: [plan.md Summary](plan.md), [contracts/write-note-handler.contract.md §`invokeCli` call shape](contracts/write-note-handler.contract.md), [tasks.md T004 (4a)](tasks.md). The cli-adapter's `dispatchCli` continues to hoist `vault=` to the argv-prefix slot, so the wire-level argv shape is unchanged.

- **PSR-4 (overwrite/open argv form — supersedes FR-007 active-mode parameters wording)**: FR-007's active-mode bullet says `parameters: { content: parsed.content, overwrite: "true" }` (key=value form for `overwrite`). Live verification via `obsidian help create` confirmed the CLI uses FLAG form (bare-word `overwrite` with no `=true` value) for `overwrite` / `open` / `newtab`, while `name=` / `path=` / `content=` / `template=` are key=value. The handler emits explicit-true booleans as flags in `flags: []`, never as `parameters`. FR-007 contains the `Plan-stage research per FR-019 confirms the CLI's actual flag shape` hedge that explicitly defers placement to plan-stage; this PSR closes that hedge. Resolution: [plan.md R2 / R3](plan.md), [research.md R2](research.md), [data-model.md §Argv mapping](data-model.md), [tasks.md T004 (4a)](tasks.md).

- **PSR-5 (`file` → `name=` argv rename — supersedes FR-007 file-token wording)**: the user-facing schema field `file` (parity with `read_note`'s convention) maps to the CLI argv token `name=<value>` for the `create` subcommand, NOT `file=<value>`. The CLI is internally inconsistent here: `read` uses `file=<name>`, `create` uses `name=<name>`. The handler does the rename at argv-assembly time so MCP clients see a uniform `file=` field across `read_note` and `write_note`. Resolution: [plan.md R3](plan.md), [research.md R3](research.md), [contracts/write-note-handler.contract.md §Argv assembly invariants](contracts/write-note-handler.contract.md), [tasks.md T004 (4a)](tasks.md).

- **PSR-6 (`open` field schema declaration — supersedes FR-003 `open` default)**: FR-003 declares `open: z.boolean().optional().default(false)`. Per [research.md R6](research.md), the active-mode `superRefine` needs to distinguish "key absent" (specific-mode acceptable) from "key present" (active-mode rejected); `.default(false)` collapses both states post-parse and breaks the distinction. Resolution: schema declares `open: z.boolean().optional()` (no `.default(false)`); handler reads `parsed.open ?? false`. The `overwrite` field keeps `.default(false)` because the active-mode rule there is "must be exactly true" — `false` (the default) fails that check naturally. Bounded asymmetry; documented in [data-model.md §Per-mode field policy](data-model.md), [contracts/write-note-input.contract.md](contracts/write-note-input.contract.md), and [tasks.md T003 (3a)](tasks.md). FR-005's "default is part of the schema, not a handler-layer fallback" remains the rule for `overwrite`; `open` carries the bounded exception with the reason captured here.

These PSRs do NOT modify the structural contracts in spec FR-001..FR-019. They record the surface-level resolutions needed to ground the spec against the actual codebase. The merge-stage Constitution Compliance checklist (FR-018 + [tasks.md T014](tasks.md)) cites these PSRs in the Principle IV / V evidence so reviewers can trace each resolution to its commit.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Specific-mode create at a new vault-relative path (Priority: P1)

An MCP client (typically an LLM agent) needs to create a new note at a known vault-relative path, with given UTF-8 content. The agent calls `write_note({ target_mode: "specific", vault: "MyVault", path: "Inbox/Idea.md", content: "# Idea\n\nBody\n" })` against a vault where `Inbox/Idea.md` does not yet exist, and receives `{ created: true, path: "Inbox/Idea.md" }`. The bytes on disk match the `content` input exactly — no trimming, no line-ending normalisation, no BOM stripping.

**Why this priority**: This is the headline value of the feature. Path-locator create is the most precise of the three create-by-locator branches; agents that know the exact target path use it directly. Without this story, the typed-tool track has no write primitive and every create goes through the freeform escape hatch.

**Independent Test**: With a stub `spawnFn` injected via `deps`, calling `write_note({ target_mode: "specific", vault: "MyVault", path: "Inbox/Idea.md", content: "# Idea\n\nBody\n" })` against a stub child that exits `0` and emits a CLI-shaped success response (canonical path on stdout, or substantively equivalent — exact wording captured during plan-stage CLI characterisation per FR-019) results in (a) the spawn being invoked with binary `"obsidian"` and argv `["create", "vault=MyVault", "path=Inbox/Idea.md", "content=# Idea\n\nBody\n"]` (vault hoisted first per the adapter's argv-assembly contract), and (b) the handler returning `{ created: true, path: "Inbox/Idea.md" }`.

**Acceptance Scenarios**:

1. **Given** the `write_note` tool is registered and a stub adapter that exits `0` reporting a successful create, **When** an MCP client calls `write_note({ target_mode: "specific", vault: "MyVault", path: "Inbox/Idea.md", content: "# Idea\n\nBody\n" })`, **Then** the tool returns a successful MCP response whose JSON-serialised payload is `{ created: true, path: "Inbox/Idea.md" }` AND the adapter was invoked with argv `["create", "vault=MyVault", "path=Inbox/Idea.md", "content=# Idea\n\nBody\n"]`.
2. **Given** the `write_note` tool, **When** an MCP client calls it with `content` containing CRLF line endings (e.g., `content: "# Title\r\nBody\r\n"`), **Then** the adapter is invoked with that exact `content=…` token unchanged AND the tool returns `created: true`. `write_note` does not normalise line endings — verbatim pass-through, parity with `read_note`.
3. **Given** the `write_note` tool, **When** an MCP client calls it with `content` containing a UTF-8 BOM (`"﻿hello"`) or multi-byte characters (`"# 笔记\nNiño\n"`), **Then** the adapter is invoked with that exact byte sequence AND the call succeeds. `write_note` does not transcode, normalise (NFC/NFD), or strip BOMs.

### User Story 2 — Specific-mode create via the wikilink locator (`file=`) (Priority: P1)

When the agent has a wikilink-form name (no extension, no folder) — typically because that is how Obsidian linked to it, or because the agent is creating a note the user referred to by name — it uses `file` instead of `path`. The agent calls `write_note({ target_mode: "specific", vault: "MyVault", file: "QuickNote", content: "draft\n" })` and the CLI resolves the wikilink to a concrete on-disk location (typically a configured "default note location" or the vault root). The handler reports `{ created: true, path: "<CLI-resolved canonical path>" }` — the canonical path is derived from the CLI's response, not from the input wikilink.

**Why this priority**: P1 alongside Story 1, not P2. The two locators target the same CLI subcommand but resolve different ambiguity classes — `file=` is wikilink-resolved by Obsidian, `path=` is exact. Both are part of the typed surface that retires `obsidian_exec` for writes; downgrading either would imply the surface is incomplete.

**Independent Test**: With a stub adapter that exits `0` and reports a CLI-shaped create response naming `Inbox/QuickNote.md` as the canonical path, calling `write_note({ target_mode: "specific", vault: "MyVault", file: "QuickNote", content: "draft\n" })` results in the spawn being invoked with argv `["create", "vault=MyVault", "file=QuickNote", "content=draft\n"]` and the handler returning `{ created: true, path: "Inbox/QuickNote.md" }`.

**Acceptance Scenarios**:

1. **Given** the `write_note` tool and a stub adapter reporting `Inbox/QuickNote.md` as the canonical resolved path, **When** an MCP client calls `write_note({ target_mode: "specific", vault: "MyVault", file: "QuickNote", content: "draft\n" })`, **Then** the tool returns `{ created: true, path: "Inbox/QuickNote.md" }` AND the adapter was invoked with argv `["create", "vault=MyVault", "file=QuickNote", "content=draft\n"]`.
2. **Given** the `write_note` tool, **When** an MCP client calls it with `file` containing UTF-8 multi-byte characters (e.g., `file: "笔记"`), **Then** the adapter is invoked with that exact `file=…` token AND the call succeeds. `write_note` does not transcode, normalise, or escape the file value.

### User Story 3 — Specific-mode overwrite-default-false rejects writes against existing files (Priority: P1)

The "fail loud rather than silently overwrite" safety posture is implemented by defaulting `overwrite: false`. An agent that calls `write_note({ target_mode: "specific", vault: "MyVault", path: "Existing.md", content: "new content\n" })` against a vault where `Existing.md` already exists MUST receive a structured error indicating the file-already-exists conflict; the on-disk file's bytes MUST be unchanged. The agent's recovery is to either pick a different path/file, or re-issue the call with `overwrite: true` (Story 4).

**Why this priority**: P1. The default-false behaviour is the entire reason `overwrite` is a typed boolean rather than implicit. A schema that defaulted true (or that omitted the field) would silently destroy user data on a name collision — a security-relevant defect for any caller that doesn't know to check first. The typed surface's value over `obsidian_exec` is partly that this safety default is built into the contract, not a discipline the caller must remember.

**Independent Test**: Two handler tests. (a) With a stub adapter that simulates the CLI's known overwrite=false-on-existing response (the exact wording captured during plan-stage characterisation per FR-019), calling `write_note({ target_mode: "specific", vault: "V", path: "Existing.md", content: "new\n" })` (overwrite omitted, defaulting to false) returns an MCP error response classified per FR-019 — typically `CLI_REPORTED_ERROR` if the CLI exits 0 with an `Error:` prefix, or `CLI_NON_ZERO_EXIT` if the CLI exits non-zero. (b) With a stub adapter that records its argv, the same call confirms `overwrite=true` is NOT included in the argv (only when explicitly set, per FR-007).

**Acceptance Scenarios**:

1. **Given** the `write_note` tool and a stub adapter that simulates the CLI's overwrite-refused response for an existing path, **When** an MCP client calls `write_note({ target_mode: "specific", vault: "V", path: "Existing.md", content: "x\n" })` (overwrite omitted), **Then** the tool returns an MCP error response (`isError: true`) whose `code` is the adapter-classified code per FR-019 (typically `CLI_REPORTED_ERROR` or `CLI_NON_ZERO_EXIT`) AND the error's `message` is derived from the CLI's verbatim wording (NOT synthesised at the tool layer).
2. **Given** the same setup, **When** an MCP client calls `write_note({ target_mode: "specific", vault: "V", path: "Existing.md", content: "x\n", overwrite: false })` (overwrite explicitly false), **Then** the result is identical to the omitted case — the schema's default and an explicit `false` are observationally indistinguishable.
3. **Given** the `write_note` tool and a stub adapter that records its argv, **When** an MCP client calls `write_note({ target_mode: "specific", vault: "V", path: "P.md", content: "x", overwrite: false })`, **Then** the argv passed to the adapter does NOT contain `overwrite=true` or `overwrite=false` — `overwrite=true` is only emitted when the caller opts in.

### User Story 4 — Specific-mode overwrite-true replaces existing files (Priority: P1)

The agent that explicitly knows it wants to replace an existing file passes `overwrite: true`. The handler emits `overwrite=true` in the argv (or the equivalent CLI flag form, captured during plan-stage characterisation per FR-019). The CLI rewrites the file's bytes to the new content; the handler returns `{ created: false, path: "<canonical path>" }` — `created: false` is the signal that this was an overwrite of an existing file, not a fresh creation.

**Why this priority**: P1. The overwrite-true branch is the second half of Story 3's safety contract — it is what makes the default-false posture liveable. Without an explicit overwrite path, agents would have no typed-surface way to issue full-rewrites (rebuilding generated notes, re-rendering templates, fixing structural defects in existing pages); they would fall back to `obsidian_exec`, which is exactly the regression this BI prevents.

**Independent Test**: With a stub adapter that exits `0` and reports a CLI-shaped overwrite-success response naming the existing path as canonical, calling `write_note({ target_mode: "specific", vault: "V", path: "Existing.md", content: "new\n", overwrite: true })` results in the spawn being invoked with argv that includes `overwrite=true` AND the handler returns `{ created: false, path: "Existing.md" }`.

**Acceptance Scenarios**:

1. **Given** the `write_note` tool and a stub adapter reporting a successful overwrite of `Existing.md`, **When** an MCP client calls `write_note({ target_mode: "specific", vault: "V", path: "Existing.md", content: "new\n", overwrite: true })`, **Then** the tool returns `{ created: false, path: "Existing.md" }` AND the adapter was invoked with argv that includes the `overwrite=true` token (exact placement per the adapter's argv-assembly contract).
2. **Given** the same setup but a NEW path (i.e., `overwrite: true` is harmless when the file does not exist), **When** an MCP client calls `write_note({ target_mode: "specific", vault: "V", path: "Brand/New.md", content: "x\n", overwrite: true })`, **Then** the tool returns `{ created: true, path: "Brand/New.md" }` — `overwrite: true` is permission, not a precondition; passing it against a non-existent file is not an error.

### User Story 5 — Active-mode rewrites the focused note (Priority: P1)

When the agent declares `target_mode: "active"`, the call's intent is "rewrite whatever note is currently open in Obsidian." The schema layer (per the target-mode primitive) forbids `vault`, `file`, and `path` in this mode; per Clarifications 2026-05-08, the schema ALSO requires `overwrite: true` in active mode (the focused note exists by definition, so the same explicit-opt-in safety rule binds) and forbids `template` and `open` in active mode (their semantics are nonsensical when the locator is "current focus"). The handler routes `{ target_mode: "active", content: "<bytes>", overwrite: true }` through the adapter, which emits argv beginning with `["create"]` and no locator key=value tokens (no `template=…` or `open=…` either). The CLI rewrites the focused note's bytes; the handler returns `{ created: false, path: "<focused note's vault-relative path>" }` — `created: false` because active mode is, by definition, an overwrite of the already-focused note.

**Why this priority**: P1. The active-mode contract is the entire reason `target_mode` is a discriminated union — collapsing it to "specific only" defeats ADR-003. The agent uses active mode when it operates on the user's current focus; making that path work end-to-end (schema → adapter → CLI) is part of this BI's value.

**Independent Test**: With a stub adapter that exits `0` and reports a CLI-shaped success response naming `Daily/2026-05-08.md` as the focused path, calling `write_note({ target_mode: "active", content: "rewritten\n", overwrite: true })` results in the adapter being invoked with argv `["create", "content=rewritten\n", "overwrite=true"]` (no `vault=…`, `file=…`, `path=…`, `template=…`, or `open=…` tokens) and the handler returning `{ created: false, path: "Daily/2026-05-08.md" }`. A second test asserts that calling `write_note({ target_mode: "active", content: "x\n", overwrite: true })` against a stub adapter that surfaces `ERR_NO_ACTIVE_FILE` flows that error through to the MCP error response unchanged.

**Acceptance Scenarios**:

1. **Given** the `write_note` tool and a stub adapter reporting a successful focused-note overwrite, **When** an MCP client calls `write_note({ target_mode: "active", content: "rewritten\n", overwrite: true })`, **Then** the tool returns `{ created: false, path: "<focused path>" }` AND the adapter was invoked with argv `["create", "content=rewritten\n", "overwrite=true"]` (no locator tokens, no template/open tokens).
2. **Given** the `write_note` tool, **When** an MCP client calls `write_note({ target_mode: "active", vault: "V", content: "x", overwrite: true })` (forbidden key in active mode), **Then** the call fails at the zod parse boundary as `VALIDATION_ERROR` whose `details.issues[].path` includes `"vault"` and whose message names `"vault"` AND `"active mode"` per the target-mode primitive's contract. The same applies for forbidden `file` and `path`.
3. **Given** the `write_note` tool and a stub adapter that surfaces `UpstreamError({ code: "ERR_NO_ACTIVE_FILE", … })`, **When** an MCP client calls `write_note({ target_mode: "active", content: "x", overwrite: true })`, **Then** the tool returns an MCP error response (`isError: true`) whose JSON-serialised payload carries `code: "ERR_NO_ACTIVE_FILE"` and a recovery-hint message naming "focus a note or switch to specific mode" — propagated via the adapter's verbatim `.message` field, not synthesised at the `write_note` layer.

### User Story 6 — Schema layer rejects malformed inputs before any CLI call (Priority: P1)

Per Constitution Principle III, the zod schema is the gatekeeper. `write_note`'s schema composes the post-010 flat target-mode primitive (which enforces `vault` required in specific mode, exactly-one-of `file`/`path` in specific mode, `vault`/`file`/`path` forbidden in active mode, `additionalProperties: false` strict-mode against unknown top-level keys) and adds `content` (required), `template` (optional), `overwrite` (optional, default false), and `open` (optional, default false) as the four `write_note`-specific fields. Per Clarifications 2026-05-08, the schema's `superRefine` ALSO enforces (in active mode): `overwrite: true` is required, `template` is forbidden, `open` is forbidden. **Ten** classes of malformed input MUST fail at the parse step: (a) `target_mode === "specific"` with neither `file` nor `path`; (b) `target_mode === "specific"` with both `file` AND `path`; (c) `target_mode === "specific"` without `vault`; (d) `target_mode === "active"` with any of `vault`/`file`/`path`; (e) any input with `content` missing; (f) any input with an unknown top-level key (e.g., `pancakes: "yes"`); (g) `target_mode` absent or with a value other than `"specific"` / `"active"`; (h) `target_mode === "active"` without `overwrite: true` (or with `overwrite: false`); (i) `target_mode === "active"` with `template`; (j) `target_mode === "active"` with `open`. None of these reach the adapter; the failure is a `VALIDATION_ERROR` with the offending field path. **Vault state on disk MUST be unchanged after every validation failure.**

**Why this priority**: P1. Boundary validation is the constitutional baseline — ADR-003's promise that the schema enforces the intent contract is empty if `write_note` doesn't actually parse against the primitive. Skipping this story would mean every malformed input becomes either an opaque CLI failure (uninformative to the agent) or undefined behaviour (the CLI accepts both `file` AND `path` and resolves one in implementation-defined precedence). The unknown-top-level-key carve-out (post-010 strict mode) is the additional safety net that prevents typos like `contennt: "..."` from silently producing empty files at the CLI layer. The active-mode constraints from Clarifications 2026-05-08 (overwrite-required, template/open-forbidden) extend the boundary contract uniformly across both modes.

**Independent Test**: Ten parse-only tests, each driving the handler with a stub adapter that MUST NOT be called. Each asserts the resulting error is `VALIDATION_ERROR` with the expected `details.issues[].path` and message keywords; the stub spawn function is asserted never to have been called.

**Acceptance Scenarios**:

### User Story 7 — CLI failures (binary missing, non-zero exit, in-band Error, no active file) flow through `UpstreamError` (Priority: P1)

When the underlying CLI fails — the binary is missing, the process exits non-zero, the CLI replies with `Error: …` on stdout, the focused note is missing in active mode — the failure surfaces to the MCP client as an `UpstreamError` carrying the adapter's classified `code`. Per the cli-adapter's four-priority classification, `write_note` does not classify; it propagates whatever code the adapter raised.

**Why this priority**: P1. Constitution Principle IV requires every boundary failure to surface as a structured `UpstreamError` with a stable code. The user input names overwrite-conflict and unknown-vault-display-name as the two adversarial cases that MUST be handled rather than silently succeed; both flow through this propagation path (per FR-019). The actual code on the wire depends on what the CLI does — the spec's job is to require propagation, not to claim a single code for a failure mode the CLI controls.

**Independent Test**: Four handler tests, each driving the adapter through an injected stub: (a) stub child fails to spawn with `ENOENT` → handler returns `code: "CLI_BINARY_NOT_FOUND"`; (b) stub child exits `1` with stderr `"permission denied"` → handler returns `code: "CLI_NON_ZERO_EXIT"`; (c) stub child exits `0` with stdout `"Error: file already exists\n"` → handler returns `code: "CLI_REPORTED_ERROR"` AND `details.message` carries the verbatim CLI line; (d) stub child surfaces `ERR_NO_ACTIVE_FILE` → handler returns `code: "ERR_NO_ACTIVE_FILE"`. In all cases, the handler does NOT swallow, mask, or rewrite the adapter's error; it routes through `asToolError()` per the existing precedent at [src/tools/_shared.ts](../../src/tools/_shared.ts).

**Acceptance Scenarios**:

1. **Given** the `write_note` tool and a stub adapter whose `spawnFn` raises `ENOENT`, **When** an MCP client calls `write_note({ target_mode: "specific", vault: "V", path: "P.md", content: "x" })`, **Then** the tool returns an MCP error response with `code: "CLI_BINARY_NOT_FOUND"` AND the error's `details` carries the binary-attempted name and `PATH` per the adapter's contract.
2. **Given** the `write_note` tool and a stub adapter that exits `1` with stderr `"permission denied"` (e.g., a read-only vault), **When** an MCP client calls `write_note({ target_mode: "specific", vault: "V", path: "P.md", content: "x" })`, **Then** the tool returns an MCP error response with `code: "CLI_NON_ZERO_EXIT"` AND `details.exitCode: 1` AND `details.stderr` carries the verbatim CLI stderr.
3. **Given** the `write_note` tool and a stub adapter that exits `0` with stdout `"Error: file already exists at Inbox/Existing.md\n"`, **When** an MCP client calls `write_note({ target_mode: "specific", vault: "V", path: "Inbox/Existing.md", content: "x" })` (overwrite omitted), **Then** the tool returns an MCP error response with `code: "CLI_REPORTED_ERROR"` AND `details.message` carries the verbatim wording from the CLI — the adapter's classification is propagated unchanged.
4. **Given** the `write_note` tool, **When** any non-`UpstreamError` exception escapes the adapter (an unexpected runtime error), **Then** the handler re-throws it (does NOT wrap it as a tool error), matching the obsidian_exec / read_note precedent.

**Acceptance Scenarios**:

1. **Given** the MCP server with `write_note` registered, **When** an MCP client calls `tools/list`, **Then** the returned descriptor for `write_note` has an `inputSchema` with `type: "object"`, `additionalProperties: false`, top-level `properties` listing all eight fields (`target_mode`, `vault`, `file`, `path`, `content`, `template`, `overwrite`, `open`), and `required` listing `["target_mode", "content"]` (or equivalent — the exact `required` set is determined by the flat-emit shape post-010; vault is conditional on target_mode and is enforced by the runtime `superRefine` rather than the JSON Schema's `required`).

### User Story 9 — Empty content, template + content precedence, and other boundary inputs (Priority: P3)

Two boundary conditions named in the user input's [P3] acceptance criteria: (i) `content: ""` (empty-string content) creates a 0-byte file successfully; (ii) the `template` + `content` combination resolves to a deterministic precedence, captured during plan-stage CLI characterisation. These are nice-to-have validators, not gating regressions; they sit at P3 because failing either one is a usability concern (and a documentation gap) but not a contract violation.

**Why this priority**: P3. Empty-content writes are unusual but legitimate (touch-style file creation, scaffolding); rejecting them at the schema layer would be paternalistic. Template + content precedence is a CLI behaviour the spec records but does not own — the CLI's actual precedence is captured in `docs/tools/write_note.md` per FR-014 and validated against the live CLI during plan-stage characterisation. Neither story gates ship.

**Independent Test**: (a) Calling `write_note({ target_mode: "specific", vault: "V", path: "Empty.md", content: "" })` against a stub adapter that exits `0` with a CLI-shaped success response results in argv `["create", "vault=V", "path=Empty.md", "content="]` (or the equivalent empty-string transmission per the adapter's argv-assembly contract) AND the handler returns `{ created: true, path: "Empty.md" }`. (b) Calling `write_note({ target_mode: "specific", vault: "V", path: "Foo.md", content: "body\n", template: "DailyTemplate" })` results in argv that includes both the `content=…` and `template=…` tokens; the test does NOT assert which one "wins" at the CLI — that contract is documented in `docs/tools/write_note.md` after plan-stage characterisation per FR-019.

**Acceptance Scenarios**:

1. **Given** the `write_note` tool and a stub adapter reporting a successful create at `Empty.md`, **When** an MCP client calls `write_note({ target_mode: "specific", vault: "V", path: "Empty.md", content: "" })`, **Then** the tool returns `{ created: true, path: "Empty.md" }` AND the adapter was invoked with `content=` (empty value) included in the argv.
2. **Given** the `write_note` tool and a stub adapter, **When** an MCP client calls `write_note({ target_mode: "specific", vault: "V", path: "Foo.md", content: "body\n", template: "Daily" })`, **Then** the adapter is invoked with argv that includes both `content=body\n` and `template=Daily` tokens AND the handler returns the adapter's reported result — `write_note` does not adjudicate template-vs-content precedence; the CLI does.

### Edge Cases

- **`write_note({ target_mode: "specific", vault: "", path: "P.md", content: "x" })`** (empty-string vault): MUST fail at the zod parse boundary as `VALIDATION_ERROR` whose `details.issues[].path` includes `"vault"`. The target-mode primitive enforces `vault: z.string().min(1)`. Reasonable default — error message mirrors zod's standard `min(1)` message.
- **`write_note({ target_mode: "specific", vault: "V", path: "../../../etc/passwd.md", content: "x" })`** (path-traversal-shaped value): MUST be forwarded to the CLI verbatim. `write_note` does NOT add path-traversal defense at the tool layer — the CLI is the trust boundary for vault-relative path validation, and the underlying Obsidian binary either rejects out-of-vault paths or restricts I/O to its own scope. This matches the project's existing posture and avoids redundant policy layers. **However**, the spec REQUIRES plan-stage characterisation (FR-019) to verify that the CLI does in fact reject such inputs; if the CLI silently writes outside the vault, this BI cannot ship as written and a tool-layer reject MUST be added. Silent vault-escape is a security defect, not a usability question. (Per the user input's FILESYSTEM adversarial bullet.)
- **Reserved characters in `path` / `file`** (Windows: `:?*<>|"\`): platform-dependent. The CLI is the trust boundary; `write_note` forwards verbatim. Plan-stage CLI characterisation per FR-019 records the observed behaviour (likely platform-specific); the documented behaviour is captured in `docs/tools/write_note.md` per FR-014.
- **Unicode normalisation (NFC vs NFD)** in `path` / `file` / `content`: `write_note` does not normalise. Input bytes flow verbatim to the CLI. Round-trip equality through the on-disk path is the CLI's domain; if a future BI surfaces NFC/NFD round-trip issues, the resolution lands at the CLI layer or in a primitive amendment, not in `write_note`.
- **UTF-8 BOM at start of `content`**: forwarded verbatim. Whether Obsidian strips it on next read is the CLI's / Obsidian's behaviour, not `write_note`'s. Documented in `docs/tools/write_note.md` after plan-stage observation.
- **CRLF vs LF in `content`**: forwarded verbatim. `write_note` does NOT normalise. Same invariant `read_note` honours.
- **Very large `content`** (≳32 KiB on Windows, ≳2 MiB on Linux): the OS argv-length limit applies. The CLI invocation fails before the binary starts; the adapter classifies the failure as `CLI_BINARY_NOT_FOUND` (or a platform-specific code from the spawn layer). `write_note` does NOT impose its own size cap; the OS ceiling is documented in `docs/tools/write_note.md` per FR-014 so callers know what to expect. A future BI may add stdin-fed content to bypass argv limits; out of scope here.
- **`write_note({ target_mode: "specific", vault: "V", path: "P.md", content: "x", template: "NonExistent" })`** (non-existent template name): MUST surface as a structured error to the caller. Whichever code the CLI's response classifies to (likely `CLI_REPORTED_ERROR` if the CLI emits `Error: template not found …` on stdout, or `CLI_NON_ZERO_EXIT` if it exits non-zero) is the propagated code. Plan-stage characterisation per FR-019 captures the actual wording.
- **Unknown vault display name** (`vault: "DoesNotExist"`): the user input flags this as a CLI failure mode that may be structurally indistinguishable from success (no error prefix, exit code 0). The implementation MUST handle this case explicitly — either by pre-validating the vault display name (which requires either a `list_vaults` tool that does not yet exist, or an environment-derived registry that the CLI exposes) OR by inspecting the response for the known signature and re-classifying as `CLI_REPORTED_ERROR`. **The selection between these two options is a plan-stage research decision per FR-019.** Reasonable default for the spec stage: the response-inspection path is preferred (no new tool dependency, no environment-introspection requirement); the actual CLI response signature is captured during plan-stage research, and the handler's response-inspection logic is added to the adapter (NOT to `write_note` directly) so all typed tools benefit. If the response-inspection approach proves brittle, a `list_vaults` primitive ships in a follow-up BI and pre-validation moves there.
- **External editor has the file open** (Obsidian, VS Code, anything else) when `write_note` is invoked: observable behaviour is captured during plan-stage characterisation per FR-019 and documented in `docs/tools/write_note.md`. `write_note` does NOT lock the file or coordinate with external editors. If the OS rejects the write with EBUSY (Windows), the CLI surfaces non-zero exit; the handler propagates as `CLI_NON_ZERO_EXIT`. If the OS accepts the write and the editor reloads (or doesn't), that is the editor's behaviour, not `write_note`'s.
- **Active-mode TOCTOU**: the focused note may change between the schema's parse step and the CLI's actual invocation. The operation runs against whichever note is focused at execution time. Documented as a known limitation; `write_note` does NOT attempt to lock or pin the focused note. (Per the user input's CONCURRENCY adversarial bullet.)
- **Two concurrent `write_note` calls to the same path** (e.g., from two simultaneous MCP requests): MUST NOT race in a way that produces inconsistent vault state. The shared in-flight CLI queue (per FR-008, inherited from [006-read-note](../006-read-note/spec.md) FR-016) serializes both calls behind one channel; the second call sees the on-disk state after the first call completed. This is the existing safety posture.
- **`write_note({ target_mode: "active", file: "Foo", content: "x" })`** (forbidden key in active mode): MUST fail at the zod parse boundary, mirroring `read_note`'s behaviour. The same applies for `vault` and `path`. The error message MUST NOT include recovery instructions per the target-mode primitive's clarification.
- **`write_note({ target_mode: "specific", vault: "V", path: "P.md", content: "x", overwrite: undefined })`** (explicit-undefined in specific mode): zod's `.optional()` accepts `undefined` as the omitted form; this resolves to the schema's default of `false` (per FR-005). Behaviour is identical to omitting the key entirely. NOTE: in active mode, `overwrite: undefined` resolves to the default `false` and then fails the active-mode `overwrite: true` requirement per Clarifications 2026-05-08 Q1 — Story 6 AC#8 covers this.
- **`write_note({ target_mode: "active", content: "x", overwrite: true, template: "T" })`** (template + active mode): per Clarifications 2026-05-08 Q3, the schema rejects `template` in active mode as `VALIDATION_ERROR`. The CLI is never invoked. Story 6 AC#9 covers this.
- **`write_note({ target_mode: "active", content: "x", overwrite: true, open: true })`** (open + active mode): per Clarifications 2026-05-08 Q3, the schema rejects `open` in active mode as `VALIDATION_ERROR` regardless of the boolean value (`true` or `false`). Story 6 AC#10 covers this.
- **Caller-supplied `content` containing argv-shell metacharacters** (e.g., `content: "; rm -rf /; #"`): SAFE. The cli-adapter assembles argv as a child_process.spawn array (no shell), so no character in `content`/`template`/`file`/`path` can break out into a shell. This is a structural property of the adapter, not a runtime check. (Per the user input's SECURITY bullet.)
- **Strict-rich vs strict-naive client-class observability of unknown-key rejection** (Story 6 AC #6): strict-rich clients (Claude Desktop, MCP Inspector) read the `additionalProperties: false` from the published `inputSchema` and either forward unknown keys (in which case the bridge-side `unrecognized_keys` rejection is observable to the test) or strip them client-side (in which case the bridge-side rejection is non-observable but the schema-side invariant — no untyped passthrough at runtime — still holds). Strict-naive clients (Cowork) do not consult `additionalProperties`; they forward unknown keys to the bridge, which always observes the rejection. Tests assume the strict-rich-forwarding path (the worst case) and assert the bridge-side rejection occurs. Documented per the user input's CLIENT-CLASS adversarial bullet.

## Requirements *(mandatory)*

### Functional Requirements

#### Module layout

- **FR-001**: A new typed tool MUST be created at `src/tools/write_note/` following the existing per-surface module convention (per Constitution Principle I and the `src/tools/help/`, `src/tools/obsidian_exec/`, `src/tools/read_note/` precedents). The module MUST contain six files at minimum (three source + three co-located tests): `schema.ts` (the zod input schema, the inferred TypeScript types, and the JSON Schema export via `zod-to-json-schema`), `handler.ts` (the function that calls the adapter and maps the result to the output shape), `tool.ts` (the `registerWriteNoteTool` factory returning a `RegisteredTool` per the aggregator pattern), and one co-located test file per source file (`schema.test.ts`, `handler.test.ts`, `tool.test.ts`). Per Constitution Principle V, every new source file MUST carry an `// Original — no upstream. <one-line description>.` header.

#### Schema

- **FR-002**: The input schema MUST compose the post-010 flat `target_mode` primitive (per [010-flatten-target-mode](../010-flatten-target-mode/spec.md)) using the canonical Pattern (a) flat-extension idiom: `applyTargetModeRefinement(targetModeBaseSchema.extend({ content, template, overwrite, open })).superRefine(<write_note-specific active-mode rules>)`. Per the contract at [specs/010-flatten-target-mode/contracts/flat-target-mode.contract.md](../010-flatten-target-mode/contracts/flat-target-mode.contract.md), `.extend()` (NOT `.merge()`) is the binding API call — `.merge()` resets `unknownKeys` to `"strip"` and silently drops the strict-mode safety net for unknown top-level keys. The composed schema inherits the primitive's full per-mode contract end-to-end: `vault` required in specific mode, exactly-one-of `file`/`path` in specific mode, `vault`/`file`/`path` forbidden in active mode, top-level `additionalProperties: false` against unknown keys.

  **`write_note`-specific active-mode `superRefine` clauses** (per Clarifications 2026-05-08 Q1, Q3): when `target_mode === "active"`, the schema MUST additionally enforce: (i) `overwrite === true` (calls without explicit `overwrite: true` — including the default-false case — fail with `VALIDATION_ERROR` whose `details.issues[].path` includes `"overwrite"`); (ii) `template` MUST NOT be present (regardless of value); (iii) `open` MUST NOT be present (regardless of boolean value). These three clauses are layered as additional `superRefine` issues on top of the target-mode primitive's `applyTargetModeRefinement` output. Plan-stage decision deferred: whether to bundle these as a single `superRefine` callback at the `write_note` schema level or as three independent issues; the runtime contract (each violation surfaces as its own `details.issues[]` entry with the offending field path) binds.

- **FR-003**: The schema MUST add four `write_note`-specific fields:
    - `content: z.string()` — required, no `.min(1)` floor (empty content is valid per Story 9 AC#1). No `.max(N)` ceiling per Clarifications 2026-05-08 Q2.
    - `template: z.string().optional()` — optional in specific mode; **forbidden in active mode** per Clarifications 2026-05-08 Q3 (enforced by the active-mode `superRefine` per FR-002, not a field-level constraint). No `.min(1)` floor; an empty-string template in specific mode is forwarded to the CLI.
    - `overwrite: z.boolean().optional().default(false)` — optional with default `false` in specific mode; **MUST be explicitly `true` in active mode** per Clarifications 2026-05-08 Q1 (enforced by the active-mode `superRefine` per FR-002 — the field's default-false coercion still applies, but the active-mode refinement then rejects the resulting `false`). The default is part of the schema, not a handler-layer fallback.
    - `open: z.boolean().optional().default(false)` — optional with default `false` in specific mode; **forbidden in active mode** per Clarifications 2026-05-08 Q3 (enforced by the active-mode `superRefine` per FR-002 — neither `open: true` nor `open: false` is permitted; the key MUST NOT be present at all).

  **Field-level vs `superRefine` enforcement**: per zod's idioms, field-level constraints (`.optional()`, `.default(false)`, type) apply uniformly across both modes; mode-specific constraints (forbidden-in-active, required-true-in-active) live in the `superRefine` per FR-002. This split keeps the field declarations terse and the mode-specific rules co-located with the other target-mode refinements.

- **FR-004**: The schema MUST NOT add `.describe()` annotations to any field. Parameter documentation lives in `docs/tools/write_note.md` (per ADR-005's progressive-disclosure design).

- **FR-005**: The schema MUST export the inferred TypeScript type via `z.infer<typeof writeNoteInputSchema>` (or equivalent name). No hand-written `interface WriteNoteInput` or `type WriteNoteInput = { … }` is permitted (Constitution Principle III). The output type MUST also be a zod schema (or its `z.infer<…>`): `writeNoteOutputSchema = z.object({ created: z.boolean(), path: z.string() }).strict()`. Hand-rolled output types are forbidden by the same principle.

- **FR-006**: The schema MUST export a JSON Schema document via `zod-to-json-schema(...)` (mirroring `obsidian_exec` and `read_note`). The exported JSON Schema MUST round-trip through the schema-stripping utility at registration time per FR-011. The emitted JSON Schema MUST be the post-010 flat shape (one top-level `type: "object"`, `properties` with all eight fields typed inline, `additionalProperties: false`, no `oneOf` envelope) — verifiable via the consolidated drift detector at [src/tools/_register.test.ts](../../src/tools/_register.test.ts) which automatically covers every registered tool.

#### Handler

- **FR-007**: The handler MUST route the validated input through the cli-adapter's `invokeCli` entry point (or its bounded counterpart through the shared queue per FR-008). The adapter call shape is `invokeCli({ command: "create", parameters: <derived>, flags: [], target_mode: <validated> }, deps?)`. The `parameters` record is derived from the validated input as follows:
    - **Specific mode**: `parameters` includes `vault: parsed.vault`, exactly one of `file: parsed.file` or `path: parsed.path`, `content: parsed.content`, optionally `template: parsed.template` (only if defined), conditionally `overwrite: "true"` (only if `parsed.overwrite === true` — the schema's default false MUST NOT emit `overwrite=true` in argv per Story 3 AC#3), conditionally `open: "true"` (only if `parsed.open === true`, same default-false-omit rule). The adapter's argv assembler hoists `vault=` to first position automatically.
    - **Active mode**: `parameters` is `{ content: parsed.content, overwrite: "true" }`. Per Clarifications 2026-05-08 Q1 the schema guarantees `parsed.overwrite === true` after parse in active mode (otherwise the parse would have failed); the handler therefore unconditionally emits `overwrite=true` in active-mode argv. Per Clarifications 2026-05-08 Q3 the schema guarantees `parsed.template === undefined` and `parsed.open === undefined` after parse in active mode, so the handler never emits `template=…` or `open=…` tokens in active-mode argv. The schema layer has already rejected `vault`/`file`/`path` so the handler does not include them; the adapter's defensive locator-stripping is a redundant safety net.
    - In both modes, `flags` is `[]` — `write_note`'s typed surface is the structured-key form. (The CLI's `newtab` flag is excluded from this BI per the user input's Out of Scope; agents that need it use `obsidian_exec`.)

  Plan-stage research per FR-019 confirms the CLI's actual flag shape (whether `overwrite` is `overwrite=true` as a key=value or a `--overwrite` flag in `flags: []`). If the CLI uses flag form, the handler's argv-assembly shifts to populate `flags` accordingly. The structural contract (no argv emission for default-false booleans, structured value emission for explicit-true booleans) binds; the placement (parameters vs flags) is a plan-stage decision.

- **FR-008**: The handler MUST execute through the same single-in-flight CLI queue that gates `obsidian_exec` and `read_note` (per [006-read-note](../006-read-note/spec.md) FR-016). `registerWriteNoteTool`'s `RegisterDeps` MUST accept `queue: Queue`, and `src/server.ts` MUST pass the *same* `queue` instance to `registerWriteNoteTool`, `registerReadNoteTool`, and `registerObsidianExecTool` so concurrent CLI invocations across all three tools serialize through one channel. This forecloses the "two concurrent writes to the same path" race in Edge Cases.

- **FR-009**: The handler MUST emit structured log events around the adapter call, parity with `read_note` (per [006-read-note](../006-read-note/spec.md) FR-017). At minimum: `logger.callStart(...)` BEFORE the adapter invocation (with a fresh `callId`, `command: "create"`, the resolved `vault` if present else `null`, `queueDepth`); `logger.callEndSuccess(...)` on the success path (with `callId`, `durationMs`, the output's `created` boolean, and `pathBytes` = `Buffer.byteLength(path, "utf8")`); `logger.callEndFailure(...)` on the failure path (with `callId`, `errorCode` from the propagated `UpstreamError.code`, `durationMs`). The `Logger` type is the existing one at [src/logger.ts](../../src/logger.ts); `RegisterDeps` MUST accept `logger: Logger` and `src/server.ts` MUST pass the same logger instance to all tool registrations.

- **FR-010**: The handler MUST return a successful tool-call result whose JSON-serialised payload conforms to the output schema: `{ created: <bool>, path: "<canonical vault-relative path>" }`. The `created` boolean is derived from the CLI's response — true when the CLI reports a fresh creation, false when the CLI reports an overwrite. Plan-stage research per FR-019 captures the CLI's exact response wording for both cases; the handler's parsing logic is locked against that wording. The `path` field is the canonical vault-relative path the CLI reports (which may differ from the input `file`-form locator's wikilink). The handler MUST NOT trim, normalise, or post-process the path; it propagates the CLI's reported value verbatim.

### Out of Scope

The following are explicitly excluded from this feature and remain the responsibility of separate BIs or future work (mirroring the user input's "Out of scope" enumeration plus consequential exclusions surfaced during spec drafting):

- **Streaming or chunked writes**: the entire `content` is in-memory per call. A future BI may add stdin-fed content for argv-limit relief; out of scope here.
- **Frontmatter-only mutations**: a separate future BI (`write_property`) ships frontmatter surgery without rewriting the body.
- **Append operations**: a separate future BI (`append_note`) ships text-append against an existing file without overwriting.
- **Surgical sub-content writes**: a separate future BI (`patch_heading`) ships heading-anchored updates.
- **The CLI's `newtab` flag**: open-in-current-tab is the safer default. Agents that genuinely want new-tab behaviour fall back to `obsidian_exec`. If empirical use surfaces a need, a future BI ships a typed `open_in_new_tab` boolean (or extends `open` to an enum).
- **Backups before overwrite**: overwrite is destructive by design (caller opts in explicitly). If a caller wants a backup-then-write workflow, they compose `read_note` + `write_note(<new path>)` themselves; the typed surface does not orchestrate.
- **Atomicity guarantees beyond what the CLI provides**: interrupted writes (power loss, kill -9) leave the on-disk state to the OS, not the wrapper. If empirical use surfaces a real-world atomicity defect, the resolution lands at the CLI layer or in a primitive amendment.
- **Path-traversal validation at the tool layer**: out of scope per Edge Cases — UNLESS plan-stage characterisation per FR-019 surfaces that the CLI does not reject `../`-shaped paths, in which case the spec is amended pre-ship to add a tool-layer reject. This is a precondition, not an open question.
- **Unicode normalisation (NFC/NFD) at the tool layer**: forwarded verbatim. If round-trip equality through the on-disk path becomes a problem, the resolution is at the primitive layer (a normalisation policy that all typed tools inherit) or at the CLI layer.
- **A `list_vaults` primitive for vault-display-name pre-validation**: out of scope. The unknown-vault edge case is handled by response-inspection in the cli-adapter (per Edge Cases). If response-inspection proves brittle, `list_vaults` ships as a follow-up BI and pre-validation moves there.
- **Output capping** (rejecting writes that exceed N MB): out of scope. The OS argv-length ceiling is the de-facto cap; documented in `docs/tools/write_note.md` per FR-014.
- **New error codes**: out of scope per the user input. `write_note`'s entire failure surface is covered by `VALIDATION_ERROR` + the adapter's four codes. If a `write_note`-specific failure mode emerges (e.g., "the named template resolves to a file that no longer exists at write time"), a future amendment introduces a code with the same rigour as [005-help-tool](../005-help-tool/spec.md) introduced `HELP_TOOL_NOT_FOUND` / `HELP_DOCS_MISSING`.
- **Refactoring the existing `obsidian_exec` tool**: out of scope. `obsidian_exec` and the typed tools co-exist; `obsidian_exec`'s schema, descriptions, error codes, and tests are unchanged by this BI (per SC-009).
- **Modifying ADR-003, ADR-005, or ADR-006**: this BI implements all three; it does not change them. Conflicts surface as new ADRs per the project's amendment procedure.

## Assumptions

- **The user input's `(required)` annotation on `target_mode` and `content`** is binding. The schema MUST NOT make either field optional. The user input's `(optional)` annotation on `template`, `overwrite`, and `open` is also binding; `vault` is "required in specific mode, forbidden in active mode" by reference to the target-mode primitive's contract.
- **The CLI's `create` subcommand** accepts the argv shape `["create", "vault=…", "file=… | path=…", "content=…"]` plus optional `template=…` / `overwrite=true` / `open=true` tokens, and emits a structured success or failure response on stdout/stderr that the adapter can classify. This is the precondition the tool is built on; if the CLI uses a different shape (e.g., `--overwrite` flag form instead of `overwrite=true` key=value), plan-stage characterisation per FR-019 captures the actual shape and the handler's argv-assembly retargets accordingly. Reasonable default: argv assembly is `[command, vault=…, locator=…, content=…, …extras]`; the structural contract (no argv emission for default-false booleans, structured emission for explicit-true booleans, verbatim content forwarding) binds, the placement does not.
- **The CLI's response signal for `created: true` vs `created: false`** is captured during plan-stage research per FR-019 (cases (i)–(iii)). Plausible signals include: a stdout marker (`"Created Inbox/Idea.md"` vs `"Updated Inbox/Idea.md"`); an exit code distinction; or a JSON-shaped response. The handler's response-parsing logic locks against whichever signal the live CLI uses. If the CLI does not distinguish (e.g., always emits `"OK"` regardless), this BI is amended pre-ship to either (a) accept the indistinguishability and infer `created` from a pre-call existence check (a second CLI call, increasing latency and TOCTOU surface), or (b) request a CLI-layer enhancement to disambiguate. The amendment lands before merge per FR-019.
- **The CLI's `read` subcommand convention applies symmetrically to `create`**: argv-array passing, no shell, verbatim UTF-8 forwarding, structured error responses. If `create` deviates (e.g., expects content via stdin rather than argv), the handler's invocation shape adjusts; the structural contracts in FR-001..FR-018 bind regardless.
- **The active-mode CLI invocation** for `create` (`obsidian create` with no vault/file/path) overwrites the focused note in the focused vault with the supplied `content`. If no vault is focused, the CLI surfaces `Error: no active file` (or equivalent), classified by the adapter as `ERR_NO_ACTIVE_FILE`. This is consistent with the `read` subcommand's active-mode behaviour observed during BI-026 (002-detect-cli-errors) reproductions; if `create`'s active-mode behaviour differs, plan-stage characterisation per FR-019 case (viii) captures the difference and the spec is amended.
- **The `template` parameter** maps directly to a CLI flag of the same name. Whether template + content interaction is "template wins, content is appended", "content wins, template is the bootstrap shape", or "template alone (content ignored)" is captured in research.md per FR-019 case (vi). The spec does NOT pre-commit to a precedence; the handler forwards both verbatim and the documentation reflects the observed behaviour.
- **The `overwrite` and `open` boolean schema defaults** (`false`) are part of the schema's parsed output, NOT of the handler's logic. After `schema.parse(args)` returns in **specific mode**, `parsed.overwrite` and `parsed.open` are guaranteed booleans (no `undefined`); the handler's "emit only when true" rule (FR-007) is a serialisation choice on top of guaranteed booleans, not a fallback chain. After `schema.parse(args)` returns in **active mode** (per Clarifications 2026-05-08 Q1, Q3), `parsed.overwrite` is guaranteed to be exactly `true` (otherwise the parse would have failed) and `parsed.template` and `parsed.open` are guaranteed to be `undefined` (any presence in input fails the parse). The handler relies on these post-parse guarantees in its argv assembly per FR-007. This avoids the "two-place default" and "two-place validation" anti-patterns.
- **The post-010 strict-mode `additionalProperties: false`** is the source-of-truth for the unknown-top-level-key rejection (Story 6 AC#6). `write_note` does NOT add `.strict()` redundantly at the tool layer; it inherits strict mode from `targetModeBaseSchema.extend(...)` per [010-flatten-target-mode](../010-flatten-target-mode/spec.md) FR-002 (which verified that `.extend()` preserves strict mode while `.merge()` does not).
- **The cli-adapter's `deps.spawnFn` test seam** is the canonical test-injection mechanism. `write_note`'s handler tests inject `deps.spawnFn` via the adapter call, NOT by mocking the adapter module itself. This matches `read_note`'s and `obsidian_exec`'s test patterns.
- **No new error codes** are introduced. Per Constitution Principle IV, error codes are the project's stable API. `write_note`'s entire failure surface is covered by `VALIDATION_ERROR` (zod) and the adapter's four codes. If empirical use surfaces a `write_note`-specific failure mode, a future amendment introduces the code with the same rigour as ADR-006 / [005-help-tool](../005-help-tool/spec.md).
- **Constitution Principle II binds the test-set co-location** at `src/tools/write_note/{schema,handler,tool}.test.ts`. The cases enumerated in FR-016 are the minimum set; reviewers may add more.
- **Constitution Principle V binds the module-header convention**: `// Original — no upstream. <one-line description>.` is required on every new source file in `src/tools/write_note/`. The Markdown documentation file at `docs/tools/write_note.md` is exempt per [005-help-tool](../005-help-tool/spec.md) FR-019.
- **Plan-stage decisions to defer** to `/speckit-plan`: the exact module file layout (e.g., whether to add a sub-helper for argv assembly), the exact wording of the top-level `description` field, the exact wording of `docs/tools/write_note.md`, the registration order in `src/server.ts`, the choice between key=value form (`overwrite=true`) and flag form (`--overwrite`) in argv assembly (depends on the CLI's actual contract), the response-parsing shape for `created` true/false, and the response-inspection logic for unknown-vault re-classification (whether it lives in the cli-adapter or in `write_note`). The structural contracts in FR-001 through FR-019 bind; the surface-level wording does not.
- **The user input's "BI" framing** does not bind a specific BI number. This feature is the next typed-tool wrap after [006-read-note](../006-read-note/spec.md) (BI-003); the architecture document MAY pre-assign a BI number (e.g., BI-005), and if so, the plan stage references that number. The spec does not invent one.
