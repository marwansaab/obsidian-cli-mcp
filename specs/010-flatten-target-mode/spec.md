# Feature Specification: Flatten `targetModeSchema` and Retire the Wrap-Branch Envelope Helper

**Feature Branch**: `010-flatten-target-mode`
**Created**: 2026-05-07
**Status**: Draft
**Input**: User description: features 007 and 009 ship a working but accumulating fix for the bug introduced in 006-read-note. The `read_note` tool's published `inputSchema` derives from `targetModeSchema` — a `ZodEffects<ZodDiscriminatedUnion>` — which `zod-to-json-schema` emits as a top-level `anyOf`. To bridge that to the MCP `Tool.inputSchema` contract, [src/tools/_shared.ts](../../src/tools/_shared.ts) carries ~140 lines of envelope synthesis: wrap branch, `oneOf` rewrite, top-level `properties` union, top-level `required` intersection, Pattern (a) `allOf` walking, leaf widening with cross-branch string-discriminator surfacing. A three-group drift detector at [src/tools/_register.test.ts](../../src/tools/_register.test.ts) exists to police this pipeline. None of this complexity would be necessary if the zod schema feeding `zodToJsonSchema` were a flat `z.object({...})`. This feature replaces the discriminated-union encoding of the target-mode rules with a flat `z.object` + `superRefine`, deleting the envelope helper's wrap branch, the synthesis logic, the Pattern (a)/(b) handling, and the drift detector groups that exist solely to verify them. The runtime semantics — what inputs zod accepts, what inputs it rejects, the `VALIDATION_ERROR` payload shape — are preserved exactly. The TypeScript type of `TargetMode` flattens, and ADR-003's "discriminated union" wording is amended to "flat object with `superRefine`" to reflect the new encoding.

## Background *(non-normative — context only)*

`0.2.1` (feature 009) ships a working fix: the published `inputSchema` for `read_note` is well-formed for both strict-rich (Claude Desktop) and strict-naive (Cowork) MCP clients, the drift detector observes the wire shape, and the typed-tool roadmap is unblocked. The fix is correct and merged. This feature is not a bug fix.

It is a structural simplification. Three branches (007, 008, 009) accumulated against a single root cause: the published JSON Schema is mechanically derived from a zod schema (`targetModeSchema`) whose top-level construct (`ZodDiscriminatedUnion`) does not survive the zod → JSON Schema → MCP `inputSchema` pipeline cleanly. Each fix bridged the gap downstream — by widening the envelope helper, by adding companion exports (007), by adding the registry pipeline (008), by synthesising top-level `properties` (009). None of them changed the input.

The cleaner alternative — encode the same target-mode rules with a flat `z.object` + `superRefine` rather than a discriminated union — produces a JSON Schema that satisfies the MCP `Tool.inputSchema` contract directly, with no envelope synthesis. The discriminated union exists today because zod 3.x's `z.discriminatedUnion` provides type narrowing at the consumer site (`if (input.target_mode === "specific") { input.vault /* string */ }`); the flat encoding loses that narrowing (`input.vault` becomes `string | undefined`). The trade is: simpler publication pipeline + smaller test surface vs. weaker type narrowing in handler code.

The control case is informative. [src/tools/obsidian_exec/schema.ts](../../src/tools/obsidian_exec/schema.ts) is a flat `z.object({...}).strict()`. Its published `inputSchema` is the natural `zodToJsonSchema` output. It works through every MCP client class observed under 007/008/009 without any envelope handling. `read_note`'s post-009 published shape will, after this feature, look structurally identical (one less property, different property roster, otherwise the same kind of object).

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Future typed tools follow the same flat-`z.object` pattern as `obsidian_exec` (Priority: P1)

A developer (or AI coding assistant) is asked to add a new typed tool that needs target-mode behaviour — e.g. the planned `write_note` or `append_note`. Today, the canonical reuse pattern is "extend `targetModeSchema` via `targetModeSchema.and(z.object({ <fields> }))`" (Pattern (a)) or "build a fresh discriminated union over the per-mode base schemas" (Pattern (b)). After this feature, the canonical pattern is "extend the flat `targetModeSchema` via `targetModeSchema.merge(z.object({ <fields> }))` (or equivalent flat composition)". The author writes only zod; the published `inputSchema` is the natural `zodToJsonSchema` output; no envelope helper sits between the schema and the wire.

**Why this priority**: The publication pipeline is the source of complexity. Replacing the input shape eliminates the helper's wrap branch entirely, which is the largest standing surface in [_shared.ts](../../src/tools/_shared.ts). Pattern (a)/(b) re-use guidance simplifies in lock-step: there is one pattern, it matches `obsidian_exec`'s shape, and every future tool inherits a clean published descriptor for free.

**Independent Test**: Add a synthetic typed tool whose registered schema is `targetModeSchema.merge(z.object({ note_text: z.string() }))` (the flat-extension equivalent of feature 009's Pattern (a)). Assert via the drift detector that its published `inputSchema` has `type: "object"`, contains `target_mode`, `vault`, `file`, `path`, and `note_text` in `properties`, has `required: ["target_mode", "note_text"]`, and `additionalProperties: false`. Assert via runtime parse that `vault` is required when `target_mode === "specific"` (enforced by `superRefine`, not by the type-level `required` array), that `vault`/`file`/`path` are forbidden when `target_mode === "active"`, that exactly one of `file` / `path` is required when `target_mode === "specific"`. Then remove the synthetic tool — it exists only as a fixture.

**Acceptance Scenarios**:

1. **Given** the flat `targetModeSchema` is the only target-mode schema export, **When** a developer registers a tool whose schema is `targetModeSchema.merge(z.object({ note_text: z.string() }))`, **Then** the published `inputSchema` is a single flat object with all five property names in `properties`, no `oneOf`, no `allOf`, and `type: "object"` at the root.
2. **Given** the same registered tool, **When** the runtime receives `{ target_mode: "specific", vault: "v", path: "p", note_text: "x" }`, **Then** zod accepts it.
3. **Given** the same registered tool, **When** the runtime receives `{ target_mode: "specific", note_text: "x" }` (missing vault), **Then** zod returns `VALIDATION_ERROR` with an issue at `path: ["vault"]` whose message names "vault is required in specific mode" or equivalent.
4. **Given** the same registered tool, **When** the runtime receives `{ target_mode: "active", vault: "v", note_text: "x" }`, **Then** zod returns `VALIDATION_ERROR` with an issue at `path: ["vault"]` whose message names "vault is not allowed in active mode" — identical to today's behaviour.

4. **Given** the post-010 package is installed, **When** any client invokes `read_note` with a malformed input (both `file` and `path` in specific, vault in active, no locator in specific), **Then** the runtime returns `VALIDATION_ERROR` with the same per-issue path and message text feature 009 already produces — verified by reusing 009's existing target-mode test fixtures.

### Edge Cases

- **Pattern (a) consumers from feature 009**: today the synthetic Pattern (a) fixture in [_register.test.ts:415-434](../../src/tools/_register.test.ts#L415-L434) constructs `targetModeSchema.and(z.object({ note_text: z.string() }))`, producing a `ZodIntersection` whose JSON Schema has top-level `allOf`. After flattening, the canonical extension is `targetModeSchema.merge(...)` (a single `z.object`), and the `allOf` shape vanishes. The fixture deletes; the assertion that "future typed tools inherit the fix" moves to a flat-extension fixture (User Story 1 above).
- **Pattern (b) consumers**: today's Pattern (b) fixture constructs a fresh discriminated union with union-level `superRefine`. After this feature, that pattern is no longer canonical (it would still work, but the publication pipeline can no longer guarantee a single `properties` map for it; it would re-trigger the wrap-branch problem we are deleting). Pattern (b) is removed from the documented reuse framework as a follow-up to feature 004's spec; the cross-feature impact is recorded in the documentation update (FR-013).
- **Type narrowing in `read_note`'s handler**: today [handler.ts:21-27](../../src/tools/read_note/handler.ts#L21-L27) reads `input.vault` (typed as `string`) inside the `target_mode === "specific"` branch. After flattening, `input.vault` is `string | undefined`. The handler must either assert non-null (`input.vault!`), check explicitly (`if (input.vault !== undefined) ...`), or read through a runtime invariant comment naming the `superRefine` guarantee. Choice deferred to `/speckit-clarify` (see Open Questions C1).
- **Existing `targetModeSpecificSchema` / `targetModeActiveSchema` exports**: defined at [target-mode.ts:58-74](../../src/target-mode/target-mode.ts#L58-L74). Six cases in [target-mode.test.ts](../../src/target-mode/target-mode.test.ts) call them directly. The exports either delete (cases migrate to call `targetModeSchema` with the matching `target_mode` literal) or persist as named refinements over the flat schema. Choice deferred to `/speckit-clarify` (C2).
- **`targetModeSpecificBaseSchema` / `targetModeActiveBaseSchema` exports**: defined at [target-mode.ts:43-66](../../src/target-mode/target-mode.ts#L43-L66). After flattening, these per-branch base schemas have no consumer (the flat schema does not compose from them). They delete, and feature 004's spec — which documented Pattern (b) reuse atop these bases — is amended in lock-step (FR-013).
- **`applyTargetModeSpecificRefinement` / `applyTargetModeActiveRefinement` exports**: defined at [target-mode.ts:52-72](../../src/target-mode/target-mode.ts#L52-L72). After flattening, the only caller of these helpers is the flat schema's own `superRefine` body. The helpers either inline (caller becomes the only call site) or persist for future callers that need to apply the same rules to a different base. Default per Assumptions: inline.
- **`zod-to-json-schema` output for the flat schema**: produces `additionalProperties: false` by default for `.passthrough()` schemas? Or does `.passthrough()` flip it to `true`? `obsidian_exec` uses `.strict()` and gets `false`. The flat target-mode schema must keep `.passthrough()` (the runtime currently allows extras) but the published shape's `additionalProperties` value must be **explicitly chosen** by FR-006, not inherited from a library default that may shift across versions.
- **Legacy `additionalProperties: true` on `read_note`'s wire shape**: feature 009 set this to `true` to satisfy strict-naive Cowork. After flattening, the natural shape can be `false` (every accepted property is in `properties`) — a tighter, more useful descriptor for clients that visualise tool argument hints. SC-001 verifies Cowork still accepts the tighter shape.
- **CLI behaviour**: zero changes. The CLI adapter, argv assembly, and `obsidian-cli read` invocation are downstream of the parsed input and do not care whether the schema was a discriminated union or a flat object.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: `targetModeSchema` MUST be encoded as a flat `z.object({...}).superRefine(...)` (or equivalent flat composition that produces a single top-level `z.object` at the type level), not as a `ZodDiscriminatedUnion` or `ZodEffects<ZodDiscriminatedUnion>`. The schema's top-level shape MUST satisfy `(zodSchema.constructor.name === "ZodEffects" && zodSchema._def.schema.constructor.name === "ZodObject")` or `zodSchema.constructor.name === "ZodObject"`.
- **FR-002**: The runtime semantics of `targetModeSchema.parse(input)` MUST be preserved exactly. Every input the `0.2.1` schema accepts, the post-010 schema accepts. Every input the `0.2.1` schema rejects, the post-010 schema rejects with the same error code (`VALIDATION_ERROR`), the same per-issue `path` array, and a per-issue `message` whose semantic content matches (exact text wording is permitted to vary; the rule each message names — "vault required in specific mode", "exactly one of `file` or `path` must be provided", "<key> is not allowed in active mode" — must match).
- **FR-003**: The 31 acceptance cases in [src/target-mode/target-mode.test.ts](../../src/target-mode/target-mode.test.ts) MUST continue to assert the same rules. Cases that test the specific/active discrimination via narrowed types MAY be rewritten to call `targetModeSchema` directly with a `target_mode` literal; cases that test rule semantics (XOR, forbidden keys, vault-required) MUST pass without semantic changes to their assertions.
- **FR-004**: The `TargetMode` TypeScript type emitted by `z.infer<typeof targetModeSchema>` MUST be a single flat object type (e.g. `{ target_mode: "specific" | "active"; vault?: string; file?: string; path?: string; [k: string]: unknown }`). The 008-refactor surface MUST adapt to the flatter type without behaviour change: [src/tools/read_note/handler.ts](../../src/tools/read_note/handler.ts) is the only consumer that destructures the discriminated union today.
- **FR-005**: [src/tools/_shared.ts](../../src/tools/_shared.ts) MUST shrink to remove the wrap branch entirely. After this feature, `toMcpInputSchema` MUST be either a one-line delegate (`return zodToJsonSchema(zodSchema, { $refStrategy: "none" }) as JsonSchemaObject`) or — if [src/tools/_register.ts:26](../../src/tools/_register.ts#L26) is updated to call `zodToJsonSchema` directly — the helper MUST be deleted. `unionTopLevelProperties`, `intersectionTopLevelRequired`, `stripInnerObjectType`, the wrap-branch comments, and the wrap-branch tests in [_shared.test.ts](../../src/tools/_shared.test.ts) all delete in lock-step.
- **FR-006**: The published `inputSchema` for `read_note` MUST be `{ type: "object", properties: { target_mode, vault, file, path }, required: ["target_mode"], additionalProperties: <chosen> }` — a single flat object with no `oneOf`, no `allOf`, no `anyOf`, no nested wrapping. The `additionalProperties` value is set deliberately by the schema (`.strict()` for `false`, `.passthrough()` for `true`), not inherited from a `zodToJsonSchema` default. The chosen value MUST be re-asserted by the post-010 drift detector.
- **FR-007**: The published `inputSchema` for `obsidian_exec` MUST remain unchanged from `0.2.0` / `0.2.1`: 6 properties, `required: ["command"]`, `additionalProperties: false`, `type: "object"`. The flattening MUST NOT touch [src/tools/obsidian_exec/schema.ts](../../src/tools/obsidian_exec/schema.ts) — that file is already in the canonical flat shape.
- **FR-008**: The drift detector at [src/tools/_register.test.ts](../../src/tools/_register.test.ts) MUST be refactored to drop the three feature-009 groups (unit-layer registry walk, integration-layer SDK round-trip, synthetic Pattern (a)/(b) fixtures). The post-010 detector MUST be a single group that walks the registry once, asserts each tool's published shape against a per-tool invariant table, and runs a single SDK round-trip pass to verify the wire side. The Pattern (a) fixture (if retained) MUST use the post-010 flat-extension idiom (`targetModeSchema.merge(z.object({ note_text: z.string() }))`), not the `targetModeSchema.and(...)` `ZodIntersection` form.
- **FR-009**: The Pattern (b) fixture in [_register.test.ts:436-472](../../src/tools/_register.test.ts#L436-L472) MUST be deleted. Pattern (b) is no longer a documented reuse pattern after this feature (FR-013); the fixture's only purpose was to verify the wrap branch handled fresh discriminated unions, which the wrap branch no longer exists to do.
- **FR-010**: This feature MUST NOT introduce any new error codes. The reachable error roster for `read_note` and for the server overall is unchanged. `VALIDATION_ERROR` payload shape (per-issue `path`, `code`, `message`) is preserved per FR-002.
- **FR-011**: This feature MUST be released as version `0.3.0` (minor — the published TypeScript type of `TargetMode` is a public-export breaking change, even though the runtime behaviour is unchanged). If `/speckit-plan` determines that no public TypeScript type leaks (e.g. `TargetMode` is not currently re-exported from `index.ts`), the release MAY be `0.2.2` (patch). The version determination MUST be surfaced as a clarification rather than decided silently.

- **FR-013**: ADR-003's "discriminated union" wording (line 20 of [.decisions/ADR-003 - Enforce Target Mode in Typed Tools.md](../../.decisions/ADR-003%20-%20Enforce%20Target%20Mode%20in%20Typed%20Tools.md)) MUST be amended (or superseded by a new ADR — choice deferred to `/speckit-clarify` C5) to read "flat `z.object` with a `superRefine` that enforces the per-mode rules". - **FR-014**: The publication pipeline MUST remain mechanically derived from the zod schema (Principle III). No parallel hand-written JSON Schema is permitted; if [src/tools/_shared.ts](../../src/tools/_shared.ts) becomes a one-line delegate, that line MUST call `zodToJsonSchema` rather than return a literal.

- **FR-016**: The 008-refactor surface outside of [src/target-mode/target-mode.ts](../../src/target-mode/target-mode.ts) and [src/tools/_shared.ts](../../src/tools/_shared.ts) MUST remain unchanged: `dispatchCli`, `invokeCli`, `invokeBoundedCli`, the in-flight registry, the four-priority error classification, the always-on bounds, `assertToolDocsExist`, and the `obsidian_exec` argv-assembly contract are all frozen. The flatten touches the schema and the helper that consumes it; nothing else.
- **FR-017**: Existing `targetModeSpecificBaseSchema` / `targetModeActiveBaseSchema` exports — which document Pattern (b) re-use over the per-mode bases — MUST be deleted (no consumer post-010). The `targetModeSpecificSchema` / `targetModeActiveSchema` exports either delete (preferred) or persist as named refinements over the flat schema; the choice is deferred to `/speckit-clarify` (C2). The `applyTargetModeSpecificRefinement` / `applyTargetModeActiveRefinement` helpers either inline into the flat `superRefine` body (preferred) or persist as exported helpers — also deferred to C2.

### Key Entities

- **Flat `targetModeSchema`** *(post-010)*: A single `z.object({ target_mode, vault?, file?, path? }).passthrough().superRefine(...)`, where the `superRefine` body branches on `target_mode` and applies the per-mode rules. Replaces the discriminated union at the same path ([src/target-mode/target-mode.ts](../../src/target-mode/target-mode.ts)).
- **Discriminated `targetModeSchema`** *(pre-010)*: The current `targetModeBaseUnion.superRefine(...)`. Removed by FR-001.
- **`toMcpInputSchema` post-010**: A one-line delegate to `zodToJsonSchema` (or absent, if [_register.ts](../../src/tools/_register.ts) inlines the call). FR-005 mandates the shrink.
- **Drift detector post-010**: A single registry-walk test in [_register.test.ts](../../src/tools/_register.test.ts) that iterates the registered tools and asserts each one's published `inputSchema` against a per-tool invariant table. FR-008 mandates the consolidation.
- **`TargetMode` TypeScript type**: A flat object type post-010 (FR-004). Public export from [src/target-mode/target-mode.ts](../../src/target-mode/target-mode.ts). Whether this constitutes a public-API break depends on whether it is currently re-exported from a top-level `index.ts` (FR-011).
- **Pattern (a) post-010**: Flat extension via `targetModeSchema.merge(z.object({ <fields> }))`. The single canonical reuse pattern after FR-013.
- **Pattern (b) pre-010**: Fresh discriminated union over the per-mode base schemas with union-level `superRefine`. Removed from the canonical roster by FR-013; existing tests deleted by FR-009.

## Assumptions

- **Reasonable defaults the user enumerated** (the feature is a structural simplification, not a bug fix; runtime semantics preserved exactly; ADR-003 amended in lock-step) are taken as fixed inputs and inherited rather than re-derived.
- **The `vault` non-null assertion in [handler.ts](../../src/tools/read_note/handler.ts)** is the simplest of three options for handling the flatter type narrowing.
