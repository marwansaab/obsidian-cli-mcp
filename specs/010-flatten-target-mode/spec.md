# Feature Specification: Flatten `targetModeSchema` and Retire the Wrap-Branch Envelope Helper

**Feature Branch**: `010-flatten-target-mode`
**Created**: 2026-05-07
**Status**: Draft
**Input**: User description: features 007 and 009 ship a working but accumulating fix for the bug introduced in 006-read-note. The `read_note` tool's published `inputSchema` derives from `targetModeSchema` — a `ZodEffects<ZodDiscriminatedUnion>` — which `zod-to-json-schema` emits as a top-level `anyOf`. To bridge that to the MCP `Tool.inputSchema` contract, [src/tools/_shared.ts](../../src/tools/_shared.ts) carries ~140 lines of envelope synthesis: wrap branch, `oneOf` rewrite, top-level `properties` union, top-level `required` intersection, Pattern (a) `allOf` walking, leaf widening with cross-branch string-discriminator surfacing. A three-group drift detector at [src/tools/_register.test.ts](../../src/tools/_register.test.ts) exists to police this pipeline. None of this complexity would be necessary if the zod schema feeding `zodToJsonSchema` were a flat `z.object({...})`. This feature replaces the discriminated-union encoding of the target-mode rules with a flat `z.object` + `superRefine`, deleting the envelope helper's wrap branch, the synthesis logic, the Pattern (a)/(b) handling, and the drift detector groups that exist solely to verify them. The runtime semantics — what inputs zod accepts, what inputs it rejects, the `VALIDATION_ERROR` payload shape — are preserved exactly. The TypeScript type of `TargetMode` flattens, and ADR-003's "discriminated union" wording is amended to "flat object with `superRefine`" to reflect the new encoding.

## Background *(non-normative — context only)*

`0.2.1` (feature 009) ships a working fix: the published `inputSchema` for `read_note` is well-formed for both strict-rich (Claude Desktop) and strict-naive (Cowork) MCP clients, the drift detector observes the wire shape, and the typed-tool roadmap is unblocked. The fix is correct and merged. This feature is not a bug fix.

It is a structural simplification. Three branches (007, 008, 009) accumulated against a single root cause: the published JSON Schema is mechanically derived from a zod schema (`targetModeSchema`) whose top-level construct (`ZodDiscriminatedUnion`) does not survive the zod → JSON Schema → MCP `inputSchema` pipeline cleanly. Each fix bridged the gap downstream — by widening the envelope helper, by adding companion exports (007), by adding the registry pipeline (008), by synthesising top-level `properties` (009). None of them changed the input.

The cleaner alternative — encode the same target-mode rules with a flat `z.object` + `superRefine` rather than a discriminated union — produces a JSON Schema that satisfies the MCP `Tool.inputSchema` contract directly, with no envelope synthesis. The discriminated union exists today because zod 3.x's `z.discriminatedUnion` provides type narrowing at the consumer site (`if (input.target_mode === "specific") { input.vault /* string */ }`); the flat encoding loses that narrowing (`input.vault` becomes `string | undefined`). The trade is: simpler publication pipeline + smaller test surface vs. weaker type narrowing in handler code.

The control case is informative. [src/tools/obsidian_exec/schema.ts](../../src/tools/obsidian_exec/schema.ts) is a flat `z.object({...}).strict()`. Its published `inputSchema` is the natural `zodToJsonSchema` output. It works through every MCP client class observed under 007/008/009 without any envelope handling. `read_note`'s post-009 published shape will, after this feature, look structurally identical (one less property, different property roster, otherwise the same kind of object).

## Clarifications

### Session 2026-05-07

- Q: Pattern (b) (fresh discriminated union with union-level `superRefine`) — keep as a documented reuse pattern, deprecate, or delete entirely from the roster? → A: Delete from the canonical roster; wrap branch in `_shared.ts` deletes entirely; a future consumer that genuinely needs a discriminated union re-adds a narrower wrap branch in its own feature.
- Q: `additionalProperties` policy on the flat schema (`.strict()` vs. `.passthrough()` vs. `.strip()`) — which best matches the project's runtime-rejection ergonomics and the desired wire descriptor? → A: `.strict()` — wire emits `additionalProperties: false`; unknown keys produce `VALIDATION_ERROR` with a path naming the offending key. Matches `obsidian_exec`'s control case; tightens the LLM-tool-use descriptor; surfaces client-side typos that 0.2.1 currently silently passes through. Constitutes a deliberate, narrow runtime-behaviour change vs. 0.2.1 (see FR-002 carve-out).
- Q: Handler-side type narrowing in [src/tools/read_note/handler.ts:21-27](../../src/tools/read_note/handler.ts#L21-L27) after `TargetMode` flattens to a single object type with optional fields — non-null assertion vs. explicit runtime check vs. a narrowing helper at the target-mode layer? → A: Non-null assertion (`input.vault!`) with a single-line comment naming the `superRefine` invariant. Smallest diff, idiomatic TypeScript "trust the validated input" pattern, comment documents the runtime guarantee for future readers.
- Q: ADR-003 (`Enforce Target Mode in Typed Tools`) — amend the encoding paragraph in place, or supersede with a new ADR-008? → A: Amend ADR-003 in place. The rationale (force explicit intent, boundary validation, separate co-pilot/orchestrator) is unchanged; only the encoding paragraph wording changes from "discriminated union" to "flat `z.object` with `superRefine`". Bump `updated:` frontmatter; add an "Amendment 2026-MM-DD" stanza at the bottom noting why the encoding changed (publication-pipeline simplification per feature 010). No new ADR.
- Q: Per-mode schema exports (`targetModeSpecificSchema`, `targetModeActiveSchema`), refinement helpers (`applyTargetModeSpecificRefinement`, `applyTargetModeActiveRefinement`), and per-mode base schemas (`targetModeSpecificBaseSchema`, `targetModeActiveBaseSchema`) — delete all, keep selectively, or keep everything? → A: Delete all six exports. Their consumers are six existing test cases, which migrate to call `targetModeSchema` directly with the matching `target_mode` literal. Pattern (b) is gone (clarification above), so the per-mode bases have lost their last documented consumer. Maximises the deletion footprint and leaves a single schema export to learn.
- Q: User Story 1's canonical flat-extension idiom — `targetModeSchema.merge(z.object({ note_text: z.string() }))` — is technically infeasible because `.merge()` is a `ZodObject` method and `targetModeSchema` is `ZodEffects<ZodObject>` (the `superRefine` wrapper). What is the correct idiom for a Pattern (a) consumer like `write_note`? → A: Export a sibling `targetModeBaseSchema` (the bare `z.object({...}).strict()` before `superRefine`) alongside the refined `targetModeSchema`, plus an `applyTargetModeRefinement(schema)` helper that attaches the per-mode rules to any extended base. Pattern (a) consumers do `applyTargetModeRefinement(targetModeBaseSchema.merge(z.object({ note_text: z.string() })))`. The refined `targetModeSchema` remains the canonical export for the no-extension case (`read_note`). **Research-stage refinement (see [research.md](research.md) R2):** the `.merge()` example call shown above is technically inferior to `.extend({...})` — `.merge()` resets `unknownKeys` to zod's default `"strip"`, whereas `.extend()` preserves the parent's `.strict()` mode. Both calls produce identical JSON Schema descriptors; only `.extend()` honours the FR-002 strict-mode carve-out at runtime. Plan-stage commits to `.extend({...})` as the canonical call; this clarification's `.merge()` example is preserved verbatim as the answer text but every downstream reference uses `.extend({...})`.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Future typed tools follow the same flat-`z.object` pattern as `obsidian_exec` (Priority: P1)

A developer (or AI coding assistant) is asked to add a new typed tool that needs target-mode behaviour — e.g. the planned `write_note` or `append_note`. Today, the canonical reuse pattern is "extend `targetModeSchema` via `targetModeSchema.and(z.object({ <fields> }))`" (Pattern (a)) or "build a fresh discriminated union over the per-mode base schemas" (Pattern (b)). After this feature, the canonical pattern is "extend the bare `targetModeBaseSchema` via `.extend({...})` and re-apply the refinement: `applyTargetModeRefinement(targetModeBaseSchema.extend({ <fields> }))`" (clarification 2026-05-07 / C7, refined by research R2). The split is necessary because `targetModeSchema` itself is `ZodEffects<ZodObject>` (the `superRefine` wrapper) and `.extend()` is a `ZodObject`-only method; the base export gives consumers a composable `ZodObject`, and the refinement helper re-attaches the per-mode rules to the extended shape. `.extend()` is preferred over `.merge()` because it preserves the parent's `.strict()` `unknownKeys` mode (research R2). The author writes only zod; the published `inputSchema` is the natural `zodToJsonSchema` output; no envelope helper sits between the schema and the wire.

**Why this priority**: The publication pipeline is the source of complexity. Replacing the input shape eliminates the helper's wrap branch entirely, which is the largest standing surface in [_shared.ts](../../src/tools/_shared.ts). Pattern (a)/(b) re-use guidance simplifies in lock-step: there is one pattern, it matches `obsidian_exec`'s shape, and every future tool inherits a clean published descriptor for free.

**Independent Test**: Add a synthetic typed tool whose registered schema is `applyTargetModeRefinement(targetModeBaseSchema.extend({ note_text: z.string() }))` (the flat-extension equivalent of feature 009's Pattern (a)). Assert via the drift detector that its published `inputSchema` has `type: "object"`, contains `target_mode`, `vault`, `file`, `path`, and `note_text` in `properties`, has `required: ["target_mode", "note_text"]`, and `additionalProperties: false`. Assert via runtime parse that `vault` is required when `target_mode === "specific"` (enforced by the re-applied `superRefine`, not by the type-level `required` array), that `vault`/`file`/`path` are forbidden when `target_mode === "active"`, that exactly one of `file` / `path` is required when `target_mode === "specific"`, AND that an unknown top-level key produces `VALIDATION_ERROR` with `code: "unrecognized_keys"` (verifying `.extend()` preserved `.strict()`). Then remove the synthetic tool — it exists only as a fixture.

**Acceptance Scenarios**:

1. **Given** `targetModeBaseSchema` and `applyTargetModeRefinement` are the canonical extension exports alongside the refined `targetModeSchema`, **When** a developer registers a tool whose schema is `applyTargetModeRefinement(targetModeBaseSchema.extend({ note_text: z.string() }))`, **Then** the published `inputSchema` is a single flat object with all five property names in `properties`, no `oneOf`, no `allOf`, and `type: "object"` at the root.
2. **Given** the same registered tool, **When** the runtime receives `{ target_mode: "specific", vault: "v", path: "p", note_text: "x" }`, **Then** zod accepts it.
3. **Given** the same registered tool, **When** the runtime receives `{ target_mode: "specific", note_text: "x" }` (missing vault), **Then** zod returns `VALIDATION_ERROR` with an issue at `path: ["vault"]` whose message names "vault is required in specific mode" or equivalent.
4. **Given** the same registered tool, **When** the runtime receives `{ target_mode: "active", vault: "v", note_text: "x" }`, **Then** zod returns `VALIDATION_ERROR` with an issue at `path: ["vault"]` whose message names "vault is not allowed in active mode" — identical to today's behaviour.

### User Story 2 — `read_note` continues to work end-to-end through every MCP client class (Priority: P1)

A user has `@marwansaab/obsidian-cli-mcp@<post-010>` installed and configured in any MCP client (Cowork, Claude Desktop, MCP Inspector, generic SDK consumer). The client successfully completes `tools/list`, sees `read_note`, validates outgoing arguments against its published `inputSchema`, dispatches both `target_mode: "specific"` and `target_mode: "active"` invocations end-to-end without argument stripping, and the runtime returns the note's content. The user observes no behavioural difference from `0.2.1` — same accepted inputs, same rejected inputs, same `VALIDATION_ERROR` payloads.

**Why this priority**: Equal priority. The simplification is only worth shipping if it preserves user-visible behaviour exactly. A regression here returns the project to 006-state; the whole point of the simplification is that it is at-worst behaviour-preserving.

**Independent Test**: Same manual verification ladder as feature 009's S-11 / S-12 (Cowork + Claude Desktop, both modes). The wire-shape changes — `oneOf` and the cross-branch property union both vanish, replaced by a single flat `properties` map — but the behavioural contract is unchanged. A drift-detector test in [_register.test.ts](../../src/tools/_register.test.ts) (refactored, see FR-008) re-asserts the post-feature shape.

**Acceptance Scenarios**:

1. **Given** the post-010 package is installed, **When** Cowork calls `read_note({ target_mode: "specific", vault: "<v>", path: "<p>" })`, **Then** the call returns `{ content: <stdout> }` (no `VALIDATION_ERROR`, no argument stripping).
2. **Given** the post-010 package is installed, **When** Cowork calls `read_note({ target_mode: "active" })`, **Then** the call returns `{ content: <stdout> }` for the active note.
3. **Given** the post-010 package is installed, **When** Claude Desktop (or MCP Inspector) inspects `read_note` in `tools/list`, **Then** the published `inputSchema` is a flat `{ type: "object", properties: { target_mode, vault, file, path }, required: ["target_mode"], additionalProperties: false }` with no `oneOf`, no `allOf`, no `anyOf`, no `$schema` envelope wrapping.
4. **Given** the post-010 package is installed, **When** any client invokes `read_note` with a malformed input (both `file` and `path` in specific, vault in active, no locator in specific), **Then** the runtime returns `VALIDATION_ERROR` with the same per-issue path and message text feature 009 already produces — verified by reusing 009's existing target-mode test fixtures.

### Edge Cases

- **Pattern (a) consumers from feature 009**: today the synthetic Pattern (a) fixture in [_register.test.ts:415-434](../../src/tools/_register.test.ts#L415-L434) constructs `targetModeSchema.and(z.object({ note_text: z.string() }))`, producing a `ZodIntersection` whose JSON Schema has top-level `allOf`. After flattening, the canonical extension is `applyTargetModeRefinement(targetModeBaseSchema.extend({...}))` per clarification C7 + research R2 — a single refined `z.object`, and the `allOf` shape vanishes. The fixture deletes; the assertion that "future typed tools inherit the fix" moves to a flat-extension fixture (User Story 1 above). Note: `targetModeSchema.extend(...)` and `targetModeSchema.merge(...)` directly are both infeasible (it is `ZodEffects<ZodObject>` from the `superRefine`; both methods are `ZodObject`-only), which is why the base/refinement split is the canonical idiom rather than a single-name extension. Within the base/refinement split, `.extend()` is preferred over `.merge()` because `.merge()` resets `unknownKeys` to `"strip"` and silently drops unknown keys at parse time — research R2 captures the empirical evidence.
- **Pattern (b) consumers**: today's Pattern (b) fixture constructs a fresh discriminated union with union-level `superRefine`. After this feature, that pattern is no longer canonical (it would still work, but the publication pipeline can no longer guarantee a single `properties` map for it; it would re-trigger the wrap-branch problem we are deleting). Pattern (b) is removed from the documented reuse framework as a follow-up to feature 004's spec; the cross-feature impact is recorded in the documentation update (FR-013).
- **Type narrowing in `read_note`'s handler**: today [handler.ts:21-27](../../src/tools/read_note/handler.ts#L21-L27) reads `input.vault` (typed as `string`) inside the `target_mode === "specific"` branch. After flattening, `input.vault` is `string | undefined`. Per clarification 2026-05-07 (C1): use `input.vault!` with a single-line comment naming the `superRefine` runtime invariant (vault is defined when `target_mode === "specific"`). Smallest diff, idiomatic, runtime-safe.
- **Existing per-mode schema exports** (`targetModeSpecificSchema`, `targetModeActiveSchema`, `targetModeSpecificBaseSchema`, `targetModeActiveBaseSchema`) and **refinement helpers** (`applyTargetModeSpecificRefinement`, `applyTargetModeActiveRefinement`): all six MUST be deleted per FR-017 (clarification 2026-05-07 / C2). The `refineSpecificBranch` / `refineActiveBranch` private functions inline into the flat `superRefine` body. The six existing test cases that consume these exports migrate to invoke `targetModeSchema` directly with the matching `target_mode` literal.
- **`zod-to-json-schema` output for the flat schema**: `.strict()` produces `additionalProperties: false`; `.passthrough()` produces `true`; `.strip()` produces `false` but silently strips at parse time. Per clarification 2026-05-07 (C3), the flat schema is `.strict()` — wire shows `false`, parse-time rejects unknown keys with a `VALIDATION_ERROR`. Matches `obsidian_exec`'s control case.
- **Legacy `additionalProperties: true` on `read_note`'s wire shape**: feature 009 set this to `true` to satisfy strict-naive Cowork. After flattening, the wire flips to `false` per FR-006 — a tighter, more useful descriptor for clients that visualise tool argument hints. SC-001 verifies Cowork still accepts the tighter shape (`additionalProperties: false` is strictly more conservative than `true`; any client that accepted the latter accepts the former).
- **CLI behaviour**: zero changes. The CLI adapter, argv assembly, and `obsidian-cli read` invocation are downstream of the parsed input and do not care whether the schema was a discriminated union or a flat object.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: `targetModeSchema` MUST be encoded as a flat `z.object({...}).strict().superRefine(...)`, not as a `ZodDiscriminatedUnion` or `ZodEffects<ZodDiscriminatedUnion>`. The schema's top-level shape MUST satisfy `zodSchema.constructor.name === "ZodEffects" && zodSchema._def.schema.constructor.name === "ZodObject"`. Per clarification C7 + research R2, [src/target-mode/target-mode.ts](../../src/target-mode/target-mode.ts) MUST also export: (a) `targetModeBaseSchema` — the bare `z.object({ target_mode, vault?, file?, path? }).strict()` *before* `.superRefine`, exposed for Pattern (a) consumers that need `.extend({...})` composition (`.merge()` is technically permissible but resets `unknownKeys` to `"strip"`; `.extend()` preserves `.strict()` and is the canonical idiom); (b) `applyTargetModeRefinement(schema: ZodObject) => ZodEffects<ZodObject>` — the helper that attaches the per-mode rules (XOR `file`/`path`, vault required when specific, locator-keys forbidden when active) to any extended base. The refined `targetModeSchema` itself is `applyTargetModeRefinement(targetModeBaseSchema)` and remains the canonical export for the no-extension case (`read_note`).
- **FR-002**: The runtime semantics of `targetModeSchema.parse(input)` MUST be preserved exactly for every input that satisfies the documented contract — every input the `0.2.1` schema accepts AND that uses only documented keys (`target_mode`, `vault`, `file`, `path`), the post-010 schema accepts; every input the `0.2.1` schema rejects, the post-010 schema rejects with the same error code (`VALIDATION_ERROR`), the same per-issue `path` array, and a per-issue `message` whose semantic content matches (exact text wording is permitted to vary; the rule each message names — "vault required in specific mode", "exactly one of `file` or `path` must be provided", "<key> is not allowed in active mode" — must match). **Strict-mode carve-out (clarification 2026-05-07):** the post-010 schema is `.strict()`, so any input containing an *undocumented* top-level key (e.g. a typo or a deprecated field) that `0.2.1` would have silently passed through MUST produce `VALIDATION_ERROR` with a per-issue `code: "unrecognized_keys"` (or zod's equivalent) naming the offending key. This is a deliberate, narrow runtime-behaviour change recorded in `CHANGELOG.md` under FR-012.
- **FR-003**: The 31 acceptance cases in [src/target-mode/target-mode.test.ts](../../src/target-mode/target-mode.test.ts) MUST continue to assert the same rules. Cases that test the specific/active discrimination via narrowed types MAY be rewritten to call `targetModeSchema` directly with a `target_mode` literal; cases that test rule semantics (XOR, forbidden keys, vault-required) MUST pass without semantic changes to their assertions.
- **FR-004**: The `TargetMode` TypeScript type emitted by `z.infer<typeof targetModeSchema>` MUST be a single flat object type: `{ target_mode: "specific" | "active"; vault?: string; file?: string; path?: string }` (no index signature — `.strict()` per clarification C3 omits `[k: string]: unknown`). The 008-refactor surface MUST adapt to the flatter type without behaviour change: [src/tools/read_note/handler.ts](../../src/tools/read_note/handler.ts) is the only consumer that destructures the discriminated union today.
- **FR-005**: [src/tools/_shared.ts](../../src/tools/_shared.ts) MUST shrink to remove the wrap branch entirely. After this feature, `toMcpInputSchema` MUST be either a one-line delegate (`return zodToJsonSchema(zodSchema, { $refStrategy: "none" }) as JsonSchemaObject`) or — if [src/tools/_register.ts:26](../../src/tools/_register.ts#L26) is updated to call `zodToJsonSchema` directly — the helper MUST be deleted. `unionTopLevelProperties`, `intersectionTopLevelRequired`, `stripInnerObjectType`, the wrap-branch comments, and the wrap-branch tests in [_shared.test.ts](../../src/tools/_shared.test.ts) all delete in lock-step.
- **FR-006**: The published `inputSchema` for `read_note` MUST be `{ type: "object", properties: { target_mode, vault, file, path }, required: ["target_mode"], additionalProperties: false }` — a single flat object with no `oneOf`, no `allOf`, no `anyOf`, no nested wrapping. The `additionalProperties: false` value is achieved by `.strict()` on the flat schema (clarification 2026-05-07 / C3). The drift detector asserts this value explicitly.
- **FR-007**: The published `inputSchema` for `obsidian_exec` MUST remain unchanged from `0.2.0` / `0.2.1`: 6 properties, `required: ["command"]`, `additionalProperties: false`, `type: "object"`. The flattening MUST NOT touch [src/tools/obsidian_exec/schema.ts](../../src/tools/obsidian_exec/schema.ts) — that file is already in the canonical flat shape.
- **FR-008**: The drift detector at [src/tools/_register.test.ts](../../src/tools/_register.test.ts) MUST be refactored to drop the three feature-009 groups (unit-layer registry walk, integration-layer SDK round-trip, synthetic Pattern (a)/(b) fixtures). The post-010 detector MUST be a single group that walks the registry once, asserts each tool's published shape against a per-tool invariant table, and runs a single SDK round-trip pass to verify the wire side. The Pattern (a) fixture (if retained) MUST use the post-010 flat-extension idiom `applyTargetModeRefinement(targetModeBaseSchema.extend({ note_text: z.string() }))` per clarification C7 + research R2, not the `targetModeSchema.and(...)` `ZodIntersection` form (and not the infeasible `targetModeSchema.extend(...)` / `targetModeSchema.merge(...)` forms).
- **FR-009**: The Pattern (b) fixture in [_register.test.ts:436-472](../../src/tools/_register.test.ts#L436-L472) MUST be deleted. Pattern (b) is no longer a documented reuse pattern after this feature (FR-013); the fixture's only purpose was to verify the wrap branch handled fresh discriminated unions, which the wrap branch no longer exists to do.
- **FR-010**: This feature MUST NOT introduce any new error codes. The reachable error roster for `read_note` and for the server overall is unchanged. `VALIDATION_ERROR` payload shape (per-issue `path`, `code`, `message`) is preserved per FR-002.

- **FR-013**: ADR-003's "discriminated union" wording (line 20 of [.decisions/ADR-003 - Enforce Target Mode in Typed Tools.md](../../.decisions/ADR-003%20-%20Enforce%20Target%20Mode%20in%20Typed%20Tools.md)) MUST be amended **in place** to read "flat `z.object` with a `superRefine` that enforces the per-mode rules" (clarification 2026-05-07 / C5). The amendment MUST bump the `updated:` frontmatter date and add an "Amendment 2026-MM-DD" stanza at the bottom of the ADR recording why the encoding changed (publication-pipeline simplification per feature 010). NO new ADR is created. Feature 004's Pattern (b) reuse documentation is amended in lock-step to remove Pattern (b) from the canonical pattern roster.
- **FR-014**: The publication pipeline MUST remain mechanically derived from the zod schema (Principle III). No parallel hand-written JSON Schema is permitted; if [src/tools/_shared.ts](../../src/tools/_shared.ts) becomes a one-line delegate, that line MUST call `zodToJsonSchema` rather than return a literal.

### Key Entities

- **Flat `targetModeSchema`** *(post-010)*: `applyTargetModeRefinement(targetModeBaseSchema)`, equivalent to `z.object({ target_mode, vault?, file?, path? }).strict().superRefine(...)`. The `superRefine` body branches on `target_mode` and applies the per-mode rules. Replaces the discriminated union at the same path ([src/target-mode/target-mode.ts](../../src/target-mode/target-mode.ts)).
- **`targetModeBaseSchema`** *(post-010, new export)*: The bare `z.object({ target_mode: z.enum(["specific","active"]), vault: z.string().min(1).optional(), file: z.string().optional(), path: z.string().optional() }).strict()` *before* `.superRefine`. A `ZodObject` (not `ZodEffects`), so it composes with `.extend({...})` (preferred — preserves `.strict()`) or `.merge(z.object({...}))` (resets `unknownKeys`; not the canonical idiom — research R2). Required by FR-001 / clarification C7 to give Pattern (a) consumers a working extension idiom.
- **`applyTargetModeRefinement`** *(post-010, new helper)*: A function `(schema: ZodObject) => ZodEffects<ZodObject>` that wraps any zod object whose shape includes the four target-mode keys with the per-mode rule body (XOR `file`/`path`, vault required when specific, locator-keys forbidden when active). Used internally to define `targetModeSchema = applyTargetModeRefinement(targetModeBaseSchema)` and externally by Pattern (a) consumers.
- **Discriminated `targetModeSchema`** *(pre-010)*: The current `targetModeBaseUnion.superRefine(...)`. Removed by FR-001.
- **`toMcpInputSchema` post-010**: A one-line delegate to `zodToJsonSchema` (or absent, if [_register.ts](../../src/tools/_register.ts) inlines the call). FR-005 mandates the shrink.
- **Drift detector post-010**: A single registry-walk test in [_register.test.ts](../../src/tools/_register.test.ts) that iterates the registered tools and asserts each one's published `inputSchema` against a per-tool invariant table. FR-008 mandates the consolidation.
- **`TargetMode` TypeScript type**: A flat object type post-010 (FR-004). Public export from [src/target-mode/target-mode.ts](../../src/target-mode/target-mode.ts). Whether this constitutes a public-API break depends on whether it is currently re-exported from a top-level `index.ts` (FR-011).
- **Pattern (a) post-010**: Flat extension via `applyTargetModeRefinement(targetModeBaseSchema.extend({ <fields> }))` per clarification C7 + research R2. The single canonical reuse pattern after FR-013.
- **Pattern (b) pre-010**: Fresh discriminated union over the per-mode base schemas with union-level `superRefine`. Removed from the canonical roster by FR-013; existing tests deleted by FR-009.
