# Feature Specification: Read Note Typed MCP Tool

**Feature Branch**: `006-read-note`
**Created**: 2026-05-06
**Status**: Draft
**Input**: User description: "Add Read Note — A typed MCP tool that reads the contents of a single vault note. Direct one-to-one wrap of the Obsidian CLI's `read` subcommand. Composes on the obsidian_exec primitive shipped in BI-001."

## Background *(non-mandatory context)*

This feature implements **BI-003 — Add Read Note**, the first of the typed-tool BIs committed in [.architecture/Obsidian CLI MCP - Architecture.md](../../.architecture/Obsidian%20CLI%20MCP%20-%20Architecture.md). It is the empirical validator for three foundational features that landed before it:

- **[004-target-mode-schema](../004-target-mode-schema/spec.md) (BI-029)** ships the shared `target_mode` discriminated-union schema primitive at [src/target-mode/target-mode.ts](../../src/target-mode/target-mode.ts). Its specific branch enforces `vault: z.string().min(1)` (required) and "exactly one of `file` or `path` is provided"; its active branch forbids `vault`/`file`/`path`. Per Constitution Principle III, this primitive is the single source of truth for those rules — `read_note` MUST compose on top of it rather than re-state the contract inline.
- **[003-cli-adapter](../003-cli-adapter/spec.md) (BI-028)** ships the centralised CLI adapter at [src/cli-adapter/cli-adapter.ts](../../src/cli-adapter/cli-adapter.ts) — `invokeCli({ command, parameters, flags, target_mode }, deps?)`. The adapter handles argv assembly, vault hoisting, active-mode target-locator stripping, and the four-priority error classification (`CLI_BINARY_NOT_FOUND`, `CLI_NON_ZERO_EXIT`, `ERR_NO_ACTIVE_FILE`, `CLI_REPORTED_ERROR`). `read_note` MUST route every CLI invocation through this adapter.
- **[005-help-tool](../005-help-tool/spec.md) (BI-030)** ships the schema-stripping utility at [src/help/strip-schema.ts](../../src/help/strip-schema.ts) and the `help` tool that serves Markdown documentation from [docs/tools/](../../docs/tools/). It already includes a stub `docs/tools/read_note.md` carrying the `<!-- TODO(BI-003): populate this doc when the tool ships -->` marker; this BI populates that file with real documentation. The schema-stripping utility MUST be applied at this tool's registration site so its `inputSchema` ships description-free.

The user-facing contract this feature delivers is straightforward: an MCP client calls `read_note({ target_mode: "specific", vault: "MyVault", file: "Note" })` (or with `path` instead of `file`, or with `target_mode: "active"` for the focused note) and receives `{ content: "<note text>" }`. The CLI's `read` subcommand returns the note's raw UTF-8 text on stdout; the handler maps that to the `content` field. Validation failures, CLI failures, and missing-active-file conditions all surface through the existing `UpstreamError` codespace — no new error codes are introduced by this BI.

The user's input phrases the implementation as "composes via `ObsidianCLIAdapter.execute(\"read\", args, target_mode)`"; the spec follows the actually-shipped adapter API (`invokeCli({ command, parameters, flags, target_mode }, deps?)`) per the Constitution's "spec follows the code that exists, not the code that was sketched" rule. Semantic intent is preserved.

## Clarifications

### Session 2026-05-06

- Q: Should `read_note` share `obsidian_exec`'s in-flight CLI queue (single-channel serialization across both tools), use a separate queue, or run unqueued? → A: Share `obsidian_exec`'s queue — reads serialize behind execs and behind each other through one channel. (See FR-016.)
- Q: Should `read_note` accept a `Logger` dep and emit structured logs (start/finish/error) around the adapter call, or run silent? → A: Accept logger dep, emit start/finish/error events — observability parity with `obsidian_exec`. (See FR-017.)
- Q: Should the schema layer enforce `.min(1)` on `file` and `path` at the `read_note` tool layer (tightening the primitive's permissive `.optional()` shape) to reject empty-string inputs before any CLI call? → A: No — accept the primitive's permissive shape; empty-string `file=`/`path=` forwards to the CLI and surfaces as `CLI_NON_ZERO_EXIT` or `CLI_REPORTED_ERROR`. Avoids a second refinement layer that could drift from the primitive. (See updated Edge Case for empty-string `file`.)

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Specific-mode read by `file` (wikilink) returns the note's raw text (Priority: P1)

An MCP client (typically an LLM agent) needs to read the contents of a specific note in a known vault, identified by name (the wikilink form — no path, no extension). The agent calls `read_note({ target_mode: "specific", vault: "MyVault", file: "Recipe" })` and receives `{ content: "<the raw text of Recipe.md>" }` — the same UTF-8 text the Obsidian CLI's `read` subcommand emits on stdout. No transformation, no JSON wrapping of the body, no metadata: the `content` field carries the file's bytes verbatim.

**Why this priority**: This is the headline value of the feature. `file=` (wikilink) is the most common locator the agent will reach for — it does not require the agent to know vault directory layout, only the note's name. Without this story, the typed-tool track has no read primitive and every read goes through the freeform `obsidian_exec` escape hatch (defeating the point of typed tools).

**Independent Test**: With a stub `spawnFn` injected via the adapter's `deps` parameter, calling `read_note` with `{ target_mode: "specific", vault: "MyVault", file: "Recipe" }` against a stub child that exits `0` with stdout `"# Recipe\n\nIngredients...\n"` results in (a) the stub spawn being invoked with binary `"obsidian"` and argv `["read", "vault=MyVault", "file=Recipe"]` (vault hoisted, file second per the adapter's argv-assembly contract), and (b) the handler returning a tool-call result whose `content[0].text` is the JSON-serialised `{ content: "# Recipe\n\nIngredients...\n" }`. The test runs with no real CLI binary involved.

**Acceptance Scenarios**:

1. **Given** the `read_note` tool is registered and a stub adapter that exits `0` with stdout `"# Recipe\n\nIngredients...\n"`, **When** an MCP client calls `read_note({ target_mode: "specific", vault: "MyVault", file: "Recipe" })`, **Then** the tool returns a successful MCP response whose JSON-serialised payload is `{ content: "# Recipe\n\nIngredients...\n" }` and the adapter was invoked with argv `["read", "vault=MyVault", "file=Recipe"]`.
2. **Given** the `read_note` tool is registered and a stub adapter that exits `0` with stdout that is the empty string, **When** an MCP client calls `read_note({ target_mode: "specific", vault: "V", file: "EmptyNote" })`, **Then** the tool returns `{ content: "" }` — empty notes are a successful read with empty content, not a failure surface.
3. **Given** the `read_note` tool, **When** an MCP client calls it with the `file` value containing UTF-8 multi-byte characters (e.g., `file: "笔记"` or `file: "Niño"`), **Then** the adapter is invoked with that exact `file=…` token and the call succeeds — `read_note` does not transcode, normalise, or escape the file value.

### User Story 2 — Specific-mode read by `path` (vault-relative path) returns the note's raw text (Priority: P1)

When an agent knows the exact vault-relative path to a note (e.g., `Templates/Recipe.md`) — typically because it was returned by a prior `list_notes` or `search_vault` call — it uses `path` instead of `file`. The agent calls `read_note({ target_mode: "specific", vault: "MyVault", path: "Templates/Recipe.md" })` and receives the file's text. `path` and `file` are mutually exclusive at the schema layer (Story 4); this story exercises the path-locator branch.

**Why this priority**: P1 alongside Story 1, not P2. The two locators target the same CLI subcommand but resolve different ambiguity classes — `file=` is wikilink-resolved by Obsidian (which can disambiguate between `Recipes/Bread.md` and `Drafts/Bread.md` only in some configurations), `path=` is exact and disambiguates unconditionally. An agent that has the path SHOULD use `path=`; downgrading this to P2 would imply the path-locator is optional, which the schema-level "exactly one of" rule contradicts.

**Independent Test**: With a stub `spawnFn` injected, calling `read_note({ target_mode: "specific", vault: "MyVault", path: "Templates/Recipe.md" })` against a stub child that exits `0` with stdout `"<template body>"` results in the spawn being invoked with argv `["read", "vault=MyVault", "path=Templates/Recipe.md"]` (vault hoisted, path second) and the handler returning `{ content: "<template body>" }`.

**Acceptance Scenarios**:

1. **Given** the `read_note` tool is registered and a stub adapter that exits `0` with stdout `"<template body>"`, **When** an MCP client calls `read_note({ target_mode: "specific", vault: "MyVault", path: "Templates/Recipe.md" })`, **Then** the tool returns `{ content: "<template body>" }` and the adapter was invoked with argv `["read", "vault=MyVault", "path=Templates/Recipe.md"]`.
2. **Given** the `read_note` tool, **When** an MCP client calls it with a `path` containing forward slashes representing nested directories (e.g., `path: "Project/2026/Q2/notes.md"`), **Then** the adapter is invoked with that exact `path=…` token unchanged — `read_note` does not normalise, resolve, or canonicalise the path. Path syntax is the CLI's responsibility.

### User Story 3 — Active-mode read returns the focused note's text without forwarding any vault/file/path (Priority: P1)

When an agent declares `target_mode: "active"`, the call's intent is "read whatever note is currently open in Obsidian." The schema layer (per the target-mode primitive) forbids `vault`, `file`, and `path` in this mode; the adapter layer (per the cli-adapter contract) strips them defensively even if a downstream tool somehow forwarded them. The handler does not assemble any locator parameters in active mode — it routes `{ target_mode: "active" }` through the adapter, which produces argv `["read"]` (just the subcommand, no key=value tokens).

**Why this priority**: P1. The active-mode contract is the entire reason `target_mode` is a discriminated union — collapsing it to "specific only" defeats ADR-003. The agent uses active mode when it is operating on the user's current focus; making that path work end-to-end (schema → adapter → CLI) is part of this BI's value.

**Independent Test**: With a stub adapter that exits `0` with stdout `"<active note body>"`, calling `read_note({ target_mode: "active" })` results in the adapter being invoked with argv `["read"]` and the handler returning `{ content: "<active note body>" }`. A second test asserts that calling `read_note({ target_mode: "active" })` against a stub adapter that surfaces `ERR_NO_ACTIVE_FILE` (exit `0`, stdout `"Error: no active file\n"`) flows that error through to the MCP error response unchanged.

**Acceptance Scenarios**:

1. **Given** the `read_note` tool and a stub adapter that exits `0` with stdout `"<active note body>"`, **When** an MCP client calls `read_note({ target_mode: "active" })`, **Then** the tool returns `{ content: "<active note body>" }` and the adapter was invoked with argv `["read"]` (no `vault=…`, no `file=…`, no `path=…` tokens).
2. **Given** the `read_note` tool, **When** an MCP client calls `read_note({ target_mode: "active", vault: "V" })` (forbidden key in active mode), **Then** the call fails at the zod parse boundary as `VALIDATION_ERROR` whose `details.issues[].path` includes `"vault"` and whose message contains `"vault"` AND `"active mode"` (or `"target_mode"` paired with `"active"`) per the target-mode primitive's FR-004. The same applies for forbidden `file` and `path`.
3. **Given** the `read_note` tool and a stub adapter that surfaces `UpstreamError({ code: "ERR_NO_ACTIVE_FILE", … })`, **When** an MCP client calls `read_note({ target_mode: "active" })`, **Then** the tool returns an MCP error response (`isError: true`) whose JSON-serialised payload carries `code: "ERR_NO_ACTIVE_FILE"` and the recovery-instruction `.message` from the adapter (per [003-cli-adapter](../003-cli-adapter/spec.md) FR-008(b)). `read_note` does not synthesise its own message; it propagates the adapter's verbatim.

### User Story 4 — Schema layer rejects malformed inputs before any CLI call (Priority: P1)

Per Constitution Principle III, the zod schema is the gatekeeper that catches malformed inputs at the boundary so internal code (the adapter, the CLI) never sees them. `read_note`'s schema composes the target-mode primitive (which enforces "vault required in specific mode" + "exactly one of file/path in specific mode" + "no vault/file/path in active mode") and adds nothing beyond what the primitive carries. Three classes of malformed input MUST fail at the parse step: (a) `target_mode === "specific"` with neither `file` nor `path`; (b) `target_mode === "specific"` with both `file` AND `path`; (c) `target_mode` absent or with a value other than `"specific"` / `"active"`. None of these reach the adapter; the failure is a `VALIDATION_ERROR` with the offending field path.

**Why this priority**: P1. Boundary validation is the constitutional baseline — ADR-003's promise that the schema enforces the intent contract is empty if `read_note` doesn't actually parse against the primitive. Skipping this story would mean every malformed input becomes either an opaque CLI failure (uninformative to the agent) or undefined behavior (the CLI accepts both `file` AND `path` and resolves one in implementation-defined precedence).

**Independent Test**: Four parse-only tests, each driving the handler with a stub adapter that MUST NOT be called: (a) `read_note({ target_mode: "specific", vault: "V" })` → `VALIDATION_ERROR` whose issues mention "exactly one of"; (b) `read_note({ target_mode: "specific", vault: "V", file: "F", path: "P" })` → `VALIDATION_ERROR` whose issues mention "exactly one of"; (c) `read_note({})` → `VALIDATION_ERROR` whose issues identify `target_mode` as missing; (d) `read_note({ target_mode: "specific", file: "F" })` → `VALIDATION_ERROR` whose issues identify `vault` as missing. In all four, the stub spawn function is asserted never to have been called.

**Acceptance Scenarios**:

1. **Given** the `read_note` tool and a stub adapter that records every invocation, **When** an MCP client calls `read_note({ target_mode: "specific", vault: "MyVault" })` (neither `file` nor `path`), **Then** the tool returns an MCP error response with `code: "VALIDATION_ERROR"` whose `details.issues` includes an entry whose `message` mentions `"exactly one of"`, AND the stub adapter was never invoked.
2. **Given** the `read_note` tool, **When** an MCP client calls `read_note({ target_mode: "specific", vault: "MyVault", file: "Foo", path: "Foo.md" })` (both locators), **Then** the tool returns `code: "VALIDATION_ERROR"` whose issues mention `"exactly one of"` (with paths including `"file"` and `"path"` per the primitive's superRefine), AND the stub adapter was never invoked.
3. **Given** the `read_note` tool, **When** an MCP client calls `read_note({})` (empty input — `target_mode` absent), **Then** the tool returns `code: "VALIDATION_ERROR"` whose issues identify `target_mode` as the missing required discriminator, AND the stub adapter was never invoked.
4. **Given** the `read_note` tool, **When** an MCP client calls `read_note({ target_mode: "specific", file: "F" })` (vault missing in specific mode), **Then** the tool returns `code: "VALIDATION_ERROR"` whose `details.issues[].path` includes `"vault"`, AND the stub adapter was never invoked.
5. **Given** the `read_note` tool, **When** an MCP client calls `read_note({ target_mode: "unknown", vault: "V", file: "F" })` (invalid discriminator value), **Then** the tool returns `code: "VALIDATION_ERROR"` whose issues identify `target_mode` as the offending field with an invalid-discriminator-value error, AND the stub adapter was never invoked.

### User Story 5 — CLI failures (non-existent file, non-zero exit, in-band Error) flow through `UpstreamError` (Priority: P1)

When the underlying CLI fails — the named file doesn't exist, the binary is missing, the process crashes — the failure surfaces to the MCP client as an `UpstreamError` carrying the adapter's classified `code`. Per the cli-adapter's four-priority classification, a read of a non-existent file typically surfaces as `CLI_NON_ZERO_EXIT` (the CLI exits non-zero when the locator does not resolve), but the same adapter could also produce `CLI_REPORTED_ERROR` (exit `0` with stdout `Error: …`), `ERR_NO_ACTIVE_FILE` (active mode + no open note), or `CLI_BINARY_NOT_FOUND` (binary not on PATH). `read_note` does not classify; it propagates whatever code the adapter raised.

**Why this priority**: P1. Constitution Principle IV requires every boundary failure to surface as a structured `UpstreamError` with a stable code. The user input's AC #3 names `CLI_NON_ZERO_EXIT` specifically — but the adapter's contract is what binds: the actual code depends on what the CLI does. The spec's job is to require propagation, not to claim a single code for a failure mode the CLI controls.

**Independent Test**: Three handler tests, each driving the adapter through an injected stub: (a) stub child exits `1` with stderr `"file not found: nonexistent_xyz"` → handler returns MCP error with `code: "CLI_NON_ZERO_EXIT"`, the adapter's `details` (command, stdout, stderr, exitCode, signal) preserved on the response; (b) stub child exits `0` with stdout `"Error: File not found\n"` → handler returns `code: "CLI_REPORTED_ERROR"`, `details.message: "Error: File not found"`; (c) stub child fails to spawn with `ENOENT` → handler returns `code: "CLI_BINARY_NOT_FOUND"`. In all cases, the handler does NOT swallow, mask, or rewrite the adapter's error; it routes through `asToolError()` per the existing `obsidian_exec` precedent.

**Acceptance Scenarios**:

1. **Given** the `read_note` tool and a stub adapter that exits `1` with stderr `"file not found"`, **When** an MCP client calls `read_note({ target_mode: "specific", vault: "V", file: "nonexistent_xyz" })`, **Then** the tool returns an MCP error response (`isError: true`) whose JSON-serialised payload has `code: "CLI_NON_ZERO_EXIT"` and `details.exitCode: 1`. The error is constructed via `asToolError()` (the existing helper at [src/tools/_shared.ts](../../src/tools/_shared.ts)), matching the obsidian_exec precedent.
2. **Given** the `read_note` tool and a stub adapter that exits `0` with stdout `"Error: File not found\n"`, **When** an MCP client calls `read_note({ target_mode: "specific", vault: "V", file: "X" })`, **Then** the tool returns an MCP error response with `code: "CLI_REPORTED_ERROR"` and `details.message: "Error: File not found"` — the adapter's classification is propagated unchanged.
3. **Given** the `read_note` tool and a stub adapter whose `spawnFn` raises `ENOENT`, **When** an MCP client calls `read_note({ target_mode: "specific", vault: "V", file: "F" })`, **Then** the tool returns an MCP error response with `code: "CLI_BINARY_NOT_FOUND"`. The error's `details` carries the binary-attempted name and PATH per the adapter's contract.
4. **Given** the `read_note` tool, **When** any non-`UpstreamError` exception escapes from the adapter (an unexpected runtime error), **Then** the handler re-throws it (does NOT wrap it as a tool error) — this matches the obsidian_exec precedent at [src/tools/obsidian_exec/tool.ts:59](../../src/tools/obsidian_exec/tool.ts#L59) where `if (err instanceof UpstreamError) … else throw err;`. Reasonable default — the SDK's outer error envelope handles the unexpected case.

### User Story 6 — Tool registration emits a stripped JSON Schema and a `help`-aware top-level description (Priority: P2)

Per ADR-005 (BI-030), every tool registered with the MCP server passes its `zod-to-json-schema` output through the schema-stripping utility before handing the schema to the SDK, AND its top-level `description` is a concise verb-led summary that mentions `help("<tool_name>")` as the recovery path for full parameter docs. `read_note` MUST satisfy both contracts.

**Why this priority**: P2 because the underlying machinery (the strip utility and the help tool) already exists and is gated by the registry-consistency test introduced in [005-help-tool](../005-help-tool/spec.md) FR-017 / SC-011. This BI's job is to slot `read_note` into that pipeline correctly. P2 not P1 because correctness of the strip and the description is structural — Stories 1–5 are the value contract.

**Independent Test**: Two registration-level tests: (a) a JSON walk over the registered `read_note` tool's `inputSchema` asserts no `description` key at any depth inside the schema's `properties` tree (per the strip utility's contract — Story 1 AC#5 of the help-tool spec generalised to `read_note`); (b) the registered tool's top-level `description` field is non-empty and contains the literal substring `"help"` (case-insensitive) plus a reference to `help("read_note")` (or `help({ tool_name: "read_note" })`) per Story 3 AC#1 of the help-tool spec. The existing registry-consistency test in `src/server.test.ts` automatically asserts the `docs/tools/read_note.md` file exists and is non-stub once this BI replaces the stub body.

**Acceptance Scenarios**:

1. **Given** the MCP server is running with `read_note` registered, **When** an MCP client calls `tools/list`, **Then** the returned descriptor for `read_note` has an `inputSchema` with zero `description` keys at any depth inside the `properties` tree (or any nested `oneOf`/`anyOf`/`items`/`additionalProperties` construct).
2. **Given** the same `tools/list` response, **When** the descriptor for `read_note`'s top-level `description` is inspected, **Then** it is a concise verb-led summary, non-empty, and contains the literal substring `"help"` (case-insensitive) along with a reference to invoking it with the tool's name (e.g., `help("read_note")` or `help({ tool_name: "read_note" })`).
3. **Given** the `help` tool, **When** an MCP client calls `help({ tool_name: "read_note" })`, **Then** it returns the full Markdown body of `docs/tools/read_note.md` — the file no longer carries the `<!-- TODO(BI-003): … -->` stub marker because this BI populates it. The body documents the input schema (target_mode + vault + file/path), the output shape (`content: string`), the propagated error codes (`VALIDATION_ERROR`, `CLI_NON_ZERO_EXIT`, `CLI_REPORTED_ERROR`, `ERR_NO_ACTIVE_FILE`, `CLI_BINARY_NOT_FOUND`), and at least one usage example per branch (`file=`, `path=`, `target_mode: "active"`).

### Edge Cases

- **`read_note({ target_mode: "specific", vault: "", file: "F" })`** (empty-string vault): MUST fail at the zod parse boundary as `VALIDATION_ERROR` whose `details.issues[].path` includes `"vault"`. The target-mode primitive enforces `vault: z.string().min(1)` — an empty string is below the floor. Reasonable default: the error message mirrors zod's standard `min(1)` message; no custom message required.
- **`read_note({ target_mode: "specific", vault: "V", file: "" })`** (empty-string `file`): per the target-mode primitive's `file: z.string().optional()` (no `.min(1)`), an empty string is structurally valid at the schema layer — it counts as "provided" for the "exactly one of" refinement. The CLI is then invoked with `file=` (no value). Per Clarifications 2026-05-06 Q3, this BI explicitly does NOT tighten the primitive's shape at the `read_note` layer: empty-string `file` (and the symmetric empty-string `path`) forwards to the CLI verbatim and surfaces as `CLI_NON_ZERO_EXIT` or `CLI_REPORTED_ERROR` per the cli-adapter's classification. Rationale: a tool-layer `.min(1)` would create a second refinement layer that could drift from the primitive; the existing CLI failure path is already structured. If a future BI adds `.min(1)` it MUST do so at the primitive (BI-029 amendment) so every typed tool inherits.
- **`read_note({ target_mode: "specific", vault: "V", path: "../../../etc/passwd" })`** (path-traversal-shaped value): MUST be forwarded to the CLI verbatim. `read_note` does NOT add path-traversal defense at the tool layer — the CLI is the trust boundary for vault-relative path validation, and the underlying Obsidian binary either rejects out-of-vault paths or restricts I/O to its own scope. This matches the project's existing `obsidian_exec` posture and avoids having two redundant policy layers (the tool's and the CLI's) that could drift.
- **`read_note({ target_mode: "active", file: "Note" })`** (forbidden key in active mode): MUST fail at the zod parse boundary as `VALIDATION_ERROR` whose `details.issues[].path` includes `"file"` and whose message contains `"file"` AND `"active mode"` per the target-mode primitive's FR-004. The same applies for forbidden `vault` and `path`. The error message MUST NOT include recovery instructions per the primitive's clarification 2026-05-06 Q2.
- **`read_note({ target_mode: "active", vault: undefined })`** (forbidden key with explicit-`undefined` value): MUST fail with the same `"vault is not allowed in active mode"` error as the value-bearing case. Per the primitive's edge-cases, presence of the property — regardless of value — counts as "passing" the forbidden key. The check is implemented via `Object.hasOwn(record, key)` in the primitive's `refineActiveBranch`.
- **`read_note({ target_mode: "specific", vault: "V", file: "F", extra: "x" })`** (unknown extra key): MUST succeed at the schema layer. The target-mode primitive's per-branch base schemas use `.passthrough()` (per the primitive's FR-005) so that downstream tool composition works; `read_note` does NOT add `.strict()` on top, because doing so would forbid the very composition the primitive is designed to support. The `extra: "x"` key is then dropped during adapter argv assembly only if the tool's per-branch schema does not surface it as a parameter. Reasonable default: `read_note` does NOT forward unknown keys to the CLI as parameters — only the validated `vault`/`file`/`path` are passed in. (See FR-006 below for the explicit forwarding rule.)
- **CLI returns stdout with leading whitespace before content** (e.g., `"\n  # Title\n…"`): `content` carries the stdout bytes verbatim, including leading whitespace. `read_note` does not trim, normalise, or strip BOMs.
- **CLI returns stdout containing CRLF line endings** (`"# Title\r\n…"`): `content` carries CRLF bytes verbatim. `read_note` does not normalise line endings.
- **CLI returns very large stdout** (e.g., a 50 MB note): the cli-adapter (per [003-cli-adapter](../003-cli-adapter/spec.md) Out of Scope) does NOT cap output size. `read_note` does not impose its own cap. If the CLI's output is bounded by Obsidian itself, that bound applies; otherwise the call returns the full content. Reasonable default — typed-tool output capping is a future enhancement (a follow-up BI), not part of this feature. (See Out of Scope.)
- **Read of a binary file via the markdown vault** (e.g., a `.png` resource that happens to live alongside notes): the CLI's `read` subcommand is the trust boundary. If the CLI emits the binary bytes on stdout, `content` carries them as a UTF-8 string (which may produce mojibake for non-UTF-8 bytes). Reasonable default — `read_note` does not detect or special-case binary content; the CLI is expected to either reject binary reads or transcode. If empirical experience surfaces issues, a future amendment adds either a content-type field on the output or a binary-rejection refinement.
- **`tool_name` argument missing or extra positional arguments**: the MCP SDK's `CallToolRequest` envelope carries a single `arguments` object; positional arguments are not part of the protocol. Outside the spec's concern.

#### Module layout

- **FR-001**: A new typed tool MUST be created at `src/tools/read_note/` following the existing per-surface module convention (per Constitution Principle I and the `src/tools/help/` and `src/tools/obsidian_exec/` precedents). The module MUST contain six files at minimum (three source + three co-located tests): `schema.ts` (the zod input schema, the inferred TypeScript types, and the JSON Schema export via `zod-to-json-schema`), `handler.ts` (the function that calls the adapter and maps the result to the output shape), `tool.ts` (the `registerReadNoteTool` factory returning a `RegisteredTool` per the aggregator pattern), and one co-located test file per source file (`schema.test.ts`, `handler.test.ts`, `tool.test.ts`). Per Constitution Principle V, every new source file MUST carry an `// Original — no upstream. <one-line description>.` header.

#### Schema

- **FR-002**: The input schema MUST compose the `target_mode` primitive (per [004-target-mode-schema](../004-target-mode-schema/spec.md)) using **Composition Pattern (b)** (per-branch divergent extension) per FR-005 of that spec — call `.extend({...})` on each per-branch BASE schema export (`targetModeSpecificBaseSchema`, `targetModeActiveBaseSchema`), wrap each result with the corresponding refinement helper (`applyTargetModeSpecificRefinement`, `applyTargetModeActiveRefinement`), then re-build a `z.discriminatedUnion("target_mode", […])`. Pattern (b) is the constitutionally-correct shape for this tool because the active branch carries no additional fields and the specific branch carries no additional fields beyond what the primitive already enforces — the composed schema's per-branch shape exactly mirrors the primitive, but expressed as a `read_note`-owned schema so the tool has its own typed handle for `z.infer<typeof …>`. The schema MUST satisfy the target-mode primitive's full contract end-to-end: vault required in specific mode, exactly-one-of file/path in specific mode, no vault/file/path in active mode.

  **Reasonable default — pattern selection**: pattern (b) is the *direct* mapping ("`read_note` is exactly the primitive shape, no extra fields"). Pattern (a) (`targetModeSchema.and(z.object({}))`) would also satisfy the contract but is structurally noisier and provides no additional value. Plan-stage MAY revisit if a tool-specific field is added (Out of Scope today).

  **Plan-stage P1 resolution (2026-05-06)**: Literal Pattern (b) — applying `applyTargetModeSpecificRefinement` / `applyTargetModeActiveRefinement` to per-branch BASE schemas, then rebuilding `z.discriminatedUnion(...)` — is structurally infeasible because `z.discriminatedUnion` requires `ZodObject` branches and the refinement helpers return `ZodEffects` (documented at [src/target-mode/target-mode.ts:78-83](../../src/target-mode/target-mode.ts#L78-L83)). For the **zero-extra-fields case (this BI's exact shape)**, re-exporting `targetModeSchema` directly as `readNoteInputSchema` is structurally equivalent to a literal Pattern (b) implementation: the parsed shape, failure modes, and inferred type are identical. Constitution Principle III (single source of truth) is preserved. See [research.md P1](./research.md#p1--schema-composition-tactic-fr-002) for the full rationale + alternatives. Future typed-tool BIs that DO add tool-specific fields (e.g., BI-004 `read_heading`) will need a BI-029 amendment that exports the private refinement bodies (or equivalent dispatcher helper) — that amendment lands with its first concrete consumer, not in this BI.

- **FR-003**: The schema MUST NOT add `.describe()` annotations to any field. Per [004-target-mode-schema](../004-target-mode-schema/spec.md) FR-007, parameter documentation lives in `docs/tools/read_note.md` (per ADR-005's progressive-disclosure design); the schema is annotation-free so the documentation responsibility is single-sourced.

- **FR-004**: The schema MUST export a JSON Schema document via `zod-to-json-schema(...)` (mirroring the existing `obsidianExecInputJsonSchema` and `helpInputJsonSchema` precedents at [src/tools/obsidian_exec/schema.ts](../../src/tools/obsidian_exec/schema.ts) and `src/tools/help/schema.ts`). The exported JSON Schema MUST round-trip through the schema-stripping utility (`stripSchemaDescriptions`, per [005-help-tool](../005-help-tool/spec.md)) at registration time per FR-008 below.

- **FR-005**: The schema MUST export the inferred TypeScript type via `z.infer<typeof readNoteInputSchema>` (or equivalent name — the exact export names are a plan-stage decision, but the inference-not-hand-written rule binds per Constitution Principle III). No hand-written `interface ReadNoteInput` or `type ReadNoteInput = { … }` is permitted.

#### Handler

- **FR-006**: The handler MUST route the validated input through the cli-adapter's `invokeCli` entry point. The adapter call shape is `invokeCli({ command: "read", parameters: <derived>, flags: [], target_mode: <validated> }, deps?)`. The `parameters` record is derived from the validated input as follows:
    - **Specific mode**: `parameters` is `{ vault: parsed.vault, ...(parsed.file !== undefined ? { file: parsed.file } : {}), ...(parsed.path !== undefined ? { path: parsed.path } : {}) }`. The adapter's argv assembler hoists `vault=` to the first key=value position automatically (per cli-adapter FR-005); `read_note` does not need to reorder.
    - **Active mode**: `parameters` is `{}` (empty record). The adapter would strip any forwarded `vault`/`file`/`path` defensively (per cli-adapter FR-003), but the schema layer has already rejected those keys so the handler does not include them.
    - In both modes, `flags` is `[]` — the `read` subcommand takes no flags in this BI's scope. (Future amendments MAY add output-format or range-filter flags; out of scope here per the user input.)

- **FR-007**: The handler MUST return a successful tool-call result whose JSON-serialised payload is `{ content: <stdout> }` — the CLI's stdout (as captured by the adapter, UTF-8) is mapped verbatim to the `content` field. The MCP envelope follows the existing `obsidian_exec` precedent: `{ content: [{ type: "text" as const, text: JSON.stringify({ content: stdout }) }] }`. The handler MUST NOT trim, transform, or post-process the stdout — `content` is the raw bytes the CLI emitted.

- **FR-017**: The handler MUST emit structured log events around the adapter call, parity with `obsidian_exec`'s logger surface (per Clarifications 2026-05-06 Q2). At minimum: a `logger.callStart(...)` event BEFORE invoking the adapter (carrying a fresh `callId`, `command: "read"`, the resolved `vault` if present (else `null`), and `queueDepth` derived from the shared queue); a `logger.callEndSuccess(...)` event on the success path (carrying `callId`, `durationMs`, and `stdoutBytes` = `Buffer.byteLength(content, "utf8")` — the byte count, NOT `content.length` which counts UTF-16 code units and would mis-report for multibyte UTF-8 content; this matches the `obsidian_exec` precedent at [src/tools/obsidian_exec/handler.ts:234](../../src/tools/obsidian_exec/handler.ts#L234)); and a `logger.callEndFailure(...)` event on the failure path (carrying `callId`, `errorCode` from the propagated `UpstreamError.code`, and `durationMs`). The `Logger` type is the existing one at [src/logger.ts](../../src/logger.ts); `RegisterDeps` MUST accept `logger: Logger`, and `src/server.ts` MUST pass the same logger instance to `registerObsidianExecTool`, `registerReadNoteTool`, and (where applicable) any other tool factory in the aggregator list. Reasonable default — exact event payload fields beyond the minimum (e.g., whether to include the input's locator shape `"file"` / `"path"` / `"active"`) are a plan-stage decision; the structural contract (three events per call, callId-correlated, with the named minimum fields) binds.

#### Registration

- **FR-008**: The tool's registration in `tool.ts` MUST follow the existing `RegisteredTool` aggregator pattern (per `src/tools/_shared.ts` and the `obsidian_exec` / `help` precedents):
    - Apply `stripSchemaDescriptions()` to the JSON Schema before constructing the descriptor's `inputSchema` (per [005-help-tool](../005-help-tool/spec.md) FR-006).
    - Catch `ZodError` from `schema.parse(args)` and surface it via `asToolError({ code: "VALIDATION_ERROR", message, details: { issues } })`.
    - Catch `UpstreamError` from the handler and surface it via `asToolError({ code, message, details })` — the adapter's code is propagated unchanged.
    - Re-throw any non-`UpstreamError` exception (mirroring the obsidian_exec precedent at [src/tools/obsidian_exec/tool.ts:59](../../src/tools/obsidian_exec/tool.ts#L59)).

- **FR-009**: The tool's top-level `description` field MUST be a concise verb-led summary that mentions `help("read_note")` (or `help({ tool_name: "read_note" })`) as the recovery path for full parameter docs, per [005-help-tool](../005-help-tool/spec.md) Story 3 / FR-015–FR-016. Reasonable default wording: `'Read a note from an Obsidian vault. Returns the note\'s raw text. Call help({ tool_name: "read_note" }) for full parameter docs.'` — exact wording is a plan-stage decision; the structural contract (verb-led summary + `help` mention with the tool's own name) binds.

- **FR-010**: The tool MUST be registered with the MCP server via `src/server.ts` alongside the existing `obsidian_exec` and `help` tools — i.e., the server's tool-registration list grows by one entry. The registration order is a plan-stage decision; alphabetical (`help`, `obsidian_exec`, `read_note`) or registration-of-introduction order are both acceptable.

- **FR-016**: The handler MUST execute through the same single-in-flight CLI queue that gates `obsidian_exec` (per Clarifications 2026-05-06 Q1). `registerReadNoteTool`'s `RegisterDeps` MUST accept `queue: Queue`, and `src/server.ts` MUST pass the *same* `queue` instance to both `registerObsidianExecTool` and `registerReadNoteTool` so concurrent CLI invocations across both tools serialize through one channel. Rationale: the Obsidian CLI is presumed stateful — it interacts with a live Obsidian app over an IPC channel — and two concurrent invocations on the same channel can race or deadlock; serialization is the existing safety posture and the cost (bounded read throughput) is acceptable for an agent-driven tool. A future BI may relax this if empirical evidence shows reads can run safely in parallel.

### Out of Scope

The following are explicitly excluded from this feature and remain the responsibility of separate BIs or future work (mirroring the user input's "Out of scope" enumeration plus consequential exclusions surfaced during spec drafting):

- **Range-based or partial reads** (e.g., "read lines 100–200 of this note"): out of scope. A future BI (the user input names `read_heading` / BI-004) ships a heading-anchored read.
- **Batch reads** (e.g., reading multiple notes in one tool call): out of scope. Wave 2 BI `batch_read_notes` ships this.
- **Format negotiation** (returning JSON / structured content / a parsed AST): out of scope. The CLI returns raw text; `read_note` returns the same. Per the user input: "The CLI returns raw text; no JSON wrapping."
- **Output capping** (rejecting reads that exceed N MB): out of scope. The cli-adapter does not cap; `read_note` does not cap. If a future need surfaces, it lands as a typed-tool-wide cap policy in its own BI.
- **Path-traversal validation at the tool layer**: out of scope. The CLI is the trust boundary for vault-relative path syntax (per Edge Cases). Adding tool-layer validation would create two redundant policy layers.
- **Read-time mutation of the file** (e.g., format-on-read, link-resolution): out of scope. `read_note` is a pure read; mutations are write_note's territory.
- **Caching `content` across calls**: out of scope. The MCP server is stateless; each call hits the CLI fresh.
- **Adding flags to the `read` subcommand** (e.g., `--copy`, `--no-frontmatter`): out of scope. This BI ships a flagless wrapper. If empirical use surfaces a need, a future amendment adds the flag plumbing.
- **Refactoring the existing `obsidian_exec` tool to use `invokeCli`**: out of scope per [003-cli-adapter](../003-cli-adapter/spec.md) Out of Scope. The two implementations co-exist.
- **Modifying ADR-003, ADR-004, or ADR-005**: this feature implements all three; it does not change them. Conflicts surface as new ADRs per the project's amendment procedure.

## Assumptions

- **The user input's `(optional)` annotation on `vault`** is reconciled by reference to [004-target-mode-schema](../004-target-mode-schema/spec.md) FR-003, which the user input separately commits to ("MUST inherit the Target Mode discriminated union from BI-029"). The primitive enforces `vault: z.string().min(1)` (required) on the specific branch — that is the binding contract. Specific mode without vault MUST fail at the parse boundary (Story 4 AC#4). The user input's `(optional)` is interpreted as "the field is part of the input shape but only inhabited by specific mode," not as "the schema accepts specific mode without vault." The reconciliation is documented here rather than treated as a clarification because the constitution commits the spec to the primitive's contract whenever the two diverge in wording.
- **The user input's `ObsidianCLIAdapter.execute("read", args, target_mode)` shape** is a sketch of an adapter that does not exist; the actually-shipped adapter at [src/cli-adapter/cli-adapter.ts](../../src/cli-adapter/cli-adapter.ts) exposes `invokeCli(input, deps?)` per [003-cli-adapter](../003-cli-adapter/spec.md) FR-002. The handler routes through the actually-shipped API. Semantic intent (one centralised CLI invocation primitive) is preserved.
- **The `read` subcommand of the Obsidian CLI** accepts the argv shape `["read", "vault=…", "file=…"]` (or `path=…`) and emits the note's raw text on stdout when the locator resolves; on failure, it either exits non-zero with a useful stderr or exits 0 with `Error: …` prefix on stdout. This is the precondition the tool is built on; if the CLI's `read` subcommand has a different invocation shape, the user input's "direct one-to-one wrap" claim is empty and the spec's argv assembly (FR-006) re-targets to the actual shape. Reasonable default per the existing project pattern: argv assembly is `[command, vault=…, file=… | path=…]`. Plan-stage MAY confirm against the CLI's documentation.
- **The CLI's `read` subcommand returns UTF-8 text** for markdown notes. The handler does not transcode. Non-UTF-8 content (e.g., from a corrupted file) is surfaced verbatim as a UTF-8 string — which may produce mojibake. Per Edge Cases, this is the CLI's domain; if it surfaces as a real problem, a future BI adds detection.
- **The active-mode CLI invocation** (`obsidian read` with no vault/file/path) reads the focused note in the focused vault. If no vault is focused, the CLI exits 0 with stdout `Error: no active file` (or substantively equivalent), which the adapter classifies as `ERR_NO_ACTIVE_FILE` per [003-cli-adapter](../003-cli-adapter/spec.md) FR-008(b). This is an empirical observation from the BI-026 (002-detect-cli-errors) reproductions, not a verified-against-current-CLI fact for this BI.
- **The schema-stripping utility from [005-help-tool](../005-help-tool/spec.md) is applied at registration time** per FR-008. The strip is mandatory; bypassing it would leave `read_note`'s `inputSchema` carrying the per-field `description` annotations from the target-mode primitive's base schemas — except the primitive ships annotation-free per its FR-007, so the strip is a no-op for `read_note`'s base schema. This BI nevertheless applies the strip for consistency with the registration pattern; a tool that skips the strip would silently break ADR-005's contract for any future amendment that adds annotations.
- **The cli-adapter's `deps.spawnFn` test seam** (per [003-cli-adapter](../003-cli-adapter/spec.md) FR-002) is the canonical test-injection mechanism. `read_note`'s handler tests inject `deps.spawnFn` via the adapter call, NOT by mocking the adapter module itself. This matches the existing `obsidian_exec` and adapter test patterns and keeps the test surface aligned with the runtime composition.
- **No new error codes** are introduced by this BI. Per Constitution Principle IV, error codes are the project's stable API; adding one without justification creates noise. `read_note`'s entire failure surface is already covered by `VALIDATION_ERROR` (zod), `CLI_NON_ZERO_EXIT` / `CLI_REPORTED_ERROR` / `ERR_NO_ACTIVE_FILE` / `CLI_BINARY_NOT_FOUND` (adapter). - **The first typed-tool BI that lands on top of the adapter + target-mode + help-tool foundations** is this BI; it is the empirical validator of all three. If implementation surfaces a real-world need not anticipated in this spec or in the foundation specs, the resolution is an amendment to whichever spec carries the gap, not a silent retrofit in `read_note`.
