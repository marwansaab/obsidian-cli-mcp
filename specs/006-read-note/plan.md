# Implementation Plan: Read Note Typed MCP Tool

**Branch**: `006-read-note` | **Date**: 2026-05-06 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from [specs/006-read-note/spec.md](./spec.md)

## Summary

Implement [BI-003 — Add Read Note](../../.architecture/Obsidian%20CLI%20MCP%20-%20Architecture.md) by shipping the first typed-tool MCP surface composed on top of the three foundation features that landed before it (the target-mode primitive from BI-029, the cli-adapter from BI-028, and the help tool's schema-stripping utility from BI-030).

The deliverable is a new per-surface module at [src/tools/read_note/](../../src/tools/read_note/) with the project's standard layout: `schema.ts` (the zod input schema + the `zod-to-json-schema` JSON Schema export + the inferred `ReadNoteInput` type), `handler.ts` (the function that calls `invokeCli` and maps the adapter's stdout to `{ content: <stdout> }`, bracketed by `logger.callStart` / `logger.callEndSuccess` / `logger.callEndFailure` events per FR-017), and `tool.ts` (the `registerReadNoteTool` factory returning a `RegisteredTool` per the aggregator pattern at [src/tools/_shared.ts](../../src/tools/_shared.ts)). All four files carry `// Original — no upstream.` headers per Constitution Principle V; co-located `*.test.ts` files cover happy + failure + boundary paths per Principle II and FR-013.

The handler's only path to the CLI is through `invokeCli({ command: "read", parameters, flags: [], target_mode }, deps?)` from [src/cli-adapter/cli-adapter.ts](../../src/cli-adapter/cli-adapter.ts) — there is zero direct `child_process.spawn` use in the read_note module (SC-003). Per Clarifications 2026-05-06 Q1 (FR-016), the handler routes through `deps.queue.run(...)` so reads serialize behind every other in-flight CLI call across the server (the same `Queue` instance powers `obsidian_exec` today; [src/server.ts](../../src/server.ts) passes one queue to both factories). Per Clarifications 2026-05-06 Q2 (FR-017), the handler accepts a `Logger` and emits the three structured events documented in the spec — observability parity with `obsidian_exec`.

The schema layer composes the target-mode primitive. Because read_note adds zero tool-specific fields beyond what the primitive carries, the constitutionally-correct shape (per [src/target-mode/target-mode.ts:78-83](../../src/target-mode/target-mode.ts#L78-L83) — z.discriminatedUnion requires ZodObject branches, the refinement helpers return ZodEffects) is to **re-export `targetModeSchema` as `readNoteInputSchema`** rather than rebuild the union with `.extend({})` no-ops. The plan's P1 below (Phase 0) documents this resolution against the spec's FR-002 wording, which mandates Pattern (b) but is structurally infeasible for the zero-extra-field case without reaching into the primitive's private refinement bodies. Re-exporting keeps Constitution Principle III's "single source of truth" intact and produces a schema structurally identical to a hand-wired Pattern (b) dispatcher.

The tool registers in [src/server.ts](../../src/server.ts) alongside `obsidian_exec` and `help`, with both `logger` and `queue` (FR-016 + FR-017) threaded through `registerReadNoteTool({ logger, queue })` exactly as `registerObsidianExecTool` is wired today. Per Clarifications 2026-05-06 Q3, the schema does NOT add `.min(1)` to `file` or `path` — empty-string locators forward to the CLI verbatim and surface as `CLI_NON_ZERO_EXIT` / `CLI_REPORTED_ERROR` per the cli-adapter classification.

The existing stub at [docs/tools/read_note.md](../../docs/tools/read_note.md) (`<!-- TODO(BI-003): … -->` per BI-030's FR-012) is replaced with a populated body covering input schema, output shape, error codes, and one usage example per branch (FR-011). The registry-consistency test at [src/server.test.ts](../../src/server.test.ts) (per BI-030 FR-017 / SC-011) automatically picks up the new tool — it asserts that `docs/tools/read_note.md` exists once `read_note` is registered. No new error codes are introduced (the entire failure surface is covered by `VALIDATION_ERROR` + the four adapter codes); the canonical errors contract at [specs/001-add-cli-bridge/contracts/errors.contract.md](../001-add-cli-bridge/contracts/errors.contract.md) is unchanged by this BI.

Constitution Principles I (per-surface module under `src/tools/read_note/` with downward imports), II (co-located `*.test.ts` covering every public surface), III (zod schema is the single source of truth via re-export of the primitive; types via `z.infer`), IV (every failure surface flows through `UpstreamError` and `asToolError`; zero new codes), and V (original-contribution headers on every new source file) all bind `Y`. The plan resolves seven plan-stage deferrals from the spec (schema composition tactic, top-level description wording, server registration order, log-event payload extras, doc body structure, test-injection pattern, and a small additive to the primitive that may be needed by future typed-tool BIs but is *not* needed here) in the Phase 0 Research section below.

**Plan-stage decisions resolved during this Phase 0** (full rationale + alternatives in [research.md](./research.md)):

- **P1 (FR-002, schema composition tactic)**: Re-export `targetModeSchema` from `src/target-mode/target-mode.ts` as `readNoteInputSchema`. The spec's FR-002 mandates Pattern (b) (per-branch `.extend({...})` + refinement-helper wrap + re-built discriminated union); however, [src/target-mode/target-mode.ts:78-83](../../src/target-mode/target-mode.ts#L78-L83) documents that `z.discriminatedUnion` requires `ZodObject` branches and the helpers (`applyTargetModeSpecificRefinement`, `applyTargetModeActiveRefinement`) return `ZodEffects` — making literal Pattern (b) infeasible. The alternative — extending the bases with `{}` and writing a union-level superRefine that dispatches to `refineSpecificBranch` / `refineActiveBranch` — requires those private bodies to be exported from the primitive, which is a BI-029 amendment. Since `read_note` adds zero extra fields, the compose-then-refine output is structurally identical to the primitive itself; re-export delivers the same observable behaviour with zero structural divergence. The spec's "tool has its own typed handle for `z.infer<typeof …>`" requirement is satisfied because `readNoteInputSchema` is a local export with a local inferred type alias `ReadNoteInput` — downstream code uses `import type { ReadNoteInput } from "./schema.js"`. **Documented FR-002 deviation**: a reviewer reading FR-002 literally will see the spec mandates Pattern (b) but the implementation re-exports; this plan is the authority for that resolution. Future typed-tool BIs (e.g., BI-004 `read_heading`) that DO add tool-specific fields will need either (a) a BI-029 amendment exporting the refinement bodies or a public dispatcher helper, or (b) a one-off duplication of the dispatcher pattern — that decision lands in the *next* tool's plan, not here.

- **P2 (FR-009, top-level description wording)**: The pinned string for `READ_NOTE_DESCRIPTION` is:
  ```
  Read a note from an Obsidian vault. Returns the note's raw UTF-8 text as { content: <stdout> }. Specific mode: vault + exactly one of file (wikilink) or path (vault-relative). Active mode: no locator — reads the focused note. Call help({ tool_name: "read_note" }) for full parameter docs and the error-code roster.
  ```
  Verb-led ("Read"), states the input shape and output, mentions both branches, and references `help({ tool_name: "read_note" })` per FR-009 + BI-030 FR-015. ~270 chars — comparable to `obsidian_exec`'s post-BI-030 description (~340 chars). Alternative wordings considered: shorter ("Read a note. Call help(\"read_note\") for docs.") rejected as too lossy — agents need to know about the two branches to construct a valid call without round-tripping through `help`; longer (mentioning specific error codes) rejected as duplicating `docs/tools/read_note.md` content per BI-030 ADR.

- **P3 (FR-010, server.ts registration order)**: Alphabetical by tool name — `help`, `obsidian_exec`, `read_note`. Matches the alphabetical ordering precedent set by [src/server.ts:49-52](../../src/server.ts#L49-L52) (where `obsidian_exec` and `help` are currently registered) — though the existing order is `obsidian_exec` before `help`, registration-of-introduction. Reordering to alphabetical is a one-line change inside this BI's `src/server.ts` edit; the choice keeps `tools/list` output deterministic and grep-discoverable. Alternative considered: keep registration-of-introduction order (`obsidian_exec`, `help`, `read_note`) — equally valid; rejected because alphabetical scales better as the typed-tool series grows (BI-004 `read_heading`, BI-005 `write_note`, etc.). Either order satisfies FR-010; the choice is purely cosmetic.

- **P4 (FR-017, log-event payload extras beyond the named minimum)**: The three events carry exactly the FR-017-named fields plus one ergonomic extra on `callStart` to aid debugging — the locator shape (`"file"` / `"path"` / `"active"`). Specifically:
    - `logger.callStart({ callId, command: "read", vault: parsed.target_mode === "specific" ? parsed.vault : null, queueDepth, locator })` where `locator` is `"file"` / `"path"` / `"active"` derived from the parsed input. Adding `argv` is rejected because the cli-adapter assembles argv internally and the read_note handler never has it; emitting an empty/dummy `argv` would mislead.
    - `logger.callEndSuccess({ callId, durationMs, stdoutBytes })` where `stdoutBytes` is `Buffer.byteLength(content, "utf8")` (NOT `content.length` — the latter counts UTF-16 code units, which mismatches the obsidian_exec precedent at [handler.ts:234](../../src/tools/obsidian_exec/handler.ts#L234) that counts the byte buffer length). The `stderrBytes` field is omitted since the adapter discards stderr on the success path for read calls.
    - `logger.callEndFailure({ callId, errorCode, durationMs })` where `errorCode` is the propagated `UpstreamError.code`. Additional fields (`exitCode`, `signal`) from the obsidian_exec failure-event shape are omitted on read_note's failure events because the read_note handler does not have direct access to them — the adapter wraps them inside `details`. Future plan amendment may surface them via the queue's plumbing if observed-need surfaces.

- **P5 (FR-011, docs/tools/read_note.md body structure)**: The Markdown body follows this section ordering, mirroring the structure pattern set by `obsidian_exec.md` (the post-BI-030 non-stub doc) for cross-tool consistency — precise wording is left to the implementation step, but the section list binds:
    1. **Overview** — one-paragraph summary; opens with "Read a note's raw text from an Obsidian vault" and names the two target modes.
    2. **Input Schema** — input fields enumerated by branch: specific (`target_mode: "specific"`, `vault`, exactly one of `file`/`path`) and active (`target_mode: "active"`, no other keys). Cross-link to [target-mode primitive](../../specs/004-target-mode-schema/spec.md) for the discriminator's full contract.
    3. **Output** — `{ content: string }` — UTF-8 raw text, verbatim from CLI stdout, no transformation.
    4. **Errors** — table of propagated codes: `VALIDATION_ERROR`, `CLI_NON_ZERO_EXIT`, `CLI_REPORTED_ERROR`, `ERR_NO_ACTIVE_FILE`, `CLI_BINARY_NOT_FOUND`. Cross-link to [errors contract](../../specs/001-add-cli-bridge/contracts/errors.contract.md).
    5. **Examples** — three code blocks showing one example per branch: `read_note({ target_mode: "specific", vault: "MyVault", file: "Recipe" })`, `read_note({ ..., path: "Templates/Recipe.md" })`, `read_note({ target_mode: "active" })`.
    6. **References** — bullet list with links to [the cli-adapter spec](../../specs/003-cli-adapter/spec.md), [target-mode primitive spec](../../specs/004-target-mode-schema/spec.md), and [help tool spec](../../specs/005-help-tool/spec.md).
  No `// Original — no upstream.` header (Markdown is exempt per BI-030 FR-019). No `<!-- TODO(BI-003): … -->` stub marker (the entire point of this BI's doc work).

- **P6 (FR-013, test-injection pattern)**: Tests inject the stub adapter via `deps.spawnFn` per the cli-adapter's existing test-seam convention. The `RegisterDeps` shape passed to `registerReadNoteTool` exposes `spawnFn?: SpawnLike` and `env?: NodeJS.ProcessEnv` as optional pass-through fields (not consumed by registration directly — they flow into the handler's `ExecuteDeps` and through `invokeCli`). For schema tests, no deps are needed (the schema is imported and parsed directly). For handler tests, a minimal `deps` object is constructed in each test body: `{ logger: stubLogger, queue: createQueue(), spawnFn: stubSpawn }` — the `Queue` is a real one (it's lightweight and queueing behaviour isn't under test here; per FR-016 / Q1 queue *sharing* is the integration concern, not queueing semantics). The `stubLogger` is a plain object with three captured methods (`callStart`, `callEndSuccess`, `callEndFailure`) — no production `Logger` instance needed. For tool-registration tests, the same minimal deps shape is passed to `registerReadNoteTool`. **No mock of the `cli-adapter` module itself** — tests exercise the real adapter with a stub `spawnFn`, matching the existing `obsidian_exec` test pattern at [src/tools/obsidian_exec/handler.test.ts](../../src/tools/obsidian_exec/handler.test.ts). This is the canonical seam per the cli-adapter contract from BI-028.

- **P7 (FR-013 (e), TODO-marker absence test mechanism)**: The TODO-marker absence assertion lives in `src/tools/read_note/tool.test.ts` (NOT in the registry-consistency block at `src/server.test.ts`). Rationale: (a) the registry-consistency block is generic across all tools and asserts file existence + schema strip, not per-tool content; (b) the TODO-marker check is read_note-specific (no other tool's stub-removal status is part of this BI's scope); (c) co-locating it with the other read_note registration tests keeps the failure mode discoverable from the read_note module's vantage. Implementation: `expect(readFileSync("docs/tools/read_note.md", "utf8")).not.toContain("<!-- TODO(BI-003)")`. Path resolution from `import.meta.url` mirrors the help tool's path-resolution pattern (see [src/tools/help/handler.ts](../../src/tools/help/handler.ts)); the test does NOT use `process.cwd()` (which would couple it to the runner's working directory). Alternative considered: assert in `server.test.ts` that no doc file under `docs/tools/` contains a TODO marker for any registered tool — rejected because it would couple this BI's test to all other tools' stub-removal status, creating spurious failures when (e.g.) BI-005 lands the `write_note` doc.

- **P8 (Reasonable default — primitive amendment NOT needed for this BI)**: Future typed-tool BIs that add tool-specific fields beyond the primitive's shape (e.g., BI-004 `read_heading` with a `heading: string` field) WILL need either (a) a BI-029 amendment exporting `refineSpecificBranch` / `refineActiveBranch` so consumers can hand-wire the union-level superRefine dispatcher Pattern (b) requires, OR (b) the primitive exporting a public dispatcher helper that takes a discriminated-union builder and returns a refined union. **Neither lands in THIS BI.** Read_note's zero-extra-fields case is solved by P1 (re-export). The primitive amendment is documented here as a known follow-up so future BIs don't waste plan time re-discovering it; the actual amendment lands in the BI that first needs it. Alternative considered: prophylactically amend BI-029 in this BI — rejected because YAGNI and because amending a foundation feature in a downstream BI's plan inverts the dependency direction.

**ADR-003 / ADR-004 / ADR-005 alignment**: This feature implements all three ADRs simultaneously — ADR-003 (`target_mode` discriminated union) via P1's re-export, ADR-004 (centralised cli-adapter) via the handler's exclusive use of `invokeCli` (SC-003), and ADR-005 (token-optimized progressive disclosure) via P2's caveman description + the FR-008 schema-strip + the FR-011 `docs/tools/read_note.md` body. No ADR amendment, no new ADR. The [Architecture document](../../.architecture/Obsidian%20CLI%20MCP%20-%20Architecture.md) names BI-003 as this implementation; this feature IS BI-003.
