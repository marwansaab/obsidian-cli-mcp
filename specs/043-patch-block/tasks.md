# Tasks: Patch Block

**Input**: Design documents from `/specs/043-patch-block/` — [spec.md](spec.md), [plan.md](plan.md), [research.md](research.md), [data-model.md](data-model.md), [contracts/](contracts/), [quickstart.md](quickstart.md).
**Prerequisites**: plan.md ✓, spec.md ✓, research.md ✓, data-model.md ✓, contracts/ ✓.

**Tests**: Co-located `*.test.ts` unit tests are MANDATORY per Constitution Principle II (every MCP tool ships with happy-path + failure-or-boundary tests in the same change that adds it). Tests are NOT optional. T0 live-vault scenarios from [research.md §R11](research.md) run at `/speckit-implement` time per CLAUDE.md `## Test Execution`.

**Organisation**: Tasks are grouped by user story. Setup + Foundational (Phase 1 + 2) build the shared Zod schemas, the pure helpers (`block-scan.ts`, `block-edit.ts`), and the module scaffold. User stories then layer the handler logic incrementally — US1 ships the happy-path MVP for `target_mode: "specific"` (all three block shapes, fs read-modify-write, EXTERNAL_EDITOR_CONFLICT classification folded in, registration); US2 layers in the diagnostic-error coverage (`BLOCK_NOT_FOUND`, `BLOCK_ON_HEADING`, `NOTE_NOT_FOUND`); US3 adds active-mode focused-file resolution.

**Cohort divergences from sibling BI-040 (patch_heading)** worth noting upfront:
- Single placement mode (`replace`) — no append/prepend tasks.
- No race-detection task — FR-026 publishes last-write-wins as the cross-invocation contract; no `BLOCK_RACE` discriminator.
- No P4 durability user story — `EXTERNAL_EDITOR_CONFLICT` classification (FR-021) is folded into US1's atomic-write task (T014) rather than split into a separate phase, because the spec admits no user story carrying it.
- Five-shape block scanner (paragraph / list-item / separately-placed / on-heading-atx / on-heading-setext) — broader classification surface than sibling's ATX-only walker because `BLOCK_ON_HEADING` routing requires recognising both heading shapes.

## Phase 3: User Story 1 - Surgical replace tied to a named block-reference (Priority: P1) 🎯 MVP

**Goal**: Ship the happy-path MVP. Caller invokes the tool with `target_mode: "specific"` + `vault` + `file`/`path` + `block_id` + `content` and the wrapper performs the fs read-modify-write through the ADR-009 substrate. All three block shapes (paragraph, list-item, separately-placed) land correctly via the dispatched surgery; line-ending and trailing-newline conventions are preserved; the success envelope echoes the locator + the matched block shape. Includes registration through the boot spine so the tool is callable from MCP clients, AND the `EXTERNAL_EDITOR_CONFLICT` classification (FR-021) folded into the atomic-write task — cohort divergence from sibling BI-040 which split this into a separate P4 phase.

**Independent Test**: Run [quickstart.md](quickstart.md) examples 1–4 and 10 against the authorised test vault — replace a paragraph by block-id, replace a list-item by block-id, replace a separately-placed-marker block (table), clear a paragraph's body via empty content, and (on Windows) verify the external-editor-conflict path surfaces `EXTERNAL_EDITOR_CONFLICT` correctly. Verify each response shape matches the envelope contract exactly AND the post-edit file's bytes outside the targeted block (and the `^block-id` marker itself) are unchanged from the pre-call state.

This MVP includes the full happy-path fs read-modify-write for `target_mode: "specific"`, the `EXTERNAL_EDITOR_CONFLICT` classification on the fs error catch site, the registration tail (factory + server wire + baseline update), and the metadataCache invalidation eval per ADR-009.
