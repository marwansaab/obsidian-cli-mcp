# Feature Specification: Patch Block

**Feature Branch**: `043-patch-block`
**Created**: 2026-05-25
**Status**: Draft
**Input**: User description: "Add Patch Block — a typed capability that lets a vault author surgically replace the body tied to a specific `^block-id` block-reference inside a note, leaving the block-reference marker itself and the rest of the note untouched. Block-references are the only Obsidian anchor that survives heading renames, list-item reordering, and table edits — heading paths and line numbers are too fragile to rely on. Closes the surgical-write gap that today forces callers to rewrite the whole note (thousands of tokens per round-trip, risk of unrelated edits) when a single paragraph, list item, or table row needs updating. Single placement mode (replace). Block-id locator only. Targets either an explicit note (vault + vault-relative path) or the note currently focused in the user's editor. Fails loud on unresolvable block-ids, missing notes, missing focused notes, and structurally invalid block-id input. Out of scope: heading-body patches (Patch Heading), frontmatter writes (Set Property), full-note rewrites (Write Note), creating new `^block-id` markers (markers are emitted by Write Note / Patch Heading when writing surrounding content), block markers inside fenced code blocks (treated as content, not anchor structure), transactional guarantees across concurrent external writes (last-write-wins beyond a single invocation)."

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Surgical replace tied to a named block-reference (Priority: P1)

A vault author issues a request naming a note (either explicitly via vault + vault-relative path, or implicitly via the focused-note locator), the bare block-id of an existing `^block-id` marker inside that note, and the replacement content. The wrapper rewrites only the block attached to that marker — paragraph text, list-item line, table, callout, or other block-shaped element — leaves the `^block-id` marker in its original position relative to the block, and leaves every other byte in the note unchanged. The response identifies the patched note's vault-relative location and the matched block-id so the caller can attribute the change or roll it back.

**Why this priority**: The whole feature exists to give authors a surgical write surface anchored to block-references — the only Obsidian anchor that survives heading renames, list-item reordering, and table edits. Without P1 there is no MVP. The three block shapes (paragraph, list item, table-or-other) cover the realistic authoring intent; the marker-preservation invariant is what makes the operation usable inside a vault where other notes link to `[[note#^block-id]]`. P1 stands alone as a viable shipped slice: an author who can replace a paragraph, a list item, or a tabular block by block-id already has every authoring shape this feature aims to deliver, plus the response payload needed to track or undo the change.

**Independent Test**: Stand up a vault with a note containing at least three `^block-id` markers — one attached to a paragraph, one to a list item inside a list with sibling items, one to a table or other block-shaped element. Issue the operation against each marker with replacement content known by inspection. Verify that (a) the targeted block's body now carries the new content, (b) the `^block-id` marker remains attached to the same block at its conventional position for that block shape, (c) every other heading, list item, paragraph, frontmatter field, and trailing-newline byte in the note is identical to the pre-call state, and (d) the response payload names the vault-relative note path and the matched block-id.

**Acceptance Scenarios**:

1. **Given** a note that contains the block-reference `^foo` attached to a paragraph, **When** the caller invokes the tool with that note, the block-id `foo`, and replacement content, **Then** the paragraph attached to `^foo` carries the new content, the `^foo` marker remains in place, and no other byte of the note is altered.
2. **Given** a note where `^foo` is attached to a list item inside a list with sibling items, **When** the caller invokes the tool with that note, the block-id `foo`, and replacement content, **Then** only that list item's content is replaced, the `^foo` marker stays attached to it, list indentation is preserved, and surrounding list items and headings are untouched.
3. **Given** a note where `^foo` is attached to a table or other block-shaped element (callout, blockquote, indented code), **When** the caller invokes the tool with that note, the block-id `foo`, and replacement content, **Then** that block's content is replaced, the `^foo` marker remains at its conventional position relative to the block (typically the line immediately following the block), and surrounding content is unchanged.
4. **Given** a successful patch in any of the above shapes, **When** the call completes, **Then** the response identifies the patched note's vault-relative location and the block-id that matched, so the caller can attribute the change or roll it back.

### User Story 2 - Distinguish unresolvable-target failures through typed errors (Priority: P2)

A vault author who supplies a block-id that does not occur in the target note, a note path that does not correspond to an existing note in the vault, or a structurally malformed block-id input, receives a typed, structured error identifying which of the three failure modes occurred. The file on disk is not modified; no note is created. The caller can switch on the failure mode directly — no prose parsing.

**Why this priority**: P1 gives the happy path. Without P2, an author whose block-id is one character off (or whose note path is wrong, or whose block-id input is structurally invalid) has no way to learn which problem they hit other than reading the note and the inputs side-by-side. Worse, an undetected note-not-found could be misread as a silent success and a future caller could re-read the would-have-been-patched location and find stale content. P2 is the diagnostic contract that makes failure self-describing — "block-reference not found", "note not found", and "block-id malformed" are three programmatically distinguishable states. With P1+P2 the caller has both the operation and the feedback loop needed to use it reliably.

**Independent Test**: With P1 in place, issue the operation naming a block-id that does not occur in the target note and verify the response is a typed error programmatically distinguishable as "block-reference not found", that the error payload names the missing block-id, and that the file on disk is byte-identical to its pre-call state. Issue the operation against a note path that does not exist in the vault and verify the response is a typed error programmatically distinguishable as "note not found" and that no note is created. Issue the operation with a structurally invalid block-id (empty string, leading `^`, characters outside the supported alphabet, exceeding the input-length cap) and verify the response is a typed error programmatically distinguishable as "block-id malformed" and that no filesystem access occurred.

**Acceptance Scenarios**:

1. **Given** a note that does not contain the named block-reference, **When** the caller invokes the tool, **Then** the operation fails with a typed error programmatically identifying the failure as "block-reference not found", the error payload names the missing block-id, and the note's content remains exactly as it was.
2. **Given** a note path that does not correspond to an existing note in the vault, **When** the caller invokes the tool, **Then** the operation fails with a typed error programmatically identifying the failure as "note not found" and no note is created.
3. **Given** a structurally invalid block-id input (empty, contains characters outside the supported alphabet, leading `^`, or exceeds the input-length cap), **When** the caller invokes the tool, **Then** the request is rejected at the input-validation boundary before any filesystem access or subprocess invocation, and the typed error names the malformed input and the reason.
4. **Given** the named block-reference is attached to a heading line (`# Heading ^foo`), **When** the caller invokes the tool with block-id `foo`, **Then** the operation fails with a typed error naming the block-id and directing the caller to the Patch Heading capability rather than rewriting the heading line.

### User Story 3 - Operate on the focused note (Priority: P3)

A vault author working with a note focused in their editor patches a block-reference inside that note without restating its location. The edit lands on the focused note's matching block. When no note is focused, the operation fails with a typed error directing the author to focus a note or to supply an explicit note locator.

**Why this priority**: P1 already accepts an explicit note locator (vault + vault-relative path), so the feature is shippable without P3. P3 layers a convenience surface on top: an author who is already working in a note does not need to restate its path on every patch. This is cohort parity with the active-mode surface across the rest of the typed-tool family — every typed tool that accepts a note locator also accepts a focused-note variant. P3 stands independently — it can be added after P1+P2 without disturbing them, and P1 callers who always pass an explicit locator are unaffected.

**Independent Test**: With P1+P2 in place, focus a known note in the user's editor (the test fixture writes the focused-note state directly), issue the operation using the focused-note locator (no explicit vault/path supplied), and verify the edit lands on the focused note's matching block. Clear the focused-note state and issue the same request, and verify the operation fails with a typed error directing the author to focus a note or to supply an explicit locator — and that no filesystem access occurred.

**Acceptance Scenarios**:

1. **Given** a note is focused in the author's editor and contains the named block-reference, **When** the caller submits a focused-note patch request, **Then** the edit is applied to that focused note's matching block in the same way as when the note is named explicitly.
2. **Given** no note is focused, **When** the caller submits a focused-note patch request, **Then** the operation fails with a typed error directing the caller to focus a note or to supply an explicit locator; no filesystem access is performed and no write occurs.

### Edge Cases

- **Block-reference marker inside a fenced code block**: a `^block-id` token appearing inside a `` ``` `` fence is content, not an anchor — it is NOT an eligible target. A request naming that block-id fails as "block-reference not found" rather than rewriting code-fence content.
- **Block-reference marker inside an indented code block or HTML comment**: same rule as fenced code — opaque content, not anchor structure.
- **Same block-id repeated inside the same note**: a note authored with two `^foo` markers is structurally malformed per Obsidian's contract (wikilinks `[[note#^foo]]` resolve ambiguously). The wrapper's behaviour follows the underlying parser: whichever block the parser binds the id to is the one patched. This is acknowledged as a pre-existing authoring bug surfaced — not introduced — by this tool.
- **Block-id present on a heading line** (`# Heading ^foo`): the block-id resolves to the heading's marker line, not to a paragraph or list-item body. The wrapper fails loud with a typed "block-on-heading" error directing the caller to Patch Heading rather than (a) rewriting the heading text (which would break wikilinks to `[[note#Heading]]` elsewhere in the vault) or (b) duplicating Patch Heading's replace-section-reach contract under a second locator surface. Cohort parity: the published scope split between Patch Heading (heading bodies) and Patch Block (non-heading block bodies) is preserved.
- **Replacement content is empty (zero bytes)**: accepted as the legitimate "clear the body" operation — the block's content becomes empty, the `^block-id` marker is preserved at its conventional position for the block shape. Cohort parity with Patch Heading's `replace` mode accepting empty content.
- **Replacement content contains newlines for a single-line block shape (list item, table row)**: accepted at the input-validation boundary; whether the resulting markdown remains a structurally valid list item or table row is the caller's responsibility. The wrapper does not invent shape-specific structural validation.
- **Note with carriage-return line endings (CRLF)**: the operation preserves the note's existing line-ending convention.
- **Note without a trailing newline**: the operation preserves the absent trailing newline — a note that did not end with `\n` still does not afterwards; a note that did, still does.
- **Block-reference marker with surrounding whitespace variations** (`text ^foo`, `text  ^foo`, marker on its own line after a table): the marker convention per block shape is whatever the underlying Obsidian parser recognises; the wrapper preserves the marker bytes and their position relative to the block.
- **Block-id input supplied with a leading `^`** (e.g. `^foo` instead of `foo`): rejected at the input-validation boundary with a typed "leading-caret" error rather than silently stripping. The caret is the delimiter in `[[note#^foo]]` wikilink syntax, not part of the identifier; accepting both forms invites ambiguity in error messages and round-trip identifiers.

## Requirements *(mandatory)*

### Functional Requirements

#### Locator and addressing

- **FR-001**: System MUST accept a block-id locator addressing a single existing `^block-id` marker inside the target note. The block-id input is the bare identifier — the leading `^` character is the wikilink delimiter, not part of the identifier, and MUST NOT appear in the input.
- **FR-002**: System MUST match the block-id against the marker text exactly — case-sensitive, no whitespace trimming on the segment itself, no fuzzy matching.
- **FR-003**: System MUST treat block-id matching as case-sensitive throughout (consistent with Obsidian's own treatment).
- **FR-004**: System MUST validate the block-id against the supported alphabet at the input-validation boundary: alphanumeric characters plus the hyphen-minus (`-`) and underscore (`_`) characters, length within the cohort input-length cap (1000 UTF-16 code units, per cohort parity with BI-033, BI-038, BI-039, BI-040).
- **FR-005**: System MUST accept either an explicit note locator (vault + vault-relative path) or a focused-note locator referencing the note currently open in the author's editor. The two locator shapes are exclusive — the caller supplies one or the other, not both.
- **FR-006**: When the focused-note locator is supplied but no note is focused, System MUST fail with the existing active-mode cohort error (whichever `code` + `details.code` pair the active-mode ADR specifies for "no focused note"; this BI introduces no new sub-discriminator for this condition). No filesystem access MUST occur and no write MUST be performed.

#### Placement mode and body semantics

- **FR-007**: System MUST support exactly one placement mode — *replace*. The block tied to the named `^block-id` marker has its body content swapped for the supplied replacement content. The operation has no `append` or `prepend` mode; callers needing those shapes use Write Note or Patch Heading.
- **FR-008**: For paragraph blocks (a paragraph whose trailing token is `^block-id`), the body is the paragraph text excluding the trailing block-marker token. The replacement content becomes the new paragraph; the `^block-id` marker re-attaches at the conventional paragraph-trailing position (the same position relative to the new content that it occupied relative to the old content).
- **FR-009**: For list-item blocks (a list-item line whose trailing token is `^block-id`), the body is the list-item content excluding the trailing block-marker token. The replacement content becomes the new list-item content at the same list nesting and indentation; the `^block-id` marker re-attaches at the conventional list-item-trailing position; surrounding list items and the list's overall structure are preserved.
- **FR-010**: For table, callout, blockquote, indented-code, and other block-shaped elements whose `^block-id` marker conventionally sits on a separate line immediately following the block, the body is the block's full content as the underlying parser delimits it. The replacement content becomes the new block; the `^block-id` marker remains at its original position relative to the block (the line immediately following the new block content).
- **FR-011**: System MUST treat content inside a fenced code block as opaque body. A `^block-id` token appearing inside a fence is NOT an eligible target; requests naming such a block-id MUST surface as "block-reference not found" per FR-016.
- **FR-012**: System MUST preserve the note's existing trailing-newline convention.
- **FR-013**: System MUST preserve the note's existing line-ending convention (LF, CRLF, or other) when applying the patch.
- **FR-014**: System MUST NOT modify YAML frontmatter under any circumstance. Frontmatter writes are handled by the frontmatter-write capability.
- **FR-015**: System MUST NOT create a new `^block-id` marker. The operation replaces the body tied to an existing marker; emitting a new marker is part of writing the surrounding content via Write Note or Patch Heading.

#### Response shape

- **FR-016**: On a successful patch, System MUST return a response identifying (a) the vault-relative location of the patched note and (b) the block-id that matched. The response shape supports caller attribution and rollback. Cohort parity: write-side tools echo the locator for write-verification (per the project's read-vs-write echo convention).

#### Failure modes and atomicity

- **FR-017**: When the supplied block-id does not match any eligible `^block-id` marker in the target note, System MUST fail with `code: "CLI_REPORTED_ERROR"` + `details.code: "BLOCK_NOT_FOUND"` and the error payload MUST name the missing block-id. The file on disk MUST NOT be modified.
- **FR-018**: When the supplied note path does not correspond to an existing note in the vault, System MUST fail with `code: "CLI_REPORTED_ERROR"` + `details.code: "NOTE_NOT_FOUND"` (cohort parity with the existing read-side `NOTE_NOT_FOUND` discriminator). No note MUST be created.
- **FR-019**: When the block-id input is structurally invalid (empty, contains characters outside the supported alphabet per FR-004, supplied with a leading `^`, or exceeds the input-length cap), System MUST reject the request at the input-validation boundary, before any filesystem access or subprocess invocation, with `code: "VALIDATION_ERROR"` + `details.code: "INVALID_BLOCK_ID"` + `details.reason` set to one of `"empty"`, `"contains-invalid-chars"`, `"leading-caret"`, or `"too-long"`.
- **FR-019a**: When the supplied block-id resolves to a `^block-id` marker attached to a heading line (e.g. `# Heading ^foo`), System MUST fail with `code: "CLI_REPORTED_ERROR"` + `details.code: "BLOCK_ON_HEADING"`, naming the resolved block-id, before any write. The wrapper does not rewrite heading-line text under this surface; heading-body edits belong to Patch Heading per the published scope split. The file on disk MUST NOT be modified.
- **FR-020**: A single invocation MUST NOT leave the note on disk in a half-written or otherwise inconsistent state at any observable instant. The wrapper inherits single-invocation atomicity from the underlying read-modify-write substrate (BI-016 reliable writer).
- **FR-021**: When the target note is open in an external editor with unsaved changes AND the reliable-writer substrate signals that condition to the wrapper, System MUST fail loud with `code: "CLI_REPORTED_ERROR"` + `details.code: "EXTERNAL_EDITOR_CONFLICT"` + `details.reason` set to `"unsaved-changes"` or `"file-locked"` matching the OS-level signal; the file on disk MUST NOT be modified. When the substrate does not signal the condition, the edit lands and the editor sees a refreshed file on next focus — this detection-capability caveat is documented in the tool's help text. Cohort parity with Patch Heading FR-021.
- **FR-022**: For every failure mode in FR-017 through FR-021 (inclusive of FR-019a), System MUST NOT modify the file on disk. "Failed loud" means the typed error is the only side effect visible to the caller and the filesystem.

#### Out of scope at the contract boundary

- **FR-023**: System MUST NOT accept locators other than the block-id locator defined in FR-001. Heading paths, line offsets, character offsets, and CSS-selector-style locators are out of scope.
- **FR-024**: System MUST NOT accept placement modes other than `replace`. Append and prepend semantics under a block-reference are out of scope; callers needing those shapes use Write Note or Patch Heading.
- **FR-025**: System MUST NOT take a backup of the note's original content before replace. Replace is destructive by design; callers needing a backup take one before submitting.
- **FR-026**: System MUST NOT publish any transactional contract across concurrent invocations beyond the single-invocation atomicity of FR-020. The published cross-invocation contract is last-write-wins; callers needing stronger guarantees coordinate externally.

### Key Entities

- **Note**: A markdown file inside the vault, addressed by vault + vault-relative path (or via the focused-note locator). Contains an optional YAML frontmatter block followed by a body composed of paragraphs, lists, tables, fenced code, callouts, and other markdown constructs. Has a line-ending convention (LF / CRLF) and a trailing-newline convention (present / absent) that the operation preserves.
- **Block reference**: An Obsidian anchor formed by a `^block-id` token attached to a specific block in a note's body. Survives heading renames, list-item reordering, and table edits. The convention by which the marker attaches to its block depends on the block shape (paragraph: trailing token on the paragraph's final line; list item: trailing token on the item's line; table / callout / blockquote / indented-code: on a separate line immediately following the block). The marker text itself is the literal `^` character followed by the bare identifier.
- **Block-id**: The bare identifier portion of a block reference (the substring following the `^` delimiter). Alphanumeric plus `-` and `_`, case-sensitive, bounded by the cohort input-length cap. Resolves a block reference uniquely inside a single note.
- **Block body — replace semantics**: The portion of the note rewritten under the `replace` placement mode. Block-shape-dependent: for paragraphs and list items, the trailing block-marker token is preserved and the rest of the block's text content is swapped; for tables and other block-shaped elements whose marker sits on a separate following line, the full block content is swapped and the marker line is preserved at its original position.
- **Focused note**: The note the user currently has open in their editor. Resolved at request time; failure to resolve (no note focused) surfaces a typed error per FR-006.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: A vault author can replace the body tied to any paragraph, list-item, table, or other block-shaped `^block-id` marker with a single request, without restating any part of the note outside the targeted block.
- **SC-002**: For every patch operation that succeeds, the note's bytes outside the targeted block — every other block's body, every other section's content, the frontmatter, the line-ending convention, the trailing-newline convention, and the `^block-id` marker itself — are unchanged from the pre-call state.
- **SC-003**: Every failure mode (block-reference not found, note not found, malformed block-id, no focused note, external-editor conflict) surfaces as a typed, programmatically distinguishable error. No failure mode produces a silently-empty success, a silently-wrong write, or a half-written file.
- **SC-004**: Repeat invocations of the same request against an unchanged vault produce identical outcomes (same write, same response) — the operation is deterministic given a fixed input and a fixed vault state.
- **SC-005**: An author's payload size for a single patch is bounded by the size of the replacement content plus the locator metadata — *not* by the size of the surrounding note. A 50-byte replacement of a block inside a 50,000-byte note moves 50 bytes plus the locator, not 50,050 bytes round-trip.
- **SC-006**: The response payload identifies the patched note's vault-relative location and the matched block-id, enabling caller attribution and rollback without re-reading the note.

## Assumptions

- **Reliable-writer substrate**: The underlying read-modify-write substrate (per BI-016 "reliable writer") provides the single-invocation atomicity that FR-020 publishes and the external-editor unsaved-changes signal that FR-021 surfaces. This specification does not invent new atomicity guarantees; it describes the observable contract the wrapper publishes given that substrate.
- **Cross-invocation concurrency**: Per the user's explicit out-of-scope statement, the wrapper publishes no transactional contract across concurrent invocations beyond single-invocation atomicity. The cross-invocation behaviour is last-write-wins; callers needing stronger guarantees coordinate externally. This is the deliberate scope contract, not a defect to be patched later.
- **Locator schema cohort parity**: The "explicit locator vs. focused-note locator" duality (FR-005, FR-006) follows the locator-shape conventions established by the existing typed-tool cohort (cf. ADR-004 / ADR-013 — `target_mode`, `vault`, `path`, and the active-mode TOCTOU concerns documented across the active-mode tool family). The plan phase confirms the exact schema shape; the spec describes the contract.
- **Error code cohort parity**: Typed errors surface under the existing top-level error codes per Constitution Principle IV — zero new top-level codes; Principle IV streak preserved. The discriminators `BLOCK_NOT_FOUND`, `BLOCK_ON_HEADING`, and `EXTERNAL_EDITOR_CONFLICT` parallel the read-side `NOTE_NOT_FOUND` / `HEADING_NOT_FOUND` (cohort) and Patch Heading's `EXTERNAL_EDITOR_CONFLICT` respectively. `INVALID_BLOCK_ID` under `VALIDATION_ERROR` parallels `INVALID_BASE_PATH` / `INVALID_VIEW_NAME` / `INVALID_HEADING_PATH`. The "no focused note" condition reuses the existing active-mode cohort discriminator.
- **Block-id alphabet**: Per FR-004, alphanumeric plus `-` and `_`. This matches the alphabet Obsidian's own block-id generator emits and what wikilink resolvers accept. The plan phase will confirm against the upstream parser; the spec describes the contract surfaced at the wrapper boundary.
- **Empty replacement content**: Accepted as the legitimate "clear the body" operation, cohort parity with Patch Heading's `replace` mode accepting empty content (BI-040 FR-018a). The block's text content becomes zero bytes; the `^block-id` marker is preserved.
- **Multi-line replacement into single-line shapes**: Accepted at the input-validation boundary. Whether the resulting markdown remains a structurally valid list item or table row is the caller's responsibility — the wrapper does not invent shape-specific structural validation. Surfacing a typed error for "newline in single-line shape" would push a parser-aware validation surface onto the boundary that the underlying read-modify-write substrate does not require.
- **Block-id on heading line**: Resolved via auto-mode default to a typed rejection routing the caller to Patch Heading (FR-019a, `BLOCK_ON_HEADING`). Alternatives considered: (a) rewrite the heading text — rejected because it silently mutates heading-path resolution for every wikilink `[[note#Heading]]` elsewhere in the vault; (b) replace the heading's section reach — rejected because it duplicates Patch Heading's `replace` mode under a second locator surface, violating the published scope split. `/speckit-clarify` may revisit; the documented default is option (a) per cohort fail-loud-and-route convention.
- **Same-id-twice ambiguity**: A note authored with two `^foo` markers is structurally malformed under Obsidian's own contract — wikilinks `[[note#^foo]]` resolve ambiguously regardless of which tool reads them. The wrapper's behaviour follows the underlying parser's binding choice; the wrapper does NOT introduce a new fail-loud path for this condition because doing so would surface a pre-existing authoring bug as a wrapper-specific error, breaking cohort parity with every other read-side tool that touches the same parser.
- **Markdown convention**: The note's block-marker syntax follows Obsidian's published convention: `^block-id` as the trailing token on a paragraph or list-item line, or on a separate line immediately following a table / callout / blockquote / indented-code block. Setext-style headings and other non-ATX heading shapes inherit the same block-marker conventions per the underlying parser; specifics deferred to the plan phase.
- **Focused-note resolution mechanism**: The mechanism by which the wrapper learns which note is "focused" is whatever the existing focused-note locator surface (per ADR-004 / ADR-013 active-mode tools) already provides. No new focus-detection mechanism is introduced for this BI.
- **Size ceiling**: There is no wrapper-imposed maximum on note size or replacement-content size beyond the input-validation-boundary input-length cap on the locator strings themselves (1000 UTF-16 code units per string at the boundary, per cohort parity with BI-033 / BI-038 / BI-039 / BI-040). Full-note size is governed by the substrate, not by this BI.
