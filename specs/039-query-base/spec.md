# Feature Specification: Query Base

**Feature Branch**: `039-query-base`
**Created**: 2026-05-20
**Status**: Draft
**Input**: User description: "Add Query Base — a typed capability that lets an agent retrieve the rows of a named view inside an Obsidian Bases (`.base`) file, returning the matched rows as structured JSON. Predictable JSON shape (array of objects keyed by column name). Structured, distinguishable errors for missing file vs. missing view. Empty view returns success with zero rows. Out of scope: non-JSON output formats, view enumeration, base-file enumeration, write-side (create/mutate base files or rows), active-file targeting without an explicit path, querying without a view name."

## Clarifications

### Session 2026-05-20

- Q: Row identity field — does each returned row carry a wrapper-guaranteed source-note locator, or are rows pure passthrough of view-defined columns? → A: **Wrapper guarantees a reserved `path: string` (vault-relative) field on every row, alongside the view's view-defined columns.** Cohort parity with ADR-003's `path=` locator across the entire read-side surface (`read_note`, `read_property`, `read_heading`, `outline`, `links`, `delete`, `rename`, `move`) — the next-hop call from a row is literal: `read_note({ target_mode: "specific", vault, path: row.path })`. The BI-0048 2026-05-18 live probe locked the upstream emission shape as `[{"path": "...", "id": "..."}]` when the view selects `file.path` — `path` is the natural JSON key for the `file.path` Bases column (last dotted segment), not `file`. Reserving `file:` would invent a wrapper-side name that contradicts upstream and forces every caller to mentally translate `row.file` → `path=row.file`. The wrapper validates `path`'s presence on every row; if a view does not include `file.path` in its selection, the wrapper synthesises the field from row metadata so the contract holds regardless of view-author choices. Collision rule: if a view defines a column also named `path`, the wrapper-injected `path` wins; the view-defined column surfaces as `path_view` so callers can disambiguate without loss. The worked example in [[Best Practices#Vault and Knowledge Hygiene]] (next-id allocation via per-type index Base) already presumes the `path` key shape.
- Q: Column order guarantee — does the response carry an explicit column-order vector, or rely on JSON object key insertion order? → A: **Envelope carries an explicit `columns: string[]` field listing column names in view-declared order, with the reserved `path` at index 0 as the leading entry.** The successful response shape moves from a bare array to an envelope object: `{ columns: string[], rows: Array<RowObject> }` (plus the truncation signal added in a later question). The `columns` vector lets tests assert column order exactly without trusting JSON.parse insertion-order semantics, and it surfaces the view's schema even on empty-rows responses (FR-006) where the agent cannot introspect from `rows`. Each row object's keys mirror the `columns` vector; insertion-order is preserved as a belt-and-braces secondary guarantee, but the `columns` vector is the authoritative ordering. Collision rule (per Q1): when a view defines a column named `path`, `columns` still lists the reserved `path` at index 0 (the wrapper-injected entry), and the view-defined column appears under `path_view` at whatever index the view declared for it. Both names are visible in the `columns` vector so callers can disambiguate without re-probing the row shape.
- Q: Error top-level code mapping — under which existing top-level code do `BASE_NOT_FOUND` and `VIEW_NOT_FOUND` surface? → A: **Both `BASE_NOT_FOUND` and `VIEW_NOT_FOUND` surface under top-level `code: "CLI_REPORTED_ERROR"`, distinguished by `details.code: "BASE_NOT_FOUND"` vs. `details.code: "VIEW_NOT_FOUND"`.** Cohort parity with the entire "missing thing" family in the read-side surface — `VAULT_NOT_FOUND` and `NOTE_NOT_FOUND` already surface under `CLI_REPORTED_ERROR`, distinguished by `details.code`. Callers handle the whole family with one pattern-match site: `if (err.code === "CLI_REPORTED_ERROR") switch (err.details.code) { case "BASE_NOT_FOUND": ...; case "VIEW_NOT_FOUND": ...; case "VAULT_NOT_FOUND": ... }`. Constitution Principle IV streak preserved (no new top-level codes — the fifteen-tool zero-new-codes streak holds). `INVALID_BASE_PATH` remains under `VALIDATION_ERROR` per FR-011 / FR-012; it is a caller-input issue (schema-level), not a "thing exists or not" issue (CLI-discovered).
- Q: Truncation signal shape — what exact fields on the envelope signal that the row set was capped at 1000? → A: **`truncated: boolean` always present on the success envelope, plus `total_rows: number` required when `truncated: true` and omitted when `truncated: false`.** Agent learns both whether truncation occurred AND by how much in a single round-trip — can decide between narrowing the view filter, switching to a sibling view, or accepting the 1000-row sample without probing iteratively. Bases natively maintains row counts for views (per the underlying Obsidian 1.9+ data model), so `total_rows` is cheap for upstream to surface; planning takes the obligation to ensure the count is filled even when the wrapper halts enumeration at 1000. The boolean is kept (rather than C's `total_rows`-only derivation) because `truncated: false` reads cleaner at the caller's switch site than `total_rows === rows.length` and avoids the corner case where a view legitimately matches exactly 1000 notes (where derivation-based detection would falsely signal truncation if `total_rows` were defined as a hint rather than a ground truth). Both `truncated` and `total_rows` are top-level envelope fields, sibling to `columns` and `rows`.
- Q: Canonical tool name — what string does the operation register under for MCP discovery and source-tree layout? → A: **`query_base`. ADR-010 mechanical mapping from the upstream CLI subcommand `base:query` (composite `namespace:action` form).** ADR-010's composite-subcommand rule reverses `namespace:action` to `action_namespace`, lowercase, underscore-joined — yielding `query_base`. Same rule that produced `set_property` (from `property:set`) and `context_search` (from `search:context`); both are shipped precedent. Direct 1:1 wrap of a native CLI subcommand — the exact pattern ADR-010 was authored for; no ADR-010 point-3 grandfather clause applies. **Bases-family cohort coherence:** the relevant cohort is NOT `read_note` / `read_property` / `read_heading` (those are eval-composed under ADR-010 point-3 grandfather clause — `read_property` composes on `properties format=json`, `read_heading` composes on `eval`; there is no ADR-010-enforced `read_*` cohort, it is a coincidence of eval-composition). The actual cohort is the four-member Bases family, all ADR-010-anchored: `bases` (BI-0049, single-word) → `bases`; `base:query` → `query_base` (this BI); `base:views` → `views_base` (BI-0082); `base:create` → `create_base` (BI-0083). Picking `read_base_view` would break ADR-010's mechanical rule AND orphan the query tool from its three siblings (which all retain the `*_base` suffix). **Source layout**: `src/tools/query_base/{index,schema,handler,*.test}.ts` — three production files plus three co-located test files, matching the 16-tool cohort precedent (Principle I per-surface module triplet; the constitution's literal `{schema, tool, handler}.ts` template is a template form not realised by any tool today). **Documentation slot**: alongside `bases`, `views_base`, `create_base` in the Bases-family cohort, not in the read-side surface.
- Q: Reserved `id` field alongside `path`? — does the wrapper guarantee `id` on every row (parity with the reserved `path`), passthrough it only when upstream emits it, suppress it, or treat it as any other non-reserved column? → A: **Pure passthrough. The wrapper reserves exactly one row field — `path` — and treats every other key (including `id`, `name`, `tags`, `created`, `updated`, frontmatter properties, formula-derived columns) identically: surfaced verbatim with whatever name and type Bases emits, no synthesis, no suppression, no wrapper guarantee about presence, type, or cross-query stability.** Decision criterion is three-pronged — universality, semantic stability, synthesis cost — and `path` clears all three while `id` clears none confidently. Universality: every vault file has a path; `id` exists only when a view selects an id-producing column AND the underlying note carries one. Semantic stability: `path` is the next-hop locator with vault-model-fixed semantics across the entire cohort (`read`, `read_property`, `links`, `backlinks`, `find_and_replace` subtree scoping); `id`'s meaning depends on the view's column selection, the vault's frontmatter convention, and possibly the Bases plugin version. Synthesis cost: synthesising `path` when upstream omits has a defensible source (the row's underlying file binding); synthesising `id` requires inventing a value with no stable referent (path-as-id duplicates the reserved field, row-index-as-id falsifies cross-query stability, content-hash-as-id invents a semantic Bases does not expose). The BI-0048 2026-05-18 probe showed `id` alongside `path` only because the Type ID Index Base view in that vault deliberately selects an `id`-producing column under this vault's frontmatter convention (BI-NNNN, ADR-NNN, TC-NNNNN identifiers) — the observation is descriptive of one vault's view-author choice, not generalisable. Constitution Principle IV alignment — no silent fallbacks; the wrapper does not invent values to satisfy a reserved-field obligation. **Future-proofing**: if a later empirical investigation establishes that Bases exposes a stable, rename-resilient, cross-query file-identity field (whether spelled `id` or otherwise), that finding can elevate the relevant field to a reserved second locator in a follow-on BI; until then, this BI's contract stays minimal and lets the view layer decide which non-`path` columns appear.
- Q: View-name matching semantics — how is the supplied `view_name` matched against the views declared in the `.base` file? → A: **Exact case-sensitive match, no whitespace trimming.** `"My View" ≠ "my view"`; `" my view "` does NOT match a view named `"my view"`. Cohort parity with `read_property` (exact property-name match) and `read_note` (case-sensitive path match). No fuzzy matching, no case-folding, no whitespace normalisation. A case-mismatched or whitespace-padded `view_name` MUST surface a clean `VIEW_NOT_FOUND` error rather than a silent "helpful" match — silent fixes mask caller typos and produce non-deterministic behaviour when views in the same file legitimately differ by case (`"open"` and `"Open"`). Agents typically obtain `view_name` from the `.base` file's declared view list (parseable from the file's YAML) or from a follow-up `views_base` call (BI-0082); both sources hand the agent the exact string, so the agent has no need for fuzzy resolution. Determinism beats friendliness here.
- Q: Input length caps on `base_path` and `view_name` — are the two locator strings bounded at the input-validation boundary, and at what length? → A: **Uniform 1000 UTF-16 code units cap on each, enforced at the input-validation boundary BEFORE any filesystem access or subprocess invocation, parity with BI-033 (1000-char `query` cap) and BI-038 (1000-char `pattern` / `replacement` caps).** Over-cap `base_path` surfaces `code: "VALIDATION_ERROR"` + `details.code: "INVALID_BASE_PATH"` + `details.reason: "too-long"`. Over-cap `view_name` surfaces `code: "VALIDATION_ERROR"` + `details.code: "INVALID_VIEW_NAME"` + `details.reason: "too-long"`. Both reject before any filesystem access or upstream invocation; no vault read occurs, no subprocess spawns. Typical caller usage is far below 1000 chars (paths are tens of chars; view names are 5–30 chars); over-cap input is virtually always a paste-a-whole-note accident the caller wants surfaced quickly rather than executed. No new top-level codes — REUSE of `VALIDATION_ERROR` (Constitution Principle IV streak preserved). The `INVALID_VIEW_NAME` `details.code` is a new sub-discriminator introduced in this BI for `view_name`-specific validation issues (parity with how `INVALID_BASE_PATH` carries `path-traversal` / `wrong-extension` / `too-long` sub-states); `details.reason` distinguishes `"too-long"` from `"empty"` (FR-011).
- Q: Natural-order determinism when view defines no sort — what is the canonical row order for views that declare no explicit sort, given that SC-003 promises determinism across repeat invocations? → A: **Wrapper post-sorts `rows` by the reserved `path` field ascending — only when the view defines no explicit sort. When the view defines an explicit sort, that sort governs as the primary ordering signal, and the wrapper applies `path`-ascending as a tiebreaker for equal sort-key values.** Cohort parity with BI-038's two-level deterministic ordering ("path-ascending across notes, then line-number-ascending within each note") — wrapper-controlled deterministic baseline rather than upstream-emission-trust. SC-003 (determinism across repeat invocations) holds independent of Bases' internal emission-order stability; if a future Bases version changes its walk order, the wrapper-applied `path`-asc baseline absorbs the churn. Tests against no-sort fixtures can assert exact row order. Agents observing repeated invocations against unchanged vaults see identical row order regardless of upstream emission semantics. The wrapper does NOT re-sort when the view declares an explicit sort — view-author intent is preserved on the primary sort axis; only equal-key rows are broken by `path` ascending. Tertiary tiebreakers (when `path` is itself equal across rows, e.g., two virtual rows synthesised from the same note) are upstream's emission order.
- Q: Malformed `.base` file handling — how does the operation surface a `.base` file that exists on disk but is structurally unusable (empty, invalid YAML, valid YAML missing required Bases-schema keys, unsupported future schema version, or non-Bases content with a `.base` extension)? → A: **New sub-discriminator `details.code: "BASE_MALFORMED"` under top-level `code: "CLI_REPORTED_ERROR"`. Distinct from `BASE_NOT_FOUND` (file missing on disk) and `VIEW_NOT_FOUND` (file fine, view missing inside it).** Three states in the "something is wrong with the base" family, each with its own `details.code`: file missing → `BASE_NOT_FOUND`; file present but structurally unusable → `BASE_MALFORMED`; file fine but the named view does not exist → `VIEW_NOT_FOUND`. The caller's remediation path differs across all three: BASE_NOT_FOUND → fix the filename; BASE_MALFORMED → fix the file content or restore from backup; VIEW_NOT_FOUND → use a valid view name. When the wrapper can determine the failure reason, `details.reason` surfaces a sub-state — `"empty"` (zero-byte file), `"invalid-yaml"` (YAML syntactically broken), `"missing-required-key"` (YAML parses but the Bases-schema-required keys are absent), `"unsupported-schema-version"` (file declares a Bases schema version newer than the wrapper understands), or `"unknown"` (upstream errored out but the wrapper cannot pinpoint the reason). No new top-level error code (Constitution Principle IV streak preserved — sixteen-tool zero-new-codes streak after BI-038 / BI-039); the new `BASE_MALFORMED` `details.code` is a sub-discriminator per ADR-015, parity with how `INVALID_BASE_PATH` carries multiple sub-state reasons.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Retrieve rows of a named view as structured JSON (Priority: P1)

An agent issues a request naming a vault-relative `.base` file and a view inside it. The response is a JSON envelope `{ columns, rows, truncated, total_rows? }` where `columns` lists the view's column names in view-declared order (with a reserved `path` at index 0), `rows` carries up to 1000 rows that view selects in the order the view defines, `truncated` is always present (false when the row set fit in the cap, true otherwise), and `total_rows` reports the upstream's full match count when `truncated: true` (omitted when `truncated: false`). Each row object is keyed by the names in `columns`. When the named view exists but matches zero notes, the response is the same envelope with `rows: []`, the populated `columns` vector, and `truncated: false` — not an error. The agent can iterate rows by column name, learn the view's schema even from a zero-row response, and decide narrowing strategy from `total_rows` when truncated, without writing a parser.

**Why this priority**: The whole feature exists to give agents a structured retrieval surface over Obsidian Bases. Without P1 there is no MVP. The predictable JSON shape and the empty-success behaviour are intrinsic to the contract — a caller cannot trust an array-of-rows response if the shape varies or if "no matches" is ambiguous with "something went wrong". This is the slice every other behaviour builds on; with only P1 the agent can already reason over vault-tabular data and replace per-caller parser code.

**Independent Test**: Stand up a vault with a `.base` file declaring at least two views. Each view exposes a known column set and matches a known set of notes. Issue the operation naming the first view and verify the response envelope's `columns` field lists that view's columns in view-declared order (with reserved `path` at index 0), `rows` contains row objects keyed by `columns`, the row set matches the view's declared row order, and no rows from the sibling view appear. Issue the operation against a view known to match zero notes and verify the response is the envelope `{ columns: [...], rows: [] }` — the `columns` vector is still populated so the agent learns the view's schema from the empty response.

**Acceptance Scenarios**:

1. **Given** a `.base` file at the supplied vault-relative path containing a view with the supplied name that matches at least one note, **When** the agent invokes the operation, **Then** the response is a JSON envelope `{ columns, rows }` where `columns` lists the view's column names in view-declared order (with reserved `path` at index 0) and `rows` carries the rows that view selects in the order the view defines, with each row an object keyed by `columns`.
2. **Given** a `.base` file containing multiple views, **When** the agent names one view, **Then** only that view's rows appear in `rows` and only that view's column names appear in `columns`; sibling views' rows and column shapes are not included.
3. **Given** a view that exposes its default column set (no explicit column selection), **When** the agent reads any returned row, **Then** the row carries every column the view exposes by default (plus the reserved `path` field).
4. **Given** a view that exists but matches zero notes, **When** the agent invokes the operation, **Then** the response is the envelope `{ columns: [...], rows: [] }` — `columns` is still populated so the agent learns the view's schema, and `rows: []` is a successful empty result, not an error.

### User Story 2 - Distinguish missing file from missing view through typed errors (Priority: P2)

An agent that targets a `.base` file or a view that does not exist receives a typed, structured error identifying which of the two is missing. The two states are programmatically distinguishable in the response envelope — the agent can branch on "wrong path" vs. "wrong view name" without parsing prose. Neither state is silently surfaced as an empty success.

**Why this priority**: P1 gives the happy path. Without P2, an agent that calls the operation and gets back an empty array has three indistinguishable interpretations: the file is missing, the view is missing, or the view matched zero notes. That ambiguity forces the agent into pre-flight checks (does the file exist? does the view exist?) and defeats the purpose of a typed retrieval surface. P2 is the diagnostic contract that makes the response self-describing: success-with-rows, success-with-empty, error-missing-file, and error-missing-view are four distinct states the caller can switch on directly.

**Independent Test**: With P1 in place, issue the operation naming a `.base` file that does not exist and verify the response is `code: "CLI_REPORTED_ERROR"` + `details.code: "BASE_NOT_FOUND"`. Issue the operation naming a `.base` file that exists but a view name that does not exist inside it, and verify the response is `code: "CLI_REPORTED_ERROR"` + `details.code: "VIEW_NOT_FOUND"`. Verify the two errors are programmatically distinguishable via `details.code`. Issue the operation against an existing view that matches zero notes and verify the response is the successful envelope `{ columns: [...], rows: [], truncated: false }` — neither error.

**Acceptance Scenarios**:

1. **Given** the requested `.base` file does not exist at the supplied vault-relative path, **When** the agent invokes the operation, **Then** the response is a typed error identifying the missing file — not a raw crash, not a silent empty array.
2. **Given** the `.base` file exists but contains no view of the supplied name, **When** the agent invokes the operation, **Then** the response is a typed error identifying the missing view, distinct in the envelope from the missing-file error.
3. **Given** the view exists and matches zero notes, **When** the agent invokes the operation, **Then** the response is the successful envelope `{ columns: [...], rows: [] }` — neither of the two error envelopes above.

### Edge Cases

- The `.base` file contains a view whose row order is determined by a configured sort or grouping: rows arrive in the order the view defines (the view's own sort) on the primary axis, with `path` ascending as the tiebreaker for equal sort-key values (FR-003).
- The `.base` file contains a view that declares no explicit sort: rows arrive in `path`-ascending order — the wrapper-controlled deterministic baseline. Two repeated invocations against an unchanged vault produce the same row order even if a future Bases version changes its internal walk order.
- The view declares an explicit sort that produces equal sort-key values for two notes (e.g., sort by `tags` where both notes have the same tag list): those notes' rows are ordered by `path` ascending relative to each other (FR-003 tiebreaker), regardless of which note Bases emits first.
- The `.base` file contains a view with a default column set (no explicit selection): each row carries every column the view exposes by default.
- A column value in a returned row is empty / null at the source: the row still appears; the cell surfaces as `null` (or the upstream's chosen sentinel) verbatim per FR-014. Absent cells are not coerced to the empty string.
- The view matches notes whose paths contain Unicode (NFC vs. NFD), spaces, or punctuation: rows surface verbatim — no normalisation, parity with the existing read-side surface.
- Two views in the same `.base` file share a column name but expose different cell content: the agent names one view; the response carries only that view's column shape, regardless of the sibling.
- A view defines a column literally named `path` (collision with the wrapper-injected source-note locator): the row carries `path` set to the wrapper-injected vault-relative source-note path AND `path_view` set to the view-defined column's value. Neither value is silently dropped. Callers that didn't know the view defined a `path` column see no surprise — the source-note locator is always at `row.path`.
- A view does not include `file.path` (or any equivalent file reference) in its column selection: the wrapper STILL surfaces a `path` field on each row, synthesised from the row's underlying note metadata. View-author column choice does NOT affect whether the row is self-identifying.
- A view selects a Bases column that emits `id` (or any other non-`path` field — `name`, `tags`, `created`, `updated`, a frontmatter property, a formula result): the column surfaces verbatim under whatever key Bases emits, with no wrapper synthesis or suppression (FR-002d). Callers iterating over rows with views that did NOT select an `id`-producing column see no `id` field; this is correct behaviour, not a bug. Cross-query caching keyed on `id` is not supported by the wrapper contract — only `path` is guaranteed.
- The `.base` file declares views named `"My View"` and the caller passes `view_name: "my view"`: the operation surfaces `VIEW_NOT_FOUND` per FR-005 / FR-005a — exact case-sensitive matching does NOT fold case. The error's `details.view_name` echoes back exactly what the caller supplied so the diff is visible.
- The caller passes a `view_name` with leading or trailing whitespace (e.g., `" my view "` or `"my view\n"`): the operation surfaces `VIEW_NOT_FOUND` per FR-005 / FR-005a — whitespace is NOT trimmed before matching. Callers obtained the view name from a precise source (the `.base` file's YAML or a `views_base` response) and should not paste with extra whitespace.
- The `.base` file declares two views differing only by case (e.g., `"open"` and `"Open"`): each `view_name` matches exactly one declared view. The exact-case-match rule makes this disambiguation deterministic with no "ambiguous match" failure mode.
- The `.base` file exists at the supplied path but is a zero-byte / empty file: the operation surfaces `code: "CLI_REPORTED_ERROR"` + `details.code: "BASE_MALFORMED"` + `details.reason: "empty"` per FR-005b. The caller can distinguish this from BASE_NOT_FOUND (wrong filename) and remediate by restoring the file content.
- The `.base` file exists but contains invalid YAML (syntactic break, unterminated string, indentation error): the operation surfaces `code: "CLI_REPORTED_ERROR"` + `details.code: "BASE_MALFORMED"` + `details.reason: "invalid-yaml"`. The caller remediates by fixing the YAML.
- The `.base` file parses as YAML but is missing keys required by the Bases schema (e.g., no `views:` block): the operation surfaces `code: "CLI_REPORTED_ERROR"` + `details.code: "BASE_MALFORMED"` + `details.reason: "missing-required-key"`. The caller remediates by adding the missing structure.
- The `.base` file declares a Bases schema version the wrapper does not understand (forward-incompatible): the operation surfaces `code: "CLI_REPORTED_ERROR"` + `details.code: "BASE_MALFORMED"` + `details.reason: "unsupported-schema-version"`. The caller's remediation is to use a wrapper version that supports the new schema or downgrade the `.base` file.
- The `.base` file exists but upstream surfaces a generic parse / structure error the wrapper cannot pin to a specific reason: the operation surfaces `code: "CLI_REPORTED_ERROR"` + `details.code: "BASE_MALFORMED"` + `details.reason: "unknown"`. Caller knows the file is the problem but not the precise structural fault.
- The caller passes a `base_path` containing `../`, a leading `/` or `\`, a drive-letter prefix (e.g., `C:\`), or a control character: the operation rejects at the input-validation boundary with `code: "VALIDATION_ERROR"`, `details.code: "INVALID_BASE_PATH"`, `details.reason: "path-traversal"` before any filesystem access. No data is read.
- The caller passes a vault-relative `base_path` whose canonical resolution (e.g., through a symlink) lies outside the vault root: the operation rejects with `code: "PATH_ESCAPES_VAULT"` (REUSE of the existing ADR-009 code). No data is read.
- The caller passes an empty `base_path: ""`: the operation rejects at the input-validation boundary with `code: "VALIDATION_ERROR"` + `details.code: "INVALID_BASE_PATH"` + `details.reason: "empty"`. No vault access occurs.
- The caller passes an empty `view_name: ""`: the operation rejects at the input-validation boundary with `code: "VALIDATION_ERROR"` + `details.code: "INVALID_VIEW_NAME"` + `details.reason: "empty"`. No vault access occurs.
- The caller passes a `base_path` longer than 1000 UTF-16 code units (e.g., accidentally pasting a note's contents): the operation rejects at the input-validation boundary with `code: "VALIDATION_ERROR"` + `details.code: "INVALID_BASE_PATH"` + `details.reason: "too-long"` BEFORE any filesystem access. Catches "I forgot to set the path field" caller bugs as a hard error.
- The caller passes a `view_name` longer than 1000 UTF-16 code units: the operation rejects at the input-validation boundary with `code: "VALIDATION_ERROR"` + `details.code: "INVALID_VIEW_NAME"` + `details.reason: "too-long"` BEFORE any filesystem access or subprocess invocation.
- The caller passes a `base_path` that does not end in `.base` (e.g., names a `.md` note or an arbitrary file): the operation rejects at the input-validation boundary with `code: "VALIDATION_ERROR"`, `details.code: "INVALID_BASE_PATH"`, `details.reason: "wrong-extension"`. No vault access occurs.
- The caller does not supply a `vault` field: the operation runs against the focused vault (cohort default for read-side tools).
- The caller supplies a `vault` display name the registry does not know: the operation surfaces `code: "CLI_REPORTED_ERROR"`, `details.code: "VAULT_NOT_FOUND"`, `details.reason: "unknown"`. A registered-but-closed vault surfaces the same `details.code` with `details.reason: "not-open"`.
- The view's row set changes between two repeated invocations because notes were edited in the interim: each invocation reflects the current vault state at read time. No fingerprinting or drift detection (read surface — same TOCTOU posture as `read_note`, `read_property`, `pattern_search`).
- The view matches more than 1000 rows: the envelope carries the first 1000 rows in view-defined order, `truncated: true`, and `total_rows` set to the upstream's full match count (FR-013). The caller can decide between narrowing the view filter, switching to a sibling view, or accepting the 1000-row sample in a single round-trip.
- The view matches exactly 1000 rows: `truncated: false`, `total_rows` omitted, `rows.length === 1000`. The agent does NOT see a false-positive truncation signal.
- A column value in a returned row is `null`, a number, a boolean, or a nested object: the value is surfaced verbatim as its native JSON type (FR-014). Cells are not coerced to strings.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST expose a typed read-side capability that accepts a vault-relative `.base` file path and a view name, and returns the rows that view selects.
- **FR-002**: The successful response MUST be a JSON envelope object with at least two fields: `columns: string[]` and `rows: Array<RowObject>`. `columns` MUST list the column names in the order the view defines, with the reserved `path` at index 0 as the leading entry. `rows` MUST be a JSON array whose entries are row objects keyed by the names in `columns`. Each row object MUST carry every key listed in `columns` (and no other top-level keys). Row-object key insertion order MUST match the `columns` vector as a secondary guarantee, but the `columns` vector is the authoritative ordering signal — callers MUST be able to recover view-declared column order from `columns` without trusting JSON.parse insertion semantics.
- **FR-002a**: Reserved source-note locator — the wrapper MUST guarantee a `path: string` field (vault-relative path to the source note) on every row, regardless of whether the view's column selection includes `file.path`. The wrapper validates `path`'s presence on every row and synthesises the field from row metadata when the view did not select `file.path`. The reserved `path` gives callers a guaranteed locator for the next-hop call (e.g., `read_note({ vault, path: row.path })`).
- **FR-002b**: Reserved-key collision rule — when a view defines a column also named `path`, the wrapper-injected `path` field MUST win. The view-defined column's value MUST surface under the key `path_view` in the same row, and `path_view` MUST appear in the `columns` vector at the index the view declared for the original `path` column. Both names MUST be visible in `columns` so callers can disambiguate without re-probing the row shape. The wrapper MUST NOT silently drop either value.
- **FR-002c**: Empty-rows schema introspection — when `rows` is the empty array (per FR-006), the envelope MUST still carry the populated `columns` vector. The agent MUST be able to learn the view's schema from a zero-row response without a follow-up call.
- **FR-002d**: Passthrough posture for non-reserved row keys — the wrapper reserves exactly one row field: `path` (per FR-002a, with the collision rule in FR-002b). Every other key in a row — including `id`, `name`, `tags`, `created`, `updated`, frontmatter-derived properties, formula-derived columns, and any future Bases-emitted column — MUST be surfaced verbatim under whatever name and type Bases emits, with no wrapper synthesis, no wrapper suppression, and no wrapper guarantee about presence, type, or cross-query stability. The wrapper MUST NOT single out any non-`path` key for special-case handling. Whether a given non-`path` column appears in `rows` is governed entirely by the view's column selection.
- **FR-003**: Returned rows MUST appear in a deterministic order following these rules: (1) When the view declares an explicit sort, that sort governs the primary ordering signal. (2) When two or more rows share an equal primary sort key (or when the view declares no explicit sort), the wrapper MUST break ties / impose the baseline order by the reserved `path` field ascending. (3) Tertiary tiebreakers (when `path` is itself equal across rows, e.g., two synthetic rows bound to the same note) fall back to upstream's emission order. Caller-supplied order, filesystem-traversal order, and upstream's emission order MUST NOT determine the primary or path-tiebreaker ordering. Cohort parity with BI-038's wrapper-controlled deterministic baseline. SC-003 holds independent of Bases' internal walk-order stability.
- **FR-004**: When the supplied `base_path` resolves to no `.base` file in the vault, the operation MUST return a typed error with top-level `code: "CLI_REPORTED_ERROR"` and `details.code: "BASE_NOT_FOUND"`. The error MUST include the offending vault-relative path in `details` so the caller can echo it back to the user. This error MUST be programmatically distinguishable in the response envelope from the missing-view error (FR-005) and from a successful empty-rows response (FR-006).
- **FR-005**: When the `.base` file exists but contains no view of the supplied `view_name`, the operation MUST return a typed error with top-level `code: "CLI_REPORTED_ERROR"` and `details.code: "VIEW_NOT_FOUND"`. The error MUST include the offending view name and the resolved base-file path in `details`. This error MUST be programmatically distinguishable in the response envelope from the missing-file error (FR-004) and from a successful empty-rows response (FR-006). Cohort parity with `VAULT_NOT_FOUND` and `NOTE_NOT_FOUND` — both surface under `CLI_REPORTED_ERROR` with a `details.code` discriminator, so callers handle the whole "missing thing" family with one switch site.
- **FR-005a**: View-name matching MUST be exact case-sensitive string comparison against the views declared in the `.base` file. No case-folding, no whitespace trimming on either side, no fuzzy / Levenshtein / shared-prefix matching. A `view_name` that differs from a declared view's name by case (`"My View"` vs. `"my view"`) or by leading / trailing whitespace MUST surface `VIEW_NOT_FOUND` per FR-005, not a silent match. Cohort parity with `read_property` (exact property-name match) and `read_note` (case-sensitive path match).
- **FR-005b**: Malformed-base handling — when the `.base` file exists on disk but is structurally unusable (zero-byte / empty file, YAML syntactically invalid, YAML parses but the Bases-schema-required keys are absent, file declares a Bases schema version newer than the wrapper understands, or `.base`-extension file written by a non-Bases tool), the operation MUST return a typed error with top-level `code: "CLI_REPORTED_ERROR"` and `details.code: "BASE_MALFORMED"`. The error MUST be programmatically distinguishable from `BASE_NOT_FOUND` (FR-004, file missing on disk) and from `VIEW_NOT_FOUND` (FR-005, file fine, view missing). When the wrapper can determine the failure reason, `details.reason` MUST sub-discriminate among `"empty"`, `"invalid-yaml"`, `"missing-required-key"`, `"unsupported-schema-version"`, or `"unknown"`. The error MUST include the resolved base-file path in `details` so the caller knows which file failed.
- **FR-006**: When the named view exists and matches zero notes, the operation MUST return a successful envelope with `rows: []` and the populated `columns` vector (per FR-002c) — not an error. `truncated` MUST be `false` and `total_rows` MUST be omitted.
- **FR-007**: When the `.base` file contains multiple views, the response MUST carry only the named view's rows. Rows belonging to sibling views in the same `.base` file MUST NOT appear in the response.
- **FR-008**: A view exposing its default column set (no explicit column selection) MUST surface every column the view exposes by default in each returned row.
- **FR-009**: Vault selection — the operation MUST accept an optional `vault` field. When absent, the operation runs against the focused vault. When present, the supplied display name MUST be resolved via the project's lazy vault registry (parity with the existing read-side cohort: `read_note`, `read_property`, `pattern_search`). An unknown or closed vault MUST surface `code: "CLI_REPORTED_ERROR"` with `details.code: "VAULT_NOT_FOUND"` and a sub-discriminating `details.reason` ("unknown" vs. "not-open").
- **FR-010**: Path-traversal safety — `base_path` inputs containing `../`, a leading `/` or `\`, a drive-letter prefix, or a control character MUST be rejected at the input-validation boundary BEFORE any filesystem access (Layer 1, ADR-009). After resolution, the canonical path of the target MUST be verified to lie under the canonical vault root (Layer 2); on escape, the operation MUST surface `code: "PATH_ESCAPES_VAULT"`.
- **FR-011**: Input validation (empty) — an empty `base_path: ""` MUST be rejected at the input-validation boundary with `code: "VALIDATION_ERROR"` + `details.code: "INVALID_BASE_PATH"` + `details.reason: "empty"`. An empty `view_name: ""` MUST be rejected with `code: "VALIDATION_ERROR"` + `details.code: "INVALID_VIEW_NAME"` + `details.reason: "empty"`. Both reject BEFORE any vault access or subprocess invocation.
- **FR-011a**: Input validation (length caps) — `base_path` MUST be capped at 1000 UTF-16 code units; over-cap input surfaces `code: "VALIDATION_ERROR"` + `details.code: "INVALID_BASE_PATH"` + `details.reason: "too-long"`. `view_name` MUST be capped at 1000 UTF-16 code units; over-cap input surfaces `code: "VALIDATION_ERROR"` + `details.code: "INVALID_VIEW_NAME"` + `details.reason: "too-long"`. Both caps enforced BEFORE any filesystem access or subprocess invocation, parity with BI-033 `query` cap and BI-038 `pattern` / `replacement` caps.
- **FR-012**: Extension filter — a `base_path` that does not end with the `.base` extension (case-insensitive on the extension) MUST be rejected at the input-validation boundary with `code: "VALIDATION_ERROR"`, `details.code: "INVALID_BASE_PATH"`, `details.reason: "wrong-extension"`. No filesystem access occurs.
- **FR-013**: Row-count cap and truncation signal — the response MUST cap `rows` at 1000 entries (parity with BI-033 `search_vault_content` and BI-035 `context_search`). The success envelope MUST always carry a `truncated: boolean` field. When `truncated: false`, the envelope MUST omit `total_rows` and `rows` MUST contain every row the view matched. When `truncated: true`, `rows` MUST contain the first 1000 rows in view-defined order AND the envelope MUST carry `total_rows: number` reporting the upstream's full count of matching rows for the view. A view that legitimately matches exactly 1000 notes MUST surface `truncated: false` (not a false-positive truncation signal). Unbounded payloads are rejected as they risk blowing the agent's context window; capping with a typed `truncated` + `total_rows` pair lets the agent decide between narrowing the view filter, switching to a sibling view, or accepting the 1000-row sample without iterative probing.
- **FR-014**: Column-value typing — each cell in a returned row MUST preserve the upstream's native JSON type (number, boolean, null, nested object, ISO-date string). String-coercion of all cells is rejected as it would defeat the typed-retrieval value proposition the feature exists to deliver. Callers that need a string projection can stringify per-cell at the call site; callers that want typed values cannot recover them from a coerced response. Parity with the read-side cohort (`read_property` surfaces upstream's typed value as-is).
- **FR-015**: Side-effect freedom — the operation MUST be a pure read. Invoking it MUST NOT mutate any note, the `.base` file, or any vault metadata. No write-queue acquisition occurs (parity with the read-side cohort).
- **FR-016**: Error envelope discipline — all failures MUST surface through the project's typed `UpstreamError` envelope (Constitution Principle IV). No new top-level error codes are introduced — this BI continues the zero-new-codes streak; the count becomes sixteen tools after BI-039 ships. New failure states are expressed via `details.code` sub-discrimination per ADR-015. Top-level code map for the new states: `BASE_NOT_FOUND` → `CLI_REPORTED_ERROR` (FR-004); `BASE_MALFORMED` → `CLI_REPORTED_ERROR` with sub-states `empty` / `invalid-yaml` / `missing-required-key` / `unsupported-schema-version` / `unknown` (FR-005b); `VIEW_NOT_FOUND` → `CLI_REPORTED_ERROR` (FR-005); `INVALID_BASE_PATH` → `VALIDATION_ERROR` with sub-states `path-traversal` (FR-010), `wrong-extension` (FR-012), `empty` (FR-011), `too-long` (FR-011a); `INVALID_VIEW_NAME` → `VALIDATION_ERROR` with sub-states `empty` (FR-011) and `too-long` (FR-011a); `VAULT_NOT_FOUND` → `CLI_REPORTED_ERROR` (FR-009, cohort reuse); `PATH_ESCAPES_VAULT` → top-level (FR-010, ADR-009 reuse). Callers handle the whole missing-thing-or-broken-thing family at a single switch site keyed on `details.code`.

### Key Entities

- **Base file**: A vault-relative file with the `.base` extension. The native Obsidian 1.9+ tabular artifact. May contain one or more named views.
- **View**: A named selector inside a base file. Defines a column set, a row-selection rule over vault notes, and a row order. The agent names exactly one view per invocation.
- **Row**: A single record returned by a view. Surfaces as a JSON object keyed by the view's column names PLUS a reserved `path: string` field (vault-relative path to the source note) the wrapper guarantees on every row regardless of view-author column selection. On collision with a view-defined column also named `path`, the wrapper-injected `path` wins and the view-defined column surfaces as `path_view`.
- **Column**: A named field in a view's output schema. The view's declared column order determines (a) the order of names in the envelope's `columns` vector and (b) the row objects' key insertion order (secondary guarantee). The reserved `path` column always occupies index 0 in `columns`; a view-defined collision with `path` surfaces under `path_view`.
- **Vault**: The Obsidian vault the base file lives in. Resolved via the project's lazy vault registry; absent vault field defaults to the focused vault.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: An agent retrieves the rows of a named view inside a `.base` file in a single call, without writing custom code to parse subprocess text output. Replaces the current escape-hatch (raw subprocess output) for the bases-query use case.
- **SC-002**: An agent receiving a response can programmatically distinguish four states without parsing prose strings: success-with-rows, success-with-empty-rows, error-missing-file, and error-missing-view. The four states are surfaced by distinct, type-narrowable envelope discriminators.
- **SC-003**: Repeated invocations of the operation against an unchanged vault and an unchanged `.base` file return the same rows in the same order — the row-order contract is deterministic and matches the view's declared order.
- **SC-004**: Naming a view in a multi-view `.base` file returns only that view's rows. An agent inspecting the response cannot mistakenly read a sibling view's rows.
- **SC-005**: An accidental call with a malformed `base_path` (path traversal, wrong extension, empty string) is rejected at the input-validation boundary before any filesystem access; no data is read and no log noise is generated beyond the validation refusal.

## Assumptions

- The upstream CLI exposes the underlying bases-query operation as the `base:query` subcommand (composite `namespace:action` form), returning structured output the wrapper converts to the FR-002 envelope. Tool name `query_base` follows from `base:query` mechanically via ADR-010's composite-reversal rule.
- The relevant cohort for naming and documentation is the Bases family (`bases`, `query_base`, `views_base`, `create_base`) — all ADR-010-anchored 1:1 wraps of `base:*` subcommands — NOT the read-side surface. `read_property` and `read_heading` are ADR-010 point-3 grandfather composites (eval-composed; no `base:*` anchor) and do not constrain Bases-family naming.
- The `.base` file format is the Obsidian 1.9+ native bases schema. Earlier or incompatible base-like artifacts are out of scope.
- Focused-vault default mirrors the existing read-side cohort (`read_note`, `read_property`, `pattern_search`). No new vault-resolution machinery is introduced.
- No active-file fallback in v1 — the caller must supply an explicit `base_path` (the user input listed active-file targeting as out of scope, behaviour unconfirmed against the live surface).
- No "list all views in a base file" or "list all base files in the vault" capability in v1 — both are deferred to separate features, per the user input's out-of-scope clauses.
- No write-side capability in v1 — creating `.base` files, adding views, or mutating rows is deferred. The operation is read-only and acquires no write queue.
- No new top-level error codes (Constitution Principle IV — the zero-new-codes streak count becomes sixteen tools after BI-039 ships, parity with the FR-016 framing). New states surface via `details.code` sub-discrimination, parity with the existing read-side error contract.
- The `details.code` values for the new states map to existing top-level codes per FR-016: `BASE_NOT_FOUND` and `VIEW_NOT_FOUND` under `CLI_REPORTED_ERROR` (cohort parity with `VAULT_NOT_FOUND` / `NOTE_NOT_FOUND`); `INVALID_BASE_PATH` under `VALIDATION_ERROR`. No new top-level codes are introduced.
- TOCTOU posture matches the read-side cohort — the response reflects vault state at read time; no fingerprinting, no drift detection between repeated invocations.
- Row-count cap (FR-013) is locked at 1000 rows, parity with BI-033 / BI-035, with the truncation signal shape pinned per the Q4 clarification: `truncated: boolean` always present, `total_rows: number` required when truncated and omitted otherwise. Operator-tunable caps were considered and rejected for v1 — a fixed cap keeps the response-shape contract deterministic across deployments.
- Column-value typing (FR-014) is locked at native-type preservation, parity with `read_property` and the read-side cohort. String-coercion was considered and rejected — it would defeat the typed-retrieval value proposition.
